## 1. FAILING test at HEAD: `@quereus/store` DDL generator omits `TEMP`

`yarn test` fails one test in the **`@quereus/store`** workspace (NOT `@quereus/quereus`):

```
@quereus/store › DDL generator › includes TEMP for temporary tables
  AssertionError: expected 'CREATE TABLE "main"."tmp" ("x" INTEGE…' to include 'TEMP'
  at packages/quereus-store/test/ddl-generator.spec.ts:107:19
```

The store's DDL generator emits `CREATE TABLE "main"."tmp" (…)` for a temporary table
instead of a `CREATE TEMP TABLE …` form, so the assertion that the generated DDL
includes `TEMP` fails.

**Why it is pre-existing / not mine:** `git stash`-ing this ticket's entire working tree
and re-running `yarn workspace @quereus/store test` reproduces the identical failure
(287 passing, 1 failing). The store's DDL serializer is unrelated to the optimizer
MV-rewrite rules this ticket changes.

Reproduce:

```
yarn workspace @quereus/store test
```

The `@quereus/quereus` suite itself is fully green (4652 passing, 9 pending, 0 failing),
including this ticket's new aggregate-rollup specs.

## 2. LATENT bug worked around: `WHERE pk_col = const` dropped under `GROUP BY` on a composite-PK table

No existing spec exercises this shape, so it does **not** fail the suite at HEAD — but it
is a real correctness bug discovered while implementing the rollup arm.

A grouped-aggregate query whose `WHERE` filters on a column that is part of a **composite**
primary key but is **not** in the `GROUP BY` silently drops the `WHERE` predicate:

```sql
create table bt (d integer not null, r integer not null, total integer null, primary key (d, r));
insert into bt values (1,10,150),(1,20,null),(2,10,7),(2,20,3);

select d, sum(total) from bt where r = 10 group by d;
--   actual:   [1,150] | [2,10]      (the `r = 10` filter is ignored: 2 → 7+3 = 10)
--   expected: [1,150] | [2,7]       (only r=10 rows: 2 → 7)
```

The same query against a single-column-PK table returns the correct rows, so the trigger
is the composite PK (a non-leading PK member filtered to a constant and absent from the
`GROUP BY`). Likely an FD/constant-binding interaction (`r = const` + PK `(d, r)` ⇒ `d`
becomes a key) in the aggregate/predicate optimization path
(`rule-groupby-fd-simplification` / predicate pushdown / filter elimination) that drops
the filter as "redundant" when it is not. Reproduces with **no materialized view**:

```
node --import ./packages/quereus/register.mjs -e "import('./packages/quereus/src/core/database.ts').then(async ({Database})=>{const db=new Database();await db.exec('create table bt (d integer not null, r integer not null, total integer null, primary key (d, r))');await db.exec('insert into bt values (1,10,150),(1,20,null),(2,10,7),(2,20,3)');for await (const row of db.eval('select d, sum(total) from bt where r = 10 group by d')) console.log(JSON.stringify(Object.values(row)));await db.close();})"
```

**Work-around applied in this ticket:** the aggregate-rollup rewrite re-aggregates over the
MV's backing, whose PK is the (often composite) MV group key, so a rollup with a residual
`WHERE` over the backing produces exactly this bug-triggering shape. The matcher therefore
**forgoes a rollup that needs a residual filter** (`matchAggregateFragmentToMv` →
`fail('rollup-residual')`). Exact-key rewrites (a direct scan, no `GROUP BY`) answer residual
queries correctly and are unaffected. Once the base bug is fixed, that forgo can be relaxed
and the equivalence harness extended with rollup-plus-residual shapes.
