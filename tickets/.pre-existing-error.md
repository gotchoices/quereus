# Pre-existing test failure

## Command

```
yarn test:store
```

(equivalently, isolated: `yarn workspace @quereus/quereus test --store --grep "41.2-alter-column"`)

Run from repo root. `test:store` re-runs the `packages/quereus` SQL-logic suite against
the LevelDB store module.

## Failing test

- **File:** `packages/quereus/test/logic/41.2-alter-column.sqllogic`
- **Test:** `SQL Logic Tests (Store Mode) › File: 41.2-alter-column.sqllogic › should execute statements and match results or expected errors`
- **Location:** assertion at `41.2-alter-column.sqllogic:112`

Excerpt of the failing block (section 7, "SET DATA TYPE text -> integer converts existing rows in place"):

```sql
create table t_conv (id integer primary key, v text);
create index t_conv_v on t_conv (v);
insert into t_conv values (1, '10'), (2, '9');
alter table t_conv alter column v set data type integer;

select id, v, typeof(v) from t_conv order by id;   -- passes
select id from t_conv where v = 9;                   -- FAILS here (line 112)
→ [{"id": 2}]
```

Error:

```
Error: [41.2-alter-column.sqllogic:112] Row count mismatch. Expected 1, got 0
Block:
select id from t_conv where v = 9;
```

After `alter column v set data type integer`, the (full, non-partial) index `t_conv_v`
does not find the row by its new numeric value in store mode — the index scan for `v = 9`
returns 0 rows. The preceding `select ... typeof(v)` confirms the column values were
converted to integers, so the defect is in the store backend's secondary-index rebuild
after an in-place column type conversion, not in the value conversion itself.

## Why this is pre-existing (not from this ticket)

- **Store-mode only.** The exact same file **passes in the default memory-backed run**
  (`yarn test` — the whole logic suite is green, 6920 passing; and
  `yarn workspace @quereus/quereus test --grep "41.2-alter-column"` passes). It fails only
  under `--store` (LevelDB).
- **My diff cannot reach this path.** This ticket
  (`bug-partial-index-predicate-ignores-table-qualifier`) only changes
  `compilePredicate` (adds an optional owning-table-name arg + a foreign-`table`-qualifier
  rejection) and its call sites, which pass the owning table name. Every call site is
  guarded by `indexSchema.predicate ? compilePredicate(...) : undefined` (or the equivalent
  early `if (!uc.predicate) return`). The failing test's index `t_conv_v (v)` has **no
  partial `WHERE` predicate**, so `compilePredicate` is never invoked on its code path —
  the change is inert here.
- The failure is in `alter column ... set data type` + full-index rebuild on the store
  backend, a subsystem this ticket does not touch.
