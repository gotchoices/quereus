# Pre-existing test failure: flaky HLC `receive()` clock-merge test

## Command
```
yarn workspace @quereus/sync test
```
(package `@quereus/sync`, run from repo root)

## Failing test
`HLC (Hybrid Logical Clock) > HLCManager > receive > should ignore remote opSeq when merging the clock`
(`packages/quereus-sync/test/clock/hlc.spec.ts:264-281`)

Excerpt:
```
AssertionError: expected 1781584146150n to equal 1781584146149n
  -1781584146150n
  +1781584146149n
  at Context.<anonymous> (packages\quereus-sync\test\clock\hlc.spec.ts:276:32)
```

## Why it is pre-existing (not this ticket's diff)
- The test was introduced by the **foundation** ticket (commit `56f73fdb`,
  `ticket(implement): sync-hlc-opseq-foundation`), not by this ticket's implement
  commit `ad38c281` — which never touches `hlc.spec.ts` (verify:
  `git show --stat ad38c281` lists no `hlc.spec.ts`).
- It is a genuinely **flaky, timing-dependent** test, independent of this ticket's
  changes. Root cause: the test creates two `HLCManager`s and calls `receive()` on
  each (`hlc.spec.ts:273-274`). `HLCManager.receive()` reads `BigInt(Date.now())`
  (`hlc.ts:315`) and, because both managers' `wallTime` (1000n) and the remote
  `wallTime` (2000n) are far below real wall-clock time, `maxWall` resolves to
  `now` in both calls. The test then asserts `r2.wallTime === r1.wallTime`
  (`hlc.spec.ts:276`). When the two `Date.now()` reads straddle a millisecond
  boundary, the wallTimes differ by 1 and the assertion fails.
- Confirmed empirically: the file passes 6/6 in isolation (both reads land in the
  same ms) but failed once inside the full suite where a 1 ms boundary fell
  between the two `receive()` calls.

## Work-around applied
None. The failure is intermittent and does not affect this ticket's tests
(`transaction-commit.spec.ts` and the migrated specs all pass). No code or test
was disabled.

## Suggested fix (for the triage pass)
Make the test deterministic — assert that `receive()` ignores remote `opSeq` for
the *clock merge* without depending on `Date.now()` coincidence. Options:
- Seed both managers' `wallTime` **above** the current wall clock (e.g. a far-future
  `wallTime`) so `maxWall` derives from the stored/remote clock, not `now`, making
  both `receive()` results equal by construction; or
- Compare the two results' `counter`/relationship rather than their absolute
  `wallTime`, or inject a fixed clock.
