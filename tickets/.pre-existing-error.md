## Failing test

`view persistence: generateMaintainedTableDDL fixed point` ›
`always emits a fully-qualified (schema.name) MV name; explicit-default USING normalizes away`
— `packages/quereus/test/view-mv-ddl-persistence.spec.ts:627`

Error excerpt:

```
AssertionError: expected 'CREATE TABLE "main"."v" (PRIMARY KEY ()) maintained as select 1 as a'
  to match /^create materialized view main\.v /i
```

`generateMaintainedTableDDL` is stringifying a maintained table as
`CREATE TABLE … maintained as …` instead of the expected
`create materialized view main.v …` form, so the whole
`generateMaintainedTableDDL fixed point` block (DDL round-trip / fixed-point
assertions) is red.

## Why it is pre-existing (not from this ticket)

- This ticket's diff is confined to the **refresh reshape** path
  (`packages/quereus/src/runtime/emit/materialized-view-helpers.ts`:
  `classifyBackingReshape` / `ReshapePlan` / `reshapeOpToChange` /
  `reshapeBackingInPlace`) plus two test files. It does **not** touch MV/DDL
  stringification (`generateMaintainedTableDDL`, `ast-stringify`, the view
  persistence emitter).
- Verified by `git stash` of all three changed files and re-running
  `view-mv-ddl-persistence.spec.ts` on the clean tree: the
  `generateMaintainedTableDDL fixed point` failures reproduce unchanged, so they
  predate this work.
- This is the same MV-DDL-stringify subsystem the ticket's Phase 4 note already
  called out (`mv-rename-propagation.spec.ts:70`, "regenerated DDL re-parses").

## Store-side fallout from the same root cause

The full `quereus-store` suite
(`node --import ./packages/quereus-store/register.mjs … "packages/quereus-store/test/**/*.spec.ts"`)
is also red in the same subsystem — **22 failing, all rehydrate/round-trip**:

- `packages/quereus-store/test/mv-rehydrate-adopt.spec.ts` — 14 failing
  (`materialized-view adopt-without-refill at rehydrate`)
- `packages/quereus-store/test/view-mv-persistence.spec.ts` — 8 failing
  (`StoreModule view / materialized-view persistence`)

Both rehydrate a persisted catalog by **re-parsing the generated MV DDL**, so the
`generateMaintainedTableDDL` defect above (emits `CREATE TABLE … maintained as …`
instead of `create materialized view …`) breaks the round-trip and these
rehydration assertions. Confirmed pre-existing: with all three of this ticket's
files stashed and `@quereus/quereus` `dist` rebuilt clean, both files reproduce
the identical 14 / 8 failure split.

## Work-around taken

None needed for this ticket — every failure is outside the reshape diff
(DDL stringify / catalog rehydration), confirmed reproducible on the clean tree.
The new reshape regression tests
(`packages/quereus/test/materialized-view-refresh-reshape.spec.ts`,
`packages/quereus-store/test/mv-store-backing.spec.ts`) all pass.
