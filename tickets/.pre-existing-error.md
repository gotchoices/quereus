# Pre-existing: CHECK constraint added via `ALTER TABLE … ADD CONSTRAINT` cannot be dropped or renamed by name

**Commands run** (from `packages/quereus`, against a freshly built `dist`):

```js
import { Database } from './dist/src/index.js';
const db = new Database();
await db.exec('create table x (id integer primary key, v integer not null)');
await db.exec('alter table x add constraint chk check (v > 10)');   // OK — catalog shows checkConstraints: ["chk"]
await db.exec('alter table x drop constraint chk');                 // ERR
await db.exec('alter table x rename constraint chk to chk2');       // ERR (same scenario on a fresh table)
```

**Failure:**

```
Named constraint 'chk' not found in table 'x'
```

(thrown by `resolveNamedConstraintClass` in `packages/quereus/src/schema/table.ts`, reached from
`runDropConstraint` / `runRenameConstraint` in `packages/quereus/src/runtime/emit/alter-table.ts`).

**Scope observed:**
- Affects only CHECK constraints added via `ALTER TABLE … ADD CONSTRAINT`. The same
  add→drop roundtrip with a UNIQUE constraint works (`alter table x2 add constraint uq unique (v)`
  → `drop constraint uq` → OK).
- CHECK constraints declared inline at `CREATE TABLE` drop/rename fine (covered green by
  `test/logic/41.6-alter-drop-rename-constraint.sqllogic`).
- The catalog record *does* carry the added constraint (`getTable(...).checkConstraints`
  lists `chk`, and it keeps enforcing inserts) — so the drop/rename path appears to resolve
  the table through a different (module-side / pre-add) `TableSchema` instance than the
  catalog one. Suspect: ADD CONSTRAINT CHECK updates the catalog without routing through
  `module.alterTable` (no index to build, unlike UNIQUE), leaving the module-cached schema
  that DROP/RENAME later resolves against without the constraint.

**Why pre-existing:** reproduced at HEAD (`469878b5`, with my working tree stashed and
`dist` rebuilt from clean) with **no materialized view involved at all** — the
mv-staleness ticket touches only the MV manager's schema-change listener
(`core/database-materialized-views.ts`) and MV helpers (`runtime/emit/materialized-view-helpers.ts`),
neither of which executes in the repro. No existing test exercises add-CHECK-then-drop
(41.6 drops inline-declared constraints; 41.8 adds UNIQUE/FK).

**Work-around taken in my ticket:** the new
`test/logic/53.3-materialized-view-constraint-only-ddl.sqllogic` declares every
to-be-dropped/renamed constraint inline at `CREATE TABLE`, and its ADD CONSTRAINT CHECK
section only asserts post-add behavior (never drops the added CHECK). No code was
disabled.
