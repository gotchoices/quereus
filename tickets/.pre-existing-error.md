# Pre-existing error: prior plan commit deleted the entire `packages/quereus` engine

## What happened

The immediately-preceding tess commit on branch `view-updates-lens`:

```
1cf7326c ticket(plan): view-write-decomposition-optional-update-hardening
```

was a **plan-stage** ticket transition (it should only have moved/created ticket
files under `tickets/`). Instead it also **deleted the entire `packages/quereus`
main SQL-engine package** — 901 files changed, **250,548 deletions** — leaving
`packages/quereus/` empty and untracked at HEAD. The working tree was clean
(deletion committed). `packages/quereus` still exists intact on `main` and `dev`,
and in this branch's parent commit `3a5a7ef5`.

This was discovered while starting the implement ticket
`1-view-write-decomp-stitch-key-unique-guard`, whose target files
(`packages/quereus/src/schema/lens-compiler.ts`, etc.) had vanished.

## Verification it is pre-existing (not mine)

- `git show --stat 1cf7326c` → 901 files, `250548 deletions`, deleting
  `packages/quereus/package.json`, `src/**`, `test/**`, `bench/**`, etc.
- `git ls-tree -r HEAD -- packages/quereus | wc -l` → `0` before my work.
- `git ls-tree -r 3a5a7ef5 -- packages/quereus | wc -l` → `898` (full package).
- `git ls-tree main -- packages/quereus` and `dev` → present and intact.

## Work-around applied (so my ticket could proceed)

I restored the package from the parent commit, which is byte-identical to the
pre-deletion state:

```
git checkout 3a5a7ef5 -- packages/quereus
```

`git diff --stat 3a5a7ef5 -- packages/quereus` is now empty. My ticket's
stitch-key-uniqueness-guard changes are layered on top of this restoration, so
the commit for this ticket will (a) restore the accidentally-deleted engine and
(b) add the guard. The legitimate parts of `1cf7326c` (the ticket-file moves)
are untouched.

## Suggested follow-up for triage

Confirm no *other* state was lost in `1cf7326c` (only `packages/quereus` and the
ticket-file moves appear in its diff). Consider why a plan-stage runner produced
a 250k-line deletion — it may indicate a `git add -A` after a transient empty
working tree, and could recur on other branches/runs.
