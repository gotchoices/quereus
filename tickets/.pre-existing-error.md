## Why it is pre-existing (not mine)
- The failure is at **DB-open** in the LevelDB store/catalog layer — a native file-lock acquisition on a freshly-created temp dir — *before any SQL in the file runs*. It is an `IO error: LockFile ... being used by another process`, not an assertion mismatch.
- It reproduces in **complete isolation** (`--grep "41.3-alter-rename-propagation"`), with my new test file not even loaded.
- It still reproduces after **stashing all my changes and rebuilding the engine at the committed HEAD** (`534f08a3`) — identical error.
- My diff touches only the **memory** module's `dropColumn` (`packages/quereus/src/vtab/memory/layer/manager.ts`), the constraint-builder/alter-table emit, and `.sqllogic`/docs — none of which is reachable at store DB-open / LevelDB lock acquisition. A memory-module edit cannot affect a LevelDB catalog `LOCK`.
- This ticket's own new store assertions pass in isolation: `yarn test:store --grep "41.3-alter-add-column-unique"` → 1 passing.

Almost certainly a Windows + LevelDB environmental lock-contention flake (e.g. real-time AV/indexer briefly holding the just-created `LOCK` file, or a native handle not yet released across the per-file `afterEach`/`beforeEach` boundary).
