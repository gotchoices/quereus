# Pre-existing error: memory vtab consolidation drops deletes when base is already populated

Surfaced while implementing `10.25-module-add-constraint-unique-fk` (memory `ADD CONSTRAINT`
re-validation + retry). The failure is **not caused by that feature** — it reproduces using only
pre-existing operations (`create index`, `alter table add column`, `delete`, `create unique index`).

## Test command

From repo root (built dist):

```
node --import ./packages/quereus/register.mjs node_modules/mocha/bin/mocha.js \
  "packages/quereus/test/alter-add-constraint.spec.ts" --colors
```

The `UNIQUE › rejects an add over duplicated existing data and leaves the constraint absent`
case tripped it, but the root cause is independent of the new feature (see standalone repro below).

## Minimal standalone repro (no new feature involved)

```js
import { Database } from './packages/quereus/dist/src/index.js';
const db = new Database();
await db.exec('create table t (id integer primary key, email text, v integer)');
await db.exec("insert into t values (1, 'a@x', 1), (2, 'b@x', 2), (3, 'a@x', 3)");
await db.exec('create index iv on t (v)');     // schema change #1 → consolidates committed layer into base = {1,2,3}
await db.exec('delete from t where id = 3');    // tombstones id=3 in a committed transaction layer
// select shows {1,2} correctly here
await db.exec('create unique index ue on t (email)');  // schema change #2 → consolidation
//   → throws "UNIQUE constraint failed: t (email)"
//   because base still physically contains the deleted id=3 ('a@x'), colliding with id=1 ('a@x')
```

`alter table t add column note text null` in place of `create index iv` reproduces it identically.
A normal `select` returns the correct rows (the layer stack applies the tombstone); only the
*physical base primary tree* is wrong, which is what an index build scans.

## Root cause

`packages/quereus/src/vtab/memory/layer/manager.ts` → `copyTransactionDataToBase` (~L2302).

When a schema change consolidates the current committed transaction layer into the base layer, the
method iterates the layer's merged view (deletes correctly applied) and **inserts** every row into
`this.baseLayer.primaryTree`. It never **removes** base rows that are absent from that merged view
(i.e. rows deleted in the transaction layer). This is harmless the first time (base starts empty, so
`insert`-only yields the correct set), but once base has been populated by a prior consolidation, a
later consolidation that includes a delete leaves the deleted row physically resident in base. The
trailing `rebuildAllSecondaryIndexes()` then rebuilds indexes from the corrupted base, so the stale
row participates in UNIQUE enforcement (and would be visible to any base-direct scan).

## Suggested fix

`allRows` collected in `copyTransactionDataToBase` is the complete merged view of the layer (it
already reflects deletes — verified by the empty-base case working). So the base primary tree should
be **replaced** by `allRows`, not unioned with it: clear/rebuild `baseLayer.primaryTree` from a fresh
tree and insert `allRows`, then `rebuildAllSecondaryIndexes()` as today. (A `BaseLayer` helper that
rebuilds its primary tree from a row list keeps the manager from reaching into the tree directly.)

Impact is broader than this ticket: any DELETE followed by a schema change (after base was populated
by an earlier schema change) can resurrect the deleted row.

## Work-around applied in this ticket

The new tests avoid asserting the "delete the offending row, then re-run the same ADD CONSTRAINT on
the same table ⇒ succeeds" chain, since that path depends on the buggy consolidation. The atomic
rejection + constraint-absent + not-enforced behavior of `ADD CONSTRAINT` is still asserted (it does
not depend on this bug). The ticket's stated "re-run after removing the dup ⇒ succeeds" edge case is
deferred to this fix. No production code was changed to mask the failure.
