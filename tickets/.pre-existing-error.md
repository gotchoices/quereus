# Pre-existing failure: "No row context" projecting only a computed column while filtering on a pruned column

**Command:** `node test-runner.mjs --grep "authored"` (from `packages/quereus`) — surfaced while adding
`test/logic/93.5-authored-inverse.sqllogic` for ticket `authored-inverse-write-path`.

**Failing statement shape** (no new syntax involved):

```sql
create table t (id integer primary key, code20 text not null);
insert into t values (3, 'C1');
create view v_plain as select id, case code20 when 'A1' then 'A' else 'C' end as code from t;
select code from v_plain where id = 3;   -- ✗
```

**Error:**

```
QuereusError: No row context found for column code. The column reference must be
evaluated within the context of its source relation.
    at resolveAttribute (src/runtime/context-helpers.ts:158)
    at Object.run (src/runtime/emit/column-reference.ts:4)
    ... via runtime/emit/project.ts
```

The same shape fails without a view (`select code from (select id, case … end as code from t) where id = 3`),
and works when the filtered column is also projected (`select id, code from v_plain where id = 3`),
when there is no WHERE, or when the WHERE references the projected column itself. So the trigger is:
**outer projection keeps ONLY a computed column while the WHERE references a different (pruned) column** —
likely a project-merge / pushdown interaction dropping the row context for the merged computed expression.

**Why pre-existing:** reproduced byte-identically at HEAD (`6a9a40b7`, branch `view-updates-lens`) with all
ticket changes stashed and a clean `yarn build` (exit 0). None of the ticket's changes are on the failing
path (no `with inverse` clause involved).

**Workaround taken for the ticket:** the new `93.5-authored-inverse.sqllogic` avoids the shape — it selects
`id, code` (or filters on the projected column) instead of `select code … where id = …`. No code disabled.
