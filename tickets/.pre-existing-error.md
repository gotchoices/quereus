# Pre-existing test failure (flagged by ticket `store-external-row-write`)

## Command

From repo root:

```
yarn workspace @quereus/store run test
```

(equivalently: `node --import ./packages/quereus-store/register.mjs node_modules/mocha/bin/mocha.js "packages/quereus-store/test/view-mv-persistence.spec.ts" "packages/quereus-store/test/mv-rehydrate-adopt.spec.ts"`)

## Result

22 failing, all in two specs in `packages/quereus-store/test/`:

- `mv-rehydrate-adopt.spec.ts` — "materialized-view adopt-without-refill at rehydrate" (14 failures)
- `view-mv-persistence.spec.ts` — "StoreModule view / materialized-view persistence" (8 failures)

Representative failures / error output:

```
1) materialized-view adopt-without-refill at rehydrate
   adopts a store backing after a clean shutdown: the sentinel survives and serves:
   AssertionError: expected [] to deeply equal [ 'main.mv' ]

15) StoreModule view / materialized-view persistence
   an MV over a store table rebuilds its backing on reopen and keeps maintenance live:
   clean rehydrate: expected -1 to equal +0   (rehydrated table count 0, expected 1)
```

Underlying cause surfaced in stderr during rehydration:

```
[StoreModule] Failed to rehydrate DDL entry, skipping: Module 'memory' connect failed
during import for table 'mv': Memory table definition for 'mv' not found. Cannot connect.
  DDL: CREATE TABLE "main"."mv" (...) maintained as select id, v from base
```

i.e. a `maintained as` (materialized-view backing) catalog entry fails to reconnect
through the `memory` module on reopen, so no backings are rehydrated/adopted.

## Why this is pre-existing (not caused by this ticket)

- This ticket adds `StoreTable.applyExternalRowChanges` / `readRowByPk`,
  `StoreModule.getTableForExternalWrite`, an `ExternalRowOp` export, and a new
  `external-row-write.spec.ts`. None of that touches MV rehydration, catalog
  import, the `memory` module, or `maintained as` DDL.
- Confirmed by reproduction: with this ticket's source changes `git stash`-ed
  away, the SAME 22 failures occur (18 passing / 22 failing) at HEAD
  (`3949fa1a`). The failures are independent of this diff.

## Work-around

None required. This ticket's own new tests (`external-row-write.spec.ts`, 8/8
passing) and the unrelated store specs are green; the 22 MV-rehydration failures
were neither introduced nor worked-around here.
