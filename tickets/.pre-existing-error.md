# Pre-existing test breakage from ticket 6.1 (maintained-table-unified-model)

Two distinct pre-existing problems, both landed by commit `a203b3dc`
(`ticket(implement): maintained-table-unified-model`) and **not** caused by this
ticket (6.5 identity-preserving reshape). The first I fixed in-passing because it
blocks the *entire* suite from loading; the second I flag here per the
pre-existing-failure rules (it is a partly-structural DDL-format test migration,
outside this ticket's scope).

## Command
`yarn test` (root) / `yarn workspace @quereus/quereus test`
→ `mocha "packages/quereus/test/**/*.spec.ts" --bail` (loads all spec files first)

---

## 1. Stale import of a renamed export — FIXED in-passing (suite-blocking)

6.1 renamed the DDL generator export `generateMaterializedViewDDL` →
`generateMaintainedTableDDL` (`packages/quereus/src/schema/ddl-generator.ts`) but
left four files importing the old name. Because mocha loads every `*.spec.ts`
before running, the load-time `SyntaxError: ... does not provide an export named
'generateMaterializedViewDDL'` **aborts the whole quereus suite — zero tests
run**:

- `packages/quereus/test/mv-backing-module.spec.ts`
- `packages/quereus/test/mv-rename-propagation.spec.ts`
- `packages/quereus/test/view-mv-ddl-persistence.spec.ts`
- `packages/quereus/test/util/schema-equivalence.ts` (imported by equivalence specs)

**Fix applied:** mechanical rename of the stale import (and doc-comment mentions)
`generateMaterializedViewDDL` → `generateMaintainedTableDDL` in those four files.
The new export is the exact same `(table: MaintainedTableSchema) => string`
function — no behavior change. This was required just to let *any* test
(including this ticket's own) run.

---

## 2. Stale `createMaterializedView` DDL-format expectations — FLAGGED (not fixed)

With the imports fixed, the suite now loads and 2157 tests pass, but two of the
unmasked files assert the **old** canonical-DDL form. 6.1 changed the canonical
maintained-table DDL from `create materialized view …` to the unified
`create table T (…) [using …] maintained as <body> …` (the parser's
`maintained as` clause — `ast.ts:300`, `parser.ts:2567`), so
`generateMaintainedTableDDL` now emits `CREATE TABLE …`, which `parse()` returns
as `type: 'createTable'`. These assertions were never updated and now fail:

- `packages/quereus/test/mv-rename-propagation.spec.ts:70`
  `expect(parse(ddl).type).to.equal('createMaterializedView')` — should be
  `'createTable'`. (This file's other DDL checks are substring-based and pass.)
- `packages/quereus/test/view-mv-ddl-persistence.spec.ts` — 14 failing
  (lines 554, 623, 627, 635, 642, 652 [a 9-iteration fixed-point loop], 706, 875):
  `parse(...).type === 'createMaterializedView'`, `^create materialized view`
  regexes, and a couple of **structural** assertions reading `.tags` / `.columns`
  off the parsed node, which differ in shape between a `createMaterializedView` and
  a `createTable` AST node (so they need real rework, not a pure find/replace).

Authoritative count: the full quereus suite (no `--bail`) is **5859 passing /
15 failing** — the 1 + 14 above, **all** pre-existing 6.1 DDL-format mismatches.
This ticket's own changes introduce **zero** new failures (one genuine reshape
regression they did cause — `query-rewrite-join.spec.ts` "select* join
ALTER+refresh" — was reworked to the new in-place semantics and passes).

### Failing test names (first `--bail` stop)
`MV rename propagation: derived fields and events › TABLE rename re-keys
sourceTables/bodyHash/sql and fires materialized_view_modified` — assertion
"regenerated DDL re-parses" (`createTable` actual vs `createMaterializedView`
expected). `view-mv-ddl-persistence` fails the same way further along.

### Why pre-existing / not chased here
The canonical DDL form is a 6.1 design decision (confirmed: the parser accepts
`create table … maintained as …`, `schema()` and catalog persistence emit via
`generateMaintainedTableDDL`). Updating these assertions correctly — especially
the structural `.tags`/`.columns` reads against the `createTable` AST — belongs to
finishing 6.1's test migration, not to identity-preserving reshape. Per the
pre-existing-failure rule this ticket flags it rather than chasing it.

### quereus-store: same root cause
`yarn workspace @quereus/store test` fails **22** tests at HEAD, all in
`packages/quereus-store/test/view-mv-persistence.spec.ts` ("StoreModule view /
materialized-view persistence") — MV catalog persistence / rehydration, which
round-trips the canonical maintained-table DDL. Verified pre-existing: the count
and set are **identical** with and without this ticket's changes (502 passing /
22 failing in both, since the reshape path this ticket touches is not exercised by
the persistence/rehydrate path). Same 6.1 DDL-migration root cause; flagged, not
fixed.

### Work-around for *this* ticket's validation
This ticket's own changes were validated by running the relevant specs directly
(they sit alphabetically before the first `--bail` stop, so they also run in the
full pass): `materialized-view-refresh-reshape.spec.ts` (9 passing),
`logic.spec.ts` (incl. `51-materialized-views.sqllogic`), and
`mv-backing-module.spec.ts`. The quereus-store MV suites were run via
`yarn workspace @quereus/store test`.
