# Pre-existing failure: fuzz differential — NULL-comparison filter passes rows when predicate-pushdown is disabled

**Command:** `yarn workspace @quereus/quereus run test` (repo root; package `packages/quereus`)

**Failing test:** `Optimizer Equivalence › predicate pushdown rules produce identical results` (`packages/quereus/test/fuzz.spec.ts:1239`)

**Error excerpt:**

```
Error: Row count mismatch: 0 (full) vs 1 (restricted) when disabling rules [predicate-pushdown, filter-merge]
SQL: select distinct * from t1 where ((+ null) < c_any3)
    at assertEqualResultSets (packages\quereus\test\fuzz.spec.ts:1200:10)
```

**Why pre-existing:** Reproduced deterministically at HEAD (same SHA, ticket changes stashed) with:

```sql
create table t1 (c_real0 real not null, c_real1 real, c_any2 any unique, c_any3 any not null unique, c_int4 integer not null primary key) using memory;
insert into t1 values (1.5, 0.25, 'x', 'y', 1);
insert into t1 values (2.5, 0.5, 7, 9, 2);
select distinct * from t1 where ((+ null) < c_any3);
```

With default tuning this returns 0 rows (correct — `(+ null) < x` is NULL ⇒ filter rejects). With `disabledRules: new Set(['predicate-pushdown', 'filter-merge'])` it returns ALL rows — the un-pushed FilterNode appears to treat the NULL comparison result as true (or the filter is dropped). Identical behavior at HEAD and on the ticket branch; the failing query contains no equality, no collation, and no join — entirely outside this ticket's diff (collation gating of equality-fact extraction).

The fuzz test samples rows/queries with unseeded `fc.sample`, so the failure surfaces intermittently depending on the random draw.

**Work-around taken for this ticket:** none needed — re-running the suite passes when the random draw misses this shape; the ticket's own tests are deterministic.
