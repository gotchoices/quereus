# Pre-existing store-mode test failure

## Test command

From `packages/quereus`:

```
QUEREUS_TEST_STORE=true node test-runner.mjs --store
```

(equivalently `yarn test:store` from the repo root). The runner passes `--bail`,
so it stops at the first failure. Running mocha directly **without** `--bail`
confirms this is the ONLY failure across the whole store-mode suite:

```
QUEREUS_TEST_STORE=true node --import ./packages/quereus/register.mjs \
  node_modules/mocha/bin/mocha.js "packages/quereus/test/**/*.spec.ts" --reporter min
# => 6324 passing, 14 pending, 1 failing
```

## Failing test

- Suite: `SQL Logic Tests (Store Mode)`
- File: `packages/quereus/test/logic/51.7-maintained-table-attach-detach.sqllogic`
- Failing statement: `alter table tgt add column extra text null;` / the following
  `alter table tgt drop column extra;` (a structural ALTER expected to SUCCEED
  after the table was detached via `alter table tgt drop maintained`).

Error:

```
QuereusError: cannot ALTER 'tgt': it is a materialized view — its shape is
defined by the view body (drop and recreate to change it)
  at Object.run (.../runtime/emit/alter-table.ts:52)
```

## Why this is pre-existing (not caused by ticket 3-store-leveldb-shared-root)

Root cause: after `alter table tgt drop maintained` (detach), the schema-manager
schema for `tgt` is correctly de-maintained (`derivation` cleared), but a
subsequent structural ALTER (`add column`) re-introduces the maintained
`derivation` on the rebuilt schema. The next ALTER then sees `tgt` as a
materialized view again and rejects it. This is in the StoreModule + isolation
layer's `drop maintained` → `add column` schema-reconstruction path
(`packages/quereus-store`, `packages/quereus-isolation`, engine
`runtime/emit/alter-table.ts`) — **none of which this ticket touches**.

It was isolated to a backend-agnostic StoreModule/isolation bug by running the
exact failing scenario through **both** an in-memory `KVStoreProvider` (built over
`InMemoryKVStore`) and the LevelDB provider, wrapped in the same
`createIsolatedStoreModule`. **Both backends fail identically**, so the KV
backend (the only thing this ticket changed) is not implicated.

Why it surfaces only now: the store-mode suite runs with `--bail`. This ticket's
shared-root rewrite fixed two genuinely-new failures that previously bailed
earlier in the run (a sublevel-name-byte-range rejection for a table named
`"Table With Spaces"`, and a closed-handle reuse in `dropIndex`). With those
fixed, the suite now reaches `51.7` and exposes this latent, pre-existing failure.
On `main` the suite would reach `51.7` too (the old per-directory layout never hit
the two new failures), so `51.7` is red on `main` in store mode as well.

## Work-arounds applied

None. The failure is a single isolated `.sqllogic` file and does not affect the
LevelDB shared-root work. No test was disabled. The rest of the store-mode suite
(6324 passing) validates this ticket's changes.
