# Pre-existing: table-qualified CHECK refs are accepted at DDL time but unenforceable at runtime

## Command / context

While implementing `schema-differ-predicate-table-qualifier-reconcile` (differ-side
reconcile of qualified CHECK / partial-WHERE refs), new end-to-end tests in
`packages/quereus/test/declarative-equivalence.spec.ts` tried to verify post-apply
CHECK *enforcement* via inserts and hit a runtime failure. Repro (HEAD, no renames,
no declarative schema involved):

```js
// node --import ./packages/quereus/register.mjs <script>, from the repo root
import { Database } from './packages/quereus/src/core/database.js';
const db = new Database();
await db.exec(`create table t (id integer primary key, qty integer, constraint chk check (t.qty > 0))`);
await db.exec('insert into t values (1, 5)');
// → QuereusError: t.qty isn't a column
```

## Failure

`insert into t …` on ANY table whose CHECK expression uses a **table-qualified
self-reference** (`check (t.qty > 0)`) fails at plan-build time:

```
QuereusError: t.qty isn't a column
    at resolveColumn (src/planner/resolve.ts:55)
    at buildExpression (src/planner/building/expression.ts:82)
    at buildConstraintChecks (src/planner/building/constraint-builder.ts:46)
    at buildInsertStmt (src/planner/building/insert.ts:539)
```

The DDL parses and registers fine; the differ canonicalizes/diffs the qualified form
fine (folded by `lowerExprIdentifiers`, exercised by existing tests); but the
constraint-check build scope only registers bare column names for the NEW-row
context, so the qualifier never resolves and the table is effectively read-only.

## Why pre-existing

Fails on the direct `create table` + `insert` path with no schema-differ involvement
whatsoever — the failing frames (`planner/resolve.ts`, `planner/building/
constraint-builder.ts`) are untouched by this ticket, which only modified
`schema/schema-differ.ts`, `planner/mutation/scope-transform.ts` (a clone-depth fix),
and two test files. The existing qualified-CHECK test (`a TABLE-qualified CHECK ref
case change … does not churn`) only diffs and never inserts, which is why the gap was
latent.

## Workaround taken in this ticket

The three new qualified-CHECK rename tests in `declarative-equivalence.spec.ts`
assert the stored catalog `namedConstraints[].definition` after apply (forward
propagation + convergence — the ticket's actual subject) instead of asserting
runtime enforcement via inserts. Enforcement coverage remains on the unqualified
CHECK tests, which are unaffected. Suggested fix direction for triage: seed the
constraint-check build scope with the owning table's name as a qualifier (the same
implicit binding `renameColumnInCheckExpression` uses), or reject qualified
self-refs at CREATE time rather than at first write.
