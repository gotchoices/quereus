# Pre-existing test failure: sync-coordinator StoreManager LRU eviction

## Command

From the repo root:

```
yarn test
```

The failure surfaces in the `@quereus/sync-coordinator` workspace. Reproduced (once) with:

```
yarn workspace @quereus/sync-coordinator test
```

## Failing test

`packages/sync-coordinator/test/store-manager.spec.ts:151`

```
1) StoreManager
     LRU eviction
       should evict LRU store when maxOpenStores reached:

    AssertionError: expected 2 to equal 3
    + expected - actual

    -2
    +3

    at Context.<anonymous> (packages\sync-coordinator\test\store-manager.spec.ts:151:36)
```

The test opens three stores (`db-a`, `db-b`, `db-c`), releasing each immediately, then asserts
`manager.openCount === 3` before opening a fourth to force an eviction. It observed `2`, so one
of the three released stores had already been closed before the assertion ran.

## Why this is pre-existing

It is a **flake, not a deterministic failure**, and it cannot be caused by the diff under review.

- The ticket that produced this diff (`docs-mv-split`) changes only Markdown files under `docs/`
  and comment text in source files. It touches nothing under `packages/sync-coordinator` at all —
  not source, not tests.
- The failure is timing-dependent: `yarn test` was run three times from the same working tree.
  Runs 1 and 2 reported `128 passing, 0 failing` for `@quereus/sync-coordinator`; run 3 reported
  `127 passing, 1 failing`. Running the package's suite alone also passed (`128 passing`).
- The shape of the assertion points at a race between `release()` (which drops the refcount to
  zero) and the manager's background close/eviction sweep: a store released at refcount zero can
  be closed before the test asserts on `openCount`, so the count reads `2` instead of `3`.

The test was not skipped, disabled, or loosened.
