**DESIGN RESOLVED (2026-06-07) — promote to implement.** The load-bearing inner-flag exposure
call is decided as **model (b): surface inner flags as readable columns of the outer view**,
excluded from the data-column arity/dedup. Rationale: it is the natural extension of the
existing single-level invariant — flags are already appended after the data columns and kept
out of the union schema/dedup (`set-operation-node.ts:69-71`); model (a) (hide-and-re-derive)
is strictly more complex and loses the row-by-row readability the membership model needs.
Design the implementation against (b).

## Problem

The membership-flag-as-appended-column model does not nest. Two distinct failures, both
verified empirically (in-memory tables A/B/C):

### 1. A flagged subtree operand fails to PLAN (column-count mismatch)

`A union[inA,inSub] (B union[inB,inC] C)` parses fine, but creating the view throws:

```
SET operation column count mismatch: left has 2, right has 4
```

Cause: the inner `(B union[inB,inC] C)` plans to a `SetOperationNode` whose attributes are
`[id, x, inB, inC]` (data columns + its two appended flag columns — `set-operation-node.ts`
`buildAttributes`). The outer constructor validates `left.columns.length === right.columns.length`
(`set-operation-node.ts:47`) — 2 (A) vs 4 (inner subtree) → throw. Any inner flag inflates the
operand's arity and breaks outer alignment.

### 2. A subtree operand is not writable (recursion bottoms out unwritable)

A flag-less subtree operand *does* plan and read correctly (a subtree flag's read already works
generically — the probe is `tuple ∈ <subtree output>`), but every **write** that touches the
subtree branch throws, because the per-branch recursion (`set-op.ts` → `propagate`) reaches the
flag-less `(B union C)` subtree, which is not a writable body:

```
update V set inSub = true   → cannot write through view '__setop_right':
                              view body operator 'SetOperation' is not updateable in phase 1
update V set x = 99          → (same — data fan-out hits the subtree branch)
delete from V where …        → (same)
update V set inA = false     → OK (leaf branch)
```

So a set-op view with a subtree operand is readable but inconsistently writable, and the static
surface still advertises the data columns `is_updatable='YES'` (a catalog-honesty over-claim).

## Desired behavior (model b)

A `SetOperationNode` whose operand is itself a (possibly flagged) `SetOperationNode` is a
first-class, readable, **writable** body:

- **Column alignment compares DATA columns only.** Use `dataColumnCount()`
  (`set-operation-node.ts:60`) for the outer arity check and the union schema, not data+flags.
  Per model (b), an inner operand's flag columns are **surfaced as the outer view's columns**
  under a defined projection rule, but never count toward the data-column set-identity/dedup or
  the outer arity check (the same treatment single-level flags already get — extend the existing
  invariant, don't special-case it).
- **A subtree operand is recursively writable.** Writing a subtree flag (`set inSub = true`)
  recurses into the subtree's own `put` (insert/delete into `(B∪C)`, which itself fans out to
  its leaves); data/delete fan-out through a subtree operand recurses the same way. A flag-less
  `union`/`intersect`/`except` subtree must become writable through the same per-branch fan-out
  the membership path uses.
- **Lineage/static surface.** The subtree-operand `set-op-branch` `existence` `UpdateSite` is
  already registered generically; the static `column_info`/`view_info` surface must agree with
  the now-real writability (and stop over-claiming when a subtree branch is non-writable).

## Edge cases & interactions

- `A union[inA,inSub] (B union[inB,inC] C)` creates, plans, reads: `inA`/`inB`/`inC`/`inSub`
  agree row-by-row with `tuple ∈ <that operand's data relation>`; outer keys stay on data
  columns only (no flag in any claimed key — Key-Soundness Invariants 1–2 at all depths).
- `set inSub = true/false` inserts into / deletes from the `(B∪C)` subtree (recursing to leaves);
  data/delete fan-out reaches every member leaf through subtree operands. No
  `'SetOperation' is not updateable` runtime throw.
- The flag-less-subtree read fixture keeps reading correctly (regression).
- `intersect` / `except` subtree operands (not just `union`) as the inner node.
- Static surface (`column_info`/`view_info`) honesty when a subtree branch is genuinely
  non-writable.
