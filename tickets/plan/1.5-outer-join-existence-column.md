## Why

Outer-join write-through has an inherent ambiguity: a NULL on the non-preserved side means *either* "no matching row exists" *or* "the matched row's value is NULL." Today Quereus carries the join predicate as a **guard** on `null-extended` lineage (`docs/view-updateability.md` § Outer Joins / § The Update Site Model) and must **infer** insert-vs-update intent from which columns a mutation writes. Dataphor's `include rowexists` solved this by manifesting the match as an explicit boolean column; in practice it unlocked clean, unambiguous outer-join updateability.

This reifies the existing guard as a first-class value: the existence column **is** the guard, evaluated to a boolean. Reading it tells you whether the side matched; writing it drives insert/delete of that side. It is the natural control surface for the outer-join write-through that `view-write-through-shape-gaps` wires.

## Model

Add a new `UpdateSite` kind — **`existence`** — for a column whose value is the truth of an outer-join source `S`'s match guard:

- **Read.** The column projects `true` when a row from `S` matched (guard holds), `false` when `S` is null-extended for this row.
- **Write (the payoff).** For a left join `A ⟕ B on <pred>` with existence column `hasB`:
  - `set hasB = true` while B absent → **insert** into B; join-key columns supplied from the preserved side via EC, other B columns from base defaults / `default_for`; fails with the existing `no-default` diagnostic if a NOT-NULL B column has no source. (This is exactly the documented "null-extended ⇒ rewrite as insert" path, now explicitly triggered rather than inferred.)
  - `set hasB = false` → **delete** the matching B row.
  - `set hasB = true` is a no-op when B already matches; `set hasB = false` is a no-op when B is already absent.
  - A column write on B composes with the existence write: `set y = 5, hasB = true` while B absent → insert B with `y = 5`.
  - **Insert** through the view: `hasB = true` ⇒ insert both sides under the join predicate; `hasB = false` ⇒ insert only the preserved side (row is null-extended through the view).
- The existence column itself has no base column — it is `existence`-sited, not `base`/`computed`. `column_info` / `view_info` report it appropriately (writable through its insert/delete effect, but not mapped to a `base_column`).

## The flag is a derived combinator predicate (shared with set-op membership)

The existence value is **derived at the join from the match guard**, never a column stored inside the operand. The tempting desugaring `left join (select *, true as inY from Y)` does **not** work: the outer join null-extends the constant, so `inY ∈ {true, NULL}` — reading it then needs `inY IS NOT NULL` (the very null-extension test the column was meant to replace), and a write cannot cleanly express "absent" (`= false` vs `= NULL`). Computing it at the combinator yields a clean `{true, false}`.

This is the **principle shared with `set-operator-membership-columns`**: an existence/membership flag is a derived predicate at the combinator over operand **data** relations — never a stored operand column, so it never re-enters the operator's own semantics. The two symptoms of getting it wrong are this outer-join `{true, NULL}` case and the set-op dedup/schema-pollution case. The generalized `existence` `UpdateSite` (carrying a join-side *or* set-branch component reference) is the one surface both tickets share — and because membership is defined on the **data** tuple (flags never participate in identity), the flags compose and nest freely.

## FD ramifications

A flag interacts with every FD surface Quereus tracks (`fds` / `equivClasses` / constant FDs / `domainConstraints` / `updateLineage`). Two invariants are **load-bearing**; the rest is reuse.

- **Invariant 1 — key-determined, never in a key.** When the node has a key (a 1:1 join's preserved PK; a distinct union's data columns), `computePhysical` must emit `key → flag` so DISTINCT-elimination / ORDER-BY-trailing-key pruning / join-elimination still fire when a flag is projected. But the flag must **never** be claimed as part of a key — over-claiming there lets those same rules drop real rows. The **Key Soundness** property harness must cover flag-bearing nodes.
- **Invariant 2 — constant-true below the combinator, free boolean above.** Within its own component the flag is a constant FD (`∅ → flag = true`); the combinator *releases* it into a key-determined free boolean. This is the FD statement of "derived at the combinator, never stored in the operand" — the stored-column bug is exactly emitting the constant-true and letting the operator (null-extension / dedup) mutate it.

The rest reuses existing machinery:

- **New attribute, own provenance.** The flag carries information not FD-derivable from the other columns (that is its purpose), so it gets its own attribute ID — never folded into another column's FD nor made an EC member of a base column.
- **Write-default via the branch predicate.** `set flag = true` asserts the component's predicate FDs on the row — the same constant-FD-supplies-defaults path view-updateability already uses (`where color='red'` ⇒ default `color='red'`). No new defaulting mechanism; the flag just needs its branch-predicate bindings attached. Symmetrically `where flag` is a semijoin/guard predicate the normalizer can push down.
- **Domain + NOT NULL.** The flag's domain is `{true,false}` NOT NULL (the clean-boolean point vs `{true,NULL}`) — a boolean `domainConstraint` and a genuinely-NOT-NULL column, consistent with the default-NOT-NULL stance.
- **EC consumed, not created.** The join-existence write pulls the shared key from the join-key equivalence class to supply the absent side; it mints no new EC.
- **Bag vs set.** On a distinct union `key → flag` holds (clean); on `union all` (a bag, no key) it does not, and provenance is per-occurrence — the FD framing of the open bag-semantics question in `set-operator-membership-columns`.
- **Pruning / scope.** An unused flag is a semijoin probe, so dead-column-eliminate it when not selected; a live flag widens `getChangeScope` to every component it probes (one that might otherwise have been join-eliminated).

## Syntax

Anchored at the join (existence is a property of the match; the planner must bind it to the exact side + guard). Recommended canonical form — **reuse the existing `exists` keyword**, no new keyword:

```sql
from A left join B on B.aid = A.id exists as hasB
-- full outer: one per side
from A full outer join B on B.aid = A.id exists left as aEx, exists right as bEx
```

- `exists` is already a reserved keyword (`TokenType.EXISTS`, `parser/lexer.ts`) used only as `exists (<subquery>)` and `if [not] exists`. The clause form `exists [<side>] as <name>` is always followed by `as` or a side token, **never `(`** — so one-token lookahead after `exists` disambiguates the clause from the existence predicate, and the clause appears only after a complete `on` predicate where no new expression can begin (there is no infix `exists`, so a finished predicate cannot absorb a trailing `exists`). This needs **no new keyword** and creates no grammar conflict — preferred over an `include` contextual keyword.
- Rejected alternatives: `include exists as` (adds a keyword for no benefit); `with exists as` (`with` is overloaded — CTE / `with tags` / `with context`).
- Minor parser note: comma-separated clauses (`exists left as a, exists right as b`) disambiguate from a new FROM source by the post-comma token (`exists` ⇒ another clause).
- **Additive, not breaking.** `exists` cannot appear after a complete join `on` predicate today, so the clause occupies currently-unused grammatical space — confirm this against the real grammar and document it (the breaking-vs-additive check + version-bump governance is shared with `set-operator-membership-columns`, where the post-`union` position is the riskier one).

Optional complementary sugar (desugars to the same `existence` site; decide whether to ship alongside or defer):

```sql
select A.*, exists(B) as hasB from A left join B on B.aid = A.id where exists(B);
```

- `exists(<source-alias>)` in projection/predicate position — unambiguous because FROM aliases are unique, gives two-sided full-outer existence for free, and composes in `where`. The parser distinguishes it from the standard `exists (select …)` by the operand being a bare in-scope source alias rather than a subquery.

The plan stage should finalize **which form(s) ship** and the exact keyword; the read-side projection and the write-side semantics are the substance, the surface is negotiable.

## Relationship to the tag surface

The existence column is the outer-join half of a directive to **remove the `quereus.update.*` routing tags**. For an outer join it replaces the routing role of `delete_via` / `target` / `exclude` (which side a write hits) with a per-row, explicit boolean. The set-op half is `set-operator-membership-columns`. The value-supply tag `default_for` is **retained** (it has no column equivalent). The actual removal of the redundant routing tags from the reserved-tag registry + docs is gated on both column mechanisms landing — see `remove-update-routing-tag-surface`. Implementation note (load-bearing for that downstream work): the `existence` `UpdateSite` should carry a **generalized relational-component reference** (join-side *or* set-branch), not a hard-coded join side, so the set-op membership work is an extension rather than a parallel mechanism.

## Dependency

The **write** semantics (existence-flip ⇒ insert/delete) require outer-join write-through, which `view-write-through-shape-gaps` wires — hence the `prereq`. The **read** projection (guard-as-boolean) is independent and could land first if the plan stage splits this into a read-only front half and a write-enabled second half.

## Expected behaviour / use cases

```sql
create view AB as
  select A.id, A.x, B.y, exists(B) as hasB
  from A left join B on B.aid = A.id;

select * from AB;                              -- hasB reflects the match per row
update AB set hasB = true  where id = 7;       -- inserts B(aid=7, y default) if absent
update AB set y = 5        where id = 7;        -- ordinary update on the matched B
update AB set hasB = false where id = 7;       -- deletes the matching B row
insert into AB (id, x, y, hasB) values (9, 'q', 3, true);  -- inserts A and B
insert into AB (id, x, hasB)    values (9, 'q', false);    -- inserts only A
```

## Tests (TDD seeds)

Acceptance is the **View Round-Trip Laws** property block (`test/property.spec.ts`), extended in the outer-join family added by `view-write-through-shape-gaps`:
- existence column reads agree with the guard across matched / null-extended rows;
- `hasB` false→true with all B NOT-NULL columns defaulted → PutGet shows the B row appears; with an undefaulted NOT-NULL B column → rejects with `no-default` (not a silent drop);
- `hasB` true→false → the B row disappears, A untouched (PutGet);
- GetPut: writing the read-back `hasB` value back is a no-op on the base;
- AST round-trip: `parse(stringify(ast)) ≡ ast` for the chosen syntax (`emit-roundtrip` + the structural comparator).
- **Key Soundness** (`test/property.spec.ts`): a flag-bearing node never over-claims — the flag is `key → flag` determined but never appears in a claimed key, and the node's `isSet` / keys are unchanged by adding the flag column (FD ramifications, Invariants 1–2).
