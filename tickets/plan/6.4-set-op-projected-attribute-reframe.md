## Triage redirect (2026-06-13, human sign-off): explore the projected-attribute reframe first

Asked for a go/no-go on the product-coordinate membership model, the dev redirected:
**"explore if we can accomplish the same thing using projected attributes, since
this would add to the predicate."** That is the brief for the plan pass — neither
the full build nor a shelve, but a research pass on a potentially-subsuming
alternative.

**The reframe.** The product model invents membership *flags* (`union exists left
as inA, …`) plus novel write machinery (σ-guard threading onto `set-op-branch`
components, coordinate-addressed multi-target fan-out, `checkSatisfiability`
contradiction rejection). The alternative: a branch instead projects an ordinary
**discriminating attribute** carrying its predicate value, and membership/leaf
addressing falls out of the *existing* predicate-driven view-update framework —
no set-op-specific write code:

```sql
-- product model (novel):           projected-attribute reframe (reuse):
(A where color='red'                 select id, x, 'red'  as kind, 'A' as src from A where color='red'
   union exists left as inA …)         union all
…                                    select id, x, 'red'  as kind, 'B' as src from B where color='red'
                                       union all
                                     select id, x, 'large' as kind, 'A' as src from A where size='large'
                                     …
```

Writing `… set kind='red', src='B'` then routes to `B where color='red'` through
the **constant-FD default recovery** the view-mutation substrate already does
(`where color = 'green'` defaults `color` — § Predicate-Driven View Updateability):
the projected literal is a constant FD, the equality predicate recovers the omitted
base column, and write routing by writable-presence columns picks the leg. The
multi-target fan-out becomes an ordinary multi-leg `union all` write, and the
"contradiction" case becomes a `predicate-contradiction` the existing
`rule-filter-contradiction` / `sat-checker` already raise for an equality clash —
reached from the normal path, not a new set-op gate.

## Questions for the plan pass to settle (does the reframe actually cover it?)

- **Read parity.** Can a consumer read "is this row in branch X" as well through a
  projected discriminator (`kind`, `src`) as through a membership flag? The flag is
  a boolean over a *probe*; the attribute is a stored/derived value. For the fixtures
  the parent ticket cares about (the `{inX,inY}×{inA,inB}` grid), is a projected
  `(axis1, axis2)` tuple an adequate — or better, more honest — surface?
- **The reused-name merge.** The product model's headline is collapsing like-named
  flags across sibling subtrees into one coordinate column whose value is
  `tuple ∈ <union of like-named leaves>`. Does a projected attribute reproduce that,
  or is it strictly the *sum* (tagged-position) model the reframe naturally
  expresses? If the reframe only does the sum model cleanly, that may be the honest
  scope — the sum model already ships via 6.2/6.3.
- **Write addressing without sat-check.** Confirm that constant-FD default recovery
  + the existing contradiction rule subsume coordinate co-satisfiability addressing
  for the literal-discriminator case, and characterize where they do NOT (a
  non-literal / range discriminator that the FD framework cannot fold to a constant
  — likely out of scope, matching the framework's existing boundary).
- **Does anything genuinely require the membership-flag form?** If a real use case
  needs writable *boolean membership over an arbitrary σ-guard* that a projected
  attribute cannot express, name it — that residue is the only thing that would
  justify reopening the bespoke build. If none surfaces, recommend shelving the
  product model in favor of the projected-attribute pattern (documented as the
  idiom) and close this ticket.

The original product-coordinate design is preserved verbatim below as the
comparison baseline — build it only if the plan pass finds the reframe cannot
cover a use case the dev wants.

## Why this was blocked (original — superseded by the redirect above)

The parent plan ticket (`set-op-membership-nested`, 6.2) was promoted **on request** and says,
verbatim and twice: *"Confirm appetite for this esoteric feature before investing."* Plan-stage
research is now complete, and it changed the picture enough that re-confirming appetite is the
honest call:

1. **The scope is materially larger than the ticket assumed.** The ticket framed the remaining
   work as "just per-leaf addressing" on top of the landed foundations. Verified (2026-06-08),
   that is false — two further foundations were required just to make the fixtures reachable, now
   filed as implement tickets `set-op-leftwrap-arity` (6.2) and `set-op-leftwrap-write` (6.3).
   Those two deliver the **sum model** (structurally-distinct, distinct-named nested subtrees:
   `(A∪B)∪(C∪D)` with read + the unambiguous-fan-out writes) — clearly-valuable work within the
   established appetite, no sign-off needed.

2. **What remains — the product-coordinate model — is genuinely novel and costly.** It is
   writable *product coordinates* over set-op branch membership with multi-target fan-out: an
   unusual UX with no precedent I'm aware of in any SQL engine. It is ~2 substantial implement
   tickets touching the read surface (planner node), the analysis layer (guard threading), the
   write decomposition, and the satisfiability checker. Building the wrong thing here is
   expensive and easy (e.g. the human may only have wanted `set <subtreeFlag> = true` for a
   single subtree, not the full product/sum/multi-target/contradiction calculus).

**Decision requested (one of):**
- **(a) Proceed** — unblock; convert the design below into the two implement tickets in
  "## Build plan" and build on top of 6.3.
- **(b) Narrow** — e.g. only `set <subtreeFlag> = true` routed to a single, explicitly-chosen
  leaf (a policy knob, no product calculus); say which subset.
- **(c) Shelve** — the sum model (6.2 + 6.3) is sufficient; move this to `backlog/`.

The sum model (6.2 + 6.3) stands on its own and ships regardless of this decision.

## The model (fully designed — ready to build on sign-off)

### The fixture this enables

```sql
create view U4 as
      ((select id, x from A where color = 'red')
         union exists left as inA, exists right as inB
       (select id, x from B where color = 'red'))
  union exists left as inX, exists right as inY
      ((select id, x from A where size = 'large')
         union exists left as inA, exists right as inB
       (select id, x from B where size = 'large'));
```

`inA`/`inB` are **reused** across the two sibling subtrees; `inX`/`inY` name the subtrees. This
yields a `{inX,inY} × {inA,inB}` product coordinate grid; each leaf is a **conjunction** of one
coordinate per axis (`(B where color='red') ≡ inX ∧ inB`).

### Read: reused names merge into one coordinate column

Today, two subtrees each declaring `inA` surface as **two `inA` columns** (a name collision;
6.2 deliberately leaves this as-is). The product model **collapses like-named flags into one
column** whose value is `tuple ∈ <union of all like-named leaves>`:
`inA ≡ tuple ∈ (A where color='red' OR size='large')`, `inX ≡ tuple ∈ <the red subtree>`. Distinct
names stay distinct (that is the sum model — tagged tree positions). The engine chooses neither;
it derives `flag ≡ tuple ∈ <union of like-named leaves>` uniformly and lets conjunction address
the leaf.

- Locus: `set-operation-node.ts` membership/attribute surface + `update-lineage.ts`. Merge by
  (lowercased) name across both operands at every depth; the merged read probe is the OR of the
  member leaves' presence.
- Open sub-question for sign-off: do we **require** structural parallelism for a name reuse
  (validate the like-named subtrees share leaf relations), or merge purely by name and let
  conjunction sort it out? The parent ticket says merge purely by name; flagging it because a
  reused name across non-parallel subtrees yields a defined-but-surprising union.

### Analysis: thread the real σ-guard (currently a `true` placeholder)

`SetOperationNode.membershipLineage()` registers each flag's `existence` `UpdateSite` with
`guard = { type:'literal', value:true }` — an explicit placeholder (`set-operation-node.ts:301`;
its own doc comment says "the write half computes the real conjunction of σ predicates down to
the branch's base for predicate-honest leaf addressing"). The product model needs the **real
accumulated σ** each branch carries (the `where color='red'` on the red subtree, the `id=1` on a
contradiction subtree) threaded onto the `set-op-branch` component, because:
- leaf addressing = "the leaf whose accumulated guard is co-satisfiable with the set-true
  coordinates' guards";
- contradiction detection = sat-check over those guards.

Locus: `set-operation-node.ts` (capture each operand's accumulated `FilterNode`/σ predicate as
the guard) + `update-lineage.ts` (carry it through).

### Write: coordinate addressing, multi-target fan-out, one rejection

Writing a (now-merged) coordinate flag `set inX = true, inB = true, inY = false, inA = false`:
1. Collect the **set-true** coordinates' guards (one per axis).
2. **Sat-check co-satisfiability** of their conjunction via `checkSatisfiability`
   (`analysis/sat-checker.ts`, the primitive behind `rule-filter-contradiction`). If provably
   `unsat` → reject with a `predicate-contradiction` diagnostic (the **only** rejection).
3. The **target leaf set** = every leaf whose accumulated guard is co-satisfiable with the
   conjoined set-true guards. **Single-target** = one-hot / mutually-exclusive coordinates pin
   one leaf (`inX∧inB` over red vs base-B → `B where color='red'`). **Multi-target** =
   co-satisfiable coordinates (color vs size) fan the write to *every* consistent leaf — the
   predicate-honest fan-out, a feature, not an error.
4. Emit one recursive `MutationRequest` per target leaf, sharing the one up-front capture
   (reusing 6.1's `fanBranch*` machinery).

Locus: `set-op.ts` — replace the current "surfaced-inner-flag write → reject" and
"`set <subtreeFlag>=true` → reject" paths with the coordinate-addressing decomposition; reuse
`checkSatisfiability` for the contradiction gate.

Contradiction example to reject: a `U` whose two subtrees are `where id=1` / `where id=2`;
`set inX = true, inY = true` conjoins `id=1 ∧ id=2` → `unsat` → `predicate-contradiction`. Two
**different** attributes (`color='red' ∧ size='large'`) are co-satisfiable → legal multi-target
write, no rejection.

- **`set-op-product-coordinate-read`** (prereq `set-op-leftwrap-write`): reused-name flag merge

## Edge cases & interactions (for the eventual implement tickets)

- **Subtree + leaf read agreement** — leaf-flag and subtree-flag reads agree with `tuple ∈
  <union of like-named leaves>` row-by-row (`inA ≡ A where color='red' or size='large'`,
  `inX ≡ the red subtree`).
- **Single-target write (conjunction)** — `inX=true,inB=true,inY=false,inA=false` targets exactly
  `B where color='red'`; PutGet: the row appears there and nowhere else.
- **Multi-target write (fan-out)** — `inX=inY=inA=inB=true` over co-satisfiable axes upserts the
  row into all four consistent leaves (A and B, as red AND large); the row appears in every
  targeted base.
- **Contradiction rejection** — `inX∧inY` over mutually-exclusive `id=1`/`id=2` rejects with
  `predicate-contradiction` (not a silent partial write).
- **Co-satisfiable vs contradictory** — different attributes co-exist (no reject); two values of
  one attribute are exclusive (reject). Cover both.
- **Sat-checker scope limits** — `checkSatisfiability` returns `unknown` outside its supported
  fragment (non-literal guards, >64 conjuncts, NULL comparisons). Decide the conservative default
  for `unknown`: **accept (fan-out) and document**, OR reject. (Recommended: treat `unknown` as
  co-satisfiable → fan-out, since the model is predicate-honest and a false-reject is worse than a
  possibly-empty leaf write that the leaf's own correlation filters out. Confirm.)
- **Key Soundness** — a nested distinct union carrying flags at both levels stays keyed by its
  data columns; no flag (leaf or subtree, merged or not) ever enters a claimed key, all depths.
- **Halloween safety** — coordinate fan-out reuses the ONE up-front capture across all target
  leaves.
- **Interaction with the sum model** — distinct-named subtrees (no reuse) keep the 6.2/6.3 sum
  behavior unchanged; only reused names trigger the merge/product path.
- **`except` / `intersect` subtrees under the product model** — inherits the
  `set-op-membership-nested-except` membership-gating; a flag-less non-union boundary stays
  deferred.
