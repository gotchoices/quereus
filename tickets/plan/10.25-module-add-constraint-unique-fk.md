# Module `ALTER TABLE ADD CONSTRAINT` for UNIQUE / FOREIGN KEY

## Why

`alter-table-drop-rename-constraint` (prereq, ticket 10.2) wired the declarative constraint
**lifecycle** (add / drop / rename by name) into `computeTableAlterDiff` / `generateMigrationDDL`.
DROP and RENAME work end-to-end for all three classes (CHECK / UNIQUE / FOREIGN KEY). **ADD does
not**: `ADD CONSTRAINT CHECK` is handled engine-side in `runtime/emit/add-constraint.ts`
(`runAddCheck` mutates `checkConstraints` directly), but `ADD CONSTRAINT UNIQUE` / `FOREIGN KEY`
route through `module.alterTable({ type: 'addConstraint', ... })`, and **both built-in modules
(memory + store) throw `StatusCode.UNSUPPORTED`** for any constraint type.

Consequence (documented in 10.2 as gap #1): the differ *emits* `ALTER TABLE … ADD <fragment>` for
a declared-absent UNIQUE/FK named constraint, so a **declarative add of a UNIQUE/FK to an
already-existing table fails at apply** with `UNSUPPORTED`. The migration batch rolls back
atomically (no partial state — `runBatchedMigrationLoop` + `endSchemaBatch(error)` discards the
overlay), so it is safe, but it is a poor UX for a plausible action (declaring a new `unique`/`fk`
on an existing table). Fresh `CREATE TABLE` and idempotent re-apply are unaffected (the constraint
arrives via CREATE TABLE / name-matches).

This also **blocks ticket 10.3** (`alter-constraint-body-change-drop-add`): a UNIQUE/FK **body**
change is realized as drop+add, and the add half hits this same wall. 10.3's CHECK path works;
its UNIQUE/FK path cannot land until this exists. (10.3's prereq has been updated to depend on this
ticket.)

## Goal

Implement `addConstraint` for UNIQUE and FOREIGN KEY in both built-in modules so the existing
`runAddConstraintViaModule` path (already wired by 10.2's prereq chain) succeeds, **re-validating
existing rows** against the new constraint and failing with `StatusCode.CONSTRAINT` (atomic — no
schema mutation) when current data violates it.

## Design sketch (to be refined in plan)

### Memory module (`MemoryTableManager.addConstraint`)
- **UNIQUE**: append to `uniqueConstraints`, then build the implicit covering index the same way
  `ensureUniqueConstraintIndexes` does (named after the constraint), reusing `populateNewIndex`
  (which already raises `CONSTRAINT` on the first duplicate among in-scope rows — honoring partial
  predicate + per-column collation). On success register the `implicitCoveringStructures` record;
  on the duplicate error, roll the manager + base schema back to the pre-add snapshot (mirror the
  rollback in `dropConstraint` / `renameConstraint`).
- **FOREIGN KEY**: append to `foreignKeys`; validate existing child rows against parent existence
  (respect `pragma foreign_keys` — when off, add without the validating scan, matching how FK
  enforcement is gated elsewhere). Decide whether a missing-parent row aborts (consistency with
  CHECK/UNIQUE re-validation) or is grandfathered; default to **abort** for parity with the other
  classes unless a capability says otherwise.
- Route memory `module.alterTable` + `MemoryTable.alterSchema` `addConstraint` arms to it; emit the
  same `alter` schema-change event as drop/rename.

### Store module (`store-module.ts` `addConstraint`)
- **UNIQUE**: append to `uniqueConstraints` (store enforces inline UNIQUE by full-scan — no separate
  index store to build), validate existing rows by a full scan for duplicate keys (NULLs distinct,
  per SQL UNIQUE), `saveTableDDL`. On violation, throw `CONSTRAINT` without persisting.
- **FOREIGN KEY**: append to `foreignKeys`, validate existing child rows against the parent store
  (respect `pragma foreign_keys`), `saveTableDDL`.

### Cross-cutting
- The runtime `AddConstraintNode` path (`runAddConstraintViaModule`) needs no change — it already
  routes UNIQUE/FK through `module.alterTable`. Confirm the `schema.addTable` + `table_modified`
  notification (cache invalidation) fire identically to the drop/rename paths.
- **Isolation module** `translateOverlayRow` already no-ops `addConstraint` (no row-shape change) —
  confirm no change needed.

## Edge cases & tests
- UNIQUE add over a table with an existing duplicate → `CONSTRAINT`, schema unchanged, enforcement
  not partially installed (atomicity).
- UNIQUE add over conforming data → enforces going forward; covering index appears in
  `index_info` (memory); idempotent re-declare ⇒ empty diff.
- FK add with `pragma foreign_keys = true` over orphan child → `CONSTRAINT`; with `= false` →
  add succeeds, enforcement deferred to writes.
- Named vs unnamed: only **named** constraints are differ-reachable (10.2 excludes auto-names), but
  the primitive should also accept the explicit `ALTER TABLE … ADD UNIQUE (...)` unnamed form.
- Declarative add of UNIQUE/FK to an existing table converges via `apply schema` and is idempotent
  (the assertion 10.2 could not make — gap #1).
- Run the new behavioral `.sqllogic` under both `yarn test` and `yarn test:store`.
- **Rewrite `test/alter-add-constraint.spec.ts`**: it currently asserts UNSUPPORTED for UNIQUE/FK —
  flip those to assert success + enforcement once implemented.

## Decision to confirm with the human
Until this lands, 10.2 leaves the differ **emitting** a doomed UNIQUE/FK ADD (fails atomically at
apply). The alternative interim behavior would be to **hard-error early** in the differ /
`generateMigrationDDL` for non-CHECK adds (clearer than a low-level UNSUPPORTED at apply). The
review of 10.2 chose to keep emit-and-fail (documented, atomic) on the expectation that this ticket
makes the add actually work. If this ticket is deferred indefinitely, revisit that choice.
