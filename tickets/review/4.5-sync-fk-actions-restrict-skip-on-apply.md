# Review: Skip RESTRICT on the trust-the-origin apply path

## What this change does (distilled)

On the external-row-change apply path (`Database.ingestExternalRowChanges` with
`applyForeignKeyActions: true`), the FK-actions facet now **skips all parent-side RESTRICT
enforcement** and propagates only the non-RESTRICT actions (cascade / set-null / set-default)
at every cascade depth. Re-enforcing RESTRICT on the receiver wedged the sync stream (throw →
no metadata commit → re-resolve → re-apply identical batch → throw forever). Trust-the-origin:
the origin already enforced RESTRICT at its own commit.

Mechanism: a `Database`-level **apply-mode suppression flag** set for the duration of the
batch (mutex held; restored in a `finally`, save-and-restore for nesting) and honored at the
RESTRICT enforcement sites.

## ⚠️ #1 REVIEW FOCUS — the ticket plan was incomplete; scope was expanded

The ticket asserted the fix was "complete" by gating **only** the three runtime RESTRICT
asserts in `foreign-key-actions.ts` (reached from `dml-executor.ts:723/797/938`, the seam,
and MV-maintenance). **This is necessary but NOT sufficient, and I verified that empirically.**

I wrote the ticket's depth-≥1 nested-cascade test (`p ← c on delete cascade ← g on delete
restrict`) and it **threw** with only the runtime gates in place:

```
ConstraintError: CHECK constraint failed: _fk_g_cid (not exists (select 1 from main.g where g.cid = OLD.id))
  at checkCheckConstraints (runtime/emit/constraint-check.ts)
  at executeSingleFKAction (foreign-key-actions.ts)   ← the cascade DELETE FROM c
```

The **primary** RESTRICT enforcer for cascade DML is the **plan-time** parent-side FK check —
a synthesized `NOT EXISTS` `CHECK` node (`buildParentSideFKChecks` → `ConstraintCheck` node →
`checkCheckConstraints` → `throwForAction`), constraint kind `fk-parent`. The runtime
`assertTransitive…` pre-check is documented as *defense-in-depth* behind it. When a cascade
DELETE/UPDATE re-enters the DML pipeline via `_execWithinTransaction`, it is planned like any
DML and carries that plan-time check, which fires **before** the gated runtime assert. The
suppression flag is runtime state and cannot reach plan construction, so the plan-time check
was un-suppressed and still threw.

**Fix added (NOT in the ticket plan):** `constraint-check.ts checkCheckConstraints` now
early-`continue`s a constraint entry when `metadata.kind === 'fk-parent' &&
rctx.db._isFkRestrictSuppressed()`. With both gates the depth-≥1 case commits.

Why `kind === 'fk-parent'` is exactly the right (and narrow) predicate — **please verify
this reasoning independently**:
- `fk-parent` checks are synthesized **only** for RESTRICT / NO ACTION FKs. CASCADE / SET
  NULL / SET DEFAULT produce *actions*, not parent-side checks (see `buildParentSideFKChecks`
  and `parent-fk-check-gate.spec.ts`). So `fk-parent` ⟺ a parent-side RESTRICT — gating all of
  them equals "suppress parent-side RESTRICT", nothing more.
- `'check'` (user CHECKs), NOT NULL (handled in `checkNotNullConstraints`, a separate path),
  and `'fk-child'` (child-side FK existence) are **untouched** — confirm none of these get
  swept up. In particular `fk-child` stays active: a cascade UPDATE that re-keys a child still
  validates the new FK value (the adapter wrote the parent's row before the seam call, so it
  resolves). This is a deliberate scope boundary; flagging it for scrutiny.

`constraint-check.ts` runs for **every** DML statement, so the added condition is on a hot
path. It short-circuits on the cheap `kind` comparison before the flag call; when the flag is
never set (all normal DML) behavior is identical. Confirm there is no measurable overhead and
no behavior change off the apply path (the 195-test FK/lens/constraint sweep below is the
evidence, but a second look is warranted).

## Known gaps / things the reviewer should poke at

- **Test floor, not ceiling.** The 4 new tests are happy-path + the two correctness
  properties. Not covered by a dedicated test (the ticket marked these optional):
  - **REPLACE-eviction RESTRICT** (`dml-executor.ts:723`): a cascade UPDATE forcing a non-PK
    UNIQUE eviction whose evicted row has RESTRICT children. Covered by construction (same
    `assertTransitive…` + same `fk-parent` plan-time check), but unverified end-to-end.
  - **Cascade UPDATE** (`on update cascade`) reaching a RESTRICT child on apply — only DELETE
    cascade is exercised. The gating is op-agnostic, but a UPDATE-path test would harden it.
  - **Lens-routed apply.** The apply seam passes `lensRouted = false` (external = physical
    basis write), so `assertLensRestricts…` is never reached on this path today; I still gated
    it (belt-and-suspenders, per the ticket). No test exercises the lens assert *under
    suppression* because it is unreachable from the seam — confirm that's acceptable.
- **`fk-child` left active under suppression** (see #1) — a judgment call. If trust-the-origin
  should also skip child-side existence re-checks on cascade re-entry, that's a follow-up, but
  I believe leaving it active is correct (the merged parent rows are present).
- **Global-assertion path untouched** — the flag suppresses only FK RESTRICT, never the
  commit-time global-assertion sink (report mode). The detect-and-notify story for replica-only
  referential invariants is unchanged; `docs/sync.md` documents expressing them as global
  assertions.
- No `.pre-existing-error.md` written — the suite was fully green before and after.

## Decision context (already signed off, do not re-litigate)

Skipping RESTRICT on apply (vs. throwing) is the intended trust-the-origin policy; a
replica-only RESTRICT invariant is, by design, not enforced on apply and is expressed as a
global assertion if the receiver must be notified. The only design change *I* made beyond the
ticket is **where** the suppression had to be applied (adding the plan-time `fk-parent` gate),
not **whether** to suppress.
