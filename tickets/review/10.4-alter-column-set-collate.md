# REVIEW: ALTER TABLE … ALTER COLUMN … SET COLLATE

## What landed

`ALTER TABLE t ALTER COLUMN c SET COLLATE <name>` — a new sibling of the existing
`SET DATA TYPE` / `SET DEFAULT` / `SET NOT NULL` verbs — changes a column's collation
in place, and collation is now projected into the declarative diff catalog so the
differ detects collation drift and emits `SET COLLATE` migrations.

Unlike tags (non-behavioral, hash-stable, catalog-only), **collation is semantic**:
it changes `=` / `ORDER BY` and participates in the schema hash. The memory module
re-keys / re-sorts every PK / UNIQUE / index that orders by the column and
re-validates uniqueness under the new collation; a set unique under `BINARY` that
collides under `NOCASE` is rejected with `CONSTRAINT` and rolled back.

## How it works (data flow)

```
SQL  ──parse──>  alterColumn{ setCollation }  ──build──>  AlterTableNode
   ──emit (runAlterColumn)──>  validateCollationForType (reject unknown, CREATE-time error shape)
   ──>  module.alterTable({type:'alterColumn', setCollation})
            memory: update ColumnSchema.collation; propagate into PK-def + index columns;
                    rebuildAllSecondaryIndexesStrict() then rebuildPrimaryTreeStrict()
                    (both throw CONSTRAINT on a collision → catch restores prior schema);
            store:  schema-only collation update (fixed table-level key encoding)
   ──>  schema.addTable(updated) + notify table_modified

diff schema:  catalog.collation (default BINARY)  vs  declared COLLATE constraint
   ──>  computeColumnAttributeChange sets ColumnAttributeChange.collation on drift
   ──>  generateMigrationDDL emits `ALTER COLUMN c SET COLLATE n`  (after SET DATA TYPE,
        before SET DEFAULT / SET NOT NULL)
```

## Use cases / behaviors the tests assert (the floor — extend, don't trust)

**`41.7-alter-column-collate.sqllogic` (memory + store):**
- `SET COLLATE NOCASE` flips `=` semantics on a non-key column (`'ABC'` matches 2 rows).
- `SET COLLATE NOCASE` flips `ORDER BY` (case-folded ordering).
- `table_info().collation` round-trip; canonical uppercase spelling; `SET COLLATE BINARY` restores default.
- Unknown collation → `Unknown collation 'nosuchcollation' for type 'TEXT'`; column untouched.
- Re-applying the same collation is an idempotent no-op.

**`41.7.1-alter-column-collate-unique.sqllogic` (memory-only):**
- UNIQUE column distinct under both collations → ALTER succeeds; later NOCASE-colliding insert rejected.
- UNIQUE column distinct under BINARY but colliding under NOCASE → ALTER rejected (`CONSTRAINT`), table unchanged + still writable.
- PK column colliding under NOCASE → rejected + rolled back.
- PK column non-colliding → succeeds; later NOCASE-colliding PK insert rejected.

**`declarative-equivalence.spec.ts` (added this run):**
- Declared `COLLATE NOCASE` drift converges via `SET COLLATE`, changes `=` semantics, idempotent re-apply.
- Absent `COLLATE` == explicit `COLLATE BINARY` (no spurious diff).
- A collation change **moves the schema hash** (inverse of the tag-hash-stable assertions).

## Known gaps / things to scrutinize (honest handoff)

1. **Store is schema-only for SET COLLATE.** The LevelDB store updates the column
   collation in the schema (so query-layer `=` / `ORDER BY` / `table_info` pick it
   up) but does **not** physically re-key or re-validate existing-row uniqueness at
   ALTER time — its physical key encoding uses a fixed table-level collation
   (`encodeOptions`). Consequence: a store table whose existing values collide under
   the new collation will **not** reject the ALTER the way memory does. This is a
   deliberate, documented limitation (see `41.7.1` header + store-module comment +
   `docs/sql.md` store-module note). **Reviewer decision:** is this memory/store
   divergence acceptable for now, or should it spawn a follow-up fix ticket? Note
   `41.7.1` is in `MEMORY_ONLY_FILES` precisely because of this.

2. **Memory rollback path is subtle.** On a strict-rebuild throw, the catch restores
   `originalManagerSchema` and runs the **non-strict** `rebuildAllSecondaryIndexes()`;
   the primary tree is only swapped on full success so it's already original. Verify
   there's no window where the secondary-index map is left half-cleared after a
   colliding-UNIQUE rollback. (`41.7.1` case 2 asserts the table is still writable
   afterward, but a reviewer should reason through the ordering: secondary strict →
   primary strict.)

3. **Explicit `CREATE INDEX` re-sort is untested.** `41.7.1` covers an inline column
   `UNIQUE` constraint (implicit covering index) and the PK. An explicit
   `create index idx on t(name)` followed by `SET COLLATE` exercises the same
   `rebuildAllSecondaryIndexesStrict` path but has no dedicated test; a non-unique
   index re-sort has no easily-assertable query effect, which is why it was skipped.
   A `create unique index` (`derivedFromIndex`) variant is likewise untested but
   shares the code path. Consider adding coverage if you want belt-and-suspenders.

4. **PK-column collation drift via the declarative differ** is covered for a
   non-key column only in `declarative-equivalence.spec.ts`. The differ compares
   collation regardless of PK membership (so it should emit `SET COLLATE` for a PK
   column too), and `PrimaryKeyColumnDefinition.collation` is propagated by the
   memory manager — but there is no end-to-end declarative test that a PK-column
   `COLLATE` change converges. Minor gap.

5. **`committed.*` snapshot / savepoint interaction** with a mid-transaction
   `SET COLLATE` is reasoned about in the ticket (collation is schema-level) but not
   explicitly tested. Low risk; worth a glance.
