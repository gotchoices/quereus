## What shipped

Extracted the normative view-updateability claims into `docs/invariants.md` as a new `VU`
area, mirroring the completed `MV-*` shape. This is docs-and-checker only — **no engine
source or test was modified**.

**Area registration**
- `scripts/check-docs.mjs`: added `VU` to `INVARIANT_HEADING` (placed after `MV`, in area order).
- `docs/doc-conventions.md` § The invariant register: added `VU` to the allowed-area list.
- `docs/invariants.md`: new `## VU — View updateability` heading + short preamble, placed
  after the `MV` area and before the reserved `## RT` placeholder (i.e. with the live areas).

**Eight invariants** (each with `code:` / exactly one `guard:` / `doc:`, ≤120 words):
- **VU-001** — view/CTE/subquery-in-`from` DML rewrites to base tables and re-plans through
  the base builders; no view flag / `instead of` / `with check option`.
- **VU-002** — updateable iff a deterministic decomposition exists at plan time, else a
  structured `MutationDiagnostic`; never silently dropped or widened.
- **VU-003** — PutGet: a write never escapes the view predicate; computed columns reject.
- **VU-004** — GetPut: read-a-row-and-write-it-back is a base no-op.
- **VU-005** — forward/backward lineage agreement.
- **VU-006** — a non-invertible (`opaque`) column is read-only unless a `with inverse` is authored.
- **VU-007** — a shared/surrogate key is evaluated once at the envelope and threaded to every
  member via the equivalence class.
- **VU-008** — a `limit` / `offset` / `distinct` body rejects rather than widen.

**Back-links** added at the owning satellite section for each invariant (5 doc sections; the
`> **Invariant:** [VU-00X](invariants.md#…)` full-slug form). The satellites were **not**
edited to restate the invariants — the register is the normative text.

## For the reviewer — where to push

- **Guard coarseness (VU-003/004/005/008).** All four point at the same `guard:` string
  `View Round-Trip Laws`, and three share the same `doc:` anchor. The checker only verifies
  the string appears in the spec file, not that a distinct `it()` exercises each law — so the
  guards are honest but coarse. The section *does* contain per-law tests (the 61 above split
  into PutGet / GetPut / lineage / reject-widen families). Worth confirming each law is
  individually red-able, not just the block as a whole.
- **VU-001 / VU-002 guards are the soft ones.** VU-001 uses a path-only sqllogic guard
  (`93.4-view-mutation.sqllogic`); VU-002 leans on the property block's `expectMutationReject`
  cases. Neither is a dedicated "rewrite-to-base equivalence" or "diagnostic completeness"
  test. This matches the ticket's self-guarding allowance, but a stronger guard is a fair ask
  if you want them harder.
- **Doc-vs-register tension in `vu-operators.md` (observation, not fixed — see Review
  findings).** § Distinct says "Mutations apply to all base rows that collapse to the affected
  view row" and § Sort, Limit, Offset says "Pure passthrough" — describing an aspirational
  Phase-2 operator semantics that **contradicts VU-008's shipped Phase-1 reject**. Per
  doc-conventions the register wins; I left the operator doc unchanged to keep the diff scoped
  to the register + back-links. Decide whether to reconcile the operator doc (it does note the
  Phase-1 limits elsewhere — overview § Current limitations, and vu-roundtrip frames the
  reject as the property the laws catch).
- **Candidate I kept vs. the ticket's suggested `doc:` for VU-008.** The ticket suggested
  `doc: vu-operators.md`; I pointed VU-008 at `vu-roundtrip.md § The three round-trip laws`
  instead, because that section actually states the reject-don't-widen behavior (vu-operators
  § Distinct states the opposite, above). If you'd rather the back-link live on the operator
  section, that requires reconciling that section's text first.
