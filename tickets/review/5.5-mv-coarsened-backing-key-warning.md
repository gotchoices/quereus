# Coarsened backing key + key-coarsening warning — implement summary

## What landed

The collation-weakening materialized-view shape (`select handle collate nocase
as handle, email from Contact_v1` over a BINARY-keyed source — the
parallel-migration-table pattern of docs/migration.md) now **creates** instead
of throwing `cannotMaterialize(... 'no provable unique key')`:

1. **Shared derivation** (`deriveCoarsenedBackingKey`, new module
   `planner/analysis/coarsened-key.ts`). When `keysOf(root)` is empty: walk a
   **row-preserving single-source chain** (Project / Filter / Sort / Retrieve /
   Alias / AssertedKeys / physical access nodes — joins, set ops, aggregates,
   DISTINCT, window, TVFs, LIMIT/OFFSET all abstain) to the one base table,
   then trace each output column through **value-preserving passthrough
   chains** (bare column ref / `CollateNode` / no-op `CastNode`, transitively
   across producing-expr hops). If the source PK is fully covered, the covering
   outputs are the coarsened backing key K'. Never enters `keysOf` — it is an
   MV-create policy, not a planner key fact.
2. **Backing keyed on K'** with the *output* collations (`deriveBackingShape`;
   `buildBackingTableSchema` already stamps pkDefinition collation from output
   column types). Both create-path consumers — `deriveBackingShape` and
   `buildFullRebuildPlan`'s bag-reject — call the same pure function over the
   same fully-optimized body, so they agree by construction (no threading was
   needed).
3. **Key-coarsening warning** when any K' column's output collation is strictly
   coarser than the source key's enforcement collation (BINARY is finest:
   refinement/equality derive a true key silently; any other difference is
   conservatively coarsening). Channel decided per the ticket's latitude:
   **structured logger** (`runtime:emit:materialized-view:warn`) emitted in
   `materializeView` after successful registration, **plus a documented record
   stamp** `MaterializedViewSchema.coarsenedKey: CoarsenedKeyInfo` (columns +
   per-column source→output collations; recomputed at create/import/refresh
   shape-rebuild, never serialized — so no DDL round-trip or differ churn).
   The lens advisory surface was NOT used: imperative MV create does not flow
   through the lens prover/deploy-report pipeline (see Gaps).
4. **Row-time eligibility**: `buildInverseProjectionPlan`'s projector
   classification now resolves through value-preserving wrappers
   (`resolveValuePreservingSourceCol`, exported from the new module; the old
   single-hop `resolveSourceCol` was deleted). The canonical migration body
   registers on the ordinary inverse-projection arm — per-row upsert keyed
   under the backing PK's output collation realizes the LWW merge with zero
   new maintenance machinery. Coarsened bodies the arm declines (e.g. a
   WHERE-IN-subquery body) ride the full-rebuild floor, whose collation-keyed
   `replace-all` diff is likewise LWW (verified by test).
5. **Coverage prover collation gate**: `proveCoverage` now requires every
   projected UC *and* PK column to carry the same collation as its base column
   (`'collation-mismatch'`, a new `CoverageFailureReason`). Defense-in-depth —
   a collation-changing projection mints a fresh attribute id and already
   failed projection coverage; the gate makes the guarantee explicit.
6. **Unchanged rejections**: key-dropping projections, collated GROUP BY keys
   (grouping collapses rows — a lineage key would be a false identity; the
   `collated-groupby-key-completeness` backlog ticket is the real fix there),
   recursive CTEs, etc. all still bag-reject with the same diagnostic.

## Behavioral contract (what to validate)

Pinned in `test/logic/51.5-materialized-views-coarsened-key.sqllogic`
(passes in both memory and store modes) and
`test/coarsened-backing-key.spec.ts`:

- **Create over clean data** succeeds; backing PK = K' at NOCASE; record
  stamped `{ columns: ['handle'], weakened: [{ column, BINARY → NOCASE }] }`.
- **Colliding seed data** (`'Bob'`,`'bob'`) fails loudly at fill ("must be a
  set"), nothing half-registered (name reusable).
- **Steady-state LWW**: colliding insert merges into the shared backing row;
  each sibling edit re-asserts its own image (deterministic oscillation).
- **REFRESH during a collision window is loud** ("must be a set") — refresh is
  a re-fill, same posture as create-fill. Pinned deliberately; see Gaps.
- **Delete-one-sibling anomaly**: deleting one colliding source row deletes
  the *shared* derived row while the sibling lives; the sibling's next edit or
  a REFRESH re-asserts it (both covered).
- **Multi-column passthrough body** (`primary key (a, b)`, only `a` collated)
  derives the composite K' with only `a` reported weakened.
- **Refining direction** (BINARY output over NOCASE-keyed source) is a true
  key — accepted silently, no stamp.
- **Floor-routed coarsened body** (WHERE-IN subquery) creates, stamps, and
  LWW-merges through the floor's replace-all diff.
- **Prover gate**: a doctored NOCASE-constraint vs BINARY-output mismatch
  yields `NotCovers('collation-mismatch')`; the matching-collation control
  still covers.

Validation run: `yarn build` clean, `yarn lint` clean, full `yarn test`
workspace green (5815 passing in the quereus suite), plus a targeted
store-mode run of the new sqllogic file (`QUEREUS_TEST_STORE=true`).

## Known gaps / review attention

- **Warning visibility.** The warning rides the debug logger (`:warn`
  sub-channel), which is invisible without `DEBUG` enabled; the record stamp
  is the programmatic surface. There is **no SQL-level introspection** of
  `coarsenedKey` (no TVF/pragma column) and no lens-ack-style acknowledgment
  governance — the ticket allowed this choice ("decide in implementation; the
  doc contract is warn, don't reject"), but a reviewer should judge whether a
  follow-up ticket for surfacing (e.g. via the schema TVFs or the declarative
  deploy report) is warranted.
- **Inverse-projection passthrough widening applies to ALL MV bodies**, not
  just coarsened ones: collate/no-op-cast-wrapped columns previously
  classified `'expr'` (computed) now classify `'passthrough'` and may land in
  backing keys / the covering inverse map. Value-correct by construction (the
  wrappers copy the source value verbatim), and the equivalence/property
  suites pass — but this is the highest-blast-radius change; adversarial
  review should focus here. Note the lateral-TVF (`prefix-delete`) and
  join-residual arms deliberately still use the *non*-value-preserving
  `resolveTransitiveSourceCol`, so the prefix-delete arm's binary-prefix-scan
  collation invariant (its INTERNAL-error guard) is untouched.
- **Rename propagation does not re-stamp `coarsenedKey`** — a source column
  rename carries the rewritten record forward with the old column names inside
  the informational stamp until the next refresh shape-rebuild/recreate.
  Cosmetic; called out in the field's doc comment.
- **Coarsening classification is conservative for custom collations**: any
  non-equal, non-BINARY output collation counts as coarsening, so a custom
  collation that is genuinely *finer* than the source's would warn spuriously
  (no such built-in exists; deemed acceptable).
- **REFRESH-loud-during-window** is pinned as intended (re-fill posture). If
  the migration design instead wants refresh to LWW-merge, that is a one-line
  contract change (route refresh through a tolerant replace) — flag for the
  reviewer to confirm the posture.
- **Runtime collision telemetry** remains parked in backlog
  (`mv-collation-collision-telemetry`), as the source ticket specified.

## Review checklist suggestions

- Soundness of the chain-walk allowlist in `coarsened-key.ts` (any node type
  that can fan out a source row but slips through `ROW_PRESERVING_CHAIN`?).
- The inverse-projection LWW delete path: `applyInverseProjection`'s UPDATE
  arm deletes the old image's key then upserts the new — under a coarsened key
  a collation-equal sibling's row is the one deleted; confirm no consumer of
  the reported `BackingRowChange`s (cascade, sync change-log) is confused by a
  merge-shaped update.
- `backingShapeMatches` / adopt-gate parity: K' derivation is deterministic
  over a re-planned body, so refresh takes the data-only fast path and the
  store adopt gates compare equal — covered by the refresh round-trip test,
  but the store rehydrate path itself is only exercised indirectly.
