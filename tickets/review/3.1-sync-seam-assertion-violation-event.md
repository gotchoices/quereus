# Review: surface inbound assertion violations to the host; converge instead of diverging

## What this ticket did (sync half of detect-and-notify)

The engine seam already gained a **report mode** in the prereq ticket
`sync-seam-assertion-report-mode` (`Database.ingestExternalRowChanges(batch,
{ assertionFailureMode: 'report' })` → returns `IngestExternalChangesResult.assertionViolations`
instead of throwing, and the batch commits). This ticket wired the sync layer to it:

- **`store-adapter.ts`** — the single end-of-invocation incremental seam call now
  passes `assertionFailureMode: 'report'` (alongside the existing
  `applyForeignKeyActions`) and copies the returned `assertionViolations` into
  `ApplyToStoreResult.assertionViolations`. The bootstrap/finalize paths are
  untouched (they skip the seam entirely), so they never carry violations.
- **`protocol.ts`** — `ApplyToStoreResult` gains optional
  `assertionViolations?: AssertionViolation[]` (imported from `@quereus/quereus`).
- **`events.ts`** — new `AssertionViolationEvent { assertion, samples }`,
  `SyncEventEmitter.onAssertionViolation`, the listener set, and
  `SyncEventEmitterImpl.emitAssertionViolation`. Exported from the package index.
- **`admission.ts`** — `applyDataToStore` emits one `onAssertionViolation` per
  returned violation **only on the success branch** (after the whole-batch-throw
  catch and after `throwIfApplyErrors` passes). Convergence is not aborted; the
  metadata commit that follows proceeds. Because every ingress modality (wire,
  non-streaming snapshot, streaming flush) routes through `applyDataToStore`, all
  paths get the event for free — and bootstrap paths carry no violations, so none
  fires there.

Net effect: the old "throw → uncommitted metadata → idempotent retry suppresses
the seam batch → MV/watch for the violating row never re-driven → silent
divergence" chain is gone. The violating row's MV maintenance and watch capture
land on the **first** apply; the host learns about the broken local invariant
via the event.

## Why it matters

An incremental materialized view (or `Database.watch` subscriber) over a synced
base table previously diverged silently for a row whose inbound merged state
tripped a local global assertion — base table had the row, MV did not, until a
manual refresh. This restores convergence-on-first-attempt and gives the host a
typed signal to drive its own policy (alert / audit / out-of-band reconciliation).

## Distinction the reviewer should hold onto

Two failure shapes that must stay orthogonal in `applyDataToStore`:

- **Assertion violation** (this ticket): collected in
  `result.assertionViolations`, surfaced on the **success** branch, does **not**
  abort, metadata commits. Detect-and-notify.
- **Per-change storage failure** (separate concern,
  `sync-apply-per-change-errors-ignored`): collected in `result.errors`, aborts
  via `throwIfApplyErrors` **before** the success branch, no metadata committed,
  whole batch re-resolves next sync. Unchanged.

Order in `applyDataToStore` matters: the abort gate (`throwIfApplyErrors`) runs
**before** the violation emission, so a batch with both a storage error and an
assertion violation aborts and emits no violation event (correct — the data did
not fully land).

## Honest gaps / things to probe (reviewer: treat tests as a floor)

- **No explicit bootstrap "no event fires" test.** Argued by construction: the
  adapter passes report mode only on the incremental seam call, which bootstrap
  flushes skip, and `bootstrapFinalize` returns no `assertionViolations`. A
  defensive test (a bootstrap path with an assertion present asserting zero
  `onAssertionViolation`) would close this by observation rather than argument —
  consider adding if you want belt-and-suspenders.
- **FK-actions interplay** is covered only indirectly (the existing
  `foreign-key-actions opt-in` tests still pass, and report mode is passed
  alongside `applyForeignKeyActions`). No single test asserts both report mode +
  `applyForeignKeyActions: true` detect-and-notify on a violation while a cascade
  also applies. Low risk (the two seam options are independent) but unproven
  together.
- **`samples` shape** is asserted only as non-empty, not by content. The engine
  caps and shapes them (`MAX_VIOLATION_SAMPLES`, assertion SELECT output, not full
  rows) — intentionally loose here so the test does not pin engine-internal
  formatting, but a reviewer may want a tighter assertion if the host contract
  should guarantee more.
- **Multiple-violations-in-one-apply** (one event each) is implemented (the loop
  in `applyDataToStore`) but not separately tested — the single-assertion case is
  the only coverage.
- The `onAssertionViolation` listener fires **synchronously inside** the apply,
  before `commitMetadata`. A listener that throws would propagate out of the apply
  (same as the other emitters here, which do not guard listener exceptions). If
  host-listener isolation is desired, that is a broader event-emitter concern, not
  scoped here.
