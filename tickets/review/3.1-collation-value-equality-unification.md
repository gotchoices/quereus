# Unify SQL value equality on one function

## What changed

`util/comparison.ts` had two ways to ask "are these two SQL values the same
value?" that disagreed on numeric-storage-class cross-representation pairs
(`sqlValuesEqual(5n, 5)` was `false`; `compareSqlValuesFast` said they were
equal). Every real caller of the old `sqlValuesEqual` was a "did this value
change?" check (FK parent-column change detection, constraint-check old/new
diffing, DML value-identical-upsert suppression, planner mutation cell
comparison) where the `===`-based divergence was a latent bug.

- Added `sqlValueIdentical(a, b)` next to `rowsValueIdentical` —
  `compareSqlValuesFast(a, b, BINARY_COLLATION) === 0`. Numeric-storage-class
  tolerant (`5n` equals `5`, `true` equals `1`), bytewise for blobs,
  canonical-JSON for OBJECT-class values, byte-exact for text.
- Rewrote `rowsValueIdentical` as a loop over `sqlValueIdentical` (was a
  duplicate inline `compareSqlValuesFast` loop) so the two contracts cannot
  drift again.
- Deleted `sqlValuesEqual` entirely.
- Migrated all 9 call sites across `foreign-key-actions.ts` (3),
  `constraint-check.ts` (1), `dml-executor.ts` (4), `multi-source.ts` (1),
  `single-source.ts` (1) to `sqlValueIdentical`.
- Updated the "Representation-sensitive keys" comment at
  `dml-executor.ts` ~331 to reflect that the bigint/number half of that
  concern is resolved; the affinity-coercion and enforcement-collation
  halves remain open (unchanged scope, documented as a separate concern at
  that site — not touched here).
- Added `sqlValueIdentical` to the `src/index.ts` public export list
  alongside `rowsValueIdentical`.
- Renamed the `sqlValuesEqual` describe block in
  `test/utility-edge-cases.spec.ts` to `sqlValueIdentical` and added two new
  cases: cross-representation identity (`5n`/`5`, `true`/`1`, `false`/`0`)
  and TEXT-vs-NUMERIC non-identity (`'1'`/`1`).
- Retargeted the hash-key/equality agreement assertion in
  `test/util/json-canonical.spec.ts` (~85-99) at `sqlValueIdentical`. Its
  fixture pairs are all OBJECT-class (no bigint/number pair), so the
  assertion held unchanged — fixture was not modified.

## Behavior change (intended)

- `sqlValueIdentical(5n, 5)` → `true` (was `false` under old `sqlValuesEqual`).
- `sqlValueIdentical(true, 1)` → `true` (was `false`).
- `sqlValueIdentical(null, null)` → `true`; `(null, 0)` → `false`. Unchanged.
- `sqlValueIdentical('1', 1)` → `false` (TEXT vs NUMERIC storage class). Unchanged.
- Blobs: bytewise, differing length → `false`. Unchanged.
- JSON objects: key-reorder-equal → `true` (canonical-string cache shared with
  `compareSqlValuesFast`'s OBJECT path — same cache instance, no perf
  regression, no semantic change).
