## What landed

RETURNING-through-view, previously rejected everywhere with the
`returning-through-view` diagnostic, is now supported for the documented shapes.
There are **two mechanisms**, chosen by the view shape:

### Single-source (all three ops, incl. computed view columns) — rewrite-onto-base-op
`planner/mutation/single-source.ts` `rewriteViewReturning()` rewrites the user's
RETURNING clause **into base terms** (each view-column reference substituted to its
base-term lineage via the same `remapper` + `makeViewColumnDescend` machinery used
for WHERE/SET) and attaches it to the rewritten base statement. The base-table
builder's own RETURNING machinery then produces a relational `ReturningNode`:
- the user's **view-spelling** is preserved as the result-column alias (so
  `returning user_id` is named `user_id`, not the base `id`);
- unqualified columns bind to **NEW** for insert/update and **OLD** for delete, so
  the result is the post-mutation (or, for delete, the deleted) view image;
- **computed** view columns re-evaluate against the post-mutation base values;
- `returning *` expands to the full view column list.

`ViewMutationNode.resultRelation()` finds the relational base op; no separate
`returning` child is built. This path is **robust** against an update that changes
a predicate column (it reads NEW/OLD, not a re-query). **MV write-through inherits
it verbatim** (an MV is single-source).

### Multi-source inner-join update/delete — re-query of the view
The view row spans both base tables, so it is not recoverable from the per-side
base ops. `building/view-mutation-builder.ts` `buildMultiSourceReturning()` builds a
**re-query** — `select <returning> from <view> [where <user where>]` — as a
separate `ViewMutationNode.returning` child with a `returningTiming`:
- **delete** → `pre`: captured **before** the base delete fires (the rows are about
  to disappear);
- **update** → `post`: captured **after** (the post-mutation image).
The columns resolve naturally against the view (computed columns re-evaluated by the
view body). `returning *` works (`select *` from the view).

### Node / emitter / block wiring
- `view-mutation-node.ts`: `getType()` returns the view's RelationType when a
  RETURNING result exists (else the scalar void shape); `getAttributes()` forwards
  the result relation's attributes; `getChildren`/`getRelations`/`withChildren`
  thread the optional `returning` child (order: baseOps, returning, envelope). The
  relational base op / `returning` child appearing in `getRelations()` makes the
  attribute-provenance walk treat the forwarded RETURNING attributes as forwarded,
  not originated.
- `runtime/emit/view-mutation.ts`: materializes RETURNING rows eagerly inside `run`
  (so base writes fire regardless of block position — matching the void path's eager
  drain), then yields them. Three branches: separate re-query (pre/post), relational
  base op (single-source), void.
- `runtime/emit/block.ts`: a `ViewMutation` is excluded from block-result selection
  only when its `getType()` is **not** a relation (void). A RETURNING view mutation
  is relational and is eligible as the block result.

`packages/quereus/test/logic/93.4-view-mutation.sqllogic` § RETURNING covers (this
is the **floor**, not a ceiling — see gaps below):
- single-source insert/update/delete RETURNING with rename + a computed column
  (`doubled = rate*2`), asserting the computed column is evaluated post-insert,
  post-update, and over OLD on delete;
- multi-row insert RETURNING (one row per inserted row, insertion order);
- a filter view: the selection-predicate constant-FD default (`color='green'`)
  appears in the RETURNING NEW row; `returning *` on a single-source view;
- multi-source inner-join: update of a child-owned column, update of a parent-owned
  column (PK hidden by the projection), an update touching **both** sides, and a
  delete — each returning the joined view-projected row; `returning *` on the join
  view.
- `93.4` line ~192: the former multi-source-delete-RETURNING rejection is now a
  positive assertion.
- `93.2-view-mutation-pending.sqllogic`: the single-source rejection was removed; a
  multi-source (join) **insert** RETURNING is asserted as still rejected.
- `53.1-materialized-view-write-through.sqllogic`: MV delete `returning *` now
  asserts the deleted view row.
- `test/quereus/view-mutation-substrate.spec.ts`: the unit test that asserted the
  rejection now asserts a **relational** `ViewMutationNode` (its attributes are the
  view-projected RETURNING columns; single-source has no separate `returning` child;
  the base op is relational; the real base `DmlExecutorNode` is still present).

**Reviewer: please probe these areas (test floor, not ceiling):**
- RETURNING with bound parameters in the predicate/expressions.
- RETURNING expressions combining multiple view columns / functions.
- The **multi-source update fragility** (below): an update that changes a column
  used in its own WHERE — assert the documented behavior and decide if it needs a
  diagnostic rather than a silent miss.
- RETURNING through an aliased / column-list-renamed (`create view v(a,b) as …`)
  join view.
- Mutation-context (`with context …`) + RETURNING together.
- RETURNING through a view whose body references a **projected-away** base column
  (see gap on column-reference scoping).

## Known gaps / honest caveats (treat my work as a starting point)

1. **Multi-source UPDATE re-query is predicate-fragile.** The post-mutation update
   re-query matches by the user predicate, so an update that **changes a predicate
   column** will not recapture that row (returns the wrong/empty set). Postgres-grade
   correctness would need per-row capture at mutation time. The single-source path
   has **no** such limitation (NEW/OLD). Documented in `docs/view-updateability.md`
   § `returning` Clauses. Not covered by a test — consider adding one and deciding
   whether to reject loudly instead of silently mis-capturing.
2. **Projected-away base columns are referenceable in RETURNING** (single-source),
   because the rewrite passes an unmapped column reference through to the base op
   (same property the existing single-source WHERE/SET rewrite has — referencing a
   non-view base column in a view WHERE already resolves against the base table).
   Consistent, but arguably a view-encapsulation leak. Confirm acceptable or file a
   follow-up to validate top-level RETURNING/WHERE column refs against the view's
   column set (the multi-source re-query naturally rejects unknown columns).
3. **OLD/NEW qualifiers through a view are not honored.** A `returning old.col` /
   `new.col` reference loses its qualifier in the single-source rewrite (so the base
   op's default NEW/OLD binding applies); the multi-source re-query has no OLD/NEW at
   all. The documented surface is unqualified / view-qualified view columns.
4. **`returning *` is supported through views** (expands to the view's columns), even
   though base-table `returning *` is currently rejected (`RETURNING * not yet
   supported`). This is an intentional convenience for the view case but is an
   asymmetry worth a glance.
5. **Multi-source (join) INSERT RETURNING is rejected** (`returning-through-view`):
   the minted shared surrogate is not threaded into a RETURNING projection. The
   coordinating ticket `view-mutation-shared-surrogate-insert` has landed, but
   surfacing the minted key through a projection is additional work — deferred.
6. **Decomposition-backed logical-table (lens) RETURNING is still rejected**
   (`decomposition.ts` keeps its `rejectReturning`); out of scope here.
7. **RETURNING row order is not guaranteed** (SQL has no ORDER BY on RETURNING). The
   sqllogic harness is order-sensitive; the tests rely on insertion order
   (single-source) and view scan/PK order (multi-source re-query), which are
   deterministic for the memory vtab but not contractually guaranteed.
