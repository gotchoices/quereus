# Review: store-and-forward outbound relay

## What landed

The `store-and-forward` disposition's **outbound half**. The prereq
(`sync-store-and-forward-hold`) made apply durably **hold** a straggler's
out-of-basis change marked *forwardable*. This ticket makes those forwardable
changes actually **reach** peers that still hold the table, by folding them into
this peer's outbound `getChangesSince` → `ChangeSet[]` return — **no new transport
surface** (client/coordinator unchanged, verified by typecheck + their test suites).

### Core change (`sync-manager-impl.ts`)
- New private `collectForwardableChanges(peerSiteId, sinceHLC?)`:
  - Source = `quarantine.listForwardable()` (from the prereq).
  - **Echo exclusion**: drops entries whose `change.hlc.siteId === peerSiteId`
    (the change's HLC siteId is the straggler origin).
  - **Watermark filter**: when `sinceHLC` defined, keeps only
    `compareHLC(change.hlc, sinceHLC) > 0`; when undefined (from-zero), keeps all.
  - Full horizon-bounded scan, no early-exit (parallels `collectAllChanges`); empty
    and zero-cost in the no-straggler case.
- `getChangesSince` merges `[...dataChanges, ...forwardable]` before
  `buildTransactionChangeSets` (only allocates the spread when `forwardable.length > 0`).
- `relayed` cumulative counter (bumped in `collectForwardableChanges`), surfaced in
  `getUnknownTableStats()` (shape now `{ ignored, quarantined, forwarded, relayed, byTable }`)
  and the `manager.ts` interface.

### The load-bearing invariant (verify carefully in review)
A forwarded change keeps its **original `hlc` + `siteId`**. The consumer advances
its per-peer `lastSyncHLC` to `max(returned ChangeSet.hlc)`, so forwardable changes
**must** be filtered `> sinceHLC` or a forwarded-only round would **regress** the
watermark and trigger a re-scan flood. The `> sinceHLC` filter makes the merged
response a safe contiguous prefix (everything ≤ the batch max is present), identical
to the change-log contract. Convergence + loop-freedom rest on original-HLC identity
+ the scalar watermark — there is **no** per-table peer-membership oracle.

New spec `store-and-forward-relay.spec.ts` (9 tests, lightweight stub harness mirroring
`unknown-table-disposition.spec.ts` — `SyncManagerImpl` + recording `applyToStore` +
`known`-set basis oracle; multi-peer via fixed `config.siteId`; a `pull()` helper that
simulates the transport advancing the per-peer watermark to `max(ChangeSet.hlc)`):

- **3-peer convergence** S→R→H: R holds C forwardable; H pulls and gets C with
  **original hlc + siteId (S)**; H materializes it (asserted via H's
  `columnVersions` carrying S's HLC) and then serves it from its own change log.
- **No re-relay after watermark advance** — re-pull at the advanced watermark omits C.
- **Ping-pong loop-freedom** between two non-holders R1/R2 — C exchanged exactly
  once, each re-holds exactly one forwardable entry (HLC-keyed idempotent), receiver
  re-disposes (assert `forwarded` bumped), then quiesces (no re-send, no growth).
- **Echo exclusion** — author S pulling never gets its own C back (from-zero + delta).
- **Below-watermark not relayed** — `HLC ≤ sinceHLC` filtered; no watermark regression.
- **Ordering** — forwarded `orders` change (wall 1500) interleaves between two live
  change-log writes (1000, 2000) in global HLC order.
- **From-zero** — all forwardable changes relayed, re-grouped into the original txn.
- **GC vs relay** — pruned at the horizon ⇒ no longer relayed.
- **Relay telemetry** — `relayed` counts re-offers (held once, relayed many).

## Honest gaps / where to push (your tests are a floor)

1. **Lightweight harness, not real-engine end-to-end.** The relay specs assert at the
   SyncManager/CRDT-metadata layer (deliberately mirroring `unknown-table-disposition.spec.ts`),
   **not** through a real `Database` + `StoreModule` like `echo-loop-quiescence.spec.ts`.
   A real-engine straggler→relay→holder test (actual SQL tables, `select * from` on the
   holder) would harden the materialization claim. Considered out of scope here; worth a look.
2. **Bound interaction under truncation is verified by argument + an ordering test,
   not by a forced-truncation test.** I did not add a test that sets a tiny `batchSize`
   and feeds a *merged* (change-log + forwardable) set exceeding it to assert the
   forwardable tail is deferred and re-collected next round (still `> sinceHLC`). The
   reasoning (comment in `getChangesSince`) mirrors the existing bounded-extraction
   contract, but an explicit truncation test would close the loop. **Highest-value gap.**
3. **`relayed` counter over-counts under truncation.** It is bumped at collection time
   by candidate count, before `buildTransactionChangeSets` may defer some past the
   batch bound (re-counted next round). Documented as "relay activity, not distinct
   deliveries"; not directly tested (default batchSize=1000 never truncates in the specs).
   If the over-count is judged misleading, either count post-grouping or drop the counter
   (it is the ticket's optional item).
4. **Watermark is a local test variable, not durable peer-state.** `pull()` tracks the
   watermark in-test; it does not exercise `updatePeerSyncState`/`peerStates`. Faithful to
   how the transport advances the watermark, but the durable peer-state path is untested here.
5. **Multi-hop depth ≤ 2.** Ping-pong covers S→R1→R2→R1. A 3+ relay chain to a distant
   holder converges by the same identity argument but isn't explicitly tested.
