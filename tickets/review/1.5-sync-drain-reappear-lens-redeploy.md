# Review: low-latency scoped drain when a lens redeploy re-maps a basis table back

## What this ticket did

Hooked the **second** reappearance path (the first, inbound `create_table`, landed in the
`sync-drain-reappear-inbound-ddl` prereq): a local `apply schema` lens redeploy that re-maps a
basis table from `detached` back into the basis now drains that table's held out-of-basis
changes **immediately** — as a separate post-commit apply unit — instead of waiting up to one
periodic-sweep interval.

The change is small and entirely additive; it reuses the prereq's `drainOnReappear` config flag
and exported `drainReappearedTables(ctx, tables)` helper. No new flag, no new helper, no worker
changes.

### Implementation (`sync-manager-impl.ts`, `recordLensDeployment`)

- Imported `drainReappearedTables` from `change-applicator.js`.
- Declared a `reappeared: Array<{schema; table}>` before the per-key classification loop.
- Inside the loop, after the record is built, push `{schema: record.schema, table: record.table}`
  whenever `wasDetached && !isDetached` (the existing `wasDetached = prior?.state === 'detached'`
  and `isDetached = state === 'detached'` locals — i.e. the precise `detached → present`
  transition).
- After `if (wroteAny) await batch.write()` **and** the `onBasisTableLifecycle` emit loop, call
  `await drainReappearedTables(this, reappeared)`.

Ordering rationale (verify): the drain runs **after** the lifecycle batch is durable and its
events have fired, so the basis oracle (`getTableColumnNames`) sees the re-attached table and
the held changes LWW-resolve against the present basis. Symmetric with the inbound-DDL path,
which drains after the admitting `applyChanges` batch commits.

Advisory guarantee (verify): `drainReappearedTables` early-returns on `!drainOnReappear`/empty
and per-table try/catches + swallows failures, so a drain throw can never abort the deploy
(itself advisory bookkeeping wrapped in try/catch by the store-module forwarder).

### Why the precise `detached → present` transition (not "inBasis newly true")

Held quarantine entries only ever exist for tables that were out of basis, so the precise
transition skips spurious scoped `quarantine.list` scans for brand-new tables (which have no
`prior` and so are excluded anyway). It is a cheapness optimization, **not** a correctness
gate — the oracle gate inside `drainHeldChanges`/`drainTableGroup` makes any over-trigger a
harmless no-op (an absent table returns `undefined` column names ⇒ the group is skipped, 0
drained).

## Tests added (`basis-lifecycle-recorder.spec.ts`)

A new `describe` block `recordLensDeployment — low-latency drain on detached → re-mapped` with its
own harness (mutable `getTableSchema` oracle + `applyToStore` in-memory store + a `failApply`
crash hook), mirroring the drain harness in `unknown-table-disposition.spec.ts`. Five cases:

1. **Detached → re-mapped triggers drain.** map → detach → seed held → flip oracle present →
   re-map. Asserts held entries drained without an explicit `drainHeldChanges`, store value
   materialized, column version written, one `onHeldChangesDrained` with `applied + skipped ===
   drained`, and `onRemoteChange` keyed by the held change's original origin site.
2. **Idempotent re-deploy does not drain.** A re-deploy with no `detached → present` transition
   fires no drained event; held entries for a still-detached table (present in the oracle so a
   drain *would* land if one fired) survive.
3. **Brand-new table does not drain.** First-ever classification (no `prior`) with held entries +
   oracle present fires no drain; the entries remain drainable via an explicit sweep.
4. **`drainOnReappear: false` defers to the sweep.** Detach + seed + re-map with the flag off:
   held entries survive the deploy; a later explicit `drainHeldChanges()` clears them.
5. **Drain failure is swallowed.** `failApply` armed for the re-mapped table: the deploy
   completes (lifecycle record re-classified `directly-mapped`, `onBasisTableLifecycle` emitted),
   no drained event, entries stay held; a later sweep drains cleanly with no double-apply.

## Known gaps / things for the reviewer to probe

- **No end-to-end (real `Database` + `StoreModule`) test for the lens-redeploy path.** The
  prereq's e2e (`sync-drain-e2e.spec.ts`) covers the inbound-`create_table` revival through the
  real store adapter (live SQL row, `Database.watch`, MV maintenance). This ticket's tests are at
  the CRDT-metadata + in-memory-stub level only — the same level as the prereq's
  `unknown-table-disposition.spec.ts` blocks. A real `apply schema` redeploy that detaches then
  re-maps a basis table with held stragglers is **not** exercised end-to-end. If the reviewer
  wants parity with the inbound path, that is a candidate fix/follow-up ticket (extend
  `sync-drain-e2e.spec.ts` with a redeploy-driven revival).
- **Case sensitivity of the scoped drain lookup.** The drain calls
  `quarantine.list(record.schema, record.table)`, and `qt:` keys are built case-sensitively from
  the held change's `schema`/`table`. `record.schema`/`record.table` is the display name from
  `db.schemaManager` enumeration (`table.schemaName`/`table.name`). If a real basis table is
  enumerated with different casing than the inbound change carried, the scoped scan could miss
  (the periodic no-arg sweep would still catch it). This is the **same** latent concern the
  inbound-DDL path already has (it uses `migration.schema`/`migration.table`); the tests here use
  all-lowercase names so they don't surface it. Worth a reviewer's eye on whether the codebase
  guarantees consistent casing across these seams.
- **`detached → present` covers any in-basis target state** (`directly-mapped`,
  `derivation-source-only`, `unreferenced`) — the tests only exercise `detached → directly-mapped`.
  The other targets are also valid reappearances (the table is back in the basis, oracle resolves
  it), and the code handles them uniformly, but they are untested. Low risk; flagging for
  completeness.

## Suggested review focus

- Confirm the drain truly runs **after** the lifecycle batch + events (ordering invariant), and
  that no path lets a drain throw escape `recordLensDeployment`.
- Confirm the `wasDetached && !isDetached` condition matches the ticket's intended
  `detached → present` semantics and excludes brand-new tables and idempotent re-deploys.
- Sanity-check the new test harness against the reference harness in
  `unknown-table-disposition.spec.ts` for any drift in the `applyToStore` / oracle stub contract.
