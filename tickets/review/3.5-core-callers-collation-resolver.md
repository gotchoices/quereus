# Review: core callers resolve collations against the database

## What landed

Two subsystems inside `packages/quereus` were still passing collation *names* to
`compareSqlValues(a, b, name)`, which resolves them against a process-global map holding
only the three built-ins. A collation registered with `db.registerCollation(...)` — or a
built-in the embedder *replaced* — was invisible there.

**Materialized-view maintenance.** `MaterializedViewManagerContext` gained
`getCollationResolver()` (supplied structurally by `Database`, no wiring change at the one
construction site). Each backing primary-key descriptor is now a `BackingPkColumn` —
`{ index, desc?, collation?, collationFn }` — built once per plan by
`resolveBackingPkColumns()`. The three per-row helpers in `-apply.ts`
(`residualRowMatchesKey`, `backingPkEqual`, `residualRowMatchesBasePrefix`) and the two
sites in `-views.ts` (`lookupCoveringConflicts`' UNIQUE match and self-conflict skip) now
call `compareSqlValuesFast` with a pre-resolved function; `CoarseningWatchColumn` carries a
resolved `sourceCollationFn`. No plan shape changed for callers outside these files.

**Satisfiability checker.** `checkSatisfiability` gained a sixth parameter,
`collationResolver?: CollationResolver`. Every column mentioned by the conjuncts, domains,
or bindings has its declared collation resolved once, before the conjunct loop. `BINARY`
short-circuits; otherwise the supplied resolver is used, else `builtinCollationResolver`.
If any name is unresolvable the whole check returns `'unknown'` — the already-supported
"cannot prove unsatisfiable" answer — so the `Filter` stands. Both call sites now pass a
real resolver (`rule-filter-contradiction.ts` from `ctx.db`, `set-op.ts` from the leg's
`PlanningContext`); no call site is left without one.

## The bugs this actually fixes (all reproduced on `main` before the change)

Each of these returned rows that both conjuncts individually return, yet the conjunction
returned nothing, because the optimizer proved the `WHERE` unsatisfiable:

1. **Custom collation degraded to BINARY.** `db.registerCollation('NOCASE', lengthOnly)`;
   `select id from t where x = 'a' and x = 'b'` on a `collate nocase` column returned `[]`
   though each conjunct alone returns the row.
2. **`COLLATE` wrapper erased.** `where x collate nocase = 'a' and x collate nocase = 'A'`
   on a BINARY column returned `[]`.
3. **Value-changing `CAST` erased.** `where x = '1' and cast(x as integer) = 1` returned
   `[]`.

(2) and (3) were **not in the ticket's scope** — they came from `unwrap()` in the same file
stripping `CastNode` and `CollateNode` unconditionally. Both are the same defect class the
ticket names (a false `'unsat'` deletes rows), both were reachable from plain SQL, and both
are one-line fixes, so they are fixed here rather than deferred: `unwrap` now strips only a
**no-op cast** (`isNoOpCast`, target logical type == operand's) and never a `COLLATE`. This
change is strictly conservative — it can only turn `'unsat'` into `'unknown'`/`'sat'`, i.e.
lose a fold, never add one. `constraint-extractor.ts` already reasoned this way for
`COLLATE`; the two are now consistent.

Also fixed, though harder to reach from SQL: `lookupCoveringConflicts` compared a source
`UNIQUE` column's values under the global collation, so a covering materialized view would
admit a duplicate under a redefined `NOCASE` (test 1 in the new MV spec fails without the
change).

## Deviation from the ticket's design

The ticket said the resolver's throw-on-unregistered-name "would be an internal bug at this
point since the name came from a column type". **It is not.** `validateCollationForType`
only checks names for types that declare a supported list (TEXT), so
`create table t (id integer primary key, k integer collate frobnicate)` reaches the planner
with an unregistered name — see the `backlog/feat-ddl-accepts-registered-collations`
ticket, which documents exactly this hole. Passing that name to `db.getCollationResolver()`
from an optimizer rule made `select * from t where k = 1` throw where it previously
succeeded (verified). `checkSatisfiability` therefore catches the resolver's throw, logs on
the `warn` channel, and treats the collation as unresolvable → `'unknown'`. An optimization
rule must not turn a working query into an error.

- `test/optimizer/predicate-contradiction.spec.ts` — 6 unit tests on collation resolution
  (BINARY unsat / NOCASE sat / no-resolver+custom-name → unknown / resolver makes it
  decidable both ways / throwing resolver → unknown / no declared collation → BINARY) and
  6 end-to-end tests, three of which are the reproductions above.
- `test/mv-custom-collation-maintenance.spec.ts` (new) — a covering-MV `UNIQUE` conflict
  under a redefined `NOCASE` is rejected; a non-conflict is not invented; a collation-equal
  `UPDATE` keeps the MV consistent (exercises the self-conflict skip); the covering path and
  `uniqueEnforcementCollations` are asserted to agree on the enforcement collation and to
  resolve to the *same function object* through the database resolver. Plus three direct
  unit tests of the `-apply.ts` per-row helpers with a custom `collationFn`.
- `yarn build` (whole monorepo), `yarn workspace @quereus/quereus run lint`, `yarn test` —
  all clean. 6633 quereus tests passing, nothing skipped or disabled.

## Known gaps — treat these as the floor, not the finish line

- **The three `-apply.ts` helpers are only unit-tested.** The residual-recompute and
  prefix-delete maintenance arms they serve need an aggregate or lateral-TVF body, and the
  discriminating case (two group keys that are collation-equal but byte-different) is
  **unreachable through SQL today**: `GROUP BY` normalizes its keys through
  `resolveKeyNormalizer`, a hard-coded three-name switch that ignores the database registry
  entirely, so the grouping splits the rows before maintenance ever sees them. That is a
  separate reachable wrong-results bug — `select k, sum(v) from src group by k` returns two
  groups where `where` and `distinct` on the same column agree there is one — filed as
  `fix/bug-key-normalizer-ignores-database-collations` with a verified repro. It also hits
  window `PARTITION BY`, the hash-join Bloom filter, and `AS OF`. **Reviewer: if you want
  end-to-end coverage of those helpers, that ticket must land first.**
- **`CoarseningWatchColumn.sourceCollationFn` has no custom-collation test.** The existing
  `mv-coarsening-collision-telemetry.spec.ts` suite (which passes) covers the built-in
  names only. Reaching a *custom* source collation on a coarsened key needs a shape where
  the source column's collation is non-BINARY and the output collation is strictly coarser;
  I did not find one that both the coarsened-key deriver and column DDL accept.
- **`set-op.ts`'s `collationResolver` has no dedicated test.** The flagless set-op write
  oracle now receives a real resolver, and `test/logic/93.6-set-op-flagless-write.sqllogic`
  stays green, but no test drives a custom-collated leg discriminator. A false `'unsat'`
  there would skip a leg the incoming row belongs in.
- **`yarn test:store` not run** (routinely exceeds the 10-minute agent idle budget). The
  diff touches no store code, but `54-covering-mv-enforcement.sqllogic` runs under the store
  module too.
- **Plan-build-time freeze.** `resolveBackingPkColumns` bakes the comparator into the plan.
  A bare `db.registerCollation(...)` after an MV is registered leaves that MV's plan on the
  old comparator (schema changes do re-register). Same exposure as index comparators, which
  resolve once at table open. Parked as a `NOTE:` at `resolveBackingPkColumns`
  (`database-materialized-views-plan-builders.ts` ~91), per the ticket's instruction — not
  a ticket.

## Things worth a reviewer's skepticism

- `resolveColumnCollations` walks every conjunct to collect column indices before the main
  loop — a second tree walk on a path that previously did none. It is skipped entirely when
  no `getCollation` is supplied, and `markUnknownForColumns` was refactored to share the
  walk, but the rule now pays it on every `Filter` it visits. If optimizer planning time
  regresses, that is where to look.
- The `'unknown'` bail is whole-check, not per-column: one unresolvable collation on column
  `b` prevents proving a contradiction on column `a`. The ticket explicitly chose this
  ("do not add a storage-class carve-out"), and it is sound, but it is a real loss of
  folding on a schema with a bad collation name.
- `checkSatisfiability`'s `tryResolve` swallows the resolver's exception (logged, not
  rethrown). Verify you agree that an optimizer rule is the right place to be lenient.
- The three unit tests of the `-apply.ts` helpers cast partial literals through
  `as unknown as ForwardResidualPlan`. They read only `backingPkDefinition` /
  `basePrefixLength`; if either helper grows a new field read, those casts hide it.
