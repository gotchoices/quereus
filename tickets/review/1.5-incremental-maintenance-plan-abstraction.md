### Phase 2 — equivalence harness (`test/incremental/maintenance-equivalence.spec.ts`)

`fast-check` + Mocha/ts-node. For each covering-index body shape × random
insert/update/delete sequence it asserts, as order-insensitive multisets:

> `read(select * from mv)` == `evaluate(<body> against source)`

The oracle is a **live re-evaluation of the MV's own defining SELECT against the source
table** — a genuinely independent computation from the materialized backing, so any
maintenance bug surfaces as a multiset mismatch. Asserted at three points per run:
baseline (committed seed), in-transaction (reads-own-writes), and post-`ROLLBACK`
(maintenance reverted in lockstep).

Four shapes, all over `src (id pk, a, b, k)`:
1. `select id from src` — single-column passthrough, PK only, no predicate.
2. `select id, a, b from src where k > 5` — multi-column + partial WHERE; the small `k`
   domain drives predicate-scope transitions.
3. `select b, a, id from src` — column-reordering projection (permutation, not identity).
4. `select id, a from src` — passthrough exercised by key-changing UPDATEs (the shared
   mutation generator rewrites `id` itself).

The generator emits inserts / non-key updates / **key-changing** updates / deletes against
a small id space (1–6) so collisions and scope transitions recur. PK-collision constraint
violations are swallowed per-mutation, justified by statement-level atomicity (a plain ABORT
reverts both the source write and its row-time backing maintenance — see
`53-...rowtime.sqllogic` §2), leaving source/MV still equivalent; any *non*-constraint error
(e.g. an `INTERNAL` from maintenance) propagates and fails the property.

## Known gaps / where to push (treat tests as a floor)

These are deliberate scope boundaries or honest limits — candidates for adversarial probing
or follow-on tickets, not necessarily defects:

- **Stub-arm guards are untested.** The `'full-rebuild'` / `'residual-recompute'` throw
  paths are unreachable through any public API (the builder never emits them), so no test
  exercises the `INTERNAL` guard. A white-box unit test that hand-constructs a stub plan and
  asserts `applyMaintenancePlan` throws would lock the guard wording — I did not add one
  because it requires reaching into private manager state. Worth a reviewer call on whether
  to add it now or defer to the cost-gate ticket (1.6), which makes `'full-rebuild'`
  reachable.
- **Harness omits NULL/3-valued predicates.** Generated column values are non-null integers,
  so the `where k > 5` shape never exercises a NULL predicate (NULL ≠ TRUE ⇒ out of scope).
  That edge is covered *deterministically* by `53-...rowtime.sqllogic` §14, not by the
  property harness. A reviewer may want the arbitraries widened to include NULLs.
- **Single-column PK only.** All four shapes key on `id`. Compound-PK maintenance and
  `order by`-seeded physical keys are covered deterministically by `53-...rowtime.sqllogic`
  §5/§15/§17 but **not** by the property harness. Natural future extension.
- **Positional row comparison.** The multiset comparator uses `Object.values(row)`, relying
  on `select * from mv` and the body sharing an identical column order. True for these shapes
  (MV columns derive from the body); a reviewer should confirm this assumption holds before
  new shapes are added that could alias/duplicate column names.
- **`numRuns: 40`, sequences ≤ 10** — modest budget chosen to keep the suite fast (~1s for
  all four shapes). Raise if deeper coverage is wanted.
- **Type exports widened.** `MaintenancePlan` and its three arm interfaces are now `export`ed
  (the predecessor was private). Intentional — the cost-gate (1.6) and general-bodies (3)
  tickets build on them — but confirm no unwanted surface leak.
