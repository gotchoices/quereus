## What landed

The AST rewrite is **retired**. The substrate is now the single propagation path for all
view/MV mutations; the single-source case is its trivial one-base-op path.

**New files**
- `planner/nodes/view-mutation-node.ts` — `ViewMutationNode extends PlanNode`
  (`nodeType = PlanNodeType.ViewMutation`). Holds `baseOps: readonly PlanNode[]` (the ordered,
  fully-built base-table DML subtrees) plus an unused `returning?` slot reserved for 3.2.
  Modeled as a **drop-in for the `SinkNode`** the rewrite used to return: `getType()` is a scalar
  INTEGER (affected-row count), `getChildren()` = baseOps, `getRelations()` = baseOps filtered to
  relational (= `[]` while children are Sinks, mirroring how `BlockNode` treats DML sinks),
  `withChildren` rebuilds, `computePhysical` = `{readonly:false, idempotent:false, deterministic:false}`.
- `runtime/emit/view-mutation.ts` — `emitViewMutation`: children are emitted as params (the scheduler
  drains each Sink, firing its base write, in list order); `run` returns `null` (side-effect only —
  RETURNING-through-view is rejected here).
- `planner/mutation/single-source.ts` — the **relocated** single-source projection-and-filter rewrite
  (`rewriteViewInsert/Update/Delete`, `analyzeView`, `MutableViewLike`, expression-transform helpers).
  This is the exact body of the deleted `building/view-mutation.ts`, with import paths fixed and
  `MutableViewLike` exported.
- `planner/building/view-mutation-builder.ts` — `buildViewMutation(ctx, view, req)`: calls `propagate`,
  re-plans each `BaseOp.statement` via `buildInsertStmt/buildUpdateStmt/buildDeleteStmt`, wraps in a
  `ViewMutationNode`.
- `test/quereus/view-mutation-substrate.spec.ts` — the parity regression (see below).

**Modified**
- `planner/mutation/propagate.ts` — keeps `classifyViewBody` / `ViewBodyClassification`; **adds**
  `BaseOp`, `MutationRequest`, `propagate(ctx, view, req): BaseOp[]`, and re-exports `MutableViewLike`.
  `propagate` calls the relocated `rewriteView*` and resolves `BaseOp.table` via
  `buildTableReference(...).tableRef`.
- `planner/building/{insert,update,delete}.ts` — the view/MV dispatch site now does
  `return buildViewMutation(ctx, view, { op, stmt })` instead of `return buildXStmt(ctx, rewriteViewX(...))`.
  (The dual `getView ?? getMaterializedView` lookup is unchanged, so MV write-through rides the same path.)
- `planner/nodes/plan-node-type.ts` — adds `ViewMutation = 'ViewMutation'`.
- `runtime/register.ts` — registers `emitViewMutation` for `PlanNodeType.ViewMutation`.
- `runtime/emit/block.ts` — the block result `valueIndex` now excludes **both** `Sink` and
  `ViewMutation` (both are void side-effect statements), so a view mutation in a multi-statement
  batch does not shadow a prior statement's value — matching base-table DML behavior.
- `docs/view-updateability.md` — Phase-1 implementation note flipped from "AST rewrite" to "via
  substrate (single-source = one base op)"; notes `building/view-mutation.ts` removed and the new
  surfaces; multi-source / RETURNING / tag rows left as not-yet-shipped.

## Design decisions (for the reviewer to scrutinize)

- **Design A (reuse-maximizing) was chosen, as the ticket recommended.** `ViewMutationNode` wraps the
  *fully-built* base-table DML subtrees (Sink-topped for non-RETURNING). For single-source the list has
  exactly one entry == today's rewrite output, so parity is **structural** (the wrapped subtree is
  byte-identical to what the rewrite re-planned).
- **`ViewMutationNode` is a `SinkNode`-shaped drop-in, not a relational node.** The ticket text said
  "well-behaved RelationalPlanNode", but the node it replaces (`SinkNode`) is itself scalar-typed and
  non-relational, and RETURNING-through-view is rejected in this phase — so a scalar (row-count) type is
  the faithful, parity-preserving choice. It still exposes `getChildren`/`getRelations`/`withChildren`
  so the optimizer and the change-scope / binding walks descend into the base DML subtrees. **Reviewer:
  confirm this is acceptable, or whether 3.2's RETURNING work should promote it to relational then.**
- **New import cycle `propagate.ts ↔ single-source.ts`** (propagate imports `rewriteView*`; single-source
  imports `classifyViewBody`). Both uses are function-call-time, not module-eval-time, so the ES-module
  cycle resolves cleanly and the build is green — but it is a new cycle worth a look.
- **`block.ts` exclusion of `ViewMutation`** is load-bearing for batch parity but is a global emit change;
  worth confirming it has no unintended effect on other batch shapes. When RETURNING-through-view lands
  (3.2) this exclusion must be revisited so a RETURNING view mutation surfaces its rows.

## Where to push hardest (honest gaps — tests are a floor, not a finish line)

1. **`getChangeScope()` / `Database.watch`.** The ticket asked for a change-scope assertion on a view
   mutation; the parity spec is **plan-shape only** and does not add a dedicated change-scope test. The
   design relies on the change-scope/binding walks descending via `getChildren` (as they do for a Sink
   under a Block). The view/MV sqllogic suite exercises the end-to-end write path and is green, but a
   **direct change-scope test for a view mutation is still missing** — add one or manually confirm the
   base-table write is reported.
2. **Batch/result-shadowing (`block.ts`).** Confirm `select 1; insert into gv …;` returns the `select`'s
   value (not null) and that no other batch shape regressed from the `ViewMutation`-exclusion change.
3. **Plan-snapshot / EXPLAIN.** No `test/plan` fixtures cover view DML (grep found none), so snapshot
   churn was nil — but `ViewMutationNode` now appears as the top node for view DML in any future EXPLAIN;
   sanity-check its `toString`/`getLogicalAttributes` rendering.
4. **MV write-through** rides the same dispatch; `53.1` is green, but spot-check that the row-time
   maintenance hook still fires correctly now that the plan top is `ViewMutationNode` rather than `Sink`.
5. **`BaseOp.table` resolution** in `propagate` calls `buildTableReference` a second time (the base
   builder resolves it again). Harmless/redundant for v1; confirm it doesn't double-register anything in
   the schema-dependency tracker.
6. **Concurrency note for 3.2:** the scheduler does not await sibling params before starting the next, so
   multiple base ops (3.2 multi-source) could execute concurrently/out-of-order. Irrelevant here (one
   base op) but the multi-source ticket must address ordering (likely chaining, not bare params).
