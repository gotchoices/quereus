## What changed

`IsolatedTable.findMergedUniqueConflict` used to check a non-PK UNIQUE constraint by
scanning the **entire** underlying table and doing one overlay point-lookup **per row**
(`isolated-table.ts`, old ~1291-1338). It now splits the merged view — this connection's
overlay superimposed on the underlying committed rows — into two disjoint halves and
searches each the cheap way:

```
merged view  =  (overlay rows)  ∪  (underlying rows with no overlay entry)
```

- **Phase 1 `findOverlayUniqueConflict`** — full-scans the (small, in-memory) overlay,
  skipping tombstones and the writer's own PK(s). A matching live overlay row is the
  conflict; its overlay value is what the merged view holds for that PK.
- **Phase 2 `findUnderlyingUniqueConflict`** — looks up underlying rows matching the
  constrained columns, skipping the writer's own PK(s) and any PK the overlay already
  owns (Phase 1's territory). The lookup is an **index seek** when
  `canSeekForConstraint` allows it, else the old full scan. `getOverlayRow` now fires
  only for the few candidates a seek returns, not once per underlying row.

Both phases run one shared matcher (`rowMatchesUniqueConstraint`) so they compare
identically (per-column enforcement collation + partial-UNIQUE predicate against the
merged row).

**Seek gate (`canSeekForConstraint`).** Seek only when the constraint is index-derived
(`derivedFromIndex` resolves to a live entry in `tableSchema.indexes`) **and every key
column's enforcement collation is BINARY**. The BINARY gate is load-bearing: the store's
physical index key bytes come from a separate encoder that ignores the DB collation
registry (`backlog/debt-store-index-keys-use-column-collation`), so a `NOCASE` seek for
`'B@X'` would physically miss a committed `'b@x'` the full scan catches. A `NOTE:` at the
gate flags when to widen it.

The seek `FilterInfo` is built by the new `makeSecondaryIndexEqSeekFilter`
(`filter-info.ts`), which reads seek values in **index-key order** (`index.columns[i].index`),
not `uc.columns` order, and populates EQ `constraints` so a module ignoring the index hint
still filters residually instead of returning the whole table.

## Behaviour changes reviewers should confirm

- **Conflict tie-break.** Phase 1 runs first, so when a constraint was *already* violated,
  an overlay-side conflict is now named in preference to an underlying one (affects the
  `UNIQUE constraint failed` `existingRow` / REPLACE eviction target). Under a satisfied
  constraint at most one conflicting row exists, so this only changes *which* row is named
  in the pre-violated case.
- **Correctness is collation-gated, not module-gated.** On memory the seek IS served
  (MemoryTable's `buildScanPlanFromFilterInfo` turns the EQ constraints into an index
  seek), so the memory tests genuinely exercise the seek arm — the ticket's TODO note
  ("only the store serves the seek") is imprecise. The store's real `analyzeIndexAccess`
  seek is exercised by `yarn test:store` and the existing `isolated-store.spec` UNIQUE
  tests.

## How it was validated

- `yarn test` — all workspaces green (quereus 6867, isolation 224, others).
- `yarn test:store` — LevelDB-backed logic tests green (6862 passing, 14 pending), which
  drives the store's actual secondary-index seek arm.
- `yarn lint` + `tsc -p tsconfig.test.json --noEmit` (isolation) — clean; `yarn docs:check`
  — clean.

New tests in `isolation-layer.spec.ts` (describe: *two-phase merged UNIQUE check (index
seek)*):

- **Four merged-view scenarios, parameterised over `unique(email)` (no index, full-scan
  arm) and `create unique index ux on t(email)` (seek arm):** overlay-side conflict
  (the case a naive seek loses), overlay-side resolution, tombstoned conflict, tombstone
  revival (`selfPks` honored in both phases).
- **`unique(email collate nocase)`** still catches a case-differing committed collision —
  proves the BINARY gate declines the seek and falls back to the NOCASE full scan.
- **Composite `create unique index ux on t(b, a)`** — proves seek values are read in
  index-key order, not column order.
- **NULL constrained value** — no seek built, two NULLs coexist (the outer NULL guard).
- **`OR REPLACE` eviction** returns the same `evictedRows` shape (user-facing `[id,email]`,
  no stray tombstone column) whether the conflict was found overlay-side (Phase 1) or
  underlying-side (Phase 2).
- **Two UNIQUE constraints, one `OR REPLACE`** — the first eviction's tombstone is skipped
  (not re-reported) by the second constraint's Phase 1; the row is evicted exactly once.
- **Row-count proof** — wraps the underlying module, counts rows yielded from `query()`
  during a constrained insert against 100 committed rows: `≤ 5` on the BINARY-index seek
  arm, `≥ 100` on the `collate nocase` full-scan arm. This is the only test that proves the
  optimisation *happened* rather than merely stayed correct.

## Known gaps / where to push (treat tests as a floor)

- **`evictedRows` shape is asserted via `IsolatedTable.update()`'s return value**, not by
  observing the downstream DML eviction pipeline (change-tracking / FK CASCADE / covering-MV
  maintenance that `evictedRows` ultimately drives). A stronger test would seed an
  `ON DELETE CASCADE` child and confirm the cascade fires identically for overlay-side vs
  underlying-side evictions.
- **No store-mode row-count assertion.** The counting-module trick is memory-specific; the
  store's real seek is only exercised indirectly (existing `isolated-store.spec` UNIQUE
  swap tests + `yarn test:store`). A dedicated store-mode seek-count test would harden the
  arm the ticket most cares about.
- **`drop index` mid-transaction with a stale `UniqueConstraintSchema`** is handled by
  `canSeekForConstraint`'s resolve-or-fall-back (returns null, no `!`-assert), but is only
  covered defensively — existing tests (`isolation-layer.spec.ts` DROP UNIQUE INDEX cases)
  clear the constraint entirely, so the stale-UC-naming-a-dropped-index path is not directly
  triggered. Low risk; flagged for completeness.
- **Partial-UNIQUE + seek interaction on the store**: the predicate is re-run in both phases
  regardless of whether the physical index excludes out-of-scope rows, so it is correct
  either way, but there is no explicit store-mode partial-UNIQUE-with-seek test beyond the
  memory `collate nocase`/index scenarios.
