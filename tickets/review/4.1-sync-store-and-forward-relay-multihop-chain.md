# Multi-hop relay chain (depth ≥ 3) — review handoff

## What was done

Added a `describe('multi-hop relay chain (depth >= 3)', ...)` block to
`store-and-forward-relay.spec.ts`, reusing the file's existing **metadata-layer**
harness verbatim (`makePeer`, `col`, `changeSet`, `pull`, `flatten`, `tablesIn`,
`maxSetHLC`, `RETIRED`). One new local helper, `buildChain()`, lives inside the
block (no new top-level helpers) and constructs a fresh `S → R1 → R2 → R3 → H`
topology per spec: `S` is the straggler origin (a bare siteId, never a peer);
`R1/R2/R3` are `store-and-forward` relays with `known: ['main.users']` (no
`orders`); `H` holds `['main.users', 'main.orders']`.

No production code changed — this is a pure test addition over already-shipped
relay behavior. `docs/sync.md` left unchanged (the relay section's loop-freedom is
stated "across hops" with no depth bound; the depth-2 ping-pong there is only an
illustration).

Four specs were added:

- **Three-hop delivery, exactly once, origin preserved.** `S→R1` direct apply,
  then from-zero `pull(R1→R2)`, `pull(R2→R3)`, `pull(R3→H)`. Asserts each relay
  re-disposed the forwarded change (`getUnknownTableStats().forwarded === 1` on
  R2/R3) and holds **exactly one** forwardable entry after the full chain; H does
  **not** hold (it materializes); H's `columnVersions` carries C once with S's
  origin `siteId` + HLC (`compareHLC === 0`, not re-stamped to any relay); and H
  serves C onward from its own change log (chain doesn't dead-end at the first
  holder).
- **Chain quiescence.** From-zero pulls advance each consumer's per-peer watermark
  to C's HLC; re-pulling each hop at the advanced watermark excludes `RETIRED`
  (the `> sinceHLC` filter stops re-send after one exchange at every depth), and
  each relay still holds exactly one entry (no unbounded growth).
- **Echo exclusion at depth.** `R3.getChangesSince(S)` (from-zero and delta) never
  re-offers C to its author — origin `siteId` is S two relays deep.
- **Back-relay loop-freedom.** A reverse `pull(R3→R2)` re-offers C to a peer that
  already holds it → still exactly one entry on R2 (HLC-keyed idempotent), mirroring
  the depth-2 ping-pong one hop deeper.

## Known gaps / where to push (treat tests as a floor)

- **Metadata layer only.** Like every spec in this file, the harness uses a
  recording `applyToStore` stub and a `known`-set basis oracle — it does **not**
  materialize a real SQL row. H's "materialization" is verified through the sync
  metadata stores (`columnVersions.getColumnVersion`), not a `select`. Real-engine
  materialization of a relayed change is the separate
  `store-and-forward-relay-e2e.spec.ts` (the 4.x e2e ticket). The multi-hop chain
  is **not** exercised end-to-end through a real `Database`/`StoreModule` — if the
  reviewer wants depth ≥3 over the real engine, that's a new e2e spec, not an
  inline fix.
- **Fixed depth 3, not parameterized.** The chain is hard-coded to R1/R2/R3
  ("three or more"; 3 exercises the >2 transition where the R2→R3 step is the
  inductive repeat of R1→R2). There is no property-style sweep over arbitrary N, so
  a regression that somehow broke only at depth ≥4 specifically would slip through.
  Given the per-change inductive identity argument this is low-risk, but it is a
  real coverage edge.
- **Single-change straggler.** The chain carries one column change C. Multi-change
  straggler transactions traversing the full chain are not covered here (the
  file-level `from-zero path` spec covers a 2-change txn at depth 1 only). The
  idempotence/identity argument is per-change, so risk is low, but a reviewer
  wanting transaction-grouping integrity across hops would add a multi-change C.
- **Back-relay is one hop.** The loop-freedom reverse pull is `R3→R2` only; a full
  reverse traversal back to S is not walked (echo exclusion at S is covered
  separately by the existing depth-2 echo spec and the new depth-3 echo spec).
- **Watermark realism.** Quiescence relies on the harness `pull` helper advancing
  `since = max(ChangeSet.hlc)`, mirroring what `sync-client`/`coordinator` do. If
  the reviewer doubts that fidelity, the real transport's watermark advance is the
  thing to cross-check against `packages/quereus-sync-client` / `sync-coordinator`.
