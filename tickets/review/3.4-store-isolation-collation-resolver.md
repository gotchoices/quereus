# Review: store and isolation table layers resolve collations against their database

## What changed and why

A *collation* is a named text-sorting rule. `BINARY`, `NOCASE`, and `RTRIM` ship built in;
an application registers more with `db.registerCollation(name, comparator)`, and may replace
`NOCASE` / `RTRIM` (not `BINARY`).

`StoreTable` (persistent key-value tables) and `IsolatedTable` (the read-your-own-writes
transaction overlay) resolved collation names by calling the three-argument
`compareSqlValues(a, b, name)`. That overload reads a **process-global** registry holding only
the three built-ins with their built-in meanings, and silently falls back to BINARY on a miss.
Both classes already carry `db: Database`, so the fix is to resolve through
`db.getCollationResolver()` — added by `3.2-collation-resolver-seam`, which throws
`no such collation sequence: X` rather than degrading to BINARY.

### Engine (`packages/quereus`)

- **New export `resolveCollationFunctions(resolver, names)`** in `util/comparison.ts`. Maps a
  `(string | undefined)[]` of declared collation names to `CollationFunction[]`, `undefined ⇒
  BINARY`. Both extension packages and the memory vtab now share it instead of each writing the
  same `.map(name => name ? resolver(name) : BINARY_COLLATION)`.
- **`MemoryTable.comparePrimaryKey` was BINARY.** Not in the ticket's file list, but it is the
  comparator `IsolatedTable.getComparePK()` *prefers* when the underlying table exposes it — so
  an isolation overlay over a memory table merged under BINARY while the memory table's own
  layer BTrees sorted under the declared collation. That is exactly the "overlay vs base table
  disagreement" the implement ticket names as an edge case, and it lives here. It now resolves
  PK collations through `db.getCollationResolver()`, memoized on the `TableSchema` object
  identity (an `alter table` installs a fresh frozen schema, invalidating the entry).

### Store (`packages/quereus-store/src/common/store-table.ts`)

- `protected readonly collationResolver` bound once in the constructor. The resolver *closure*
  is cached (it reads the live registry); resolved *functions* are not cached beyond the
  comparator or constraint check that hoisted them.
- **Pushed-constraint re-check.** `compareValues(a, op, b, collation?: string)` became
  `compareValues(a, op, b, collationFunc: CollationFunction)` and calls `compareSqlValuesFast`.
  `matchesFilters(row, filterInfo)` gained a required third parameter,
  `collations: ReadonlyMap<number, CollationFunction>`, built once per scan by the new
  `resolveFilterCollations(filterInfo, indexCollationNames?)`. All four scan arms (PK point, PK
  range, secondary index, full scan) hoist it above their row loop. The index arm feeds
  `indexColumnCollations(access.index)` in as the override, preserving the previous
  index-column-COLLATE-beats-declared-column-collation rule.
- **Three `UNIQUE` conflict paths** (`findUniqueConflictViaIndex`, `findUniqueConflict`,
  `findUniqueConflictViaCoveringMv`) resolve `uniqueEnforcementCollations(...)` once, above the
  candidate loop, and compare with `compareSqlValuesFast`.

### Isolation (`packages/quereus-isolation/src/isolated-table.ts`)

- `private readonly collationResolver` bound once in the constructor.
- New `getPkCollations()` — PK-column comparison functions, memoized on `TableSchema` identity.
  Backs both the fallback `getComparePK()` comparator (used when the underlying table exposes no
  `comparePrimaryKey`, i.e. every store-backed table) and `keysEqual()` (which decides "same
  logical key", so a wrong answer becomes a false PK-relocation and then a false PK conflict).
- `checkUniqueAgainstUnderlying` resolves its enforcement collations once, above the scan.

### Untouched on purpose

The name-less `compareSqlValues(a, b)` calls (`store-table.ts` ~1713/~1719, `isolated-table.ts`
~769, `quereus-sync/src/sync/store-adapter.ts` ~476) are BINARY by intent.

## The DDL restriction that shapes every test

`create table t (v text collate NOSPACE)` is **rejected today** — `validateCollationForType`
(`packages/quereus/src/schema/table.ts`) checks a column's `COLLATE` against a static list on the
logical type (`BINARY`/`NOCASE`/`RTRIM` for TEXT), never against the connection's registry. So a
custom collation reaches these code paths by exactly two routes, and both suites use both:

1. **Overriding a built-in** — `db.registerCollation('NOCASE', noSpace, stripSpaces)`. The
   discriminating probe: a comparison that reached the global registry still gets the built-in
   case-folding comparator.
2. **A custom name on an index column** — `create unique index ix on t (v collate NOSPACE)`.

`backlog/feat-ddl-accepts-registered-collations` tracks the gap. It is pre-existing, not this
ticket's leftover. (The implement ticket's suggested test "custom collation orders a persisted
primary key" is not writable for this reason, and would fail for a second reason — see the
key-encoder defect below.)

## What to exercise

173 (isolation) + the rest, **0 failing**.

### Store — `packages/quereus-store/test/custom-collation.spec.ts` (7 cases)

Run: `node --import ./packages/quereus-store/register.mjs node_modules/mocha/bin/mocha.js "packages/quereus-store/test/custom-collation.spec.ts" --reporter spec`

- `still rejects a case-variant duplicate under the built-in NOCASE` — regression guard. NOCASE
  resolves in *both* registries, so this cannot catch a wrong-registry mistake; it catches one
  that breaks enforcement outright.
- `lets an overriding NOCASE that does not fold case admit a case variant` — registers a
  case-*sensitive* NOCASE; `'ABC'` must insert alongside `'abc'`. Fails pre-fix (the global
  NOCASE raises `UNIQUE`).
- `collides two rows the overriding NOCASE considers equal` — the inverse direction: `'ab'`
  must collide with `'a b'`. Fails pre-fix (no violation).
- `enforces an index-derived UNIQUE under a custom collation named on the index column` —
  `create unique index … (code collate NOSPACE)`; `'ab'` collides with `'a b'`. Fails pre-fix.
- `enforces a PARTIAL index-derived UNIQUE under a custom collation` — same, plus
  `where active = 1`; the out-of-scope row inserts freely.
- `resolves an empty collation-name list without throwing` — the ticket's degenerate-array case.
- `raises rather than falling back to BINARY after a reopen` — db1 registers NOSPACE, creates
  the index, persists; db2 rehydrates the catalog **without** registering it and the first
  insert raises `no such collation sequence: NOSPACE`. Pre-fix the insert silently succeeded.

### Isolation — `packages/quereus-isolation/test/collation-resolver.spec.ts` (8 cases)

Run: `node --import ./packages/quereus-isolation/register.mjs node_modules/mocha/bin/mocha.js "packages/quereus-isolation/test/collation-resolver.spec.ts" --reporter spec`

- `treats a collation-equal PK rewrite as an in-place update, not a relocation` — the headline.
  With NOCASE overridden to ignore spaces, `update t set k = 'ab' where k = 'a b'` is a no-op on
  the key. Pre-fix `keysEqual` used the global NOCASE, called the two keys distinct, took the
  relocation path, and resolved the "new" key back to the same underlying row — a false PK
  conflict.
- `restores the base row on rollback`, `shadows the base row exactly once when only a non-key
  column is staged` — merge/shadowing under the overlay.
- `rejects an insert whose PK collation-equals a committed row` / `accepts an insert whose PK
  the overridden NOCASE keeps distinct` — the two sides of `keysEqual`.
- `enforces an index-derived UNIQUE under a custom collation named on the index column` —
  `checkUniqueAgainstUnderlying` against a committed row, inside a transaction.
- `still rejects a case-variant duplicate under the built-in NOCASE` — regression guard.
- `raises rather than falling back to BINARY` — `collate FROBNICATE` on an INTEGER PK (INTEGER
  declares no supported-collation list, so DDL accepts the name and the comparator build raises).

## Known gaps — please treat as the floor, not the finish line

1. **`IsolatedTable.getComparePK()`'s fallback arm is not directly covered.** It runs only when
   the underlying table exposes no `comparePrimaryKey` — i.e. a store-backed isolated table,
   never a memory-backed one. Writing that test needs `@quereus/store` (which does depend on
   `@quereus/isolation`, so `packages/quereus-store/test/isolated-store.spec.ts` is the right
   home), but a discriminating case there requires a custom PK collation, which trips the
   key-encoder defect below rather than the comparator. So the arm is exercised only indirectly,
   by the store suite's existing isolated-store coverage under built-in collations.

2. **`findUniqueConflictViaCoveringMv` (`store-table.ts` ~2012) has no custom-collation test.**
   Reaching it needs a row-time covering materialized view linked to the UNIQUE, and the
   `findRowTimeCoveringStructure` collation gate declines a finer/incomparable index-derived
   UNIQUE. The resolve was hoisted and the compare switched to `compareSqlValuesFast`; the
   change is mechanical and type-checked, but unproven at runtime under a custom collation.

3. **Both `pkCollationCache`s (memory + isolation) key on `TableSchema` identity, not on the
   collation registry.** A `db.registerCollation` that *replaces* a collation after a comparator
   was built is not picked up. This matches the engine-wide contract documented at
   `Database.registerCollation` ("registration is not retroactive"), and both caches carry a doc
   comment saying so — but it is a deliberate choice worth a second opinion.

4. **`yarn test:store` (the LevelDB sqllogic re-run) was deferred** — it routinely exceeds the
   10-minute agent idle budget. The store paths this ticket touches are covered by the Mocha
   store suite (777 cases, green), but the LevelDB-backed sqllogic pass has not run. Worth a
   human or CI invocation before this is considered fully validated.

5. **`resolveFilterCollations` is called once per `query()` / `scanPKRange()` / `scanIndex()`
   invocation, not memoized across calls.** It walks `filterInfo.constraints` and does one
   registry lookup per distinct constrained column. Fine now — it is O(constraints), dwarfed by
   the scan's I/O. If a point-lookup-heavy profile ever shows it, memoize on the `FilterInfo`.
   Parked as a `NOTE`-free judgement call; no code comment, since the method's doc comment
   already says "ONCE per scan".

## A real defect found and filed, not fixed here

`tickets/fix/bug-store-key-encoder-ignores-database-collations.md`.

The store encodes text keys into sort-preserving bytes through a **separate, process-global
encoder registry** (`packages/quereus-store/src/common/encoding.ts`) holding only `BINARY` /
`NOCASE` / `RTRIM` with their built-in meanings, and falling back to the `NOCASE` encoder
(lowercasing) for any unrecognized name. It never consults the database. So comparison and
storage now disagree:

```ts
db.registerCollation('NOCASE', noSpace, stripSpaces);   // "ignore spaces"
await db.exec(`create table t (k text collate NOCASE primary key, v text) using store`);
await db.exec(`insert into t values ('a b', 'one')`);
await db.exec(`insert into t values ('ab',  'two')`);   // expected PK conflict
```

Observed on the working tree (verified by a throwaway spec): **no error, both rows persist.**
Reachable today via a `NOCASE`/`RTRIM` override; it does not need the DDL restriction lifted.
This is out of this ticket's scope (different file, different registry, and it needs a design
decision about where the store gets a key normalizer from) but it is a live duplicate-primary-key
bug, so it went to `fix/` rather than `backlog/`.

A `NOTE:` at the divergence site (`encodeText` in `encoding.ts`) points at the ticket, and
`docs/sql.md` § COLLATE gained a "Store caveat — physical key bytes" paragraph telling embedders
to keep store primary-key collations to the three built-ins with their built-in meanings until
it lands. Delete that paragraph when the fix ships.

## Review findings

- The pushed-constraint hot path (`matchesFilters`) previously re-resolved a collation name per
  constraint per row via `compareSqlValues`'s internal lookup; it now resolves once per scan.
  Strictly faster, but the `matchesFilters` / `compareValues` signature change is the widest
  blast radius in this diff — both are `protected`, and `HarnessStoreTable` in
  `packages/quereus-store/test/store-ryow.spec.ts` extends `StoreTable` (it does not override
  either method). Worth confirming no out-of-tree subclass is implied by the `protected` marker.
- `MemoryTable.comparePrimaryKey` was changed although it sits outside the ticket's file list.
  The justification is above; if a reviewer disagrees that it belongs here, it should move to its
  own ticket rather than be reverted — the isolation overlay is wrong without it.
