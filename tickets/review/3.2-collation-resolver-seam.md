# Review: one official way to resolve a collation name, per database

## What landed

A *collation* is a named text-sorting rule (`BINARY`, `NOCASE`, `RTRIM`, plus whatever
the embedder registers). Collations live per `Database` instance, but several comparison
paths resolved names against an older process-global registry in `util/comparison.ts` and
silently fell back to byte order (`BINARY`) on a miss. This ticket adds the single
resolution primitive the follow-on tickets will adopt, and closes the silent fallback in
the emission path.

Four pieces:

- **`CollationResolver`** (`types/logical-type.ts`) — `(name: string) => CollationFunction`.
  Exported from `src/index.ts`.
- **`Database.getCollationResolver()`** — lazily-created bound closure, stable identity
  across calls, reads the live per-database registry. Fast-paths the exact name `BINARY`;
  otherwise `_getCollation(name)` (case-insensitive); on a miss it throws
  `QuereusError('no such collation sequence: X', StatusCode.ERROR)` — wording matches SQLite.
  Deliberately no `checkOpen()`: it runs at comparator-construction time and reading a
  closed database's registry is harmless.
- **`getCollationResolver()` on `DatabaseInternal`** — so `quereus-store` and
  `quereus-isolation` can reach it through their existing `this.db as DatabaseInternal` cast.
- **`builtinCollationResolver(name)`** (`util/comparison.ts`) — database-free, `switch`-based,
  returns `undefined` for non-built-ins. Written as a `switch` (not a registry read) so it
  survives the deletion of the deprecated global map in `3.6-retire-global-collation-registry`,
  and so it never observes a caller's override of a built-in name.

`EmissionContext.resolveCollation` now keeps its `BINARY` fast path, still records the
collation dependency via `getCollation()` (plan invalidation intact), and delegates the
miss to `db.getCollationResolver()` so build-time and emit-time resolution report the same
error from the same place.

## Behavior change the reviewer must look at first

**`test/collation-isolation.spec.ts` had one assertion that pinned the defect.** Its
"custom collation registered on one Database is not visible from another" case asserted
that a second database running `ORDER BY name COLLATE REVERSE` (never registered there)
returns rows in **BINARY order**. That is exactly the silent fallback this ticket family
exists to close, so the assertion is now inverted: db2 must reject with
`no such collation sequence: REVERSE`. The implement ticket claimed this suite would stay
green unchanged; it was wrong, and the edit is intentional rather than a test papering
over a regression. Confirm you agree with the direction before anything else.

Consequence for embedders: opening a persisted database whose DDL names a custom collation
now **throws** when that collation was not re-registered on the connection, where it
previously returned byte-ordered rows. That tradeoff is stated in the plan ticket and is
now documented in `docs/sql.md` §4.4 under COLLATE.

## How to exercise it

```ts
const db = new Database();
const resolve = db.getCollationResolver();

resolve('NOCASE');            // → NOCASE_COLLATION
resolve('nocase');            // → same (case-insensitive)
resolve('NOPE');              // throws QuereusError: no such collation sequence: NOPE
resolve('');                  // throws (no silent BINARY)

db.registerCollation('REVERSE', cmp);
resolve('REVERSE');           // → cmp  (resolver was obtained BEFORE registration)
```

Two databases each registering `REVERSE` with different comparators get different
functions from their own resolvers; neither leaks. `db.registerCollation('NOCASE', ...)`
still overrides `NOCASE` for that database (a pre-existing, tested capability);
`BINARY` alone cannot be overridden, because of the fast path — pre-existing
`EmissionContext` behavior, kept and now documented on the resolver.

## Known gaps — treat these as the starting point, not the finish line

- **Nothing adopts the resolver yet.** `EmissionContext` is the only caller. The memory
  vtab, `quereus-store`, and `quereus-isolation` still resolve through the deprecated
  global registry; that is tickets 3.3–3.5. The global registry in `util/comparison.ts`
  is untouched (`compareSqlValues`, `compareWithOrderBy`, `createOrderByComparator`,
  `compareRows` all still route through it), and `3.6` retires it.
- **`yarn test:store` was not run.** Only the memory-backed suite. If a persisted table's
  DDL can carry a custom collation, the store path is where the new throw would first bite
  an embedder — worth a look.
- **Dependency-tracker keys are not normalized.** `EmissionContext.getCollation` records
  the dependency under the raw name (`collation:nocase` vs `collation:NOCASE` are distinct
  keys). Harmless today — `validateCapturedSchemaObjects` re-resolves through
  `_getCollation`, which upper-cases — but it means the same collation can appear twice in
  a plan fingerprint. Pre-existing; left alone rather than widened into this ticket. Not
  filed as a ticket because nothing currently misbehaves.
- **No emitter was found producing an unregistered name.** The ticket warned to watch
  `effectiveCollationOfTypes` (`emit/join.ts`) and the `attr.type.collationName` reads in
  `distinct.ts` / `set-operation.ts` / `async-gather.ts`. Those all guard with
  `collationName ? resolve(...) : BINARY_COLLATION`, and the full suite is green, so no
  producer needed fixing. That is an absence of evidence from the existing tests, not a
  proof — a query shape the suite does not cover could still reach the new throw.
- **Resolver identity vs. result caching.** The resolver is bound once, but results are not
  cached: an index whose comparator was built before a `registerCollation` call keeps the
  old comparator. Pre-existing; recorded as a `NOTE:` tripwire at
  `Database.registerCollation` rather than filed as work.

## Review findings

- Tripwire parked at `packages/quereus/src/core/database.ts`, immediately above
  `registerCollation`: collation registration is not retroactive — comparators resolve once
  at construction time, so re-registering a collation does not rebuild indexes or cached
  plans that already captured the old comparator. Fine today; becomes work only if
  retroactive re-registration is ever wanted.
