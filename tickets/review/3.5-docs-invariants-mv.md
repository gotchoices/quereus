## What landed

`docs/invariants.md` gained a populated `## MV — Materialized views` area: **MV-001 through
MV-024**, matching the format `docs-invariants-optimizer` established (heading `### MV-nnn — …`,
one or more `code:` pointers, exactly one `guard:`, one `doc:`, body ≤ 120 words). No entry uses
`guard: none` — every one names a real test.

The 24 entries, grouped:

- **The core promise** — MV-001 (a materialized view is a faster plain view; observational, immediate,
  not eventual), MV-002 (maintenance rides the writing transaction), MV-005 (MV-over-MV cascade
  completes in the originating transaction).
- **The per-statement contract** — MV-003 (bounded-delta arms apply per row, because the UNIQUE
  enforcement probe reads the backing mid-statement), MV-004 (full-rebuild is the one deferred arm,
  and is never a covering structure — which is *why* deferring it is safe).
- **Coverage and rejection** — MV-006 (no body is rejected for its shape; exactly five non-shape
  create-time rejections), MV-007 (the strategy gate can be slow, never wrong), MV-008 (the
  replicable-derivation gate is host-conditional and `pragma nondeterministic_schema` does not lift it).
- **Identity and storage** — MV-009 (exactly one schema object), MV-010 (the maintained relation is
  always a set), MV-011 (the coarsened-key carve-out, derived once and read by both sides),
  MV-012 (read-only to user DML), MV-013 (`bodyHash` is the identity of the definition).
- **The module boundary** — MV-014 (every privileged operation routes through the backing host),
  MV-015 (a host owes ordered keyed storage), MV-016 (`applyMaintenance` reports exactly what it
  realized — so a value-identical write reports nothing), MV-024 (a durable backing is refilled
  unless provably adoptable).
- **Constraints and ingestion** — MV-017 (declared CHECK / child-FK validated on derived rows before
  the cascade), MV-018 (the validator tracks its constraint-only dependencies), MV-019 (secondary
  UNIQUE enforced by the host, post-batch), MV-020 (parent-side referential actions), MV-021 (the
  ingestion seam trusts its origin).
- **Schema change** — MV-022 (a stale view serves its snapshot and propagates nothing), MV-023 (a
  rename rewrites the stored body).

Back-links (`> **Invariant:** [MV-nnn](invariants.md#mv-nnn--full-slug)`) were added at 24 sections
across the six MV docs, using the full-heading-slug convention `docs-invariants-optimizer` pinned.
`yarn docs:check` resolves every one of them.

## What to verify

`yarn docs:check` is green and `yarn lint` is green. The checker validates *pointers*, not
semantics — so the review's job is the semantics. Suggested reading order:

1. Read `docs/invariants.md` § MV end-to-end against the code, which is the whole point of the
   register. It should take one sitting.
2. Spot-check the sharpest claims, which are the ones most likely to be subtly wrong:
   - **MV-003 / MV-004 together.** The claim is that per-row apply is load-bearing for enforcement
     visibility, *and* that deferring the full-rebuild floor does not violate it because a
     full-rebuild backing is never read mid-statement. If a future arm becomes both deferred and
     a covering structure, both entries are false at once. Confirm `lookupCoveringConflicts` really
     reads only `'inverse-projection'` backings.
   - **MV-007.** "The gate can only be slow, never wrong" rests on *every* strategy in the sound set
     being correct and an empty sound set resolving to the floor. Verify `selectMaintenanceStrategy`
     has no path that returns a strategy the arm builder declined.
   - **MV-011.** The soundness argument is that `deriveCoarsenedBackingKey` is called over the same
     fully-optimized body in both `buildFullRebuildPlan` and `deriveBackingShape`, so the two agree
     "by construction". That is an argument about call sites, not a proof — worth confirming.
3. Confirm the doc-drift correction below is the right call in the right direction.

## The docs and the code disagreed — and the docs were wrong

**The count of create-time rejections.** `docs/materialized-views.md` (twice),
`docs/mv-maintenance.md`, and the `buildMaintenancePlan` doc-comment all asserted **four**
non-shape create-time rejections. The code has **five**. The missing one is at
`packages/quereus/src/core/database-materialized-views-plan-builders.ts:844`:

```ts
if (sourceBases.length === 0) throw cannotMaterialize(mv.name, 'its body reads no source table');
```

It is reachable and distinct from the "no relational output" reject: `create materialized view v as
select 42` produces relational output and has a key, passes determinism, reaches the floor, and
throws here. It is **guarded** — `test/logic/53.3-materialized-view-constraint-only-ddl.sqllogic:358`
and `:377` both assert the `reads no source table` error.

**My call: the code is right, the docs had drifted.** The reject is sound (nothing could ever dirty a
sourceless plan, so nothing could maintain it), it is tested, and it predates this ticket. I corrected
all four prose sites rather than filing a ticket, since `docs/doc-conventions.md` says the register is
normative and the topic doc is the one to fix. **This is the finding a reviewer should second-guess
first**: if you think the sourceless reject is itself the bug (i.e. a sourceless body should be
accepted as a constant relation and never maintained), then MV-006 as written is wrong and the fix
belongs in `tickets/fix/`, not in the prose. I do not think so, but I am recording the alternative
because MV-006 explicitly says "adding a sixth is a breaking change" — the count is now load-bearing.

Note also that `mv-maintenance.md` already described a "fifth, host-conditional" replicable gate; that
paragraph now reads "a sixth", and the two host-conditional gates (non-replicable function,
non-replicable collation) remain outside the count, as MV-008 states.

## What I could not confirm against the code

Listed honestly, worst first:

- **MV-024 (adopt-without-refill) is the weakest entry.** The five-part adopt gate — body hash, backing
  shape, module identity, same-module sourcing, and a crash-durable trust basis — lives in
  `rehydrateCatalog` in `packages/quereus-store`, which I did **not** open. The entry's `code:` points
  at the engine-side `backingShapeMatches`, which is only gate 2. Everything about the clean-shutdown
  marker and the durable stale-MV set is taken from `mv-backing-host.md` prose on faith. A reviewer
  should either verify it in the store package and add a second `code:` pointer, or downgrade the
  entry to cover only the "refill by default" half, which I did verify.
- **MV-015 is an obligation the engine does not enforce.** The cost contract (O(log n) keyed ops plus
  an ordered prefix scan) is a normative statement in `vtab/backing-host.ts` about what a *module*
  must provide. There is no runtime check; a module that advertises the capability without the
  ordered prefix scan fails silently and slowly. The guard cites the memory reference implementation,
  which trivially satisfies it. This is a real invariant but its guard is weak, and I could not make
  it stronger without writing a conformance harness, which was out of scope.
- **MV-019 cites only the memory host.** `mv-constraints.md` says the store host implements the same
  post-batch check as `StoreTable.enforceSecondaryUniqueForMaintenance`. I verified the memory symbol
  (`vtab/memory/layer/manager.ts` — `enforceSecondaryUniqueOnMaintenance`) and did **not** open the
  store one. The guard is a `.sqllogic` file, which runs against memory by default; `yarn test:store`
  would exercise the store path but I did not run it.
- **MV-001's `code:` pointers are indicative, not exhaustive.** The invariant is a property of the
  whole subsystem; I pointed at `maintainRowTime` and `materializeView` because they are where the
  promise is established and kept. No single symbol implements "MV ≡ faster view".

## Things I deliberately did not do

- **No `debt-` backlog ticket was filed.** The ticket predicted the MV-vs-plain-view differential
  harness might not exist and nominated it as the one allowed debt ticket. **It exists**:
  `packages/quereus/test/incremental/maintenance-equivalence.spec.ts` is exactly that oracle — it
  reads the MV backing and compares against the defining `SELECT` evaluated live against the sources,
  across randomized mutation batches, in-transaction and after rollback, with a negative self-test
  (`assertEquivalent throws when the oracle body does not match the backing`) proving the oracle can
  go red. It is MV-001's guard and the strongest guard in the register. Nothing else in the MV area
  needed `guard: none`, so there was no second candidate.
- **I did not strip restating prose from the topic docs.** `doc-conventions.md` says a back-linked
  section "does not restate the invariant". Honouring that literally would mean deleting large
  passages of `mv-maintenance.md` and `mv-constraints.md`, which is a much larger blast radius than a
  register-population ticket should have. The back-links are in place; the de-duplication is a
  separate call. **Flagging this as a reviewer decision, not an oversight.**
- **No `OPT-*` cross-reference by ID.** The MV area preamble says the read-side rewrite and the
  coverage prover are optimizer concerns, but there is **no OPT invariant covering either** — the OPT
  area stops at FD machinery. I linked to `docs/optimizer-rules.md` by section instead of by ID. If
  the reviewer wants the ID form the ticket asked for, the OPT area needs entries first.
- **`LENS-*` stays empty**, as instructed. Covering structures are documented in `mv-constraints.md`
  and appear in the register only where they constrain MV behaviour (MV-003, MV-004, MV-019).

## Tripwires parked

- `scripts/check-docs.mjs` — extended the existing `NOTE` on `parseInvariantBlocks`. A block's body
  runs to the next `### ` heading, so an area's `## <Area>` heading **and any preamble prose under it**
  are charged to the *previous* invariant's 120-word budget. This tripped for real: my first MV
  preamble pushed `OPT-058` to 133 words and failed the build. The note now records that an area
  preamble must stay short (~30 words of headroom), and what to change if preambles ever need length.
