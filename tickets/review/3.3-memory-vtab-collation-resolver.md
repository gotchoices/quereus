# Review: memory vtab resolves collations against its own database

## What a collation is, and what was wrong

A *collation* is a named text-sorting rule. `BINARY`, `NOCASE`, and `RTRIM` are built in;
an application can register more with `db.registerCollation(name, comparator)`, and can
replace `NOCASE`/`RTRIM` with its own comparator (but not `BINARY`).

Everything under `src/vtab/memory/` used to turn a collation *name* into a comparison
function by calling the process-global `resolveCollation()` in `util/comparison.ts`, or by
passing a name into the three-argument `compareSqlValues(a, b, name)` (same lookup,
internally). That global registry only ever holds the three built-ins, and it silently
fell back to `BINARY` on a miss. So a memory table's primary key, secondary indexes, index
range seeks, plan-filter bound checks, and `UNIQUE` enforcement all ignored whatever the
connection had registered, and byte-ordered instead.

`3.2-collation-resolver-seam` added `db.getCollationResolver()` — a `(name) => comparator`
that reads the live per-database registry and *throws* `no such collation sequence: X` on a
miss. This ticket threads it through the memory vtab.

## What shipped

**The resolver rides on the objects that already reach every comparison site.** `ScanPlan`
was left alone (it still carries collation *names*; it is logged and compared as plain data
in tests).

- `Layer` gained `readonly collationResolver: CollationResolver`. `BaseLayer` takes it as a
  constructor argument; `TransactionLayer` copies it from its parent — which is what keeps a
  child layer's `compareKeys` built from the same function that ordered the parent's
  secondary BTree nodes it inherits.
- `MemoryIndex` takes a `collationResolver` as its **third** constructor argument (before
  `primaryKeyComparator`).
- `createPrimaryKeyFunctions(schema, collationResolver)` takes it as a second argument.
  Deliberately **not** defaulted — a default would re-open the hole.
- `MemoryTableManager` binds `db.getCollationResolver()` once in its constructor and hands
  it to `BaseLayer` and `createPrimaryKeyFunctions`. (The original ticket asked for a
  constructor parameter supplied by `MemoryTable`; the manager is actually constructed in
  `module.ts` and already holds `db`, so taking it from `this.db` is the same thing with
  less plumbing.)
- `scanLayer(layer, plan)` resolves the plan's collation names **once**, then delegates to a
  private `scanLayerResolved(layer, plan, collations)` that passes the resolved functions
  down its own recursion (multi-seek → single key, multi-range → single range). Neither
  narrowing changes which columns the collations describe, so nothing re-resolves.
- `plan-filter.ts` exports `resolveScanCollations(plan, resolver)` → `ResolvedScanCollations`
  (`{ equalityPrefix: CollationFunction[], bound: CollationFunction }`), and
  `planAppliesToKey` now takes that as a fourth argument. All the per-row
  `compareSqlValues(a, b, name)` calls in `plan-filter.ts` and `scan-layer.ts` became
  `compareSqlValuesFast(a, b, fn)`.

**Beyond the ticket's TODO list**, three `compareSqlValues(a, b, name)` calls in
`manager.ts`'s `UNIQUE` enforcement (`checkUniqueViaIndex` ~1223, `checkUniqueViaMaterializedView`
~1291, `checkUniqueByScanning` ~1340) were also converted. They had the same defect: a
`UNIQUE` constraint on a column whose collation the connection overrode was enforced by the
global comparator. Each resolves its per-column collations once, above the candidate loop.
`checkUniqueViaIndex` gained an early `return null` when the index yields no candidate PKs —
that is the overwhelmingly common insert, and it must not pay for the resolves.

`compareSqlValues(a, b)` with no collation argument (BINARY) is untouched — `manager.ts` ~342/
~1006/~1018, `table.ts` ~429, `utils/predicate.ts`. `src/vtab/` now imports nothing from the
deprecated global registry (`grep -rn "resolveCollation\|registerCollation" packages/quereus/src/vtab/`
is empty), which is what `3.6-retire-global-collation-registry` needs.

## The ticket's headline test could not be written as specified — read this first

The ticket asked for `create table t (k text collate REVERSE primary key)`. **That DDL is
rejected today**, and not by anything this ticket touches:

```
Unknown collation 'REVERSE' for type 'TEXT' on column 'k' (expected one of: BINARY, NOCASE, RTRIM)
```

`validateCollationForType` (`schema/table.ts` ~194) checks a column's `COLLATE` against a
**static list on the logical type** (`TEXT_TYPE.supportedCollations`), never against the
connection's registry. So a custom-named collation can never be declared on a TEXT column.
Two things do work, and the new suite uses both:

- **Overriding a built-in.** `db.registerCollation('NOCASE', myComparator)` is allowed, and
  `collate nocase` passes DDL. This reaches the primary-key comparator.
- **A custom name on an *index* column.** `IndexColumnSchema.collation` is not gated by
  `supportedCollations`, so `create index ix on t (v collate REVERSE)` is accepted and
  reaches `MemoryIndex`.

I filed `backlog/feat-ddl-accepts-registered-collations.md` for the gap (the gate is
simultaneously too strict for TEXT and too loose for INTEGER/REAL/BLOB, which accept any
name). **Do not treat that as this ticket's leftover** — it is a pre-existing schema-layer
limitation, and closing it would change `102.1-unique-edge-cases.sqllogic`'s pinned wording.

## Behavior changes an embedder can observe

1. **A replaced `NOCASE`/`RTRIM` now governs a memory table's key order and `UNIQUE`
   collisions.** Previously the query layer honored the override while the storage layer did
   not — the two disagreed.
2. **An unregistered collation now raises at table/index construction** instead of silently
   byte-ordering. Concretely: `create table t (k integer collate frobnicate primary key)`
   used to succeed (INTEGER declares no supported-collation list, so DDL waves the name
   through; the old `resolveCollation` warned and returned BINARY). It now fails with
   `Module 'memory' create failed for table 't': no such collation sequence: FROBNICATE`.
   For a **TEXT** column the DDL-level `Unknown collation '…' for type 'TEXT'` still fires
   first — verified, and `102.1-unique-edge-cases.sqllogic` / `41.7-alter-column-collate.sqllogic`
   still pass.

## How to exercise it

`packages/quereus/test/vtab/memory-collation-per-database.spec.ts` (11 tests, all new):

| What it pins | Pre-fix behavior |
|---|---|
| PK order under a per-database `NOCASE` override → `c, b, a` | `a, b, c` |
| A database that did *not* override still gets `a, b, c` | same (guard) |
| Index scan over `collate REVERSE` → `c, b, a` | `a, b, c` |
| Range seek `v > 'b'` under `REVERSE` → `a` | `c` |
| Transaction layer over an inherited `REVERSE` index stays ordered | — |
| A collation function is never called with `null` | — |
| Two databases, `REVERSE` registered with opposite comparators, each sorts its own way | both `a, b, c` |
| `UNIQUE` still rejects a case-variant duplicate under built-in `NOCASE` | same (guard) |
| `UNIQUE` rejects under an *overridden* `NOCASE` | same (guard) |
| Unregistered collation raises `no such collation sequence` | table created, byte-ordered |
| TEXT column reports the DDL `Unknown collation` error first | same (guard) |

The range-seek and index-order tests drive `manager.scanLayer(layer, plan)` directly. That
is deliberate: **the planner will not choose an index whose collation differs from the
query's**, so there is no SQL that reaches those code paths for a `REVERSE` index. If you
think there is, that would be a genuine finding.

Direct-construction tests (`memory-index-pk-value-identity.spec.ts`,
`performance-sentinels.spec.ts`, `memory-vtable.spec.ts`) got a resolver rather than a
default: `test/util/builtin-collation-resolver.ts` exports `testBuiltinCollationResolver`,
a built-ins-only resolver that throws on an unknown name.

## Known gaps / where to look hardest

- **`scanLayer`'s multi-seek branch builds `createPrimaryKeyFunctions(seekSchema, …)` per
  scan** for its dedup encoder, right next to `layer.getPkExtractorsAndComparators(seekSchema)`
  which builds the same thing again. That duplication predates this ticket; I threaded the
  resolver through both rather than deduplicating them. Worth a look.
- **`BaseLayer.createSecondaryIndexes` catches and logs `MemoryIndex` construction errors**
  (~155). An unknown collation now throws there. I convinced myself it is unreachable — the
  index could not have been created in the first place with an unregistered collation, since
  `addIndexToBase` does *not* swallow — but that reasoning depends on no path ever removing a
  collation from the registry, and the registry has no `unregisterCollation`. If you find one,
  the swallow turns a bad collation into a missing index and then `Secondary index 'x' not found`
  at scan time.
- **`uniqueEnforcementCollations(schema, uc)` returns `(string | undefined)[]`.** I map
  `undefined → 'BINARY'`. Check that matches what `quereus-store` / `quereus-isolation` do
  with the same helper; `test/unique-enforcement-collation.spec.ts` pins their agreement and
  still passes, but it may not cover the `undefined` slot.
- **`checkUniqueViaIndex`'s `schema.columns[col].collation ?? 'BINARY'`** — `ColumnSchema.collation`
  is typed optional but `columnDefToSchema` always fills it. The `?? 'BINARY'` is defensive; if
  it can actually be `undefined`, the old code passed `undefined` to `compareSqlValues`, which
  also defaulted to BINARY. Same result, but confirm.
- **No test covers a `RTRIM` override**, only `NOCASE`. And no test covers a composite
  primary key whose *second* column carries an overridden collation.
- **Tripwire recorded** (not a ticket): `resolveScanCollations` runs once per `scanLayer` call.
  Noted as a `NOTE:` comment at its definition in `plan-filter.ts` — if a workload of very many
  tiny scans ever makes per-scan setup visible, memoize on the `ScanPlan` via a `WeakMap`.
