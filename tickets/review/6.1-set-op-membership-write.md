### Architecture (read this first ‚Äî it diverges from the ticket's literal model)

The ticket described two complementary mechanisms (per-row conditional partition *and*
recursion into `propagate`). The shipped design **combines** them and adds the load-bearing
piece the ticket under-specified ‚Äî a plan-level **capture**:

- **Halloween-safety forced the capture.** The DML executor drains its source **lazily**
  (`runtime/emit/dml-executor.ts`: `for await ‚Ä¶ of rows`), so a branch op that referenced
  the view `U` directly (which reads the very branches being written) would read a table
  mid-write. So every probe-driven write materializes the affected rows ‚Äî their data
  columns **and** their membership-probe flags ‚Äî **once, up front**, into the same
  context-backed `__vmupd_keys` relation the multi-source path uses, via the existing
  `ViewMutationNode.identityCapture` side input. **No runtime changes were needed** ‚Äî the
  capture + void/drain path already exists (the ticket's listed `runtime/emit/view-mutation.ts`
  was *not* touched; confirm you agree that path is sufficient).
- **Recursion per branch is preserved.** Each branch op is lowered to an AST `BaseOp`
  against a **synthetic branch view-like** (`__setop_left` / `__setop_right`, built from the
  AST legs `leftCore` / `compound.select`) and run back through `propagate`, so the branch's
  own single-source spine handles its œÉ predicate, renames, base routing, and
  no-default/computed rejections. Branch correlation is a NULL-safe **full-data-tuple**
  `exists (‚Ä¶ from __vmupd_keys k ‚Ä¶)`; branch columns are qualified with the synthetic
  view-name so `propagate`'s `__vm_self` self-alias correlation binds them.
- **Routing.** `buildViewMutation` intercepts a `isSetOpMembershipBody` AST (insert/update/
  delete) ‚Üí `buildSetOpMutation` ‚Üí `buildSetOpWrite` (new `planner/mutation/set-op.ts`).
  `propagate` keeps an inlined defensive guard for a nested/recursive set-op body (‚Üí
  rejects pointing at `set-op-membership-nested`). `update-lineage.resolveBaseSite` now makes
  a `set-op-branch` existence site writable-through-effect (merged with the join-side case).
  `schema.ts` reports a membership body's columns `is_updatable='YES'` / null base and the
  view insertable/updatable/deletable.

### Per-operator semantics (all verified)

Uniform across operators because the probe flags already encode each operator's branch
truth (`except` ‚áí inLeft=true/inRight=false; `intersect` ‚áí all true; `union[all]` ‚áí actual):

- **flip true** inserts the captured rows **absent** from the branch (`where not k.<flag>`)
  ‚Äî a redundant `true` is a clean no-op; `except`'s always-false right flag inserts all,
  `intersect`'s always-true flags insert none.
- **flip false** deletes the matching branch row (full-tuple `exists`).
- **data write** (`set x=v`) and **delete** fan out to every member branch (the full-tuple
  correlation restricts each branch op to its resident rows ‚Äî no flag gate needed).
- **insert-through** routes by the supplied uniform-literal flags over a VALUES source.

## Use cases to validate (acceptance gate)

All live in `test/property.spec.ts` ‚Üí `describe('Set-operation membership writes')` (16
tests) plus the updated `Set-operation membership columns` read-half block (lineage /
column_info / view_info flipped from read-only to writable). Fixture:
`U = select id, x from A union exists left as inA, exists right as inB select id, x from B`,
seeded `(1,10)`‚ààA-only, `(2,20)`/`(3,30)`‚ààboth, `(4,40)`‚ààB-only.

- **inB false‚Üítrue** inserts into B (PutGet: inB now reads true); idempotent on a row
  already in B; **rejects** when B has an undefaulted NOT NULL column (`no-default`).
- **inB true‚Üífalse** deletes the B row, A untouched, row stays visible (still inA); **both
  false** removes the row from the view.
- **except**: `set inR=true` inserts into B and the row vanishes from the view; `set
  inL=false` deletes from A. **intersect**: `set inR=false` deletes from B and drops the row.
- **composition**: `set x=5, inB=true` inserts B with x=5 and aligns A; `set x=5, inB=false`
  **rejects** (contradiction).
- **data fan-out / delete fan-out** verified against rows in A-only, B-only, and both.
- **insert-through** `(‚Ä¶, true, false)`‚ÜíA only, `(‚Ä¶, true, true)`‚Üíboth; a **flag-less**
  ambiguous insert rejects.
- **GetPut**: writing the read-back membership value (true‚Üítrue / false‚Üífalse) is a base
  no-op.
- **static-vs-dynamic agreement**: `column_info` YES columns are all actually writable.

## Known gaps / where to probe hardest (tests are a floor)

The implementation is correct on everything tested, but these are **deliberate v1 boundaries
or thin spots** the reviewer should stress ‚Äî several are over/under-claims rather than
crashes:

1. **Static surface over-claims on non-writable branches.** `column_info` / `view_info`
   report a membership body fully writable from the **AST shape alone** (membership present),
   without verifying each branch is writable. A branch that is a VALUES/DML operand or has a
   computed/`select *` leg is rejected at *write* time (`analyzeSetOpView` /
   `branchColumnNames`) but the static surface still says YES ‚Äî a static/dynamic mismatch.
   The cross-check test only exercises writable fixtures. Consider gating the static surface
   on a lightweight branch-writability probe (or documenting the over-claim more loudly).
2. **`union all` duplicate-tuple imprecision.** Identification is by full data tuple, so a
   delete / data-write on a `union all` view with **duplicate data tuples** in a branch
   touches *all* copies. Untested; documented. (The count variant is out of scope.)
3. **Data-fan-out value with column refs** requires the operand legs to use **matching
   column names** ‚Äî a leg rename of a *referenced* column is not remapped (the value is
   passed to each branch verbatim). Only literal values are tested. Probe `set x = x+1` and
   a renamed-leg view.
4. **Branch op ordering assumes independent base tables.** Ops are emitted data-fan-out-then-
   flips, left-then-right ‚Äî no FK topological sort across branches. A set-op whose two
   branches share a base table or carry an FK between them is untested; the immutable capture
   makes ordering correctness-neutral for *identity*, but FK enforcement order is not
   considered.
5. **Capture filter scope has no parent** (mirrors `createSetOperationScope`). A user WHERE
   that resolves only against `ctx.scope` (not a view column) is untested; the tested WHEREs
   are literal/`id=K`.
6. **Conservative contradiction reject.** *Any* data assignment + *any* `false` flip rejects
   (`conflicting-assignment`), even a data column that arguably wouldn't conflict. Matches the
   ticket but is broader than strictly necessary.
7. **Insert-through requires all data columns supplied** (no partial/defaulted insert-through
   through the set-op view), and a VALUES source only (SELECT/DML source rejected).
8. **RETURNING** through a set-op membership write is rejected (not recoverable).
9. **Capture alias is `k`** ‚Äî a data column literally named `k` is an untested edge.
10. **`resolveBaseSite` merge.** The `set-op-branch` and `join-side` existence cases now share
    one writable-through-effect return. Confirm the join-side path is unaffected (full suite
    is green, but eyeball the merge in `update-lineage.ts`).
