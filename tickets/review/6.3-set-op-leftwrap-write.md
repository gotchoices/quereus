description: REVIEW — write-side support for the parenthesized LEFT compound operand of a set-op membership view (the *parallel-sibling* shape `(A∪B) union[…] (C∪D)`). The write path now unwraps the parser's `select * from (compound)` left wrapper, making the wrapped LEFT operand a first-class subtree operand: data UPDATE / DELETE / `set <subtreeFlag> = false` fan out through the LEFT subtree's leaves exactly as they already do through the (direct) right subtree. Closes the 6.0 backlog gap (`set-op-membership-nested` left write branch over a `select *` wrapper). The "sum" model; the "product" model (reused flag names) stays the blocked `set-op-product-coordinate-model`.
files: packages/quereus/src/planner/mutation/set-op.ts, packages/quereus/test/property.spec.ts, docs/view-updateability.md
----

## What landed

Before this change, a write through a parallel-sibling membership view rejected:

```
delete from P1 where id = 2
  → cannot write through view 'P1': the left branch uses 'select *';
    list its columns explicitly so each maps to a writable branch column
```

The mutation path (`set-op.ts`) works off the **view AST**. `leftBranchSelect(sel)` returned the
parser's wrapper `select * from (<inner compound>) as values_N`; `branchColumnNames` →
`tryBranchColumnNames` returned `null` for the `select *` leg → reject. So the LEFT subtree of
every parallel-sibling shape was unaddressable by the write path, even though the right subtree
(a direct compound) already fanned out. Read/plan support for this shape landed in 6.2
(`set-op-leftwrap-arity`); this ticket is the write-side counterpart, reusing 6.2's shared
`unwrapPassthroughSubquery` predicate so neither path drifts on what a pure wrapper is.

### Mechanism (all in `set-op.ts`)

- **`unwrapBranchSelect(branchSelect)`** — new private helper: returns the inner compound when
  `branchSelect` is a pure `select * from (<compound>) as values_N` passthrough wrapper (via
  `unwrapPassthroughSubquery`, the same predicate `select-compound.ts` uses), else the operand
  unchanged. A no-op on a direct operand.
- **`buildBranch`** now unwraps the operand first (`effectiveSelect = unwrapBranchSelect(branchSelect)`),
  and derives `dataColNames`, `isNested`, and `branchView.selectAst` from the inner compound. This
  single unwrap is what makes the wrapped left operand a subtree operand — the existing recursion
  (`fanBranchDataUpdate` / `fanBranchDelete` → `analyzeSetOpBranches`) then fans through it to the
  leaves with **no further change**, because `analyzeSetOpBranches` reads `branch.view.selectAst`
  which is already the unwrapped inner.
- **`leftmostLeafSelect` + `branchColumnNames`** — a subtle but load-bearing addition. For a
  nested compound branch, the data-column **names** are its left-most leaf's projection (a set op
  preserves its left child's column ids at every depth). The right-spine path got away with a
  single `branchSelect.columns` read because a right-nested operand's left leg is a *direct* leaf.
  A **LEFT-spine** nest's own left leg is *itself wrapped* (depth-3: `((A∪B)∪(C∪D)) union[…] (E∪F)`),
  so a single read would see the wrapper's `*`. `branchColumnNames` now descends to the real
  left-most leaf (`leftmostLeafSelect`, unwrapping each layer) before reading names. **This is the
  one place the reviewer should look hardest** — it was found by the depth-3 test failing, not by
  the design.
- **Static surfaces walk BOTH operands** (the AST helpers `schema.ts` calls, so no `schema.ts`
  change was needed):
  - `setOpHasSubtreeOperand` now checks the unwrapped left too → `is_insertable_into = NO` for a
    parallel-sibling view.
  - `surfacedInnerFlagNames` now collects the unwrapped left subtree's inner flags (in
    `[L flags] ++ [R flags]` layout order) → those columns report `is_updatable = NO` and a write
    to one rejects with the clean `set-op-membership-nested` diagnostic (not `unknown-view-column`).
  - `isOperandWritable` unwraps the operand before classifying, mirroring `buildBranch`; the
    except/intersect boundary-flag gating threads through unchanged.
- Module docstring + `docs/view-updateability.md` updated (new "Parenthesized LEFT subtree operand"
  subsection; the read/plan section's close-out line; the static-surface paragraph).

### Why no `schema.ts` change

`deriveViewInfo` / `column_info` are driven entirely by `isSetOpMembershipBody`,
`isSetOpBranchWritable`, `setOpHasSubtreeOperand`, and `surfacedInnerFlagNames`. Fixing those
helpers cascades. The dynamic write's `analysis.surfacedInnerFlagNames` is derived **positionally
from the plan** (correct post-6.2 already), so the surfaced-left-inner-flag reject worked for the
dynamic path even before the AST-helper fix — the AST-helper fix is what makes the *static*
`column_info` surface agree.

## Use cases for testing / validation

New describe block `Parenthesized LEFT-compound operand writes` in `property.spec.ts` — **14 tests,
all green**. Base data: `A∪B = {1,2,3}`, `C∪D = {3,4,5}`, ids 1=A, 2=A&B, 3=B&C, 4=C&D, 5=D.

- **P1 (flag-less inner siblings, outer flags `inL`/`inR`)** — the exact shape that used to reject:
  - data fan-out PutGet: `update P1 set x = 99 where id = 2` reaches A and B (both left leaves);
    `id = 3` fans to both subtrees; the view re-reads the new values.
  - data value referencing a column: `set x = x + 1 where id = 2` clones/resolves per leaf.
  - delete fan-out: `delete ... where id = 2` removes from A and B; `id = 3` spans both subtrees.
  - `set inL = false where id = 3` drops from the LEFT subtree leaves **only** (B), the right
    subtree (C) keeps it → row stays visible via the right (proves the flip targets one subtree).
  - `set inL = true` rejects (`set-op-membership-nested`); contradiction guard `set x, inL = false`.
- **P2 (distinct-name flagged inner siblings)** — surfaces `inA,inB` (left) + `inC,inD` (right):
  - `set inA = true` (a surfaced LEFT-subtree inner flag) rejects cleanly, NOT `unknown-view-column`.
  - data + `set inLsub = false` fan-out works alongside the surfaced flags.
  - **static-surface agreement**: `view_info` = updatable/deletable YES, insertable NO; `column_info`
    = data + own subtree flags (`inLsub`,`inRsub`) YES, surfaced inner (`inA..inD`) NO; cross-checked
    against the dynamic writes.
- **Gated except/intersect LEFT subtree**: a non-member visible via the *right* subtree is NOT
  touched in the left (the gate AND-s `k.inLsub` into each leaf member-exists); genuine members fan.
- **Flag-less non-union LEFT subtree deferred**: `(A except B) union exists right as inRsub (C∪D)`
  rejects `set-op-membership-nested-except`; static surfaces all-NO.
- **Depth-3 LEFT nest** `((A∪B)∪(C∪D)) union[inL,inR] (E∪F)`: data + delete reach the deepest
  left-most leaf A (the test that surfaced the `leftmostLeafSelect` requirement).
- **Key Soundness under write**: the parallel-sibling view stays keyed on data columns [0,1]; no
  flag column (idx ≥ 2) appears in any claimed key.

## Honest gaps / where to probe (tests are a floor, not a ceiling)

- **`leftmostLeafSelect` is the highest-risk addition.** It assumes the left-most leaf's projected
  names positionally align to the outer data columns at every depth — true under the data-only
  arity invariant the `SetOperationNode` constructor enforces, and exercised by the depth-3 test,
  but I did NOT test a **renamed leaf column inside a non-leftmost-depth left leg**, nor a left
  subtree whose leaves use **divergent column names** across legs (the pre-existing v1 "matching-leg-names"
  caveat applies, now on the left too — a data-fan-out *value* that references a renamed column is
  not remapped). Worth an adversarial case.
- **Insert-through routing whose active flag is the LEFT subtree side** is rejected structurally
  (`buildInsertThrough` sees `branch.isNested`), but I added no explicit `insert into P1 (...) values
  (..., inL=true, ...)` test for the left — only `set inL = true`. Same code path, untested spelling.
- **`union all` LEFT subtree (bag)** — I tested `union` on the left; `union all` is union-like and
  should behave identically, but I did not pin it specifically on the left side.
- **Mixed-operator siblings** — I covered left-except/right-union and left-intersect/right-union;
  did NOT cover left-union/right-except in one view (right-except is covered by the existing
  right-nested suite, but not *combined* with a left subtree in the same statement's fan).
- **Halloween at depth on the left** is exercised implicitly by the fan-out PutGet tests and the
  `set x = x + 1` test (single up-front capture reused, no second capture), but not with a
  pathological self-referential value designed to expose a re-scan — the mechanism is identical to
  the proven right-nested path.
- **Own subtree-flag static optimism (inherited).** `column_info` reports an own subtree flag
  (`inLsub`/`inRsub`) `is_updatable = YES` — accurate for its `= false` delete, optimistic for the
  deferred `= true` insert (which the dynamic write still rejects cleanly). Unchanged honesty
  posture from `nestable-flagged-set-ops`, now applying to the left wrapper's flag too; not
  regressed here, flagged for completeness.

## Out of scope (deliberate, not defects)

- **Product model** — two siblings *reusing* flag names (shared coordinate columns), `set
  <subtreeFlag> = true`, surfaced-inner-flag *writes*, insert-through into a subtree side: all stay
  deferred to `set-op-membership-nested` / blocked `set-op-product-coordinate-model`. Reused-name
  parallel siblings are rejected at create (duplicate flags collide in the set-op output scope).
- **Flag-less non-union boundary** deferral (left or right) — documented policy, not a bug.
