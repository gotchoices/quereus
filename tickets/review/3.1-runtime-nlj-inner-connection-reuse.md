## What was built

A nested-loop join whose inner (right) side is **not** wrapped in a cache node
re-scans the inner relation once per outer row (`runtime/emit/join.ts`
`driveFromLeft`). Each re-scan re-invokes the inner scan leaf (`emitSeqScan`,
`runtime/emit/scan.ts`), which previously did `module.connect(...)` at generator
start and `disconnectVTable(...)` in its `finally` — **one connect + disconnect
per outer row**.

Now the scan leaf connects the inner virtual-table instance **once per scan-site
per execution**, reuses it across every re-scan, and disconnects once at
statement teardown.

### Mechanism

- **`runtime/types.ts`** — new optional field
  `RuntimeContext.scanConnections?: Map<symbol, VirtualTable>`: a per-execution
  cache of connected inner-scan instances.
- **`runtime/emit/scan.ts`** — each `emitSeqScan` mints a stable per-node symbol
  (`scanConnectionKey`) in its emitter closure. In `run`: cache hit → reuse and
  skip connect; miss → connect once and store. The per-invocation disconnect was
  removed from the scan's `finally` (the row slot still closes each pass). Keying
  by scan-node identity (not table name) means a self-join's two scan sites over
  one table get **distinct** instances and never share a cursor.
  - The query-method existence check moved *inside* the try/finally, so on the
    fallback (no-cache) path a query-less module's error still disconnects the
    freshly-connected instance (the original leaked it there).
- **`core/statement.ts`** — `_iterateRowsRawInternal` creates the map, passes it
  on the `RuntimeContext`, and in its `finally` disconnects every cached instance
  exactly once (runs after the consumer drains: completion, `break`, error,
  abort). `runtimeCtx` was hoisted so the `finally` can reach it.
- **`runtime/parallel-driver.ts`** + **`fork-contract.spec.ts`** — `fork()`
  propagates `scanConnections` by reference (policy `shared-cooperative`, like
  `executionMemo`); the fork-contract test pins the new field's policy (the
  `satisfies Record<keyof RuntimeContext, ForkPolicy>` clause fails to compile
  otherwise).

### Fallback (scoped, intentional)

Many `RuntimeContext` construction sites do **not** set `scanConnections`
(const-evaluator, deferred-constraint-queue, materialized-view maintenance,
`Database._executeSingleStatement`, assertions, etc.). On those the scan leaf
falls back to **owning the lifecycle**: connect + disconnect per invocation, as
before — correct, just no reuse. Only the `Statement` iteration path
(`db.eval` / `db.get` / `db.all` / `stmt.*`) gets reuse.

## Why reuse is safe here

The memory vtab reads live-at-`query()`-time, so a reused instance's later
`query()` observes the same state a fresh connect would (no visibility change).
Re-scans of one site are sequential (the NLJ drains each inner cursor before the
next outer row), so a single instance is never concurrently self-live. The read
scan connects `module.connect` **directly** and never registers a
`VirtualTableConnection`, so this is independent of the `adoptConnection` /
connection-registration path.

### New spec: `test/runtime/nlj-inner-connection-reuse.spec.ts`

Uses a self-contained **counting** vtab module (tallies connect/disconnect/query
by table name) plus the memory module for row correctness. Key cases:

- **Core discriminator** — a self cross-join over the counting module asserts
  `connectCount('t') === 2` (once per scan-site, not once per outer row),
  `disconnectCount === 2`, and `queryCount === 4` (outer once + inner re-scanned
  once per outer row — proving the inner is genuinely re-scanned, not cached).
  *Before the fix this would be 4 connects (1 outer + 3 inner).* The counting
  module's `getBestAccessPlan` reports 60000 estimated rows (> the optimizer's
  `join.maxRightRowsForCaching` of 50000) so the advisory does **not** cache the
  inner — that is how the un-cached re-scan path is forced.
- **Re-execution** — re-running a prepared statement gives 4 connects / 4
  disconnects (cache resets per execution; no cross-run reuse or leak).
- **Abort mid-inner-scan** — aborting mid-stream throws `AbortError` and every
  connect is matched by exactly one disconnect (no leak, no double-disconnect).
- **Row correctness (memory)** — self-join (`a.v < b.v`), correlated IndexSeek
  join (`i.id = o.ref`), and a correlated read within one DML statement
  (`update t set v = (select count(*) from t x where x.id < t.id)` → 0,1,2).

## Reviewer: treat these tests as a floor. Suggested probes / known gaps

- **`_executeSingleStatement` gets no reuse.** The transient exec primitive
  (`database.ts`, used by `_executeStatementBatch` and some internal
  multi-statement paths) doesn't set `scanConnections`, so a join executed
  through it still reconnects per outer row (falls back correctly, no leak).
  Not covered by a test. Possible follow-up if that path ever matters for
  perf — would be a small backlog `feat-`/`debt-` item, not a bug.
- **Concurrency of a shared cache under a real parallel scan.** `fork()` shares
  `scanConnections` by reference. If a future parallelized query drove two forks
  scanning the *same* scan node concurrently, they'd issue concurrent `query()`
  on one shared instance — only safe for a module whose `concurrencyMode`
  permits it. Dormant today (ParallelDriver has no query consumers); documented
  as a NOTE in `parallel-driver.ts` fork() and the fork-policy comment. Worth a
  glance if parallel scan consumers are on the roadmap.
- **Non-memory module visibility.** The reuse-is-safe argument (live-at-query
  reads) is verified for the memory + LevelDB store modules only. A module that
  snapshots state at `connect()` (rather than at `query()`) and is *mutated
  within the same statement* by a correlated read could in principle observe a
  different snapshot than the old per-row-reconnect behavior. The ticket flagged
  this; for `_readCommitted` memory instances a single stable snapshot per
  execution is argued to be at-least-as-correct. No such third-party module
  exists in-tree to test against — a probe worth keeping in mind if one is added.
- **Cursor sharing under exotic join shapes.** The self-join cursor-isolation
  claim rests on scan-node-keyed instances + sequential inner drain. RIGHT/FULL
  joins buffer the left and drive from the right (`driveFromRight`); the inner
  callback is still re-invoked sequentially, but a reviewer may want to add a
  RIGHT/FULL self-join case to the spec for completeness.
