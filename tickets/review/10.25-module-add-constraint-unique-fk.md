# Review: module `ALTER TABLE ADD CONSTRAINT` for UNIQUE / FOREIGN KEY

## What landed

Both built-in modules now implement `module.alterTable({ type: 'addConstraint', constraint })` for
UNIQUE and FOREIGN KEY (CHECK was already engine-side in `runtime/emit/add-constraint.ts` and is
unchanged). The existing `runAddConstraintViaModule` path now succeeds: the module re-validates the
existing rows and either installs forward enforcement or fails atomically with
`StatusCode.CONSTRAINT` (schema unchanged). Closes 10.2 gap #1 (declarative add of a named UNIQUE/FK
to an existing table now converges + is idempotent). This is the prereq 10.3 depends on for its
UNIQUE/FK body-change drop+add path.

### Implementation shape

- **`schema/constraint-builder.ts` (NEW)** — single source of truth for AST→schema:
  - `buildUniqueConstraintSchema(con, columnIndexMap)`
  - `buildForeignKeyConstraintSchema(con, columnIndexMap, childTableName, childSchemaName)`
  - `validateForeignKeyOverExistingRows(db, childSchema, fk)` — pragma-gated, MATCH SIMPLE,
    self-FK-safe (aliased correlated `not exists`), parent-absent fallback (any fully-non-NULL child
    row is an orphan). Reads via `db.prepare` + scan (takes no module schema-change latch).
  - `SchemaManager.extractUniqueConstraints`/`extractForeignKeys` **table-level** arms now delegate
    to the first two (column-level arms untouched). CREATE TABLE behavior verified byte-identical by
    the full suite.
- **Memory** (`vtab/memory/layer/manager.ts`): `addConstraint` mirrors `createIndex`/`dropConstraint`
  latch + `ensureSchemaChangeSafety()` + snapshot/restore + `'alter'` event.
  - UNIQUE: builds the covering secondary index via `addIndexToBase` → `populateNewIndex` (throws
    CONSTRAINT on first in-scope dup; NULLs distinct; per-column collation honored). **Reuse:** skips
    the build only when an existing `unique:true` index already covers the exact columns; a non-unique
    index still triggers build-and-validate.
  - FK: validates via the shared validator (before the schema swap), then appends — no physical
    structure (FK enforcement is engine-side EXISTS checks).
- **Store** (`quereus-store/.../store-module.ts`): UNIQUE → `validateUniqueOverExistingRows`
  (full-scan `seen`-set dup detect, mirrors `buildIndexEntries`; NULLs distinct; partial predicate
  honored), then `updateSchema` + `saveTableDDL` + `'alter'`. FK → shared validator, then persist.
- **Parser** (`parser.ts`): the ADD branch now routes `UNIQUE`/`FOREIGN`/`CHECK` (not just
  `CONSTRAINT`) to `tableConstraint()`, enabling the **unnamed** `ALTER TABLE … ADD UNIQUE (...)` /
  `ADD FOREIGN KEY (...)` forms the ticket requires (auto-named `_uc_…` / `_fk_…`). Additive only —
  no change to previously-accepted syntax.

## Honest gaps / things to scrutinize

1. **PRE-EXISTING memory-consolidation bug (filed `tickets/.pre-existing-error.md`).** `copyTransactionDataToBase`
   (memory manager) only *inserts* a consolidated transaction layer's rows into base; it never
   *removes* base rows tombstoned (deleted) in that layer. Latent until base is populated by a prior
   schema change, then a DELETE, then another schema change — the deleted row resurfaces in the base
   primary tree and corrupts a subsequent index build / UNIQUE validation. **Reproduced with only
   pre-existing ops** (`create index` / `alter add column` + `delete` + `create unique index`), zero
   involvement of this feature. **Consequence for this ticket:** the stated edge case "delete the
   offending row, then re-run the same ADD ⇒ succeeds" is deferred — the spec and 41.8 tests
   deliberately omit that retry-on-same-table chain (clearly commented). All other ADD behavior is
   asserted and passes. No production code masks the failure. Reviewer: the root-cause fix + a
   regression test belong in the triage/fix the runner dispatches.
2. **Store UNIQUE collation limitation (carried, documented).** The store's existing-row UNIQUE check
   is value-exact (JSON signature, BINARY), matching `buildIndexEntries` and the store's
   `CREATE UNIQUE INDEX` path — it does **not** honor a per-column NOCASE collation. Rows unique under
   BINARY but colliding under NOCASE would be accepted on the store (memory rejects them, since the
   covering index keys by collation). Documented in `docs/schema.md`.
3. **UNIQUE reuse-of-existing-index** is implemented (not always-rebuild): reuse only when an existing
   `unique:true` index covers the exact columns; otherwise build-and-validate. Worth a sanity check
   that "a pre-existing `create unique index` then declarative UNIQUE on the same cols" doesn't double
   up (the differ excludes index-derived constraints, so this is an explicit-add corner; not directly
   covered by a test — candidate for an added case).
4. **FK parent-absent** behavior: with pragma on, a fully-non-NULL child row is treated as a violation
   (mirrors `foreign-key-builder.ts` null-guards-only fallback); a child with only NULL FK rows
   survives. Covered indirectly; no dedicated "parent table genuinely missing" test (the add path
   normally has the parent present). Composite/cross-schema FK adds are also not explicitly tested
   (the builder handles them; column-count mismatch ⇒ ERROR at resolve time).
5. **Persisted-DDL is lossy for constraints (pre-existing, not introduced here).** `generateTableDDL`
   emits columns + PK + USING + tags only — **no** UNIQUE/FK/CHECK. So the store's `saveTableDDL`
   does not persist any constraint across a cold reopen (true for CREATE TABLE constraints and the
   already-shipped drop/rename too, not just this add). In-session behavior (what the suites exercise)
   is correct. Flagging in case it should be its own ticket.
6. **Isolation layer:** confirmed no edit needed — `isolation-module.ts` forwards `alterTable` to the
   underlying and `translateOverlayRow` already no-ops `addConstraint`. **Not** covered by a dedicated
   isolated-table add-constraint test; the standard suites don't drive it. Candidate gap.
7. **`emit/add-constraint.ts`** confirmed unchanged (CHECK→runAddCheck, else→runAddConstraintViaModule;
   `schema.addTable` + `table_modified` notify fire as for drop/rename).
