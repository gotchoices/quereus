# MV-over-MV row-time cascade — review handoff

## What shipped

An MV whose single source resolves to another MV's backing table is now **accepted**
and maintained transparently through a chain of arbitrary depth. A write to base `t`
maintains `mv1`'s backing; that backing write's *effective* per-row delta is then routed
to every MV reading `mv1`'s backing (`mv2`), recursively — synchronous, DAG-ordered, and
atomic within the statement (whole chain commits/rolls-back in lockstep).

Three small, localized changes plus the rejection lift:

1. **`vtab/memory/layer/manager.ts`** — new exported `BackingRowChange` type;
   `applyMaintenanceToLayer` now **returns** `BackingRowChange[]` (the realized deltas:
   `delete-key`→`delete` w/ oldRow; `upsert`→`update` if it replaced a row, else
   `insert`). The layer already looked up each op's before-image, so this is free — no
   re-read. Switch stays exhaustive.

2. **`core/database-materialized-views.ts`** — `applyRowTimeChange` returns the
   `BackingRowChange[]`; `maintainRowTime(sourceBase, change, depth=0)` now, after
   applying a plan, looks up `rowTimeBySource[backingBase]` and, when non-empty,
   recurses per backing change. Leaf fast path (`!rowTimeBySource.has(backingBase)`)
   keeps a non-chained MV at exactly today's cost. New `assertCascadeDepth` guard throws
   `INTERNAL` past `rowTime.size` (defense-in-depth; cycles are structurally impossible).

3. **`buildRowTimePlan`** — deleted the `getMaterializedViewByBackingTable(...)`
   reject block; the remaining eligibility checks evaluate against the keyed backing
   schema unchanged (mv2's source PK is mv1's backing PK, which mv2 projects → eligible).
   Updated `getMaterializedViewByBackingTable`'s doc comment in `schema/manager.ts`
   (the create-gate consumer it named is gone).

### Key insight (verify this holds)
No new dependency structure was added. `building/select.ts` rewrites a reference to
`mv1` into a `TableReference` against `_mv_mv1`, so `mv2.sourceBase === 'main._mv_mv1'`
and `rowTimeBySource['main._mv_mv1']` records the producer→consumer edge the moment
`mv2` registers. The cascade just *drives* that existing index from each backing write.

## Honest gaps / divergences from the ticket (review focus)

1. **Built against the pre-abstraction code; both hard prereqs were unlanded.** At
   implement time `incremental-maintenance-plan-abstraction` (1.5) and
   `materialized-view-rowtime-per-statement-batching` (3) were still in `implement/`, so
   there is **no** `MaintenancePlan` union / `applyMaintenancePlan` / `buildMaintenancePlan`
   and **no** per-statement `mctx`. I mapped the ticket's design onto the current names
   (`RowTimeMaintenancePlan` / `applyRowTimeChange` / `buildRowTimePlan` / `maintainRowTime`)
   and threaded a `depth` counter instead of an `mctx`. The cascade rides the existing
   `getBackingConnection`, which already resolves the **same** connection per backing
   table within a transaction — so reads-own-writes (semantic-edge-B) holds without the
   batching cache; the cache is a perf optimization, not a correctness dependency.
   **Reconciliation (whoever lands the prereqs second):** rename `applyRowTimeChange`→
   `applyMaintenancePlan`; thread `mctx` via a `descend(mctx)` that also bumps the depth;
   the cascade body is otherwise unchanged. The ticket explicitly anticipated "whoever
   lands second reconciles."

2. **Equivalence harness does not exist** — `test/incremental/maintenance-equivalence.spec.ts`
   is a deliverable of the unlanded abstraction prereq, so I could not "extend" it. The
   standalone `materialized-view-cascade.spec.ts` is the chain oracle the acceptance
   names. **Fold its chain assertions into the harness when it lands.**

3. **`BackingRowChange` is the non-discriminated `{op; oldRow?; newRow?}`**, matching the
   existing inbound change shape (not the discriminated union from `rowtime-mv-minor-cleanups`,
   unlanded). Unify when that ticket lands — it is the same shape by design.

4. **Create-time cycle/depth walk skipped** (ticket marked it optional). Cycles are
   structurally impossible (a consumer MV requires its producer to exist; sources are
   fixed at create), and the runtime `assertCascadeDepth` guard is the backstop. Reviewer:
   confirm you agree the runtime guard suffices, or request the create-time walk.

5. **Mixed-arm chains not exercisable yet.** The cascade is arm-agnostic (it routes
   whatever `BackingRowChange` a plan produces), but only the covering-index
   (inverse-projection) arm exists today — `-residual-recompute` (3.2) and `-prefix-delete`
   (3.4) are unlanded. Once they land, a chain may mix arms; add a mixed-arm chain case.

6. **Fan-out cascade (one backing, multiple dependents) is covered by design but not
   explicitly tested.** `maintainRowTime` loops over *all* keys in
   `rowTimeBySource[backingBase]`, so `mv2a` and `mv2b` both over `_mv_mv1` are both
   maintained — but no test pins this. Worth a quick added case.

7. **Two-op update emits delete+insert downstream** (not a single `update`). For a
   key-unchanged source update, `applyRowTimeChange` produces `[delete-key, upsert]`;
   `applyMaintenanceToLayer` applies them in order (the upsert sees the tombstone → emits
   `insert`), so the downstream MV receives `delete(old)` then `insert(new)`. Net-correct
   for passthrough and verified by the oracle, but reviewer should confirm no
   enforcement-timing concern (within one `maintainRowTime` call the ops apply
   synchronously in order; no external reader interleaves mid-cascade).

8. **Deeper enforcement via a downstream MV** (a constraint proved by an MV that must
   see through the chain to the base) is the **lens prover's** concern
   (`lens-prover-and-attachment`), not this ticket. `54` §7 covers only the in-reach
   level-1 covering MV whose backing then cascades; documented inline.
