## Findings from reading the docs against the code

**A real doc bug, fixed.** `docs/optimizer-conventions.md`'s `PlanNodeCharacteristics` sample
defined `guaranteesUniqueRows` as `node.physical.uniqueKeys?.some(key => key.length === 0)`.
`PhysicalProperties.uniqueKeys` **does not exist** ŌĆö it was removed when functional
dependencies became the uniqueness surface (`optimizer-fd.md` ┬¦ *Rejected alternatives* even
records the removal). The real implementation reads `hasSingletonFd` with a zero-column
`estimatedRows` fallback. The doc was wrong, not the code; I corrected the sample and recorded
the "no `uniqueKeys` field" fact inside `OPT-054`. No code change, no bug ticket.

**A seed invariant dropped.** The plan listed "rules dispatch on characteristics and physical
properties, never on `instanceof` or `nodeType` equality." It fails the four-part test. Rules
are *registered* per `nodeType` (`RuleHandle.nodeType` is how the pass manager dispatches),
15 rule sites compare `nodeType` directly, and ~20 rule files use `instanceof` ŌĆö including
`ruleSelectAccessPath` on `RetrieveNode`, which `optimizer.md` explicitly names as a
sanctioned exception. It is house style, and violating it is not a bug. It stays in
`optimizer-conventions.md`, where it belongs.

**A seed invariant sharpened.** The plan said an `'aware'` rule "consults
`PlanNodeCharacteristics.hasSideEffects` / `subtreeHasSideEffects`". 25 of the 27 `'aware'`
rules do. `cte-optimization` and `in-subquery-cache` consult nothing ŌĆö they wrap the subtree
in a `CacheNode`, which is a run-once fence and preserves the write rather than refusing it.
`optimizer.md` already hedged this ("every aware rule *that consults* `subtreeHasSideEffects`
will start refusing"), so the prose was not wrong, just imprecise. `OPT-002` states it
exactly, names the two exceptions, and says the declaration is not machine-checked against a
rule's body. **Not a code bug**; the audit gate is inert today because DML cannot appear in
expression position.

**Eviction ordering corrected.** The plan said `addFd`'s cap enforcement "prefers to evict
`'determination'` over `'unique'`." `enforceCap` actually partitions on `keyHints` **first**
and only prefers `'unique'` within each partition. `OPT-046` states both.

**A weaker guard than the doc implies.** `optimizer-retrieve.md` says "a validation invariant
enforces that no `Retrieve` nodes reach emission." `validatePhysicalNodeType` does throw ŌĆö but
only under `tuning.debug.validatePlan`, which is `false` by default in production
(`optimizer-tuning.ts:255`). In a release build a surviving `RetrieveNode` surfaces as a
missing-emitter error instead. `OPT-020`'s body says so.

**One tripwire parked in code.** `PassManager.registerPass` accepts a pass whose `rules` array
is already populated and never runs `validateSideEffectMode` over it; only `addRuleToPass`
validates. Every pass ships with `rules: []`, so nothing bypasses the gate today. `NOTE:`
comment at `packages/quereus/src/planner/framework/pass.ts` ┬¦ `registerPass`. Conditional, not
a latent defect ŌĆö no dormant path reaches it, because no such pass exists.

**One backlog ticket filed** (the ticket permitted at most one):
`debt-mechanical-guards-for-optimizer-invariants`, covering the three `guard: none` entries
where a source-scanning test looks worthwhile (`OPT-002`, `OPT-046`, `OPT-052`), modelled on
the existing static convention guard in `cost-additivity.spec.ts`. No guards were implemented
here ŌĆö this was a documentation ticket, and a new assertion in the optimizer's hot path
deserves its own plan.
