## What changed

Docs + one lint override. **No production code changed** — `characteristics.ts` was only used to verify the lint override bites, then reverted (confirm it's absent from the diff).

### 1. `docs/optimizer-conventions.md` — full rewrite
Was titled "Characteristics Over Identity" and told developers **"DON'T: Use `instanceof` checks"** — the documented root cause of the duck-typing drift the `planner-capability-brands` effort fixed. Rewritten to the agreed standard:

- **Three questions, three mechanisms** table up front:
  - concrete class identity → `instanceof`
  - cross-class capability → branded marker interface + `CapabilityDetectors` guard
  - physical properties → `PlanNodeCharacteristics` over `PlanNode.physical`
  - (`nodeType` = dispatch/serialization/EXPLAIN only, not class narrowing)
- The **"by intent" distinction**: *need one class's API → `instanceof`; need "any node that can do X" → capability guard.* Both legitimate; choosing narrower `instanceof` is not a smell.
- **Removed** blanket anti-`instanceof` guidance. The reserved warning is now the real anti-pattern: **duck-typed `as any` property-presence detectors** (`'foo' in node && typeof (node as any).foo === 'function'`).
- **Documents the brand mechanism**: `implements XCapable` + set `is<X>Capable` brand + guard in `characteristics.ts`; compiler enforces completeness.
- **Notes the cross-bundle constraint**: planner nodes are singletons in `@quereus/quereus`, plugins never receive them, so `instanceof` has no cross-realm hazard.
- **Dropped** the "Migration Patterns: From instanceof to Interface Checks" section (advocated the reverse of the decided standard).
- **Preserved verbatim**: the FD/EC/bindings section (invariant citations OPT-030…OPT-054) — that content is about physical properties and was already correct.

### 2. `packages/quereus/src/planner/framework/README.md` — reconciled
Its canonical rule shape (`if (!(node instanceof AggregateNode)) return null;`) was already correct and now agrees with the conventions doc. Added a paragraph explaining the intent choice (instanceof vs capability guard) and a cross-reference link to the conventions doc's § Node discrimination.

### 3. `packages/quereus/eslint.config.mjs` — file-scoped lint guard
Added a per-file override raising `@typescript-eslint/no-explicit-any` from `warn` to `error`, scoped to **exactly** `src/planner/framework/characteristics.ts`. Any reintroduced `as any` duck-typed detector in the canonical file now fails lint. Other planner files keep `any` at `warn` (unchanged).
