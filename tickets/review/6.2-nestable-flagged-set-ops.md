## What landed

Made a nested (subtree) set-op operand recursively writable for the **unambiguous fan-out**
operations, and made the **ambiguous inserts** into a subtree reject cleanly. Independently
valuable: any nested set-op view is now writable for data/delete, not only flagged ones.

## How it works (mechanism ‚Äî for the reviewer)

`set-op-membership-write` captures the affected view rows + their probe flags **once** into the
context-backed `__vmupd_keys` relation before any branch op fires. The nested recursion **reuses
that single capture**: a subtree's leaves share the outer's data columns (nesting preserves data
columns), so "touch the leaf rows whose data tuple ‚àà `__vmupd_keys`" is the same frozen-capture
correlation rebuilt against each inner branch. A leaf's rows ‚äÜ the subtree's, so the outer
member-exists and a re-derived inner member-exists select the same rows. This is exactly why
fan-out (update/delete) recurses cleanly while insert-into-subtree does not (no single
deterministic target leaf ‚Äî product-coordinate addressing, owned by backlog
`set-op-membership-nested`).

## Test surface added (`test/property.spec.ts` ¬ß Nested / subtree set-operation membership writes)

Schema: `A={(1,10),(2,20)}`, `B={(2,20),(3,30),(5,50)}`, `C={(3,30),(4,40)}`;
`Vn = A union[inA,inSub] (B union[inB,inC] C)` (membership by id: 1=A-only, 2=A&B, 3=B&C,
4=C-only, 5=B-only). 13 cases:

- **Data fan-out through a subtree** (A-only / B‚à©C / B-only rows) + PutGet re-read; **value
  referencing a data column** (`set x = x + 1`) recurses.
- **Delete fan-out** through the subtree; **`set inSub = false`** subtree drop.
- **Flag-less subtree operand** (`A union[inA,inSub] (B union C)`) writable for data + delete.
- **`except` subtree operand** ‚Äî delete fan-out + `set inSub = false` recurse through a non-union
  inner node.
- **Deferred rejects** ‚Äî `set inSub = true`, `set inB = true`, insert-through to the subtree side:
  each asserts the message **names `set-op-membership-nested`** and is **NOT** the phase-1
  `not updateable` message and **NOT** `unknown-view-column`. **Leaf-only insert-through**
  (`inSub = false`) still works.
- **Conflicting assignment** (`set x = 5, inSub = false`) still rejects.
- **Static-surface honesty** ‚Äî `view_info`/`column_info` agree with the dynamic write.
- **Key Soundness under write** ‚Äî nested view stays keyed on data columns, no flag in a key.

## Known gaps / where to scrutinize (tests are a floor)

- **3+ level nesting** is structurally handled by the recursion (each `analyzeSetOpBranches` may
  return nested branches) but tested only to 2 levels. A 3-level fan-out / reject case would
  strengthen confidence.
- **Left-operand subtree.** A subtree on the LEFT is not SQL-reachable (a parenthesized left
  compound flattens to a `select * from (‚Ä¶)` subquery), so `setOpHasSubtreeOperand` and the
  recursion only exercise the right operand via SQL. Left-side recursion is correct by symmetric
  construction but untested.
- **`union all` (bag) subtree fan-out + duplicate data tuples** at depth ‚Äî the binary v1 caveat
  (delete/data-write fans to ALL copies of a duplicate tuple) recurses; no nested bag-duplicate
  test was added. Worth a case.
- **Renamed referenced column at depth** ‚Äî the v1 caveat (a data-fan-out value referencing a
  renamed leg column is not remapped) recurses; only the matching-leg-name case is tested.
- **`column_info` partial on `inSub`** (own subtree flag reports `is_updatable = YES` though only
  its `= false` works) is a *deliberate* documented partial, not deep per-operand column_info
  coupling ‚Äî confirm this is acceptable rather than tightening it.
- **`intersect` subtree operand** delete/data fan-out follows the same path as the tested `except`
  case but has no dedicated test.
- **Insert-through ordering nit:** `buildInsertThrough` builds a leaf active-side insert before it
  reaches and rejects a nested active side in the same statement ‚Äî harmless (plan-time only, the
  throw discards), but a reviewer may prefer an upfront all-active-sides nested check.
