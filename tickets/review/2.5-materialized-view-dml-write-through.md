## What was implemented

Write-through is **pure routing**, exactly as the ticket framed it:

- **`view-mutation.ts`** — `analyzeView` / `rewriteViewInsert` / `rewriteViewUpdate`
  / `rewriteViewDelete` / `findViewColumn` / `rejectReturning` now take a minimal
  structural `MutableViewLike` param (`{ name; schemaName; selectAst; columns? }`)
  that **both** `ViewSchema` and `MaterializedViewSchema` satisfy. No behavior change
  for plain views — the rewrite reads only those four fields. (The `ViewSchema` type
  import was dropped.)
- **`insert.ts` / `update.ts` / `delete.ts`** — the `assertNotMaterializedView(...)`
  call is removed; the view dispatch is now
  `getView(…) ?? getMaterializedView(…)`, so an MV target routes into the same
  `rewriteView*` + re-plan path plain views use. The committed-pseudo-schema guard
  stays ahead of it.
- **`schema-resolution.ts`** — `assertNotMaterializedView` deleted (it was not part
  of any public package export; only the three builders imported it).

No new maintenance code: the rewritten write hits source `T`, which fires the
existing `_maintainRowTimeCoveringStructures` hook, so the backing syncs inside the
same statement/transaction.

## ⚠️ Reviewer focus — read these first

**1. MV-over-MV correction (divergence from the ticket's stated reasoning).**
The ticket asserts "MV-over-MV cannot exist … create is already rejected … no
separate write-through rejection is needed." **This is stale/false against shipped
code.** `database-materialized-views.ts:563-566` explicitly states MV-over-MV is *no
longer rejected at create* and is supported via the maintenance cascade — and
`53-materialized-views-rowtime.sqllogic §8` + `54 §7` + `materialized-view-diagnostics.spec.ts`
all create MV-over-MV chains successfully. So write-through **can** encounter an
MV-over-MV.

Left unguarded, `insert into outer_mv` (where `outer_mv as select … from inner_mv`)
would resolve the body source to `inner_mv`'s **backing table** `_mv_inner_mv`
(`building/select.ts` rewrites MV refs to their backing) and the rewrite would emit
`insert into _mv_inner_mv …` — a backing table that is `isReadOnly`, producing a
confusing `Table '_mv_inner_mv' is read-only` error (or worse, a direct backing
write if that guard ever weakened).

**Fix applied:** `analyzeView` now rejects when the body's single source is itself a
materialized view (added right after the existing `getView`-on-source nested-view
check), with the `nested-view` reason and message `cannot write through '<name>':
its body reads a materialized view; … write the base source instead`. This:
  - prevents silent backing-table corruption / confusing errors;
  - matches the ticket's *intent* (MV-over-MV write-through is a declared non-goal);
  - is a **strict improvement that also covers plain views over an MV** — a
    pre-existing latent hole (a plain `view v as select … from mv` would have hit the
    same backing-table problem). Reviewer should confirm this broader reach is
    acceptable (it is the honest, conservative behavior).

Decision to scrutinize: I **reject** rather than route one level down to the inner
MV's own write-through. Routing-down is the eventual full behavior but is out of this
ticket's scope (and untested); rejection is the safe, honest stop. Confirm you agree
rejection is the right call here.

**2. `51-materialized-views.sqllogic §3` flipped from read-only to write-through.**
That section previously asserted `-- error: read-only` for INSERT/UPDATE/DELETE on an
MV. Removing the rejection makes those asserts wrong, so §3 was rewritten to a brief
write-through smoke (insert/update/delete through the MV name, netting back to the
pre-write state so §4's expectations still hold). This is the only pre-existing test
that asserted the old boundary (repo-wide grep confirms).

## Use cases / test coverage (the floor, not the ceiling)

New `test/logic/53.1-materialized-view-write-through.sqllogic` (9 sections):

1. Round-trip insert/update/delete through a passthrough MV (`select id, x from t`).
2. Reads-own-writes inside an open txn + rollback reverts source **and** backing in
   lockstep; committed write-through persists.
3. Rename/reorder projection (`select uid as user_id, name as full_name`) — the
   rewrite maps each MV column to its base column.
4. Filtered MV with an **equality predicate on a projected column**
   (`where status = 'open'`): constant-FD default for an omitted predicate column;
   plan-time `predicate-contradiction` rejection; predicate-scope transition on
   update (row leaves the MV, base row survives).
5. Delete identity by a **non-PK projected column** removes exactly the matching
   backing rows.
6. RETURNING through an MV rejected (`returning through view`).
7. Deterministic **expression column read-only** (`x + 1 as y`): write to `y` raises
   the computed/`non-invertible` diagnostic; passthrough write still propagates and
   re-derives `y`; insert naming only passthrough columns works.
8. Bulk write-through: `insert … select`, multi-row update, multi-row delete.
9. MV-over-MV: level-1 write-through routes to base **and cascades** to level-2;
   level-2 write-through rejected (`reads a materialized view`).

Extended `test/logic/54-covering-mv-enforcement.sqllogic §11`: write-through to a
**row-time covering MV** enforcing `unique(x,y)` — ABORT / OR IGNORE / OR REPLACE on
a write-through INSERT, and an ABORT-default UPDATE onto an occupied UC value, all
resolved against the live backing exactly as a direct source write. **Asserts source
end-state only (store-safe).**

## Known gaps / suggested reviewer probes

- **`yarn test:store` full run was not executed** (only the 3 touched files were run
  under store mode, plus the full memory-mode suite). The store-specific covering
  paths for my §11 additions passed, but a reviewer wanting belt-and-suspenders could
  run the full store suite. The ticket only required `yarn test`.
- **RETURNING / computed-column / MV-over-MV diagnostics** are asserted by substring
  in sqllogic. A focused spec (à la `93.2-view-mutation-pending.sqllogic` /
  `materialized-view-diagnostics.spec.ts`) locking the structured `reason` codes
  (`returning-through-view`, `no-inverse`, `nested-view`) for the **MV** path
  specifically would be stronger — currently the MV path reuses the plain-view
  diagnostics, which `93.x` already pin for views.
- **`predicate-contradiction` only fires for VALUES sources with literal cells**
  (inherited from `checkContradiction`); a contradicting `insert … select` is not
  caught at plan time (it just inserts a row that won't be visible through the MV).
  This is the inherited view behavior, not a regression — worth a sentence in docs if
  the reviewer thinks users will trip on it.
- The MV path's RETURNING-rejection message still says "view" (`RETURNING through
  view '<mv>'`). Cosmetic; the shared rewrite emits it. Left as-is.
- Two untracked files predate this work and were **not** touched:
  `docs/.view-updateability.utf8.md`, `vu_temp.md` (stray; not mine to remove).
