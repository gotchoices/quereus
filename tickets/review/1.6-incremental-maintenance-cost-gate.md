## What landed

A shared **backward (maintenance-direction) cost surface** plus a create-time gate that
makes MV row-time eligibility a cost choice among structurally-sound strategies (replacing
the implicit shape-allowlist judgement).

### 1. Cost surface — `packages/quereus/src/planner/cost/index.ts`
Appended after the existing forward cost helpers; reuses `COST_CONSTANTS` + `chooseCheapest`:

- `type MaintenanceStrategy = 'inverse-projection' | 'residual-recompute' | 'full-rebuild'`
  (mirrors `MaintenancePlan['kind']`).
- `interface MaintenanceSourceStats { tableRows; distinctGroupsEstimate?; forwardBodyCost; fallbackRatio? }`.
- `maintenanceCost(strategy, changeCardinality, stats)`:
  - inverse-projection: `cc × (INDEX_SEEK_PER_ROW + PROJECT_PER_ROW)` — O(1)/row.
  - residual-recompute: with stats, `cc × rowsPerGroup × (SEQ_SCAN+FILTER+PROJECT)`
    (`rowsPerGroup = tableRows/distinctGroupsEstimate`); **no stats →
    `cc × forwardBodyCost × fallbackRatio`** (byte-for-byte the legacy `deltaPerRowFallbackRatio` heuristic).
  - full-rebuild: `forwardBodyCost`.
- `selectMaintenanceStrategy(soundStrategies, cc, stats)` — argmin (empty set → `'full-rebuild'`).
- `isFullRebuildPathological(stats)` and `shouldDegradeToRebuild(cc, stats)` — exported + unit-tested,
  but **not yet wired into the gate/runtime** (see "dormant machinery" below).
- `MAINTENANCE_REBUILD_ROW_THRESHOLD = 10_000` (first-cut constant).

### 2. Create-time gate — `database-materialized-views.ts`
- `MaintenancePlanCommon` base added to the three plan arms; carries `chosenStrategy` +
  `sourceStats` (DRY refactor — removed the duplicated `mv/sourceBase/backingSchema/backingTableName`).
- `buildMaintenancePlan`: after the soundness checks pass (covering-index shape), it builds
  `MaintenanceSourceStats` via `estimateMaintenanceStats` (from `optimizer.getStats()` +
  forward cost helpers + `optimizer.tuning.deltaPerRowFallbackRatio`), runs
  `selectMaintenanceStrategy(['inverse-projection'], ~1% of tableRows, stats)`, stores
  `chosenStrategy` + `sourceStats` on the returned plan, and keeps a defensive INTERNAL guard
  if a non-inverse strategy is ever chosen.

  **IMPORTANT design point (and a bug I hit + fixed):** the sound set for the covering shape
  is **`['inverse-projection']` only** — NOT `['inverse-projection','full-rebuild']`. The ticket
  invariant is "inverse-projection … is always cheapest for its eligible shapes and is never
  demoted to 'full-rebuild'." My first cut put both in the sound set; for a **small/empty source**
  `forwardBodyCost` (≈ `seqScanCost(0)=0.1`) is *cheaper* than inverse-projection's per-row cost
  (0.4), so argmin picked the unwired `'full-rebuild'` and the guard threw — breaking ALL MV
  creation (caught by the full suite). The fix encodes the invariant directly: `'full-rebuild'`
  is the floor for shapes where inverse projection is NOT sound, so it is not a competitor here.

### 3. deltaPerRowFallbackRatio scope — docs only
`DeltaExecutor` / assertions / watchers are **unchanged**. `docs/incremental-maintenance.md`
now states the ratio governs the detection kernel only and that MV maintenance uses
`maintenanceCost(...)` with the ratio as its stats-absent fallback (spike note + inline cost
fallback + design-decision updated). `docs/optimizer.md` has **no** `deltaPerRowFallbackRatio`
reference (confirmed via `git show HEAD:docs/optimizer.md | grep` → 0 hits), so nothing changed there.

## Honest gaps / where to dig (treat tests as a floor)

- **Dormant machinery.** `isFullRebuildPathological` (reject-at-create) and `shouldDegradeToRebuild`
  / `ResidualRecomputePlan.degradeToRebuild` (per-write runtime demotion) are implemented + unit-tested
  as pure functions but are **NOT reached by any runtime path**: the only sound/wired arm is
  inverse-projection, which never degrades and never rejects. The ticket's "with-stats crossover →
  reject", "runtime demotion → degradeToRebuild", and end-to-end demotion key-tests therefore could
  not be written as true integration tests — they activate when `materialized-view-rowtime-general-bodies`
  widens `soundStrategies` and wires the residual/full-rebuild arms. The wiring point for demotion is
  known: `dml-executor.ts` has `changes.length` (statement cardinality) → thread into `maintainRowTime`
  → residual arm calls `shouldDegradeToRebuild`.
- **`forwardBodyCost` is estimated** from `seqScanCost + filterCost? + projectCost` of the source, not
  the optimizer's physical plan cost (`optimizeForAnalysis` is pre-physical). Reasonable v1.
- **`distinctGroupsEstimate`** only populated for single-column PKs (`StatsProvider.distinctValues` is an
  optional per-column-**name** method); multi-column PKs leave it unset → residual no-stats path. Only
  matters for the (unreachable) residual arm.
- **First-cut constants:** `MAINTENANCE_REBUILD_ROW_THRESHOLD = 10_000`, create-time cardinality estimate
  `max(1, tableRows × 0.01)`. `isFullRebuildPathological` = "large source AND body costs more than a scan"
  — confirm this matches the intended reading of the ticket's threshold.
- **Harness instability during implementation.** This run hit repeated windows of corrupted / non-returned
  tool output (stale reads of a non-existent `materialized-view-maintenance.ts`; fabricated "all tests
  passing"; a spurious 1-char edit to `optimizer.md`, reverted). Every claim above that is stated as a
  number was taken from a *real* run that returned consistent output; where a re-run could not be confirmed
  it is flagged. **Reviewer: verify the diffs and re-run `yarn typecheck` + `yarn test` directly** rather
  than trusting any narrative. Working tree = `cost/index.ts`, `database-materialized-views.ts`,
  `docs/incremental-maintenance.md` (modified) + new `test/incremental/maintenance-cost.spec.ts`.
