## What shipped

`decomposition.ts` `routeAssignment` no longer rejects an optional-member / EAV-pivot UPDATE assignment. Each routed assignment now drives a per-row materialization transition, emitted as plain AST `BaseOp`s keyed off the existing anchor subquery (`<memberKey> in (select <anchorKey> from <anchor> where <pred>)`), still **anchor-last** so every branch reads the intact anchor:

### Optional columnar member (e.g. `T_c(id, c)` outer-joined onto the anchor)
- **matched** (the component row exists) → the existing per-member base UPDATE.
- **absent** (no component row for the anchor key) → a null-extended materialization INSERT: `insert into <member> (<memberKey>, <cols…>) select <anchorKey>, <values…> from <anchor> where <pred> on conflict (<memberKey>) do nothing`. The anchor key threads the member key (EC); each assigned value fills its column; unassigned value columns take base defaults. The `on conflict … do nothing` cedes the matched rows (whose member row already exists) to the matched UPDATE, so the two ops partition the affected rows.
- **all value columns of the member assigned null** → a base DELETE of the component row (the read renders an emptied row identically to absence). A *partial* null (some value columns assigned, all null, but not all of the member's value columns) is a matched UPDATE only — the row survives and an absent row stays absent.

### EAV pivot member (per attribute, independently)
- **non-null** value → upsert the `(entity, attr, value)` triple: a matched UPDATE of the value column (`set <val> = <value> where <attr> = '<attribute>' and <entity> in (<anchor subquery>)`) + a materialize INSERT `on conflict (<entity>, <attr>) do nothing`.
- **null** value → delete that attribute's triple.

### Why `on conflict do nothing` rather than the ticket's `not in (select <memberKey> from <member>)`
The ticket spelled the absent-row filter as `… and <anchorKey> not in (select <memberKey> from <member>)`. That makes the INSERT's source subquery **scan its own insert target**, which the planner cannot assign an access path to (`RetrieveNode … was not rewritten to a physical access node`). Switching the exclusion to `on conflict (<key>) do nothing` removes the self-scan and is order-independent with the matched UPDATE (the UPDATE never changes member keys, so the conflict set is stable either way). **Reviewer: validate this is behaviorally equivalent to the anti-join the ticket drew, and weigh the conflict-target caveat in gap 2 below.**

## Value handling (a deliberate v1 narrowing — see gap 1)
A non-null optional/EAV value must lower (via `substituteViewColumns`) to a **constant** (no column references) — because the matched UPDATE evaluates the value in the *member's* scope while the materialize INSERT evaluates it over the *anchor*, so only a scope-independent value survives both. A value referencing any column (member self-ref, anchor column, cross-member, subquery) is rejected `unsupported-decomposition-update`. A null literal is the delete/absent-no-op trigger. (Parameters / functions of constants are admitted — scope-independent.)

## Example
```sql
-- logical x.T over T_core(id,a) ⋈ T_b(id,b) ⟕ T_c(id,c)   [T_c optional]
update x.T set c = 200 where id = 7;
--   matched : update T_c set c = 200 where id in (select id from T_core where id = 7)
--   absent  : insert into T_c (id, c) select id, 200 from T_core where id = 7
--               on conflict (id) do nothing
update x.T set c = null where id = 7;   -- c is T_c's only value column → delete the T_c row
```

## Known gaps / risks the reviewer should weigh (tests are a floor)
1. **Constant-only values (the main v1 limitation).** Any non-null optional/EAV value that references a column rejects — including the genuinely-expressible `set c = <anchorKey>` and `set c = c + 1` (matched-update-only) cases. The generalization is the correlated-subquery form for anchor-resolvable values, or consumer 1's capture substrate for arbitrary values. Deferred; flag whether a follow-up ticket is warranted.
2. **`on conflict (<key>) do nothing` targets the member's PK.** I target `[memberKey]` (columnar) / `[entity, attr]` (EAV), assuming the shared key is the member's declared PK/unique (true for every advertisement the tests build). A decomposition member whose shared key is **not** a declared unique constraint would make the targeted conflict clause fail to match an index. Untested — consider an advertisement where the stitch key isn't the member PK.
3. **Surrogate-keyed optional members untested for UPDATE.** The materialize threads `singleKeyColumn(anchor)` → `singleKeyColumn(member)` (distinct spellings for a surrogate), but the only surrogate test covers mandatory members + INSERT. An optional member under a surrogate key isn't exercised through this path.
4. **Multi-value-column partial materialize: base-default soundness.** The view-soundness guard rejects materializing an absent row when an unassigned value column is NOT NULL or carries a declared default (it would widen the absent row's image / be uncreatable). Tested both ways (success when unassigned columns are nullable-no-default; reject on a defaulted column). A `not-null`-no-default uncovered base column is still caught by `assertNoMissingNotNull` (create-conflict via `no-default`).
5. **Same-member matched-UPDATE + materialize-INSERT ordering.** Emitted UPDATE-then-INSERT; analyzed order-independent (the UPDATE never changes member keys, the INSERT skips existing via the conflict clause). Atomicity is the existing per-statement rollback. Worth a fresh-eyes trace.
6. **No GetPut over the new ops in the columnar property test.** The existing GetPut writes only the mandatory writable columns (a, b); it does not write `c` back, so the materialize/delete branches aren't covered by a GetPut idempotence check. The PutGet oracle covers post-state, but a GetPut over `c` would harden the floor.

## Out of scope (still rejecting — verify they stay red)
- A **shared-key (identity) column** UPDATE (`unsupported-decomposition-update`).
- A **non-constant** optional/EAV value (`unsupported-decomposition-update`, gap 1).
- A WHERE touching a **non-anchor member / EAV pivot / subquery** (`unsupported-decomposition-predicate`) — unchanged.
- **Composite shared keys** (`unsupported-decomposition-key`), **computed-mapping** writes (`no-inverse`), **cross-member** value reads (a column-referencing value is subsumed by the constant-only reject).
- **RETURNING** through a decomposition-backed logical table (`returning-through-view`).
