## What was implemented

Phase 2 of FD direction B (decision record: plan ticket
`fd-determination-reader-side-preservation`). Phase 1 (`fd-kind-provenance-field`)
made `FunctionalDependency.kind: 'unique' | 'determination'` trustworthy at every
producer/transform/fan-out site; this phase makes the readers consume it and
deletes every producer-side drop gate the readers subsume.

### The reader rule (fd-utils.ts)

- **`isUniqueDeterminant(attrs, fds, columnCount, isSet)`** — new primitive. True
  iff closure(attrs) covers every column AND uniqueness is reachable: the relation
  is a set, OR some unguarded `kind:'unique'` FD has determinants ⊆ closure(attrs).
  Coverage alone over a bag proves nothing.
- **`isSuperkey` renamed `closureCoversAll`** — pure-coverage predicate; the rename
  forced a coverage-vs-uniqueness audit of every call site (all accounted for: the
  primitive's internal use, and the key-utils fallback which switched to the
  uniqueness primitive with `isSet:false`).
- **Rewritten readers**, all now threading `isSet`:
  `deriveKeysFromFds` / `hasAnyKey` / `hasSingletonFd` (≡ `isUniqueDeterminant(∅,…)`;
  zero-column early-out preserved) / `isUnique` closure branch (proper-subset guard
  dropped — an all-columns probe on a bag fails the rule on its own) / `keysOf`
  (isSet threaded; all-columns set fallback unchanged).
- **`isAssertedKey` deleted** (would have been a pure alias); its two callers
  (sort.ts / window-node.ts strict-monotonicOn) call `isUniqueDeterminant` directly.

### Gate removals (the point of direction B)

- `foldSingleSingleGated` DELETED; reference.ts (CHECK + assertion-hoist; the
  `realKeyFds` snapshot died too) and filter.ts (predicate FDs) now fold all
  producer FDs unconditionally via plain `addFd`.
- filter.ts `activateGuardedFds`: endpoint gate removed — activation is now
  unconditional `stripGuard` (kind-preserving) for every entailed FD. The
  `valueEquality`-keyed EC lift kept exactly as before.
- join-utils.ts: equi-pair FDs emitted unconditionally (both directions,
  'determination'); `equiKeyFds` probe deleted; `dropSideKeyFds` DELETED — a
  non-preserved side now gets phase 1's `downgradeUniqueFds` only (inner/cross/
  left/right arms), so the value claims survive as determinations.
- project-node.ts injective-pair endpoint gate removed (unconditional
  'determination' emission, matching returning-node, whose latent gate-less
  emission is now sound by the reader rule).

### isSet threading at callers

characteristics.ts (`guaranteesUniqueRows`/`hasUniqueKeys` → `node.getType().isSet`),
sort.ts, window-node.ts, rule-join-greedy-commute.ts, async-gather-node.ts
(crossProduct singleton probe — its stale "known gap" note rewritten), key-utils.ts
fallback branch (`isSet:false`, conservative — preserved keys mint 'unique' FDs
downstream so coverage alone must never qualify).

## Bug fixes verified (use these for review validation)

1. **Constant-pin phantom singleton** (`∅→col` pins survive projection; old
   `hasSingletonFd` read coverage as ≤1-row on a bag and dropped a required
   DISTINCT):
   `select distinct a, b from t1 where a = 1 and b = 2` over
   `t1(a,b,c pk)` rows (1,2,1),(1,2,2),(3,4,3) → DistinctNode retained, 1 row
   (HEAD gave 2).
2. **Partial-unique guarded key FD through a fanning join** (guarded 'unique'
   crosses the fanning join downgraded by phase 1, activates above, projection
   derives phantom `{c}` key over duplicated rows):
   the ticket's exact repro → DistinctNode retained, 1 row (HEAD gave 2 dups).
   Non-fanning 1:1 control: FD stays 'unique', DISTINCT still eliminated, 2 rows.

## Direction-B payoff verified (positive coverage)

Over non-keyed `pt(a,b,c, check (b = a + 1))`:
- ORDER BY: `select distinct a, b from pt order by a, b` → SortNode prunes to ONE
  key (the restored `{a}→{b}` determination feeds `computeClosure`). Note: a bare
  `from pt order by a, b` never materializes a Sort (the implicit all-columns
  `_primary_` index satisfies the ordering) — that's why the spec interposes a
  DISTINCT.
- GROUP BY: `group by a, b` collapses to `group by a` + `min(b)` picker
  (`minimalCover`), results identical.
- Derived key above DISTINCT: `keysOf(DistinctNode)` for
  `select distinct a, b from pt` now includes `[a]` (the isSet branch), and a
  stacked outer DISTINCT is eliminated.

## Known gaps / notes for the reviewer

- **FD-cap eviction stress test not added** (ticket said "only if cheap"): a
  relation with >64 CHECK determinations could in principle evict a 'unique'
  witness — phase 1's `enforceCap` unique-first bias defends this, but there is
  no dedicated stress spec. Under-claim only (sound), but worth a look.
- **Zero-column semantics choice**: `isUniqueDeterminant(∅, fds, 0, isSet)`
  returns `isSet` (semantically correct — a zero-column set is ≤1-row);
  `hasSingletonFd` keeps a `columnCount <= 0 → false` early-out to preserve the
  legacy contract (callers route zero-column ≤1-row via `estimatedRows`).
  `isUnique([], rel)` on a zero-column SET now returns true where it previously
  depended on a declared empty key — believed strictly more correct; no test
  exercises a zero-column bag/set distinction through `isUnique`.
- The key-utils fallback (`leftType`/`rightType` unavailable) passes
  `isSet:false`; if a caller exists that reaches the fallback with genuinely
  set-typed sides, preserved-key completeness is reduced there (soundness
  unaffected). I found no such caller.
- `valueEquality` EC lift in `activateGuardedFds` retains its single↔single
  shape check (the marker is only ever set on such mirrors) — intentional
  belt-and-suspenders, not a behavior change.
- Out-of-scope items from the plan record (new determination producers for
  computed projection terms; NDV/cardinality reading determinations) were NOT
  needed mid-implementation; no backlog tickets filed.
