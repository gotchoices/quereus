unrelated `## Visited Tracking Architecture` — is now `docs/optimizer-fd.md`, a

sanity-check that no normative clause was lost with them.**

## Genuine drift found and corrected

Two doc claims contradicted the code. Per the ticket ("fix the prose, note it in
the handoff, do not open a ticket"), both were corrected in place:

1. **The check-shape table claimed the `col1 = col2` bi-directional FDs are folded
   "only when an endpoint is a real declared key (else gated)".** They are not.
   `packages/quereus/src/planner/analysis/check-extraction.ts:313-314` pushes both
   directions unconditionally as `kind: 'determination'`. The endpoint gate was
   removed when the reader rule landed; the table row was never updated. It also
   directly contradicted the `TableReferenceNode` row of the per-operator table
   twenty lines away, which correctly said CHECK FDs "fold unconditionally."
2. **The IND section claimed "no producer mints [the `relation` IND variant] this
   wave".** `packages/quereus/src/schema/lens-compiler.ts:924` mints it.

A third, smaller fix: the guarded-FD section said "see § Unique constraints above",
a heading that does not exist in any doc in the repo. The sentence now names the
code site (`relationTypeFromTableSchema`) instead.

Every symbol the doc names in `fd-utils.ts` was spot-checked against the module's
export list and all still exist: `computeClosure`, `determines`, `minimalCover`,
`closureCoversAll`, `isUniqueDeterminant`, `hasAnyKey`, `hasSingletonFd`,
`deriveKeysFromFds`, `keysOf`, `isUnique`, `isAtMostOneRow`, `addFd`, `mergeFds`,
`shiftFds`, `projectFds`, `stripGuard`, `superkeyToFd`, `singletonFd`,
`addSingletonFd`, `extractEqualityFds`, `predicateImpliesGuard`, and the
EC / binding / domain / IND helper families.

The "only consumer of `PhysicalProperties.inds` is the coverage prover" claim was
verified by grep: every other `.inds` site propagates rather than reads.
