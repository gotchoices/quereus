# Identity-preserving reshape — review handoff

Part 4 of the maintained-table chain. Previously a source `alter` that shifted a
derivation's output shape made `refresh` **drop + recreate** the maintained table
(`rebuildBackingTable`) — a new incarnation, fatal to a replicated basis table's
row metadata and the cascade discipline. This ticket replaces that with an
**identity-preserving in-place reshape** and a **sited error** for the cases that
can't be expressed in place. `rebuildBackingTable` is deleted (no remaining
callers).

## What changed (design as built)

`emitRefreshMaterializedView` keeps two of three legs untouched — the
explicit-column-count error and the `backingShapeMatches` data-only fast path
(`rebuildBacking`). The third leg (shape shifted) now calls **`reshapeBacking`**:

1. **`classifyBackingReshape(current TableSchema, shape BackingShape)`** — a
   column-level delta old→new, matched **by name** (case-insensitive) via a
   two-pointer alignment with positional-rename detection. **Expressible** =
   trailing adds + drops + positional renames + per-column attribute (type /
   collation / not-null) shifts, with surviving columns' relative order preserved
   AND the physical PK unchanged. **Inexpressible** = an interleaving reorder (a
   new column landing mid-output, the canonical `select *` join add) or a
   physical-PK definition change (column set / order / direction / collation).
   - Shares the per-column predicates (`backingTypeMatches`,
     `backingNotNullMatches`, `backingCollationMatches`) with the positional
     `describeBackingShapeMismatch` (the rename-propagation "pure name shift?"
     check) — no second column-compare. `describePhysicalPkChange` compares the
     PK **by name through the rename map** (add/drop shift indices).
2. **`reshapeBackingInPlace`** (expressible) — apply the ordered module
   `alterTable` ops (renames → adds → attribute shifts → drops, data-lossless
   first) → re-register the reshaped schema + shape-updated derivation → data
   reconcile via the shared `rebuildBacking` → assert any deferred NOT NULL →
   fire **one `table_modified`** (NOT `table_removed`/`table_added`). Same host
   incarnation throughout.
   - **Added columns go in NULLABLE**; their values arrive with the reconcile and
     any NOT NULL is asserted *after* it (so a non-empty backing never trips "ADD
     NOT NULL without a default"). This is the `tightenNotNull` post-pass.
3. **Inexpressible / no-`alterTable` host** → `inexpressibleReshapeError`:
   *"the derivation's output shape changed incompatibly with table `<name>` (…);
   alter the table to the new shape and re-attach, or drop and recreate"* — the
   table and rows are left untouched, the MV stays `stale`.

## Use cases to validate (the test floor — treat as a floor, not a ceiling)

Reworked `materialized-view-refresh-reshape.spec.ts` (10 tests) + sqllogic §9/§11:

**Expressible in place** (assert no `table_removed`/`table_added`):
- Trailing source add reaching a single-source `select *` → in-place addColumn.
- **NOT NULL** trailing add → added nullable, reconcile fills, tightened to NOT
  NULL; a second refresh is the fast path (proves no re-reshape loop / convergence).
- Output column dropped (source `drop column`) → in-place dropColumn.
- Non-PK attribute shift (source `set collate`) follows through the passthrough.
- Row-time maintenance after a reshape: `read(mv) == evaluate(body)` and a source
  insert maintains the reshaped backing (maintenance re-registered on new shape).
- Producer→consumer: a producer in-place reshape fires `table_modified` (NOT
  incarnation events), the consumer goes stale via the cascade and recovers by
  its own refresh.

**Inexpressible → sited error** (assert table + rows untouched, MV stays stale):
- Interleaving `select *` join add (new column lands mid-output).
- Physical-PK collation change (re-keying the row identity).
- Explicit-column-list count shift (unchanged pre-existing behavior).

**Store host:** `mv-store-backing.spec.ts` "an in-place reshape after a source
ALTER keeps the backing in the store module" — the store module's `alterTable`
drives the same in-place path; module identity + data + live maintenance preserved.

**Join read-rewrite:** `query-rewrite-join.spec.ts` "select* join … reshapes the
MV table in place and RE-ENABLES the rewrite" (reworked: trailing add to the LAST
join table so it's expressible; the rewrite resumes after the reshape).

## Known gaps / risks (be adversarial here)

- **Mid-sequence reshape failure recovery is documented but only convergence-on-
  success is tested.** The claim "a failed reshape leaves the table stale and a
  re-run refresh re-derives the residual delta and converges" is exercised for the
  NOT NULL tighten case (second refresh = fast path) but NOT for an actual
  mid-sequence module throw (e.g. an `alterColumn` MISMATCH narrowing live rows,
  then a re-refresh after fixing the data). Worth a targeted test.
- **PK-column *type* change is permitted** (only set/order/direction/collation of
  the key are PK-definition changes per the ticket) and routes through
  `alterColumn setDataType` on a PK column — memory's re-key-on-type-change path
  there is untested. Low-likelihood (passthrough PK type shift) but unverified.
- **`describeBackingShapeMismatch` (positional) vs the new alignment classifier**
  can diverge on a pathological multi-shift like `[a,b]→[b,c]` (positional →
  "pure name shift"; alignment → "drop a, add c"). The positional form was kept
  for rename propagation deliberately (single-rename propagation never produces a
  multi-shift), so this is theoretical — confirm the reasoning holds.
- **Store reshape in an explicit transaction, and store NOT-NULL/narrowing
  alters**, are not pinned. The data-only refresh-in-txn parity test still holds;
  the *reshape* path on store inside an explicit transaction (multiple
  `alterTable` + one `replaceContents`) has no dedicated parity test. The store's
  `alterColumn`/`addColumn` arms are exercised only via the trailing-nullable-add
  case.
- **Two-table+ reshapes and renames at refresh** are classifier-covered but only
  the single-column shapes are tested end-to-end; a combined drop+rename+add in
  one refresh has no direct test.
