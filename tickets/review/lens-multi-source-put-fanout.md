## What this ticket was, and the scope decision

The original `implement/` ticket asked for the **full** put direction of n-way
decomposition: insert / update / delete fan-out, with a shared key that may be a
**surrogate** generated by a basis default and **evaluated-once-per-row-and-threaded**
across every member. Its own (authoritative, dated 2026-05-31) prereq notes said to
"build the fan-out against the envelope `view-mutation-shared-surrogate-insert` (3.6)
ships, not re-invent it," and expected the runner to defer the ticket until that
substrate landed.

Verified state at implement time:

- **The surrogate-insert envelope is NOT present.** `view-mutation-shared-surrogate-insert`
  (3.6) is still in `implement/`, and its mechanism is *explicitly undecided* (its own
  TODO: "Decide pre-evaluate vs capture-and-thread"). It owns ALL multi-source insert +
  the per-row/per-statement shared-surrogate mutation-context envelope, in the same core
  files this ticket touches. Building insert now would mean re-implementing 3.6's
  charter against an undecided surface — exactly what the ticket told us not to do.
- **A second substrate gap:** a decomposition DELETE that reaches *all* members, when
  filtered by a predicate spanning more than the anchor, needs **snapshot-consistent
  multi-member base-op execution** — the very capability `multi-source.ts` and the
  `view-mutation-lenient-multiside-delete-fanout` backlog ticket explicitly **defer**.

tess rules forbid parking on a tess prereq still in `implement/` ("deferred
automatically, not blocked") and require advancing with the best option. So this pass
**built the genuinely substrate-independent, sound slice** (DELETE + UPDATE fan-out that
need no surrogate envelope and no snapshot substrate) and split the substrate-dependent
remainder into a follow-on (`lens-multi-source-put-insert-fanout`, prereq 3.6). The
runner mis-dispatched this ticket because 3.6 is a *same-stage* (implement) prereq that
the cross-stage auto-defer does not catch — documented here so it is not mistaken for a
silent scope cut.

## What shipped (verify these)

A decomposition lens is registered as an ordinary view whose `selectAst` is the
synthesized `anchor ⋈ members` join (`compileDecompositionBody`). Routing its writes
through the generic two-table join path (`multi-source.ts`) is **unsound for a
decomposition** — that path picks a *single* delete side (would orphan the other
members), caps at two members, and rejects the outer joins optional members ride.

- **Routing** (`propagate.ts` → `decompositionStorage`): when the target's lens slot
  carries a `primary-storage` advertisement and **no** override (⇒ the body is exactly
  the synthesized decomposition), writes route to `propagateDecomposition`
  (`decomposition.ts`) *before* the `isJoinBody` path. Plain views / MVs / name-match
  lenses / overridden lenses are unaffected (a cheap `getLensSlot` returns undefined).

- **DELETE** (`decomposition.ts`) fans out to **every** member (mandatory, optional, and
  EAV pivot). The load-bearing soundness argument: members are ordered **anchor-last**,
  and each non-anchor member's identifying set is read from the **anchor alone**
  (`… where memberKey in (select anchorKey from anchor where <pred>)`), never the full
  join — so an earlier member's delete can never shrink a later member's identifying set.
  A no-`WHERE` delete is an unconditional `delete from <member>` per member (also the
  sound **singleton** full-delete path — no key to thread). An EAV pivot's delete removes
  every triple for the matched entity (keyed by `attributePivot.entityColumn`).

- **UPDATE** routes each `set` assignment to the single **mandatory, non-EAV** member
  backing it (off the advertisement's `member.columns`), and emits one per-member UPDATE
  keyed off the anchor the same anchor-last way. Multiple members update in one statement.

- **Deferred, each with a precise diagnostic** (new reasons in `mutation-diagnostic.ts`):
  `unsupported-decomposition-insert` (rides 3.6's envelope),
  `unsupported-decomposition-predicate` (WHERE references a non-anchor member — needs the
  snapshot substrate), `unsupported-decomposition-update` (optional/EAV/shared-key
  column), `unsupported-decomposition-key` (composite/absent shared key — v1 is
  single-column).

## Use cases / how to exercise

Set up a decomposition lens via an advertising module (see
`test/lens-put-fanout.spec.ts` `setup()` / `split()`), seed the **basis member tables
directly**, then write through the logical table:

```
-- columnar split: T_core(id,a) anchor + T_b(b) mandatory + T_c(c) optional, key=id
delete from x.T where id = 1;     -- removes id=1 from T_core, T_b, and T_c
delete from x.T where a = 20;     -- anchor-column predicate, fans out to all members
delete from x.T;                  -- truncates every member
update x.T set a = 99 where id=2; -- routes to anchor member T_core
update x.T set b = 999 where id=1;-- routes to mandatory non-anchor member T_b
update x.T set a=11, b=111 where id=1; -- two member updates, one statement
-- deferred (diagnostic, no write):
insert into x.T (id,a,b) values (3,30,300);  -- unsupported-decomposition-insert
delete from x.T where b = 100;               -- unsupported-decomposition-predicate
update x.T set c = 5 where id = 1;           -- unsupported-decomposition-update (optional)
update x.T set id = 9 where id = 1;          -- unsupported-decomposition-update (key)
```

EAV: `test/lens-put-fanout.spec.ts` `eavSplit()` — `delete from x.E where id=1` removes
the anchor row and every triple for that entity.

## Tests are a floor — known gaps to scrutinize

`yarn workspace @quereus/quereus test` (4161 passing, 9 pre-existing pending), `lint`,
and `build` are green. The new spec (`lens-put-fanout.spec.ts`, 12 cases) plus the
rewritten `lens-advertisement.spec.ts` "put fan-out" case cover the happy paths and each
deferral diagnostic. **Not yet covered / verify adversarially:**

- **The anchor-last + anchor-only-subquery soundness argument** is the crux — re-derive
  it. Is there a member topology (≥2 mandatory non-anchor members, or an anchor key
  spelled differently per member) where an earlier op corrupts a later op's identifying
  set? The single-column-key restriction (`unsupported-decomposition-key`) is the v1
  guard; confirm it actually fires for every composite-key path.
- **Cross-member UPDATE value references** (`rewriteAssignedValue` → `cross-source-assignment`)
  are implemented but **untested** — add `update x.T set a = b + 1` (a∈anchor, b∈other).
- **Mutation context** (`contextValues`) is threaded into each member statement but not
  asserted; a per-statement context value across a multi-member fan-out is untested.
- **Conflict / FK / RETURNING parity** (a "Key test" in the original) is **not** covered.
  RETURNING through a decomposition is rejected (`returning-through-view`); `on conflict`
  composition across member ops and FK ordering between members are untested. The fan-out
  reuses the substrate's per-`BaseOp` base-table builders + the `ViewMutationNode`
  sequencer, so conflict/FK behavior is whatever those give — confirm that is sound for a
  multi-member group, or file a ticket.
- **`yarn test:store`** (store-path write fan-out) was **not** run — deferred per the
  ticket's note; a reviewer with time should run it.
- **Mandatory-existence IND trust** (the original's "no O(n) existence probe" assertion):
  the update/delete fan-out keys members off the anchor subquery and never probes
  per-member existence, so the property holds structurally, but there is no regression
  assertion pinning it.
