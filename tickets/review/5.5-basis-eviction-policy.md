# Review: basis-table eviction policy & network-wide retirement signal (dynamic half)

Builds on the completed `basis-lifecycle-classification` static record. Adds the
**dynamic** "is any peer still writing this table directly?" signal and the
**default eviction policy** that reclaims a detached legacy table's local storage
once it has been quiet for the retention horizon. Realizes `docs/migration.md`
§ 4 Contract.

The implement-stage build is complete, builds clean across all touched packages,
and the full monorepo test suite is green (engine 6347, store 646, sync 349,
sync-client 45, others pass). Treat the tests as a floor, not a ceiling — the
honest gaps below are where to push.

## What landed (architecture)

**Dynamic signal — `lastDirectlyMappedWriteAt`.** In `change-applicator.ts`,
after a batch's data changes are admitted, `bumpLastDirectlyMappedWrites` runs
inside the admission `commitMetadata` closure (durable with the rest of the
metadata, before the watermark advances). For each *applied* (hence remote —
self-origin is skipped before resolution) change to a tracked basis table whose
lifecycle record is **not** `directly-mapped`, it bumps
`lastDirectlyMappedWriteAt = max(current, change.hlc.wallTime)`, one batched KV
write per touched table. It is **advisory** — wrapped in try/catch, a failure is
logged and swallowed (never aborts the apply). Zero overhead pre-migration: one
bounded `getAll` scan, immediate return when no records exist.

**Effective-horizon math** (pure, in `basis-lifecycle.ts`):
- `effectiveEvictHorizonMs(record, config)` → `null` (never) / `0` (immediate) /
  ms. Per-table `evictPolicy` override wins; absent ⇒ global
  `config.basisEviction.mode` (`horizon` default → `horizonMs ?? retentionHorizonMs`).
- `quietSince(record) = max(unmappedSince ?? detachedAt, lastDirectlyMappedWriteAt)`.
- `isEvictable(record, now, config)` = `state === 'detached'` AND horizon not null
  AND `now - quietSince >= horizon`. **In-basis `unreferenced` is never evictable**
  (load-bearing safety rule).

**Capture at observation time** (`recordLensDeployment`): for each in-basis table
it snapshots the `quereus.sync.evict` tag → `record.evictPolicy` and the
secondary-index names → `record.indexNames`; on detach these are carried through
(the schema + tag are gone afterward). New `record.detachedAt` is stamped on
entering `detached`, cleared on re-attach. Two new record fields
(`detachedAt`, `indexNames`) extend the prereq's reserved (`lastDirectlyMappedWriteAt`,
`evictPolicy`) shape; `basisLifecycleRecordChanged` now compares all of them.

**The sweep** — `SyncManager.evictExpiredBasisTables(now = Date.now())`,
host-driven like `pruneTombstones`/`pruneQuarantine` (no library timer). Lists
records, selects `isEvictable`, **re-reads + re-checks** each immediately before
the drop (guards a concurrent re-deploy), calls
`dropLocalTable(schema, table, indexNames)`, emits `onBasisTableEvicted`, and
deletes the record. A `dropLocalTable` throw leaves the record for the next sweep
(idempotent). No `dropLocalTable` wired (relay-only) ⇒ no-op.

**The reclaim seam.** `DropLocalTableCallback` (new) → store's
`reclaimDetachedTable(schema, table, indexNames)`, which evicts cached handles,
calls `provider.deleteTableStores(schema, table, indexNames)`, and removes the
catalog DDL — **no schema-change event** (the engine already saw the detach). The
captured-index-list approach was chosen over prefix-scanning (the store warns a
`{table}_idx_` prefix scan can clobber a sibling `{table}_idx_<x>` table).

**The knobs.** `SyncConfig.basisEviction: { mode, horizonMs? }` (default
`{ mode: 'horizon' }`). Per-table reserved tag `quereus.sync.evict =
never | immediate | <ms>` (new `SYNC_EVICT_TAG`, `'eviction-policy'` value schema,
sited view-ddl + physical-table like `quereus.sync.replicate`).

## Known gaps / reviewer focus (treat tests as a floor)

1. **The dynamic bump only fires while the table is in-basis-but-legacy**
   (`derivation-source-only` / `unreferenced`). Once `detached`, inbound writes to
   it are *out of basis* and get diverted by unknown-table detection — they never
   reach `appliedChanges`, so the clock is frozen at detach. This is correct under
   the conservative design (`quietSince` is anchored at the last in-basis foreign
   write or the unmap/detach time), but it is the subtlest part of the model —
   verify the reading holds for the directly-mapped→detached-in-one-deploy path and
   the never-directly-mapped (derivation-source-only→detached) path.

2. **The genuine between-list-and-drop race is NOT unit-tested.** The test
   (`re-deploy (re-map) before the sweep cancels eviction`) exercises the
   re-deploy-then-sweep *ordering*, which the first `isEvictable` check already
   catches; the `fresh.state !== 'detached'` re-check right before the drop is
   defensive code that the test does not force to diverge from the initial scan.
   Consider whether an interleaving test (or a comment justifying the gap) is
   warranted.

3. **`indexNames` orphan risk for pre-existing records.** A lifecycle record
   created by the prereq (before this ticket) has no `indexNames`; evicting it
   passes `[]`, so its index stores would be orphaned (data store + catalog still
   reclaimed). Same for a table first observed already-detached. "No backwards
   compat" (AGENTS.md) makes this acceptable, but confirm the boundary — a leaked
   index store is storage-only, never a correctness issue.

4. **Validation-vs-parse duplication.** The accepted shapes of `quereus.sync.evict`
   live twice: `validateEvictionPolicy` (engine `reserved-tags.ts`, structural) and
   `parseEvictPolicyTag` (sync `basis-lifecycle.ts`, semantic). The engine can't
   depend on sync, so this is deliberate; check they agree (both accept
   never/immediate case-insensitively + non-negative number-or-numeric-string).

5. **No production caller invokes the sweep yet.** `dropLocalTable` is wired in the
   worker, but nothing schedules `evictExpiredBasisTables` — identical status to
   `pruneTombstones`/`pruneQuarantine` (host-driven, no timer). Intentional, but
   note it: the feature is dormant until a host calls it.

6. **`sync-client` MockSyncManager is intentionally partial.** It `implements
   SyncManager` but omits `evictExpiredBasisTables` (and already omitted the
   prereq's `recordLensDeployment` / `getBasisTableLifecycle`). The package's tests
   run transpile-only (vitest, no test-file type-check), so this is invisible at
   runtime and the suite is green — but a `tsc` over those tests would flag the
   three missing methods. Pre-existing posture; left as-is.

7. **Worker `dropLocalTable` wiring is build-verified only** (quoomb-web builds
   clean) — no worker-harness integration test, mirroring the prereq's gap #1.
