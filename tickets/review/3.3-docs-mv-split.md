## What a reviewer should check first

### 1. Nothing was lost, and nothing quietly changed meaning

Two mechanical checks were run and both passed; re-run them if you touch anything.

- **Heading set.** Every `##`/`###`/`####` heading in the pre-split file appears in exactly one
  of the six files. `git show HEAD~1:docs/materialized-views.md | grep '^#'` versus
  `cat docs/materialized-views.md docs/mv-*.md | grep '^#'` differs only by the new titles,
  the seven stubs, `## Topic documents`, and three `## Rejected alternatives`.
- **Prose survival.** Every source line over 150 characters appears verbatim in one of the six
  files, except the 52 that were deliberately edited — link retargets, deleted history, and the
  passages listed under *Vocabulary sort* below.

**No duplicate heading slugs exist in any of the six files** (checked programmatically). The
checker's `-1` disambiguation therefore never fires, so no link resolves to the wrong heading.

### 3. The plan's inbound-link table was stale

`docs-optimizer-split-satellites` landed first (commit `52ab5b4b` / `b9db4c40`), so the four
links the plan attributed to `docs/optimizer.md` had already migrated to `optimizer-rules.md`
and `optimizer-fd.md`. The live set was re-enumerated by grep and every anchored reference was
repaired; `yarn docs:check` proves each resolves. **Link *labels* were repaired too** — eight
links still read `[materialized-views.md § X](mv-foo.md#x)` after the target moved.

### 4. Depth promotion is not uniform, and could not be

The ticket said "promote depth by one per satellite." That works for the three satellites built
from a **single** `##` section — `mv-backing-host.md`, `mv-ingestion.md`, `mv-schema-change.md`
— whose top heading became the `#` title. It cannot work for `mv-maintenance.md` and
`mv-constraints.md`, which each carry **two** `##` sections; promoting both would mint two `h1`s.
Those two got a new `#` title (`# Materialized-View Maintenance`,
`# Derived-Row Constraints and Covering Structures`) and kept their `##` sections in place.

Heading slugs are depth-independent, so **no anchor changed in either case**. Worth a look:
does `mv-constraints.md` read as one document or as two stapled together? Its opening paragraph
argues explicitly for why covering structures live there rather than in `lens.md`, per the
ticket's instruction.

### 5. Stubs make `docs:check` blind to a *missed* retarget

Every removed `##` left a two-to-three-sentence stub under its original heading, so
`materialized-views.md#maintenance-strategy`, `#covering-structures`, `#external-row-change-ingestion`,
`#schema-change-staleness`, `#derived-row-constraint-validation-…`,
`#maintenance-row-time-per-statement`, and `#backing-host-capability` **all still resolve** in
the overview. That is the ticket's stub rule working as designed, but it means the checker
cannot tell a deliberately-kept link from one that should have been retargeted and wasn't.
The retargeting was done by explicit anchor→file table, not by grep-and-hope, but this is the
one class of error a green `docs:check` does not rule out.

## Claims that looked wrong — flagged, not fixed

Per the ticket ("a wrong claim you find is a review finding, not a scope extension"):

- **`docs/architecture.md:118` contradicts `materialized-views.md`.** Architecture says a
  materialized view is *"Dual-registered (the backing `TableSchema` plus a
  `MaterializedViewSchema`)"*. The substrate section says the opposite, at length: *"realized as
  exactly **one** schema object — an ordinary `TableSchema` … One `TableSchema`, one catalog
  name, one physical incarnation."* `MaterializedViewSchema` does not appear anywhere in
  `docs/materialized-views.md`. One of the two is stale; almost certainly `architecture.md`.
  Left alone — it is outside this ticket's diff and deserves a check against `schema/derivation.ts`.

- **`packages/quereus/src/vtab/backing-host.ts:168` cited a section that never existed.** The
  comment read *"see `docs/materialized-views.md` § Row-time refresh"*. There is no
  `## Row-time refresh` heading in the pre-split file, nor anything close. It was retargeted to
  `mv-maintenance.md § Maintenance (row-time, per-statement)`, the section that actually
  describes the `MaintenanceOp` kinds the comment is annotating — a **guess at intent**, and the
  one comment retarget a reviewer should verify rather than trust.

- **`packages/quereus/src/schema/table.ts:700` cited "§ Enforcement through a row-time covering
  MV"**; the heading is `### Enforcement through a covering MV`. Corrected in passing (the file
  was already being retargeted).

- **`## Current limitations` disagreed with `## Maintenance strategy` about replicable
  determinism.** The limitations bullet described the replicable-determinism gate as unbuilt
  future work; the maintenance-strategy section documents it as wired and inert-by-default,
  firing only when a host declares `requiresReplicableDerivations`. **The bullet was rewritten**
  to say the gate exists but no host turns it on yet. This is the one place a claim's *meaning*
  changed, not just its wording — please verify against `selectMaintenanceStrategy`
  (`planner/cost/index.ts`) and the `requiresReplicableDerivations` capability flag before
  trusting it.
