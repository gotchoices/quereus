# Inline before-image on `RowDeletion` (`priorRow`) â€” IMPLEMENT handoff

The row-level companion to the per-cell before-image landed by
`sync-change-before-image-column` (now in `complete/`). An optional, purely-additive
**last-known row image** rides on a synced deletion and is persisted on the
tombstone, so a receiver can show or undo *what was removed* via `getChangesSince`
without reconstructing it beforehand.

```ts
interface RowDeletion {
  // â€¦type, schema, table, pk, hlcâ€¦
  readonly priorRow?: Row;   // row's last-known image at the origin (audit/undo)
}
interface Tombstone {
  hlc: HLC;
  createdAt: number;
  priorRow?: Row;            // persisted so it reaches the wire via getChangesSince
}
```

## What was built (matches the ticket design)

- **Type** (`protocol.ts`): `priorRow?: Row` on `RowDeletion`, documented as
  best-effort/additive, sourced from the engine `oldRow`.
- **Persistence** (`tombstones.ts`): `priorRow?: Row` on `Tombstone`.
  `serializeTombstone` keeps the **unchanged fixed 38-byte head** (30 HLC + 8
  `createdAt`) and, *only when a `priorRow` is present*, appends a JSON array of the
  row's values, each through the column-version `encodeSqlValue` helper so
  `Uint8Array`/`bigint`/`null` cells round-trip. `deserializeTombstone` is
  absent-tolerant (head-only buffer â†’ `priorRow` absent). `setTombstone` /
  `setTombstoneBatch` gained an optional trailing `priorRow?` param.
- **Source = engine `oldRow`, copied** (`recordDataEvent` delete branch): a
  `[...oldRow]` copy is set on both the persisted tombstone and the inline
  `RowDeletion`, so neither aliases the engine's row buffer. Absent when the event
  carried no `oldRow`.
- **Relay/extraction** (`resolveLogEntry`, `collectAllChanges`): conditional spread
  copies `tombstone.priorRow` onto the resolved `RowDeletion` only when present.
- **Apply** (`commitDeleteMetadata`): persists `change.priorRow` onto the tombstone,
  so a receiver relaying via `getChangesSince` re-emits it. Only the in-batch
  max-HLC winner's metadata is written, so the surviving `priorRow` is the latest
  delete's image.
- **Docs** (`docs/sync.md`): `RowDeletion.priorRow` field, a "Row before-image"
  prose section (source/persistence/best-effort/storage-cost), and the
  storage-key table row (`tb:` value now `{hlc, createdAt, priorRow?}`).

## Use cases for the reviewer to probe (adversarial targets)

- **Absent vs. present discipline.** Confirm no path leaks a phantom `priorRow:
  undefined` key onto a wire `RowDeletion` or a stored `Tombstone`. The gate is
  conditional-spread on `priorRow !== undefined` everywhere; the empty-array test
  pins the present/absent boundary.
- **Serialization edge cases.** The trailing JSON is appended after a fixed 38-byte
  head; `deserializeTombstone` keys on `buffer.byteLength > 38`. Probe: a row whose
  JSON happens to be tiny, wide rows, and the `getBigUint64(30)` read on a
  >38-byte buffer (DataView spans the whole buffer â€” verified safe).
- **Latest-image-wins under key reuse.** deleteâ†’reinsertâ†’delete and in-batch dedup
  both must surface the latest/winning image â€” adversarially construct orderings
  (e.g. an out-of-order in-batch pair) and confirm `keepMaxHLC` + the tombstone
  dedup hold.
- **`pruneExpired` / `getAllTombstones`** now deserialize longer buffers; confirm
  they still behave (they read hlc/createdAt and ignore/echo `priorRow`).

## Known gaps & honest caveats (please weigh these)

- **JSON wire transport drops `priorRow` (and the column `priorValue`/`priorHlc`).**
  `packages/quereus-sync-client/src/serialization.ts` and
  `packages/sync-coordinator/src/common/serialization.ts` â€” the two JSON transport
  (de)serializers used by the coordinator-mediated path â€” serialize only
  `type/schema/table/pk/hlc` (+ `column/value` for column changes). They were **not**
  extended here and silently omit every before-image field. So in a real
  browserâ†”coordinatorâ†”browser deployment the before-image survives **in-process and
  via `getChangesSince` re-resolution** (which all the e2e tests exercise) but does
  **not** cross the JSON boundary. This is the *same* pre-existing gap the column
  before-image has â€” the prior ticket's review deliberately did not close it. I left
  it alone to avoid scope creep, but it materially limits the audit/undo feature over
  the wire. **Recommend the reviewer decide whether to file one backlog ticket
  covering before-image transport coverage for both column and row before-images.**
- **Scope extension beyond the listed files: `quarantine.ts` + `quarantine.spec.ts`.**
  The ticket did not list quarantine, but its module contract promises the raw wire
  `Change` is "stored verbatim â€¦ full fidelity," and the prior ticket's review added
  the column before-image there for exactly that reason. For consistency I extended
  the quarantine (de)serializer to round-trip a delete's `priorRow` (new `pr?` field,
  same conditional-spread/absent-tolerant pattern). Low-risk and additive, but it is
  outside the original file list â€” flag for the reviewer.
- **LevelDB store round-trip unverified.** `yarn test:store` is slow / not
  agent-runnable in-budget. The tombstone format change is opaque bytes to the store
  and is exercised via `InMemoryKVStore`, so the LevelDB path should be unaffected â€”
  left for CI / a human out-of-band.
- **Storage cost not gated.** Every tombstone may now hold a full row image until
  retention-horizon GC (`pruneTombstones`). Bounded but non-trivial for wide rows.
  Per the ticket, the opt-out config flag is intentionally **not** added here. Not
  filed; reviewer may file a `backlog/` note if a deployment would want to gate it.
- **`yarn lint` targets `@quereus/quereus` only** (no eslint config exists for
  `quereus-sync`, which was the only package touched). The strict src+test
  typechecks above are the real type gate for this work; `yarn lint` was not run
  because it would not exercise any changed code.
