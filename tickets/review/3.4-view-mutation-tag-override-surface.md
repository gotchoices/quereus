## What shipped

The reserved `quereus.update.*` tags now have **Effect** (their shape/site validation already
landed in `schema/reserved-tags.ts`). Flow:

1. **Statement-level tag separation ÔÇö chose option (b), nearly free.** The decision in the
   implement ticket was (a) filter reserved keys out of `contextValues`, vs (b) a distinct
   statement-level tags clause. The parser **already** parsed `WITH TAGS (...)` on DML via
   `parseTrailingWithClauses` ÔÇö it just dropped the result. So (b) was the lower-risk path *and*
   keeps tags cleanly separate from mutation-context (no risk of conflating a `quereus.*` context
   var). Implemented by: adding `tags?` to `InsertStmt`/`UpdateStmt`/`DeleteStmt` (ast.ts),
   capturing the previously-dropped `WITH TAGS`, and ÔÇö the load-bearing fix ÔÇö extending the
   **pre-clause** `WITH` handlers (pre-source for insert, pre-SET for update, pre-WHERE for delete)
   to accept `TAGS` in addition to `CONTEXT`, so the doc syntax `delete from v with tags (...)
   where ...` parses (previously only trailing `WITH TAGS` worked, and `with tags ... where` was a
   parse error). ast-stringify emits the tags clause (trailing) for round-trip; verified stable.

2. **Collect / validate / merge** ÔÇö `planner/mutation/mutation-tags.ts` (new). `collectMutationTags`
   validates `view.tags` at `'view-ddl'` and `stmt.tags` at `'dml-stmt'` via the registry's
   `validateReservedTags`, raises the first `severity:'error'` diagnostic as a **sited**
   `QuereusError` (statement line/column; view-DDL errors prefixed with the view name), logs
   warnings, and merges `{ ...viewTags, ...stmtTags }` (statement wins). Called from
   `view-mutation-builder.ts` *before* any base op is built (atomic). Typed readers
   (`readPolicy`/`readDeleteVia`/`readTargetNames`/`readExcludeNames`/`readDefaultFor`) wrap
   `getReservedTag`/`getReservedTagByTemplate` ÔÇö **no re-parsing of the namespace**.

3. **Consumption** ÔÇö merged tags ride `MutationRequest.tags`:
   - `default_for.<col>` (single-source insert, `single-source.ts`): supplies an omitted-insert
     value ahead of the base column default. Slots into the same VALUES-append path as the
     constant-FD fill; resolves `<col>` as a base column first, then a view column with `base`
     lineage; unknown Ôćĺ `tag-target-not-found`. Value is a TEXT **expression** parsed via the new
     `parseExpressionString` helper (parser/index.ts).
   - `target` / `exclude` (multi-source, `multi-source.ts`): narrow the candidate base sides by
     base-table name/alias; unknown name Ôćĺ `tag-target-not-found`; removing all sides Ôćĺ
     `tag-conflict`. On UPDATE they act as a guard (an assignment to an excluded side Ôćĺ
     `tag-conflict`, never a silent drop). On DELETE they narrow the candidate set.
   - `delete_via` (multi-source delete): `'parent'` Ôćĺ FK-parent side (needs a provable FK),
     `'left_delete'` Ôćĺ left join source. `'right_insert'` is an except-only value Ôćĺ rejected for
     joins with a pointer to the join-valid forms.
   - `policy` (view-DDL): `strict` rejects any unresolved multi-side delete (won't even fall back
     to the FK-many heuristic) with `policy-strict-ambiguity`; `lenient` (default) keeps the
     FK-many fallback.

New diagnostic reasons: `tag-conflict`, `policy-strict-ambiguity` (mutation-diagnostic.ts).

## Use cases to validate (all in `test/logic/93.4-view-mutation.sqllogic`)

- **default_for, statement-level**: `insert into df_v with tags ("quereus.update.default_for.created" = '777') values (5,'x')` fills the projected-away base column `created`.
- **default_for, view-level + merge precedence**: a view declaring `default_for.created='111'` fills it; a statement-level `default_for.created='222'` overrides for that statement.
- **target / exclude narrow a multi-source delete**: `target='tg_x'` deletes only the named side; `exclude='tg_x'` routes the same delete to the other side.
- **delete_via='left_delete'** picks the left source; **delete_via='parent'** picks the FK-parent (with `on delete cascade`, removing the child too).
- **policy=strict** (view-level tag): the FK-join delete that lenient auto-routes to the child is *rejected* under strict; an explicit `target` resolves it.
- **Sited diagnostics**: unknown reserved key on a DML statement; mis-sited key on a DML statement (`policy` is view-DDL-only); unknown reserved key declared on the view DDL (surfaces at mutation); `target` naming a non-join table; `default_for` naming a non-column; UPDATE `exclude` excluding an assigned side.

Suggested adversarial angles for the reviewer (treat the tests as a floor):
- `default_for` on an `insert ... select` view (non-VALUES source) ÔÇö hits `unsupported-source`
  (same constraint as constant-FD fill); **untested** ÔÇö confirm the diagnostic is acceptable.
- statement-level `default_for` whose column the insert *also* supplies explicitly through the
  view ÔÇö `isSupplied` should make the explicit value win; **untested directly**.
- `target` + `delete_via` together where they conflict (delete_via picks an excluded side) Ôćĺ
  `tag-conflict`; **untested**.
- a single-source view carrying a `target`/`delete_via` tag ÔÇö intentionally a **no-op** (narrowing
  a one-element base set), and an unknown target name there is **not** existence-checked. Confirm
  that's the intended contract (it is per the ticket).

## Known gaps / deliberate deviations (honest handoff ÔÇö these are starting points, not done)

1. **Lenient multi-side delete fan-out is deferred.** The doc's maximal lenient reading ÔÇö an
   ambiguous join delete deletes *from every side* ("make this row not exist") ÔÇö is **not**
   implemented. Each side's delete addresses rows via a subquery over the join, and the
   view-mutation substrate runs base ops **sequentially against live state** (see
   `runtime/emit/view-mutation.ts`), so deleting one side empties the join before the second side's
   subquery runs ÔÇö both orderings lose rows. A correct fan-out needs snapshot-consistent base-op
   execution or eager key materialization. **Consequence:** lenient currently *rejects* an
   unresolved ambiguous delete (FK-many fallback or `delete-ambiguous`), and `policy=strict`'s
   observable difference from lenient is that strict rejects *even the FK-resolvable* case. This is
   a defensible reading of "reject any ambiguity" but differs from the doc's "lenient fans out"
   framing. Documented in `docs/view-updateability.md` ┬ž Inner Join (Shipped vs. intent) and the
   Status table. **Reviewer: decide whether this warrants a follow-up fix/backlog ticket for the
   snapshot-consistent fan-out.**
2. **`delete_via` join mapping is asymmetric.** `'left_delete'` picks the left side; there is no
   enum value to pick the *right* side directly (use `target` for that). `'right_insert'` is
   rejected for joins. Faithful to the enum's except-branch origin, but worth a sanity check.
3. **View tags are validated at mutation time, not at CREATE VIEW.** A view created with a
   malformed reserved tag succeeds; *every* mutation through it then errors (re-validated each
   time). Arguably validation should also run at create. Out of scope here; candidate backlog item.
4. **Multi-source insert + `default_for`** never interact: multi-source insert is rejected
   (`unsupported-multisource-insert`) before `default_for` would apply, and the outer-join
   null-extend `default_for` path (doc ┬ž Outer Joins) is not implemented (outer joins rejected).
5. **`default_for` requires a VALUES source** (inherits the constant-FD-fill constraint).
