# Review: index canonical body — per-column collation

Follow-on to `schema-differ-index-body-drift`, which deliberately EXCLUDED
collation from the index canonical body (the declared side carried only as-written
`COLLATE` while the actual side stored a resolved collation, so a naive include
churned inherited/BINARY collations). This ticket includes collation **correctly**
by resolving the declared side's effective collation the same way the engine does
at create/import time, then rendering both sides identically (BINARY elided,
non-BINARY explicit).

## What landed

Three changes funnel both diff sides through one renderer fed an already-resolved,
normalized collation:

1. **Renderer** (`ast-stringify.ts`). `canonicalIndexedColumnsToString` now renders
   each named column as `name [collate <c>] [desc]` — same order as the persistence
   renderer `indexedColumnsToString`. New helper `canonicalIndexColumnCollation(col)`
   returns the effective collation only when non-BINARY (case-insensitive compare;
   `BINARY`/absent both elide). It reads the plain `IndexedColumn.collation` (which
   both callers pre-resolve) with a **defensive fallback** to the parser's
   collate-folded form (`col.expr.collation`) so a future raw caller never silently
   drops collation. `indexedColumnBareName` is now `export`ed (the differ needs it to
   resolve the declared column's table-column collation). A genuine expression-index
   column (no resolvable name) still falls through to `expressionToString` unchanged.
   Docs on `createIndexBodyToCanonicalString` / `canonicalIndexedColumnsToString`
   reversed (was "collation excluded").

2. **Actual side** (`ddl-generator.ts`, `indexToCanonicalDDL`). Each lifted
   `IndexedColumn.collation` is set from `indexSchema.columns[i].collation` (fallback
   to `tableSchema.columns[col.index].collation`), `normalizeCollationName(... ?? 'BINARY')`.

3. **Declared side** (`schema-differ.ts`). New `declaredIndexCanonicalBody(indexStmt,
   declaredTable)` builds a normalized plain-form column list, resolving each column's
   effective collation as
   `normalizeCollationName(explicitIndexColumnCollation(col) || declaredColumnCollation(declaredTable, bareName) || 'BINARY')`,
   then calls the renderer. `explicitIndexColumnCollation` covers both the plain
   (`col.collation`) and collate-folded (`col.expr.collation`) forms;
   `declaredColumnCollation` reuses the existing `extractDeclaredCollation` to read the
   table column's `collate` constraint (BINARY default). The index loop looks up
   `declaredTables` by `indexStmt.table.name.toLowerCase()`; an index on a non-declared
   table passes `undefined` → falls back to explicit-or-BINARY.

### New regression cases (under "declarative differ stability" describe, all via `diffIndexEdit`)

- inherited BINARY, re-declared verbatim → no churn (the original common case).
- explicit index `collate nocase` added → `indexesToDrop = ['ix']`, recreate matches `/collate nocase/i`.
- inherited NOCASE (`email text collate nocase`, no explicit index COLLATE) re-declared verbatim → no churn (guards declared side reads the table-column collation).
- column collation change (`name text` → `name text collate nocase`) under a stable-named index → index drops+recreates AND `columnsToAlter` carries the `SET COLLATE NOCASE`.
- explicit `collate binary` on a BINARY column → no churn (normalization equates binary/BINARY/absent).
- composite: all collations unchanged → no churn; one column COLLATE added → recreate.
- desc index inheriting NOCASE, re-declared verbatim → no churn (name→collate→desc order symmetry).
- `policy: 'require-hint'`: a pure collation body-change recreate does NOT throw (it is a recreate, not an unhinted rename — already covered by `indexBodyRecreates` bookkeeping, no code change needed).

Also retitled the pre-existing "inherited-NOCASE unique index … (collation excluded
from body)" case — its rationale is now "both sides resolve the same collation",
and it still passes.

## ⚠️ KNOWN GAP — apply ordering (please scrutinize / consider a fix ticket)

The ticket asked to verify that apply ordering runs the column `SET COLLATE` **before**
a collation-driven index recreate, and to flag it if not. **It does not.** The ticket's
parenthetical assumption ("column alters before index creates") is **inverted** in the
actual code:

- `generateMigrationDDL` (schema-differ.ts ~1490) pushes `diff.indexesToCreate` in the
  **"Create new items"** block, which runs **before** the **"Alter existing tables"**
  loop that emits `ALTER COLUMN … SET COLLATE` (~1524). So for a "column collation
  change drives index recreate" migration the emitted order is:
  `DROP INDEX ix` → `CREATE INDEX ix ON t(name)` (reads the **old** column collation —
  the recreate DDL has no explicit COLLATE, it inherits) → `ALTER COLUMN name SET COLLATE nocase`.
- The store backend's `SET COLLATE` (`packages/quereus-store/.../store-module.ts` ~1089)
  updates the column collation and re-validates non-PK UNIQUE constraints but **does NOT
  re-key dependent secondary indexes**. So the recreated index ends up keyed under the
  OLD collation while the column lands on the NEW collation — a stale-key mismatch at
  apply time, specifically on the store backend.
- This was not previously reachable: before this ticket a column-collation-only change
  never recreated the index at all. This ticket newly makes the differ emit that
  recreate, so the ordering now matters.

**Why the test suite is green anyway:** all new tests assert the diff *decision*
(`indexesToDrop` / `indexesToCreate` / `columnsToAlter`), which is correct. None apply
the migration end-to-end for the collation case, and the suite runs on the memory vtab
(whose `SET COLLATE` re-key behavior I did not verify). So this gap is **untested at the
apply/execution level**.

Suggested fixes (a reviewer call — likely a separate fix ticket, since reordering the
global migration sequence has its own blast radius):
- (a) Reorder so a collation-driven index recreate's `CREATE INDEX` runs **after** the
  column `SET COLLATE`; or
- (b) Have the collation-driven recreate emit an explicit `COLLATE <resolved>` so it is
  order-independent (note: `createIndexToString` is also the persistence path, so this
  would need to be scoped to the recreate, not the shared emitter).

## Other notes for the reviewer

- The renderer's defensive folded-form read (`col.expr.collation`) is currently
  unreached in production (both callers pre-resolve to plain form). It is intentional
  insurance; confirm you're comfortable keeping it vs. asserting plain-form-only.
- `declaredColumnCollation` returns `'BINARY'` for an index on a non-declared table.
  That scenario isn't a coherent name-match (the index would be a pure create/drop, not
  a body compare), so it never feeds a real comparison — but worth a sanity check.
- Cross-check the `IndexColumnSchema.collation` is always set in practice (both
  `buildIndexSchema` and `importIndex` always normalize-and-set it); the `?? table
  column collation` fallback in `indexToCanonicalDDL` is defensive only.
