## What changed

The isolation layer merges a per-connection overlay against an underlying table. To interleave
two sorted streams it must know the underlying scan's sort order (primary-key order, or
`(secondaryIndexKey…, pk…)`). It used to learn that by **parsing `FilterInfo.idxStr`** with two
regexes; any string it didn't recognise silently fell back to a primary-key merge — corrupting
row order when the scan was really a secondary index.

The prereq (`iso-index-descriptor-engine-seam`, now in `complete/`) added
`FilterInfo.accessPath`: a typed, validated `AccessPath` union the planner stamps at plan time.
This ticket makes the isolation layer read that and deletes the string parsing.

Concretely in `packages/quereus-isolation/src/isolated-table.ts`:

- **`parseIndexFromFilterInfo` → `resolveScanIndex`.** Switches on `filterInfo.accessPath.kind`:
  - `fullScan` / `empty` → merge by primary key.
  - `index` with `role: 'primary'` → primary-key merge **regardless of the index's name** (an
    alias like `_primary_1` still counts as the PK).
  - `index` with `role: 'secondary'` → `(indexKey…, pk…)` merge using the descriptor's full
    `keyColumns`.
  - `unresolvedIndex` → **throw `QuereusError(INTERNAL)`** naming the index. No silent fallback:
    the layer is choosing a *comparator* for a stream the underlying already emits in index
    order, so downgrading to a PK comparator would be exactly the corruption being removed.
  - `accessPath` absent → **throw `QuereusError(INTERNAL)`** (only reachable on the merged/dirty
    path; a clean no-overlay read takes the fast path and never inspects it).
- Deleted `PK_INDEX_NAME_RE`, `SUFFIXED_PK_IDXSTR_RE`, `getIndexColumnIndices`, and the long
  comment block that justified the regex heuristics.
- **`adaptFilterInfoForOverlay`** rewritten on the engine's `retargetFilterInfoIndex`: when the
  path is the primary key under a non-canonical alias, retarget the FilterInfo's index name to
  `_primary_` (the only PK name the overlay `MemoryTable` knows) — replacing the old hand-rolled
  idxStr string surgery.

`filter-info.ts` was already collapsed onto the engine builders by the prereq; nothing further
needed there. All internal FilterInfo construction in `packages/quereus-isolation/src` already
routes through `makeFullScanFilterInfo` / `makePkPointLookupFilter`, both of which populate
`accessPath`.

`docs/design-isolation-layer.md` gained a "scan-order source of truth" subsection under **Key
Ordering** (the `accessPath` → merge-order table, the full-scan contract, and the INTERNAL error
a module earns by aliasing an index without a descriptor).

## Tests added (floor, not ceiling)

All in `packages/quereus-isolation/test/isolation-layer.spec.ts`:

1. **Secondary-index merge order across a tombstone revival with a changed index key** — stages
   an overlay, then revives a tombstoned PK at a *new* index value; asserts the raw emitted order
   is `(indexKey, pk)` and that the overlay's index key (not the shadowed underlying value)
   governs position, with a surviving underlying row interleaved.
2. **`_primary_extra`** — an index literally named like the PK family merges as a *secondary*
   index (structural via schema lookup, no longer a regex special-case).
3. **Aliased PK with `indexDescriptor`** — the existing "suffixed primary-key index name" describe
   was retargeted: `SuffixedPkMemoryModule` now supplies a `role:'primary'` descriptor named
   `_primary_1`, so its point-lookup / range-scan / secondary-index tests keep passing and now
   assert the *with-descriptor* contract.
4. **Aliased PK without `indexDescriptor`** (`NoDescriptorAliasedPkModule`) — clean read OK, dirty
   read throws `INTERNAL` naming `_primary_1`.
5. **No `accessPath`** — hand-built FilterInfo: clean read OK, dirty read throws `INTERNAL`.
6. **Empty-plan** — `accessPath: { kind: 'empty' }` resolves to a primary-key merge (no throw).
7. **`ANALYZE` inside an open transaction** over a dirty overlay succeeds (analyze hand-builds a
   full-scan FilterInfo via `makeFullScanFilterInfo`, which carries `accessPath`).
8. **Two-table commit** still flushes both overlays.

The pre-existing `fullScan()` / `fullScanFilter()` spec helpers were updated to populate
`accessPath` (they now drive the merged path, which would otherwise throw).

## Known gaps / where a reviewer should push

- **`empty`→primary is defensive, not exercised in production.** `EmptyResultNode`'s emitter
  yields nothing and never calls `query()`, so `resolveScanIndex` never sees `{ kind: 'empty' }`
  via a real runtime path. Test 6 constructs it directly (idxStr `null` + `accessPath` `empty`),
  which is a slightly artificial pairing. The branch exists so an impossible predicate can't
  surface overlay rows if that ever changes — confirm you agree it should stay a mapped case
  rather than another throw.
- **End-to-end secondary-index routing through the SQL planner is only indirectly covered.** The
  secondary-merge tests force the path with a hand-built `secondaryScanFilter`; the only
  planner-driven secondary coverage is the existing `idx_email` test. An added end-to-end SQL
  test (`create index … ; begin ; …write… ; select … order by <indexed col>`) that asserts merged
  ordering would raise the floor — if the planner picks a PK-scan-plus-sort instead, results stay
  correct but the secondary merge path isn't exercised, so such a test needs care to actually hit it.
- **Behavior change for aliasing modules (contract, but worth a callout).** Any real underlying
  module (e.g. lamina/store) that aliases an index name and does *not* return an `indexDescriptor`
  will now throw `INTERNAL` on a dirty merged read where it previously (mis)merged. That is the
  intended tightening, but it is a live behavior change — worth confirming the store/lamina path
  supplies descriptors. This run validated the memory-overlay path only (`yarn test`); the
  store-backed path (`yarn test:store`) was not run here.
- **DESC secondary-index direction.** `IndexKeyColumn.desc` is now available on the descriptor but
  `buildCompareSortKey` still folds direction in only via `underlyingTable.getIndexComparator`,
  falling back to a direction-blind `compareSqlValues`. Left untouched deliberately (a separate
  pre-existing defect the plan/engine-seam ticket already flags); noted here so it isn't
  mistaken for regression introduced by this change.
