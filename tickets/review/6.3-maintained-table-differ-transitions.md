# Review: differ recognition of attach / detach / re-attach

Part 3 of the maintained-table chain. A `declare schema` transition on a named
maintained table now applies as a **catalog flip + content reconcile**, never a
drop+recreate (which would mint a new incarnation, fatal to a replicated table's
row metadata).

## What landed

**Catalog unification (`catalog.ts`).** A maintained table (a `TableSchema`
carrying a `derivation`) now round-trips in the `tables` category like any other
table, carrying a `maintained?: { bodyHash, backingModuleName?, backingModuleArgs? }`
descriptor and rendering its DDL via `generateMaintainedTableDDL`. The
`CatalogMaterializedView` type and `SchemaCatalog.materializedViews` array were
**removed** (no other module referenced them; the equivalence harness reads MVs
from `schemaManager.getMaintainedTable`, not the catalog snapshot). Its internal
covering indexes stay out of `catalog.indexes`.

**Differ (`schema-differ.ts`).**
- `declare materialized view` items **normalize at diff time** into the table
  category (`materializedViewToDeclaredTable` → a declared table with the body in
  a `maintained` clause and NO declared columns). The standalone MV comparison
  loop and `materializedViewsToCreate/Drop` + `materializedViewTagsChanges` buckets
  are gone. The original `declaredMaterializedViews` map is retained only to render
  the CREATE branch as `create materialized view` sugar (a column-less `create
  table … maintained as` has no parseable DDL).
- `computeTableAlterDiff` gains the derivation dimension via
  `applyMaintainedTransition`: attach (`setMaintained`), detach (`dropMaintained`),
  re-attach (body-hash drift → `setMaintained`, plus a `dropMaintained` reshape leg
  when a concurrent declared-shape change is detected). A declared maintained table
  with NO authored columns (MV-sugar) **suppresses** column/constraint/PK
  comparison — the differ stays plan-free; a genuine shape mismatch surfaces at
  apply's attach shape check (sited).
- `generateMigrationDDL` ordering: `drop maintained` runs EARLY (where MV drops
  ran), `set maintained as` runs LATE (after table creates/alters). The flip's
  detach-v2 / attach-v1 pair falls out naturally.
- A maintained table's **tag-only** change routes through `ALTER MATERIALIZED VIEW
  … SET TAGS` (flagged by `maintainedTags` when the table is still maintained at
  tag time) — the runtime ALTER TABLE handler rejects tag actions on a maintained
  table, and the TABLE verb fires the wrong change event for store re-persistence.
- **Orphan** maintained table → `drop table` (structural parity).

## Honest gaps / decisions (do not assume these are solid)

1. **Backing-module change on a maintained table is a silent NO-OP** (was a
   drop+recreate while MVs had their own bucket). `CatalogTable` carries no module
   field, so the differ — like the plain-table path — does not detect a `using
   <module>(...)` change. The ticket scoped an in-place module move as destructive
   and out of scope. Pinned by two "documented gap" tests. If silent-ignore is
   unacceptable, this needs a follow-up (surface the module and emit drop+create).

2. **Differ-side idempotence workaround.** The re-attach VERB records
   `derivation.columns` explicitly while CREATE records the implicit form, so a
   re-attached sugar MV's bodyHash would diverge. `maintainedBodyMatches` papers
   over this by accepting either the as-authored OR the live-column-names hash for
   an implicit body. This is a band-aid; the proper fix is verb-side. Filed:
   `tickets/backlog/maintained-table-reattach-columns-parity-and-reshape.md`.
   Review the workaround for false-negatives (could it mask a real change?). I
   restricted the live-names variant to the implicit form so an explicit
   rename-list change is NOT masked — verify that reasoning.

3. **Shape-changing body edits on a sugar MV error at apply.** Renaming an output
   column (`select id, x` → `select id, y`) or changing the `(a,b)` rename list is
   emitted as a re-attach the verb cannot apply (strict shape check, no rename
   syntax) → sited error. The plan-free differ can't detect the shape change to
   reshape instead. Documented + pinned (`a column-list (rename) change … the verb
   cannot apply`). Same backlog ticket. Shape-PRESERVING changes (value/source/
   filter, or aliasing) work.

4. **Existing MV declarative tests were re-pointed**, not just deleted — several
   changed semantics (drop+recreate → re-attach; module/args change → no-op).
   Confirm the rewrites still assert something meaningful and didn't lose coverage
   the reviewer cares about (e.g. the old module-migration behavior is gone).

5. The "sugar vs table-form compare equal" matrix case was dropped (uncertain
   `declare schema { table X { … } maintained as … }` syntax). Form-parity is
   asserted at the hash level by the idempotence path, but a direct equivalence
   test would be stronger.
