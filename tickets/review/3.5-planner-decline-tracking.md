description: The query optimizer used to re-run rules that had already looked at an unchanged piece of the plan and declined to act, wasting planning time; now a rule that declined isn't offered the same unchanged node again, but is re-offered the moment that node actually changes.

## What shipped

A single, surgical change in `PassManager.applyPassRules`
(`packages/quereus/src/planner/framework/pass.ts:557-613`): an **ephemeral
per-node decline set**.

Inside the `while (changed)` fixpoint loop, a rule that declines (returns `null`
or the same node) on the *current node id* is added to a local
`declinedOnCurrent: Set<ruleId>` and skipped on later iterations — a declining
rule is a pure function of its input node, so re-running it on the same
unchanged node is wasted work (the case the ~1300-line filter/query-rewrite
matcher hit).

The set is **reset to empty every time a rule transforms the node** (mints a new
id). The plan piece changed, so every decliner gets a fresh shot on the new node.

Applied rules are unchanged: still tracked in `context.visitedRules`
(`markRuleApplied` / `hasRuleBeenApplied`) and inherited across a transform's
re-mint by `inheritVisitedRules` for loop prevention. Declines are **not**
inherited and **not** stored on the context.

Net effect: a strict speedup with **zero plan-output change** — only redundant
same-node re-runs are cut, never a legitimate re-offer after the node changes.

## IMPORTANT: this deviates from the ticket body — read why

The implement ticket's body proposed **inheriting declines across the transform
re-mint** ("each rule gets at most one shot per node lineage per pass"). That
approach was implemented first and **fails a real test**:

- `test/optimizer/parallel-async-gather-zip-by-key.spec.ts` →
  *"folds a derived scalar expression over the merged key"* went red.
- Cause: `ruleAsyncGatherZipByKey` (a `Project`-matching rule in the
  PostOptimization pass) declined earlier, and an earlier-pass rule that reshapes
  the `Project` **after a sibling rule transforms it** got suppressed by the
  inherited decline — leaving a two-`Project` shape the gather can't fold
  (it aborts with *"a projection references a branch key outside a full-group
  coalesce"*). The plan silently lost its `AsyncGather`.

This is exactly the soundness hazard the ticket flagged ("a changed plan here is
a signal, not a nit to silence") and it proves the optimizer's rule set is **not
decline-monotonic**: a rule that declines on shape A can legitimately apply to
shape A′ after a *sibling* rule rewrites A→A′.

Crucially, the ticket **description** already says the right thing: *"not get
re-offered **unless the plan piece actually changed**."* The ephemeral approach
implements that literally; the body's inheritance idea over-reached past it. So
this is not a fallback — it's the correct reading of the ticket's own intent. The
larger "skip the expensive matcher after any sibling transform" win the body
imagined is **not soundly achievable** and was not forced (no escalation to
`blocked/` needed: the safe, description-faithful path was available and taken).

## Known gaps / things a reviewer should probe

- **The win is ordering-dependent and modest.** Decline dedup only cuts a re-run
  when a decliner sits *after* a transforming rule in registration order **and**
  the fixpoint loop re-iterates. When the transform is the last matching rule, or
  when a decliner sits before it, the decliner still re-runs on the new node
  (correctly). No attempt was made to reorder rules to maximize the win.
- **No perf/bench numbers.** The improvement is validated structurally (call-count
  tests), not timed. `bench/` was not run. If someone wants a number, a
  representative multi-filter/multi-join query with the optimizer rule trace
  (`DEBUG=quereus:optimizer:trace:rules`) counting `rule.fn` invocations before/
  after would quantify it. The trace hooks in `framework/trace.ts`
  (`onRuleStart`/`onRuleEnd`) are still **not wired into `applyPassRules`** — a
  future counter could reuse them instead of an ad-hoc instrument.
- **Residual behavior narrowing (tripwire, recorded as a `NOTE:` at
  `pass.ts:604`).** A rule that declines but mutates shared `context` state,
  expecting to re-apply on the *same unchanged* node on the next iteration, would
  no longer get that second look. No such rule exists today (full suite green);
  the old code re-ran every decliner each iteration, so this is a genuine (very
  low-risk) narrowing. If such a rule is ever added, exclude it from the skip or
  key the skip on a context epoch.
