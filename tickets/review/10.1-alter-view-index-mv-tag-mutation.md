# Review: ALTER VIEW / ALTER MATERIALIZED VIEW / ALTER INDEX … SET TAGS

## What landed

The three remaining tagged catalog objects (views, materialized views, indexes) had **no `ALTER`
statement at all** — the top-level `ALTER` dispatch only accepted `TABLE`. This ticket adds the
missing in-place `SET TAGS` primitive for all three, following the whole-set-replacement model the
prereq (`alter-table-tag-mutation`) established for table/column/constraint tags.

New SQL (v1 scope is **`SET TAGS` only** — no other alter verb):

```sql
alter view v               set tags (cacheable = true);
alter materialized view mv set tags (owner = 'analytics');
alter index idx            set tags (purpose = 'search');
alter index idx            set tags ();   -- clear
```

Pipeline: parser dispatch (`alterStatement`) branches TABLE / MATERIALIZED VIEW / VIEW / INDEX →
new AST nodes → one generic `SetObjectTagsNode` (carries `{objectKind, schemaName, name, tags}`) →
`emitSetObjectTags` delegates to a SchemaManager setter. Reserved-tag validation runs at
**plan-build time** in the builder (matching the prereq's `buildAlterTableStmt`, NOT the emitter as
the ticket prose suggested) — `view-ddl` site for view/MV, `physical-index` for index.

SchemaManager setters (catalog-only, empty clears, `NOTFOUND` on missing):
- `setViewTags` / `setMaterializedViewTags` — re-register the in-memory schema object, **no change
  event** (mirrors the no-event view create path; MV deliberately avoids re-firing
  `materialized_view_added`, which would mislead listeners into re-registering maintenance — there
  is no `materialized_view_modified` event in the union).
- `setIndexTags` — resolves the owning table by index name, swaps the `IndexSchema`, fires
  `table_modified` (mirrors create/drop index). Hidden implicit covering indexes → `NOTFOUND`.

Declarative differ: new `viewTagsChanges` / `materializedViewTagsChanges` / `indexTagsChanges`
buckets, drift detected on a **name-matched** object (pure name match only — a rename-matched
view/index has no in-place primitive and rides drop+recreate). `generateMigrationDDL` emits the
three `SET TAGS` forms. **MV body-change and tag-change are mutually exclusive** per object: a body
change drops+recreates (recreate carries the tags); a tag-only change takes the in-place path and
does **not** re-materialize.

## Known gaps / things to probe (be adversarial here)

1. **Behavioral `quereus.update.*` view tags + no cache invalidation.** `setViewTags` fires no
   change event (faithful to the create path). But a view's `quereus.update.*` tags drive
   write-through routing, computed at plan time. An `ALTER VIEW … SET TAGS` that adds/changes/drops
   those tags will **not** invalidate an already-cached prepared-statement plan referencing the
   view. Documented in docs/sql.md, but **not behaviorally tested** — the highest-value gap to probe:
   does write-through actually re-route after `ALTER VIEW SET TAGS` on an updatable view, or does a
   cached plan serve stale routing? If it must invalidate, a view-modified event/notifier is needed.
2. **Store-backed persistence not verified.** Like the prereq's `setTableTags`, these bypass
   `module.alterTable`, so a tag-only change on a store-backed view/index is not re-persisted across
   reconnect (same gap as backlog `tag-mutation-store-persistence`). `yarn test:store` was **not run**.
3. **MV tags have no SQL introspection.** `schema()` does not enumerate materialized views, so MV
   tags are only observable via `db.schemaManager.getMaterializedView().tags` (used in the specs),
   not from SQL. Pre-existing; out of scope, but a reviewer may want a `materialized_view_info` TVF.
4. **Hidden-implicit-index branch coverage.** The `NOTFOUND` test passes, but in the memory module
   the implicit covering index is built in the `MemoryTableManager` layer and may not be present in
   the catalog `tableSchema.indexes` at all — so the `NOTFOUND` could come from the "no matching
   index" path rather than the `isHiddenImplicitIndex` guard. The observable contract holds either
   way; the guard is belt-and-suspenders for catalogs where the implicit index IS present (e.g.
   `quereus.expose_implicit_index = true`, or store-backed). Probing the **exposed-implicit** case
   would exercise the guard directly.
5. **Plain-view body drift is still undetected by the differ (pre-existing).** Only tags were added
   to the view loop; a name-matched view with a changed body produces no recreate (unchanged
   behavior). Out of scope, but worth confirming the reviewer agrees it's pre-existing.
6. **Non-`main` schema execution.** The builder defaults schema to `main` and honors `name.schema`;
   parser test covers parsing `main.v`, but a real non-main qualified `ALTER VIEW`/`INDEX`/`MV`
   execution path is not exercised end-to-end.
