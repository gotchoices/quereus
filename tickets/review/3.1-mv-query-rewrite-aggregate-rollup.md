## What landed

The headline MV-rewrite payoff: a `group by g₁,…,gₖ agg(…)` query answered from a **grouped** MV, instead of re-scanning + re-aggregating the base.

```sql
create materialized view daily as
  select d, sum(amt) as total, count(*) as cnt from sales group by d;

select d, sum(amt) from sales group by d;   -- exact-key → scan _mv_daily, residual Project (no re-aggregation)
select sum(amt) from sales;                  -- rollup    → scan _mv_daily, re-aggregate sum(total) into one group
```

Implemented as a **second arm** of the existing `ruleMaterializedViewRewrite`, registered on the `Aggregate` node type (the projection-filter arm is on `Project`):

- **Matcher** (`query-rewrite-matcher.ts`): `analyzeAggregateFragment` recognizes a logical `Aggregate(Filter?(scan(T)))` with bare-column group keys and `count(*)` / `f(bare-col)` aggregates; `matchAggregateFragmentToMv` classifies the MV (must be a single-source grouped aggregate), maps both group keys to bare source columns, and decides exact-key vs rollup.
  - **Exact-key** (query key == MV key): backing rows are the answer — passthrough `outputColumnMap` + optional residual (reuses the foundation's `buildReplacement`). Any aggregate exactly matching a stored one is admitted (incl. `count(distinct)` / `group_concat`).
  - **Rollup** (query key ⊊ MV key, incl. the empty global key): re-aggregate the stored partials. Allowlist + recombine recipes: `sum`→`sum`, `count`→`coalesce(sum,0)`, `min`/`max`→`min`/`max`, `avg`→`sum(sum)/sum(count)`. Everything else (incl. any `distinct`) ⇒ forgo.
- **Rule** (`rule-materialized-view-rewrite.ts`): dispatches `Project` → projection-filter arm, `Aggregate` → aggregate arm; `buildRollupReplacement` emits backing scan → optional residual Filter → a fresh logical `Aggregate` (recombine aggregates) → a `Project` recombining/preserving the fragment's output attribute ids. Aggregate cost gate (`exactAggCost` / `rollupAggCost` vs `recomputeAggCost`, grouping-factor backing-cardinality discount).
- **Registration** (`optimizer.ts`): the aggregate arm is a 2nd handle (`materialized-view-rewrite-aggregate`) that honors the canonical `materialized-view-rewrite` disable id, so one `disabledRules` entry (and every existing equivalence-harness / pristine-fragment control) turns off both arms.

## Soundness contract (the load-bearing dimension)

Identical to the foundation: **a false NotMatch only forgoes a speedup; a false Match returns wrong rows** — every check forgoes on doubt. Key witnesses the reviewer should scrutinize:

- **One-row-per-MV-group** (`backingPkIsGroupKey`): the backing PK must equal the MV group key (the schema-level form of `proveEffectiveKeyUnique`). This is what makes the exact-key direct scan return one row per query group and the rollup re-aggregate a *set*, not a bag. It also transparently catches MVs whose body was group-key-simplified at create time (backing PK ≠ declared group key ⇒ forgo).
- **Residual confined to group keys**: `backingColOfBaseCol` exposes only the stored group-key columns, so a residual on a non-group column ⇒ `missing-column`. A group-key residual partitions whole groups, commuting with the rollup.
- **`avg` NULL/zero correctness**: recombine is `sum(sum_x) / sum(count_x)`. The count must exclude the same NULLs `avg` does — a stored `count(x)` always qualifies; `count(*)` only when `x` is `not null` (verify this gate). Quereus `/` is **real division** (`3/2 = 1.5`, confirmed empirically), so this matches the native `avg`; over zero rows it is NULL/NULL = NULL.
- **`count`-over-zero-rows = 0**: the `coalesce(sum(cnt), 0)` wrapper is load-bearing — a bare `sum(cnt)` over an empty backing (the global `select count(*)` over an empty table) yields NULL, but `count(*)` must be 0. The number-literal `0` (not bigint) matches the native count's numeric 0.

### Suggested adversarial probes for the reviewer

- **`avg` representation under scale**: the harness uses small ints. For huge sums/counts the recombine `sum(…)` may promote to bigint while native `avg`/`count` stay JS `number` — probe whether the JSON-equal comparison hides an int-vs-bigint divergence past `2^53` (likely a non-issue for real data, but unverified).
- **`min`/`max` types**: the recombine `min`/`max` schemas default to REAL; values are correct but the *type label* may differ from the native aggregate. The harness compares values only — confirm no downstream consumer depends on the column type here.
- **Multiple covering MVs** (one finer, one exact): the cost gate should pick exact over rollup; confirm the tiebreak/cost ordering is what you expect (only single-MV cost paths are unit-tested).
- **Top-Project-over-Aggregate** (reordering / computed-over-aggregate selects, e.g. `select sum(amt)+1, d … group by d`): the rule fires on the inner `Aggregate` and preserves its attribute ids, leaving the parent Project intact — lightly exercised, worth a direct row check.
- **HAVING**: a `having` filter stays as the parent of the rewritten aggregate (the rule never absorbs it). Confirm `select d, sum(amt) from t group by d having sum(amt) > k` stays row-identical.

## Known gaps / deliberate limitations (honest list)

1. **Rollup + residual filter is forgone** (`fail('rollup-residual')`) to work around a **pre-existing** base-engine correctness bug: `WHERE pk_col = const` is silently dropped under `GROUP BY` on a composite-PK relation (reproduces with no MV — see `tickets/.pre-existing-error.md` §2). This is the biggest functional limitation; once the base bug is fixed, the forgo can be relaxed and the harness extended with rollup-plus-residual shapes. **Reviewer: confirm the forgo is sound (it is conservative) and that the underlying bug is genuinely pre-existing, not introduced here.**
2. **Multi-key query pinning a group column is forgone** (`fail('group-key-pinned')`): when ≥2 group keys and a `WHERE g = const` / `g is null` / `g1 = g2`, the base's `rule-groupby-fd-simplification` drops the determined group column and re-emits it as a picker `min` at a *shifted* output position, changing the result's column ORDER. The rewrite preserves the pristine order, so it forgoes. Range / `IN` residuals are unaffected. (This is a column-ORDER divergence, not a value divergence — the guard is conservative; a reviewer may judge whether it over-forgoes.)
3. **Exact-key `avg` requires a stored `avg`** (passthrough). The `avg`-from-sum/count recombine is only wired for the rollup arm, mirroring the ticket's framing. A user querying `select k, avg(x) … group by k` against an exact-key MV storing only `sum`+`count` will fall back to base. Easy to extend (per-row division) if desired.
4. **Cost gate is heuristic**: memory tables expose no row counts, so backing cardinality uses a grouping-factor discount (`mvGroups = baseRows × min(0.8, max(0.1, groupCount×0.2))`). It reliably picks the MV for grouped MVs but is not statistics-grounded; a highly selective base predicate is not modeled.
5. **Out of scope (parked)**: rollup over a *join* MV body (interacts with 3.2 join subsumption); `count(distinct)` / HLL-style approximate re-aggregation.
