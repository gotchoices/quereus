## What landed

The compound grammar now accepts a parenthesized query expression as a `query-term`
operand on **both** legs (previously only the right leg accepted a leading `(`, and it
rejected the inner's own ORDER BY/LIMIT). All three compound sites were unified onto one
operand reader + one left-associative tail parser. No new AST node — pure grouping unwraps
to the inner `QueryExpr`; a grouped left operand is carried as the existing
`select * from (<inner>) as values_<offset>` subquery wrapper (the same device VALUES
already used).

### Parser changes (`parser.ts`)

New private helpers (all in the SELECT region, ~line 675–900):
- `checkCompoundOperator()` — peek for `union/intersect/except/diff`.
- `parseCompoundOperator()` — decode the just-matched operator keyword (+ `all`) and the
  `exists <branch> as <name>` membership clause(s). Extracted from the two copy-pasted decodes.
- `parseCompoundOperand(withClause, requireReturning)` — reads ONE operand: a parenthesized
  `( <query-expr> )` via full `parseQueryExpr` recursion, or a bare SELECT/VALUES/DML leg.
- `wrapAsSubquerySelect(inner, startToken)` — generalized VALUES wrapper.
- `parseCompoundTail(left, withClause, startToken)` — the left-associative loop (the crux).
- `parseTrailingOrderLimit(sel, isCompoundSubquery)` — extracted ORDER BY/LIMIT/OFFSET tail.

`selectStatement` and `continueSelectAfterFrom` were rewritten to build a core `sel` then
delegate to `parseCompoundTail` + `parseTrailingOrderLimit`. The `usedParen` blocks and the
duplicated operator-decode / order-limit code are gone. Entry points `parseQueryExpr` and
`statement()` gained a leading-`(` branch.

### The associativity crux (verify this carefully)

`parseCompoundTail` keeps **unparenthesized chains byte-identical** to before: the first
**bare** operand greedily consumes the rest of the chain (its own `selectStatement` recursion)
and we `break` → today's right-leaning `A{union, B{union, C}}`. A **parenthesized** operand
iterates the loop (left-associative), wrapping the accumulator when it can't host the new
compound. The wrap condition is `acc.type !== 'select' || acc.compound`.

> **Deviation from the ticket pseudocode (intentional — please confirm):** the ticket's
> inline pseudocode wrapped only in the paren branch. That is buggy for the ticket's own
> documented outcome `(A) union (B) union C` → `SS_from(A∪B){union, C}`: when the bare `C`
> arrives, the accumulator `A{union,B}` already has a full compound slot and MUST be wrapped.
> I unified the wrap rule across both branches (paren and bare), which satisfies **every**
> documented outcome in the ticket. Trace table I validated by hand and via tests:
> - `A union B union C` → `A{union, B{union, C}}` (right-leaning, unchanged)
> - `(A) union (B) union (C)` → `SS_from(A∪B){union, C}` (left wrapper, one node)
> - `(A union B) union (C union D)` → `SS_from(A∪B){union, C{union, D}}` → builds
>   `SetOp(union, SetOp(A,B), SetOp(C,D))`
> - `(A) union B union C` → `A{union, B∪C}` (bare B greedily eats C)
> - `(A) union (B) union C` → `SS_from(A∪B){union, C}`

> **Signature deviation:** `parseCompoundTail` does NOT take `isCompoundSubquery` (the ticket
> listed it). It is genuinely unused there — order/limit suppression lives entirely in the
> separate `parseTrailingOrderLimit`, and the compound tail is always consumed regardless.

## Known gaps / things to probe (treat my work as a starting point)

1. **Round-trip NON-idempotence for `A union (select 2 order by 1 limit 1)`.** The inner
   order/limit binds inside the parens at parse (improvement, pinned by a parse-shape test),
   but `ast-stringify` emits the leg WITHOUT grouping parens, so re-parsing rebinds order/limit
   to the OUTER compound. So `parse→stringify→parse` is NOT stable for this one shape. The
   ticket explicitly required only a parse-shape test here (not a round-trip) and said
   "stringify — no change required", so I left stringify alone. **A reviewer-grade follow-up**:
   teach the compound stringifier to wrap `compound.select` in parens when the leg carries its
   own `orderBy`/`limit`. I did not, to avoid perturbing the many existing compound round-trips;
   flagging it as the one round-trip hole. (All other documented cases ARE idempotent because a
   wrapped left re-emits `select * from (...)` and a redundant-paren simple compound re-emits
   the bare form — both stable.)

2. **Parenthesized FIRST operand reachability.** Works at: top-level statement, view body, CTE
   body, and set-op legs (both sides). It is **not** reachable when the inner content of a
   FROM-subquery or an IN/EXISTS/scalar-subquery position STARTS with `(` — those positions gate
   on `tableSource`'s post-`(` lookahead and `checkSubqueryStart()`, neither of which includes
   `LPAREN`. I deliberately did NOT extend them: adding `LPAREN` there would mis-route
   value-lists (`x in ((1),(2))`) and parenthesized joins (`from (a join b)`) into the
   query-expr path. Keyword-led inner content (`from (select 1 union select 2) as t`,
   `x in (select 1 union select 2)`) is unaffected, and re-parsed wrapper forms always have a
   keyword-led FROM subquery, so round-trips are safe. **Confirm** this scoping is acceptable —
   if `from ((A) union (B))` is wanted, it needs a separate, carefully-disambiguated change.

3. **Membership WRITE through a wrapped nested branch is NOT enabled.** This ticket only enables
   the parse + read plan for `(A∪B) union exists … (C∪D)`. Making the write branches
   (`leftBranchSelect`/`rightBranchSelect`, `planner/mutation/set-op.ts`) writable when they are
   a `select * from (compound)` is the backlog ticket `set-op-membership-nested`. Existing
   single-level membership write tests are untouched (full suite green). Worth a reviewer pass:
   confirm no write path silently accepts the wrapped shape and then misbehaves (the static
   surface should report non-writable for it — spot-check `view_info`/`column_info` on a nested
   membership view).

4. **`loc` precision.** `selectStatement` no longer tracks `lastConsumedToken` granularly; it
   stamps `_createLoc(start, this.previous())` after parsing. Spans are equivalent in the common
   cases but a reviewer wanting exact error-column fidelity on compound tails may want to check a
   few error messages point sensibly (the negative tests confirm errors fire with sane messages).

## Out of scope (do not chase here)

  AST shapes and must re-validate the set-op membership write path).
- `set-op-membership-nested` (writability of wrapped nested branches).
