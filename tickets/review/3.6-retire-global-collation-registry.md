# Review: retire the process-global collation registry

## What landed

The module-level `Map<string, CollationFunction>` in `util/comparison.ts`, and the
three functions that read it — `registerCollation`, `getCollation`,
`resolveCollation` — are deleted. So are the `createLogger`/`warnLog` bindings that
existed only to serve them. Collations now live in exactly one place: on a
`Database`, registered with `db.registerCollation(name, func, options?)` and resolved
through `db.getCollationResolver()`. `builtinCollationResolver` remains as the
switch-based BINARY/NOCASE/RTRIM floor for code that has no `Database` in scope.

Signature changes that turn a would-be silent bug into a compile error:

- `compareSqlValues(a, b, collationName?)` → `compareSqlValues(a, b)`. It is BINARY,
  full stop. Passing a name is now a type error rather than a lookup against a dead map.
- `compareWithOrderBy` and `createOrderByComparator` are **deleted**. `find_references`
  showed no production callers — only tests and the `index.ts` re-export. Their `*Fast`
  siblings (which take a pre-resolved `CollationFunction`) are what emitters actually use.

`compareRows(a, b)` keeps its BINARY-only behavior and now carries a `NOTE:` saying so.

Public surface (`src/index.ts`): dropped the three registry functions,
`compareWithOrderBy`, and `createOrderByComparator`; added `normalizeCollationName`,
`createTypedRowComparator`, and `createCollationRowComparator` (the last was used
internally and documented in `docs/runtime.md` but never actually exported).

Docs updated: `packages/quereus/README.md` (plugin-development bullet),
`docs/plugins.md` (both the import list and the collation-management API block),
`docs/runtime.md` (two emitter examples that imported the global `resolveCollation`;
they now use `ctx.resolveCollation`), `docs/architecture.md` (property-test description).

## The thing a reviewer should look at first

**Prereq 3.5 left a straggler in production code.** `runtime/emit/materialized-view-helpers.ts`
still called the three-argument `compareSqlValues(ra[i], rb[i], c.collation ?? 'BINARY')`
inside `assertDerivedRowsAreSet` — the "a maintained table must be a set" gate that
runs on `create table … maintained as` (attach) and on the constraint-bearing arm of
`refresh materialized view`. Removing the overload surfaced it.

It now takes a `CollationResolver` parameter, threaded from `db.getCollationResolver()`
at both call sites, and resolves the per-column key collations once above the sort via
`resolveCollationFunctions`.

Two behavior changes fall out of that, and both are intentional:

1. A database that overrides `NOCASE` or `RTRIM` now decides derived-key identity in
   this gate. Previously the gate always compared under the *built-in* function.
2. An unregistered collation name now raises `no such collation sequence: <name>`
   instead of silently falling back to BINARY. Reachable only if a caller manufactures
   a shape whose column collation was never registered; DDL validates column collations
   at create time, so this should be unreachable in practice — worth a skeptical look.

## Tests

`yarn build` (all packages), `yarn lint`, `yarn test` — all clean. Quereus suite:
**6643 passing, 0 failing, 9 pending** (was 6641 before; +2 new).

New: `test/mv-custom-collation-maintenance.spec.ts` gains
`describe('the maintained-table "must be a set" gate resolves collations through the database')`
with two regression tests, both of which fail against the pre-change code:

- attach admits a body whose derived keys are distinct under an overridden `NOCASE`
- a constraint-bearing `refresh` does the same

Both use a `NOCASE` override that folds *nothing* (plain byte comparison), so the
source primary key admits `'aa'` and `'AA'`. Under the old global lookup the gate
resolved the real, case-folding `NOCASE`, saw one key where the source saw two, and
rejected a set that is a set.

`test/exports.spec.ts` now also asserts the **negative** surface: `registerCollation`,
`getCollation`, and `resolveCollation` are `undefined` on the package namespace.

Retargeted rather than deleted: the `compareWithOrderBy` block in
`test/utility-edge-cases.spec.ts` is now `describe('createOrderByComparatorFast')`,
driving the same assertions through a local helper, plus one new case proving text
compares through the supplied collation function. `test/property.spec.ts`'s collation
consistency property resolves its comparator from `db.getCollationResolver()` — the
same connection the query runs on — instead of the global map.

## Known gaps — please probe these

- **The set gate's *rejecting* branch is not covered under a custom collation, and I
  believe it is unreachable from SQL.** Attach refuses a body whose column collation
  differs from the declared one (`body derives collation BINARY but the table declares
  NOCASE`), so the derived key always compares under the source column's own collation
  — and the source table's primary key already deduplicated under that same collation.
  I wrote, ran, and then removed a test that tried to construct a collision; every route
  was blocked earlier in the pipeline. The reject branch stays covered by the existing
  engine-wide duplicate-key parity block (integer keys). If you can construct a
  collation-sensitive collision that reaches the gate, that is a real hole in my
  reasoning and the test should exist.
- **`yarn test:store` was deferred** — it exceeds the 10-minute agent budget. The store
  suite has a `using store(...) args: PK key collation` case exercising a NOCASE-PK set
  gate; it should be run out-of-band before this is considered done. Nothing in the diff
  is store-specific, but the set gate is shared.
- The `resolveCollationFunctions` call in `assertDerivedRowsAreSet` runs once per
  attach/refresh, above the sort — but I did not benchmark it. It replaced a per-comparison
  `resolveCollation` call, so it should be strictly faster.
- I did not touch `planner/building/schema-resolution.ts`'s identically-named
  `resolveCollation`. It is a different, database-scoped, build-time function that throws
  on an unknown name, and it stays. Confirm no confusion crept in.
