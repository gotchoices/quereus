# REVIEW: ALTER TABLE DROP/RENAME CONSTRAINT + declarative constraint lifecycle

## What landed

Two new ALTER TABLE primitives plus the declarative constraint lifecycle (add / drop / rename **by name**) for named CHECK / UNIQUE / FOREIGN KEY constraints.

### Primitives
```sql
alter table t drop constraint uq_email;
alter table t rename constraint uq_email to uq_user_email;
```
- **AST/parser**: `DROP CONSTRAINT <name>` branches off the `DROP [COLUMN]` arm; `RENAME CONSTRAINT <old> TO <new>` off the `RENAME` arm. Stringify arms added; both round-trip (property test extended).
- **Plan/emit**: handled as lightweight `AlterTableNode` arms (mirroring `dropColumn`/`renameColumn`), NOT a separate node. `runDropConstraint` / `runRenameConstraint` in `runtime/emit/alter-table.ts` resolve the class up front (clear NOTFOUND / ambiguous errors), reject derived-from-index UNIQUEs, then route through `module.alterTable` so persistent modules re-persist DDL.
- **Class resolution** is shared: `resolveNamedConstraintClass(tableSchema, name)` in `schema/table.ts` (check → unique → fk; NOTFOUND / ambiguous). `SchemaManager.setConstraintTags` was refactored onto it (DRY).
- **Module layer**: `SchemaChangeInfo` gains `dropConstraint` / `renameConstraint`. Memory module delegates to new `MemoryTableManager` methods that also **tear down / rename the implicit covering index** a named inline UNIQUE auto-builds (named after the constraint). Store module does a pure schema rewrite + `saveTableDDL` (inline UNIQUE in store is full-scan enforced — no index store to touch).

### Declarative differ
- **Fixed a real latent bug**: `computeTableAlterDiff` had always populated `constraintsToRename`, but `generateMigrationDDL` never read it — a declared constraint rename was silently dropped. It now emits `ALTER TABLE … RENAME CONSTRAINT`. (Regression test added.)
- **New** `TableAlterDiff.constraintsToDrop` / `constraintsToAdd`: detect drops (catalog-present, declaration-absent) and adds (declaration-present, catalog-absent), gathering declared named constraints from **both** table-level and column-level (named) constraints.
- **Catalog**: `namedConstraints` now surfaces only *user-addressable* names — engine auto-names (`_check_*` / `_fk_*` / `_uc_*`, via `isAutoConstraintName` = leading `_`) and `derivedFromIndex` UNIQUEs are excluded. This keeps the diff idempotent for unnamed and index-derived constraints.
- Migration order: RENAME CONSTRAINT → DROP CONSTRAINT → ALTER PRIMARY KEY → ADD CONSTRAINT (after column adds / PK change). `require-hint` extended to constraint add/drop.

## Known gaps / things a reviewer should probe

1. **ADD CONSTRAINT for UNIQUE / FOREIGN KEY is NOT module-supported (pre-existing).** The differ *emits* `ALTER TABLE … ADD <fragment>` for any declared-absent named constraint, but the memory/store modules' `addConstraint` still throws `UNSUPPORTED` for non-CHECK (unchanged by this ticket; `test/alter-add-constraint.spec.ts` still pins that). So a **declarative add of a UNIQUE/FK to an already-existing table fails at apply**. CHECK add works end-to-end. This is documented (docs/sql.md §2.7, docs/schema.md, and the `constraintsToAdd` field comment). Idempotent re-apply and fresh CREATE TABLE are unaffected (the constraint arrives via CREATE TABLE / matches by name). **Implication for the follow-on** `alter-constraint-body-change-drop-add`: its drop+add reuse works for CHECK but a UNIQUE/FK body change would hit this same wall — that ticket likely needs to add module `addConstraint` support for UNIQUE/FK first. Decide whether ADD-emit-that-fails is the right UX or whether the differ should hard-error earlier for non-CHECK adds.

2. **CHECK drop ↔ optimizer FDs / EmptyRelation folding — only behaviorally tested.** The sqllogic confirms a previously-rejected row inserts after the CHECK drop (so enforcement is gone, and `table_modified` fires to invalidate caches). I did **not** add a test that a query which folded to `EmptyRelation` via a CHECK-derived FD (e.g. `check (x = 1)` then `select … where x = 2`) stops folding after `DROP CONSTRAINT`. Worth an adversarial probe — the cache-invalidation path is the same `table_modified` the other alters use, but it's unverified here for the constraint-drop case specifically.

3. **`_`-prefix auto-name heuristic.** Constraint lifecycle deliberately ignores names starting with `_` (engine-synthesized). A user who *explicitly* names a constraint `_foo` forfeits declarative add/drop/rename of it. Documented; confirm this corner is acceptable.

4. **Ambiguity test depends on the engine allowing duplicate constraint names at CREATE TABLE** (it does — no uniqueness check on constraint names). If a future change adds duplicate-name validation, the ambiguity sqllogic case (`constraint dup check … , constraint dup unique …`) would need rework.

5. **RENAME of a UNIQUE backing index uses a full `rebuildAllSecondaryIndexes`** in the memory manager (correctness over surgery — the index data is rebuilt from the post-rename schema). Fine for rare ALTERs; flagged in case a reviewer wants a surgical key-rename instead.

6. **FK drop with `pragma foreign_keys = false`** is not explicitly exercised (the sqllogic sets it `true`). Low risk (FK enforcement reads `foreignKeys`; drop removes the entry), but unverified.

## Out of scope (correctly deferred)
- Constraint **body** drift (changed CHECK expr / FK action / UNIQUE column set) → follow-on `alter-constraint-body-change-drop-add` (reuses the `constraintsToDrop` / `constraintsToAdd` buckets established here).
- Implementing module `addConstraint` for UNIQUE/FK (see gap #1).
