## What landed

A fourth row-time maintenance arm, `'join-residual'`, for **1:1 row-preserving inner/cross
joins** (`select … from T join P on T.fk = P.id`, exactly one MV row per governed T row). It
reuses the residual kernel the aggregate arm (`3.2`) built and adds the lookup-side mechanism.

### `database-materialized-views.ts`
- **`JoinResidualPlan`** (new `MaintenancePlan` union arm) + **`ForwardResidualPlan`** (a
  structural interface naming the fields the forward residual apply path reads, satisfied by
  both `ResidualRecomputePlan` and `JoinResidualPlan`).
- **`applyResidualRecompute` → renamed `applyForwardResidual`** and re-typed to
  `ForwardResidualPlan`; it now drives **both** the aggregate `'residual-recompute'` arm and the
  join arm's **driving (T)** path unchanged (a T change = a size-1 group: delete old backing
  slice keyed on T.pk → run the T-keyed residual → upsert). `residualRowMatchesKey` re-typed
  likewise.
- **`buildJoinResidualPlan`** — admits the join: exactly 2 base refs; no aggregate-over-join;
  **no WHERE** (deferred); proves 1:1 on the driving table via the shared `proveOneToOneJoin`
  (tries each base ref as the candidate driver, so the diagnostic is the prover's own — fanout
  vs shape); restricts to **inner/cross** (an outer join would make the reverse residual unsound
  — filtering P drops null-extended rows); verifies the backing PK is *exactly* T's PK (catches
  a `keysOf` all-columns fallback); deterministic-projection check; compiles a **forward**
  (T.pk-keyed) and a **reverse** (P.pk-keyed) residual via the existing `compileResidual`.
- **`applyJoinResidual`** dispatches on which source changed (`changedBase`): T → forward
  (`applyForwardResidual`); P → **`applyLookupResidual`**.
- **`applyLookupResidual`** — the lookup-side mechanism: per affected P key (OLD ∪ NEW), run
  the reverse residual (`… where P.pk = :pk0`) against live state and **upsert** every joined
  row it returns (each carries its T.pk backing key). **No delete** — sound because for an
  inner/cross join + enforced RI + no lookup-referencing WHERE, the set `{T : T.fk = P.pk}` is
  fixed by `T.fk` (a T column a P write can't change), so a P change only refreshes the
  lookup-projected columns of existing backing rows.
- **Dual-base registration**: `planSourceBases(plan)` returns `[T]` for single-source arms and
  `[T, P]` for the join arm; `registerMaterializedView` / `releaseRowTime` index under all of
  them, so a write to **either** T or P fires maintenance.
- **`changedBase` threaded** through `maintainRowTime` → `applyMaintenancePlan` → `applyJoinResidual`.
- **`buildMaintenancePlan` routing**: the blanket `tableRefs.length > 1` reject is now
  `length > 1 && !containsAnyJoin` (a non-join multi-source body — e.g. a 2-table UNION — keeps
  its source-count diagnostic); a join body routes to `buildJoinResidualPlan` (after the TVF arm,
  which is matched first since a lateral TVF also surfaces a join). Helper `rootRelationalNode`.

### `coverage-prover.ts` (the requested refactor)
- Extracted the join-soundness descent into exported **`walkToConstrainedBase`** (the
  no-row-loss walk: row-preservation OR `innerJoinRetainsConstrainedTable`), exported
  **`proveJoinNoFanout`**, and added **`proveOneToOneJoin`** (= the walk + the fan-out gate).
  `proveCoverage` now calls `walkToConstrainedBase` (its inline fan-out gate kept in place so
  its diagnostic ordering is byte-for-byte unchanged); the MV gate calls `proveOneToOneJoin`.
  Net: the join 1:1 logic lives in one place, consumed by both the base-table coverage prover
  and the MV gate.

### Tests + docs
- `maintenance-equivalence.spec.ts`: new **1:1 inner-join suite** (`t join p on t.fk = p.id`,
  NOT-NULL FK) — property test with random mutations on **both** t and p (forward + reverse
  paths), multiset equivalence in-txn and after rollback; an explicit **removed-join-row**
  (delete-a-t-row) case; an explicit **lookup-side name-update** case. Added `execTolerant`
  helper (CONSTRAINT-tolerant exec), shared with the existing `applyMutation`.
- `53-materialized-views-rowtime.sqllogic`: § 7 `bad_join` flipped to a **fanning, non-FK**
  join asserting the `is not a provably 1:1 row-preserving join` (shape) tail; new **§ 24**
  accepts a 1:1 FK join with insert / FK-update / lookup-name-update / delete / key-change /
  txn-rollback / no-op-lookup-insert reads-own-writes.
- `materialized-views.md` + `incremental-maintenance.md`: join moved out of the deferred list;
  the `'row'`/`'pk'` join binding, the forward+reverse residual mechanism, and the lookup-side
  soundness argument documented; arm count 3 → 4.

## Review focus / known gaps & risks
1. **Build/type-check is UNVERIFIED.** Highest priority. Suspect spots: the `let tRef` /
   `topJoin` narrowing-after-`reject(): never` in `buildJoinResidualPlan`; the `RelationalPlanNode`
   type import addition; `ForwardResidualPlan` structural compatibility with `ResidualRecomputePlan`;
   the `changedBase` param added to `applyMaintenancePlan` (ensure no caller was missed).
2. **Lookup-side upsert-only soundness** is the crux — verify the argument holds for every P
   mutation (insert / payload-update / PK-update / delete) under RI, and that it is genuinely
   gated (no WHERE, inner/cross only). A WHERE referencing P, or an outer join, would break it;
   both are rejected at build — confirm the rejections actually fire (the sqllogic only covers
   the no-WHERE accept + a non-FK reject; consider adding reject cases for an outer 1:1 join and
   a `where p.col …` join if you want them locked).
3. **Diagnostic ordering** in `buildMaintenancePlan`: a join now routes *before* the
   single-source aggregate/covering arms; confirm `bad_union` (set-op) still reports "reads more
   than one source table" and no other `53-…`/`materialized-view-diagnostics.spec.ts` assertion
   regressed (I could not grep for callers of the old "joins are not supported" / "contains a
   join" messages — please verify none assert them for a join body).
4. **Within-statement convergence** (last-write-wins against live state) for mixed T+P writes,
   and the **MV-over-MV cascade** when a join MV's backing has dependents (the join arm returns
   effective `BackingRowChange[]` from both paths — confirm the cascade still works; no test
   covers a join-MV producer).
5. **`coverage-prover` refactor**: confirm the existing coverage-prover unit/sqllogic tests stay
   green (the descent was extracted verbatim; the fan-out gate stayed in `proveCoverage`).
6. **Out of scope (deferred, documented):** fanning (non-1:1) keyed joins, outer 1:1 joins,
   partial-WHERE join bodies, >2-source joins, computed-column join projections, and
   covering-MV UNIQUE enforcement through a join MV (`lookupCoveringConflicts` returns `[]` for
   `'join-residual'`).
