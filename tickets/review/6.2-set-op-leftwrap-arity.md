## What landed

**Root cause (confirmed empirically).** The parser lifts a parenthesized LEFT compound operand
`(A∪B)` into a passthrough wrapper `select * from (A∪B) as values_N` so the SELECT-level `compound`
slot can host the outer operator (`wrapAsSubquerySelect` / `parseCompoundTail`). `buildCompoundSelect`
built that wrapper as a `select *` `ProjectNode` over the inner `SetOperationNode`, so the outer node's
left child was NOT a `SetOperationNode`. `dataArity(left)` then took the `else` branch
(`left.getType().columns.length`) and counted the inner's **surfaced flag columns** as data columns
(2 data + 2 flags = 4), while the RIGHT subtree recursed correctly to 2 → the constructor's
`leftData !== rightData` check threw. A flagged parallel-sibling view could not even be planned.

**Fix (one locus, build-time AST unwrap).** New shared pure predicate
`unwrapPassthroughSubquery(sel)` in `planner/util/set-op-wrapper.ts` returns the inner query-expr of a
*pure* `select * from (<query-expr>) as <alias>` wrapper (exactly one unqualified `*`, exactly one
unaliased `subquerySource` FROM, and **none** of where / groupBy / having / distinct / all / orderBy /
limit / offset / own compound / schemaPath / withClause), else `undefined`. It is recursive/idempotent
(peels nested pure wrappers; stops at a wrapper that carries its own `compound`). `select-compound.ts`
applies it (via a local `unwrapToSelect` that only redirects to a **SELECT** inner — a VALUES/DML inner
stays the wrapper, since its data arity is already its column count) to `leftCore` AND to a
`select`-typed right operand before `buildSelectStmt`, so a parenthesized compound operand on either
side builds as a **direct `SetOperationNode`** subtree operand.

**`set-operation-node.ts` needed NO change** — once the operands build directly, the existing
`dataArity` / `flagCount` / `buildAttributes` recursion (`nestable-flagged-set-ops`) produces the sum
surface `[data] ++ [L flags] ++ [R flags] ++ [own flags]` correctly. (The file is in `files:` only
because the reviewer should confirm the no-change claim, not because it was edited.)

## Honest gaps / things for the reviewer to probe

- **Read/plan only — writes are 6.3 (`set-op-leftwrap-write`).** Verified: a write through a P2-shaped
  view is now **reachable** (the view can be created) and is **cleanly rejected** (not a crash, not
  silent corruption) with `cannot write through view 'P2': the left branch uses 'select *'` — because
  the write path's `leftBranchSelect` (`planner/mutation/set-op.ts`) does NOT yet unwrap the wrapper, so
  it sees the synthetic `select *`. The message is slightly misleading (the user never wrote `select *`).
  6.3 should reuse `unwrapPassthroughSubquery` in `leftBranchSelect` to fix this. Reviewer: confirm the
  rejection stays clean and consider whether the diagnostic should be improved now vs. in 6.3.
- **Parser limitation (pre-existing, not this ticket).** A top-level statement that *starts* with `(`
  does not consume a trailing `order by` (the LPAREN-led `parseQueryExpr` path skips
  `parseTrailingOrderLimit`). The read tests work around it by reading through a created view with the
  ORDER BY on the outer select (the idiomatic membership-view pattern). Flag if you think the parser gap
  warrants its own ticket — it is orthogonal to arity.
- **Product model is explicitly out of scope** — reused flag names → merged coordinate columns is the
  blocked `set-op-product-coordinate-model`. This ticket delivers only the **sum** surface.
- **Right-operand unwrap is defensive.** The parser does not currently emit a *pure* passthrough wrapper
  as a right operand (parenthesized right operands are pure grouping → the inner compound directly;
  wrappers-with-compound are rejected by the predicate's compound guard), so the right-side unwrap is a
  no-op on all current inputs. It is included for symmetry and 6.3 reuse. Reviewer: confirm this is
  acceptable dead-defensive code or worth a comment trim.
- **`dataArity` is checked at SetOperationNode *construction* (build time)**, before optimization, so an
  optimizer-inserted node between the outer set-op and its operands cannot retrigger the throw. Reads go
  through the full pipeline and pass, but the reviewer may want to spot-check that no rule re-wraps a
  left set-op operand in a way that changes `getType().columns.length` at emit time.
