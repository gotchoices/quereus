# Residual-recompute arm (single-source aggregates) — review handoff

## What landed

The `'residual-recompute'` arm of `MaintenancePlan` (previously a loud `INTERNAL`
stub) is now **wired for single-source aggregate bodies**. A
`create materialized view mv as select g1,…, agg(…) from T [where P] group by g1,…`
over **bare** group-by columns is admitted at create and maintained row-time:

1. **Admit** (`buildMaintenancePlan` → `buildAggregateResidualPlan`): the former blanket
   aggregate rejection is replaced by a soundness-checked admit. Requires a non-empty
   GROUP BY of bare source columns, a deterministic body, and a backing PK equal to the
   group key (every backing-PK column must resolve — transitively through any
   Project-over-Aggregate — to a GROUP BY source column).
2. **Compile** (`compileResidual`): the body is rewritten with
   `injectKeyFilter(analyzed, T, groupColumns, 'gk')` (the shared primitive), optimized,
   emitted, and the `Scheduler` cached on the plan.
3. **Apply** (`applyResidualRecompute`): derive the affected group key(s) from the
   changed row (OLD ∪ NEW, deduped on the backing key); for each — `delete-key` the old
   backing slice, run the residual bound to the key through a fresh `RuntimeContext` on
   the live `db` (reads-own-writes via `T`'s registered transaction connection), and
   `upsert` the recomputed group row. Reuses the existing `applyMaintenanceToLayer` +
   `getBackingConnection` + MV-over-MV cascade path unchanged.

This is the synchronous, in-transaction analogue of the assertion evaluator's residual
path (`database-assertions.ts:executeResidualPerTuple`).

## ⚠️ Two divergences from the ticket spec (verify the reasoning)

The original ticket assumed two things that turned out **false against the landed
substrate**; both are documented in code/docs but a reviewer should sanity-check them:

1. **`extractBindings` does NOT yield a usable `'group'` binding for general
   aggregates.** `analyzeRowSpecific`/`classifyForAggregate` only classifies a source
   `'group'` when the GROUP BY columns *cover a source unique key* (the `coversAnyKey`
   gate). For the canonical `select g, count(*) from t group by g` (g non-key), it
   returns `'global'`. So the ticket's "populate `binding` from `extractBindings`,
   reject `'global'`" mechanism would reject almost every real aggregate. **Resolution:**
   the group binding is built **directly** from the aggregate node's bare GROUP BY
   columns. (The `binding: BindingMode` field is still populated, as
   `{ kind: 'group'; groupColumns }`, for substrate parity.) — *Is deriving directly
   the right call, or should `classifyForAggregate` instead grow a "group-key-as-backing-
   key" mode?*

2. **The per-statement-flush batching prereq shipped as connection-resolution caching
   only** — no op-buffering / statement-flush boundary (see the completed
   `3-materialized-view-rowtime-per-statement-batching` review). So the ticket's
   "accumulate distinct affected keys at the statement-flush boundary + wire the cost
   gate's runtime `degradeToRebuild`" cannot be wired as written. **Resolution:**
   maintenance is **per-row** (dedup is per-change OLD ∪ NEW only). The ticket itself
   states per-row recompute is correct without batching (every change to a group
   recomputes it from live state; the last write wins), and the equivalence harness
   confirms it. `degradeToRebuild` is set `false` (dormant), like the inverse-projection
   arm's dormant cost machinery. — *Affordability gap: a bulk `update t set v=v+1` over
   N rows in M groups runs the residual once per changed row (N times), not once per
   distinct group (M). Sound but O(N) residuals. Re-wiring dedup needs the statement-
   flush boundary (a separate batching follow-up).*

## ⚠️ Pre-existing optimizer bug found + worked around (NOT my code)

A two-column `group by` whose group columns are **all equality-pinned to constants**
collapses to a *scalar* aggregate, emitting one spurious all-NULL `count=0` row over an
empty input instead of zero rows. Reproduces as a plain query (no MV):

```
select a, k, count(*), sum(b) from t where a=0 and k=6 group by a, k   -- empty match
  => [[null,null,0,null]]   -- WRONG: should be []
select k, count(*), sum(a) from t where k=6 group by k                 -- single col
  => []                     -- CORRECT
```

This directly breaks the residual for an **emptied multi-column group** (the residual
pins both group cols, hits the collapse, and re-upserts a spurious row). **Worked
around** in `applyResidualRecompute`: only residual rows whose backing key **equals the
affected key** are upserted (`residualRowMatchesKey`) — a sound invariant regardless
(the residual for key K may only contribute K's slice), which discards the spurious
null-key row so the emptied group's backing row is correctly removed. Filed as
`fix/optimizer-constant-group-aggregate-empty-input-spurious-row`. **Reviewer: the
work-around relies on the spurious row carrying NULL group cols (empirically true). If a
future optimizer path projected the bound constants with count=0, the key would match
and the filter would NOT catch it — the real fix is the optimizer.** No `.pre-existing-
error.md` was written (no test fails, since the arm works around it); a `fix/` ticket
was filed instead.

## Scope: 1:1-join arm descoped to a follow-on implement ticket

The ticket's Phase 3 (1:1 row-preserving inner/cross join, `'row'`/`'pk'` binding) is
**not** in this change. It is genuinely separable and carries its own hard sub-problems
that warrant a dedicated pass:

- a **lookup-side change** (`update p …`) cannot bind a residual keyed on `T`'s PK — one
  P row maps to many T rows — so it needs a second P-keyed reverse residual or a global
  fallback (the ticket's test plan lists this case but the mechanism is undescribed);
- the **coverage-prover refactor** (factor `proveJoinNoFanout` /
  `innerJoinRetainsConstrainedTable` / row-preservation out of `coverage-prover.ts` into
  a shared helper) to drive the join admit.

Filed as `implement/3.3-materialized-view-rowtime-residual-join` (prereq:
`materialized-view-rowtime-residual-recompute`). The residual kernel built here
(`compileResidual` / `runResidual` / `applyResidualRecompute` / `residualRowMatchesKey`)
is shape-agnostic and reusable for it — `applyResidualRecompute` already handles the
`'pk'` prefix and the `'row'` binding shape generically. The `containsAnyJoin` rejection
is kept here; the join ticket lifts it. The downstream
`3.4-materialized-view-rowtime-prefix-delete-lateral-tvf` (lateral-TVF) is unaffected —
its prereq on the residual primitive is satisfied by this landing.

## How to validate / use

```sql
create table g (id integer primary key, k integer, v integer);
insert into g values (1,1,10),(2,1,5),(3,2,7);
create materialized view ga as select k, count(*) as c, sum(v) as s from g group by k;
select k, c, s from ga order by k;          -- [{k:1,c:2,s:15},{k:2,c:1,s:7}]
insert into g values (4,2,3);               -- recompute group k=2 only
update g set k = 2 where id = 1;            -- group-key change: recompute k=1 AND k=2
delete from g where id = 3;                 -- emptied group → backing row removed
-- reads-own-writes + rollback revert hold inside begin/…/rollback
```

### Known gaps / suggested review probes

- **HAVING aggregates** (`… group by k having count(*) > 1`) incidentally work (the
  residual re-evaluates HAVING; a group dropping below the threshold returns zero rows →
  backing row removed) but are **not tested**. Worth a probe.
- **`order by <aggregate>`** aggregate MVs are *rejected* (a backing-PK column would be a
  non-passthrough aggregate value) — sound, but untested; confirm the diagnostic.
- **O(N) residual runs** on bulk single-statement writes (no statement-level dedup) — see
  divergence #2. No perf test added.
- **MV-over-MV with an aggregate producer** is not explicitly tested (the cascade path is
  arm-agnostic and should work; the residual emits delete+insert effective changes for a
  recomputed group rather than a single update — verify a downstream consumer tolerates
  that).
- The empty-group filter's reliance on NULL-keyed spurious rows (see the optimizer-bug
  section) is the sharpest soundness edge — scrutinize.
