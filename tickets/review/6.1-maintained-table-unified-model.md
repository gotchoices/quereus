# Review: unified maintained-table schema model

Part 1 of the maintained-table chain, implemented per the design resolution: a
materialized view is now ONE `TableSchema` registered under the MV's own name,
carrying `derivation?: TableDerivation` (`schema/derivation.ts`). All lifecycle
verbs are catalog flips on one record. `create/refresh/drop materialized view`
keep working and remain the canonical DDL render. There is no stored `sql` —
`generateMaterializedViewDDL(table)` renders on demand from the unified record;
module identity lives on the table's own `vtabModuleName`/`vtabArgs`
(`normalizeBackingModule` survives as the differ/catalog normalizer).

- `yarn test:store` (logic suite against LevelDB): 5863 passing, 0 failing.

- Updated 20+ test files to the unified naming (four parallel passes); the

## Key design/implementation decisions a reviewer should challenge

- **Derivation identity.** The `TableDerivation` object is shared BY REFERENCE
  across catalog swaps of the owning table (tag updates, index appends, the
  backing-column rename pass re-attaches it explicitly). Runtime state
  (`stale`, `sourceScope`) lives on it so it survives swaps. Audit for any
  remaining swap site that could drop it (`module.alterTable` returns
  derivation-less schemas — the emitter now REJECTS structural ALTERs on
  maintained tables and `renameShiftedBackingColumns` re-attaches; is any other
  path re-registering a module-returned schema for a maintained table?).
- **DML dispatch order** (the ticket's highest-risk seam): the three builders
  check `getMaintainedTable` BEFORE plain-table resolution and adapt via
  `maintainedTableViewLike`. The lookup uses `stmt.table.schema ?? null`
  (current-schema default) while `buildTableReference` may search a schema
  path — same asymmetry the old `getMaterializedView` dispatch had, but worth a
  look for temp-schema shadowing corner cases.
- **DROP TABLE on a maintained table** now drops table + derivation through
  shared `dropMaintainedTable` (unregister maintenance → unlink covering link →
  dropTable → `materialized_view_removed`). The old redirect error died; the
  sqllogic expectation was updated accordingly.
- **ALTER TABLE rename on a maintained table** is an ordinary rename plus a
  maintenance re-key and a `materialized_view_removed`(old) +
  `materialized_view_added`(new) pair so the store moves the persisted MV
  entry. OTHER structural ALTERs are rejected with a clear diagnostic (new
  behavior — previously the MV name wasn't a table so they failed with "no such
  table"; now they would silently desynchronize, hence the explicit gate).
- **Self-reference reject**: the table registers under the MV name before the
  fill, so a body referencing itself is now lexically possible; create/import
  reject it up front (`assertNoSelfReference`).
- **Rehydrate ordering gate** (`ImportCatalogOptions.pendingDerivations`): in
  the old model an MV-over-MV dependent failed to PLAN until its upstream's MV
  record existed, which ordered the store's fixpoint for free. Now the
  dependent's body plans against the upstream's phase-1 plain table, so
  `importMaterializedView` defers (throws, retried next round) when any body
  source is named in the caller-supplied pending set. `rehydrateCatalog` passes
  the other still-pending entry names each round. Covered by
  mv-rehydrate-adopt; an adversarial review of mixed adopt/refill chains is
  welcome.
- **Adopt-failure posture** is now literally "table without derivation": a
  registration failure during adopt detaches the derivation and leaves the
  plain table registered (rows preserved).
- **Import derives the shape ONCE** (before any drop) and threads it through
  adopt/refill — a planning failure can no longer drop a durable backing even
  on the no-trust path (strictly data-safer than before).
- **Create-time tags**: found+fixed during testing — `def.tags` are stamped
  onto the created/adopted table (`buildBackingTableSchema` gained a `tags`
  param) and preserved across the refresh reshape. Pinned by sqllogic § 11 and
  three previously-failing suites.

## Known gaps / intentional behavior deltas (honest list)

- `column_info('mv')` now resolves the maintained table through the base-table
  branch and reports its columns as updatable base columns (previously threw
  not-found). Harmless but semantically debatable — a reviewer may want a
  follow-up to route it through the view lineage surface instead.
- `schema()` now reports maintained tables as `type = 'materialized_view'`
  (previously the hidden backing appeared as a plain `table` row named
  `_mv_<x>`). Deliberate, pinned by the new sqllogic block.
- A store-hosted maintained table persists TWO catalog entries (its ordinary
  table bundle — required for the phase-1/adopt fast path — plus the MV-form
  entry); a memory-hosted one persists exactly one (the MV form). Same shape as
  the old model, re-keyed; the ticket's "exactly one bundle" phrasing was
  interpreted as the memory-hosted case (documented in
  docs/materialized-views.md § store host).
- The rehydrate fixpoint recomputes `pendingDerivations` from the ROUND's
  initial pending list, so a dependent whose upstream imported earlier in the
  same round defers one extra round (converges; bounded by chain depth).
- `getSchemaItem` (views-then-tables) untouched; a maintained table resolves as
  a table there — consistent with the unified model.
- Plan-text assertions in the rewrite tests now key on `"name": "<table>"`
  lines because a maintained-table TableReference serializes its derivation
  (body SQL, sourceTables) into the plan dump — a reviewer may consider whether
  serializePlanTree should elide the derivation payload (cosmetic).
- `tickets/.pre-existing-error.md` was NOT needed — no pre-existing failures
  encountered.
