# Index canonical body: concurrent column-rename reconciliation (review)

Follow-on to `index-canonical-body-collation` (landed, in `complete/`). The drift
detector compares a declared index body against the actual catalog body. The
**actual** body renders the *pre-rename* column names (a column rename has not
landed at diff time), while the **declared** body renders the *new* names — so a
same-named index over a column renamed in the same diff rendered new-name (declared)
vs old-name (actual) and produced an unnecessary drop+recreate. This reconciles the
declared index body against the in-diff column renames so a pure rename matches (no
churn — the rename rides the `RENAME COLUMN` table-alter channel) while a genuine body
edit still recreates.

## What landed (implementation)

- **`declaredIndexCanonicalBody`** (`schema-differ.ts`) gained two params:
  `colRenames: ReadonlyArray<ColumnRenameOp>` and `schemaName: string`.
  - Each resolved bare column name is inverse-mapped NEW→OLD (case-insensitive)
    **after** collation resolution. **Ordering is load-bearing**: collation resolves
    on the *new* declared name first (the declared `ColumnDef` is keyed by it), then
    the emitted name is mapped back to its old form. Reversing it would look the
    collation up under the old name and miss it (regression-guarded — see tests).
  - The partial-`WHERE` predicate is reconciled the same way the constraint CHECK
    path is: each renamed column is inverse-rewritten NEW→OLD over a `cloneExpr` copy
    via `renameColumnInCheckExpression` (seeded with the index's table so unqualified
    refs resolve). Skipped entirely when there is no `where` or no renames.
- **Index loop call site** (`computeSchemaDiff`): passes
  `columnRenamesByTable.get(indexStmt.table.name.toLowerCase()) ?? []` and
  `targetSchemaName`. The lookup is keyed by the index's *declared* (post-rename)
  table name — the same key `columnRenamesByTable` uses — so a table renamed in the
  same diff still resolves its column renames. The `columnRenamesByTable` pre-pass
  already existed (built for the constraint FK reconcile); this reuses it, no new
  pre-pass.
- **`docs/schema.md`**: the "Index body-change detection" paragraph was **stale** —
  it still said collation was *excluded* and deferred both collation and rename
  reconciliation to the (now-split) backlog ticket. Rewrote it to state collation is
  included (the prereq) and column renames are reconciled (this ticket), including the
  ordering and table-name-keyed-lookup notes. **Reviewer: this doc edit reaches
  slightly beyond this ticket's direct code (it also corrects the prereq's wording) —
  verify the collation half reads accurately too.**

The partial-WHERE reconcile was implemented per the ticket's recommendation (not
deferred), keeping indexes symmetric with the constraint CHECK path. No backlog
ticket was filed for it.

## Testing surface / validation floor

All under the `declarative differ stability` describe in
`index-ddl-roundtrip.spec.ts`. The `diffIndexEdit(baseline, modified, policy?)`
helper applies the baseline as real schema (`apply schema` → `collectSchemaCatalog`),
so the actual `definition` comes from the live engine — not a declared-to-declared
shortcut. A column rename is expressed with `quereus.previous_name` on the modified
column (baseline under the old name, modified under the new name + hint). 11 new cases:

- **pure column rename, same-named index** → no index churn; `columnsToRename`
  carries `{ oldName, newName }`. (headline)
- **end-to-end convergence** — applies the rename for real, asserts the column is
  renamed, the index *survived* (no drop+recreate), the row is readable under the new
  column name, and the re-diff is empty (idempotent — the hint does not re-trigger).
- **rename + body edit (direction flip)** → recreates AND emits the column rename;
  recreate carries the new name + `desc`.
- **composite index, one renamed + one stable column** → no churn.
- **inherited collation unchanged under a rename** → no churn (the ordering guard).
- **collation also changed under a rename** → recreates; column `SET COLLATE` rides
  the table-alter channel. (NB: the recreate DDL carries **no** explicit COLLATE — the
  index inherits the table column's collation rather than declaring its own; the
  drop+recreate is the observable effect. An initial test asserted `/collate nocase/`
  on the recreate and correctly failed — the assertion was wrong, not the code.)
- **partial-WHERE over a renamed column** → no churn (predicate reconciled NEW→OLD).
- **partial-WHERE genuine predicate edit under a rename** (`= 1` → `= 0`) → still
  recreates (precedence guard for the WHERE reconcile).
- **`require-hint` policy, column-rename-only** → does not throw (zero index
  creates/drops, so the unhinted-rename guard is untripped).
- **renamed index AND renamed column together** → index rename op only, no body
  recreate (reconcile applies on a rename-matched index too).
- **table rename, columns stable** → no index churn (body excludes the `on <table>`
  ref; column-rename lookup returns `[]`).

Build, lint (`schema-differ.ts` + spec), and the **full quereus suite** all pass:
5230 passing, 10 pending, exit 0.

## Known gaps / scrutinize

- **WHERE reconcile relies on `renameColumnInCheckExpression`'s scope walk**, which
  has a documented limitation (rename-rewriter.ts ~308): aliased subquery /
  function-source / CTE-projection inner sources in the predicate are not asked
  whether they expose the column. For an index partial WHERE, top-level unqualified
  refs resolve to the indexed table via the seed binding; subqueries inside a partial
  predicate are rare and the limitation is **symmetric on both diff sides** (no NEW
  churn vs today). Not separately tested here — covered by the rewriter's own tests.
- **`deny` policy**: column renames are not detected at all, so an index over a
  renamed column drops+recreates. This is the `deny` contract (no renames), not a
  reconcile bug — left untested deliberately (would assert the deny semantics, not
  this code).
- **No store-path test** (`test:store` not run — agent default is memory-backed).
  The reconcile is pure diff logic (no module interaction) and the convergence test
  exercises only the memory backend; a store-specific run was not done.
- The convergence test is the only **apply-path** assertion; the other 10 cases pin
  the diff DECISION. For a *pure* rename the apply path is a real `RENAME COLUMN`
  primitive (index columns are stored by ordinal, so the index needs no rewrite),
  which the convergence test confirms — but a reviewer wanting apply-level coverage of
  the *rename + body edit* recreate (does the recreate's `CREATE INDEX` read the
  post-rename column?) would need to add it; the prereq's
  `index-explicit-column-collate-apply-path` fix ticket is the relevant adjacent gap
  for explicit-index-COLLATE recreates.
