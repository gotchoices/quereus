## What landed

`FilterNode.estimatedRows` used to hardcode a `0.5` selectivity (every WHERE keeps half its rows). It now derives from the statistics layer that already existed (`context.stats.selectivity(table, predicate)` → histogram / `1/ndv` equality / null-fraction / naive fallback).

Because node accessors (`estimatedRows` / `computePhysical`) carry no `OptContext`, the stats lookup is done by a **Physical-pass rule** that holds the context and stamps the result onto a new optional `FilterNode.selectivity` field.

### Changes
- **`filter.ts`**
  - New `export const DEFAULT_FILTER_SELECTIVITY = 0.5` (last-resort default).
  - Constructor gains `readonly selectivity?: number` (5th param, after `estimatedCostOverride`).
  - `estimatedRows` getter multiplies the **logical** source rows by `this.selectivity ?? 0.5`.
  - `computePhysical` multiplies the **physical** source cardinality by the stamped selectivity — see the deviation note below; this is the branch that actually makes the feature observable.
  - `withChildren` carries `selectivity` through only when the predicate is unchanged (dropped on predicate change); tripwire `NOTE:` at that site about stale-estimate risk if a stamped Filter is ever re-sourced.
  - `withPredicate` drops `selectivity` on a predicate change (comment added).
- **`rule-filter-selectivity.ts`** (new) — Physical pass, `nodeType: Filter`, `phase: 'impl'`, `sideEffectMode: 'safe'`. Declines when already stamped, when `extractTableSchema` returns undefined (join / multi-table), or when the provider returns undefined. Clamps to `[0,1]`. Registered in `RULE_MANIFEST` next to `select-access-path`.
- **`docs/optimizer.md`** — Statistics Abstraction section documents the new filter-estimate path.
- **`filter-selectivity.spec.ts`** (new) — 6 tests (below).

## IMPORTANT deviation from the plan — reviewer please scrutinize

The ticket asserted: *"computePhysical already derives rows from `this.estimatedRows`, so it inherits the new selectivity automatically."* **This is false in practice**, and I changed `computePhysical` to compensate. Reason:

- `FilterNode.estimatedRows` reads `this.source.estimatedRows` — the **logical** getter.
- Physical access nodes (`IndexScan` / `IndexSeek`) define **no** `estimatedRows` getter, so `source.estimatedRows` is `undefined` once the Physical pass has converted `Retrieve → access node` (which happens bottom-up *before* this rule fires on the parent Filter).
- So the getter returns `undefined`, and the old `computePhysical` fell back to the raw physical source rows — **selectivity had zero effect on any real plan** (verified: a residual Filter over a range seek stayed at 100 rows, ignoring both the new stats value *and* the old flat `0.5`).

Fix: `computePhysical` now applies the stamped selectivity to the **physical** source cardinality (`childrenPhysical[0].estimatedRows`) when `this.selectivity !== undefined`, falling back to the old `min(srcRows, est)` when unstamped. The getter still applies selectivity for the rarer case where the source *does* expose a logical row count (Filter over Values/Join/another Filter). No double-counting: getter → logical, computePhysical → physical, each applies the factor once.

**If the reviewer disagrees with touching `computePhysical`, the alternative is to give access nodes a logical `estimatedRows` getter** — a larger change I deliberately avoided.

## How to validate

Run: `yarn workspace @quereus/quereus test` (full suite: **6902 passing / 13 pending / 0 failing** at handoff) and `yarn workspace @quereus/quereus run lint` (clean).

Focused: `node --import ./packages/quereus/register.mjs node_modules/mocha/bin/mocha.js "packages/quereus/test/optimizer/filter-selectivity.spec.ts"`

The spec covers:
- **Deterministic mechanics** (via direct `computePhysical([{estimatedRows: N}])` calls, no optimizer dependence):
  - non-covering predicate: `floor(srcRows * sel)` (and ≠ the flat-0.5 value);
  - covered unique key forces `estimatedRows = 1`, overriding a huge stamped selectivity (locks the branch-ordering the ticket flagged);
  - `selectivity = 0` floors to 1.
- **End-to-end through the optimizer**:
  - seeded catalog stats (`ANALYZE`, ndv 4) → Filter `selectivity === 1/ndv`, physical `estimatedRows === floor(srcRows/ndv)`, ≠ `floor(srcRows*0.5)`;
  - stats-less table → naive fallback (`0.1` equality), no crash;
  - join-source filter → selectivity stays unstamped.

## Known gaps / things to probe (your tests are a floor, not a ceiling)

- **Full-scan sources report physical `estimatedRows = 0` for memory vtabs** (pre-existing; the access node doesn't seed its physical cardinality from catalog `rowCount`). Consequence: `SELECT ... WHERE cat = 'a'` over a plain full scan yields Filter `estimatedRows = floor(0 * sel) = 0` — the selectivity is correct but invisible because the *source* is 0. The feature is only observable over sources with a positive physical cardinality (range seeks, etc.). This is an access-node cardinality gap, **out of scope** here; the broader stats work is parked in backlog `feat-conjunction-and-join-selectivity`. Worth a reviewer eye on whether it deserves its own `debt-` ticket.
- **Conjunctions (`a = 1 and b = 2`) are not decomposed** — the `CatalogStatsProvider` finds no single column child on the `AND` root and falls back to `NaiveStatsProvider`'s coarse `0.1`. Crude but not a regression vs. flat `0.5`. Parked in backlog `feat-conjunction-and-join-selectivity` (a `NOTE:` in the rule points there).
- **Golden plans: no churn observed.** The ticket predicted EXPLAIN/golden snapshot churn; none materialized because the golden corpus's Filters sit over full scans (0-row physical sources, above) and the aggregate/optimizer suites don't assert the affected estimates. All 5 golden + 1544 other plan/optimizer tests pass unchanged. If the access-node-0-rows gap is later fixed, expect golden churn *then*.
- **`withChildren` stale-selectivity tripwire** — a carried selectivity was computed against the old source's table; harmless today (Physical is the last rule-bearing pass over Filters) but noted in-code.
- I did **not** add a test for the pure logical-source getter path (Filter over a source that exposes logical `estimatedRows`); the mechanics tests exercise `computePhysical`, which is the observable surface. A reviewer wanting belt-and-suspenders could add one.
