## What landed (this run)

- **`create table … maintained as`** (`createMaintainedTable`): existence /
  `if not exists` check first (skip entirely — never half-attach), then the
  declared shape is verified against the derived body shape BEFORE any catalog
  registration (`SchemaManager.buildDeclaredTableSchema` builds without
  registering), then normal `createTable` + the shared attach core
  (attach-to-empty = the fill, applied to the connection's pending state — no
  `replaceContents` commit-first caveat). Any later failure drops the table.
  Build gates in `building/ddl.ts`: DML-body reject via `planViewBody`, arity
  vs declared count, generated-column reject, backing-host capability on the
  resolved module (using clause else session default).
- **`alter table X set maintained as`** (`attachMaintainedDerivation`) — the
  ordering is: capability → derive shape → strict shape check → cycle check →
  evaluate body once + duplicate-key pre-check → catalog flip
  (`attachDerivation`) + `registerMaterializedView` (which runs the create-time
  gates: determinism / keyed-or-coarsened / size threshold) → reconcile
  (`'replace-all'` keyed diff via `BackingHost.applyMaintenance` on the
  registered/created connection) → covering links (unlink old, link new) →
  cascade genuine changes via `db._maintainRowTimeCoveringStructures` +
  `_flushDeferredRebuilds` → `materialized_view_added` (fresh) /
  `materialized_view_modified` (re-attach) → coarsened-key warning. A gate or
  reconcile throw restores the prior record and (on re-attach) re-registers the
  prior plan (`restorePrior`; a failed prior re-register fail-safes to stale).
- **`alter table X drop maintained`** (`detachMaintainedDerivation`):
  release plan → unlink covering → catalog swap minus derivation (staleness
  leaves with it) → ONE `materialized_view_removed`, deliberately NO
  `table_modified` (consumers stay live — pinned by spec test).
- **Cycle check** (`assertNoDerivationCycle`): walks sourceTables→derivation
  edges of the live catalog; self-reference is the degenerate case; diagnostic
  names the cycle path.
- **Duplicate derived keys** (`assertDerivedRowsAreSet`): collation-aware
  sort + adjacent-compare under the table's PK collations, BEFORE any mutation;
  the error names the colliding key values.
- **Statement-cache invalidation** (`core/statement.ts`):
  `materialized_view_added/removed/modified` now invalidate dependencies of
  type `'table'` (and legacy `'view'`) — without this, a cached direct-write
  plan would survive an attach (writing the maintained table directly) and a
  cached write-through plan would survive a detach. Pinned by spec test.
- **Test-expectation migration**: `view-mv-ddl-persistence.spec.ts`'s
  `generateMaintainedTableDDL` fixed-point matrix now lifts BOTH authoring
  forms (`mvSchemaFromDDL` handles `createTable`+maintained via
  `columnDefToSchema`/`findPKDefinition`) and asserts the table form.

## Key design decisions (review these)

1. **Strict shape check, names included** (`describeAttachShapeMismatch`): the
   verbs require the body to derive the declared shape verbatim — column names
   (alias to match), types, **nullability exact in both directions**,
   collations, physical PK (order/direction/collation; a body `order by` seeds
   the derived PK via `computeBackingPrimaryKey`, so it must match the declared
   key). Rationale for two-way not-null strictness: a tolerated skew would make
   the next refresh's reshape pass "tighten" the declared column, mutating the
   frozen basis. The canonical-DDL **import** path stays lenient (declared
   names are the rename list) — that's what makes the sugar's explicit-column
   round-trip work; the verbs are the strict authoring surface.
2. **`derivation.columns` = declared column names at attach** — matching
   `maintainedImportFromTableStmt`, so attach → persist → reopen is a
   `bodyHash` fixed point (pinned by spec test). A sugar-created MV without an
   explicit column list still records `columns: undefined` (pre-existing known
   gap on first reopen, ticket `mv-table-form-implicit-columns-roundtrip`).
3. **Reconcile through the pending layer** (applyMaintenance replace-all), not
   `replaceContents` — also for the create-fill. Transactional in lockstep with
   the statement; identical content = zero writes (host's keyed-diff skip);
   changes cascade to consumers like source writes. The catalog flip itself is
   NOT transactional (DDL-in-transaction posture matches create-MV).
4. **Detach fires no table event.** Consumers of the detached table stay live;
   cached-plan invalidation rides the `materialized_view_removed` mapping
   extension instead. If a reviewer finds a consumer of `table_modified` that
   relied on detach firing it, that's the seam to look at.
5. **Registration before reconcile.** The create-time gates run via
   `registerMaterializedView` BEFORE rows are written, so a gate failure needs
   only a catalog restore. The reconcile failing after registration rolls the
   flip back; its pending writes die with the statement.

## Known gaps / candid notes for the reviewer

- **Attach diagnostics are named, not source-located**: the emitter-side shape
  /cycle/duplicate errors name the table and component but carry no line/column
  (the create-path arity/DML gates DO site at build). Acceptable per "sited"
  reading = names the first mismatching component; flag if you read it stricter.
- **`alter table … set maintained` on a missing table** surfaces
  `buildTableReference`'s generic resolution error, not a verb-specific one.
- **Synthesized all-columns PK interplay**: a declared maintained create
  without a PK gets the synthesized all-columns key; bodies rarely derive that
  exact key, so the shape check effectively forces authors to declare a PK.
  Not separately tested.
- **Write-DML rejection on maintained tables is still write-through**, not
  READONLY — unchanged; the enforcement seam is `backing-tables-readonly-
  enforcement` (backlog), keying off `derivation` presence (detach sheds it
  structurally — pinned by the detach-then-write tests).
- **Declarative differ does not consume the maintained clause** on table items
  (parses through; differ ignores it) — that is ticket 6.3
  (`maintained-table-differ-transitions`, already in implement/ with a resume
  note); docs say so explicitly (migration.md shows the declarative form via
  the MV item with a pointer).
- **Transient store-mode harness flakes observed once each** during iteration
  (a TS1005 parse error in an untouched file; a `getCheckExtraction is not
  defined` ReferenceError) — both vanished on re-run and never recurred across
  3 consecutive store-mode runs; likely concurrent-file-access noise, not
  reproduced, nothing filed.
- The maintenance-equivalence "attached-not-created variant over a couple of

  cascade-on-attach. An aggregate-body attach variant was NOT added — worth a
