# Review: lateral-TVF fan-out `'prefix-delete'` arm

## What shipped

The fourth `MaintenancePlan` arm, `'prefix-delete'`, wired for **single-source
lateral-TVF fan-out** bodies:

```sql
create materialized view mv as
  select t.id, t.x, f.value
  from t cross join lateral generate_series(1, t.x) f;
```

Each base row of `t` fans out to **N** backing rows; the backing PK is the **composite
product key** `(t.pk ∪ f.key)` that `keysOf` advertises across the lateral `JoinNode`
(`optimizer-keyed-cross-product-join-keys`), with the base PK as its **leading prefix**.
Per source row-write, maintenance deletes the base row's whole fan-out slice **by base-PK
prefix** and re-runs the TVF fan-out **residual** for that base row, upserting each
recomputed row. Reuses the residual kernel from `materialized-view-rowtime-residual-recompute`
unchanged (`injectKeyFilter` on the base `TableReferenceNode` with the `'pk'` prefix,
`runResidual`, reads-own-writes, the cost gate) and differs only in the **prefix** delete
(vs point key) and the **N-row** residual (vs ≤1).

### Phase 1 — by-prefix delete primitive (`manager.ts`)
- Re-added `{ kind: 'delete-by-prefix'; keyPrefix: SqlValue[] }` to `MaintenanceOp`.
- `applyMaintenanceToLayer` is now **async**; the new arm range-scans the primary btree
  over the prefix interval (reusing `scanLayer`'s `equalityPrefix` seek + early-term),
  collects matched rows, then `recordDelete`s each — same bookkeeping (secondary indexes,
  change tracking) as the point `delete-key` arm. Collect-then-delete avoids
  mutate-during-iterate. The three existing callers `return` the now-async result.

### Phase 2 — `'prefix-delete'` arm + shape gate (`database-materialized-views.ts`)
- `PrefixDeletePlan` added to the `MaintenancePlan` union; `applyMaintenancePlan`
  dispatches it.
- `buildMaintenancePlan` routes the lateral-TVF shape to `buildLateralTvfPrefixDeletePlan`
  **before** the generic join/TVF rejections (a lateral fan-out surfaces both a `Join` and
  a `TableFunctionCall`). Gate: 1 base ref + 1 TVF + 1 join, deterministic TVF +
  operands, TVF advertises a per-call key, base PK projected and the leading prefix of the
  backing PK with a non-empty TVF-key tail.
- `applyPrefixDelete` mirrors `applyResidualRecompute`: derive affected base key(s)
  (OLD ∪ NEW deduped) → `delete-by-prefix` + residual fan-out + upsert each row whose
  base-PK prefix matches.
- `runResidual` generalized to `(scheduler, bindParamPrefix, keyTuple)`, shared by both
  residual arms; `resolveTransitiveSourceCol` extracted to module scope (was a closure in
  the aggregate path).

`read(MV) == evaluate(body)` mid-transaction AND after rollback. Covers fan-out growth/
shrink, base-PK-moving updates (whole prefix moves), the `where k > 5` partial-WHERE

all-columns-key shape.

no-match no-op, single-row-slice byte-identical to `delete-key`, secondary-index

## Honest gaps / reviewer focus

- **Non-binary collation on the base PK is reasoned-sound but not directly tested.** The
  `delete-by-prefix` scan early-terminates on a **binary** prefix mismatch, while the
  backing btree orders by the column's collation. This is sound because the base-PK column
  carries `T`'s PK collation and `T`'s PK is unique under it — so only one base row maps to
  each collation class, its fan-out rows are binary-homogeneous, and no
  collation-equal/binary-different base-PK rows can interleave the slice. The harness uses
  integer PKs (binary), so this reasoning is **untested**. Worth a skeptical look (e.g. a
  `text collate nocase` base PK with a lateral fan-out) — if wrong, the early-term could
  miss tail rows. (The same binary-prefix-scan reasoning already underlies
  `lookupCoveringConflicts`' fast path, which *does* gate off non-binary collation.)
- **`order by` reorder → `reject` (UNSUPPORTED), not the `INTERNAL` assert the ticket
  suggested.** The base-PK-leads-prefix check is user-reachable via `order by f.value`, so
  it surfaces as a user-facing shape diagnostic (steer to a plain view) rather than a loud
  `INTERNAL`. The gate IS the invariant check; no separate defensive `INTERNAL` assert was
  added (it would be unreachable).
- **No separate `WHERE` predicate compiled.** The body's `WHERE` is part of the residual
  (an out-of-scope base row fans out to zero rows), exactly as in the aggregate arm — so
  the ticket's "compile the partial WHERE" step is subsumed by the residual. Verified by
  the partial-WHERE harness shape and §23's `ltw_fan` scope-transition cases.
- **Cost gate stays dormant (as in the aggregate arm).** `chosenStrategy` is
  `'residual-recompute'` (the shared key-filtered-re-execution cost shape); `kind` is
  `'prefix-delete'`. The synchronous reject-at-create / `degradeToRebuild` machinery is NOT
  wired — the TVF fan-out factor is unknown at create, so a "pathological fan-out" cannot be
  detected without fan-out stats. `degradeToRebuild` is `false`. This matches the prereq's
  honest stance; flag if the reviewer wants an explicit large-fan-out guard.
- **`binding: BindingMode` is carried for substrate parity but unread** by `applyPrefixDelete`
  (which uses `bindColumns` / `backingPrefixSourceCols`) — same as the aggregate arm.
- **`residualRowMatchesBasePrefix` is a defensive net.** The residual filters `T` to the
  affected base key, so every returned row already shares that base-PK prefix; no
  optimizer-mis-collapse analogue (the aggregate arm's empty-group spurious-row bug) was
  observed for the lateral shape. Kept for parity/robustness.
- **Statement-level dedup of distinct base keys deferred** (O(N) residuals on bulk single-
  statement writes), same affordability deferral as the aggregate arm.

## Follow-up (out of scope, noted in docs)

A **fanning keyed join** (a non-1:1 inner/cross join) is the natural next consumer of this
exact by-prefix machinery — it shares the by-prefix delete + composite product key,
differing only in that the "fan-out residual" is the join rather than a TVF. Deferred; not
filed as a ticket here (the existing `materialized-view-rowtime-residual-join` covers the
1:1 join; the fanning variant would be its non-1:1 sibling).
