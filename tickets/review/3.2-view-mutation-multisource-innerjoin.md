# Multi-source inner-join view mutation — implement handoff

This is the **review handoff** (slug intentionally preserved as `view-mutation-multisource-innerjoin`
because `lens-multi-source-put-fanout` prereqs this exact slug). Treat my work as a
**starting point**; my tests are a **floor**, not a ceiling. Read the "Honest gaps" and
"What the reviewer should probe" sections carefully — I deliberately scoped this ticket down
and decomposed the rest, so there is real surface I did **not** build.

## What landed (verified: build + lint + `yarn workspace @quereus/quereus test` all green — 4136 passing, 9 pending, 0 failing)

### 1. Emitter: sequential base-op execution (`runtime/emit/view-mutation.ts`)
The 3.1 emitter passed base ops as bare scheduler params. The scheduler (`runtime/scheduler.ts`
`runAsync`, line ~162) **kicks off sibling params concurrently** — it does not await one before
starting the next; their result promises are awaited together at the consuming node. So a
multi-op fan-out would interleave the base writes and lose any ordering `propagate` decided.
The emitter now wraps each base op with `emitCallFromPlan` (a sub-program callback) and **awaits
them in list order** inside `run`. For the single-source spine (one base op) this is parity with
the prior single-param form — confirmed by the green 93.1/93.3/93.4/53.1 view+MV sqllogic.

### 2. Multi-source decomposition (`planner/mutation/multi-source.ts`, dispatched from `propagate.ts`)
- `propagate()` peeks the view's `selectAst` FROM (cheap, no plan built): a join body routes to
  `propagateMultiSource`; a single-table body stays on the unchanged single-source spine.
- `analyzeJoinView` builds the **planned** body (`buildSelectStmt`), validates it is a two-table
  inner equi-join over plain base tables, and reads `root.physical.updateLineage` to route each
  output column to its owning base table (`UpdateSite.base.table` = the producing
  `TableReferenceNode`'s id). Explicit projections only.
- **UPDATE** routes each assignment to its owning side (computed/non-identity column → `no-inverse`),
  emits one base `update` per touched side, ordered FK-parent before FK-child. The SET value is
  rewritten to base terms and the owning-side qualifier stripped; a value referencing the other
  side → `cross-source-assignment`.
- **DELETE** routes to the FK-many (child) side, proved from a declared foreign key; no provable FK
  → `delete-ambiguous` (the `delete_via` tag override is `view-mutation-tag-override-surface` / 3.4).
- Both build the row-identifying predicate as a **subquery over the join body** lowered to AST
  (`<owning>.<pk> in (select <alias>.<pk> from <cloned join FROM> where <userWhere∧bodyWhere>)`),
  so the base builders are reused verbatim. Requires a **single-column PK** on each addressed side.

### 3. Diagnostics (`mutation-diagnostic.ts`)
Added `unsupported-multisource-insert`, `cross-source-assignment`, `delete-ambiguous`. Every
rejection message begins with `cannot write through view '<v>'` (so existing
`/cannot write through/i` expectations — incl. the lens `write-rejected` boundary test — keep
matching).

## Honest gaps / NOT done (decomposed into follow-on implement tickets)

- **Multi-source `insert` + per-row shared-surrogate context envelope** → `view-mutation-shared-surrogate-insert`.
  Through a key-hiding join the join key is *not* a view column, so insert fundamentally needs a
  surrogate (`default next_rid()`) resolved **once per row at the envelope and threaded across both
  base inserts**. I confirmed there is **no** existing machinery for a per-row value shared across
  sibling base ops (mutation context is per-`DmlExecutorNode`, evaluated once per statement) — this
  is genuinely new runtime infra, not a small add. The `u_core`/`u_contact` worked example and its
  sqllogic live in that ticket. `lens-multi-source-put-fanout` rides this envelope and now prereqs it.
- **RETURNING-through-view** → `view-mutation-returning-through-view`. Needs `ViewMutationNode`
  promoted to relational, post-mutation view-image capture in the emitter, and the `block.ts`
  result-shadowing exclusion revisited. Still rejected (`returning-through-view`) for both single-
  and multi-source today.
- **Restrictions of the shipped UPDATE/DELETE** (each a precise diagnostic, none silent): composite-PK
  sides, `>2`-table joins, self-joins, `select *` join bodies, comma/implicit cross joins, outer
  joins, cross-source `set` values, and the `delete_via` override (FK-many is the only delete policy
  until 3.4). The reviewer should sanity-check that none of these silently *widen* a write.
- **Conflict composition / FK ordering across ops**: ordering is decided in `propagate` (FK-parent
  before FK-child) and executed sequentially by the emitter; the per-op `or` mode rides each base
  op's own builder (reused). I did **not** add same-row same-table fan-out merging (it is an insert
  fan-out concern, deferred with insert). FK ordering for UPDATE/DELETE is low-stakes (no row is
  created mid-statement); verify this reasoning holds.
- **`query_plan()` lineage across the join**: relied on as already-threaded by
  `view-mutation-physical-lineage` (property Law 4 covers the planned join lineage statically); I did
  not add a new EXPLAIN fixture.

## What the reviewer should probe (highest-value adversarial targets)

1. **Subquery-IN identifying-predicate semantics.** Is the non-correlated `pk in (select … from
   <join> where …)` always equivalent to the intended view-row set? Especially: a parent-side
   update affecting *all* children that join it (intended, predicate-honest), and the Halloween
   case (updating the PK that also feeds the join — the subquery is evaluated once, so it is stable;
   confirm). Composite/duplicate join matches.
2. **Qualifier rewriting** (`substituteViewColumns` / `stripSideQualifier`): a user predicate or SET
   value qualified by the view name vs a base alias vs unqualified — does it always resolve to the
   right base column? Edge: a base column whose name collides with a view column name.
3. **FK-child detection** (`fkChildIndex`): self-referential or mutual FKs, FK to a different schema,
   multi-column FKs. The default-delete-to-child policy vs the doc's "delete from every side absent a
   tag" — I chose FK-child-only-when-provable + reject otherwise (the doc's named FK-style default);
   confirm that is the right call before the `delete_via` tag lands.
4. **Lineage routing correctness** when the same base column appears on both sides via the equi-join
   EC — `UpdateSite.base.table` should disambiguate, but worth a targeted check.
5. **`select *` and `view.columns` overrides**: I reject `select *` over a join; confirm that is
   acceptable vs. expanding it.

## Items deliberately out of scope (remain rejected, as the ticket allows)
Set-op / outer-join / aggregate / window / recursive-CTE bodies; nested-view/MV-over-MV join sources;
the `quereus.update.*` tag surface (3.4). All reject with structured diagnostics from the planned walk.
