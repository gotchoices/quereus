## What landed

Extends `ruleMaterializedViewRewrite` with a **join-subsumption arm** that recognizes when a
`Project(Filter?(Join(T, P)))` query is covered by a 1:1 row-preserving **inner/cross** join MV
body (the row-time `'join-residual'` shape) and rewrites it to scan the MV's backing table —
**eliminating the join at read time**.

```sql
create materialized view enriched as
  select o.id, o.customer_id, o.amt, c.name
  from orders o join customers c on o.customer_id = c.id;   -- 1:1 NOT-NULL FK → PK

select o.id, o.amt, c.name
from orders o join customers c on o.customer_id = c.id
where o.amt > 100;                                          -- → scan _mv_enriched, residual filter + project
```

### Mechanism (soundness is the whole game)

The hard question — "does this join contribute *exactly one* row per governed `T` row?" — is the
coverage prover's shared `proveOneToOneJoin` (no-row-loss descent + `proveJoinNoFanout`). A 1:1
join's output is in bijection with `T`'s governed rows, so two 1:1 joins over the **same tables,
equi-pairs, and type** produce the same relation. `matchJoinFragmentToMv`:

- runs `proveOneToOneJoin` on **both** the fragment join and the MV body join (the rule plans the MV
  body once, suppressed, caches its optimized root in a `WeakMap`);
- requires the **same driving `T`, same lookup `P`, an inner/cross top join on each side**
  (outer deferred), and **equi-pair equivalence** in `(driving-col, lookup-col)` terms — the
  load-bearing guard that rejects a join on a *different FK* to the same lookup;
- proves **projection coverage** over the joined output (every fragment output, incl. lookup-side
  columns, is a stored backing column) via stable attribute ids;
- carries the **post-join WHERE as a residual** (a join MV body has no WHERE — the create gate
  forbids a partial join body — so entailment is trivial). The residual re-binds onto the backing by
  **source attribute id** (a base-column index is ambiguous across a join).

**Read-side relaxation:** a WHERE on a *lookup-side* column is allowed here (the row-time arm forbids
it) because we only *read* the materialized join. A residual on a non-stored column ⇒ `missing-column`;
a subquery conjunct ⇒ `predicate-not-entailed` (preserving the no-subquery invariant behind
`sideEffectMode: 'safe'`).

The replacement is the foundation's emission **unchanged** (backing scan → residual `Filter` →
residual `Project`); the only new field on `RewriteMatch` is `backingColOfSourceAttrId` (attr-id-keyed
residual re-bind) plus `joinInfo` (cost inputs). Cost gate: recompute now pays **both base scans + the
join** (`hashJoinCost`, the cheaper physical, so the gate stays conservative) vs. the 1:1 backing scan,
so the rewrite wins decisively; same cheapest-wins + stable-name tiebreak.

`coverage-prover.ts` change is a **one-line export** of `pureJoinEquiAttrPairs` (handles logical
`JoinNode` + physical Bloom/Merge joins, with the pure-equi soundness gate).

  passing, 9 pre-existing pending, 0 failing), `eslint` on the changed files (clean).
- **Matcher unit + golden plans:** `test/query-rewrite-join.spec.ts` — positives (residual-free,
  driving-side residual, **lookup-side residual**), per-reason negatives (`shape` for outer join /
  different equi-pair / fanning non-1:1 join; `missing-column` for omitted lookup col;
  `source-mismatch`; `no-candidate` for stale), end-to-end rows == rule-disabled recompute, golden
  plan asserts **no `Join` survives**, a fanning near-miss keeps the join, MV self-maintenance is
  never rewritten, and a **stale-ALTER → refresh** cycle suspends then resumes the rewrite (exercises
  the cache invalidation).
- **Equivalence harness:** `test/query-rewrite-equivalence.spec.ts` 3rd block — `rewritten ==
  unrewritten` over random data across the **FK→PK boundary** (generator only emits child rows for
  existing parents), **NULL lookup columns**, **empty-lookup**, with driving + lookup residual shapes;
  plus a non-vacuous check that the rewritable queries actually rewrite (and lose the join).

## Reviewer: probe these (honest gaps / risk areas)

1. **Nested `db.getPlan` inside a query-optimization rule is novel.** The join arm plans the MV body
   *during* the Structural pass (suppressed, so the nested optimize bails on the suppression flag;
   `OptContext` is per-call so there's no shared-pass-state corruption). Tests pass and re-entrancy is
   sound, but this pattern wasn't used at query time before — worth a skeptical look at perf under
   many candidate MVs / deeply nested queries, and at whether the per-fire re-plan (cached) is
   acceptable on the hot path. The cache is a module-level `WeakMap<MaterializedViewSchema, …>`
   invalidated on staleness and storing only valid roots — verify the stale→refresh reasoning for a
   `select *` join body (the case the invalidation specifically guards).
2. **No explicit unit test for a `select *` join MV body** or a **composite-FK / composite-PK** 1:1
   join, though both are handled structurally (stored-column map from body output attrs;
   `proveOneToOneJoin` supports composite FKs, exercised by `covering-structure.spec.ts`). Adding a
   join-rewrite test for each would tighten coverage.
3. **`predicate-not-entailed` is reused** as the reason for a subquery-bearing residual conjunct —
   defensible (it's "a predicate we won't handle") but arguably deserves its own reason code.
4. **Residual determinism is not gated** — the fragment's residual WHERE may contain a
   non-deterministic function over stored columns. This matches the foundation arm (which gates only
   the MV *body*, not the query residual) and is sound (the predicate is evaluated per joined row
   identically either way), but confirm you agree with that stance.
5. **Cross join whose equi-condition lives in a separate WHERE** (`from a, b where a.fk = b.id`) is
   **not** rewritten (the equi-pairs must be in the join `ON`). Sound NotMatch; flagged so it isn't
   mistaken for a bug.
6. **Cost model** uses a 1:1 backing cardinality ≈ driving rows and `hashJoinCost`; sanity-check the
   tiny-table boundary where recompute could tie.

## Out of scope (parked upstream, unchanged here)

- Outer-join 1:1 MV bodies; multi-join MV bodies covering a sub-join of the query (partial join
  matching); rollup over a join MV.
