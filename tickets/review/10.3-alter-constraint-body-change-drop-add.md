# REVIEW: constraint BODY change → DROP + ADD CONSTRAINT in the declarative differ

## What this does

Before this ticket, the declarative differ (`computeSchemaDiff` / `computeTableAlterDiff`) could
add / drop / rename a named constraint by **name**, but a constraint whose **name was unchanged and
whose body changed** (an edited CHECK expression, a changed FK action / referenced table / columns,
a changed UNIQUE column set / `ON CONFLICT`) was invisible — the catalog projection carried only
`{ name, tags }`, so a body edit silently no-op'd at `apply schema`.

This ticket makes the differ see body changes and realize them as **drop-old + add-new**
(drop+recreate — there is no in-place "redefine" primitive; re-creation re-validates existing rows
for UNIQUE/FK). It reuses the prereq's `constraintsToDrop` / `constraintsToAdd` buckets and the
existing `generateMigrationDDL` emission — only their **population** is new.

## Design (as built)

**Carry a comparable body in the catalog.** Each `CatalogTable.namedConstraints` entry gains a
required `definition: string` — a canonical DDL **body fragment** with the `constraint <name>`
prefix and `with tags (...)` suffix **excluded**. Comparison is then `actual.definition !==
declared.definition`.

- **One rendering path, both sides.** `ast-stringify.ts` exports
  `constraintBodyToCanonicalString(tc: AST.TableConstraint)`: it strips name + tags, normalizes
  parser-default-equivalent forms, then calls the existing `tableConstraintsToString`. Both sides
  funnel through it, so fragments are byte-comparable:
  - **Declared side** (`schema-differ.ts collectDeclaredNamedConstraints`) calls it on the declared
    AST constraint (keeping the full-fragment `ddl` separately, since `ADD` still needs name+tags).
  - **Actual side** (`ddl-generator.ts constraintToCanonicalDDL(kind, schemaConstraint, tableSchema)`)
    lifts the stored constraint back into an `AST.TableConstraint` (column indices → names,
    `RowOpMask` → `RowOp[]` via the new `table.ts maskToOps`, `defaultConflict` → `onConflict`, FK
    actions / referenced columns), then calls the same renderer. Populated in `tableSchemaToCatalog`.
  - **Normalization** (so semantically-equal forms don't churn a spurious drop+recreate): bare
    CHECK's default `INSERT|UPDATE` mask is elided (matches the parser leaving `operations`
    undefined); other masks emit in fixed insert→update→delete order; FK default `RESTRICT` action
    elided; FK empty referenced-column list elided; default `ON CONFLICT ABORT` elided. FK
    `deferrable` is **dropped** from the fragment (not a tracked body channel — see gaps).

**Detect in `computeTableAlterDiff`.** For each declared named constraint matched (by name OR by
rename hint) to an actual via `constraintRenames.pairs`, compare `definition`. On drift: push the
old name to `constraintsToDrop` and the declared fragment to `constraintsToAdd`.
- **`tags` excluded from `definition`** — a tag-only change yields equal `definition`, so it takes
  the in-place `ALTER CONSTRAINT … SET TAGS` path (ticket #10), never a drop+recreate. This is the
  single most important correctness property; tested three ways (tag-only / body-only / both).
- **Rename+body precedence** — when a constraint is rename-matched AND its body differs, the
  drop+recreate (under the declared name/def) **suppresses the RENAME**.
- **require-hint** — body-change drop+add pairs are **excluded** from the add/drop count that trips
  `rename_policy = 'require-hint'` (they are deliberate recreates, not ambiguous renames). Pure
  create/drop counts are tracked separately and preserve the prior guard semantics exactly.
- **No redundant SET TAGS** — a body-change-recreated constraint is skipped in the tag-drift loop
  (its `ADD` fragment already carries the declared tags).

## How to validate / use

```sql
declare schema main { table t { id integer primary key, qty integer, constraint chk check (qty > 0) } }
apply schema main;
-- later, edit ONLY the body, keep the name:
declare schema main { table t { id integer primary key, qty integer, constraint chk check (qty >= 0) } }
apply schema main;   -- differ emits: ALTER TABLE t DROP CONSTRAINT chk; ALTER TABLE t ADD constraint chk check (qty >= 0)
```

Use cases the reviewer should exercise / sanity-check:
- **CHECK** body edit → new predicate enforced forward (see CHECK gap below re: existing rows).
- **UNIQUE** column-set / conflict change → drop tears down covering index, add rebuilds and
  re-validates existing rows; violating data aborts the apply.
- **FK** action / referenced change → re-add re-validates child rows (gated by `pragma foreign_keys`).
- **Disambiguation** — tag-only ⇒ `SET TAGS` (no drop/add); body-only ⇒ drop/add (no `SET TAGS`);
  both ⇒ drop/add carrying declared tags.
- **No spurious churn** — `declare == actual ⇒ empty diff`, idempotent re-apply (default-elided
  forms: default mask / ABORT / RESTRICT).

## Known gaps / honest flags (please scrutinize)

1. **CHECK body change does NOT re-validate existing rows.** `ADD CONSTRAINT CHECK`
   (`runtime/emit/add-constraint.ts runAddCheck`) applies the CHECK in place and enforces forward
   only — it never re-checks existing rows. So a CHECK body change is **forward-enforcing**; an
   existing row that violates the new predicate is *not* re-validated. This is a **pre-existing**
   limitation of the CHECK add path (UNIQUE/FK re-validate via `module.addConstraint`; CHECK does
   not), not introduced here. The ticket's edge-case text optimistically said CHECK "re-validates";
   the engine does not. Tests assert the honest forward-enforcement behavior and call this out.

2. **A body-change drop+recreate is NOT atomic across the two statements on the memory backend.**
   A body change emits two DDL statements (DROP then ADD), and `runBatchedMigrationLoop`
   (`runtime/emit/schema-declarative.ts`) runs each via `_execWithinTransaction`; memory schema
   mutations commit immediately and are not rolled back as a unit. So if the re-ADD fails
   re-validation (e.g. UNIQUE narrowing over violating data), the **DROP has already removed the old
   constraint** — the table ends up with neither the old nor the new constraint. (The single-statement
   prereq ADD is "atomic" only because the module validates *before* mutating.) Store modules with
   `beginSchemaBatch`/`endSchemaBatch` *may* fold the migration into one substrate commit and be
   atomic — not verified here. **This is the most important thing to review**: is non-atomic
   drop+recreate acceptable as the product's body-change realization, or should the differ/apply
   guarantee atomicity (out of scope here; would need transactional schema mutation or a
   single fused "redefine" primitive)? The spec/sqllogic tests assert only "apply aborts + data
   survives", NOT "old constraint restored". Worth a follow-on ticket if atomicity is required.

3. **FK referenced-column canonicalization corner.** `definition` emits the referenced column list
   iff the stored constraint has `referencedColumnNames`. Changing a declaration from
   `references parent` to `references parent(pid)` (when `pid` is the PK — semantically equal) will
   register as a body change and churn a (harmless) drop+recreate. Documented; not special-cased.

4. **FK `deferrable` excluded from `definition`.** A change to only the deferrable clause is not
   detected as a body change. Deferrable representation differs between AST (`deferrable` /
   `initiallyDeferred`) and schema (`deferred`), and it is a niche; excluding it keeps
   canonicalization robust. Documented limitation.

5. **Unnamed constraints out of scope.** Detection keys off names; unnamed-constraint body changes
   are not individually addressable (noted in docs, not silently claimed as covered).

6. **Test isolation only.** `41.9-...sqllogic` was verified under both `yarn test` and
   `QUEREUS_TEST_STORE=true` *in isolation* (grep). The full `yarn test:store` suite was NOT run
   (slow; not required by the ticket). The full memory suite **was** run green.
