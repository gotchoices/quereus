## What shipped

Multi-source **INSERT** through a two-table key-preserving inner-join view, plus the
per-row **shared-surrogate mutation envelope** it requires. Design source:
`docs/view-updateability.md` § Inner Join (Inserts), § Mutation Context, § Implementation Surface.

### The envelope mechanism (the load-bearing new surface)

The shared join key is not a view column, so an insert must produce a value present at
neither the user row nor the inserted view row, shared across both base tables. The
chosen design is **pre-evaluate at the envelope** (option 1 of the implement ticket; the
RETURNING-threading option was rejected — it couples to RETURNING-through-view and loses
row↔value correlation):

- `ViewMutationNode` gained an optional `envelope: { source, descriptor, mint? }`
  (`planner/nodes/view-mutation-node.ts`).
- `emitViewMutation` (`runtime/emit/view-mutation.ts`) **materializes the augmented source
  once** into an array *before* any base op runs, appends the minted key per row, and
  stashes the rows in `rctx.tableContexts` under the descriptor. Each base op reads them
  back through a new leaf `EnvelopeScanNode` (`planner/nodes/envelope-scan-node.ts` +
  `runtime/emit/envelope-scan.ts`) — the same working-table-in-context pattern recursive
  CTEs use. Because every base-op sub-program runs with the **same `rctx`**, they all read
  the identical rows: one generation per row, threaded across the fan-out, no
  "which-branch-generates-first" question. The context entry is removed in `finally`.
- Sharing is per-execution-safe: `tableContexts` is `forked` (each parallel fork owns its
  copy), and the descriptor key is unique per plan node — mirroring `recursive-cte.ts`.
  `view-mutation.ts` was added to `TABLE_CONTEXTS_MUTATION_ALLOWLIST` in
  `test/runtime/fork-contract.spec.ts`.

### Surrogate generation

The shared key per row comes from one of two paths, decided in `analyzeMultiSourceInsert`
(`planner/mutation/multi-source.ts`):

- **Directly supplied** — a supplied view column maps (via `updateLineage`) to a join-key
  base column. Its value threads to both sides (no minting, no `mint`).
- **Minted** — `integer-auto` / `per-row`: `seed + ordinal` (1-based), where
  `seed = coalesce(max(anchor.key), 0)` is evaluated **once** before fan-out (observes the
  pre-mutation state). The anchor is the FK-parent (else the left source).

### Decomposition + defaults + ordering

- One base insert per side, re-planned through the ordinary `buildInsertStmt` via a new
  optional `preBuiltSource` param (`planner/building/insert.ts`) that injects a projection
  over the `EnvelopeScanNode` — so **all** constraint / conflict / FK / base-default
  machinery is reused verbatim.
- Sides are FK-parent-before-FK-child (reuses `orderSides`).
- Per-side omitted base columns fall to their declared base defaults (base pipeline). A
  `not null` base column with no value raises `no-default`; a computed target column raises
  `no-inverse`.
- `buildViewMutation` routes an inner-join insert to `buildMultiSourceInsert` *before*
  `propagate` (which still rejects multi-source insert defensively — now unreachable on the
  supported path). Decomposition-backed logical tables keep their own (deferred) path.

## How to exercise / validate

```sql
-- Worked example (inner-join form): one minted rid per row, shared across both bases.
create table u_core (rid integer primary key, name text);
create table u_contact (rid integer primary key, email text);
create view appuser as select c.name as name, k.email as email
  from u_core c join u_contact k on k.rid = c.rid;
insert into appuser (name, email) values ('Ada','ada@x.io'), ('Lin','lin@x.io');
-- u_core (1,'Ada'),(2,'Lin'); u_contact (1,'ada@x.io'),(2,'lin@x.io')

-- Directly-supplied key (view exposes the join key): supplied value threads to both sides.
-- Mint above existing keys + FK-parent-before-child (see 93.4 § Phase 2b (b)).
-- no-default / no-inverse rejections (93.4 § Phase 2b (d),(e)).
```

## Known gaps / honest limits (treat tests as a floor)

- **Inner join only.** The doc's worked example is a `left join`; the insert path requires
  an INNER join in v1 (the test uses the inner-join form). Outer-join / optional-member
  insert is deferred and still rejected.
- **`integer-auto` mint only.** A `declared-default`-expression generator (`default
  next_rid()`) and non-integer surrogates are deferred — a non-supplied non-integer key
  raises `no-default`. The mint is computed in the emitter (JS `seed + ordinal`); it is
  deterministic-given-context (seed from pre-state, deterministic ordinals) and so replays
  identically, but the per-row values are **not** explicitly recorded in a context
  structure — a genuinely non-deterministic generator would need that recording before it
  could ride this envelope.
- **Seed reads only the anchor side's key.** `max(anchor.key)`; a pre-existing key on the
  *non-anchor* side that collides with a minted value is caught by the base PK/constraint
  pass, not pre-empted. Probe: seed the non-anchor table with a higher key and insert.
- **`lens-multi-source-put-fanout` (`decomposition.ts`) is NOT wired** to the envelope yet —
  it still raises `unsupported-decomposition-insert`. The envelope is built as a reusable
  surface (`ViewMutationNode.envelope` + `EnvelopeScanNode`) the follow-up can consume, but
  this ticket did not thread it through `propagateDecomposition` (optional/EAV members,
  `SharedKeyGenerator` strategies). Confirm the surface is shaped to thread, not reinvent.
- **`default_for` is not honored for multi-source insert** (single-source still is).
- **Lightly tested paths:** SELECT-source inserts (implemented in `buildEnvelopeSource` but
  only VALUES is covered by tests), `with context` threaded through a multi-source insert,
  `ON CONFLICT` (or ignore/replace) on the per-side base inserts, and composite-PK sides
  (should `no-default` on the uncovered PK columns). Worth a reviewer probe.
- **RETURNING through a join insert** is rejected (`rejectReturning`), consistent with the
  rest of the substrate.
- `> 2`-table joins, self-joins, `select *` join bodies, composite/expression join keys,
  cross-source values: rejected (same gates as update/delete).

## Suggested review focus
1. The `emitViewMutation` materialize-then-fan-out ordering and `tableContexts` lifecycle
   (leak/finally correctness; interaction with savepoints inside the per-side base ops).
2. `preBuiltSource` in `buildInsertStmt` — does any other facet of an insert (RETURNING,
   generated columns, upsert) interact badly when the source is injected?
3. `analyzeMultiSourceInsert` routing/anchor/`no-default` correctness on adversarial shapes
   (swapped projections like `ax_swap`, composite PK, key column that is also a value).
4. Whether the envelope surface is genuinely consumable by `decomposition.ts` insert.
