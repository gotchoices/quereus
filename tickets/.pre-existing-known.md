## ORDER BY aggregate subquery over a GROUP BY query — "Scalar subquery returned more than one row"

- **Symptom:** a grouped query whose ORDER BY contains an aggregate subquery
  throws, e.g.
  `select o.k, count(*) as cnt from o group by o.k order by (select count(*) from c), o.k`.
- **Root cause:** planner scope-leak — the ORDER BY subquery's inner aggregate is
  resolved against the *outer* aggregate's output scope and bound to the outer
  aggregate's alias, so the subquery reads a column instead of aggregating. It is
  a build-phase resolution bug, independent of decorrelation: reproduces with an
  **uncorrelated** subquery and with **all** decorrelation rules disabled.
- **Tracked by:** `fix/order-by-aggregate-subquery-scope-leak` (in-flight).
- **History:** originally recorded in `.pre-existing-error.md` by
  `3-feat-decorrelate-exists-in-select-list`, which predicted the in-flight
  `feat-decorrelate-scalar-subquery-order-by` run would fix it. That run
  determined the bug is out of its scope (its ORDER BY decorrelation cannot fix
  an orthogonal build-phase bug), removed the one `.sqllogic` line that exercised
  it (replaced with a NOTE), and filed the fix ticket above. Do not re-report.
