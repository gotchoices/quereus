## Blocked — why

This ticket's premise is that nested membership "builds on the recursive per-branch fan-out
from `set-op-membership-write`" and only adds **per-leaf addressing**. Empirical verification
(project ts-node loader, in-memory tables) shows the two layers the ticket assumes already
exist do **not**, so none of its distinctive deliverables (subtree writes, product
coordinates, multi-target fan-out, contradiction rejection) is reachable. The files this
ticket lists (`set-op.ts`, `set-operation-node.ts`, `update-lineage.ts`,
`rule-predicate-contradiction`) cannot deliver the gate without first building two
foundations that are out of this ticket's scope.

### Evidence (verified, not theorized)

**1. The grammar cannot express the acceptance fixtures.** The product-coordinate fixture
(view `U4`), the contradiction fixture (two subtrees `where id=1` / `where id=2`), and *any*
structurally-parallel sibling-subtree shape require a **parenthesized left-leg compound**,
which the parser rejects:

```
"(select id,x from A union exists left as inA, exists right as inB select id,x from B)
   union exists left as inX, exists right as inY
 (select id,x from A union exists left as inA, exists right as inB select id,x from B)"
  → PARSE FAILED: Expected statement type (SELECT, …), got '('. (line 1, col 1)

"(select 1) union (select 2)"  → PARSE FAILED (same — pre-existing limitation)
```

This matches the in-repo note at `test/property.spec.ts:2173`. → foundation
**`parenthesized-compound-set-op-legs`** (a SQL-language-design decision).

**2. The planner cannot even plan a flagged subtree operand.** The only nested shape that
*parses* — right-nested with flags at both levels — fails to create:

```
"select id,x from A union exists left as inA, exists right as inSub
 (select id,x from B union exists left as inB, exists right as inC select id,x from C)"
  → CREATE FAILED: SET operation column count mismatch: left has 2, right has 4
```

The inner subtree's flag columns (`inB`,`inC`) inflate its arity to 4; the outer
`SetOperationNode` arity check (`set-operation-node.ts:46`) rejects 2-vs-4.

**3. Subtree-flag writes and fan-out through a subtree operand throw at runtime.** A
flag-less subtree operand *does* read correctly (subtree-flag read already works generically),
but every write that reaches the subtree branch fails:

```
create view V as select id,x from A union exists left as inA, exists right as inSub
                 (select id,x from B union select id,x from C);   -- creates + reads OK
update V set inSub = true  …  → cannot write through view '__setop_right':
                                 view body operator 'SetOperation' is not updateable in phase 1
update V set x = 99 …  (data fan-out) → (same throw — fan-out hits the unwritable subtree)
update V set inA = false …  (leaf branch) → OK
```

→ foundation **`nestable-flagged-set-ops`** (data-column-only arity alignment + a recursively
writable subtree).

### What already works (no new code needed for it)

- **Subtree-flag READ** over a flag-less subtree: `inSub ≡ tuple ∈ (B∪C)` reads correctly,
  and the generic `set-op-branch` `existence` lineage/static surface already covers it
  (`set-operation-node.ts` `membershipLineage` is leaf/subtree agnostic). So the ticket's
  "subtree flag read" half is effectively already shipped by `set-op-membership-read`.

### What unblocks this ticket

1. `nestable-flagged-set-ops` lands → a flagged subtree can nest and is recursively writable
   (unblocks subtree-flag writes + nested fan-out; exercised by the right-nested form,
   independent of the grammar work).
2. `parenthesized-compound-set-op-legs` lands **and the human signs off** on the grammar
   widening → the parallel-sibling-subtree fixtures (product coordinates, multi-target,
   contradiction) become expressible.

When both land, re-pick this ticket: the per-leaf addressing, product/sum derivation,
multi-target fan-out, and contradiction rejection below are the actual remaining work and are
well-scoped on top of those foundations.

## Why (original design intent — retained)

`set-op-membership-write` covers binary (non-nested) membership writes. A nested set-op
(`((A ∪ B) ∪ (C ∪ D))`) with flags at multiple levels is well-defined because each flag is just
`tuple ∈ <that operand's relation>` — a leaf flag gives base-relation provenance, a subtree flag
gives subtree provenance, every one an independent dedup-safe boolean function of the result tuple.
This ticket completes the model: subtree-flag reads/writes, the **product coordinate system** that
falls out of reusing leaf names across structurally-parallel subtrees, multi-target fan-out, and the
single rejection rule (provable contradiction).

What this ticket adds (on top of the foundations above) is the **per-leaf addressing**: deriving a
flag as `tuple ∈ <union of like-named leaves>`, routing a conjunction of coordinates to the
addressed leaf, fanning a multi-target write out to every co-satisfiable leaf, and rejecting a
contradictory coordinate set.

## Subtree flags

A flag may name a **subtree** branch (an immediate operand that is itself a `SetOperationNode` or a
filtered sub-union), not only a leaf base relation. `set-op-membership-read` already derives the flag
as `tuple ∈ <that operand's data relation>` regardless of whether the operand is a leaf or a subtree
(the probe is against the operand's output rows). This ticket ensures:

- **Read**: a subtree flag `inX` reads `tuple ∈ R1` (R1 the subtree) — the probe is against R1's
  materialized output, dedup-safe. *(Already works — see "What already works".)*
- **Write**: `set inX = true/false` recurses through R1's own `put` (insert/delete fan-out into the
  subtree, which itself recurses to its leaves). *(Needs `nestable-flagged-set-ops`.)*

## Reused flag names factor the address (product vs sum)

When sibling subtrees are **structurally parallel** (the same leaf relations under different filters
— `R1 = (A∪B) where id=1`, `R2 = (A∪B) where id=2`), reusing `inA`/`inB` at both inner levels while
naming the subtrees `inX`/`inY` yields a **product coordinate system**: `{inX,inY} × {inA,inB}` is a
2×2 grid and each leaf is the **conjunction** of one coordinate per axis — `(B where id=1) = inX ∧ inB`.

The derivation is uniform: a top-level (subtree-shared) flag is `tuple ∈ <the union of all
like-named leaves>` (so `inA ≡ tuple ∈ (A where id∈{1,2})`). The flags are **named, writable handles
on the branch predicates**; the σ predicate each subtree carries (the `guard` on the flag's
`existence` site) is what addresses the leaf:

- **Single-target** — one-hot coordinates (mutually-exclusive subtree predicates) make the
  conjunction pin **one** leaf. `inX ∧ inB` over `id=1` (X) and base B routes the write to
  `B where id=1`. One-hot is **how you opt into single-target**, not a soundness requirement.
- **Multi-target** — setting several **co-satisfiable** coordinates true fans the write out to *every*
  leaf consistent with the flags (`inX=inY=inA=inB=true` upserts the row at all consistent leaves) —
  the predicate-honest fan-out, a feature.

**Reused names express a product** (factored coordinates, structurally-parallel subtrees); **distinct
names express a sum** (tagged tree positions, structurally-different subtrees, e.g. `(A∪B)∪(C∪D)`).
The engine chooses neither — it derives `flag ≡ tuple ∈ <union of like-named leaves>` and lets
conjunction address the leaf. Implementation: when assembling the per-branch fan-out in `set-op.ts`,
collect all `existence` sites sharing a flag name and union their branch guards for the read
derivation; for a write, intersect (conjoin) the set-true coordinates' guards to compute the target
leaf set.

> NOTE: every parallel-sibling-subtree example in this section requires
> `parenthesized-compound-set-op-legs` to be expressible at all.

## The one rejection: provable contradiction

A multi-target write needs the set coordinates **co-satisfiable**. Reuse the optimizer's existing
predicate-contradiction detection (`rule-filter-contradiction` + `analysis/sat-checker.ts`, from
`2-optimizer-predicate-contradiction-detection`, now complete) to test the conjunction of the
set-true coordinates' guards:

- Two **values of one attribute** are mutually exclusive (`id=1` vs `id=2`, `color='red'` vs
  `'blue'`) — `inX ∧ inY` over such subtrees is a `predicate-contradiction` → **reject**.
- Two **different attributes** co-exist (`color='red'` ∧ `size='large'`) — co-satisfiable, a legal
  multi-target write; no rejection.

This is the **only** rejection. One-hot is not required; it is just how single-target is expressed.

(Note: the rule lives at `rules/predicate/rule-filter-contradiction.ts` driving
`analysis/sat-checker.ts` — the ticket's original `rule-predicate-contradiction.ts` path does not
exist. The reusable co-satisfiability primitive is `checkSatisfiability` in the sat-checker.)

## Tests (acceptance gate: `test/property.spec.ts` § View Round-Trip Laws → set-op membership family)

Use the parent design's product-coordinate fixture (NOTE: unparseable until
`parenthesized-compound-set-op-legs` lands):

```sql
create view U4 as
      ((select id, x from A where color = 'red')
         union exists left as inA, exists right as inB
       (select id, x from B where color = 'red'))
  union exists left as inX, exists right as inY
      ((select id, x from A where size = 'large')
         union exists left as inA, exists right as inB
       (select id, x from B where size = 'large'));
```

- **Subtree + leaf read agreement.** Leaf-flag and subtree-flag reads agree with `tuple ∈ <union of
  like-named leaves>`: `inA ≡ tuple ∈ (A where color='red' or size='large')`, `inX ≡ tuple ∈ the red
  subtree`, etc. — assert row-by-row.
- **Single-target write (conjunction).** `set inX = true, inB = true, inY = false, inA = false`
  targets exactly the B/red leaf → `B where color='red'` (PutGet: the row appears there and nowhere
  else).
- **Multi-target write (fan-out).** `inX=inY=inA=inB=true` over the co-satisfiable axes
  (color vs size) upserts the row into **both A and B as red AND large** (all four consistent
  leaves); the row appears in every targeted base.
- **Contradiction rejection.** A contradictory coordinate over **mutually-exclusive** subtrees
  (`inX ∧ inY` where X=`id=1`, Y=`id=2`) rejects with `predicate-contradiction` (not a silent
  partial write). Build a `U` whose two subtrees are `where id=1` / `where id=2` for this case.
- **Key Soundness.** A nested distinct union carrying flags at both levels stays keyed by its data
  columns; no flag (leaf or subtree) ever appears in a claimed key (Invariants 1–2, all depths).
- **Subtree-flag read (parseable today).** Even before the foundations, add/keep a regression for
  the right-nested flag-less-subtree read (`inSub ≡ tuple ∈ (B∪C)`) — the one nested behavior that
  works now.
