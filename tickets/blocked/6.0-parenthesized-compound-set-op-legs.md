**BLOCKED — design / human sign-off.** SQL language-surface decision: whether to widen the grammar to accept parenthesized compound set-op legs (e.g. `(A union B) union (C union D)`), matching SQLite. *Unblocks when:* a human approves widening the SQL surface — the recommended implementation (generalize `parseQueryExpr` to accept a leading `(`-wrapped query expr) is in the body below.

## Problem

A set operation's **left** operand must be a bare query keyword. `parseQueryExpr`
(`parser.ts:230`) and `statement()` (`parser.ts:322`) switch on the first token's keyword
(`SELECT` / `VALUES` / `WITH` / `INSERT` / `UPDATE` / `DELETE`); a leading `(` falls to the
default branch and throws `Expected statement type (SELECT, INSERT, UPDATE, DELETE, VALUES,
CREATE, etc.), got '('`.

A set operation's **right** operand *does* accept an optional `(` (the `usedParen` path in
`selectStatement` / `continueSelectAfterFrom`), so a parenthesized nested compound is legal
only on the right.

Net: the grammar can express `A union (B union C)` (right-nested) but **not**
`(A union B) union (C union D)` (parallel siblings) nor even `(select 1) union (select 2)`.

Verified empirically (project ts-node loader):

```
"(select 1) union (select 2)"
  → PARSE FAILED: Expected statement type …, got '('. (line 1, col 1)
"(select id from A union select id from B) union (select id from C union select id from D)"
  → PARSE FAILED: Expected statement type …, got '('. (line 1, col 1)
"select 1 union (select 2 union select 3)"   → PARSED OK (right-nested only)
```

The in-repo comment at `test/property.spec.ts:2173` already documents this:
> "The engine's compound grammar does not accept a parenthesized left leg at the outer level
> (a pre-existing limitation — `create view v as (select 1) union (select 2)` fails
> identically)…"

## Why it matters

The set-op membership product-coordinate model (`set-op-membership-nested`) requires two
**structurally-parallel sibling subtrees** as the two operands of an outer set operation
(`(A∪B) union[inX,inY] (A∪B)`), so that reused leaf flag names (`inA`/`inB`) factor a product
coordinate system across the axes. The contradiction-rejection fixture likewise needs two
sibling subtrees (`where id=1` / `where id=2`). Neither is expressible without a parenthesized
left leg. This ticket is the grammar foundation those fixtures depend on.

## Desired behavior

A parenthesized query expression is accepted as a relational atom wherever a set-operation
operand or a view/top-level body is expected, and a following `union`/`intersect`/`except`
operator chains onto it as the left leg. Concretely:

- `(<query-expr>)` parses to the inner `QueryExpr` (a `SelectStmt` that may itself carry a
  `compound`), with the parentheses consumed as grouping (no AST node needed beyond the inner
  statement — or a thin grouping marker if round-trip fidelity demands it).
- `(<compound>) <op> <rightleg>` parses as a compound whose left leg is the parenthesized
  inner compound. Associativity/precedence must match the right-nested semantics already in
  place (the inner compound binds tighter than the outer operator).
- The membership clause (`exists <branch> as <name>`) attaches to the **outer** operator as
  today; an inner parenthesized compound carries its own `compound.existence`.
- Round-trips through `ast-stringify` (re-emit must re-parse to the same shape — see the
  existing `emit-roundtrip.spec.ts` compound cases). Decide whether the stringifier always
  parenthesizes a compound left leg for stability.

## Design considerations (for the promoting human)

- **Scope of the change.** The cleanest contained option is to extend `parseQueryExpr` (and
  the view-body / FROM-subquery entry it shares) to accept a leading `(` that wraps a query
  expr, rather than touching top-level `statement()` (which would also let bare
  `(select …)` statements through — possibly desirable, possibly not). The view body
  (`createStatement` → `parseQueryExpr`) is the path that matters for the membership model.
- **No janky one-off.** Per AGENTS.md ("No half-baked janky parsers; use a full-fledged
  parser or better, brainstorm with the dev"), this should be a clean grammar generalization,
  not a special-case hack for the membership fixture. SQLite accepts parenthesized compound
  operands generally; matching that is the principled target.
- **Interaction with the `usedParen` right-leg path** — unify the two so left and right legs
  share one "parenthesized compound operand" production.
