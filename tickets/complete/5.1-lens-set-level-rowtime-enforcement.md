## What this delivered

A **lock-and-document** ticket: no enforcement logic changed. The row-time
set-level class (a logical `unique` / PK whose key is answered by a **basis
`UNIQUE` + non-stale covering MV**) is enforced end-to-end with **no dedicated
lens code path**:

- the prover (`classifyKeyConstraint` in `src/schema/lens-prover.ts`) classifies a
  key `enforced-set-level{mode:'row-time'}` **only when** the body does not already
  *prove* the key AND a basis `UNIQUE` + non-stale covering MV answers it
  (`findBasisCoveringStructure`); and
- the single-source spine re-plans the lens write to that **basis table by name**
  (basis-column terms, renames handled, conflict action preserved), so the basis
  INSERT/UPDATE hits the memory layer's `checkUniqueConstraints` →
  `checkUniqueViaMaterializedView` — an O(log n) existence lookup that honors
  `ABORT` / `IGNORE` / `REPLACE`.

`collectLensSetLevelConstraints` returns `[]` for row-time (only `commit-time`
emits); `hasCommitTimeSetLevelObligation` is false for row-time; and
`rejectLensSetLevelConflictResolution` declines to reject row-time (it gates only
on the commit-time obligation). All three verified, behaviorally and by direct
obligation inspection.

The implement stage flipped `docs/lens.md` (maturity blockquote, Set-level bullet,
`lens-prover.ts` status tracker) and the module/function doc comments in
`lens-enforcement.ts`, `lens-prover.ts`, and `view-mutation-builder.ts` from
"pending" → "delivered, via the basis covering-MV path, not a new lens code path".

### Checked — correctness of the "no new code" claim (no findings; claim holds, and holds *beyond* what was tested)
Empirically confirmed via a throwaway diagnostic that the basis covering-MV path
carries row-time enforcement correctly for the two corners the implementer flagged
as untested:
- **Composite row-time key** (`unique(a,b)` + covering MV over both): classifies

Both work with no lens code. Single-column unique, rename override, replace, ignore,
abort, update-creates-duplicate, and null-distinct were already covered and verified.
**No correctness gap found** — no `fix/` ticket warranted.

### Found & fixed inline (minor) — the regression suite did not actually pin "row-time"
The crux finding. **Every** row-time assertion in the implement suite
(`collectLensSetLevelConstraints == []`, `hasCommitTimeSetLevelObligation == false`,
and even REPLACE/IGNORE succeeding) is **also satisfied by a `proved` key** — so the
suite could silently exercise a proved key while claiming to test row-time. Direct
obligation inspection exposed this concretely:

- The **re-keyed PK test was mislabeled**: its deployment (basis `unique(code)` NOT
  NULL + covering MV, logical PK on `code`) classifies **`proved`, not `row-time`**.
  A logical PK is NOT NULL, so a matching basis `UNIQUE`+`NOT NULL` *proves* the key
  outright and pre-empts row-time. A row-time PK is in fact **unreachable** for a
  faithful single-source projection (basis UNIQUE+NOT-NULL ⇒ proved; no basis UNIQUE
  ⇒ commit-time, which the commit-time suite already covers). The test still passed
  because proved keys also enforce detection + replace through the basis — but its
  docstring/comments asserting "classifies row-time, backed by the basis UC's
  covering-MV enforcement path" were **false**.

Fixes applied to `packages/quereus/test/lens-enforcement.spec.ts`:
- Added a `setLevelModes(slot)` helper that reads the slot's actual obligation modes,
  so a test can pin `row-time` vs `commit-time` vs proved/vacuous (`[]`) directly.
- Added `setLevelModes(...) == ['row-time']` assertions to the single-column,
  rename-override, and composes-with-check row-time tests; added a
  `['commit-time','row-time']` assertion to the composition-caveat test (both classes
  genuinely co-exist on the slot).
- Reworked the re-keyed-PK test to assert it is **`proved`** (no set-level obligation),
  retitled it honestly, and documented why a row-time PK is unreachable for a
  single-source projection. It still verifies detection + `or replace` resolve
  end-to-end through the basis key.
- Added two new tests pinning the previously-uncovered (but working) corners: a
  **composite row-time key** and an **`on conflict … do update` upsert**.

Suite went 43 → 45 tests; all green.
