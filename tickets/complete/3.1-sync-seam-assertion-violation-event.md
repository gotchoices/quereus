# Complete: surface inbound assertion violations to the host; converge instead of diverging

## Summary of the shipped change

The sync layer now wires the engine seam's **report mode**
(`Database.ingestExternalRowChanges(batch, { assertionFailureMode: 'report' })`, landed
by prereq `sync-seam-assertion-report-mode`) into a host-facing detect-and-notify signal:

- **`store-adapter.ts`** — the single end-of-invocation incremental seam call passes
  `assertionFailureMode: 'report'` alongside `applyForeignKeyActions`, and copies the
  returned `assertionViolations` into `ApplyToStoreResult.assertionViolations`. Bootstrap
  flushes skip the seam, so they carry none.
- **`protocol.ts`** — `ApplyToStoreResult.assertionViolations?: AssertionViolation[]`.
- **`events.ts`** — `AssertionViolationEvent { assertion, samples }`,
  `SyncEventEmitter.onAssertionViolation`, listener set, `emitAssertionViolation`; exported
  from the package index.
- **`admission.ts`** — `applyDataToStore` emits one `onAssertionViolation` per returned
  violation on the **success** branch (after the whole-batch-throw catch and after
  `throwIfApplyErrors` passes). Convergence is not aborted; the metadata commit proceeds.
  Every ingress modality routes through `applyDataToStore`, so all paths get the event.

Net: the old "throw → uncommitted metadata → idempotent retry suppresses the seam batch →
MV/watch for the violating row never re-driven → silent divergence" chain is gone. The
violating row's MV maintenance and watch capture land on the **first** apply; the host is
notified of the broken local invariant.

- Orthogonality of the two failure shapes is correct: `throwIfApplyErrors` runs **before**

**One major finding — filed, not fixed.** `tickets/backlog/sync-assertion-violation-dropped-on-mixed-error.md`:
when a single apply batch carries **both** a per-change storage error **and** an assertion
violation on the successfully-applied portion, the report-mode seam commits the violating
data but `throwIfApplyErrors` aborts **before** the emit, dropping the event — and on the
idempotent retry the violating table is suppressed from the seam batch (and not in the
commit's changed-base set), so `runGlobalAssertions` never re-evaluates it and the host is
**permanently never notified**. Data stays consistent (no MV divergence) — only the
notification is lost, in the narrow intersection of two orthogonal failure modes. Deferred
because the fix is a deliberate design decision (emit-before-abort vs. persist-and-re-emit)
that changes the abort-vs-emit ordering contract in `applyDataToStore`, not a reviewer's
inline tweak.

**Not changed (with reasons):**
- *Listener-exception isolation* — `emitAssertionViolation` runs synchronously inside the
  apply, before `commitMetadata`; a throwing host listener would propagate and trip the
  abort path. This matches every other emitter in `events.ts` (none guard listener
  exceptions); event-emitter isolation is a broader, pre-existing concern, out of scope
  here.
- *`result.assertionViolations` set only when non-empty* — leaves the field `undefined`
  otherwise; `admission.ts` guards with a truthiness check. Correct and idiomatic for an
  optional field; no change.
