# Constraint BODY change → DROP + ADD CONSTRAINT in the declarative differ — completed

The declarative differ now detects a named constraint whose **name is unchanged but whose body
changed** (edited CHECK expression, changed FK action / referenced table / columns, changed UNIQUE
column set / `ON CONFLICT`) and realizes it as **drop-old + add-new**. Each `CatalogTable.namedConstraints`
entry gained a required canonical `definition` body fragment (name + tags excluded); both the
declared-AST side and the actual-catalog side render through the single
`constraintBodyToCanonicalString` path so fragments are byte-comparable. Tags stay out of
`definition`, so a tag-only change keeps the in-place `ALTER CONSTRAINT … SET TAGS` path. Body-change
recreates are excluded from the `require-hint` add/drop count, and a rename+body change suppresses the
RENAME in favor of the drop+recreate.

Implementation shape is as described in the implement handoff. The review below records what was
independently checked, found, and done.

### Major — new ticket filed
1. **Column/table rename referenced by a named constraint churns a redundant drop+recreate.**
   Empirically confirmed during review (CHECK and UNIQUE): renaming a column (or, by inspection, the
   parent table under an FK) that a named constraint references emits a spurious `DROP CONSTRAINT` +
   `ADD CONSTRAINT` **in addition to** the rename, because the actual-side `definition` is rendered
   with the *current* identifier and the declared-side with the *new* one, so the bodies compare
   unequal — even though the rename pass already rewrites the constraint
   (`propagateColumnRename`/`rewriteTableForColumnRename`). It converges and `apply` succeeds (RENAME
   orders before DROP/ADD), so it is not an immediate correctness failure, but for UNIQUE/FK it wastes
   a full re-validation scan on a metadata-only rename and — given the non-atomic drop+recreate (gap
   #2) — opens a new failure window on a previously-safe operation. This is **newly introduced** by
   this ticket (no body comparison existed before). Beyond a review-pass inline fix (the clean fix
   rewrites the actual constraint through the detected renames at the AST level before comparing).
   Filed as `tickets/fix/constraint-body-change-rename-churn.md`.

### Minor — fixed in this pass
1. **Docs claimed CHECK body changes "re-validate existing rows" — they do not.** Both `docs/sql.md`
   (§2.7) and `docs/schema.md` (Constraint body-change detection) asserted that re-creation
   re-validates existing rows "against the new rule", lumping CHECK with UNIQUE/FK. Per the engine's
   actual behavior (handoff gap #1, confirmed against `runAddCheck`), a CHECK re-add is
   **forward-enforcing only** and never re-scans existing rows. Corrected both docs to distinguish
   UNIQUE/FK (re-validate, abort on violation) from CHECK (forward-only), and to note the
   non-atomicity of the two-statement drop+recreate. Also added the rename-churn caveat (major
   finding above) to `docs/schema.md` with a pointer to the new ticket. (Docs-only; no code change.)

- **Rename+body precedence and double-drop avoidance**: a name-matched body change is skipped in the

- **require-hint accounting**: `pureCreateCount`/`pureDropCount` correctly exclude body-change

- **Canonicalization fidelity / no spurious churn**: default-form bodies (default CHECK mask, `ABORT`,

  idempotence (`diff schema => []`) confirm no churn. The FK `references parent` vs `references
  parent(pid)` corner (gap #3) and the excluded `deferrable` channel (gap #4) are genuine, narrow,

- **Atomicity of drop+recreate (gap #2)**: the most-flagged concern. Confirmed the two statements are
  not atomic on the memory backend (`runBatchedMigrationLoop` commits each separately); the

  "redefine" primitive is out of scope); the non-atomicity is now documented more loudly in both
  docs. Not separately ticketed — the rename-churn ticket references it as the reason that path's
  redundancy is more than cosmetic.
