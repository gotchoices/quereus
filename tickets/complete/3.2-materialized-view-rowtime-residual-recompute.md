# Residual-recompute arm (single-source aggregates) — COMPLETE

## What shipped

The `'residual-recompute'` arm of `MaintenancePlan` (previously a loud `INTERNAL`
stub) is wired for **single-source aggregate bodies**: a
`create materialized view mv as select g1,…, agg(…) from T [where P] group by g1,…`
over bare group-by columns is admitted at create and maintained row-time.

- **Admit** (`buildMaintenancePlan` → `buildAggregateResidualPlan`): non-empty GROUP BY
  of bare source columns, deterministic body, backing PK == group key (every backing-PK
  column resolves transitively to a GROUP BY source column).
- **Compile** (`compileResidual`): body rewritten with `injectKeyFilter(analyzed, T,
  groupColumns, 'gk')`, optimized, emitted, `Scheduler` cached on the plan.
- **Apply** (`applyResidualRecompute`): derive affected group key(s) from the changed
  row (OLD ∪ NEW, deduped on backing key); per key — `delete-key` the old backing slice,
  run the residual bound to the key through a fresh `RuntimeContext` on the live `db`
  (reads-own-writes via T's registered transaction connection), upsert the recomputed
  group row (filtered to rows whose backing key equals the affected key).

The synchronous, in-transaction analogue of the assertion evaluator's residual path
(`database-assertions.ts:executeResidualPerTuple`), sharing the `key-filter.ts`
primitive so the two cannot drift. The 1:1-join arm and lateral-TVF arm are descoped to
`implement/3.3-materialized-view-rowtime-residual-join` and
`3.4-…-prefix-delete-lateral-tvf` respectively; the shape-agnostic kernel built here is
reusable by both.

## Review findings

**Reviewed with fresh eyes against the implement diff (`283d2d8b`), not just the
handoff.** Scrutinized for soundness, ordering, type safety, DRY/modularity, resource
cleanup, error handling, and test coverage. Disposition below.

### Correctness / soundness — checked, no defects

- **Reads-own-writes ordering (the sharpest correctness dependency).** Confirmed
  `maintainRowTimeStructures` is invoked in `dml-executor.ts` *after* `vtab.update!()`
  on every path (insert/update/delete, including REPLACE-eviction and UPSERT-update), so
  the residual scan over T always observes this row's pending write. The residual arm —
  unlike inverse-projection — re-reads T, so this ordering is load-bearing; it holds.
- **Multi-row statement "last write wins".** Maintenance interleaves per-row *after* each
  write, and every recompute reads the full live group, so the last row of a group to be
  written produces the authoritative backing row. Verified a bulk `update t set k=2 where
  k=1` (3 rows, one statement) converges to the correct single group. Sound (O(N)
  residuals — see deferred-perf note).
- **`injectKeyFilter` target matching.** `relKey = ${sourceBase}#${tableRef.id}` (lowercased
  base) matches `tryWrapTableReference`'s construction exactly; `tableRef` comes from the
  same `analyzed` plan so the id is stable. The `rewritten === analyzed` guard correctly
  rejects an un-targetable body.
- **Backing-key / group-key plumbing.** `backingPkSourceCols` (in `backingPkDefinition`
  order) and `bindColumns` (group-by order) are kept distinct and used consistently;
  `buildPrimaryKeyFromValues(keyVals, backingPkDefinition)` mirrors the inverse-projection
  key construction. `residualRowMatchesKey` uses per-column collation.
- **Empty-group filter robustness (the flagged "sharpest edge").** Verified empirically:
  an emptied **all-NULL** multi-column group (`group by a,k`, both NULL, sole row deleted)
  correctly removes its backing row — the optimizer mis-collapse does *not* fire for the
  NULL-safe `IS NULL` filter form, so the residual returns `[]` and the delete-without-
  upsert wins. The `residualRowMatchesKey` invariant is sound regardless of the bug.

### Soundness probes the handoff flagged as untested — all verified correct

- **HAVING aggregates** (`… group by k having count(*) > 1`): a group dropping below the
  threshold returns zero rows → backing row removed; rising to it → row appears. Works.
- **`order by <aggregate>`**: rejected with the group-key-as-backing-key diagnostic
  (`'s' … is not a GROUP BY source column`). Sound.
- **Computed GROUP BY key** (`group by k+1`): rejected (`GROUP BY includes a computed
  expression`).
- **MV-over-MV with an aggregate producer**: a downstream covering-index MV over an
  aggregate backing stays equivalent through insert / key-change / emptied-group cascades,
  tolerating the recomputed group's delete+upsert effective-change pair.

### Notes (no action — accepted as documented)

- **`binding: BindingMode` is populated but unread** by the apply path (which uses
  `bindColumns`/`backingPkSourceCols`). Kept deliberately for substrate parity and the
  follow-on join arm; documented in code. Acceptable.
- **O(N) residual runs on bulk single-statement writes** (no statement-level dedup of
  distinct affected groups). Sound but not affordable at scale; deferred with the
  statement-flush batching boundary (the shipped per-statement batching is connection-
  resolution caching only). Documented in code + docs.
- **Optimizer mis-collapse workaround** (`residualRowMatchesKey` discarding the spurious
  null-key `count=0` row from a constant-pinned multi-column grouped aggregate over empty
  input): the underlying bug is **pre-existing** (reproduces as a plain query; lives in
  `planner/rules/aggregate/`, untouched by this diff) and is filed as
  `fix/optimizer-constant-group-aggregate-empty-input-spurious-row`. No
  `.pre-existing-error.md` written — no test fails (the arm works around it correctly).

### Scope split — verified honest

The two ticket-spec divergences (group binding built directly from the aggregate node
rather than via `extractBindings`, which gates `'group'` on covering a source key; and
per-row rather than per-statement-batched maintenance because the batching prereq shipped
as connection-caching only) are accurate against the landed substrate and correctly
documented. The 1:1-join descope (`3.3-…-residual-join`) is genuine — the lookup-side
change problem and coverage-prover refactor are self-contained hard sub-problems — and
the kernel is genuinely shape-agnostic (`applyResidualRecompute` already handles `'pk'`).
