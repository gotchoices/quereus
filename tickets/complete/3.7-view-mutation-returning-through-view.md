## What shipped

RETURNING through a view (previously rejected everywhere) is supported for the
documented shapes via two mechanisms:

- **Single-source (all three ops, incl. computed/renamed columns)** —
  `single-source.ts` `rewriteViewReturning()` rewrites the clause into base terms
  and attaches it to the rewritten base statement; the base op plans to a relational
  `ReturningNode` which `ViewMutationNode` surfaces (NEW for insert/update, OLD for
  delete; computed view columns re-evaluated post-mutation). MV write-through inherits
  this verbatim.
- **Multi-source inner-join update/delete** — `view-mutation-builder.ts`
  `buildMultiSourceReturning()` builds a re-query of the view restricted to the user
  predicate, captured `pre` (delete) / `post` (update) as a separate
  `ViewMutationNode.returning` child.

`ViewMutationNode` is relational when a `returning` clause is present (`getType()` /
`getAttributes()` forward the result relation; `getChildren`/`getRelations`/
`withChildren` thread the optional `returning` child); `block.ts` admits a relational
view mutation as a block result. Full design: `docs/view-updateability.md` §
`returning` Clauses. Implement-stage commit: `51f55b6d`.

## Review findings

Reviewed the implement diff (`51f55b6d`) with fresh eyes against SPP / DRY /
modularity / type-safety / resource-cleanup / error-handling, then probed the
adversarial shapes the handoff flagged as a "test floor." Build, lint, and the full
`yarn workspace @quereus/quereus test` suite are green (**4164 passing, 9 pending, 0
failing**) before and after the review change.

**Probed and confirmed working** (added throwaway sqllogic probes, since removed):
- **Bound parameters** in the predicate, in a SET value, and inside a RETURNING
  expression itself — all bind correctly through the single-source rewrite.
- **RETURNING expressions combining multiple view columns / arithmetic** (e.g.
  `returning ratev + doubled as r3`) — evaluated against post-mutation NEW.
- **Column-list-renamed view** (`create view cv(aa, bb) as …`) — `returning aa, bb`
  and `returning *` both project the renamed columns.
- **`returning *`** through single-source (incl. computed columns) and through the
  join view — expands to the view's column list in order.
- **Projected-away base column referenceable in RETURNING** (gap #2) — confirmed it
  resolves against the base table and returns the value (null when unset). Consistent
  with the pre-existing WHERE/SET surface. Filed as backlog
  `view-mutation-top-level-colref-view-scope` for an explicit accept/enforce decision
  (the handoff asked for one).

**Found and FIXED inline (minor — silent correctness → loud error):**
- **Multi-source UPDATE that rewrites a column its own WHERE predicate filters on**
  (handoff gap #1). The post-mutation re-query matches by the original predicate, so
  the changed rows no longer match and RETURNING **silently returned 0 rows** (verified
  with a probe: `update rjoin set note='A' where note='a' returning …` → empty instead
  of the updated row). Silent-wrong-result is the worst failure mode, so I added a
  plan-time guard in `buildMultiSourceReturning` that compares the update's assignment
  columns against the columns referenced in its WHERE (`collectColumnRefNames`) and
  **rejects loudly** with the `returning-through-view` diagnostic. Verified it does not
  trip the existing positive multi-source update tests (all predicate on `cid`, a
  non-assigned column) and that a multi-source DELETE (captured `pre`) and an update on
  a non-assigned predicate column still work. Added a regression case to
  `93.2-view-mutation-pending.sqllogic`; updated `docs/view-updateability.md` and the
  builder JSDoc. Correct per-row capture (so this shape works rather than rejects) is
  filed as backlog `view-mutation-multi-source-update-returning-perrow`.

**Checked — clean / accepted as documented:**
- **Child wiring & param order.** `getChildren` (baseOps, returning, envelope),
  `withChildren` cursor-slicing, and the emitter's `params`/`args` indices all agree
  on the same order; `withChildren`'s `unchanged` short-circuit includes the
  `returning` child. Mutually-exclusive `returning` re-query vs relational base op vs
  envelope are dispatched cleanly in `run`.
- **Eager materialization.** The emitter materializes RETURNING rows inside `run` so
  base writes fire regardless of block position (matching the void path's eager
  drain); delete `pre` fully drains the re-query into an array before the base delete
  fires — no stale-cursor / vanished-row hazard.
- **Type widening.** `getType()` now returns `RelationType | ScalarType`; the full
  suite (incl. optimizer/plan/property tests) passing confirms no consumer broke.
  `block.ts` discriminates on `typeClass === 'relation'`.
- **OLD/NEW qualifiers through a view** (gap #3), **`returning *` asymmetry vs
  base-table** (gap #4), **multi-source insert RETURNING rejection** (gap #5),
  **decomposition-lens RETURNING rejection** (gap #6) — all confirmed as the
  documented/intended surface; #5/#6 stay rejected with structured diagnostics and are
  tracked by their own (pre-existing) follow-ups. The `deriveReturningName` table-
  qualified alias (`returning v.col` → name `"v.col"`) is a cosmetic edge of the
  documented view-qualified surface; left as-is.

**DRY note:** the guard's `collectColumnRefNames` is a third copy of a reflective
column-walk (also private in `lens-compiler.ts` / `lens-prover.ts`); a shared util
would touch unrelated files, so it was added locally with a pointer comment — minor,
not blocking.

**Not run:** `yarn test:store` (LevelDB path) — slow and not store-specific;
RETURNING-through-view rides the normal base-table DML path. Deferred to CI/out-of-band
per the handoff (carried forward as an unverified-but-low-risk caveat).
