# Complete: maintained-table attach/detach verbs

Part 2 of the maintained-table chain (after `maintained-table-unified-model`).
Implemented in commit `c04e512e` (`ticket(implement):
maintained-table-attach-detach-verbs`); see that handoff for the full design
narrative. Summary of what shipped:

- **`create table … maintained as <body>`** — declared-shape form: existence /
  `if not exists` check, declared-vs-derived strict shape verification BEFORE
  any catalog registration (`SchemaManager.buildDeclaredTableSchema`), then
  ordinary create + the shared attach core as attach-to-empty (the reconcile
  diff against an empty table IS the fill, in the connection's pending state).
  All-or-nothing: any later failure drops the just-created table.
- **`alter table … set maintained as <body>`** (`attachMaintainedDerivation`) —
  attach / re-attach: capability → strict shape check (names, types,
  nullability both directions, collations, physical PK) → derivation-cycle
  check over the live catalog → one body evaluation + duplicate-derived-key
  reject → catalog flip + maintenance registration (create-time gates) →
  verify-by-diff reconcile (`'replace-all'` keyed diff; identical content =
  zero writes; derived wins) → covering links → cascade of genuine changes to
  consumers → `materialized_view_added`/`_modified`. Gate or reconcile failure
  restores the prior record and plan (fail-safe to stale).
- **`alter table … drop maintained`** (`detachMaintainedDerivation`) —
  catalog-only detach: rows/indexes/module identity stay, staleness leaves
  with the derivation, ONE `materialized_view_removed`, deliberately no
  `table_modified` (consumers stay live).
- **Statement-cache invalidation**: `materialized_view_added/removed/modified`
  now invalidate `'table'` (and legacy `'view'`) plan dependencies, so cached
  direct-write plans don't survive an attach nor write-through plans a detach.
- The mid-migration canonical-DDL breakage
  (`maintained-table-canonical-ddl-not-rehydratable`) is resolved; the
  remaining store residuals live in `mv-table-form-implicit-columns-roundtrip`
  (plan/).

## Review findings

Reviewed by reading the implement diff first (commit `c04e512e`, scoped to the
ticket's `files:` list — the commit also carries unrelated concurrent work, as
the handoff warned), then the live state of every touched source file, the
helpers the new code calls (`attachDerivation`, `registerMaterializedView` /
`unregisterMaterializedView` idempotency + release-before-build ordering,
`getBackingConnection` parity of `resolveAttachConnection`,
`collectSourceTables` name normalization for the cycle walk, `'replace-all'`
in `backing-host.ts` / memory layer manager, parser verb plumbing), both new
test files, and the touched docs. Behavioral suspicions were verified
empirically with scratch probes before filing/fixing.

**Minor — fixed in this pass:**

1. **Cycle diagnostic rendered the path wrong for multi-node cycles** —
   `[target, ...path].reverse().concat(target)` produced
   `main.b1 → main.a1 → main.a1` (duplicated target, inverted direction).
   Fixed to data-flow order closed on the target
   (`main.a1 → main.b1 → main.a1`); pinned by a new spec test
   (`maintained-table-attach-detach.spec.ts` › cycle diagnostic).
2. **`set maintained` accepted a target with generated columns** (verified:
   attach succeeded and the generation expression was silently bypassed) while
   the create form rejects them — the gate was only in `building/ddl.ts`.
   Added the mirror gate to the `setMaintained` build case; both forms now
   reject, pinned in sqllogic § 8 (neither form's reject was previously
   tested).
3. **Stale doc claim in sql.md §2.10** — "`using module(...)` is parsed for
   forward compatibility but v1 always backs the view with the in-memory table
   module" contradicted the code (the clause resolves a backing-host module
   end-to-end) and the adjacent text this ticket added. Rewritten to match
   materialized-views.md.
4. **"Sited error" wording** in materialized-views.md's attach step 2 — the
   alter-path shape errors are named, not source-located (the handoff flagged
   this exact ambiguity). Softened to "an error naming the first mismatching
   component".
5. **Test-coverage gaps closed**: `insert defaults` on the ALTER attach verb
   (only the create form was covered — sqllogic § 10 extension), and the
   aggregate-body attach variant the handoff explicitly left unverified
   (sqllogic § 5 extension: strict shape check over derived aggregate
   types/key, post-attach incremental maintenance, both suites). The aggregate
   case demonstrated the strict check's authoring cost: a `count(*)` column
   must be declared `real null` (engine quirk, see major finding 2) and the
   PK collation must be pinned `collate binary` to pass the store suite.

**Major — filed as new tickets:**

1. **`tickets/backlog/maintained-table-declared-constraint-semantics.md`** —
   declared CHECK (and FK / secondary UNIQUE) constraints on a maintained
   table are silently bypassed by the create-fill, the attach reconcile, and
   steady-state maintenance (verified empirically: a CHECK-violating body row
   stores without complaint in both phases). The table form newly exposes
   this — the MV sugar could never declare constraints. Needs a design
   decision (reject like generated columns / validate derived rows / document
   as informational) plus a stance on detach-with-violating-rows.
2. **`tickets/backlog/aggregate-return-type-fidelity.md`** — `count(*)`
   derives `REAL` nullable (the `createAggregateFunction` default), so the
   strict declared-shape check forces table-form authors to declare a count
   column as `real null`. Pre-existing engine quirk, now a direct authoring-
   surface wart; easy fix with the 51.7 aggregate case as the regression net.

**Checked, no findings (explicitly):**

- **Attach core ordering/rollback** (the ticket's named review focus): every
  gate runs before catalog/data mutation; `restorePrior` correctly handles
  fresh-attach (unregister — verified idempotent) vs re-attach (re-register
  prior plan, fail-safe to stale); `registerMaterializedView`'s
  release-before-build means a failed re-attach registration has already
  released the prior plan, which `restorePrior` re-registers — correct.
  `createMaintainedTable`'s catch drops the just-created table; the
  pre-create shape check uses an unregistered schema build so failures leave
  the name free (pinned in sqllogic § 2).
- **Event/invalidation contract** (second named focus): one event per verb,

  attach/detach plan flips — all pinned by the implementer's spec tests,
  which I read critically and found sound (the dispatch-capture and
  event-capture techniques measure the right seams).
- **Strict-shape-check decisions** (third named focus): the two-way

  The synthesized all-columns-PK interplay (declared create without PK)

  component.

- **Duplicate-key pre-check**: collation-aware under the table's PK
