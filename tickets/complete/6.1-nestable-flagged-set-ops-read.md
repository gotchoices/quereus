## What landed

Generalized the single-level membership-flag invariant to nestable flagged set-ops, **read
path only**. `A union[inA,inSub] (B union[inB,inC] C)` now CREATEs, plans, and reads — it
previously threw at construction (`SET operation column count mismatch`) because the inner
subtree's flags inflated its total arity.

- `planner/nodes/set-operation-node.ts`: module helpers `dataArity` (recursive) / `flagCount`;
  constructor arity check on data arity only; `dataColumnCount()` now public + recursive;
  `hasSurfacedFlags` getter; `buildAttributes`/`getType` emit `[data] ++ [L flags] ++ [R flags]
  ++ [own flags]`; own-flag physical indices shifted by `ownFlagBase`; the all-data superkey FD
  determines every surfaced flag (inner and own).
- `runtime/emit/set-operation.ts`: `dataColCount = plan.dataColumnCount()`; `runWithMembership`
  → `runWithSurfacedFlags`, selected on `plan.hasSurfacedFlags`, buffering each operand's full
  row, keying probe sets on the data columns, emitting an operand's flag slice when present else
  default-false.
- `planner/building/select-compound.ts`: no code change — `createSetOperationScope` iterates the
  surfaced columns positionally, so an ORDER BY can reference an inner flag at depth.
- `docs/view-updateability.md`: new "Nestable flagged set-ops (read half)" paragraph.

### Static-writability over-claim on nested views — verified read-reachable, **not** new, deferred to 6.2
The handoff asked the reviewer to "verify this is genuinely read-only and not a latent over-claim
reachable from a read." Empirically it **is** read-reachable: `select * from view_info('Vn')` /
`column_info('Vn')` on a nested flagged view report fully insertable/updatable/deletable, while
every actual write is rejected (`the left branch projects 2 columns but the set operation exposes
4 data columns` — `analyzeSetOpView`'s `attrs.length - flags.length` over-counts the surfaced
inner flags as data columns).

**Disposition: documented, no inline fix, no new ticket.** It is *not* a defect this ticket
introduces in isolation: the flag-less nested shape `A union[…] (B union C)` — which predates this
ticket ("the one nested shape that already worked") — over-claims identically (`view_info` all-YES,
write rejected with `SetOperation is not updateable in phase 1`). This is exactly the
**one-level-deep over-claim `isSetOpBranchWritable` already documents** in
`planner/mutation/set-op.ts` ("a known, out-of-scope one-level-deep over-claim"). `set-op.ts` and
`func/builtins/schema.ts` are unchanged by this ticket. The real fix (recursive `dataColumnCount()`
in `analyzeSetOpView` + making nested bodies genuinely writable) is owned by **`nestable-flagged-set-ops`
(6.2)**, which is gated on this ticket — filing a new ticket would duplicate it. A conservative
inline "return false for nested" was rejected: it would churn against 6.2's design and only
partially address the broader pre-existing deferral. The **data-read path — this ticket's actual
deliverable — is exact**; only the optimistic metadata lags until 6.2. Added a one-sentence
honesty note to the read-half doc paragraph recording this.

### Documented residual gaps (acceptable for the read foundation; not findings)
- **Optimizer physical surface on a flag-less-outer-over-flagged-operand.** `computePhysical`
  gates its FD/domain/binding surface on `hasMembershipColumns` (own flags), so `A union (B∪[inB,inC]
  C)` surfaces `inB`/`inC` as readable columns but the *outer* node publishes no `key → flag` FD
  for them (the inner node carries its own). Correctness unaffected (data reads exact, keys sound);
  a missed optimization only. Consistent with the binary case's deferred sibling-prune.
- **Left-operand surfaced flags (`[L flags]`).** No SQL-reachable test — a parenthesized left
  compound flattens to a `select * from (…)` subquery, so `leftFlagCount` is 0 via SQL. The code
  path is exercised only structurally; correct by the symmetric construction.
- **`dataArity` couples to `instanceof SetOperationNode`.** If an optimizer rule ever wrapped the
  inner set-op between outer and inner, the recursion would miscount. Not reachable for the tested
  shapes (full create-view + select pipeline stays green); noted for maintainers.
- **3+ level nesting** follows the same recursive rule but is only tested to 2 levels.
