## What landed

A pure-refactor lift of the per-MV row-time maintenance plan into a tagged union, plus a
property harness that is the correctness oracle for the shipped arm.

- `RowTimeMaintenancePlan` (private) → `export type MaintenancePlan`, a tagged union:
  - `'inverse-projection'` (`InverseProjectionPlan`) — the former plan verbatim + a `kind`
    discriminant. The only arm the builder produces.
  - `'full-rebuild'` (`FullRebuildPlan`) — stub; identity fields only.
  - `'residual-recompute'` (`ResidualRecomputePlan`) — stub; carries `binding: BindingMode`
    (type-only import) + `degradeToRebuild`.
- `applyRowTimeChange` → `applyMaintenancePlan` — a `switch (plan.kind)` dispatching
  `'inverse-projection'` to the new private `applyInverseProjection` (old body byte-for-byte)
  and throwing `INTERNAL` (loud unreachable guard) for the two stub arms.
- `buildRowTimePlan` → `buildMaintenancePlan` — shape allowlist unchanged; return now carries
  `kind: 'inverse-projection'`, return type widened to `MaintenancePlan`.
- `lookupCoveringConflicts` gained an early `kind !== 'inverse-projection'` skip guard.
- New `test/incremental/maintenance-equivalence.spec.ts` — `fast-check` harness asserting
  `read(select * from mv) == evaluate(<body> against source)` as order-insensitive multisets,
  at baseline / in-transaction / post-rollback, over four covering-index body shapes with
  random insert/update/key-changing-update/delete sequences.
- Docs (`materialized-views.md`, `incremental-maintenance.md`) updated to name the
  `'inverse-projection'` arm and point at the harness as the oracle.

Public entry points (`registerMaterializedView`, `unregisterMaterializedView`,
`maintainRowTime`) unchanged in name & signature; `dml-executor.ts` correctly untouched
(it reaches maintenance via `db._maintainRowTimeCoveringStructures` → `maintainRowTime`).

## Review findings

### What was checked

- **The lift is semantics-preserving.** Read the full implement diff before the handoff.
  The `'inverse-projection'` body is the former `applyRowTimeChange` byte-for-byte; the
  builder's shape allowlist is unchanged. Regression gates green:
  `53-materialized-views-rowtime.sqllogic`, `54-covering-mv-enforcement.sqllogic`,
  `materialized-view-diagnostics.spec.ts`.
- **The equivalence harness is a sound oracle, not a tautology.** Confirmed via the planner
  (`planner/building/select.ts`) that Quereus only resolves an MV when explicitly named —
  there is no automatic MV-rewrite that would answer the oracle query (`select … from src`)
  from the MV backing. `select * from mv` reads the *maintained backing*; the oracle re-scans
  `src`. The two sides are genuinely independent computations, so a maintenance bug surfaces
  as a multiset mismatch. Verified the four shapes (incl. the column-permutation shape) are
  positionally comparable as `Object.values(row)`.
- **Union field access is safe.** `maintainRowTime` reads `backingSchema`/`backingTableName`,
  present on all three arms.
- **No stale call sites / docs.** Grepped `applyRowTimeChange` / `buildRowTimePlan` /
  `RowTimeMaintenancePlan` across `packages/` and `*.md` — every remaining hit is an
  intentional historical mention in JSDoc/docs, no live call site. Only the two intended docs
  reference the plan type; no other doc is stale.
- Build clean (`yarn build`), lint clean (`yarn lint`, exit 0), full suite green
  (`yarn test` — 3930 passing, 9 pending, up from 3928 by the +2 guard tests below).

### What was found and done (all minor, fixed inline)

- **Missing exhaustiveness guard (fixed).** `applyMaintenancePlan`'s `switch` covered all
  three arms but had no `default`; `noImplicitReturns` is off in this package, so a future
  fourth arm would have silently fallen through and returned `undefined` typed as
  `BackingRowChange[]`. Added a `default` branch with the codebase's `const
  exhaustiveCheck: never = plan` convention (mirrors `vtab/memory/layer/manager.ts`) so a new
  arm becomes a compile error.
- **Loud guards had zero test coverage (fixed).** The stub-arm `INTERNAL` throw is the novel
  surface of this change but was unreachable through any public API and untested — a
  regression that softened it to a silent no-op would not have been caught. Added a white-box
  test (`stubbed-arm guard` describe) that mutates a registered plan's `kind` to each stub arm
  in place and drives a real source write through the maintenance path, asserting the guard
  fires with `StatusCode.INTERNAL` and locked wording. Both arms covered; both pass.

### What was examined and deliberately left as-is

- **`lookupCoveringConflicts` returns `[]` (silent) on stub arms while `applyMaintenancePlan`
  throws.** This asymmetry is safe: both paths are unreachable today, and any stub-arm source
  write still trips the loud `applyMaintenancePlan` guard during maintenance, so the net
  behavior is fail-loud regardless. The `return []` is documented as a defensive skip. Not
  worth a change with its own subtleties.

### Major findings

- None. The implementer-flagged gaps (harness omits NULL/3-valued predicates, single-column
  PK only, MV-over-MV chains, positional comparison) are legitimate scope boundaries, each
  covered deterministically elsewhere (`53-materialized-views-rowtime.sqllogic`
  §5/§14/§15/§17) and named as natural extensions of follow-on tickets
  (`incremental-maintenance-cost-gate`, `materialized-view-rowtime-general-bodies`,
  `materialized-view-rowtime-mv-over-mv-cascade`). No defect; no new ticket filed.
