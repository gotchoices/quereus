# Unify SQL value equality on one function

## What changed

`util/comparison.ts` had two ways to ask "are these two SQL values the same
value?" that disagreed on numeric-storage-class cross-representation pairs
(`sqlValuesEqual(5n, 5)` was `false`; `compareSqlValuesFast` said equal). Every
real caller of the old `sqlValuesEqual` was a "did this value change?" check
(foreign-key parent-column change detection, check-constraint old/new diffing,
DML value-identical-upsert suppression, planner mutation cell comparison) where
the `===`-based divergence was a latent bug.

- Added `sqlValueIdentical(a, b)` — `compareSqlValuesFast(a, b, BINARY_COLLATION) === 0`.
  Numeric-storage-class tolerant (`5n` equals `5`, `true` equals `1`), bytewise
  for blobs, canonical-JSON for object-class values, byte-exact for text.
- Rewrote `rowsValueIdentical` as a loop over `sqlValueIdentical` (was a
  duplicate inline `compareSqlValuesFast` loop) so the two contracts cannot drift.
- Deleted `sqlValuesEqual`; migrated all 9 call sites.
- Exported `sqlValueIdentical` from `src/index.ts`.

## Behavior change (intended)

- `sqlValueIdentical(5n, 5)` → `true` (was `false`); `(true, 1)` → `true` (was `false`).
- `(null, null)` → `true`; `(null, 0)` → `false`; `('1', 1)` → `false`. Unchanged.
- Blobs bytewise, differing length → `false`. Unchanged.
- JSON objects: key-reorder-equal → `true`, via the canonical-string cache shared
  with `compareSqlValuesFast`'s object path (same cache instance — no perf
  regression, no semantic change).

## Review findings

### Verified correct

- **Behavior change is a net fix, not a regression.** `util/key-serializer.ts`
  already normalized `5n`, `5`, and `true`/`1` to the same key (asserted by
  `test/util/json-canonical.spec.ts` lines 68-78). The old `sqlValuesEqual`
  therefore *disagreed with the hash-key path* — a row could hash to one key yet
  be reported as "changed". The new function agrees with both the key serializer
  and `compareSqlValues`. That inconsistency is what this ticket closed.
- **All 9 call sites are change-detection or contradiction checks**, all of which
  want storage-class-tolerant equality. Read each in context; no site relied on
  the old `===` strictness. The two planner sites (`multi-source.ts` ~1012,
  `single-source.ts` ~1111) previously raised a spurious `predicate-contradiction`
  diagnostic for a `5n`-vs-`5` literal pair; they no longer do.
- **No orphaned references.** Repo-wide grep for `sqlValuesEqual` across all `.ts`
  and `.md` (excluding `node_modules`, `tickets/`) returns zero hits — no other
  package consumed the removed public export, and no doc named it. Docs in `docs/`
  never referenced either helper, so nothing was left stale.
- **No duplicated inline `===` equality** hiding at the migrated call sites; the
  implementer flagged this as unaudited. Checked: `compareSameType` /
  `compareSqlValuesFast` is now the single implementation reached by every path.
- `sqlValueIdentical` is referenced by `rowsValueIdentical` before its own
  declaration — legal (function declarations hoist), and it keeps the row-level
  contract doc as the primary one. Left as is.
- `yarn build` clean, `yarn workspace @quereus/quereus run lint` clean,
  `yarn test` 6585 passing / 9 pending (pre-existing skips) / 0 failing.

### Fixed in this pass (minor)

- **Missing unit coverage for `rowsValueIdentical`.** It is exercised only
  indirectly (three spec files use it as a helper); nothing pinned its own
  contract. Added a `rowsValueIdentical` describe block in
  `test/utility-edge-cases.spec.ts` covering delegation to the new per-value
  contract (`[5n, true, 'a']` vs `[5, 1, 'a']`), text-vs-numeric non-identity,
  blob identity, and the differing-width and empty-row cases.
- **Missing NULL-vs-non-NULL case** in the new `sqlValueIdentical` block, despite
  the implement handoff claiming that behavior was covered. Added
  `(null, 0)`, `(null, '')`, `(0, null)` → all `false`.

### Parked as a tripwire (not a ticket)

- `compareNumbers` (`util/comparison.ts` ~173) returns `0` when either operand is
  `NaN`, because both relational tests are false — so a `NaN` would compare
  "equal" to every number, in sort order *and* now in `sqlValueIdentical`. This
  is unreachable today: `runtime/emit/binary.ts` maps non-finite arithmetic
  results to `NULL`, and the affinity/coercion paths reject `NaN`. It is
  pre-existing (`compareSqlValuesFast` always behaved this way) and orthogonal to
  this ticket, but the migration widened the set of callers that inherit it.
  Recorded as a `NOTE:` comment at `compareNumbers`, naming the condition that
  would trip it (a `NaN` reaching storage via a bound parameter that skips
  affinity).

### Considered, deliberately not filed

- **No end-to-end test of the four caller families** with a bigint-vs-number pair
  round-tripping through a real storage layer — the implement handoff called this
  out as a gap. Not filed: the behavior is pinned at the only place it is
  decided (`sqlValueIdentical`), every caller is a thin delegation to it, and an
  integration test would exercise the storage module's representation choices
  rather than this function's contract. The `.sqllogic` suite already covers all
  four call sites' surrounding logic and passes unchanged.
- **The `key`/`sqlValueIdentical` agreement assertion** in
  `json-canonical.spec.ts` (~95-97) is now near-tautological: with
  `sqlValueIdentical` defined as `compareSqlValuesFast(...) === 0`, it restates
  the `key`/`compareSqlValues` assertion on the line above. Left in place — it is
  cheap, and it is precisely the invariant that would break if someone
  reintroduced a divergent equality helper. Documenting rather than deleting.
- The "Representation-sensitive keys" gap at `dml-executor.ts` ~331
  (affinity coercion and enforcement collation are still not applied when
  matching an `on conflict` target) remains open and is explicitly out of this
  ticket's scope. The comment there already states the shared fix. No new ticket
  filed — it is already documented at the exact site that would implement it.
