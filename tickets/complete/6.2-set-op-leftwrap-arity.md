## What landed

**Root cause.** The parser lifts a parenthesized LEFT compound operand `(A∪B)` into a passthrough
wrapper `select * from (A∪B) as values_N` (`wrapAsSubquerySelect` / `parseCompoundTail`) so the
SELECT-level `compound` slot can host the outer operator. `buildCompoundSelect` built that wrapper as a
`select *` `ProjectNode` over the inner `SetOperationNode`, so the outer node's left child was NOT a
`SetOperationNode`. `dataArity(left)` then took the `else` branch (`getType().columns.length`) and
counted the inner's **surfaced flag columns** as data (2 data + 2 flags = 4) while the RIGHT subtree
recursed to 2 → the constructor's `leftData !== rightData` check threw.

**Fix (one locus, build-time AST unwrap).** New shared pure predicate `unwrapPassthroughSubquery(sel)`
in `planner/util/set-op-wrapper.ts` returns the inner query-expr of a *pure* `select * from
(<query-expr>) as <alias>` wrapper, else `undefined`; recursive/idempotent (peels nested pure wrappers,
stops at an inner that carries its own `compound`). `select-compound.ts` applies it (via local
`unwrapToSelect`, which redirects only to a **SELECT** inner) to `leftCore` AND a `select`-typed right
operand before `buildSelectStmt`, so a parenthesized compound operand on either side builds as a direct
`SetOperationNode` subtree operand. `set-operation-node.ts` needed **no change** — the existing
`dataArity` / `flagCount` / `buildAttributes` recursion then produces the sum surface
`[data] ++ [L flags] ++ [R flags] ++ [own flags]` correctly.

### Findings & disposition

- **MINOR (fixed inline)** — `isPassthroughWrapper` omitted the legacy `union` / `unionAll` SelectStmt
  fields from its purity guard. They are dead in current parser output (compounds ride the `compound`
  slot), so this was not a live bug, but the project's parallel predicate
  `isNonRowReducingProjection` (`planner/mutation/lens-enforcement.ts`) *does* guard them. Since the
  helper is exported/shared and accepts any `SelectStmt`, the omission was a latent drift risk. Added
  `&& sel.union === undefined && !sel.unionAll` with an explanatory comment. No behavior change on real
  inputs; re-verified green.
- **MINOR (fixed inline)** — the `unwrapPassthroughSubquery` docstring claimed it was "Shared by …
  the write path (`set-op.ts`)", which is not true today (write-path reuse is 6.3). Corrected the
  docstring to state the write path does NOT yet unwrap and that `set-op-leftwrap-write` will add it.

### Probed, no finding

- **Right-operand unwrap = dead-defensive code.** The parser does not currently emit a *pure*
  passthrough wrapper as a right operand (the V4 `((C∪D))` test passes via pure grouping, not via this
  unwrap). The implementer flagged it. Left as-is: it is conservative (the predicate cannot misfire),
  carries a clear comment, mirrors the left-side handling, and is the natural reuse point for 6.3. Not
  worth a trim.
- **`dataArity` checked at construction (build time), pre-optimization.** Spot-checked that the set-op
  rules do not re-wrap a left operand in a way that changes `getType().columns.length` at emit time;
  reads go through the full pipeline and pass. No issue.
- **Parser limitation (pre-existing, orthogonal).** A top-level statement that *starts* with `(` does
  not consume a trailing `order by` (the LPAREN-led `parseQueryExpr` skips `parseTrailingOrderLimit`).
  The read tests correctly work around it by reading through a created view. This predates the ticket
  and is unrelated to arity — no ticket filed here; flag to the human if the parser gap warrants one.
- **Product model (reused flag names → merged coordinate columns)** — explicitly out of scope; rejected
  at create today (P3); deferred to `set-op-product-coordinate-model`. Correct boundary.
