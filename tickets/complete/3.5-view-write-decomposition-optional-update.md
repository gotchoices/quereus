## What shipped

`decomposition.ts` `routeAssignment` no longer rejects an optional-member / EAV-pivot UPDATE.
Each routed assignment drives a per-row materialization transition emitted as plain AST `BaseOp`s
keyed off the anchor subquery, **anchor-last** so every branch reads the intact anchor:

- **Optional columnar member** — matched → base UPDATE; absent → null-extended materialize INSERT
  (`insert into <member> (<memberKey>, <cols…>) select <anchorKey>, <values…> from <anchor>
  where <pred> on conflict (<memberKey>) do nothing`); every value column assigned null → base
  DELETE. A partial-null (some value columns null, not all) is a matched UPDATE only.
- **EAV pivot member** (per attribute) — non-null → upsert the triple (matched UPDATE + materialize
  INSERT `on conflict (<entity>, <attr>) do nothing`); null → delete the triple.
- **Value handling (v1 narrowing)** — a non-null value must lower to a **constant** (scope-
  independent across the member-scoped UPDATE and the anchor-scoped INSERT); a column-referencing
  value rejects `unsupported-decomposition-update`. A null literal is the delete/no-op trigger.
- Still rejecting: shared-key (identity) UPDATE, non-constant optional/EAV value, non-anchor
  predicate, composite key, computed mapping, cross-member read, RETURNING-through-view.

## Review findings

**Verdict: sound. One minor doc inconsistency fixed inline; no correctness bugs found; three
deferred corners filed as backlog hardening + one feature follow-up.**

### Checked

- **Read the implement diff (b83cf316) with fresh eyes before the handoff**, then the full current
  `decomposition.ts` (both pages), the test diffs, and all four touched docs.
- **Equivalence of `on conflict do nothing` to the ticket's drawn anti-join** (the implementer's
  flagged substitution): traced the runtime — `do nothing` fires only on a detected PK/UNIQUE
  conflict (`dml-executor.ts` `matchUpsertClause` → `processInsertRow`). When the stitch key is
  the member PK (every supported advertisement), the conflict path correctly cedes matched rows
  to the UPDATE; the matched UPDATE never changes member keys, so the conflict set is stable and
  the UPDATE-then-INSERT order is order-independent. **Behaviorally equivalent to the anti-join
  for a PK/unique stitch key.** (The non-unique-stitch-key caveat → backlog, see Gap 2 below.)
- **Anchor-last ordering / pre-mutation snapshot**: confirmed the optional matched-UPDATE +
  materialize-INSERT are emitted before the anchor's own op, so a predicate over an anchor column
  being updated (`set a = 1, c = 5 where a = 0`) reads pre-mutation `a`. Sound.
- **Lingering all-null member rows** (a partial-null UPDATE that leaves every value column null,
  or a function-valued null like `nullif(1,1)` that the syntactic `isNullLiteral` does not catch):
  traced every subsequent op (matched-update / materialize / delete) and confirmed an all-null
  physical row is **observationally equivalent to absence** — the read renders it as absent and
  every write branch keys off PK existence, so no view-image divergence. The syntactic null
  detection is therefore observationally sound, just not physically minimal.
- **Scope-independence of admitted values**: `exprHasColumnRef` rejects any column-referencing
  value (member self-ref, anchor column, correlated subquery). A column-free uncorrelated subquery
  slips through but is genuinely constant across rows — sound.
- **Conflicting-assignment / case folding**: `noteTarget` keys columnar on the basis column and
  EAV on `attr:<lower>`, so `set p = 1, P = 2` (same attribute) and duplicate-rename collisions
  reject view-aware. Correct.
- **`assertNoMissingNotNull` + the value-column-default soundness gate**: a NOT-NULL uncovered base
  column rejects (`no-default`); an unassigned mapped value column that is NOT NULL or carries a
  declared default rejects `unsupported-decomposition-update` (would widen the absent row's image).
  Tested both ways in the new multi-value-column describe block.
- **No external callers broken**: `routeAssignment` / `RoutedAssignment` signature + type changes
  are file-local; the only caller (`decomposeUpdate`) is updated.
- **Lint**: `yarn workspace @quereus/quereus lint` — clean (exit 0), before and after my edit.
- **Tests**: `yarn workspace @quereus/quereus test` — **4680 passing, 9 pending (exit 0)**, matching
  the handoff. The extended PutGet property oracles (columnar numRuns 100, EAV numRuns 80) over
  random matched/absent/delete sequences plus the deterministic branch pins are a strong soundness
