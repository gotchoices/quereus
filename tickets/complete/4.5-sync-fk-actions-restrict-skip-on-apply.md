# Skip RESTRICT on the trust-the-origin apply path (COMPLETE)

## What shipped

On the external-row-change apply path (`Database.ingestExternalRowChanges` with
`applyForeignKeyActions: true`), the FK-actions facet **skips all parent-side RESTRICT
enforcement** and propagates only the non-RESTRICT actions (cascade / set-null /
set-default) at every cascade depth. Re-enforcing RESTRICT on the receiver wedged the sync
stream (throw → no metadata commit → re-resolve → re-apply identical batch → throw forever).
Trust-the-origin: the origin already enforced RESTRICT at its own commit.

Mechanism: a `Database`-level apply-mode suppression flag (`_setFkRestrictSuppressed` /
`_isFkRestrictSuppressed`) set for the duration of the batch (mutex held; save-and-restore in
a `finally` for nesting), honored at **two** enforcement layers:
- the three runtime RESTRICT asserts in `foreign-key-actions.ts` (reached from the DML
  executor, the removed seam call, and MV maintenance), and
- the **plan-time** parent-side FK check (`constraint-check.ts`, `metadata.kind === 'fk-parent'`),
  which is the primary enforcer a depth-≥1 cascade re-entering the DML pipeline would trip.

## Review findings

### Scope checked
Read the full implement diff (`10bb88bc`) with fresh eyes before the handoff: all 4 source
files, the rewritten spec, and the `docs/sync.md` § Apply-time-validation rewrite. Verified
the central design claims by reading the surrounding code (not just the diff), re-ran lint +
the full quereus suite + the sync suite, and added one hardening test.

### Correctness — the `fk-parent` gating predicate is exact (VERIFIED)
The handoff asked for independent verification that `metadata.kind === 'fk-parent'` is the
precise predicate for "parent-side RESTRICT, nothing more". Confirmed:
- `kind: 'fk-parent'` is synthesized in exactly **one** place (`foreign-key-builder.ts:434`),
  inside `buildParentSideFKChecks`, and that loop does `if (action !== 'restrict') continue;`
  (line 350) — so **every** `fk-parent` check is a RESTRICT check. CASCADE / SET NULL / SET
  DEFAULT produce *actions*, not parent-side checks. Gating all `fk-parent` ⟺ suppressing
  parent-side RESTRICT.
- `'check'` (user CHECKs), NOT NULL (separate `checkNotNullConstraints` path), and `'fk-child'`
  (child-side existence) are untouched — the gate names only `'fk-parent'`. Confirmed by
  grepping every producer/consumer of the literal `'fk-parent'`.
- No RESTRICT `fk-parent` check is ever *deferred* past the flag window: RESTRICT is
  `needsDeferred: false` / immediate, so none is queued to fire at the caller's later commit
  (where the flag would already be restored). The runtime gate `continue`s before the
  deferred-queue branch, consistent with this.

### Correctness — flag lifecycle (VERIFIED)
- Flag is set/restored only while the exec mutex is held (set at line 150, after
  `_acquireExecMutex`), so no concurrent statement on the connection observes it; `Database`
  is per-connection, so no cross-connection leak.
- Save-and-restore (not a blind reset to `false`) supports nesting; restored in `finally` so a
  non-RESTRICT mid-batch throw still re-arms RESTRICT for later normal statements. Both
  properties have dedicated tests (post-batch normal-delete; NOT NULL throw mid-batch).
- When the facet is off, the flag is never touched (neither set nor restored) — correct.

### Performance (VERIFIED, no concern)
The added `constraint-check.ts` condition is on the per-DML hot path but short-circuits on the
cheap `metadata.kind === 'fk-parent'` comparison before the flag read; off the apply path
behavior is byte-identical. The 195-test FK/lens/constraint sweep is unchanged.

### Docs (VERIFIED current)
`docs/sync.md` § Apply-time validation fully reflects the new reality: the
RESTRICT-not-enforced-on-apply paragraph names all three honoring layers (runtime asserts,
nested cascade / MV-maintenance, plan-time `fk-parent`), and limitation (E) is correctly
downgraded to moot (RESTRICT never throws on apply). Other RESTRICT-mentioning docs
(`lens.md`, `materialized-views.md`, etc.) describe general FK semantics, not the sync-apply
path — correctly left alone; the apply-path policy is a sync concern and lives in `sync.md`.

### Tests — added one (minor, fixed in this pass)
The implementer flagged cascade-**UPDATE** reaching a RESTRICT child as an untested gap
("only DELETE cascade is exercised; the gating is op-agnostic"). I added the UPDATE analog of
the depth-≥1 DELETE test (`a nested cascade UPDATE reaching a RESTRICT grandchild COMMITS`).
It exercises the plan-time `fk-parent` gate on the UPDATE cascade-reentry path: re-keying a
parent cascades `on update cascade` to a child column referenced by an `on update restrict`
grandchild; the gate suppresses the RESTRICT, the child re-keys, the grandchild dangles, the
batch commits. **6358 passing** (was 6357).

Building this test surfaced and clarified the apply-path contract (worth recording):
- A depth-0 external change never reaches a gate — the adapter writes storage and the seam
  only *propagates actions* (RESTRICT is not an action). The gates matter only at cascade
  **re-entry** (depth ≥1), which is what tests #2 (DELETE) and the new #3 (UPDATE) cover.
- A cascade UPDATE necessarily re-keys a column that is itself a child-side FK, whose deferred
  `fk-child` check is **deliberately not suppressed**. That check is only satisfiable if the
  parent's *new* key is already in storage — the apply contract ("adapter writes storage
  before the seam call"). My first draft omitted the `directWrite` staging of the parent
  re-key and tripped `_fk_c_id` at commit; staging it (delete-old + insert-new, mirroring the
  delete-cascade tests) fixed it. This empirically confirmed the RESTRICT (`fk-parent`)
  suppression works on the UPDATE path — the only throw that ever occurred was the
  un-suppressed child-side check, exactly as designed.

### Untested-but-sound (acknowledged, no ticket)
- **REPLACE-eviction RESTRICT** (`dml-executor.ts:723`): covered by construction (same runtime
  assert + same `fk-parent` plan-time check, both gated); eviction is runtime-only so there is
  no plan-time check for the evicted row, only the gated assert. No e2e test — low marginal
  value over the inspection.
- **Lens-routed apply** (`assertLensRestrictsForParentMutation`): the seam passes
  `lensRouted = false`, so this assert is unreachable from the apply path today; gating it is
  belt-and-suspenders per the ticket. Acceptable — no reachable path to test.

### Observation — broader wedge class (considered; NO ticket filed)
The flag suppresses only parent-side RESTRICT. A non-RESTRICT cascade-derived failure on apply
(SET NULL onto a NOT NULL column, a CHECK, or a dangling child-side FK) still throws at commit
and rolls the batch back — the same throw → re-apply → throw wedge, via a different constraint
kind (the implementer's NOT NULL test documents this as accepted behavior). I judged this **out
of scope and not a defect**: on a well-formed origin those states can't arise (the origin would
have thrown at its own commit), and the only way they reach a replica is a *stricter replica-only
constraint* — which is exactly the "replica-only invariant not enforced on apply" case the ticket
already documents, with the prescribed remedy being a **global assertion** (detect-and-notify).
Filing a ticket for a state the trust-the-origin model implies cannot occur would be over-reach;
recording it here for a human to escalate if real-world schema divergence makes it concrete.

## Decision context (signed off; not re-litigated)
Skipping RESTRICT on apply (vs. throwing) is the intended trust-the-origin policy; a
replica-only RESTRICT invariant is, by design, not enforced on apply and is expressed as a
global assertion if the receiver must be notified. The only design expansion during implement
was *where* suppression had to be applied (adding the plan-time `fk-parent` gate), not
*whether* to suppress — verified correct above.
