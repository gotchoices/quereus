## What landed

`docs/invariants.md` is a new register: a preamble, then `## OPT — Query optimizer` holding
28 entries `OPT-001` … `OPT-058` (ascending, gaps deliberate), then empty stub headings for
the reserved `MV` / `RT` / `SCH` / `SYNC` / `LENS` areas.

Each entry has a heading `### OPT-nnn — <title>`, one or more `code:` pointers, exactly one
`guard:` line, one `doc:` back-link, and a body under 120 words. `yarn docs:check` (Check B)
validates every pointer resolves and every named symbol still appears in its file. 21 of 28
entries point at a real test or runtime assertion; five say `guard: none — <reason>`; two are
self-guarding.

23 `> **Invariant:** [OPT-nnn](invariants.md#full-slug)` back-links went into the topic docs.
`optimizer-conventions.md`'s eight numbered conventions became six back-link blockquotes plus
one convention kept as prose (equivalence-class closure for bindings — advice, not an
invariant). `doc-conventions.md` states that the register is the normative text and that
back-links must use the full heading slug, since an invariant heading's em dash slugifies to a
double hyphen and the short `#opt-014` form does not resolve.

The implement stage also corrected a genuinely wrong code sample in
`optimizer-conventions.md`: `PlanNodeCharacteristics.guaranteesUniqueRows` was shown reading a
`physical.uniqueKeys` field that does not exist. The real implementation reads `hasSingletonFd`
with a zero-column `estimatedRows` fallback. Confirmed against
`packages/quereus/src/planner/framework/characteristics.ts:134`.

- **OPT-024.** All three sub-claims hold in `rule-select-access-path.ts`: the role pickers use

  relation matches nothing.
- **OPT-050.** All six extraction sites named in `optimizer-fd.md` apply the collation gate,

### Found and fixed in this pass (minor)

- **`OPT-002` was wrong about `in-subquery-cache`.** The entry named it as one of two `'aware'`
  rules that "refuse nothing". It does refuse: `rule-in-subquery-cache.ts:54` returns `null`
  when `!PlanNodeCharacteristics.isFunctional(source)`. `isFunctional` is
  `isDeterministic && isReadOnly`, and `isReadOnly` reads `physical.readonly` — a *different*
  purity signal from the `hasSideEffects` / `subtreeHasSideEffects` the entry named, and a
  strictly stronger one. Only `cte-optimization` consults nothing. Rewrote the entry to say
  so.
- **`OPT-022` was wrong that growth never consults cost.** True on the `supports()` path.
  False on the index-style fallback: `fallbackIndexSupports` in `rule-grow-retrieve.ts:380-404`
  refuses when `accessPlan.cost >= seqScanCost(estimatedRows)` and the plan provides no
  required ordering. Growth over that path is therefore cost-gated. Rewrote the sentence, kept
  the conclusion it was supporting (the boundary is still a deterministic function of the plan,
  so join-enumeration cost comparisons remain meaningful), and pinned `fallbackIndexSupports`
  as a `code:` symbol so a rename now fails `docs:check`.
- **Two `guard:` pointers named things that are not tests.** `OPT-054`'s guard was the bare
  token `isSet`, which matches a type field, a helper option, and file prose in
  `keysof-isunique.spec.ts`; it would keep passing after the guarding test was deleted. Now
  names the test title `falls back to the all-columns key for a set with no smaller key`.
  `OPT-058`'s guard was `Singleton equivalence law`, which appears only as a section comment in
  `property.spec.ts`; now names `checkSingletonEquivalence`, the function the property run
  actually invokes (`property.spec.ts:2028`).
- **`optimizer-retrieve.md` overstated `OPT-020`'s guard.** It said "a validation invariant
  enforces that no `Retrieve` nodes reach emission." `validatePhysicalNodeType` runs only under
  `tuning.debug.validatePlan`, off by default. `OPT-020`'s body already said this; the topic
  doc did not. Per the precedence rule the register introduced, the topic doc is the one to
  fix. Fixed.

On the implementer's review-focus question #2 — whether `OPT-020` should instead assert what
production actually does — the entry as written is correct: the invariant is "no logical-only
node reaches emission", and a weak guard is a guard-strength problem, not a wrong statement.
Both texts now name the debug flag.

On review-focus #4, the dropped seed invariant ("rules never dispatch on `instanceof` or
`nodeType`"): agreed with the implementer's call to drop it. Rules are registered per
`nodeType`, and `optimizer.md` sanctions `instanceof` on `RetrieveNode`. It is house style.
No ticket.

On review-focus #3, splitting `OPT-002` into separate `'safe'` and `'aware'` entries: not
done. The rewritten entry states both accurately inside the word cap. Splitting would burn an
ID for no gain.

### Tripwires (parked, not ticketed)

- **`guard:` pointers are substring-matched.** The checker documents this as intended strength
  for `code:` pointers, but the consequence for `guard:` is sharper: a common token or a
  section comment resolves, so a guard can name something that is not a test and still pass.
  Both instances above were exactly that. Extended the existing `NOTE:` at
  `scripts/check-docs.mjs` § `checkInvariantPointer` with the guard-specific advice (write the
  exact `describe`/`it` title, or the function driving the law). Conditional: it only matters if
  someone writes a lazy guard pointer, and the fix — stripping comments before matching — is
  disproportionate today.
- **`PassManager.registerPass` does not validate a pre-populated `rules` array.** Parked by the
  implementer as a `NOTE:` at `packages/quereus/src/planner/framework/pass.ts` §
  `registerPass`. Verified: every pass ships with `rules: []`, so no dormant path reaches it.
  Correctly a tripwire, not a defect.

### Not fixed, and why

The implementer's `OPT-002` caveat stands after my pass: I confirmed that each `'aware'` rule
file consults *a* purity signal, but not that every subtree-moving branch *within* each file
consults it before moving. A rule could gate on one branch and skip on another. Proving that
needs per-branch dataflow, not a source scan, and no proposed mechanical guard would catch it
either. The entry says the declaration is not machine-checked against a rule's body, which is
the honest statement of exactly this gap.
