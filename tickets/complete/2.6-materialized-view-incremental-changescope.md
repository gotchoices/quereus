# Incremental-MV change-scope projection (completed)

A `select` from an `on-commit-incremental` materialized view now projects, in
`analyzeChangeScope`, onto the MV's **source** tables rather than reporting its
(never-user-written) `_mv_*` backing table. A `Database.watch` on such an
MV therefore fires on a source mutation. A `manual` MV keeps reporting its
backing table (its cadence is `refresh`, which writes that table; it is
user-observable state).

## What shipped

- **`MaterializedViewSchema.sourceScope?: ChangeScope`** (`schema/view.ts`) ‚Äî
  cached source-union scope for an incremental MV; type-only `ChangeScope`
  import (no runtime cycle). Not serialized; repopulated on re-registration.
- **`buildSourceUnionScope(sourceTables)`** (`change-scope.ts`) ‚Äî conservative v1
  union: one `{kind:'full'}`, `columns:'all'` watch per source.
- **Cache population** in `MaterializedViewManager.registerMaterializedView`
  (`database-materialized-views.ts`), right after the incremental-policy guard ‚Äî
  the single point an MV becomes incremental (create emitter + `apply schema`
  both reach it via `db.registerMaterializedView`).
- **`SchemaManager.getMaterializedViewByBackingTable`** (`schema/manager.ts`) ‚Äî
  O(1) reverse lookup, prefix-guarded by `backingTableNameFor('')` and confirmed
  against the MV's own `backingTableName`.
- **`MaterializedViewSourceResolver` + analyzer projection** (`change-scope.ts`) ‚Äî
  optional `resolveMaterializedViewSource` on `analyzeChangeScope`; a resolved MV
  scope replaces the backing watch and is folded in with `unionScopes` (so the
  MV + a direct source read dedup to the source once). Gated on
  `!dmlWithoutReturning`; param binding still runs last.
- **`Statement.getChangeScope` wiring** (`core/statement.ts`) ‚Äî resolver backed by
  `getMaterializedViewByBackingTable`, returning `sourceScope` only for
  `on-commit-incremental`.
- **Docs** ‚Äî `materialized-views.md`, `change-scope.md`, `optimizer.md`.

## Review findings

Adversarial pass over the implement diff (`62c36593`). Read the full source +
test + docs diff first, then probed edge cases with throwaway specs.

**Verified correct (no change needed):**

- **Behavioral contract.** Incremental MV ‚Üí source watch; manual MV ‚Üí backing
  watch; MV+source read ‚Üí source once; end-to-end watch fires on source
  mutation. All hold (tests pass).
- **Resolver soundness.** `getMaterializedViewByBackingTable` is prefix-guarded
  *and* confirms `backingTableName`, so a regular table can't misfire ‚Äî and a
  user table can't even collide with a backing-table name (the backing table
  occupies it). Cross-schema MV/source qualified names parse correctly.
- **Graceful fallback.** If `sourceScope` is `undefined` (a hypothetical
  catalog-load-without-registration path ‚Äî none exists today), the resolver
  returns `undefined` and the analyzer reports the backing table. Sound.
- **Double-reference / param-binding interaction.** Repeated MV refs fold via the
  idempotent `unionScopes`; `bindParameters` runs last on the unioned scope.
- **No runtime import cycle.** `view.ts` uses a type-only `ChangeScope` import;
  `database-materialized-views.ts` imports the `buildSourceUnionScope` value from
  `change-scope.js` (no back-edge). Confirmed by clean `tsc --noEmit`.
- **Stale-source-drop edge (the ticket's flagged concern).** Probed: after
  `DROP TABLE src`, **both** executing `select * from mvi` and calling
  `getChangeScope()` throw the *identical* "stale; drop and recreate (Table 'src'
  not found)" error at plan-build. `getChangeScope` throws *before* producing a
  scope, so the resolver never yields a watch on the dropped table ‚Äî the
  watch-rejects path is unreachable. Full parity with execution; the ticket's
  reasoning holds. Locked with a regression test.
- **Only one production caller** (`Statement.getChangeScope`) needs the resolver,
  and it is wired. The analyzer-level test helper intentionally omits it.

**Minor ‚Äî fixed inline (this pass):**

- **Test floor was thin** (3 integration tests only checked table *names*). Added:
  (1) `change-scope.spec.ts` ‚Äî the projected source watch is `{kind:'full'}` /
  `columns:'all'` (locks the v1 conservative contract a column/row-specific MV
  read could otherwise silently sharpen); (2) `change-scope.spec.ts` ‚Äî stale MV
  after source drop ‚Üí `getChangeScope` throws cleanly (regression guard for the
  edge above); (3) `change-scope-analyzer.spec.ts` ‚Äî a `buildSourceUnionScope`
  unit suite covering full/all shape, multi-source schema honoring, and the
  unqualified-name `main` fallback (the N-source path is unreachable via SQL today
  because join bodies are incremental-ineligible, so unit coverage is the only
  guard). Suite: **3761 passing, 9 pending** (was 3756; +5). Lint + tsc clean.

**Considered, intentionally not changed:**

- **DRY of the `schema.table` parse.** `buildSourceUnionScope` and
  `linkCoveredUniqueConstraints` both do `indexOf('.')` + `'main'` fallback (and
  `global.ts` uses `split('.')`). The implementer consciously mirrored the
  existing helper. Because `sourceTables` always carries a schema-qualified name
  (`collectSourceTables` builds `schema.table`), the `'main'` fallback is dead in
  practice. Extracting a shared `parseQualifiedName` is a cross-cutting cleanup
  beyond this ticket ‚Äî not worth coupling here.
- **`buildSourceUnionScope` does not self-dedup.** `normalizeWatches` only sorts;
  dedup-by-table lives in `unionScopes`. The helper is always fed a deduped
  `Set`-derived list, so adding defensive dedup would be unnecessary
  over-engineering. Documented in the new test's comment.

**No major findings ‚Äî no new fix/plan/backlog tickets filed.** The documented v1
imprecisions (whole-table `full` projection rather than per-source row/group;
multi-source/join incremental eligibility; auto-rebind of watchers on schema
change) remain correct follow-up scope and were already recorded in
`docs/materialized-views.md` ¬ß Limitations and `docs/change-scope.md`
¬ß Known-imprecisions.
