# Module `ALTER TABLE ADD CONSTRAINT` for UNIQUE / FOREIGN KEY — completed

Both built-in modules implement `module.alterTable({ type: 'addConstraint', constraint })` for
UNIQUE and FOREIGN KEY (CHECK remains engine-side in `runtime/emit/add-constraint.ts`). The module
re-validates the existing rows and either installs forward enforcement or fails atomically with
`StatusCode.CONSTRAINT` (schema unchanged). Closes 10.2 gap #1 (declarative add of a named UNIQUE/FK
to an existing table converges + is idempotent). Prereq for 10.3's UNIQUE/FK drop+add path.

Implementation shape, behavior, and the full validation matrix are as described in the implement
handoff; the review below records what was independently checked, found, and done.

### Major — new ticket filed (not fixed here)
3. **Store DDL round-trip silently drops ALL table constraints across a cold reopen.** The store
   persists schema as `generateTableDDL(tableSchema)` and re-parses it on `rehydrateCatalog`, but
   `generateTableDDL` emits only columns + PK + USING + tags — **no** UNIQUE / FK / CHECK (verified:
   zero constraint tokens in `ddl-generator.ts`). So this feature's store output (and any CREATE
   TABLE constraint, and the already-shipped drop/rename) is correct in-session but not durable.
   Pre-existing and broad; the green declarative-equivalence round-trips use the *AST→SQL* path
   (`emit/ast-stringify.ts`), a different emitter, so they never exercised the lossy schema→DDL path.
   Also noted: the downstream `10.3` ticket's premise that `generateTableDDL` "already serializes
   constraints" is false and must be reconciled. Filed
   `tickets/fix/store-ddl-roundtrip-drops-constraints.md`.

### Checked — acceptable / no action
- **Atomicity / rollback:** memory `addConstraint` catch restores the prior schema and rebuilds
  secondary indexes (discarding a half-built covering index); FK validates before the schema swap;
  store validates before `updateSchema`/`saveTableDDL`. Failed-add cases assert constraint-absent +
  not-enforced. Correct.
- **Store UNIQUE collation limitation:** the existing-row UNIQUE check is value-exact (BINARY),
  matching the store's `CREATE UNIQUE INDEX` path; it does not honor a per-column NOCASE collation.
  Documented in `docs/schema.md` / `docs/sql.md`. Carried, acceptable.
- **FK parent-absent / composite / cross-schema:** the shared builder + validator handle these
  (aliased correlated `not exists`, self-FK-safe, column-count mismatch → ERROR at resolve). Covered
  indirectly; no dedicated parent-genuinely-missing or composite case, but the logic is exercised by
  the success/orphan/NULL matrix. Acceptable for this scope.
- **Isolation layer:** `isolation-module.ts` forwards `alterTable` to the underlying generically and
  `translateOverlayRow` no-ops `addConstraint`; no edit needed. Not driven by a dedicated isolated
  add-constraint test, but the forwarding is structural.
- **Store post-`updateSchema` `saveTableDDL` I/O failure** is non-atomic (in-memory schema updated,
  DDL not persisted) — but this mirrors every other store alter arm; pre-existing pattern, not
  introduced here.
