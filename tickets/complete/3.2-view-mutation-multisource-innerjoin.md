# Multi-source inner-join view mutation — COMPLETE (reviewed)

Shipped in implement commit `f7ba968c`; reviewed and finalized here. Multi-source
key-preserving inner-join `update` / `delete` write-through lands, decomposed via
`planner/mutation/multi-source.ts` off the planned body's `updateLineage`, identified by a
subquery over the join body, executed as an ordered (FK-parent-before-child) sequential base-op
fan-out through the rebuilt `ViewMutationNode` emitter. Insert, RETURNING, and the broader join
shapes remain diagnosed-and-rejected (decomposed into 3.6 / 3.7 follow-on tickets).

## Review findings

- Read the full implement diff with fresh eyes: the new `multi-source.ts` (557 lines), the
  `propagate.ts` routing peek, the `single-source.ts` export-widening (`transformExpr` /

- Confirmed AST/schema shapes used by the decomposition (`ResultColumnExpr.expr`, `InExpr`,
  `ForeignKeyConstraintSchema.{referencedTable,referencedSchema}`, `primaryKeyDefinition[].index`,
  `UpdateSite.base.{table,baseColumn,inverse}`) match the code.

### Adversarial probes — all PASS
- **Subquery-IN identifying predicate / Halloween.** Updating a PK that is *also* the join key
  (`update vj set tid = 9 where tid = 1`, the case removed from `93.2-pending`): the IN-subquery
  is evaluated once and stable; the row is re-keyed and correctly drops out of the view. Predicate
  on the column being updated (`set note='X' where note='a'`) targets the original matches only,
  no cascade.
- **Predicate honesty.** A parent-side update via `where cid=K` touches exactly the parent of the
  matched child and is visible to every sibling child joining that parent — intended. A delete
  with a parent-side predicate deletes all matching children, parents untouched. The property
  test's dangling (unjoined) children are never perturbed.
- **Qualifier / aliasing.** Swapped projection (view col `cid` ← `p.pv`, view col `pv` ← `c.cid`,
  with base columns of those same names present) routes by lineage, not by name. View-name-qualified
  predicate (`where jv.cid = 1`) resolves. Explicit `create view v (k,n) as ...` column-list
  override routes correctly.
- **Body-WHERE conjunction.** A filtered join view (`... where p.pv > 150`) honors its own WHERE
  for both predicate-less `update` and `delete` (only the exposed rows are touched).
- **Rejections (none silently widen).** Composite-PK side, `>2`-table join, self-join, cross-source
  SET value, `select *` over a join, and DELETE with no provable FK-child all reject with structured
  `cannot write through` / `cannot delete through` diagnostics. A LEFT-join `insert` rejects on the
  outer-join *shape* (not as "multi-source insert"), preserving the lens `write-rejected` boundary.

### Findings disposition
- **Minor — FIXED.** `docs/view-updateability.md` referenced a non-existent follow-on slug
  `view-mutation-multisource-insert`; corrected to the real ticket `view-mutation-shared-surrogate-insert`.
- **Minor — ADDED.** 11 permanent regression cases appended to `test/logic/93.4-view-mutation.sqllogic`
  (swapped-name aliasing, view-name qualifier, body-WHERE conjunction, column-list override, and the
  composite-PK / cross-source-SET / self-join / 3-table / no-FK-delete rejections). The implementer's
  93.4 cases covered the canonical happy paths and three rejections; these close the aliasing and
  shape-rejection gaps.
- **Major — FILED (backlog).** `view-mutation-subquery-nested-colref-substitution`: `transformExpr`
  does not descend into nested subquery / `exists` operands when substituting view-column references,
  so a view column referenced inside a subquery in a DML predicate or SET value can silently re-bind
  to a same-named base column (wrong write, not an error). **Pre-existing** — shared with the
  single-source spine, which reuses the same `transformExpr`; this ticket inherited it. Low frequency
  but a silent-correctness failure mode, hence tracked.

### Observations (no action — by design / acceptable)
- Diagnostic-prefix uniformity: `delete-ambiguous` begins "cannot delete through view" and
  `returning-through-view` begins "RETURNING through view" rather than the "cannot write through
  view" prefix the handoff describes. These match existing single-source patterns and no test/grep
  depends on the uniform prefix for them — left as-is (the variants read more clearly).
- The two per-side identifying subqueries deep-clone their FROM but **share** the single
  `idPredicate` WHERE AST node (the doc-comment's "never share AST nodes" is slightly inaccurate).
  Re-planning each base op twice is non-mutating in practice (the both-sides-update path is green),
  so this is a latent-only risk; not worth churn. Noted for the 3.6/3.7 follow-ons that extend this
  file.
- Source matching is by table **name** only (schema ignored), so a cross-schema join of same-named
  tables would be mis-rejected as a self-join. Genuinely rare; within the documented restriction set.
