## What shipped

Binary set-operation membership-column write-through (`planner/mutation/set-op.ts`). A
membership column **is** the branch presence: writing it drives the branch's existence
(`set inB = true` ⇒ insert into B, `= false` ⇒ delete from B). Data-column writes and
deletes fan out to every member branch via the runtime membership probe; insert-through
routes by the supplied literal flags. Every probe-driven write is decomposed into per-branch
base ops (each lowered through `propagate` against a synthetic branch view-like) over an
up-front, Halloween-safe capture of the affected rows + their probe flags, riding the
existing `ViewMutationNode.identityCapture` side input (no new runtime substrate). Covers
`union` / `union all` / `except` / `intersect`; nested/subtree writes and non-literal
membership values stay deferred.

See the implement-stage handoff (in git history: `git show 01bc66af`) for the full
architecture writeup. This ticket records the review pass over that work.

## Review findings

Reviewed the implement-stage diff (`01bc66af`) with fresh eyes against every aspect angle
(Halloween-safety, correlation/scope correctness, per-operator semantics, type safety,
error paths, DRY/reuse, the static-surface/dynamic agreement, resource cleanup), then
re-derived the substrate (capture materialization order, `ViewMutationNode` wiring,
single-source self-alias correlation, `resolveBaseSite` merge) from the source rather than
the handoff. Ran lint + typecheck + the full suite, and probed the implementer's listed
gaps directly with throwaway fixtures.

### Bugs found and fixed inline (minor)

- **Bound parameters in the WHERE of a membership write threw a planner error.** The
  affected-row capture (`buildSetOpCapture`) resolved the user WHERE against a *parentless*
  `RegisteredScope` (gap #5 in the handoff — "capture filter scope has no parent"). A
  parameter (`update U set inB = true where id = ?` / `:name`) — the dominant real-world
  DML form — failed at plan time with `? isn't a parameter` (from `resolveParameter`),
  because the parameter scope was unreachable. Fixed by parenting the capture's view-column
  scope to `ctx.scope` (`set-op.ts:212`): a view output column still shadows ambient symbols
  (registered scope is checked first), and a base-only / projected-away / unknown name in
  the WHERE still rejects with `Column not found` (verified `secret` — a projected-away base
  column — and `nonexistent` both still reject, so view encapsulation is preserved). This
  fixes parameters for update / delete / and (transitively) the data-column refs across all
  membership-write forms. Added a regression test (`bound parameters resolve in the WHERE of
  every membership-write form`) exercising anonymous `?`, named `:p`, delete-by-param,
  insert data-params, and the unknown-column rejection.

### Filed as a new ticket (major-by-honesty, no correctness impact)

- **Static surface over-claims on non-writable branches** (handoff gap #1) →
  `tickets/backlog/set-op-membership-static-surface-branch-writability.md`. `column_info` /
  `view_info` report a membership body fully writable from the AST shape alone (a flag is
  present), without checking each branch is writable. A computed / `select *` / non-SELECT
  operand leg is rejected at *write* time but the static surface still says `YES` — a
  catalog-honesty gap, and an inconsistency with the join path (which *does* gate its static
  surface on decomposability). Not a data-correctness bug (the write rejects cleanly), so
  deferred to backlog rather than fixed in this pass.

### Checked and confirmed correct (no change needed)

- **Halloween-safety.** The capture (`Project(Filter_{userWhere}(SetOperationNode))`) is
  emitted as a side input and fully drained into a `Row[]` by `emitViewMutation` *before*
  any base op fires (`runtime/emit/view-mutation.ts` — `collectRows(args[captureIdx])` then
  `runBody`), and torn down in a `finally`. So a branch insert/delete cannot perturb the
  affected set out from under a sibling op. The handoff's claim that no runtime change was
  needed holds — `view-mutation.ts` was untouched and the existing identity-capture path is
  sufficient.
- **Per-branch correlation / scope.** `buildMemberExists` qualifies branch columns with the
  synthetic branch-view name; `propagate`'s single-source spine binds them to the lowered
  base row via the `__vm_self` self-alias, and `guardTopLevelScope` does not descend into
  the `exists` subquery (so `k.*` capture refs are not mis-rejected). Verified resolution for
  pk WHERE, non-pk data-column WHERE, and a correlated subquery WHERE.
- **Per-operator semantics.** `except` (always-false right flag ⇒ insert-all / left-delete),
  `intersect` (always-true flags ⇒ insert-none / right-delete), `union[all]`, idempotent
  redundant flips, both-false row removal, data fan-out and delete fan-out across A-only /
  B-only / both — all pass, including a `set x = x + 1` data fan-out (column-ref value over
  matching leg names).
- **`resolveBaseSite` merge** (handoff gap #10). The `set-op-branch` and `join-side`
  existence cases now share one writable-through-effect return; the full suite's
  join-existence write tests stay green, and the merged branch returns exactly what the
  join-side case returned before. Confirmed unaffected.
- **`propagate` nested-set-op guard.** A directly/recursively reached membership body
  rejects with a pointer to `set-op-membership-nested` (the AST peek is inlined to keep the
  `set-op.ts → propagate` dependency one-directional). Correct.
- **`k`-named data column** (handoff gap #9). A data column literally named `k` does not
  collide with the `__vmupd_keys` table alias `k` (branch columns are view-name-qualified) —
  verified a `select k, x from A union …` view writes cleanly.

### Deliberately left as documented v1 boundaries (verified as clean rejections / acceptable)

`union all` duplicate-tuple imprecision (gap #2 — largely academic given keyed base tables
reject duplicate full tuples), renamed-leg referenced-column remapping (gap #3), cross-branch
FK ordering (gap #4), conservative contradiction reject (gap #6), insert-through
all-columns-required / VALUES-only (gap #7), and RETURNING rejection (gap #8). Each is
documented in `docs/view-updateability.md` § Set-operation membership writes and behaves as
a clean diagnostic, not a crash.
