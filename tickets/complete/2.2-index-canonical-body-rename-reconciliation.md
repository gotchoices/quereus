# Index canonical body: concurrent column-rename reconciliation (complete)

Follow-on to `index-canonical-body-collation`. The drift detector compares a declared
index body against the actual catalog body. The **actual** body renders the
*pre-rename* column names (a column rename has not landed at diff time), while the
**declared** body renders the *new* names — so a same-named index over a column renamed
in the same diff rendered new-name (declared) vs old-name (actual) and produced an
unnecessary drop+recreate. This reconciles the declared index body (column list **and**
partial-`WHERE` predicate) against the in-diff column renames so a pure rename matches
(no churn — the rename rides the `RENAME COLUMN` table-alter channel) while a genuine
body edit still recreates.

## What landed (implementation)

- **`declaredIndexCanonicalBody`** (`schema-differ.ts`) gained `colRenames` and
  `schemaName` params. Each resolved bare column name is inverse-mapped NEW→OLD
  (case-insensitive) **after** collation resolution (ordering is load-bearing —
  collation resolves on the new declared name first, then the emitted name is mapped
  back). The partial-`WHERE` predicate is reconciled the same way the constraint CHECK
  path is — each renamed column inverse-rewritten NEW→OLD over a `cloneExpr` copy via
  `renameColumnInCheckExpression`.
- **Index loop call site** passes the index table's renames (keyed by the index's
  declared/post-rename table name, reusing the existing `columnRenamesByTable` pre-pass)
  and `targetSchemaName`.
- **`docs/schema.md`** "Index body-change detection" paragraph rewritten (collation
  inclusion + column-rename reconcile + the qualified-`WHERE` exception — see findings).

### Found

- **MINOR (edge, benign churn) — table-qualified self-reference in a partial `WHERE`
  predicate under a concurrent *table* rename spuriously recreates.** The reconcile
  rewrites column *names* inside the predicate but not the *table qualifier*, so
  `where t_old.active = 1` → `where t_new.active = 1` (declared) vs `t_old.active = 1`
  (actual) diverges and drops+recreates. **Empirically confirmed** on this branch.
  This is benign (the recreate is a valid migration — no data loss, no wrong schema,
  just an unnecessary rebuild), **rare** (unqualified predicate refs are idiomatic),
  and **symmetric with the constraint CHECK path's identical limitation** (which is in
  fact slightly worse — it seeds the rewriter with the *old* table name, so a qualified
  declared ref misses the column reconcile too).
  - **Disposition:** doc claim "a *table* rename alone never churns an index body"
    corrected inline (`docs/schema.md`) to scope it to the column list and call out the
    qualified-`WHERE` exception. Code fix (shared index-WHERE + constraint-CHECK
    qualifier reconcile, plus aligning the two seeds) filed as backlog
    `schema-differ-predicate-table-qualifier-reconcile`.

- **MINOR (coverage) — no test pinned the *unqualified* partial-`WHERE`-under-table-rename
  no-churn case** (the implementer's table-rename test had no predicate).
  - **Disposition:** added inline — "a table rename with an UNQUALIFIED partial-WHERE
    predicate does not churn the index" (the 12th case), which also documents the
    qualified exception for the next reader.

### Not found / explicitly empty

- **No correctness bugs.** The reconcile produces correct diffs in every case
  exercised; the only defect is benign churn in the rare qualified-predicate edge above.
- **No DRY / modularity issues.** The index path reuses the constraint path's
  `cloneExpr` + `renameColumnInCheckExpression` + the shared `columnRenamesByTable`
  pre-pass; no new pre-pass, no duplicated rewrite logic.
- **No resource-cleanup / type-safety / lint issues.**

### Carried-forward gaps (from the implement handoff, accepted as out of scope)

  rewriter's own tests.
- `deny` policy: renames undetected by contract (an index over a renamed column
  drops+recreates) — the deny semantics, not a reconcile bug.
- No store-path run (memory backend default); the reconcile is pure diff logic, and the

- Apply-level coverage of the *rename + body-edit recreate* path is pinned only at the
  diff-decision level (the convergence test covers the pure-rename apply path).
