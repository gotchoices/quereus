`Database.watch(scope, handler)` ships: any `ChangeScope` value
(analyzed, deserialized, or hand-built literal) registers as a
post-commit subscription via the shared `DeltaExecutor` kernel.
Validation rejects unbound params, missing tables, and missing
columns synchronously; handler errors are caught and never roll the
commit back; schema changes on a watched table dispose the
subscription.

## Review findings

### Scrutiny matrix

| Aspect | Verdict |

### Minor â€” fixed in this pass

1. **`DeltaExecutor.runAll` did not actually snapshot subscriptions.**
   The implementer's handoff claimed the kernel iterates a snapshot per commit;
   the code iterated the live `Set` directly. ES spec for `Set` iteration visits
   entries added during iteration â€” so a handler that called `db.watch(...)`
   synchronously inside its fire could have its new subscription fire on the
   *same* commit (before it was meant to). Replaced the loop with a snapshot
   pass that also skips entries unsubscribed by a peer handler mid-fire.
   `packages/quereus/src/runtime/delta-executor.ts`.

2. **`baseKeyFor` / `relKeyForWatch` did not lowercase the qualified
   table name.** `QualifiedName`'s contract says "lowercased", but
   hand-built `ChangeScope` values may not honor it. The change log is
   keyed lowercased; an uppercase `dependencies` entry would never match
   `getChangedBaseTables()` and the subscription would silently never
   fire. Lowercased defensively at the helper boundary. Added a test
   asserting a hand-built scope with `'Main'/'T'` resolves and fires.
   `packages/quereus/src/runtime/delta-executor.ts`,
   `packages/quereus/test/logic/change-scope.spec.ts`.

3. **`literalValues: ... | null` was a dead type.** The docstring on
   `literalValuesOnly` claimed it could return `null`; the implementation
   always returned an array. The downstream `?? []` coalescers were
   dead. Narrowed the type to `ReadonlyArray<ReadonlyArray<SqlValue>>`
   end-to-end and dropped the nullish handling. No behavior change.
   `packages/quereus/src/runtime/delta-executor.ts`.

4. **Missing schema-change-invalidation test.** Implementer flagged it
   as a "floor not ceiling" gap. Added: subscribe â†’ `DROP TABLE` â†’
   re-create same table â†’ mutate â†’ assert handler did not fire and
   `unsubscribe()` is still safe.
   `packages/quereus/test/logic/change-scope.spec.ts`.

### Minor â€” design notes, no change

5. **`DeltaSubscription.dispose()` is empty for watcher subscriptions.**
   The `DeltaSubscription` contract says "Release any external resources
   this subscription holds," but the watcher path delegates capture-spec
   disposal to the caller (`WatcherManager` owns the disposers). Works
   because the manager wraps everything, but worth noting if a third
   `DeltaExecutor` consumer is added. Not worth restructuring for v1.

6. **Sequential handler awaits inside `runAll`.** A slow async handler
   delays subsequent watchers and the return of `commitTransaction`
   (data is already durable; only the SQL call's promise is delayed).
   Acceptable for v1; if it becomes a problem, the kernel could fire
   handlers in parallel with `Promise.allSettled`. Not documented in
   `change-scope.md` either way; readers can infer from "watcher
   handler runs post-commit, sequentially."

## What landed (verbatim from implementer handoff, for archive)

Public exports added (re-exported from `packages/quereus/src/index.ts`):
- `Subscription`, `WatchEvent`, `MatchedWatch`, `WatchHandler` from
  `planner/analysis/change-scope.ts`.
- `subscriptionFromChangeScope`, `ChangeScopeTableInfo`,
  `SubscriptionFromChangeScopeContext`,
  `SubscriptionFromChangeScopeResult` from `runtime/delta-executor.ts`
  (used by `Database.watch`).
- `Database.watch(scope, handler): Subscription`.

`WatcherManager` (`core/database-watchers.ts`) owns the watcher half:
constructs its own `DeltaExecutor`; on `watch()` validates the scope,
resolves columns, registers capture-spec demand, builds the
`DeltaSubscription` via `subscriptionFromChangeScope`, returns a
`Subscription` whose `unsubscribe` is idempotent and releases all
resources. `runPostCommit()` is invoked from
`TransactionManager.commitTransaction` after every connection commits
and before the change log is cleared. Errors caught + logged.
Schema-change listener disposes subscriptions whose table is
removed/modified (no auto-rebind).

`subscriptionFromChangeScope` maps the public scope to a
`DeltaSubscription`:

| `WatchScope.kind` | Internal `BindingMode` | Notes |
| --- | --- | --- |
| `full` | `{kind: 'global'}` | Capture spec for non-PK named columns when `columns !== 'all'`. |
| `rows` | `{kind: 'row', keyColumns}` | Apply intersects kernel tuples against literal `values`. |
| `groups` | `{kind: 'group', groupColumns}` | Hits = kernel-emitted tuples. |
| `rowsByGroup` | `{kind: 'group', groupColumns}` | Apply intersects kernel tuples against literal `values`. |

txnId is an opaque `txn:<n>` from a monotonic counter inside
`WatcherManager` (stable within one `Database`, not portable).

## Known limitations carried forward (v1)

- **Column tracking on `full` watches is best-effort (over-fires).**
  A `{kind: 'full', columns: {a, b}}` watch registers capture demand
  for `a` and `b`, but the watcher always fires on any change to the
  table â€” the change-log API doesn't yet expose a per-row "did column
  X change" predicate. Sound (never misses) but coarse. Documented in
  `docs/change-scope.md`. A future ticket can plumb a precise
  predicate through `TransactionManager`.
- **No auto-rebind on schema change.** A drop/modify on a watched
  table disposes the subscription; callers re-subscribe against the
  new schema. Documented.
- **Watcher handler errors do not roll commits back.** Asymmetric
  with assertions by design. Documented.
