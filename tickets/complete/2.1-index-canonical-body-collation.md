# Index canonical body — per-column collation (complete)

Follow-on to `schema-differ-index-body-drift`, which deliberately EXCLUDED
collation from the index canonical body. This ticket includes collation
correctly: both diff sides resolve each column's *effective* collation the same
way the engine does at create/import time (explicit index COLLATE → table-column
collation → BINARY, normalized), then render both sides through one canonical
renderer (BINARY elided, non-BINARY explicit). An unchanged inherited/BINARY
collation renders identically (no churn); a genuine collation change diverges
(drop+recreate).

## What landed (implementation)

- **Renderer** (`ast-stringify.ts`): `canonicalIndexedColumnsToString` now emits
  `name [collate <c>] [desc]`; new `canonicalIndexColumnCollation` returns the
  effective collation only when non-BINARY; `indexedColumnBareName` exported for
  the differ.
- **Actual side** (`ddl-generator.ts`, `indexToCanonicalDDL`): each lifted column
  carries `normalizeCollationName(IndexColumnSchema.collation ?? table-column
  collation ?? 'BINARY')`.
- **Declared side** (`schema-differ.ts`, `declaredIndexCanonicalBody` +
  `explicitIndexColumnCollation` + `declaredColumnCollation`): resolves each
  declared column's effective collation against the matched declared table and
  renders the plain-form column list through the same renderer.
- **Catalog doc** (`catalog.ts`): `CatalogIndex.definition` doc updated (collation
  now included).
- Tests: 9 new collation cases + 1 retitled.

## Review findings

Diff reviewed fresh (commit `de8513c3`) before the handoff. The CORE logic —
canonical-body collation rendering and the diff DECISION — is correct, symmetric,
and well-tested. Verified directly:

- **Renderer / resolution symmetry** — both `buildIndexSchema` (manager.ts ~2068)
  and `importIndex`/`resolveImportedIndexColumn` (~2470) resolve the effective
  collation identically and store it on `IndexColumnSchema.collation`, which
  `indexToCanonicalDDL` lifts; the declared side resolves the same way. Both feed
  one renderer → byte-comparable. `normalizeCollationName` (trim+upper) and
  `extractDeclaredCollation` (upper, BINARY default) agree on both sides. ✔
- **Direction preservation through the folded form** — the parser folds
  `col COLLATE x` into `{expr, direction}`; `declaredIndexCanonicalBody` reads the
  bare name off `col.expr` and carries `col.direction`, so the canonical BODY for
  `email collate nocase desc` is correct on both sides. ✔ (BODY only — see the
  apply-path gap below.)
- **Test helper genuinely exercises the actual side** — `diffIndexEdit` does
  `apply schema` → `collectSchemaCatalog`, so the actual `definition` comes from
  the real engine, not a declared-to-declared shortcut. ✔
- **require-hint policy** — a pure-collation recreate is a body change, already
  excluded from the unhinted-rename guard; covered, no code change needed. ✔
- **Lint** clean; **full quereus suite** 5219 passing, 10 pending, exit 0.

### MAJOR — filed as `tickets/fix/index-explicit-column-collate-apply-path.md`

The differ now EMITS recreate DDL for an explicit-index-COLLATE change, but the
live engine cannot apply it. Confirmed end-to-end (memory backend):

- `apply` of `…index ix on t (email collate nocase)` fails with
  `Indices on expressions are not supported yet.` — `buildIndexSchema`
  (manager.ts ~2053) throws on any `indexedCol.expr`, and the parser folds *every*
  explicit COLLATE into an `expr`. So the live `CREATE INDEX … (col COLLATE x)`
  path is unsupported (only the import path unwraps it).
- `createIndexToString` additionally drops the trailing `desc` for the folded
  form (`email collate nocase desc` → `email collate nocase`).
- The COLUMN-collation-driven recreate (plain index form, e.g.
  `index ix on t (name)` while `name` gains `collate nocase`) applies cleanly —
  only the EXPLICIT-index-COLLATE form (Case B) breaks.

This is a pre-existing live-path limitation (a brand-new declared index with an
explicit COLLATE was always unapplicable), but this ticket extends the blast
radius to RECREATES of existing indexes. The new tests masked it by asserting only
the diff DECISION, never applying the modified schema. The fix ticket requires:
support the collate-folded form in `buildIndexSchema`, emit direction for it in
`createIndexToString`, add apply-level assertions, and re-enable the pending test.

### MAJOR (secondary) — folded into the same fix ticket

Apply ordering: `generateMigrationDDL` emits `indexesToCreate` BEFORE the
`SET COLLATE` column alter, so a collation-driven recreate's `CREATE INDEX` reads
the column's OLD collation. Confirmed emitted order
(`DROP INDEX` → `create index` → `ALTER COLUMN … SET COLLATE`). Currently benign:
the memory backend returns correct collation-aware results after this sequence
(verified — NOCASE lookup found the case-differing row), and the store keys
secondary indexes under a single TABLE-LEVEL collation (not per-column), so a
per-column SET COLLATE never re-keys them anyway. Becomes a real stale-key hazard
only on a future backend that keys secondary indexes by per-column collation and
resolves index collation from the column at CREATE INDEX time. Captured with two
candidate fixes in the fix ticket.

### Minor — reviewed, no change

- `declaredIndexCanonicalBody`'s trailing `|| 'BINARY'` is unreachable
  (`declaredColumnCollation` always returns a non-empty string). Harmless
  belt-and-suspenders, consistent with the codebase's defensive style. Left as-is.
- `canonicalIndexColumnCollation`'s defensive folded-form read
  (`col.expr.collation`) is unreached in production (both callers pre-resolve to
  plain form). Intentional insurance, documented. Kept.
- `declaredColumnCollation` returning `'BINARY'` for an index on a non-declared
  table never feeds a real comparison (such an index is a pure create/drop, not a
  body compare). Correct.
