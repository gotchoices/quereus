### Found and fixed in this pass

All four were the same defect class: prose describing code the project does not have. Per
`docs/doc-conventions.md`, a doc that has drifted is worse than no doc, and future work
leaves the docs entirely. The implementer flagged all four honestly and declined to act on
them, calling the first "the single biggest open decision." Each is a bounded, well-scoped
correction, so all four were fixed here rather than deferred.

1. **`ApplyNode` does not exist** (`docs/optimizer-retrieve.md`). ~1,300 words described an
   `ApplyNode` abstraction that "replaces traditional `JoinNode` semantics". Verified: no
   such symbol anywhere under `packages/quereus/src`, and nothing outside the doc
   referenced it. The design proposal was condensed into a new **Phase 4 – Correlated
   push-down** entry under `docs/todo.md § Push-down & Federation Roadmap`, preserving the
   substance (the `Apply(left, right, predicate, outer)` shape, the SQL mapping rationale,
   why it exists — push-down, not execution) with four checklist items. Its place in the
   doc is now a short `## Correlated and lateral access` section describing what actually
   happens today: a `JoinNode` driven by the nested-loop emitter, with the right-side
   `RetrieveNode` re-assessed per outer row.
2. **QuickPick's `### Algorithm Design`** (`docs/optimizer-joins.md`) showed a
   `class QuickPickOptimizer` with `async optimizeJoins()` and a `Math.random()` shuffle.
   Verified absent. The real rule is `rules/join/rule-quickpick-enumeration.ts` — a plain
   `(node, context) => PlanNode | null`. Replaced the fabricated class with a four-step
   description of what the rule does: `extractJoinGraph`, side-effect refusal, greedy tours
   (10× cross-product penalty, `maxTours`/`timeLimitMs` bounded, start relation alternating
   between the two smallest) plus one greedy bushy plan, and adoption only below 90% of
   baseline cost. Also corrected "a min-heap of best plans" (it keeps a single running
   best).
3. **`estimatedCostFromNode.getTotalCost()`** (`docs/optimizer-joins.md § Integration
   Points`) names no symbol in the tree. The real surface is `PlanNode.getTotalCost()`, used
   by `estimatePlanCost`. Corrected, and the entry now also records `sideEffectMode: 'aware'`
   and the actual `tuning.quickpick` fields (`enabled`, `maxTours`, `timeLimitMs`,
   `minTriggerCost`).
4. **`## Optimization Pipeline Architecture § Phase Sequencing Strategy`** (hub) described
   four phases, of which phase 1's method was **wrong** (it said "bottom-up sliding";
   `ruleGrowRetrieve` is registered on `PassId.Structural`, which runs **top-down** — and
   the retrieve doc said so, contradicting the hub), and phases 3–4 described passes that do
   not exist ("traditional dynamic programming" join enumeration; a "cost-precise" advanced
   push-down phase) while duplicating `docs/todo.md § Push-down & Federation Roadmap`
   verbatim in spirit. Replaced with `### Why the segment boundary comes first`: the
   rationale for growing segments before enumerating joins (structural growth never consults
   cost, so it is deterministic; fixing segments first is what makes the enumerator's
   cardinalities meaningful), a pointer to `#multi-pass-optimization-system` for the real
   pass list, and a pointer to `todo.md` for the roadmap. Verified the priority ordering it
   now asserts: `grow-retrieve` priority 10 < `predicate-pushdown` priority 20 within
   Structural, and `quickpick-join-enumeration` sits in the Physical pass (lower priority
   runs first, per `framework/registry.ts:58`).

Two further corrections found while verifying the above:

5. **Duplicated `ruleGrowRetrieve` description** (`docs/optimizer-retrieve.md`). The file
   carried both `### Dynamic support growth with ruleGrowRetrieve` and
   `### ruleGrowRetrieve Design`, describing the same rule with different wording and
   contradicting each other on traversal direction. Merged into one accurate section
   (algorithm, worked example, key properties). Two "establishes Quereus as a powerful
   federation engine"-style marketing sentences went with the deletion.
6. **`## Visited Tracking Architecture § Integration with Advanced Optimizations`** (hub)
   claimed "Each tour gets fresh context / Best plan selected across all contexts" for
   QuickPick. False: tours are pure plan construction inside one rule invocation and never
   touch `OptContext`. Rewritten to state what is actually true of the interaction — the
   rule costs candidates outside the visited set and records exactly one transform.
