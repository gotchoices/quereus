# Complete: retire the process-global collation registry

## What shipped

The module-level `Map<string, CollationFunction>` in `util/comparison.ts` and its three
readers — `registerCollation`, `getCollation`, `resolveCollation` — are gone, along with
the `createLogger`/`warnLog` bindings that existed only to serve them. A collation name
now has meaning in exactly one place: on a `Database`. Register with
`db.registerCollation(name, func, options?)`; resolve with `db.getCollationResolver()`,
which is case-insensitive, fast-paths the non-overridable `BINARY`, and throws
`no such collation sequence: <name>` rather than silently degrading to byte order.
`builtinCollationResolver` remains as the switch-based BINARY/NOCASE/RTRIM floor for code
with no `Database` in scope.

Two signature changes convert a would-be silent bug into a compile error:

- `compareSqlValues(a, b, collationName?)` → `compareSqlValues(a, b)`. It is BINARY, full
  stop.
- `compareWithOrderBy` and `createOrderByComparator` are deleted. They had no production
  callers; their `*Fast` siblings, which take a pre-resolved `CollationFunction`, are what
  emitters actually use.

Public surface (`src/index.ts`): dropped the three registry functions,
`compareWithOrderBy`, and `createOrderByComparator`; added `normalizeCollationName`,
`createTypedRowComparator`, and `createCollationRowComparator` (the last was used
internally and documented in `docs/runtime.md` but never exported).

`runtime/emit/materialized-view-helpers.ts`'s `assertDerivedRowsAreSet` — the "a maintained
table must be a set" gate, reached on `create table … maintained as` and on the
constraint-bearing arm of `refresh materialized view` — was the last production caller of
the three-argument `compareSqlValues`. It now takes a `CollationResolver`, threaded from
`db.getCollationResolver()` at both call sites, and resolves the per-column key collations
once above the sort via `resolveCollationFunctions`.

Docs updated: `packages/quereus/README.md`, `docs/plugins.md` (import list and the
collation-management API block), `docs/runtime.md` (two emitter examples that imported the
global `resolveCollation`; they use `ctx.resolveCollation` now), `docs/architecture.md`.

## Review findings

### Checked

- **Read the implement diff first, before the handoff summary.** Re-derived the call graph
  independently rather than trusting the summary's claims.
- **Dead-symbol sweep.** `resolveCollation` / `registerCollation` / `getCollation` /
  `compareWithOrderBy` / `createOrderByComparator` across all `packages/`, `docs/`, and
  READMEs. The only surviving `resolveCollation` identifiers are two different, legitimate,
  database-scoped functions: `planner/building/schema-resolution.ts` (build-time, throws on
  unknown name) and `EmissionContext.resolveCollation` (emit-time, records the dependency
  for plan invalidation). No confusion crept in; both are reached only with a `Database` in
  hand. No caller passes a collation name to a comparison utility anywhere.
- **Three-argument `compareSqlValues` call sites.** All remaining calls are two-argument.
  The three in production (`rule-select-access-path.ts`, `stats/analyze.ts`,
  `vtab/memory/layer/manager.ts`) were already BINARY-intentional.
- **Resolver equivalence at the set gate.** The old path upper-cased the name inside
  `resolveCollation`; the new path goes through `Database._getCollation`, which applies
  `normalizeCollationName` (trim + uppercase). Same normalization, so no case-sensitivity
  regression on declared `collate` names. The batch resolve sits after the `rows.length < 2`
  early return, so a trivial gate still costs nothing.
- **`docs/plugins.md` collation block against the real signature.** The documented
  `db.registerCollation(name, func, optionsOrNormalizer?)` and
  `db.getCollationResolver(): CollationResolver` match `core/database.ts` exactly, including
  the `{ normalizer?, replicable? }` union.
- **Plugin surface.** `sample-plugins/` and `plugin-loader/` never touched the global
  registry; `util/plugin-helper.ts` already routed plugin collations through
  `db.registerCollation`. Confirmed, no change needed.
- **`yarn build` (all packages), `yarn lint`, `yarn test`.** All clean.
  **6645 passing, 0 failing, 9 pending** (implement handed off 6643; this review added 2).

### Found and fixed in this pass

- **The handoff's central "known gap" was wrong, and the gap was a real coverage hole.**
  The implementer believed the set gate's *rejecting* branch was unreachable under a
  collation-sensitive comparison, having only tried bodies whose derived key is the source
  table's own primary key — where the source PK has already deduplicated under that very
  collation. But the derived key need not be a source key. With
  `create table src (id integer primary key, x text collate nocase not null)` holding
  `'aa'` and `'AA'`, the body `select x from src` derives a colliding `nocase` key, and the
  gate rejects it with `produces duplicate rows for primary key ('AA')`. The gate runs
  *before* the coverage prover's bag check, so its verdict is observable: under a database
  that overrides `NOCASE` to fold nothing, the same attach passes the gate and is rejected
  later, by the prover, with a different message. That pair discriminates database-resolved
  from globally-resolved collation on the rejecting branch, and the pre-change code failed
  it. Added `describe('the rejecting branch also decides under the database collation')` to
  `test/mv-custom-collation-maintenance.spec.ts` with both halves, and corrected the
  now-false reasoning in the parent describe's comment.

  (Note for the reader: the gate asserting on `produces duplicate rows for primary key`
  rather than on `must be a set` is deliberate — the coverage prover's bag error *also*
  contains the phrase "must be a set", so the looser pattern matches both errors and proves
  nothing. The first version of this test passed vacuously for that reason.)

- **The new `NOTE:` on `compareRows` stated something untrue.** It claimed the function
  "backs DISTINCT ordering in a BTree whose caller has already applied collation-aware key
  normalization, so the collation has been folded into the bytes before this ever runs."
  No such caller exists. `compareRows`'s sole production caller is the recursive-CTE
  `union` (DISTINCT) dedup BTree in `runtime/emit/recursive-cte.ts`, which feeds it raw,
  unnormalized rows. The `distinct` and `set-operation` emitters — the ones that *are*
  collation-aware — use `createCollationRowComparator` and never touch `compareRows`.
  A future reader trusting that comment would have concluded the recursive-CTE dedup was
  collation-correct when it is plain BINARY. Rewrote the comment to say what is actually
  true, and why BINARY is nonetheless correct there: SQLite's recursive queue table carries
  no `COLLATE` either, so byte dedup is the matching behavior, not an oversight.

### Found, filed as tripwires (not tickets)

- **Recursive-CTE `union` dedup is BINARY while compound `union` is collation-aware.**
  Fine today — it matches SQLite, and the divergence is in SQLite too. It only becomes work
  if Quereus ever decides its recursive queue should carry the body's declared collations.
  Parked as the `NOTE:` on `compareRows` in `util/comparison.ts`, at the exact site, telling
  a future implementer to take pre-resolved `CollationFunction`s rather than re-introduce a
  name parameter.

### Found, not actioned

- **`docs/review.html` still describes the global registry as live** ("Deprecated global
  registry is still the live fallback inside `compareSqlValues`"). Left alone on purpose:
  the file is a dated snapshot — `<title>Quereus Code & Design Review — July 2026</title>` —
  not a living document. Editing it would falsify the record of what the review found. The
  finding it records is precisely what this ticket chain (3.4 → 3.6) closed.

- **Historical "used to" comments in sibling test suites**
  (`quereus-store/test/custom-collation.spec.ts`,
  `quereus-isolation/test/collation-resolver.spec.ts`,
  `quereus/test/vtab/memory-collation-per-database.spec.ts`) reference the three-argument
  `compareSqlValues` and the process-global registry. All three are phrased in the past
  tense as *what the code used to do*, explaining why the suite exists. Accurate as written.

### Deliberately empty

- **No major findings, so no new `fix/`, `plan/`, or `backlog/` tickets.** The one real
  defect (the missing rejecting-branch coverage) was a test gap fixable in this pass, and
  the one wrong comment was a comment. Nothing in the diff is architecturally unsound: the
  change strictly narrows an API surface, and every removal is enforced by the type checker
  rather than by convention.

- **No performance findings.** The gate's `resolveCollationFunctions` runs once per attach
  or refresh, above the sort, replacing a per-comparison `resolveCollation` lookup — it is
  strictly less work on a path that already sorts the full derived row set. Not benchmarked,
  and not worth benchmarking: the sort dominates by orders of magnitude.

- **No resource-cleanup or error-handling findings.** The diff introduces no allocations,
  no handles, and no `catch`. Its one new throw path (`no such collation sequence` from the
  set gate) replaces a silent BINARY fallback and matches what
  `resolveBackingPkColumns` already did on the same schema, so the two agree.

## Deferred to a human / CI

`yarn test:store` was not run — it exceeds the 10-minute agent idle budget and is therefore
not agent-runnable. The store suite has a `using store(...) args: PK key collation` case
that exercises a NOCASE-PK set gate. Nothing in the diff is store-specific and the store
package's own collation suite passes under `yarn test`, but the set gate is shared code, so
one out-of-band `yarn test:store` before release is the honest bar.
