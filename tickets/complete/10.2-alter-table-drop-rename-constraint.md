# COMPLETE: ALTER TABLE DROP/RENAME CONSTRAINT + declarative constraint lifecycle

Two new ALTER TABLE primitives (`DROP CONSTRAINT <name>`, `RENAME CONSTRAINT <old> TO <new>`) for
named CHECK / UNIQUE / FOREIGN KEY constraints, plus the declarative constraint lifecycle
(rename / drop / add **by name**) wired into the schema differ. See the implement-stage commit
`045e56f3` for the full design; the summary there is accurate.

## Review findings

### Verdict
Implementation is solid, well-decomposed, and matches the handoff. Build clean, lint clean, full
suite green (**4798 passing / 0 failing / 9 pending**, memory). One minor gap fixed inline; one
documented limitation formalized as a follow-on ticket. No correctness regressions found.

### What was checked

**Diff read end-to-end** (parser → AST → stringify → plan node → build → runtime emit → module
layer → memory manager → store module → isolation module → differ → catalog → docs → tests), with
fresh eyes before the handoff.

- **Class resolution DRY** — `resolveNamedConstraintClass` (schema/table.ts) is the single
  check→unique→fk resolver; `SchemaManager.setConstraintTags`, both runtime emitters, the memory
  manager, and the store module all share it. Confirmed the refactor of `setConstraintTags` onto it
  preserves the prior NOTFOUND/ambiguous semantics. ✓
- **Runtime gating** (`runDropConstraint` / `runRenameConstraint`) — resolves class up front (clear
  NOTFOUND/ambiguous), rejects `derivedFromIndex` UNIQUEs before any module call, checks rename
  collision (case-insensitive, allows case-only self-rename), then routes through `module.alterTable`
  so persistent modules re-persist DDL. Order of checks is correct. ✓
- **Memory covering-index teardown/rename** — `dropConstraint` tears down the implicit covering
  index (auto-named after the constraint) and deletes the `implicitCoveringStructures` record;
  `renameConstraint` renames the index in lock-step and `rebuildAllSecondaryIndexes()` so the base
  layer's index map matches the new key. Both wrap in the SchemaChange latch with full rollback to
  the pre-op snapshot on error. The named-UNIQUE-reusing-a-pre-existing-index corner (index keeps
  its own name, not torn down/renamed) is consistent — that index is the user's. ✓
- **Store module** — pure schema rewrite + `saveTableDDL` for all three classes; inline UNIQUE in
  store is full-scan enforced so there is no index store to touch. `derivedFromIndex` rejection is
  upstream (runtime), so no store can be stranded. ✓
- **Differ** — the real latent dead-code bug it fixes (`constraintsToRename` populated but never
  emitted by `generateMigrationDDL`) is genuine and now covered by a dedicated regression test.
  `constraintsToDrop` / `constraintsToAdd` gather declared named constraints from **both**
  table-level and column-level (named) constraints, symmetric with the catalog's `namedConstraints`.
  Auto-names (`_`-prefix) and `derivedFromIndex` UNIQUEs are excluded on both sides → idempotent.
  Migration order (RENAME → DROP → ALTER PK → ADD → DROP COLUMN) is sound. ✓
- **Exhaustiveness** — isolation module and both memory dispatch points (`module.ts`, `table.ts`)
  handle the two new `SchemaChangeInfo` arms; isolation's `translateOverlayRow` correctly no-ops
  (no row-shape change). ✓
- **Docs** — `docs/sql.md` §2.7 and `docs/schema.md` (Migration Order + lifecycle + rename
  detection) read accurately against the code, including the honest UNIQUE/FK ADD limitation. ✓

### Tests — checked happy path, edges, errors, regressions, interactions
The implementer's tests cover enforcement on/off after drop (CHECK/UNIQUE/FK), name change after
rename (via `*_constraint_info` TVFs), error cases (NOTFOUND, ambiguous, rename collision,
derived-from-index rejection), memory covering-index teardown/rename, declarative add/drop/rename
convergence + idempotency, the dead-rename regression, and the emit round-trip property. This is a
genuinely strong floor.

### Findings & disposition

1. **[MINOR — fixed inline] Optimizer-folding invalidation on CHECK drop was untested**
   (handoff gap #2). CHECK constraints lift FDs/constant-bindings onto `TableReferenceNode`
   (`getCheckExtraction`), which the filter-contradiction rule uses to fold contradictory predicates
   to `EmptyRelation`. I verified by probe that `DROP CONSTRAINT` of a `check (x = 1)` correctly
   stops `where x = 2` from folding (the `table_modified` notification invalidates the cached plan)
   — **behavior is correct**. Added a permanent regression guard as section 8 of
   `test/logic/41.6-alter-drop-rename-constraint.sqllogic` (runs under both memory and store; both
   verified passing in isolation).

2. **[MAJOR — ticketed] Declarative ADD of a UNIQUE/FOREIGN KEY to an existing table fails at
   apply** (handoff gap #1). The differ emits `ALTER TABLE … ADD <fragment>` for any declared-absent
   named constraint, but `ADD CONSTRAINT` for non-CHECK routes through `module.alterTable`, and both
   built-in modules throw `StatusCode.UNSUPPORTED` (CHECK is handled engine-side in
   `runtime/emit/add-constraint.ts` and works). The migration batch rolls back atomically (verified
   via the `runBatchedMigrationLoop` / `endSchemaBatch(error)` overlay-discard path) — safe but poor
   UX. This is **pre-existing** (the module gate predates this ticket; `alter-add-constraint.spec.ts`
   still pins it), but this ticket newly made it *declaratively reachable*. It also **blocks the
   UNIQUE/FK path of follow-on ticket 10.3** (`alter-constraint-body-change-drop-add`), whose
   drop+add realization of a body change needs a working ADD.
   - **Filed `tickets/plan/10.25-module-add-constraint-unique-fk.md`** to implement module
     `addConstraint` for UNIQUE/FK (the proper fix — re-validates existing rows, atomic on
     violation).
   - **Added it as a prereq of ticket 10.3** so 10.3's UNIQUE/FK body-change path can't hit the same
     wall.
   - Kept the current emit-and-fail behavior (documented + atomic) rather than hard-erroring early in
     the differ — the resolution is to make the ADD work, not to special-case its absence. Flagged
     the interim-UX decision for the human in the new ticket.

3. **[OBSERVATION — not actioned, pre-existing & out of scope] `ADD CONSTRAINT CHECK` does not
   round-trip through `module.alterTable`** (`runAddCheck` mutates `checkConstraints` directly and
   only fires `table_modified`). For a **store-backed** table this means an engine-path CHECK add may
   not call `saveTableDDL`, so its DDL might not persist across reconnect. This predates this ticket
   and lives outside its diff, but the new declarative CHECK-add lifecycle makes it more reachable.
   Not chased here (reconnect persistence is non-trivial to assert in the current harness and is a
   separate concern); flagged for a future pass if store DDL persistence of engine-path CHECK adds
   proves to be a real gap.

- **Error handling** — clear, typed `QuereusError`s with correct `StatusCode`s (NOTFOUND / ERROR /
