# Maintained-table differ transitions ‚Äî COMPLETE

Part 3 of the maintained-table chain. A `declare schema` transition on a named
maintained table applies as a **catalog flip + content reconcile**, never a
drop+recreate. A maintained table round-trips in the `tables` catalog category
with a `maintained` descriptor; the differ recognizes attach (`set maintained
as`), detach (`drop maintained`), and body-drift re-attach as non-destructive
alter ops via `applyMaintainedTransition`. `generateMigrationDDL` orders detach
EARLY and attach/re-attach LATE. Tag-only changes on a still-maintained table
route through `ALTER MATERIALIZED VIEW ‚Ä¶ SET TAGS`.

See the implement-stage commit (`ticket(implement): maintained-table-differ-transitions`)
for the full landing summary.

### Correctness ‚Äî scrutinized, no defects found
- **Idempotence workaround (`maintainedBodyMatches`) ‚Äî verified sound, no false
  negatives.** The dual-hash tolerance (accept the implicit `columns: undefined`
  hash OR the live-column-names hash) only ever differs in the rename-list
  *prefix* of the canonical string; the body SQL and `insert defaults` come from
  the declared clause in BOTH variants. A genuine body/defaults edit changes that
  shared portion and fails both variants, so it cannot be masked. The live-names
  variant is correctly restricted to the implicit form (`declared.columns ===
  undefined`), so an explicit rename-list change (`mv (a,b)`‚Üí`(a,c)`) is still
  detected. The implementer's stated reasoning holds.
- **Tag routing across every transition combo ‚Äî verified correct.**
  `markMaintainedTagRoute` gates `maintainedTags` on `liveMaintained &&
  !dropMaintained`. Walked all cases (attach+tags, pure re-attach+tags,
  reshape-leg re-attach+tags, detach): in each, the SET TAGS verb (ALTER TABLE vs
  ALTER MATERIALIZED VIEW) matches the table's maintained-ness at the point the
  alter block runs, given the early-detach / late-attach ordering. The
  body+tags-both-changed test pins this.
- **Cross-schema rendering ‚Äî verified.** `set maintained as` renders via
  `astToString` over an `alterTable` AST whose `table` identifier carries the
  schema; `alterTableToString` emits it through `expressionToString` (handles
  `schema.name`). `drop maintained` uses the same `schemaPrefix` as every other
  ordered op. Symmetric and correct.
- **Fresh create of the declared-shape table form ‚Äî verified.**
  `applyTableDefaults` spreads the stmt (preserves `maintained`), and
  `createTableToString` emits the `maintained as` clause. Confirmed end-to-end by
  a new test.
- **Drop ordering (orphan maintained table + its source in one apply) ‚Äî

  drop via FK-ordered `tablesToDrop`). Probed both the both-drop and the
