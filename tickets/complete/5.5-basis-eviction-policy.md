# Basis-table eviction policy & network-wide retirement signal — COMPLETE

The dynamic half of the legacy-table retirement model (`docs/migration.md` § 4
Contract), built on the static `basis-lifecycle-classification` record. Adds the
network-wide "is any peer still writing this table directly?" signal
(`lastDirectlyMappedWriteAt`) and the default retention-horizon eviction policy
that reclaims a detached legacy table's local storage once it has been quiet.

## What shipped (unchanged from the implement handoff)

- **Dynamic signal** — `bumpLastDirectlyMappedWrites` (change-applicator), inside
  the admission `commitMetadata` closure: each applied (remote) write to a tracked
  non-`directly-mapped` basis table bumps `lastDirectlyMappedWriteAt =
  max(current, change.hlc.wallTime)`. Advisory (try/catch-swallowed), batched,
  zero-overhead pre-migration.
- **Effective-horizon math** (`basis-lifecycle.ts`) — `effectiveEvictHorizonMs`
  (per-table override > global mode), `quietSince = max(unmappedSince ??
  detachedAt, lastDirectlyMappedWriteAt)`, `isEvictable` (detached **only** +
  horizon elapsed).
- **Capture at observation time** — `recordLensDeployment` snapshots the
  `quereus.sync.evict` tag → `evictPolicy` and the secondary-index names →
  `indexNames` while in-basis, stamps `detachedAt` on entering `detached`, and
  carries both through detach (schema + tag are gone afterward).
- **The sweep** — `SyncManager.evictExpiredBasisTables(now?)`, host-driven (no
  library timer). Re-reads + re-checks each candidate before the drop, calls
  `dropLocalTable(schema, table, indexNames)` → store `reclaimDetachedTable`,
  emits `onBasisTableEvicted`, clears the record. Idempotent on drop failure;
  no-op when no `dropLocalTable` is wired.
- **The knobs** — `SyncConfig.basisEviction: { mode, horizonMs? }` (default
  `{ mode: 'horizon' }`); per-table reserved tag `quereus.sync.evict =
  never | immediate | <ms>` (`SYNC_EVICT_TAG`, `'eviction-policy'` value schema).

## Review findings

### Adversarial verification of the load-bearing correctness claims

- **Gap #1 — dynamic bump fires only while in-basis-but-legacy — CONFIRMED
  CORRECT.** Traced both paths against the recorder's stamping
  (`sync-manager-impl.ts:413-432`):
  - *directly-mapped → detached in one deploy:* `wasMapped→!isMapped` stamps
    `unmappedSince = now`; `!wasDetached→isDetached` stamps `detachedAt = now`; the
    table was never bumped (bumps skip `directly-mapped`), so `quietSince = now`
    (the detach instant). Eviction measured from detach. ✓
  - *never-directly-mapped (derivation-source-only → detached):* `unmappedSince`
    never set → `quietSince` falls back to `max(detachedAt, lastDirectlyMappedWriteAt)`;
    in-basis foreign writes bumped the clock, but `detachedAt ≥` any of them, so
    `quietSince = detachedAt`. ✓
  Once `detached`, inbound writes are out-of-basis and diverted by unknown-table
  detection (`change-applicator.ts:166-177`) — never reaching `appliedChanges`, so
  the clock is correctly frozen at detach. A straggler write after eviction is
  quarantined, not lost (composition test). The conservative model holds:
  over-counting only delays reclamation; under-counting cannot occur.
- **Gap #2 — the between-list-and-drop re-check is defensive, not adversarially
  unit-tested — ACCEPTED.** In the single-threaded event loop the only interleaving
  point is the `await this.basisLifecycle.get(...)` between the `list()` scan and
  the drop; a concurrent `applyChanges` (clock bump) or `recordLensDeployment`
  (re-map) there is caught by the re-read + `isEvictable(fresh, …)` re-check. The
  `re-deploy (re-map) before the sweep` test covers the primary scenario; the
  re-check carries a clear justifying comment (`sync-manager-impl.ts`). Forcing a
  true mid-sweep interleave would require mocking `get` to mutate — low value over
  the existing comment + ordering test. Left as-is; no ticket filed.
- **In-basis `unreferenced` never auto-dropped — CONFIRMED for every
  policy/mode.** `isEvictable` gates on `state === 'detached'` *first*, before any
  horizon or policy resolution, so the rule is invariant under default / global
  `immediate` / per-table `immediate`-tag. The `unreferenced` + `immediate`-mode
  test exercises the strongest case.

### Static aspects checked

- **Type safety / API** — `change.hlc.wallTime` is `bigint` ms-since-epoch
  (`clock/hlc.ts`), consistent with `Date.now()` ms math after `Number(...)`;
  `BasisLifecycleStore.{get,put,delete,getAll,list}` all present and keyed
  lowercased via `buildBasisLifecycleKey` (so display-cased re-reads resolve);
  `TableSchema.{indexes,tags,derivation}` confirmed real fields (not test-only).
- **Validate-vs-parse agreement (gap #4)** — `validateEvictionPolicy` (engine) and
  `parseEvictPolicyTag` (sync) agree on accepted shapes: `never`/`immediate`
  case-insensitively and non-negative number / numeric-string / bigint. The engine
  can't depend on sync, so the deliberate duplication is justified; the divergence
  on empty-string (engine errors, sync falls back) is unreachable past a valid
  deploy.
- **Resource cleanup / error handling** — `reclaimDetachedTable` mirrors
  `destroy`'s map-clear-before-await ordering and idempotent provider/DDL no-ops;
  the sweep's drop-throw path retains the record for retry and emits only after
