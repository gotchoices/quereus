# Complete: one official way to resolve a collation name, per database

A *collation* is a named text-sorting rule (`BINARY`, `NOCASE`, `RTRIM`, plus whatever the
embedder registers with `db.registerCollation`). Collations live per `Database`, but several
comparison paths resolved names against an older process-global registry and silently fell back
to byte order on a miss. This ticket added the single resolution primitive that tickets 3.3–3.6
adopt, and closed the silent fallback in the plan-emission path.

## What shipped

- **`CollationResolver`** (`types/logical-type.ts`) — `(name: string) => CollationFunction`,
  exported from `src/index.ts`.
- **`Database.getCollationResolver()`** — a lazily-created bound closure with stable identity
  across calls, reading the live per-database registry. Resolves case-insensitively (and now
  tolerates surrounding whitespace); fast-paths the exact name `BINARY`; throws
  `QuereusError('no such collation sequence: X')` on a miss, matching SQLite's wording.
- **`getCollationResolver()` on `DatabaseInternal`** — so `quereus-store` and `quereus-isolation`
  reach it through their existing `this.db as DatabaseInternal` cast.
- **`builtinCollationResolver(name)`** (`util/comparison.ts`) — database-free `switch` over the
  three built-ins, returning `undefined` otherwise. Survives deletion of the deprecated global map
  in ticket `3.6-retire-global-collation-registry` and never observes a caller's override.
- **`EmissionContext.resolveCollation`** and **build-time `resolveCollation`**
  (`planner/building/schema-resolution.ts`) both delegate misses to the resolver, so the two report
  the same error from the same place. Plan-invalidation dependency recording is unchanged.

**Behavior change for embedders.** Opening a persisted database whose DDL names a custom collation
now throws if that collation was not re-registered on the connection, where it previously returned
byte-ordered rows. `test/collation-isolation.spec.ts` had an assertion pinning the old silent
fallback; the implement stage inverted it, and this review confirms that direction — a query that
silently returns a different ordering than the database that wrote it is the defect this ticket
family exists to close. Documented in `docs/sql.md` §4.4 under COLLATE.

## Review findings

**Read first:** the implement-stage diff (`git show 828d59d2`) before its handoff summary.

### Correctness — one real defect found and fixed inline

- **`registerCollation('BINARY', cmp)` was half-applied, and which half you got depended on the
  spelling in the SQL text.** The resolver (and `EmissionContext.resolveCollation`) fast-path the
  exact string `'BINARY'` to the built-in comparator, but `registerCollation` happily stored an
  override under the map key `BINARY`. So `order by x collate BINARY` used the built-in byte
  comparator while `order by x collate binary` used the embedder's override — two different sort
  orders for the same query, decided by letter case. The implement-stage doc comment asserted
  "`BINARY` alone cannot be overridden", which was only true for the canonical spelling. Fixed by
  making `registerCollation` reject a `BINARY` registration outright with `MisuseError` rather than
  silently half-honoring it. That turns the fast path's correctness precondition into an enforced
  invariant instead of a comment. Nothing in the repo registered `BINARY`, so no call site moved.
  Covered by a new test asserting both spellings are rejected and both still resolve to the
  built-in comparator.

- **Collation-name keys were uppercased but not trimmed**, while `normalizeCollationName` (the
  function whose entire job is canonicalizing these names) trims *and* uppercases. `registerCollation`,
  `_getCollation`, `_getCollationNormalizer`, `_isCollationReplicable`, and the new
  `builtinCollationResolver` each hand-rolled `name.toUpperCase()`, so `registerCollation(' X ', f)`
  stored an unreachable key. All five now call `normalizeCollationName`. New tests cover
  whitespace-tolerant resolution.

- **The handoff claimed build-time and emit-time resolution "report the same error from the same
  place." They did not.** `planner/building/schema-resolution.ts` has its own `resolveCollation`
  that threw `Collation not found: X` and never touched the resolver. (It also currently has zero
  call sites — the emitters all go through `EmissionContext`.) Rather than leave a second, divergent
  resolution primitive in a ticket whose whole point is having exactly one, it now delegates its
  lookup to `db.getCollationResolver()`, keeping its cache and dependency recording. The claim in
  the `EmissionContext` doc comment is now true.

### Gaps from the handoff — closed

- **`yarn test:store` was not run.** Run here: `6595 passing`, exit 0. The new throw does not reach
  the store-backed logic suite.
- **No emitter produces an unregistered name.** Re-checked the sites the plan ticket flagged
  (`effectiveCollationOfTypes` in `emit/join.ts`; the `attr.type.collationName` reads in
  `distinct.ts` / `set-operation.ts` / `async-gather.ts`) — each guards with
  `collationName ? resolve(...) : BINARY_COLLATION`. Still an absence of evidence rather than a
  proof, but both suites are green.

### Aspect sweep — nothing else found

Checked single-purpose (the resolver does one thing), DRY (the duplicate resolution primitive above
was the one violation, now removed), modularity (the `DatabaseInternal` seam is the right place —
no new import edge from the extension packages into `core/`), type safety (no `any`; the resolver
type is exported), resource cleanup (nothing allocated per call — the closure is bound once), and
error handling (`QuereusError` with `StatusCode.ERROR`, no swallowed exceptions). The deliberate
absence of `checkOpen()` on `getCollationResolver` is sound: it runs at comparator-construction
time, and reading a closed database's collation registry has no side effects.

### Tripwires (recorded, not filed as tickets)

- **Registration is not retroactive** — comparators resolve once at construction time, so
  re-registering a collation does not rebuild indexes or cached plans that captured the old
  comparator. Parked by the implementer as a `NOTE:` above `Database.registerCollation`; verified in
  place. Becomes work only if retroactive re-registration is ever wanted.
- **Plan-dependency keys are not normalized** — `EmissionContext.getCollation` records
  `collation:nocase` and `collation:NOCASE` as distinct keys for one collation. Harmless today
  (validation re-resolves case-insensitively) but the same collation can appear twice in a plan
  fingerprint. Parked as a `NOTE:` above `EmissionContext.getCollation`; only matters if fingerprint
  identity ever has to be exact.

### New tickets filed

None. The one defect found was a two-line fix with a direct test, and the rest were either handoff
gaps closed in this pass or genuinely conditional concerns parked as tripwires.

### Remaining scope (already ticketed, not a review finding)

Nothing adopts the resolver yet beyond `EmissionContext` and the build-time path. The memory vtab,
`quereus-store`, and `quereus-isolation` still resolve through the deprecated global registry —
tickets 3.3–3.5 — and `3.6-retire-global-collation-registry` deletes it. `compareSqlValues`,
`compareWithOrderBy`, `createOrderByComparator`, and `compareRows` still route through the global
map, untouched here by design.
