# Multi-hop relay chain (depth ≥ 3) — completed

## What shipped

A `describe('multi-hop relay chain (depth >= 3)')` block (4 specs) appended to
`store-and-forward-relay.spec.ts`, reusing the file's metadata-layer harness
(`makePeer`, `col`, `changeSet`, `pull`, `flatten`, `tablesIn`, `maxSetHLC`,
`RETIRED`) plus one block-local `buildChain()` helper that stands up a fresh
`S → R1 → R2 → R3 → H` topology per spec. No production code changed — pure test
addition over already-shipped relay behavior. `docs/sync.md` unchanged.

The four specs cover: three-hop delivery (origin HLC + siteId preserved, exactly
one entry per relay, holder materializes and serves onward); chain quiescence
(watermark stops re-send after one exchange at every depth); echo exclusion two
relays deep (from-zero and delta); and back-relay loop-freedom one hop deeper than
the existing ping-pong.

This was the second of two implement tickets split from the `deep-coverage` plan;
its sibling (`real-engine-e2e`) already landed. The plan scoped this ticket
explicitly as the **metadata-layer** chain test — reaches a holder several hops
away, exactly once, original `hlc` + `siteId` intact, chain quiesces — and all of
that is delivered (plus two extra adversarial specs).

  `packages/quereus` — the explicit `tsc` pass above is the equivalent
  signature-drift / unused-symbol check for this file.

### Correctness / adversarial scrutiny
- **Assertions are non-vacuous.** Confirmed each load-bearing assertion fails on a
  real regression: `tablesIn(...).include(RETIRED)` fails if a hop drops the relay
  (empty sets); `listForwardable().lengthOf(1)` fails if re-dispose doesn't run
  (0) or duplicates (>1); `siteIdEquals(cv.hlc.siteId, S)` fails if any relay
  re-stamps origin; `compareHLC(cv.hlc, C.hlc)===0` fails on HLC re-stamp.
- **Quiescence watermark logic checks out.** From-zero pull returns only C (relays
  have no local writes, `known` is `main.users` only), so each consumer's
  `max(ChangeSet.hlc)` watermark equals `C.hlc`; the re-pull's `> sinceHLC` filter
  correctly excludes C. No stray relay-origin changes can perturb the max.
- **No state-leak across the chain.** R1's forwardable count is only mutated by its
  own apply; being pulled *from* is read-only, so the post-chain "still exactly
  one entry" assertions on R1/R2/R3 are sound.
- **Holder disposition (`quarantine`) is irrelevant** because H has `orders` in its
  basis — consistent with the existing depth-2 holder; not a defect.

### Findings disposition
- **Major (new tickets):** none. The implementation fully meets the planned scope.
- **Minor (fixed inline):** none required. Code is DRY (block-local `buildChain`,

- **Docs:** `docs/sync.md` § Store-and-forward relay reviewed against the change.
  It states loop-freedom "across hops" with **no depth bound** and points at

### Accepted gaps (documented, not defects — none warrant a ticket)
These are beyond the planned scope, low-risk given the per-change inductive
identity argument, and explicitly *not* known defects:
- **Metadata layer only.** Like every spec in this file, materialization is
  asserted via `columnVersions`, not `select`. Real-engine materialization is the
  separate (already-landed) `store-and-forward-relay-e2e.spec.ts`; a real-engine
  *multi-hop* e2e was never scoped by the `deep-coverage` plan and is not filed —
  filing it now would be getting ahead of demonstrated need.
- **Fixed depth 3** (not a property-style sweep over arbitrary N). Depth 3
  exercises the >2 inductive transition; a regression isolated to depth ≥4 only
  would slip through, but the per-hop identity is uniform.
- **Single-change straggler** across the chain (the file's from-zero spec covers a
  2-change txn at depth 1). Transaction-grouping integrity across hops is not
  re-exercised at depth; per-change idempotence makes this low-risk.
- **Back-relay is one hop** (`R3→R2`), not a full reverse traversal to S; echo
  exclusion at S is covered separately.
