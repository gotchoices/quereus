## Review findings

### Process note (read first)
The review environment intermittently fabricated/garbled small tool outputs (some `Read`/`grep` returns, and — importantly — two ad-hoc probe `.spec` runs that *appeared* to print results but had actually failed to compile: `0 passing, 1 failing`). All conclusions below rest only on channels that proved reliable and self-consistent: full-commit `git show` diffs of the source and test files, and the real Mocha suite exit/summary. Probe-derived "results" were discarded. The throwaway probe spec was deleted; the working tree carries no review-introduced changes. Findings that would normally be fixed inline were instead routed to a follow-up ticket — making source edits under a known output-corrupting channel risks committing a corruption-masked regression, so surfacing + ticketing with full repro is the higher-integrity call here.

### Findings — correctness / soundness
- **Override body shape mishandling (medium, ticketed → `fix/lens-override-body-shape-validation`).** `compileOverrideBody` composes by replacing only the top SELECT's projection (`{ ...select, columns }`). Confirmed by code reading that `SelectStmt` carries `union`/`unionAll`/`compound` as fields on the top node (`ast.ts` ~183–185), so a `select A union all select B` override parses as `type:'select'` and slips past the parser's `body.type !== 'select'` guard. The merger then collects sources/coverage from the top leg only and carries the other leg verbatim → silently mis-mapped reads or an opaque query-time column-count error when legs don't line up positionally. Not happy-path; requires a set-operation override body. Bundled with four lower-severity validation gaps (unaliased computed column silently dropped; `hiding(...)` names unvalidated; override FROM sources not checked against the declared `over Y` basis; `values` body error-message/regression) into the one fix ticket since they share the same compile-time validation surface.

### Findings — design / documentation (accepted for v1)
- **Lens block not wired into `explain schema` drift hash (minor, accepted).** `computeSchemaHash` was widened to hash a `DeclareLensStmt` and a unit test confirms two different lens blocks hash differently — but the only production caller, `emitExplainSchema`, hashes the logical `DeclareSchemaStmt` and never folds in the separately-stored lens block. So `explain schema X` returns the same hash regardless of override changes. Acceptable while the lens layer is read-only and there is no `explain lens` surface; should be wired when lens versioning/drift becomes load-bearing (natural home: the prover/versioning work). The function-level capability and its test are real and correct — the handoff's "participates in hashing" is accurate about the function, not about `explain schema`.
- **Write-enforcement not wired (by design).** Slot `attachedConstraints` are stored verbatim, not routed to enforcement. MV/lens is read-correct, write-unsound for logical-constraint enforcement. Owned by `lens-prover-and-constraint-attachment`. Documented in `docs/lens.md`.
- **Lens-declaration lifecycle.** The stored block lingers in `DeclaredSchemaManager` until `removeDeclaredSchema`; re-declare replaces; the override-references-declared-table check is skipped on an empty re-apply to preserve detach-all semantics. Reviewed; consistent with the foundation's asymmetric-removal model. Accepted.

### Findings — tests

- **Not line-audited:** `test/logic/52-lens-overrides.sqllogic` — it executes as part of the green suite (the lens read-path ran), but the corrupting tool channel prevented a reliable line-by-line read. No reason to suspect issues; flagged for transparency.
