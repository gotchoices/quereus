# View-mutation substrate core — completed

The AST rewrite is retired. The substrate is the single propagation path for all view/MV
mutations; the single-source case is its trivial one-base-op path. INSERT/UPDATE/DELETE against a
view or materialized view dispatch (in `building/{insert,update,delete}.ts`, via the unchanged
`getView ?? getMaterializedView` lookup) to `buildViewMutation(ctx, view, { op, stmt })`, which
calls `propagate` → re-plans each `BaseOp.statement` through the base-table builder → wraps in a
`ViewMutationNode`. The emitter drives each wrapped base-table DML subtree (a `Sink`-topped
subtree) as a param child, in list order, and yields nothing (RETURNING-through-view stays
rejected). See the implement-stage handoff (commit `e9a13f17`) and `docs/view-updateability.md`
for the architecture. This file records the **review pass**.

### Correctness — investigated, no defects
- **Optimizer descends into the wrapped DML subtree (the biggest risk — RESOLVED OK).** The
  optimizer traverses children via `getChildren()` (`framework/pass.ts` top-down/bottom-up both
  read `node.getChildren()`; `optimizer.ts optimizeChildren` likewise), and
  `ViewMutationNode.getChildren()` returns `baseOps`. So the byte-identical base-table DML subtree
  is optimized and `withChildren`-rebuilt exactly as the retired rewrite's output was (it sat
  under a `SinkNode`, whose children were likewise walked). Not a bug.
- **`getRelations()` returning `[]` is correct, and the node's comment is accurate.** Confirmed
  `isRelationalNode(n) ⟺ n.getType().typeClass === 'relation'` (`plan-node.ts`), and
  `SinkNode.getType()` is `typeClass: 'scalar'` — so the one Sink-topped base op is *not*
  relational and `baseOps.filter(isRelationalNode)` is `[]`. `ViewMutationNode` itself is also
  scalar-typed, so relational-only walks (e.g. `reference-graph.visitAllChildren`, which consults
  `getRelations()` only when `isRelationalNode(node)`) still reach the subtree through the
  unconditional `getChildren()` pass. No code change made here (an earlier candidate comment edit
  was based on a misreading of `isRelationalNode` and was discarded — the original comment is
  right).
- **Affected-row count / return-value parity.** `emitViewMutation.run` returns `null`. This is
  correct parity: `emitSink.run` also returns `null` — the row count is a side effect of draining
  the DML, not a return value. The wrapped `Sink` child is evaluated as a param, so it still
  drains its `DmlExecutor` and fires the writes; `ViewMutation` only sequences. `changes`
  reporting is unaffected (driven by the unchanged `DmlExecutor`).
- **`block.ts` result-shadowing exclusion.** `findLastIndex(stmt ≠ Sink && stmt ≠ ViewMutation)`
  is correct: both are void side-effect statements, so a view mutation in a multi-statement batch
  no longer shadows a prior statement's value — matching base-table DML. Must be revisited when
  RETURNING-through-view lands (3.2) so a RETURNING view mutation surfaces its rows.
- **Emitter registration.** `emitViewMutation` is registered for `PlanNodeType.ViewMutation`;
  confirmed transitively — end-to-end view-write sqllogic executes (would throw "no emitter"
  otherwise) and is green.
- **Dispatch sites.** `insert`/`update`/`delete` builders each route the
  `getView ?? getMaterializedView` hit through `buildViewMutation` identically; MV write-through
  rides the same path (53.1 green).
- **Type-safety of `buildBaseOp` casts** (`op.statement as AST.InsertStmt` …) are sound — the
  `op.op` discriminant guarantees the matching union member. A discriminated `BaseOp` union would
  drop the casts; left as-is (acceptable; `BaseOp` is a 3.2 extension surface).
- **Import cycle `propagate.ts ↔ single-source.ts`** — call-time only (no module-eval-time use on
  either side), resolves cleanly, build green. Acceptable.

### Minor — examined, deliberately left in place (documented for 3.2)
- **`BaseOp.table` is currently dead, and resolving it does redundant work.** `propagate` calls
  `resolveBaseTable` → `buildTableReference` (which builds a full `TableReferenceNode` and
  registers a schema dependency) purely to populate `BaseOp.table`, but **no consumer reads
  `.table`** (verified: the builder uses only `op.op`/`op.statement`; the node and emitter never
  touch it). The base-table builder re-resolves the same table when re-planning. Net cost: one
  extra `buildTableReference` per view mutation at **plan time only** (plans are cached); the
  schema-dependency registration is idempotent (keyed by schema+table+kind), so there is no
  double-tracking of consequence. Left in place because `BaseOp.table` is the natural extension
  point for **3.2's FK-ordered multi-source fan-out** (which must order ops by table); removing it
  now would only force 3.2 to re-add it. **3.2 should either consume `.table` or make its
  resolution lazy** so the work isn't wasted in the single-source path.
