## What landed

Extends `ruleMaterializedViewRewrite` with a **join-subsumption arm** that recognizes when a
`Project(Filter?(Join(T, P)))` query is covered by a 1:1 row-preserving **inner/cross** join MV
body and rewrites it to scan the MV's backing table ‚Äî eliminating the join at read time.

```sql
create materialized view enriched as
  select o.id, o.customer_id, o.amt, c.name
  from orders o join customers c on o.customer_id = c.id;   -- 1:1 NOT-NULL FK ‚Üí PK

select o.id, o.amt, c.name
from orders o join customers c on o.customer_id = c.id
where o.amt > 100;                                          -- ‚Üí scan _mv_enriched, residual filter + project
```

Soundness pivots on the coverage prover's shared `proveOneToOneJoin` (no-row-loss descent +
`proveJoinNoFanout`) run on **both** the fragment join and the MV body join, plus same-(T, lookup,
type) and equi-pair equivalence, then projection/residual subsumption over the joined relation. The
residual re-binds onto the backing by stable **source attribute id**. See `docs/optimizer.md`
¬ß *Materialized-view query rewrite (read side)* and `docs/materialized-views.md` ¬ß *Join subsumption*
for the full contract.

### MAJOR ‚Äî found and fixed inline (silent wrong results / planning crash)

**Stale MV-body-root cache + positional backing misalignment for a `select *` join MV body across a
source `alter` + `refresh`.** Two compounding defects in the same area the implementer flagged as
risk #1/#2:

1. **Lazy cache invalidation gap.** `MV_BODY_ROOT_CACHE` was only invalidated when `plannedMvBodyRoot`
   ran while `mv.stale === true`. A `refresh` clears `stale` *without* firing the rule, so the sequence
   *fresh query (caches body root) ‚Üí `alter` source (stale, no query) ‚Üí `refresh` (stale cleared) ‚Üí
   query* served the **pre-ALTER** body root. Reproduced: returned customer `id` values instead of
   `name`. Even the *tested* stale-window path re-derived a body that then hit defect 2.
2. **Positional backing misalignment.** The arm keys backing columns by the MV body's output
   **position**. For a `select *` join body, a source column add makes the re-planned body
   **interleave** the new column (e.g. `orders.extra` at output pos 2) while the refreshed backing does
   not reorder to match ‚Äî so the position map read the wrong backing column, or ran off its end
   (`colRefOnto` ‚Üí `backingAttrs[oob]` undefined ‚Üí planning crash during `getPlan`). Reproduced both a
   crash and wrong rows.

   Explicit-column join MV bodies (the only shape the implementer's tests exercised) are **immune** ‚Äî
   their output column set is stable across a source column add/drop ‚Äî which is why the gap shipped
   green.

**Fix (sound, conservative ‚Äî declining is always safe):**
- `query-rewrite-matcher.ts` ‚Äî new `backingAlignsWithBody` gate in `matchJoinFragmentToMv`: verifies
  the backing columns still correspond positionally (count + name) to the MV body root's output before
  trusting the position-keyed map; a mismatch forgoes the rewrite (`no-candidate`) ‚Üí correct base
  recompute. Catches both the length-mismatch (crash) and reorder (wrong-row) cases.
- `rule-materialized-view-rewrite.ts` ‚Äî `plannedMvBodyRoot` now also validates cache freshness via
  `cachedBodyRootIsCurrent` (a source `alter` swaps the `TableSchema` object identity, so a cached root
  referencing a superseded source object is re-derived), and deletes the entry when a body fails to
  re-plan. Defense-in-depth + re-derives against current schema rather than serving a superseded plan.
- Regression test added (`query-rewrite-join.spec.ts`): a `select *` join MV across `alter`+`refresh`
  **with no intervening stale-window query** ‚Äî asserts the rewrite forgoes (no crash) and rows stay
  exactly correct. This also fills the previously-absent `select *` join-body coverage (risk #2),
  exercising the positive (fresh) rewrite too.
- `docs/optimizer.md` ‚Äî documented the backing-alignment invariant and the desync forgo.
- Full suite green afterward: `yarn test` **4673 passing** (+1 mine), 0 failing; `typecheck` clean;
  `eslint` clean on all changed files.

The remaining *optimization* gap (a `select *` join MV no longer rewrites after a desyncing
`alter`+`refresh` until the alignment can be re-established) is filed as backlog
`mv-join-rewrite-schema-evolution` ‚Äî the behavior is now always **correct**, just unoptimized for that
narrow case.

### Reviewed and accepted (no change)

- **Join/equi-pair soundness.** `findJoinAssignment` + `equiPairsByBaseCol` correctly reject a
  different FK to the same lookup (`shape`), a fanning non-1:1 join, and an outer join. The
  bidirectional driver trial is safe: a 1:1 FK join fixes the driver, and either successful assignment
  yields the same backing mapping. Confirmed by the per-reason negative tests.
- **Cross-join-with-WHERE-equi not rewritten** (implementer risk #5) ‚Äî the equi lands in the `Filter`,
  not the `JoinNode` ON, so `pureJoinEquiAttrPairs` returns `[]` ‚Üí declines. Sound NotMatch, not a bug.
- **Cost gate** declines on tie (`cost >= baseCost`), uses the cheaper `hashJoinCost` for the recompute
  floor ‚Äî conservative; the tiny-table boundary (risk #6) can only *forgo*, never mis-rewrite.
- **Residual determinism not gated** (risk #4) ‚Äî matches the foundation arm; sound because the residual
  is evaluated per joined row identically either way.
- **`predicate-not-entailed` reused for a subquery conjunct** (risk #3) ‚Äî defensible reason reuse; left
  as-is (a dedicated code is cosmetic).
- **Nested `db.getPlan` re-entrancy** (risk #1) ‚Äî the suppression flag makes the nested optimize bail
  (`rule-materialized-view-rewrite.ts:93`), `OptContext` is per-call; no shared-pass-state corruption.
  The cache freshness fix above strengthens this path.
- **Equivalence harness** (FK‚ÜíPK boundary, NULL lookup cols, empty-lookup, driving+lookup residuals)
  and the golden-plan / per-reason / self-maintenance tests all pass and are non-vacuous.

### Not found

- No SPP/DRY/type-safety/resource-cleanup issues in the diff. The `pureJoinEquiAttrPairs` export is a
  clean one-line reuse. No `any`, no eaten exceptions (the `plannedMvBodyRoot` catch is intentional and
  now nulls the cache). Composite-FK / composite-PK 1:1 joins (risk #2) are handled structurally by
  `proveOneToOneJoin` (exercised by `covering-structure.spec.ts`); left to the backlog enhancement to
  add a join-rewrite-specific test rather than blocking completion.
