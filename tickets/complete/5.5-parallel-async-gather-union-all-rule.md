## What landed

- **New rule** `packages/quereus/src/planner/rules/parallel/rule-async-gather-union-all.ts` (~130 lines). Walks an arbitrary tree of `SetOperationNode(op='unionAll')` (left-deep, right-deep, balanced) plus any unionAll-`AsyncGatherNode` children already produced by an inner firing (the load-bearing absorption that keeps the "single gather per chain" invariant under bottom-up traversal), flattens leaves, then folds into one `AsyncGatherNode({ kind: 'unionAll' })`. Two gates: (1) `every(child.physical.concurrencySafe === true)`; (2) `max(child.physical.expectedLatencyMs ?? 0) >= tuning.parallel.gatherThresholdMs`. `preserveAttributeIds = node.getAttributes()` pins the outermost SetOp's attributes (which equal the leftmost-recursive leaf's attributes — same as `AsyncGather`'s `unionAll` default would produce).
- **Tuning knob** `tuning.parallel.gatherThresholdMs` (default 25 ms) added to `optimizer-tuning.ts`. Calibrated to the synthetic `HighLatencyMemoryModule` (`expectedLatencyMs = 25`) shared with the fan-out rule's spec.
- **Registration** at `PassId.PostOptimization` priority 17 — after `asof-strategy-select` (11, so leaves' physical properties are finalized) and before `materialization-advisory` (30, so any cache the advisory injects lands inside gather branches).
- **Spec** `test/optimizer/parallel-async-gather.spec.ts` — 16 specs covering 3-way / 2-way folds, single-SELECT no-op, mixed-branches (locks the "max-of-children" v1 decision), all-local no-fold (the load-bearing local-only invariant), raised-threshold no-fold, minBranches override, `UNION` (distinct) rejection, `disabledRules` honored, execution-equivalence (multiset), attribute-ID preservation via column-name check, Sort-above-gather (skipped under strict-fork — pre-existing), recursive CTE untouched, idempotence, default tuning has `gatherThresholdMs > 0`, concurrencyCap clamped to N.
- **Docs.** `docs/optimizer.md` § "Async gather UNION ALL" added (gate semantics, threshold default, pass placement, attribute-ID story, idempotence, tuning knobs, out-of-scope items). `docs/runtime.md` updated with a "Recognition rule (unionAll)" paragraph linking to the optimizer doc.

### What was checked

- **Rule code path** (`rule-async-gather-union-all.ts`): both gates match the doc; the `collectUnionAllChildren` helper correctly bounds itself to `op === 'unionAll'` for SetOps and `combinator.kind === 'unionAll'` for AsyncGather absorption (it does NOT absorb `crossProduct` gathers — confirmed at line 122). ✓
- **Pass placement** (`optimizer.ts:554-560`): registered at PostOptimization priority 17, after asof-strategy-select (11) and before materialization-advisory (30). The justification comment is accurate. PostOptimization is BottomUp (`framework/pass.ts:151`), so inner subtrees rewrite first — this makes the AsyncGather-child absorption load-bearing, confirmed via the idempotence test and the rule's own comment at lines 99-112. ✓
- **Attribute-ID preservation**: `node.getAttributes()` on the outer `SetOperationNode` returns `left.getAttributes()` via `buildAttributes`, which for left-deep chains is `A`'s attributes (the leftmost-recursive leaf). For balanced / right-deep shapes the rule's flattened `children[0]` is also the leftmost-recursive leaf, so `AsyncGather.buildAttributes` (which uses `preserveAttributeIds` if set, else `children[0].getAttributes()`) lands on the same answer. The `preserveAttributeIds` argument is therefore redundant with the `unionAll` default, but harmless and explicit — kept as-is. ✓
- **Re-entry safety**: after rewrite the root is `AsyncGatherNode`, so the matcher (`node instanceof SetOperationNode`) rejects on a re-pass. The `applyRules` registry also caches via `markRuleApplied` keyed on node id. ✓
- **Constructor validation**: `AsyncGatherNode.validateConstruction` enforces `children.length >= 2` and per-column-count equality across branches. The rule only fires when flattened `children.length >= tuning.minBranches` (default 2), and SetOperationNode's constructor has already validated column counts pairwise — so by transitivity all flattened leaves share column count. ✓
- **physical-properties parity** between `SetOperationNode(unionAll).computePhysical` (drops `monotonicOn`, `fds`, `equivClasses`, `constantBindings`, `domainConstraints`) and `AsyncGatherNode(unionAll).computePhysical` (drops the same set plus `ordering`): identical from the downstream consumer's perspective. ✓
- **Recursive CTE**: uses `RecursiveCTENode`, not chained `SetOperationNode`s — the rule's matcher rejects automatically. Locked by an explicit test. ✓
- **Disabled-rules integration**: rule id `'async-gather-union-all'` is checked against `context.tuning.disabledRules` in `registry.ts:198`. Spec exercises this path. ✓
- **concurrencyCap clamp**: `Math.max(1, Math.min(tuning.concurrency, children.length))` — clamps to N when N < concurrency, floors at 1 when concurrency=0 (though `tuning.concurrency` defaults to 8 and the AsyncGather constructor would reject a non-positive cap). ✓

### Findings — open / documented, deferred to follow-up

2. **Sort-above-AsyncGather under `QUEREUS_FORK_STRICT=1` violates the strict-fork contract.** The Sort emitter (`runtime/emit/sort.ts:35`) calls `withAsyncRowContext(ctx, …)` on the parent rctx while the gather's iterator still holds active forks. This is pre-existing — any manually constructed Sort-on-AsyncGather plan trips the same path. The spec uses `it.skip` under strict-fork; the non-strict path validates correctness end-to-end. Implementer flagged this; worth a follow-up (`fix-sort-above-async-gather-strict-fork`) to fix the Sort emitter's lifecycle, but not blocking for this rule.

3. **`materialization-advisory` interaction is unpatched.** The advisory runs at priority 30 (after the gather at 17) and inspects each gather child individually. If a child were both advisory-eligible (the recognized types are `Block`, `ScalarSubquery`, `Exists`, `In`, `Insert`, `Update`, `Delete`, `CTE`, `RecursiveCTE`, `Returning`, `ScalarFunctionCall`, `CaseExpr` — see `optimizer.ts:580-593`) *and* high-latency (the only way the gather rule fires), the advisory could wrap it in a `CacheNode` *inside* the gather branch, which materializes serially before the gather sees a row and defeats the prefetch overlap. In the current tree, none of those node types are common leaves of a unionAll chain (a typical branch is a `Project`/`Filter`/`SeqScan` chain over a table), so the interaction is theoretical. No test exercises it because no fixture combines a high-latency module with an advisory-eligible leaf. Implementer flagged; deferred to a follow-up if the case surfaces with a remote-vtab plugin.

4. **Mixed-branches "max-of-children" gate fires the rule.** Per the v1 design, a 3-way chain with one slow branch and two local branches *will* fold. Whether that's a wall-clock win depends on per-branch fork overhead vs. the parallelism payoff. The spec at `parallel-async-gather.spec.ts:115-126` pins this decision; adaptive per-branch evaluation is parked. Implementer flagged.

5. **Cost gate is inert in production until a remote-vtab plugin declares non-zero `expectedLatencyMs`.** Same posture as the fan-out rule (cf. complete/4.5-parallel-fanout-lookup-join-rule.md, finding 2). The synthetic `HighLatencyMemoryModule` in the spec is the only consumer. The next remote-vtab plugin landing must set `VirtualTableModule.expectedLatencyMs > 0` on its module class to exercise either rule end-to-end.

6. **`crossProduct` and `zipByKey` recognition.** Opt-in only per parent ticket scope; `crossProduct` has no recognition rule, `zipByKey` is parked in backlog as `parallel-async-gather-zip-by-key`. Not in scope here.

7. **The `if (tuning.minBranches < 2) return null;` guard is mildly redundant** with the subsequent `if (children.length < tuning.minBranches) return null;` (the AsyncGather constructor's own `children.length < 2` precondition would catch a misconfigured `minBranches=1` setting at construction time). Kept as a fast-path defensive guard; not load-bearing. No action.

8. **Attribute-ID preservation test asserts on column names (`Object.keys(base[0])` vs `Object.keys(rewr[0])`)**, not on raw attribute IDs read off the plan. Same shallowness as the analogous fan-out rule test (cf. complete/4.5, finding 1's resolution). Acceptable for v1: column-name equality is the user-visible manifestation of attribute-ID stability, and the rewrite explicitly passes `preserveAttributeIds` through the `AsyncGatherNode` constructor — so the rule is structurally incapable of dropping or renumbering IDs without a code change. No follow-up.
