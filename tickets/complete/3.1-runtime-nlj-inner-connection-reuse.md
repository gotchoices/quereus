## What shipped

A nested-loop join whose inner (right) side is **not** wrapped in a cache node
re-scans the inner relation once per outer row. Previously each re-scan did
`module.connect(...)` at scan-generator start and `disconnect(...)` in its
`finally` — one connect + disconnect per outer row. Now the scan leaf connects
the inner virtual-table instance **once per scan-site per execution**, reuses it
across every re-scan, and disconnects once at statement teardown.

### Mechanism (as implemented)

- **`runtime/types.ts`** — new optional `RuntimeContext.scanConnections?:
  Map<symbol, VirtualTable>`, a per-execution cache of connected inner-scan
  instances.
- **`runtime/emit/scan.ts`** — each `emitSeqScan` mints a stable per-node symbol
  (`scanConnectionKey`) in its emitter closure. On a cache hit → reuse and skip
  connect; on a miss → connect once and store. Per-invocation disconnect removed
  from the scan `finally` (row slot still closes each pass). No cache present
  (transient/analysis contexts) → fall back to owning the lifecycle
  (connect+disconnect per invocation). Keyed by scan-node identity, so a
  self-join's two scan sites get distinct instances.
- **`core/statement.ts`** — `_iterateRowsRawInternal` creates the map, passes it
  on the `RuntimeContext`, and in its `finally` disconnects every cached instance
  exactly once on all exit paths (completion, `break`, error, abort).
- **`runtime/parallel-driver.ts`** + **`fork-contract.spec.ts`** — `fork()`
  propagates `scanConnections` by reference (policy `shared-cooperative`); the
  fork-contract test pins the policy (typecheck-enforced).

Only the `Statement` iteration path (`db.eval` / `db.get` / `db.all` / `stmt.*`,
including the trace path) gets reuse; every other `RuntimeContext` construction
site falls back to per-invocation connect+disconnect (correct, no reuse).

## Review findings

Adversarial pass over commit `179c1f93`. Diff read first with fresh eyes, then
the handoff. Lint + full test suite run green. Result: **no defects found, no
inline fixes, no new tickets.** Details of what was checked:

### Correctness — connect/disconnect lifecycle (CHECKED, clean)
- Cache miss connects+stores (ownership deferred to statement teardown); cache
  hit reuses without connecting; teardown disconnects each cached instance
  exactly once. Verified against every exit path.
- **No leak on error/abort:** a cached instance whose `query()` throws is *not*
  disconnected in the scan `finally` (`ownsInstance` is false) and is released
  by the statement `finally` instead. The fallback (no-cache) path sets
  `ownsInstance = true` and disconnects itself.
- `disconnectVTable` swallows a per-instance disconnect error (catch+log), so one
  failing disconnect in the teardown loop does not strand the rest.
- **Double-disconnect is safe.** In a memory self-join the inner scan adopts the
  outer's already-registered connection, so two distinct `MemoryTable` instances
  reference one `MemoryTableConnection`; teardown calls `disconnect()` on both,
  but `MemoryTableManager.disconnect` is idempotent (early-returns once the
  connection id is gone from its map). The counting-module test (distinct
  instances, no sharing) pins `disconnectCount === connectCount`.
- **Latent pre-existing leak actually fixed by this change.** The query-method
  existence check moved *inside* the try/finally. Before, a query-less module on
  the fallback path connected an instance and then threw *before* the try,
  leaking it. Now it disconnects. (Defensive only — a table either has `query`
  or never worked; but the fix is correct.)

### Visibility / read-your-writes (CHECKED, reuse is neutral)
Deep-verified for the memory module (the real scan lifecycle) by reading
`vtab/memory/table.ts` (`ensureConnection`, `query`) and `module.ts`
(`concurrencyMode = 'reentrant-reads'`, `scanSnapshotIsolation = true`):
- `query()` reads `conn.pendingTransactionLayer ?? conn.readLayer` **fresh on
  every call** off the held connection *reference* (not a snapshot captured at
  connect). A reused scan instance therefore observes the writer's live pending
  layer on each re-scan — identical to the old per-row-reconnect behavior.
- The DML writer's registered connection persists for the whole statement (its
  own `getVTable`/`disconnect` in the DML executor `finally`), so the connection
  the scan adopts never swaps mid-statement. Reuse changes only the *connect
  count*, not which layer is read.
- The correlated-read-within-one-DML test passes; the count(id<current) case is
  visibility-insensitive by construction, but the layer analysis above covers the
  visibility-sensitive case too.
- **Non-memory modules:** the LevelDB store path flows through the same generic
  scan connect/disconnect lifecycle; the implementer ran `yarn test:store`
  (6891 passing) and the semantics argument (direct `module.connect`, no
  `adoptConnection` registration change) holds. Not re-run in this review pass —
  no source change since the implement commit that ran it.

### Cursor isolation (CHECKED, clean)
- Self-join's two scan sites get distinct symbol keys → distinct instances → no
  shared cursor. Where the inner *adopts* the outer's connection (memory), the
  two concurrent read iterators over one connection are `reentrant-reads`-safe
  (each captures its own immutable start-layer). Both are pre-existing
  properties; the memory self-join row-correctness test pins them.

### Join-shape coverage (CHECKED — deliberately not expanded, with reason)
The handoff suggested adding a RIGHT/FULL self-join spec case. **Not added on
purpose:** the reuse logic lives entirely in `emitSeqScan` and is keyed by
scan-node identity, which is join-type-agnostic — RIGHT/FULL (`driveFromRight`)
still re-invokes the inner scan callback sequentially, exercising the exact same
`scan.ts` path the existing cross-join test already pins. A RIGHT/FULL
permutation would add a new join driver but not a new scan-reuse code path, so it
is low marginal value. Recorded here rather than filed or silently skipped.

### Fallback paths (CHECKED, clean)
`_executeSingleStatement`, const-evaluator, deferred-constraint-queue,
materialized-view maintenance, assertions, etc. do not set `scanConnections` →
the scan owns connect+disconnect per invocation (correct, no reuse). No leak on
any of these; verified by the fallback branch in `scan.ts` and the full green
suite.

### Types / lint / docs (CHECKED, clean)
- No `any`; `VirtualTable` imported for the map value type.
- `yarn workspace @quereus/quereus run lint` — 0 errors (eslint + `tsc -p
  tsconfig.test.json --noEmit`, which enforces the `satisfies Record<keyof
  RuntimeContext, ForkPolicy>` fork-contract clause).
- `docs/runtime.md` gained an "Inner-scan connection reuse" section and two
  fork-policy table rows (`executionMemo`, `scanConnections`); both accurate to

### Tests run
- Targeted: `nlj-inner-connection-reuse.spec.ts` + `fork-contract.spec.ts` —
  12 passing (8 pending are strict-mode gated, unrelated).
- Full: `yarn workspace @quereus/quereus run test` — **6896 passing, 0 failing**,

- The new spec is a solid floor: connect-once discriminator (2 not 4),
  re-execution (cache resets per run), abort mid-inner-scan
  (disconnect==connect, no double-disconnect), and memory row-correctness for
  self-join / correlated IndexSeek / correlated DML.

### Tripwires (parked, not tickets)
- **Concurrent shared `scanConnections` under a future parallel scan.** `fork()`
  shares the map by reference; two forks scanning the *same* scan node would
  issue concurrent `query()` on one shared instance — only safe for a module
  whose `concurrencyMode` permits it. Dormant today (ParallelDriver has no query
  consumers; the sequential NLJ re-scan is never concurrently self-live).
  Recorded as a `NOTE` in `runtime/parallel-driver.ts` `fork()` and the
  fork-policy doc/comment — verified present. Not a ticket.

that snapshots at `connect()`) are correct-but-no-reuse or hypothetical
