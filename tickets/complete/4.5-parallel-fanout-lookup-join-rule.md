## What landed

The implement-stage handoff (review ticket) accurately describes the surface on disk. Headlines:

- **New `PhysicalProperties` fields** (`nodes/plan-node.ts`):
  - `expectedLatencyMs?: number` ŌĆö first-row latency, default 0; propagated as `max(children)` by the base `PlanNode.physical` getter.
  - `concurrencySafe?: boolean` ŌĆö siblings-on-shared-connection safety; propagated as `every(c.concurrencySafe !== false)` (defaults true on absence).
- **Leaf source** in `TableReferenceNode.computePhysical`: `concurrencySafe = getModuleConcurrencyMode(vtabModule) !== 'serial'`; `expectedLatencyMs = vtabModule.expectedLatencyMs` when > 0.
- **`VirtualTableModule.expectedLatencyMs`** ŌĆö optional per-module hint; in-tree modules omit (treated as 0).
- **Tuning** (`optimizer-tuning.ts`): `tuning.parallel = { minBranches: 2, branchSetupCost: 1.0, concurrency: 8 }`.
- **Rule** `rule-fanout-lookup-join.ts` registered in `PassId.Structural` at priority 23, one ahead of `join-elimination` (24). Walks the Project's pass-through wrappers (Filter/Sort/LimitOffset/Distinct/Alias) down to a `JoinNode`, descends the `.left` spine collecting joins, and for each candidate `.right` reuses the join-elimination primitives (`isAndOfColumnEqualities`, `checkFkPkAlignment`, `lookupCoveringFK`, `isRowPreservingPathToTable`, `extractTableSchema`). FK indices are translated from "left-of-this-join attribute index" to "outer-attribute index" by looking up each left attribute's id in an outer-attr map ŌĆö this is the safety net that keeps the per-join alignment honest when the join's `.left` is a wide combined relation. INNER branches additionally require non-null FK + row-preserving path.
- **Cost gate** fires only when `(N ŌłÆ cap) ├Ś max(expectedLatencyMs) > N ├Ś branchSetupCost`, with `cap = min(tuning.concurrency, N)`. Local-only chains (`expectedLatencyMs = 0`) leave the gate inert; concurrency-bound chains (N > cap) cross it.
- **Output construction.** Each branch's `child` is the lookup wrapped in `FilterNode(node.scope, lookup, joinCondition)` ŌĆö outer-side `ColumnReferenceNode`s in the condition resolve through `rctx.context` (the outer's `RowSlot` is set on the parent ctx before each fork). `preserveAttributeIds` pins the outer-then-branches wide layout verbatim so the surrounding `ProjectNode` rebuild retains all attribute IDs. The chain wrappers above the topmost join are rebuilt bottom-up around the new FanOut.
- **Docs.** `docs/optimizer.md` gained a "Fan-out lookup join (FKŌåÆPK)" section (cost-gate semantics, tuning knobs, relationship to `join-elimination`, v1 limitations). `docs/runtime.md` updated to reflect the now-defined physical-properties fields, replaced the stale "rule will eventually source `concurrencyCap`" sentence, and added a Recognition+cost gate paragraph.

### Findings ŌĆö minor (fixed inline)

1. **`preserves output attribute IDs across the rewrite` spec used an IIFE that returned an unawaited Promise inside `try/finally`, restoring tuning before planning ran.** `db.eval(sql)` returns an async generator iterator synchronously; the planning + optimization actually happen on the first `.next()` call. Inside the IIFE the `finally` block ran `updateTuning(before)` *synchronously* between `results(...)` being called and the iterator being consumed via `Promise.all([...])` ŌĆö so the "baseline" query was planned with the rule **re-enabled** rather than disabled, defeating the test's intent. The test passed spuriously because both runs had the rule enabled and produced identical output.

   Rewrote the test to `await` each `results(...)` inside the tuning window, restoring `before` only after the await resolves. Added a `planRows`-based sanity-check assertion to confirm the second run's plan actually contains `FANOUTLOOKUPJOIN`, so the test would now catch regressions where attribute IDs diverge across the rewrite. All 10 tests still pass. (`test/optimizer/parallel-fanout.spec.ts:259-285`.)

### Findings ŌĆö open / documented, deferred to follow-up

2. **No production-shipping module declares non-zero `expectedLatencyMs`.** The cost gate is inert in tree by design; the synthetic `HighLatencyMemoryModule` in the spec is the only consumer. The next remote-vtab plugin will need to set `VirtualTableModule.expectedLatencyMs` on its module class to exercise the rule end-to-end. Already called out by the implementer; no action this pass.

3. **Cost-gate formula edge case at `cap=1`.** `Math.max(1, Math.min(tuning.concurrency, N))` floors `cap` at 1, so with user-supplied `tuning.parallel.concurrency = 1` the formula yields `(N-1) ├Ś latency > N ├Ś branchSetupCost`, which can be true even though a 1-concurrent fan-out has the same wall-clock as the original nested-loop chain (one wave of cap=1 still costs N├Ślatency, not 1├Ślatency). Not exposed by the default tuning (cap=8) or the spec (cap=2 ŌåÆ still parallel). Acceptable as v1 behavior; flag for the next pass-through if anyone wires up a tuning UI. The implementer's gap #2 already covers the broader "formula is a simplification" caveat.

4. **The rule abandons the whole cluster if any candidate branch fails recognition.** Documented in the implementer's gap #4 ("the rule does not currently 'skip the bad branch and cluster the rest'"). Mirrors join-elimination's per-rewrite conservatism; revisit if real-world chains demand partial clustering.

5. **The chain walk only sees joins that are stacked left-deep with no intermediate wrappers between them.** If a `FilterNode`/`SortNode` sits between two `JoinNode`s on the spine, the walker stops at the first non-`JoinNode` and the joins above the wrapper are recognized in isolation (often falling below `minBranches` and bailing). Mirrors typical parser output (left-deep join chains with all filters above the chain) and matches `join-elimination`'s shape assumptions. Not a regression.

6. **Aggregate-rooted recognition (analogous to `ruleJoinEliminationUnderAggregate`) is deferred.** Per the original plan; project-rooted recognition covers the motivating example.

7. **Connection-per-branch acquisition not implemented.** v1 reuses the outer's connection; `'serial'`-mode branches serialize via the lock and get no concurrency win. Correctness preserved; payoff is module-mode-gated.

8. **`FanOutBranchSpec` still carries no per-branch equi-pair surface**, so `FanOutLookupJoinNode.computePhysical` propagates FDs with empty equi-pair lists. Tightening is a node-side follow-up.

9. **At-most-one recognition should eventually cover aggregate-shaped branches.** A correlated scalar aggregate subquery (e.g. `(select json_group_array(...) from child where fk = outer.key)`) yields exactly one row per outer row, so it is an at-most-one branch the v1 emitter already handles ŌĆö only this rule's recognition pass doesn't route it through the fan-out node yet. Extending recognition to cover that shape subsumes the proposed `array` branch mode entirely (no new `FanOutBranchMode`, no JSON shaping in the node ŌĆö the query expresses the shape). Tracked in `tickets/backlog/parallel-fanout-lookup-join-aggregate-branch-recognition.md`; the relational 1:n product case is `tickets/backlog/parallel-fanout-lookup-join-cross-mode.md`.

### Findings ŌĆö empty categories (explicit)

- **Security / correctness regressions:** none observed. The rule's failure modes are all conservative (`return null` ŌåÆ no rewrite); the FanOut constructor's runtime validations would catch any malformed branch shape.
- **Type safety:** all touched files type-clean. `ChainEntry` is a discriminated union; `RecognizedBranch` is a tight interface; no `any` introduced.
