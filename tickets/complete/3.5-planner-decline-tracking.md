## What shipped

An **ephemeral per-node decline set** in `PassManager.applyPassRules`
(`packages/quereus/src/planner/framework/pass.ts:557-618`). Inside the
`while (changed)` fixpoint loop a rule that declines (returns `null` or the same
node) on the current node id is added to a local `declinedOnCurrent` set and
skipped on later iterations of that same unchanged node. The set is **reset to
empty every time a rule transforms the node** (mints a new id), so every decliner
gets a fresh shot on the changed plan piece.

Applied rules are unchanged: still tracked via `context.visitedRules`
(`markRuleApplied` / `hasRuleBeenApplied`) and inherited across a re-mint by
`inheritVisitedRules`. Declines are **not** inherited and **not** stored on the
context. Net effect: strict speedup, zero plan-output change.

### Deviation from implement-ticket body (upheld on review)

The implement ticket's body proposed *inheriting* declines across the transform
re-mint. That was tried first and broke a real test
(`test/optimizer/parallel-async-gather-zip-by-key.spec.ts` →
*"folds a derived scalar expression over the merged key"*): the optimizer rule
set is **not decline-monotonic** — a rule that declines on shape A can legitimately
apply to shape A′ after a *sibling* rule rewrites A→A′. The ephemeral approach
implements the ticket **description** literally ("not re-offered *unless the plan
piece actually changed*"). The implementer's judgment here was correct and is
well-documented in code + docs; review upholds it.

## Review findings

Adversarial pass over commit `e796aeb4`. Read the diff first, then handoff.
Checked from SPP / DRY / soundness / convergence / resource / type-safety angles.

**Correctness / soundness — clean.** Traced the fixpoint by hand across the
tricky interleavings: (a) decliner-before-transformer in one for-pass, (b)
transformer-before-decliner, (c) a third rule declining on N1 after a mid-loop
transform reset. In every case the implicit keying of `declinedOnCurrent` to
"current node id" (via reset-on-transform) stays consistent with
`hasRuleBeenApplied`'s explicit node-id keying — no decliner is wrongly
suppressed on a changed node, none is wrongly re-run on an unchanged one. The
gather canary (`parallel-async-gather-zip-by-key`) passes, confirming the
description-faithful reading preserves the non-monotonic re-offer.

**Convergence — unchanged.** The `changed` flag is set only on transform, exactly
as before; the decline skip never touches it. Fixpoint termination and the
`maxRulesFired` guard are untouched (`pass-manager.spec.ts` still green). No
non-termination or perf-regression path — the change strictly reduces `rule.fn`
calls.

**Tests — adequate for the change's altitude.** 4 new unit tests cover the win
(same-node re-run cut), soundness (re-offer after sibling transform — the guard
against anyone re-introducing inheritance), fresh-rule offering post-transform,
and a 3-node fan regression. They exercise the mock harness; the real-rule
integration signal is the gather canary + full suite. Verified the test helpers
(`makeRule`/`relNode`/`makeContext`) and the hand-traced call counts match the
asserted expectations. No coverage gap worth a ticket.

**Docs — verified current.** `docs/optimizer.md` § Rule Application Control and
`framework/README.md` Loop Detection bullet both read accurately against the code
and record *why* decline inheritance was rejected (with the
`ruleAsyncGatherZipByKey` counterexample). No stale doc found among touched files.

**Tripwire (recorded, not ticketed).** `pass.ts:604-611` carries a `NOTE:`: a
rule that declines *but mutates shared `context` state* expecting a second look
at the same unchanged node next iteration would no longer get it. No such rule
exists today (full suite green); the old code re-ran every decliner each
iteration, so this is a genuine, very-low-risk narrowing. Correctly parked as a
code comment per tripwire rules — conditional ("fine now; only if such a rule is
added"), not a latent defect. If it ever trips: exclude that rule from the skip
or key the skip on a context epoch.

**Known gaps (from handoff, confirmed non-blocking).**
- The win is ordering-dependent and modest (only cuts a re-run when a decliner
  sits after a transformer in registration order and the loop re-iterates). No
  rule reordering attempted — out of scope, correctly not forced.
- No perf/bench numbers; improvement validated structurally via call-count tests,
  not timed. Acceptable — a bench would be a separate effort.
- Trace hooks (`onRuleStart`/`onRuleEnd` in `framework/trace.ts`) still not wired
  into `applyPassRules`. Pre-existing gap, unrelated to this ticket; a future
  rule-firing counter could reuse them. Not filed.

**Findings requiring action: none.** No minor inline fixes, no major follow-up
tickets. Empty because the change is genuinely small and surgical and the
soundness hazard the ticket flagged was correctly navigated.
