## What shipped

The reserved `quereus.update.*` tags now have **Effect** (shape/site validation already lived in
`schema/reserved-tags.ts`):

1. **Statement-level tag separation** via a distinct `WITH TAGS (...)` clause on DML
   (`tags?` on `InsertStmt`/`UpdateStmt`/`DeleteStmt`). The pre-clause `WITH` handlers
   (pre-source / pre-SET / pre-WHERE) accept `TAGS` alongside `CONTEXT`; `ast-stringify` emits the
   clause (trailing) for round-trip.
2. **Collect / validate / merge** — `planner/mutation/mutation-tags.ts` validates `view.tags` at
   `'view-ddl'` and `stmt.tags` at `'dml-stmt'` through the registry's `validateReservedTags`,
   raises the first error as a **sited** `QuereusError`, logs warnings, and merges
   `{ ...viewTags, ...stmtTags }` (statement wins). Called from `view-mutation-builder.ts` before
   any base op is built. Typed readers wrap `getReservedTag`/`getReservedTagByTemplate` — no
   re-parsing.
3. **Consumption** — merged tags ride `MutationRequest.tags`:
   - `default_for.<col>` (single-source insert) supplies an omitted-insert default ahead of the
     base column default (VALUES-append path; value is a TEXT **expression** parsed via
     `parseExpressionString`).
   - `target` / `exclude` (multi-source) narrow the candidate base sides; UPDATE treats them as a
     guard (`tag-conflict` on an assignment to an excluded side), DELETE narrows the candidate set.
   - `delete_via` (multi-source delete): `'parent'` → FK-parent, `'left_delete'` → left source;
     `'right_insert'` (except-only) rejected for joins.
   - `policy` (view-DDL): `strict` rejects any unresolved multi-side delete; `lenient` (default)
     keeps the FK-many fallback.

New diagnostic reasons: `tag-conflict`, `policy-strict-ambiguity`.

### Major — filed to backlog

- **Lenient multi-side delete fan-out is deferred** (`tickets/backlog/view-mutation-lenient-multiside-delete-fanout.md`).
  The doc's maximal lenient reading — an ambiguous join delete deletes from *every* side — is not
  implemented; the substrate runs base ops sequentially against live state, so a per-side join
  subquery is invalidated by the prior side's delete. Shipped behavior **rejects** ambiguity (safe,
  tested, documented). A correct fan-out needs snapshot-consistent base-op execution or eager key
  materialization. The implementer explicitly asked the reviewer to decide on a follow-up — filed.

### Minor — documented, no fix

- **Reserved tags on a *base-table* (non-view) DML are silently ignored.** `stmt.tags` is consumed
  only in `view-mutation-builder.ts`; a base-table `insert/update/delete ... with tags
  ("quereus.update.target" = ...)` parses but is never validated or applied. This is consistent
  with the single-source-view no-op contract (statement tags currently mean something only for view
  mutations) and is low-impact, so it is not fixed here. A future hardening could warn on a reserved
  `quereus.update.*` key carried by a base-table DML; not worth a ticket on its own today.

- **`target`/`exclude` name resolution is order-sensitive on a pathological alias collision.**
  `applyTargetExclude` maps both each side's table name and its alias into one `Map`; if one side's
  alias equals the other side's *table name*, the later `set` wins. Realistically unreachable (a
  user would have to alias side A to side B's table name) and harmless given the
  `tag-target-not-found` / `tag-conflict` guards. Noted only for completeness.

### Not found / empty categories

- **Regressions** — none. Full suite green; the change is additive (a new optional clause + a new
  optional `tags` field consumed only on the view-mutation path; the single-source cost is unchanged
  when no tag is present, guarded by the early `undefined` return in `collectMutationTags`).
- **Resource cleanup** — N/A; the change is pure plan-time AST/analysis with no I/O, handles, or
  long-lived state.
- **Performance** — N/A; `collectMutationTags` returns early (`undefined`) for the common no-tag
  case, leaving propagation cost identical to before.

## Known gaps / deliberate deviations (carried forward from implement, reviewer-confirmed)

1. **Lenient multi-side delete fan-out deferred** — see Major above (backlog ticket filed).
2. **`delete_via` join mapping is asymmetric** — `'left_delete'` picks the left side; the right
   side is reached via `target` (no direct enum value). Faithful to the enum's except-branch origin.
3. **View tags are validated at mutation time, not at CREATE VIEW** — a view with a malformed
   reserved tag is created successfully; every mutation through it then errors. Candidate future
   hardening; out of scope here.
4. **Multi-source insert + `default_for` never interact** — multi-source insert is rejected
   (`unsupported-multisource-insert`) before `default_for` would apply; the outer-join null-extend
   `default_for` path is not implemented (outer joins rejected).
5. **`default_for` requires a VALUES source** (inherits the constant-FD-fill constraint).
