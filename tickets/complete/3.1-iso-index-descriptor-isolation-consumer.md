## What shipped

The isolation layer's overlay/underlying merge picks its comparator from the planner's typed
`FilterInfo.accessPath` (`resolveScanIndex`) instead of regex-parsing the free-text `idxStr`.
Unknown/unresolved index → `QuereusError(INTERNAL)` rather than a silent (and wrong) primary-key
merge. `adaptFilterInfoForOverlay` retargets an aliased-PK FilterInfo to the overlay's canonical
`_primary_` via the engine's `retargetFilterInfoIndex`. The old regex helpers
(`PK_INDEX_NAME_RE`, `SUFFIXED_PK_IDXSTR_RE`, `getIndexColumnIndices`) are deleted. Docs gained a
"scan-order source of truth" subsection under **Key Ordering**.

See the implement commit (`ticket(implement): iso-index-descriptor-isolation-consumer`) for the
full change description; this file records only the review.

## Review findings

Reviewed the implement-stage diff (isolated-table.ts, the spec, the design doc) against the typed
`AccessPath` union (`packages/quereus/src/vtab/index-descriptor.ts`), the `retargetFilterInfoIndex`
helper, and the merge call flow.

**Checked — correctness / call flow.** `resolveScanIndex` is the gate in `mergedQuery` (line 416),
run *before* any overlay adapt or query, so both the missing-`accessPath` and `unresolvedIndex`
throws fire before a bad comparator can be chosen. `role` is read authoritatively over `name` in
both `resolveScanIndex` and `adaptFilterInfoForOverlay`; secondary merge uses the descriptor's full
`keyColumns`. No defect found.

**Fixed inline (minor — type safety).** The `resolveScanIndex` switch on `path.kind` omitted the
exhaustiveness `never`-guard that every other discriminated-union switch in this codebase carries
(e.g. `isolation-module.ts` `translateOverlayRow`, `store-table.ts`, `select.ts`). Because
`noImplicitReturns` is off in these packages, a future 5th `AccessPath` kind would compile clean and
make `resolveScanIndex` return `undefined`, crashing at the `indexInfo.type` read one line later.
Added a `default: { const _exhaustive: never = path; throw QuereusError(INTERNAL) }` guard (imports
`AccessPath` as a type). Rebuilt isolation + reran its suite: **210 passing**.

**Checked — the implementer's open "store path unvalidated" concern; resolved, no ticket.** The
worry was that the new `unresolvedIndex` throw is a live behavior change for a store/lamina module
that aliases an index name without supplying an `indexDescriptor`. Verified the store module
(`quereus-store`) advertises its PK as the canonical `_primary_` (`store-module.ts:2149/2185/2429`)
and its secondary indexes by their real schema name (`:2350`) — both resolve through
`resolveIndexDescriptor` to a proper `index` descriptor (`primary`/`secondary`), **never** to
`unresolvedIndex`. So the store-backed path cannot hit the new throw; `test:store` was not required.
Only an external aliasing module (lamina's `_primary_1`) reaches it, which the engine-seam prereq's
contract now requires to return a descriptor.

**Tripwires (conditional; parked in place, not ticketed).**
- DESC secondary-index direction: `buildCompareSortKey` still folds column direction in only via
  `underlyingTable.getIndexComparator`, falling back to a direction-blind `compareSqlValues`.
  Unchanged by this ticket; a separate pre-existing defect already flagged upstream. Not a
  regression here.
- `empty`→primary branch in `resolveScanIndex` is defensive: `EmptyResultNode` never calls
  `query()`, so no real runtime path constructs `{ kind: 'empty' }`. Test 6 exercises it directly.
  Confirmed it should stay a mapped primary-key case rather than a throw (an impossible predicate
  must not surface overlay rows in the wrong order if that ever changes).

**Test coverage — adequate, one incremental gap noted, not ticketed.** Happy path, tombstone
revival with a changed index key, `_primary_extra` structural-secondary classification, aliased-PK
with/without descriptor, missing-`accessPath`, empty-plan, ANALYZE-over-dirty-overlay, and
two-table commit are all covered (7 new specs + retargeted suffixed-PK suite). End-to-end
*planner-driven* secondary-index merge is only covered indirectly (the existing `idx_email` test +
hand-built `secondaryScanFilter`); an added `create index … ; begin ; write ; select … order by
<indexed col>` SQL test would raise the floor but needs care to actually hit the merge path (the
planner may pick PK-scan-plus-sort). Left as a floor-raiser, not filed — the secondary merge path
is exercised structurally today.

**Docs.** `docs/design-isolation-layer.md` "scan-order source of truth" subsection read and
confirmed accurate to the shipped `resolveScanIndex` (the `accessPath`→merge-order table, full-scan
contract, and the INTERNAL-error contract all match the code).
