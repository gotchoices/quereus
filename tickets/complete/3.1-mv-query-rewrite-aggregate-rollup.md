The aggregate arm of `ruleMaterializedViewRewrite` recognizes a logical `Aggregate(Filter?(scan(T)))` answered from a grouped MV and rewrites it to scan (exact-key) or scan + re-aggregate (rollup) the MV's backing table. See `docs/materialized-views.md § Aggregate rollup` and `docs/optimizer.md § Aggregate-rollup arm` for the architecture, the decomposable-aggregate recombine table, and the soundness witnesses.

### What was checked

- **Read the full implement diff fresh** — matcher (`analyzeAggregateFragment` / `matchAggregateFragmentToMv` + recipe helpers), rule (`rewriteAggregate` / `buildRollupReplacement` / the shared `buildBackingSource` refactor + cost surface), the second optimizer registration, all three test files, and both doc updates.
- **Build / lint / full suite** — `yarn build` green; `yarn lint` clean (0); full `@quereus/quereus` suite **4656 passing, 9 pending, 0 failing** (was 4654 — +2 from the regression tests below).
- **Soundness contract** (a false Match returns wrong rows; every check must forgo on doubt):
  - **One-row-per-group witness** (`backingPkIsGroupKey`) — verified it requires the backing PK to equal the MV group key exactly, so the exact-key scan returns one row per query group and the rollup re-aggregates a set, not a bag. Conservative.
  - **Residual confined to group keys** — `backingColOfBaseCol` exposes only stored group-key columns; a residual on a non-group column ⇒ `missing-column`. Verified.
  - **`avg` recombine** (`sum(sum)/sum(count)`) — the count must exclude the same NULLs `avg` does; confirmed `count(x)` always qualifies and `count(*)` only when the column is declared `NOT NULL` (the `aggregate-not-decomposable` forgo for nullable-`x` + only-`count(*)` is unit-tested). Real division matches native `avg`; NULL/NULL over zero rows ⇒ NULL.
  - **`count`-over-zero-rows = 0** — the `coalesce(sum(cnt), 0)` wrapper restores 0 (not NULL) for the empty global group; the number-literal `0` matches the native count's numeric type (verified empirically — see below).
  - **The one residual-exposure path I could not close by reading** — a grouped MV storing a *bare non-group column* whose value a residual could then filter — was verified **unreachable**: Quereus rejects a grouped select that picks a bare non-group, non-FD column at create time, so any stored bare column is a group key or FD-determined by it, and filtering an FD-determined column post-aggregation equals filtering pre-aggregation. Sound.
- **Forgo guards are conservative** (lose only a speedup, never correctness): `rollup-residual` (works around the pre-existing composite-PK filter-drop bug) and `group-key-pinned` (avoids the base's FD-simplification column reorder) — both confirmed sound and unit-tested.
- **Cost gate / tiebreak** — exact preferred over rollup by cost; stable lowercased-name tiebreak. Multiple-covering-MV case exercised (see regression tests) and stays row-identical.
- **Docs** — read both touched docs end-to-end against the code; they accurately reflect the two sub-cases, the recombine table, the soundness witnesses, the pass placement / second-registration mechanics, and the cost surface. Up to date.

No divergence.
- **No major findings.** The two forgo guards are conservative, the soundness witnesses hold, and the only unverified-by-reading path (residual on a stored non-group column) is unreachable by create-time validation.
