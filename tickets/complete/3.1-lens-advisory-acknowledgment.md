## What shipped

Governance layer atop the prover's advisory channel: in-source acknowledgment of
coded+sited+fingerprinted advisories (`quereus.lens.ack.<code>[:<target>]`),
fingerprint-driven re-surfacing on material change, and a per-(logical-table)
escalation policy (`quereus.lens.policy.{error-on,require-ack}`). Deploy report
gained `acknowledged: AcknowledgedAdvisory[]`; the `quereus_lens_advisories(logical_schema)`
TVF expands the full advisory list (active / re-surfaced / acknowledged /
acknowledged-unconditional). No new enforcement semantics — pure governance.

See the implement commit `2c52bcf8` and `docs/lens.md` § Acknowledging advisories
for the full mechanics. This file records the review pass.

### Minor — fixed in this pass

- **Policy code prefix silently no-op'd the bare form.** The ack *tag* names a code
  bare (`quereus.lens.ack.no-backing-index`), but `error-on`/`require-ack` compared
  the policy CSV against the fully-qualified `w.code` (`lens.no-backing-index`).
  A developer mirroring the ack-tag spelling (`require-ack = 'no-backing-index'`)
  would have silently escalated *nothing*. Fixed: `parseCodeCsv` now normalizes
  every entry through `advisoryCodeBase`, and the governance compares against the
  advisory's code base — so both `lens.no-backing-index` and `no-backing-index`
  escalate identically. (`lens-ack.ts`.) Regression test added.
- **Untested live paths covered.** Added three tests to `lens-ack.spec.ts`:
  constraint-*scoped* acks (the `collectConstraintAcks` path — previously only
  table-level acks were exercised; confirmed `constraint uq unique (col) with tags`
  parses in logical DDL and the ack suppresses only its own constraint), a sibling-
  constraint-still-surfaces case, and the bare-code policy form above. 8 → 11 tests.

### Minor — noted, not fixed (low impact)

- **`lens.partial-override` carries no `fingerprintInputs`** (set in
  `lens-prover.ts:checkPartialOverride`). An ack on it works only *untargeted* and
  its fingerprint depends on table name alone, so it never re-surfaces when the
  authored/gap-filled column split changes. Acceptable for an informational table-
  level advisory; if anti-fatigue is ever wanted there, the prover would populate
  `fingerprintInputs` (authored/gap-filled column lists). Left as-is.
- **TVF column names `constraint` / `column` are SQL keywords.** `select *` works;
  selecting them by bare name needs quoting. This is the *established* convention
  in this codebase (`schema.ts` foreign-key TVF uses `table`/`from`/`to`), so it
  was left consistent rather than renamed.
- **Double meta-warning surface for an empty rationale.** An empty-rationale ack
  produces both a reserved-tags validation warning (logged) and an ack-governance
  report warning. Different channels, both intentional; not deduplicated.

### Major — filed as a follow-up ticket

- **`tickets/fix/lens-policy-code-validation.md`** — an *unrecognized* policy code
  (typo / stale, e.g. `lens.no-backing-indx`) is silently dropped: the escalation
  fails **open**. The author believes a code is escalated to a hard error, but the
  deploy stays advisory. The reserved-tag layer can't catch it (no advisory-code
  vocabulary there); validation must live where the code set is known. Ticket specs
  validating policy entries against an exported acknowledgeable-code set and
  surfacing unknown entries (warning vs error decided at implement). Out of scope to
  fix inline because it needs a deliberate severity/vocabulary design choice.

### Deviation confirmed (implementer asked)

- **Escalation policy is per-(logical-table), not schema-wide.** The source ticket
  framed it as "per-schema tag or engine option"; there is no schema-level tags AST,
  and an engine option would plumb through `database.ts`. The in-source per-table
  reserved tag is the most consistent with the ack philosophy (version-controlled,
  review-visible) and is **accepted**. A global engine-option source remains a
  reasonable future ergonomic and is already noted in `docs/lens.md` and the
  shipped-status bullet — no separate ticket filed (it is a genuine future-only
  enhancement, not a gap in this slice).

### Empty categories (explicit)

- **Type safety** — no `any`; the new interfaces are fully typed; nothing to fix.
- **Resource cleanup** — the TVF is a pure async generator over an in-memory
  report; no handles to release. Tests close every DB. Nothing to fix.
- **Performance / scalability** — governance is O(advisories × acks) per table on a
  small per-table tag surface; the fingerprint is a single FNV-1a pass over a short
  canonical string. No concern.
