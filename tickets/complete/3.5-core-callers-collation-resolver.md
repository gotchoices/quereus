# Complete: core callers resolve collations against the database

## What landed

Two subsystems inside `packages/quereus` stopped passing collation *names* to
`compareSqlValues(a, b, name)` — which resolved them against a process-global map holding
only the three built-ins, making `db.registerCollation(...)` invisible.

**Materialized-view maintenance.** `MaterializedViewManagerContext` gained
`getCollationResolver()` (satisfied structurally by `Database`). Each backing primary-key
descriptor is now a `BackingPkColumn` — `{ index, desc?, collation?, collationFn }` — built
once per plan by `resolveBackingPkColumns()`. The three per-row helpers in `-apply.ts`
(`residualRowMatchesKey`, `backingPkEqual`, `residualRowMatchesBasePrefix`), the two sites
in `-views.ts` (`lookupCoveringConflicts`' UNIQUE match and self-conflict skip), and
`CoarseningWatchColumn.sourceCollationFn` all call `compareSqlValuesFast` with a pre-resolved
function.

**Satisfiability checker.** `checkSatisfiability` gained a sixth parameter,
`collationResolver?: CollationResolver`. Every column mentioned by the conjuncts, domains, or
bindings has its declared collation resolved once, before the conjunct loop. `BINARY`
short-circuits (it cannot be overridden — `registerCollation` rejects it, so the fast path is
sound); any unresolvable name makes the whole check return `'unknown'`, so the `Filter`
stands. Both call sites pass a real resolver.

**Three wrong-results bugs fixed** (all reproduced on `main` first): a custom collation
degraded to BINARY (`x = 'a' and x = 'b'` on a `collate nocase` column with a redefined
NOCASE); an erased `COLLATE` wrapper (`x collate nocase = 'a' and x collate nocase = 'A'`);
an erased value-changing `CAST` (`x = '1' and cast(x as integer) = 1`). The last two came
from `unwrap()` stripping `CastNode`/`CollateNode` unconditionally; it now strips only a
no-op cast (`isNoOpCast`) and never a `COLLATE`. Also fixed: `lookupCoveringConflicts`
compared a source `UNIQUE` column under the global collation, letting a covering MV admit a
duplicate under a redefined `NOCASE`.

## Review findings

### Major — filed as a ticket

- **`constraint-extractor.ts` has the same value-changing-`CAST` defect the implementer fixed
  in `sat-checker.ts`, and it is reachable today with wrong rows.** The sat-checker's `unwrap`
  comment cites `unwrapCast` as "already reasoning this way" — that is true for `COLLATE`
  only. `unwrapCast` strips *every* `CastNode`, so `select x from t where cast(x as integer) = 1`
  on `create table t (x text primary key)` plans to a bare `INDEXSEEK` on `x` for the integer
  `1`, with the predicate dropped, and returns **no rows**; the same query against a
  non-indexed column correctly returns `'1'` and `'1abc'`. Verified with `query_plan` and by
  executing both shapes. Filed as `fix/bug-cast-stripped-from-seek-constraints` with the
  repro, the plan output, and scope notes. Out of this ticket's diff (different file, not
  touched by it) and not a one-line fix — `unwrapCast` feeds `isColumnReference` and the
  covered-key path.

### Minor — fixed in this pass

- **`lookupCoveringConflicts`' new comment was wrong and invited an unsound refactor.** It
  claimed the resolved collations are "the UC's own enforcement collation … see
  `uniqueEnforcementCollations` for the same name selection." They are not: the code uses the
  *declared source column* collation, and `uniqueEnforcementCollations` prefers the index's
  per-column collation when the constraint is `derivedFromIndex`. The declared collation is
  deliberate — this method generates a candidate *superset* that the re-validators then filter
  under the index collation, sound only because `findRowTimeCoveringStructure` declines any MV
  failing `coveringMvHonorsIndexCollation`. Rewrote the comment to say so and to forbid the
  "DRY" merge. Retitled and re-commented the corresponding test (which only ever exercised the
  non-index-derived case where the two coincide) so its name no longer over-claims.
- **`tryResolve`'s doc comment cited a repro that does not reproduce.** It said
  `create table t (id integer primary key, k integer collate frobnicate)` then
  `select * from t where k = 1` "throws where it previously succeeded". That query throws at
  *emit* (`emitComparisonOp` → `EmissionContext.resolveCollation`) at HEAD, with or without the
  sat-checker change. The leniency is nonetheless load-bearing, just for a different shape:
  add `check (k > 0)` and the column gains a `DomainConstraint`, so `resolveColumnCollations`
  resolves `k` even though no emitted comparison touches it — `select id from t where id = 1`
  then throws without the catch. Confirmed by enabling the `sat-checker:warn` channel and
  seeing the swallowed throw. Comment corrected to the real repro.
- **The leniency had no test.** Added an end-to-end regression to
  `predicate-contradiction.spec.ts` for exactly that shape (unregistered collation on an
  unrelated `CHECK`ed column; the query must run, not throw).

### Checked, no defect found

- **`resolveColumnCollations` collects a superset of the columns `absorb` ever compares.**
  Domains contribute `d.column`, bindings `b.attrs`, and `collectColumns` walks the same
  `'expression' in child` edges the shape matchers descend, so `cmp`'s `?? BINARY_COLLATION`
  fallback is never the path an unresolvable collation takes. The refactor of
  `markUnknownForColumns` onto `collectColumns` is behavior-preserving.
- **The `BINARY` short-circuit is sound.** `Database.registerCollation` rejects `BINARY`
  outright, so the checker's fast path can never disagree with the runtime comparator. Both
  `normalizeCollationName(name) === 'BINARY'` (checker) and `collationName === 'BINARY'`
  (resolver) reach the same comparator for every spelling.
- **`unwrap`'s remaining no-op-cast strip cannot mint a false `'unsat'`.** For
  `cast(x as text) = lit` the checker uses `x`'s declared collation while emit may use BINARY;
  a coarser checker collation can only turn `'unsat'` into `'sat'` — a lost fold, never wrong
  rows.
- **`pkCollationFns[i]` cannot be `undefined`.** `sourcePk` is built from `pkBackingCols`,
  which is `pkDef`-length, and the loop is guarded on `sourcePk.length === newSourcePk.length`.
- **`buildPrimaryKeyFromValues` ignores `collation`,** so widening `PrimaryKeyColumnDefinition`
  to `BackingPkColumn` changes no key bytes; the extra `collationFn` field is inert there.
- **One `MaterializedViewManager` construction site** (`Database`), so the new context method
  needed no other wiring.
- **`ctx.db.getCollationResolver()` per `FilterNode` visit is not a per-rule allocation** — the
  resolver closure is memoized on `Database`.
- **Byte-canonical dedupe in `applyForwardResidual`** (`canonKeyValues`) sits next to the now
  collation-aware `residualRowMatchesKey`. Not a defect: its own comment establishes that both
  residuals read the same live state, so surviving keys are byte-identical.
- **Docs.** `docs/optimizer.md`'s sat-checker section and `docs/sql.md`'s collation section
  were read against the new code and are accurate, including the new `GROUP BY` byte-grouping
  caveat (which is the `fix/bug-key-normalizer-ignores-database-collations` ticket's subject).
