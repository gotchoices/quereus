# Inline before-image on `RowDeletion` (`priorRow`) — COMPLETE

The row-level companion to the per-cell before-image (`sync-change-before-image-column`).
An optional, purely-additive **last-known row image** rides on a synced deletion and
is persisted on the tombstone, so a receiver can show or undo *what was removed* via
`getChangesSince` without reconstructing it beforehand.

```ts
interface RowDeletion {
  // …type, schema, table, pk, hlc…
  readonly priorRow?: Row;   // row's last-known image at the origin (audit/undo)
}
interface Tombstone {
  hlc: HLC;
  createdAt: number;
  priorRow?: Row;            // persisted so it reaches the wire via getChangesSince
}
```

`priorRow` is sourced from the engine `oldRow` (copied, no aliasing), persisted on
the tombstone as a JSON tail after the fixed 38-byte head (absent → unchanged
38-byte head), relayed via both `resolveLogEntry` and `collectAllChanges`, and
re-persisted on apply (`commitDeleteMetadata`) so the winning/latest delete's image
survives key reuse and in-batch dedup.

## Review findings

### Scope reviewed
Read the implement diff (`8b6797df`) with fresh eyes before the handoff, then traced
every producer/consumer of `RowDeletion.priorRow` and `Tombstone.priorRow`:

- **Type / docs** (`protocol.ts`, `docs/sync.md`): additive optional field, accurate
  best-effort prose, storage-key table updated to `{hlc, createdAt, priorRow?}`. Docs
  match reality; the new code block is well-formed.
- **Serialization** (`tombstones.ts`): fixed 38-byte head + JSON tail only when
  present. Verified the absent/present boundary — a present `priorRow` always
  serializes to ≥ 40 bytes (min JSON `[]`), so `byteLength > 38` is an exact
  present/absent discriminator and the empty-array case is distinguishable from
  absent. `DataView(buffer.buffer, byteOffset, byteLength)` correctly handles
  sub-buffer views; `getBigUint64(30)` is safe at the 38-byte boundary. Cells go
  through `encodeSqlValue`/`decodeSqlValue` so `Uint8Array`/`bigint`/`null`
  round-trip.
- **Producer** (`recordDataEvent` delete branch): `[...oldRow]` copy onto both the
  tombstone and the inline change; absent when no `oldRow`. No engine-buffer aliasing.
- **Relay** (`resolveLogEntry`, `collectAllChanges`): conditional-spread, sourced
  from freshly-deserialized tombstones (no shared-buffer aliasing).
- **Apply / dedup** (`commitDeleteMetadata`, `commitChangeMetadata`): verified only
  the max-HLC winner per pk reaches `commitDeleteMetadata` (`keepMaxHLC` +
  `deleteWinners` map), so the surviving `priorRow` is the latest/winning delete's
  image. In-batch losers never persist.
- **Quarantine** (`quarantine.ts`): out-of-listed-files extension by the implementer,
  but correct — honors the module's "verbatim wire fidelity" contract with the same
  conditional-spread/absent-tolerant pattern as the column before-image.
- **Other `setTombstone` callers**: only the two production call sites
  (`recordDataEvent`, `commitDeleteMetadata`) create deletion tombstones; both
  updated. Snapshot-reconstructed tombstones legitimately carry no `priorRow`
  (documented).

### Major findings
**One — filed, not blocking this ticket.** The JSON wire transport
(`serializeChangeSet`/`deserializeChangeSet` in `quereus-sync-client` and
`sync-coordinator`) silently drops **all** before-image fields — `priorRow` here and
`priorValue`/`priorHlc` from the column ticket. So over a coordinator-mediated
browser↔browser path the before-image does not survive, even though it works
in-process and via `getChangesSince` re-resolution. This is a pre-existing gap the
column ticket's review missed (it only fixed the *quarantine* serializer). Filed as
`tickets/backlog/sync-before-image-wire-transport.md` covering both features.

### Minor findings (fixed in this pass)
- **Incremental relay path was untested despite the handoff claiming it.** The new
  e2e tests all drain via `getChangesSince(peer)` (no `sinceHLC`), which routes to
  `collectAllChanges`; the `collectChangesSince` → `resolveLogEntry` path (the
  realistic incremental sync path) re-emits `priorRow` from a *separate* code site
  that nothing exercised. Added an e2e test that passes a zero `sinceHLC` to force
  that path and asserts `priorRow` round-trips. **307 passing** (was 306).

### Known gaps carried forward (not bugs)
- **LevelDB store round-trip unverified.** `test:store` is slow / not agent-runnable

  exercised via `InMemoryKVStore`, so the LevelDB path should be unaffected — left
