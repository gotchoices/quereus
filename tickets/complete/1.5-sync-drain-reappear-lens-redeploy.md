# Complete: low-latency scoped drain when a lens redeploy re-maps a basis table back

## What shipped

A local `apply schema` lens redeploy that re-maps a basis table from `detached` back into the
basis now drains that table's held out-of-basis changes **immediately** — as a separate
post-commit apply unit — instead of waiting up to one periodic-sweep interval (~5 min). This is
the second reappearance path; the first (inbound `create_table`) landed in the
`sync-drain-reappear-inbound-ddl` prereq.

The change is small and entirely additive — no new flag, no new helper, no worker changes. In
`SyncManagerImpl.recordLensDeployment` (`sync-manager-impl.ts`):

1. A `reappeared: Array<{schema, table}>` is declared before the per-key classification loop.
2. Inside the loop, after each record is built, `{schema, table}` is pushed whenever
   `wasDetached && !isDetached` (the precise `detached → present` transition, using the existing
   `wasDetached`/`isDetached` locals).
3. After `if (wroteAny) await batch.write()` **and** the `onBasisTableLifecycle` emit loop,
   `await drainReappearedTables(this, reappeared)` runs — reusing the prereq's exported advisory
   helper and the `SyncConfig.drainOnReappear` flag.

The drain runs after the lifecycle batch is durable and its events have fired, so the basis
oracle (`getTableColumnNames`) sees the re-attached table and the held changes LWW-resolve
against the present basis — symmetric with the inbound-DDL path.

## Review findings

Adversarial pass over the implement diff (`git show a77e50d7`), read with fresh eyes before the
handoff summary. Cross-checked against the shared `drainReappearedTables`/`drainHeldChanges`
machinery and the sibling inbound-DDL path.

### Checked and confirmed correct

- **Ordering invariant.** The drain call sits after `batch.write()` and after the
  `emitBasisTableLifecycle` loop, so the re-attached table's lifecycle record is durable and the
  oracle resolves it before any held change is replayed. Matches the code comment and is
  symmetric with the inbound path's post-`admitGroup`/post-`emitRemoteChanges` placement.
- **Advisory safety (no drain throw escapes the deploy).** Double-guarded: (a)
  `drainReappearedTables` wraps each table's `drainHeldChanges` in its own `try/catch` +
  `console.warn`, and (b) the production caller — `quereus.worker.ts`
  `setLensDeploymentListener` (lines 678-683) — is itself documented as advisory-guarded by the
  store module. Verified directly by test 5 (the deploy completes, records durable, entries stay
  held after `failApply` throws).
- **Transition semantics.** `wasDetached && !isDetached` correctly fires only on
  `detached → present` and excludes both brand-new tables (`prior` undefined ⇒ `wasDetached`
  false) and idempotent re-deploys (no state change). Covered by tests 2 and 3.
- **No dedup needed.** Unlike the inbound path (which dedups via a `Map` because multiple
  migrations in one batch can name the same table), `recordLensDeployment` iterates a
  `Set<string>` of relation keys, so each table is pushed at most once. Correct design choice,
  not an oversight.
- **`drainOnReappear: false` gate.** The flag defers to the host sweep; the `drainHeldChanges`
  primitive itself is not gated by it. Covered by test 4.
- **Docs.** `docs/migration.md` § 4 Contract and `docs/sync.md` § Revival/drain both read
  correctly against the new reality — the "Low-latency reappearance" bullet is now split into
  the inbound-`create_table` and lens-redeploy sub-cases, and the test-coverage map names the
  new spec block. Confirmed by reading both files in full against the diff.

### Found and fixed inline (minor)

- **Untested "any in-basis state" claim.** The code comment asserts `detached → present` covers
  *any* in-basis target (`directly-mapped`, `derivation-source-only`, `unreferenced`), but the
  implementer's five tests only exercised `detached → directly-mapped`. Added a sixth test —
  *"detached → unreferenced (back in basis but unmapped) also drains"* — which re-maps the table
  into the basis via the db enumeration while the snapshot maps nothing to it, classifying it
  `unreferenced`, and asserts the held change still drains. This pins the behavioral claim that
  the gate is `detached → present`, not `detached → directly-mapped`. (425 passing, was 424.)

### Found, not fixed — filed as follow-up (see below)

- **No real-engine e2e for the lens-redeploy trigger.** The inbound path has
  `sync-drain-e2e.spec.ts` (real `Database` + `StoreModule` + `createStoreAdapter`); this
  ticket's tests are at the CRDT-metadata + in-memory-stub level only. The *drain machinery*
  (`drainReappearedTables` → `drainHeldChanges` → `admitGroup` → store adapter) is shared and
  already e2e-proven via the inbound path, so the only uncovered seam is the
  `recordLensDeployment` `detached → present` trigger driving a real `apply schema` redeploy —
  which is exhaustively covered at the metadata level here. Low value (duplicative), but a real
  parity gap; filed as `tickets/backlog/sync-drain-reappear-lens-redeploy-e2e.md` rather than
  done inline (an e2e harness with a real detach/re-map redeploy is too much for a review pass).

### Noted, no action (out of scope for this ticket)

- **Case sensitivity of the scoped drain lookup.** `drainHeldChanges` builds `qt:` scan bounds
  case-sensitively from `record.schema`/`record.table` (the display name from schema-manager
  enumeration); if a real basis table were enumerated with different casing than the held
  change carried, the scoped scan could miss (the no-arg periodic sweep would still catch it).
  This is a **pre-existing, shared** latent concern — the inbound-DDL path has the identical seam
  (it uses `migration.schema`/`migration.table`) — and is **not introduced by this ticket**. Not
  fixed here; if it warrants attention it should be addressed once for both paths, not bolted
  onto this redeploy ticket.
