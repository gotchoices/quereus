## Summary

Write-through is **pure routing**, as scoped:

- `view-mutation.ts` — `analyzeView` / `rewriteView{Insert,Update,Delete}` /
  `findViewColumn` / `checkContradiction` / `rejectReturning` now take a minimal
  structural `MutableViewLike` (`{ name; schemaName; selectAst; columns? }`) that
  both `ViewSchema` and `MaterializedViewSchema` satisfy. No behavior change for
  plain views.
- `insert.ts` / `update.ts` / `delete.ts` — `assertNotMaterializedView(...)`
  removed; the view dispatch is now `getView(…) ?? getMaterializedView(…)`, so an
  MV target routes into the same rewrite + re-plan path. The committed-pseudo-schema
  guard stays ahead.
- `schema-resolution.ts` — `assertNotMaterializedView` deleted (no other callers).
- The rewritten write hits source `T`, firing the existing
  `_maintainRowTimeCoveringStructures` hook, so the backing syncs inside the same
  statement/transaction.
- **MV-over-MV correction**: `analyzeView` rejects when the body's single source is
  itself a materialized view (`nested-view` reason, message `… reads a materialized
  view …`). This also covers a plain view over an MV — a pre-existing latent hole.
