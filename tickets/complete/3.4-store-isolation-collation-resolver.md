# Complete: store and isolation table layers resolve collations against their database

## What shipped

A *collation* is a named text-sorting rule. `BINARY`, `NOCASE`, and `RTRIM` ship built in;
an application registers more with `db.registerCollation(name, comparator)`, and may replace
`NOCASE` / `RTRIM` (not `BINARY`).

`StoreTable` (persistent key-value tables) and `IsolatedTable` (the read-your-own-writes
transaction overlay) used to resolve collation names through the three-argument
`compareSqlValues(a, b, name)`, which reads a **process-global** registry holding only the three
built-ins with their built-in meanings and silently falls back to BINARY on a miss. Both classes
now resolve through `db.getCollationResolver()` (the seam added by `collation-resolver-seam`),
which raises `no such collation sequence: X` rather than degrading to BINARY.

### Engine (`packages/quereus`)

- `resolveCollationFunctions(resolver, names)` in `util/comparison.ts` — maps a
  `(string | undefined)[]` of declared collation names to `CollationFunction[]`,
  `undefined ⇒ BINARY`.
- `resolveUniqueEnforcementCollations(schema, uc, resolver)` in `schema/unique-enforcement.ts`
  (added during review) — the `uniqueEnforcementCollations` + `resolveCollationFunctions` pair
  that all four out-of-package UNIQUE re-validators were writing out by hand.
- `MemoryTable.comparePrimaryKey` now delegates to `createPrimaryKeyFunctions` — the same
  factory the table's own layer BTrees are keyed by — so it reproduces the table's native key
  order exactly. (The implement pass made it collation-aware with a bespoke loop; review found
  that loop still diverged on two other axes. See *Review findings*.)

### Store (`packages/quereus-store/src/common/store-table.ts`)

- `protected readonly collationResolver`, bound once in the constructor. The resolver *closure*
  reads the live registry; resolved *functions* are hoisted no further than the comparator or
  constraint check that resolved them.
- Pushed-constraint re-check: `compareValues` takes a pre-resolved `CollationFunction` and calls
  `compareSqlValuesFast`. `matchesFilters` takes a `ReadonlyMap<number, CollationFunction>` built
  once per scan by `resolveFilterCollations(filterInfo, indexCollationNames?)`. All four scan
  arms (PK point, PK range, secondary index, full scan) hoist it above their row loop; the index
  arm passes `indexColumnCollations(access.index)` as the override, preserving the
  index-column-`COLLATE`-beats-declared-column-collation rule.
- All three `UNIQUE` conflict paths resolve once, above the candidate loop.

### Isolation (`packages/quereus-isolation/src/isolated-table.ts`)

- `private readonly collationResolver`, bound once at connect.
- `getPkKeyShape()` — per-PK-column comparison functions *and* sort directions, memoized on
  `TableSchema` identity. Backs the fallback `getComparePK()` comparator (used whenever the
  underlying table exposes no `comparePrimaryKey` — i.e. every store-backed table) and
  `keysEqual()`, which decides "same logical key".
- `checkUniqueAgainstUnderlying` resolves its enforcement collations once, above the scan.

### Deliberately untouched

The name-less `compareSqlValues(a, b)` calls in `store-table.ts`, `isolated-table.ts`, and
`quereus-sync/src/sync/store-adapter.ts` are BINARY by intent.

## The DDL restriction that shapes every test

`create table t (v text collate NOSPACE)` is **rejected today** — `validateCollationForType`
(`packages/quereus/src/schema/table.ts`) checks a column's `COLLATE` against a static list on the
logical type (`BINARY`/`NOCASE`/`RTRIM` for TEXT), never against the connection's registry. So a
custom collation reaches these code paths by exactly two routes, and both suites use both:
**overriding a built-in** (`db.registerCollation('NOCASE', noSpace, stripSpaces)`), and **a
custom name on an index column** (`create unique index ix on t (v collate NOSPACE)`).

`backlog/feat-ddl-accepts-registered-collations` tracks the gap. It is pre-existing.

### Major — fixed in this pass (a confirmed, reachable data bug)

**A `DESC` primary key made an isolation overlay show duplicate rows.** The merge iterator
aligns overlay and underlying entries by a primary-key comparator, so that comparator must
reproduce the underlying's native key order. `MemoryTable.comparePrimaryKey` and
`IsolatedTable`'s fallback comparator both compared **ascending**, ignoring a
`primary key (k desc)` declaration, while both the memory table's BTrees and the store's encoded
key bytes order descending. Reproduced on the working tree before fixing:

```
create table t (k integer, v text, primary key (k desc)) using isolated;
insert into t values (1,'a'), (2,'b'), (3,'c');
begin;
update t set v = 'B' where k = 2;
select k, v from t;   -- observed: (2,'B'), (3,'c'), (2,'b'), (1,'a')  ← k=2 twice
```

The implement pass fixed the *collation* axis of this same "overlay comparator must match the
base table's ordering" contract and stopped there; `comparePrimaryKey`'s documented contract is
"must use the module's native key ordering", and a bespoke value-by-value loop cannot honor that.
`MemoryTable.comparePrimaryKey` now delegates to `createPrimaryKeyFunctions` — the layer BTrees'
own factory — which folds in DESC direction, per-column logical-type semantics (temporal types
have their own comparator, which the loop also bypassed), and the connection-resolved collation.
`IsolatedTable`'s fallback arm now applies each PK column's declared direction. Seven regression
cases added to `isolation-layer.spec.ts`, run against both an underlying that exposes
`comparePrimaryKey` and one that hides it (the store-shaped arm, which the implement handoff
correctly flagged as uncovered — it is now covered), plus a composite mixed-direction key.
Verified separately that a store-backed `primary key (k desc)` table scans descending and now
merges correctly.

### Minor — fixed in this pass

- **DRY.** `resolveCollationFunctions(resolver, uniqueEnforcementCollations(schema, uc))` was
  written out at four call sites across two packages. Collapsed into
  `resolveUniqueEnforcementCollations(schema, uc, resolver)`, exported from `@quereus/quereus`
  alongside the helper it wraps. The header comment on `unique-enforcement.ts` (which asserted
  the store and isolation re-validators import `uniqueEnforcementCollations` directly) was
  updated to match.
- **Stale JSDoc.** `StoreTable.matchesFilters` still documented a `collationOverride` parameter
  that the diff had removed and replaced with a pre-resolved `collations` map. Rewritten, and it
  now states why a column absent from the map is unreachable rather than merely defaulted.
- **Misleading test name.** `custom-collation.spec.ts`'s "resolves an empty collation-name list
  without throwing" declares no UNIQUE constraint at all; what it actually exercises is a scan
  with no pushed constraints, i.e. `resolveFilterCollations` returning an empty map. Retitled and
  its comment corrected.
- **Docs.** `docs/design-isolation-layer.md` § Key Ordering described the memory vtab's ordering
  as `compareSqlValues()` and § Collation Considerations said nothing about *which* registry a
  name resolves against, nor that `comparePrimaryKey` is optional and has a fallback. Both
  sections rewritten. `docs/sql.md` was already updated by the implement pass and is accurate.

### Tripwires — recorded, not ticketed

- `resolveFilterCollations` rebuilds its map on every `query()` / `scanPKRange()` / `scanIndex()`
  call. It is O(constraints) and dwarfed by scan I/O today. Parked as a `NOTE:` on the method in
  `store-table.ts`, pointing at memoizing on the `FilterInfo` if a point-lookup-heavy profile
  ever shows it.
- Both PK comparator caches (`MemoryTable.pkComparatorCache`,
  `IsolatedTable.pkCollationCache`) key on `TableSchema` object identity, not on the collation
  registry, so a `db.registerCollation` that *replaces* a collation after a comparator was built
  is not picked up. This matches the contract `Database.registerCollation` documents engine-wide
  ("registration is not retroactive"); both carry a doc comment saying so. Not a defect unless
  that contract changes.
- The store encoder's `NOCASE` fallback (`encoding.ts`) is a *live* bug, not a tripwire, and is
  filed — see below.

### Major — filed, not fixed (carried forward from the implement pass, re-verified here)

`tickets/fix/bug-store-key-encoder-ignores-database-collations.md`. The store encodes text keys
into sort-preserving bytes through a **separate, process-global** encoder registry
(`packages/quereus-store/src/common/encoding.ts`) holding only `BINARY` / `NOCASE` / `RTRIM` with
their built-in meanings, falling back to the `NOCASE` encoder (lowercasing) on any unrecognized
name. It never consults the database, so comparison and storage disagree: with `NOCASE` overridden
to ignore spaces, `'a b'` and `'ab'` both persist as separate primary-key rows instead of raising
a conflict. Reachable today via a `NOCASE`/`RTRIM` override — it does not need the DDL restriction
lifted. Out of scope here (different file, different registry, and it needs a design decision
about where the store obtains a key normalizer), so it went to `fix/` rather than `backlog/`.
A `NOTE:` at `encodeText` points at the ticket, and `docs/sql.md` § COLLATE carries a "Store
caveat — physical key bytes" paragraph telling embedders to keep store primary-key collations to
the three built-ins until it lands. **Delete that paragraph when the fix ships.**

### Checked and found clean

- **`protected` blast radius.** `matchesFilters` and `compareValues` changed signature. The only
  in-tree subclass is `HarnessStoreTable` (`store-ryow.spec.ts`), which overrides neither. No
  other package extends `StoreTable`, `IsolatedTable`, or overrides `MemoryTable.comparePrimaryKey`.
  `yarn lint` type-checks spec call sites too, so drift would have surfaced.
- **`resolveFilterCollations` vs `matchesFilters` skip conditions.** Both walk
  `filterInfo.constraints` under the identical `iColumn < 0 || argvIndex <= 0` guard, so no
  constraint can reach `matchesFilters` without an entry in the map. The `?? BINARY_COLLATION`
  there is unreachable-by-construction, and now says so.
- **`PrimaryKeyColumnDefinition.collation` vs `ColumnSchema.collation`.** The implement pass
  resolved PK collations from `schema.columns[pk.index].collation` in one place and the engine
  resolves from `pkDef[i].collation` in another. Confirmed at every construction site
  (`schema/table.ts`) that the PK definition's collation is always `col.collation || 'BINARY'`,
  so the two agree. The delegation to `createPrimaryKeyFunctions` removes the second source.
- **`findUniqueConflictViaCoveringMv`** (`store-table.ts`) still has no custom-collation runtime
  test — reaching it needs a row-time covering materialized view linked to the UNIQUE, and the
  `findRowTimeCoveringStructure` collation gate declines a finer/incomparable index-derived
  UNIQUE. The change there is mechanical (hoist the resolve, switch to `compareSqlValuesFast`) and
  type-checked. Left as-is; it is not a defect, only unproven at runtime. Flagging it here rather
  than filing a ticket, because constructing the shape is a test-fixture problem, not a code one.
