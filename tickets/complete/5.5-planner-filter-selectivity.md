## What landed

`FilterNode.estimatedRows` no longer hardcodes `0.5` selectivity. A new Physical-pass rule (`rule-filter-selectivity`) consults `context.stats.selectivity(table, predicate)` — histogram / `1/ndv` equality / null-fraction / naive fallback — and stamps the result onto a new optional `FilterNode.selectivity` field, which both `estimatedRows` (logical) and `computePhysical` (physical) then multiply into the source cardinality.

The node-accessor-carries-no-context problem was solved with a context-holding rule (same pattern as `select-access-path`). `DEFAULT_FILTER_SELECTIVITY = 0.5` remains only as the last-resort default. Details, deviation rationale, and known gaps are in the implement handoff (commit `7a1667cf`).

## Review findings

Adversarial pass over commit `7a1667cf`. Read the full source diff before the handoff; validated the implementer's flagged deviation independently. Lint clean (exit 0), full suite **6902 passing / 13 pending / 0 failing**.

### Verified sound (no change)

- **The `computePhysical` deviation is correct.** The implementer overrode the plan (which claimed selectivity flows automatically) and applied selectivity to the *physical* source cardinality directly. Confirmed necessary: physical access nodes (`SeqScan`/`IndexScan`) expose no logical `estimatedRows`, so the getter yields `undefined` after the `Retrieve → access-node` conversion and the stamp would be lost. The getter (logical source) and `computePhysical` (physical source) apply the factor **once each, to different consumers** — no double-counting. Formulas are identical (`max(1, floor(rows * sel))`), only the source differs.
- **`estimatedCostOverride` dropped by the rule is harmless.** The rule mints `new FilterNode(..., undefined, clamped)`, passing `undefined` for the cost override. Grepped every `new FilterNode(` call site — no Filter is ever constructed with a numeric cost override, so nothing is lost.
- **Edge cases hold.** `selectivity = 0` floors to 1 (matches empty-source min-1 convention); clamp to `[0,1]` guards a misbehaving provider; the covered-unique-key branch still forces `estimatedRows = 1`, overriding any stamped fraction (tested); the idempotence guard treats a stamped `0` as "already stamped" correctly (`0 !== undefined`).
- **No plan churn is expected and none occurred.** Structural-pass cost comparisons run before the Physical rule, so they still see the flat `0.5` — plan selection is unchanged. The stamp only affects the final physical `estimatedRows` of residual Filters. Consistent with the observed 0 golden-plan churn.
- **Docs (`docs/optimizer.md`) accurate** against the code as landed. `NOTE:`/tripwire citations to backlog `feat-conjunction-and-join-selectivity` resolve to a real ticket.

### Major — filed as a ticket

- **Access-node physical cardinality ignores catalog `rowCount` → this feature is invisible over full scans.** `TableReferenceNode.estimatedRows` (reference.ts:90) returns the *static* `tableSchema.estimatedRows`, not `tableSchema.statistics?.rowCount`, so after `ANALYZE` a full-scan source still reports `0` (memory vtab) and `floor(0 * sel) = 0`. The selectivity is computed correctly then discarded because the source is 0; only positive-cardinality sources (range seeks) show the effect. Pre-existing and broader than this ticket (it starves the whole cost model of base cardinality), so **not fixed inline** — filed as backlog `debt-access-node-catalog-cardinality` (confirmed the existing `feat-conjunction-and-join-selectivity` ticket does **not** cover it; that one is about combining selectivities, not seeding base rows). That ticket also carries the golden-plan churn this feature deferred.

### Tripwires (recorded in-code, not ticketed)

- `withChildren` carries a stamped `selectivity` through when the predicate is unchanged even if the source changes; the carried value was computed against the old source's table. Harmless today (Physical is the last rule-bearing pass over Filters). Already tagged with a `NOTE:` at the site (filter.ts) — reader is instructed to also drop selectivity on source change if a later pass ever re-sources a stamped Filter.
- Covered-key detection re-runs `extractConstraints` on every `computePhysical` re-mint. Pre-existing; already carries an in-code `NOTE:` pointing at ticket `3-planner-plan-time-mechanical` with a WeakMap-cache sketch if it ever shows hot.

### Conjunction / join gap — already parked

`a = 1 and b = 2` is not decomposed (the whole `AND` reaches the provider, finds no single column, falls back to naive `0.1`); filters over joins stay at default `0.5`. Crude but **not a regression** vs. the old flat `0.5`. Already parked in backlog `feat-conjunction-and-join-selectivity`, with a `NOTE:` in the rule pointing there. No action.

### Not covered by tests (accepted)

The pure logical-source getter path (Filter over a Values/Join source that exposes logical `estimatedRows`) has no dedicated assertion. The arithmetic is byte-identical to the `computePhysical` stamped branch, which **is** tested three ways, so the risk is negligible and stubbing a full `RelationalPlanNode` source for a getter-only test is not worth the fragility. Noted, no test added.
