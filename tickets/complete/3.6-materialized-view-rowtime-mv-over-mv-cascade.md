# MV-over-MV row-time cascade — completed (reviewed)

## What shipped

An MV whose single source resolves to another MV's backing table is **accepted** and
maintained transparently through a chain of arbitrary depth. A write to base `t`
maintains `mv1`'s backing; that backing write's *effective* per-row delta
(`BackingRowChange`) is then routed to every MV reading `mv1`'s backing, recursively —
synchronous, DAG-ordered, and atomic within the statement.

- **`vtab/memory/layer/manager.ts`** — `applyMaintenanceToLayer` returns
  `BackingRowChange[]` (the realized deltas: `delete-key`→`delete` w/ oldRow when a row
  existed; `upsert`→`update` if it replaced a row, else `insert`; absent-key delete →
  nothing). Free — the layer already looks up each op's before-image.
- **`core/database-materialized-views.ts`** — `applyMaintenancePlan` /
  `applyInverseProjection` return the `BackingRowChange[]`; `maintainRowTime(sourceBase,
  change, cache?, depth=0)`, after applying a plan, looks up `rowTimeBySource[backingBase]`
  and, when non-empty, recurses per backing change threading the per-statement
  `BackingConnectionCache` and `depth+1`. Leaf fast path
  (`!rowTimeBySource.has(backingBase)`) keeps a non-chained MV at today's cost. New
  `assertCascadeDepth` throws `INTERNAL` past `rowTime.size` (cycle backstop).
- **`buildMaintenancePlan`** — the `getMaterializedViewByBackingTable(...)` reject block
  is gone; the remaining eligibility checks evaluate against the keyed backing schema
  unchanged. `getMaterializedViewByBackingTable`'s doc comment in `schema/manager.ts` was
  updated (it remains live — `statement.ts` still uses it for `Database.watch`
  source-scope resolution; only its create-gate consumer was removed).

> (`incremental-maintenance-plan-abstraction`, `materialized-view-rowtime-per-statement-batching`,

### Correctness — no bugs found
- **DAG-ordered, no double-maintenance.** Every MV has exactly one source (single-source
  eligibility gate), so the dependency structure is an **in-forest** (each node one
  parent) — never a diamond. The cascade is a depth-first forest walk; no node is
  maintained twice for one source write. This is the load-bearing property behind
  "DAG-ordered + atomic" and it holds structurally.
- **Recursion safety.** The recursive `maintainRowTime(backingBase, …)` iterates a
  *different* `rowTimeBySource` set than the one being walked, and nothing in the
  maintenance path mutates `rowTimeBySource` (only register/unregister do), so there is no
  iterator-invalidation. Cache + depth are threaded correctly (`…, cache, depth + 1`).
- **Depth guard.** `depth > rowTime.size` is a sound over-bound: a valid chain of N MVs
  reaches max recursion depth N−1, and `rowTime.size ≥ N`, so the guard never fires for a
  legal forest and still terminates a (structurally-impossible) cycle. Defense-in-depth,
  correct.
- **Effective-delta routing & the two-op update.** A key-unchanged source UPDATE yields
  `[delete-key, upsert]`; `applyMaintenanceToLayer` applies them in order (the upsert sees
  the tombstone → emits `insert`), so the downstream MV receives `delete(old)` then
  `insert(new)`. Net-correct for passthrough and confirmed by the chain oracle. Within one
  `maintainRowTime` call ops apply synchronously in order; no external reader interleaves
  mid-cascade. No enforcement-timing concern.
- **Reads-own-writes / atomicity.** Cascade writes ride the same per-statement backing
  connection a select/enforcement scan resolves to (semantic-edge-B). A depth-≥2 backing
  connection registers lazily and `registerConnection` replays the active savepoint stack
  (incl. the statement-atomicity savepoint), so the whole chain reverts in lockstep. The
  cold enforcement path (`lookupCoveringConflicts`) correctly omits the cache and
  re-resolves the same connection deterministically.
- **Exhaustiveness / error handling.** `applyMaintenancePlan` switch is exhaustive with a
  `never` check; the unwired `full-rebuild`/`residual-recompute` arms throw loud
  `INTERNAL`; backing-not-found throws `INTERNAL`. Clean.

### Type safety / SPP / DRY
`BackingRowChange` is a discriminated union (narrow on `op`, no non-null asserts); the
inbound-source and cascade shapes are unified by design (DRY). Functions are small and
single-purpose. `typecheck` = 0, `lint` = 0 (observed).

### Findings disposition

**Minor — FIXED INLINE (added + verified).**
*Fan-out from a cascade backing* (one inner MV feeding **two** outer MVs) is handled by
the code — `maintainRowTime` loops over *every* key in `rowTimeBySource[backingBase]` — but
was not pinned by a dedicated test. `53` §6 exercises the identical multi-consumer loop at
the *base* level, and §21 exercises recursion+cache, so fan-out-at-a-cascade-backing was the
only untested corner of the cascade's breadth dimension. **Closed**: the case below was added
to `53` §8 (after the 3-level-chain block) and the file re-runs green (1 passing; full
targeted suite 24 passing, exit 0):

```sql
-- Fan-out at a cascade backing: one inner MV (`fo1`) feeds TWO outer MVs with
-- distinct backing keys. A single source write maintains the inner backing once,
-- whose effective per-row change is routed to BOTH dependents.
create table fo (id integer primary key, x integer);
create materialized view fo1 as select id, x from fo;
create materialized view fo2a as select id, x from fo1;
create materialized view fo2b as select x, id from fo1 order by x;
insert into fo values (1, 10), (2, 20);
select id, x from fo2a order by id;
→ [{"id":1,"x":10},{"id":2,"x":20}]
select x, id from fo2b order by x;
→ [{"x":10,"id":1},{"x":20,"id":2}]
update fo set x = 99 where id = 1;
select id, x from fo2a order by id;
→ [{"id":1,"x":99},{"id":2,"x":20}]
select x, id from fo2b order by x;
→ [{"x":20,"id":2},{"x":99,"id":1}]
delete from fo where id = 2;
select id, x from fo2a order by id;
→ [{"id":1,"x":99}]
select x, id from fo2b order by x;
→ [{"x":99,"id":1}]
```

**Minor — noted, no action (acceptable as-is):**
- *`rollback to savepoint` through a chain.* §13 covers savepoint replay for a single
  level and §8 covers full rollback for a chain; the intersection (a `rollback to
  savepoint` reverting a *deeper* cascade level) is not pinned. The savepoint-replay
  mechanism is shared and tested at depth 1, so this is low-value; left as-is.
- *Chained REPLACE / downstream `update` BackingRowChange.* In practice an in-scope
  key-unchanged update decomposes to `delete+insert` downstream, so the discriminated
  `update` arm is rarely the cascade input for the covering shape; it is handled
  identically and the direct (non-chained) REPLACE/upsert paths are tested in `53`
  §10–§12. No action.

**Deferred to other tickets (correctly out of scope) — agreed:**
- *Mixed-arm chains* (#5): only the `inverse-projection` arm is wired today; a chain
  cannot yet mix arms. The cascade is arm-agnostic (routes whatever delta a plan produces).
  Revisit once `-residual-recompute` / `-prefix-delete` land.
- *Create-time cycle/depth walk* (#4): skipped intentionally; cycles are structurally
  impossible (a consumer requires its producer to pre-exist; sources fixed at create) and
  the runtime `assertCascadeDepth` is the backstop. Reviewer concurs the runtime guard
  suffices.
- *Equivalence-harness fold* (#2): docs now reference
  `test/incremental/maintenance-equivalence.spec.ts` (a plan-abstraction deliverable). The
  standalone `materialized-view-cascade.spec.ts` chain oracle still lives separately; folding
  its chain assertions into the equivalence harness is a tidy-up, not a correctness gap.
- *Deeper enforcement via a downstream MV* (#8): the lens prover's concern
  (`lens-prover-and-attachment`), not this ticket; `54` §7 covers only the in-reach
  level-1 covering MV.
