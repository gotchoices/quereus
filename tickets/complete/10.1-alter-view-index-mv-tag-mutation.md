# ALTER VIEW / MATERIALIZED VIEW / INDEX ‚Ä¶ SET TAGS ‚Äî completed

Adds the in-place `SET TAGS` primitive for the three remaining tagged catalog
objects (views, materialized views, indexes), following the whole-set-replacement
model the prereq (`alter-table-tag-mutation`) established. Parser dispatch ‚Üí
generic `SetObjectTagsNode` ‚Üí `emitSetObjectTags` ‚Üí SchemaManager setter.
Reserved-tag validation at plan-build (`view-ddl` for view/MV, `physical-index`
for index). Declarative differ gains `viewTagsChanges` / `materializedViewTagsChanges`
/ `indexTagsChanges` buckets (name-match only) and `generateMigrationDDL` emits the
three `SET TAGS` forms; an MV body change drops+recreates while a tag-only change
takes the in-place path (mutually exclusive).

See the implement commit `ticket(implement): alter-view-index-mv-tag-mutation`
for the full design rationale.

## Review findings

**Process.** Read the full implement diff with fresh eyes before the handoff
summary, across every aspect angle (SPP, DRY, modularity, maintainability,
performance, resource cleanup, error handling, type safety) and every test angle
(happy path, edge, error, regression, interaction). Ran `typecheck` (clean),
`lint` (clean), and the full quereus suite (**4786 passing / 0 failing**, +1 from
a test added this pass). Several of the implementer's flagged gaps were probed
with throwaway scripts to confirm actual behavior rather than trust the prose.

### Major ‚Äî filed as new ticket(s)

- **Stale cached write-through plan after `ALTER VIEW ‚Ä¶ SET TAGS` (CONFIRMED bug).**
  ‚Üí `tickets/fix/view-tag-mutation-plan-invalidation.md`. A view's
  `quereus.update.default_for.<col>` tag is consumed at plan time; the view
  setters fire **no** schema-change event and views are **not** tracked as plan
  dependencies (the `SchemaDependency` union has no `view`), so an already-prepared
  write-through statement keeps the old default. Reproduced: after altering
  `default_for.created` from `100`‚Üí`200`, a re-executed cached `INSERT INTO v`
  still wrote `100`. Fixing it properly needs view-dependency tracking + a
  view/MV-modified event ‚Äî infrastructure, hence a separate fix ticket, not an
  inline patch. (This was the implementer's flagged gap #1; now grounded with a
  repro.)

### Minor ‚Äî fixed inline this pass

- **Default schema resolution inconsistent for the MV path.** The builder
  hardcoded `schemaName = name.schema || 'main'`. `CREATE`/`DROP MATERIALIZED VIEW`
  resolve the default via `getCurrentSchemaName()`, so under a switched current
  schema an unqualified `ALTER MATERIALIZED VIEW ‚Ä¶ SET TAGS` would look in `main`
  and throw NOTFOUND while the MV lives in the current schema. Changed all three
  builders to `name.schema || ctx.schemaManager.getCurrentSchemaName()`
  (behavior-preserving ‚Äî defaults to `main`; now consistent with MV create/drop).
  Note: a full non-`main` end-to-end is still blocked by a *separate, pre-existing*
  table search-path limitation (table resolution uses `main` even when the current
  schema is switched) ‚Äî out of scope here.
- **Docs oversold updatable-view tag routing.** `docs/sql.md` ¬ß2.7 implied
  `SET TAGS` reliably re-routes write-through; added an explicit "known limitation"
  caveat about cached prepared statements, pointing at the new fix ticket.
- **Untested exposed-implicit-index addressability (implementer's gap #4).**
  Verified by probe that the implicit covering index IS present in
  `tableSchema.indexes` even when hidden ‚Äî so the existing hidden‚ÜíNOTFOUND test
  *does* exercise the `isHiddenImplicitIndex` guard (not the no-match path); the
  guard is load-bearing, not dead code. Added a regression test confirming an
  **exposed** implicit covering index (`quereus.expose_implicit_index = true`) is
  addressable by `setIndexTags` ‚Äî exercises the guard's `=== false` branch for an
  implicit (not ordinary) index.

### Scope extension ‚Äî folded into an existing backlog ticket

- **Store-backed persistence (implementer's gap #2).** Extended
  `tickets/backlog/tag-mutation-store-persistence.md` to cover view/MV/index, with
  the key wrinkle for the change-event-subscription option: `setIndexTags` fires
  `table_modified` (coverable by a `table_modified` listener) but `setViewTags` /
  `setMaterializedViewTags` fire **no event at all**, so a store fix must also
  observe a view/MV change. `yarn test:store` was not run this pass (memory-backed
  suite only).

### Verified correct ‚Äî no action

- Reserved-tag validation mirrors the prereq's `buildAlterTableStmt` exactly (same
  registry, sites, and no-op warning logger).
- `freezeTags` whole-set/clear semantics (`{}` ‚Üí `undefined`) consistent across all
  setters; re-registration via `addView`/`addMaterializedView`/`addTable` replaces
  by lowercased key.
- Differ pure-name-match vs rename-match distinction is correct (maps keyed by

- Index resolution is schema-unique; `setIndexTags` fires `table_modified` (matches

- Docs (`docs/sql.md`, `docs/schema.md`) read accurately against the code.

- Plain-view body drift undetected by the differ (gap #5) ‚Äî confirmed pre-existing;
