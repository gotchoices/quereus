# Complete: memory vtab resolves collations against its own database

## What a collation is, and what was wrong

A *collation* is a named text-sorting rule. `BINARY`, `NOCASE`, and `RTRIM` are built in;
an application can register more with `db.registerCollation(name, comparator)`, and can
replace `NOCASE`/`RTRIM` with its own comparator (but not `BINARY`).

Everything under `src/vtab/memory/` used to turn a collation *name* into a comparison
function by calling the process-global `resolveCollation()` in `util/comparison.ts`. That
global registry only ever holds the three built-ins, and it silently fell back to `BINARY`
on a miss. So a memory table's primary key, secondary indexes, index range seeks,
plan-filter bound checks, and `UNIQUE` enforcement all ignored whatever the connection had
registered, and byte-ordered instead.

`3.2-collation-resolver-seam` added `db.getCollationResolver()` — a `(name) => comparator`
that reads the live per-database registry and *throws* `no such collation sequence: X` on a
miss. This ticket threaded it through the memory vtab.

## What shipped

The resolver rides on the objects that already reach every comparison site. `ScanPlan` was
left alone — it still carries collation *names*, and is logged and compared as plain data.

- `Layer` gained `readonly collationResolver: CollationResolver`. `BaseLayer` takes it as a
  constructor argument; `TransactionLayer` copies it from its parent, which is what keeps a
  child layer's `compareKeys` built from the same function that ordered the parent's
  secondary BTree nodes it inherits.
- `MemoryIndex` takes a `collationResolver` as its third constructor argument.
- `createPrimaryKeyFunctions(schema, collationResolver)` takes it as a second argument,
  deliberately not defaulted — a default would re-open the hole.
- `MemoryTableManager` binds `db.getCollationResolver()` once in its constructor.
- `scanLayer(layer, plan)` resolves the plan's collation names once, then delegates to a
  private `scanLayerResolved(layer, plan, collations)` that passes the resolved functions
  down its own recursion.
- `plan-filter.ts` exports `resolveScanCollations(plan, resolver)` → `ResolvedScanCollations`,
  and `planAppliesToKey` takes it as a fourth argument. Per-row
  `compareSqlValues(a, b, name)` calls in `plan-filter.ts` and `scan-layer.ts` became
  `compareSqlValuesFast(a, b, fn)`.
- Three `compareSqlValues(a, b, name)` calls in `manager.ts`'s `UNIQUE` enforcement
  (`checkUniqueViaIndex`, `checkUniqueViaMaterializedView`, `checkUniqueByScanning`) were
  converted too — they had the same defect. Each resolves its per-column collations once,
  above the candidate loop.

`compareSqlValues(a, b)` with no collation argument (BINARY) is untouched. `src/vtab/`
now imports nothing from the deprecated global registry, which is what
`3.6-retire-global-collation-registry` needs.

## Behavior changes an embedder can observe

1. **A replaced `NOCASE`/`RTRIM` now governs a memory table's key order and `UNIQUE`
   collisions.** Previously the query layer honored the override while the storage layer
   did not — the two disagreed.
2. **An unregistered collation now raises at table/index construction** instead of silently
   byte-ordering. `create table t (k integer collate frobnicate primary key)` used to
   succeed; it now fails with `no such collation sequence: FROBNICATE`. For a TEXT column
   the DDL-level `Unknown collation '…' for type 'TEXT'` still fires first.

Collation registration remains non-retroactive — comparators are resolved when a structure
is built, so a collation re-registered after a table exists does not reorder it. That
contract is documented at `Database.registerCollation`.

## The headline test the implement ticket could not write

`create table t (k text collate REVERSE primary key)` is rejected today by
`validateCollationForType` (`schema/table.ts`), which checks a column's `COLLATE` against a
static list on the logical type, never against the connection's registry. Two shapes do
work and the suite uses both: overriding a built-in (`db.registerCollation('NOCASE', …)`,
which reaches the primary-key comparator), and a custom name on an *index* column
(`create index ix on t (v collate REVERSE)`, which reaches `MemoryIndex`).
`backlog/feat-ddl-accepts-registered-collations` tracks the gap; it is a pre-existing
schema-layer limitation, not this ticket's leftover.

## Review findings

Everything below was checked against the implement-stage diff (`95cf732e`) read first,
before the handoff summary.

### Verified correct — the three claims the handoff asked review to confirm

- **`checkUniqueViaIndex`'s new `if (existingPKs.length === 0) return null` cannot skip a
  conflict.** `existingPKs` is the index bucket for the new row's key, looked up under the
  index's own collation-aware comparator; an empty bucket means the old code's `for` loop
  ran zero iterations and fell through to the same `return null`. Behavior identical, minus
  the collation resolves.
- **`scanLayerResolved`'s recursion cannot receive a narrowed plan whose collations
  differ.** The multi-seek branch sets `equalityKey`, which short-circuits `planAppliesToKey`
  before any prefix or bound compare. The multi-range branch narrows only `lowerBound` /
  `upperBound`, and `buildScanPlanFromFilterInfo` assigns every range in a multi-range plan
  the *same* leading-index-column collation (`scan-plan.ts`, "Every range bounds the leading
  index column, so they share its collation"). Nothing re-resolves and nothing needs to.
- **`uniqueEnforcementCollations`'s `undefined` slot.** Memory now maps `undefined →
  'BINARY'`. `quereus-store/store-table.ts` and `quereus-isolation/isolated-table.ts` pass
  the possibly-`undefined` name straight into `compareSqlValues(a, b, name)`, whose default
  parameter is `'BINARY'`. Same comparator, so the three agree.
  `checkUniqueViaIndex`'s `schema.columns[col].collation ?? 'BINARY'` is the same
  substitution and equally sound — the old code passed `undefined` into the same default.

### Found and fixed in this pass

- **The UNIQUE-override test did not discriminate.** `enforces UNIQUE under the overriding
  database NOCASE comparator` used `REVERSE_NOCASE`, which still folds case, so it passed
  whichever comparator the storage layer picked — it could not have caught the original
  bug. Added `lets an overriding NOCASE that does not fold case admit a case variant`,
  which registers a case-*sensitive* `NOCASE` and asserts `'ABC'` inserts alongside
  `'abc'` (the global registry's `NOCASE` raised `UNIQUE`), and still collides on an exact
  duplicate.
- **No `RTRIM` override coverage** (the handoff flagged this). Added
  `enforces UNIQUE under an overriding RTRIM comparator`: the database's `RTRIM` ignores
  *leading* spaces, so `'x  '` inserts and `'  x'` collides — the exact inverse of the
  built-in.
- **No composite-primary-key coverage** (the handoff flagged this). The single-column and
  composite PK comparators are built by different functions and only the composite one
  resolves a collation for a non-leading column. Added
  `applies the overridden collation to a trailing key column`.
- **`TransactionLayer` rebuilt its primary-key functions on every call**, and its
  constructor built them twice (once via `getPkExtractorsAndComparators`, once in
  `initializeSecondaryIndexes`). Each build resolves every PK column's collation. Now built
  once into a `private readonly pkFunctions`, valid because `tableSchemaAtCreation` never
  changes for a layer.
- **`scanLayer`'s multi-seek branch built a second, throwaway `createPrimaryKeyFunctions`
  purely to obtain the PK encoder**, right beside `layer.getPkExtractorsAndComparators()`
  which had already built the same thing (the handoff called this out as worth a look).
  The layer method's return type is now the named `PkExtractorsAndComparators` and carries
  `primaryKeyEncoder`; the duplicate build is gone. `IN`-list seeks no longer pay for it.

### Found, filed as a ticket — `backlog/bug-rename-column-drops-partial-index`

Investigating the handoff's `BaseLayer.createSecondaryIndexes` swallow (it catches and logs
any `MemoryIndex` construction error) turned up something worse than the unreachable
collation case the handoff reasoned about. The swallow is **load-bearing today**, and what
it hides is a live defect:

`ALTER TABLE … RENAME COLUMN` runs the memory module's rebuild *before*
`propagateColumnRename` rewrites partial-index predicate syntax trees. So a partial index
whose `WHERE` names the renamed column makes `compilePredicate` raise
`unknown column '<old name>'` during the rebuild — the catch drops that index from the map
that then replaces the live one, and nothing ever rebuilds it. Verified directly: the base
layer's secondary-index map is `['ix']` before the statement and `[]` after, while the
catalog still reports the index with a correctly rewritten predicate.

I removed the catch, saw `RENAME COLUMN rewrites an unqualified predicate reference`
(`test/index-ddl-roundtrip.spec.ts`) fail, and restored it — the fix is the rename ordering,
not the catch, and that is a change to `ALTER` semantics across every vtab module, not an
inline review edit. The catch now carries a comment naming the ticket and explaining that
it is not a design choice. The handoff's original concern (an unregistered collation landing
in the same silent-drop hole) is recorded there too; it remains unreachable, since an index
naming one could never have been created and the per-database registry has no
`unregisterCollation`.

### Tripwires — noticed, parked in place, not filed

- `resolveScanCollations` runs once per `scanLayer` call. The implementer left a `NOTE:` at
  its definition in `plan-filter.ts`: if very many tiny scans ever make per-scan setup
  visible, memoize on the `ScanPlan` via a `WeakMap`. Confirmed the note is there and
  accurate; a BINARY-only plan costs one empty array and one branch.
- `store-table.ts` re-derives `uniqueEnforcementCollations` per insert and already carries
  its own `NOTE:` to that effect. Out of this diff, left alone.

### Checked, nothing found

- **Docs.** `docs/sql.md`'s COLLATE section is accurate as written — I re-read it against
  the code. `docs/plugins.md` and `docs/runtime.md` still show the deprecated global
  `resolveCollation` in example code; that is `3.6-retire-global-collation-registry`'s
  subject (it retires the symbol), not a stale statement about memory tables. No other doc
  describes memory-table collation resolution.
- **Other `Layer` implementors.** Only `BaseLayer` and `TransactionLayer` implement it, so
  the new required `collationResolver` member breaks nothing outside this package.
- **Resolver liveness.** `getCollationResolver()` returns a memoized closure that reads the
  registry on each call, so binding it once in `MemoryTableManager`'s constructor does not
  freeze the registry — only the comparator each structure captured at build time.
- **`compareSqlValuesFast` vs `compareSqlValues`.** The latter is literally the former with
  a name→function step; NULL and storage-class ordering are shared. Every converted call
  site is semantically identical.
- **Backlog scoping question** (the handoff asked whether the too-loose `INTEGER`/`REAL`/`BLOB`
  half of `feat-ddl-accepts-registered-collations` should split into a `bug-`). Left as one
  ticket: both halves are the same missing registry consult in the same function
  (`validateCollationForType`), and two tickets would collide on the same lines.
