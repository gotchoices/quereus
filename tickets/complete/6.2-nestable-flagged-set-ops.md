## What landed (implement)

Made a nested (subtree) set-op operand recursively writable for the unambiguous fan-out
operations, and made the ambiguous inserts into a subtree reject cleanly. See the implement
commit (`ticket(implement): nestable-flagged-set-ops`, `04f59054`) for the full mechanism:

- `analyzeSetOpView` data-col count fixed to the recursive `SetOperationNode.dataColumnCount()`
  (`findSetOpNode` descends the spine); `surfacedInnerFlagNames` added.
- `SetOpBranch.isNested`; `analyzeSetOpBranches` (membership-free subtree analysis reusing the
  OUTER `dataColCount`); recursive `fanBranchDataUpdate` / `fanBranchDelete` threading the OUTER
  `analysis` so the single up-front capture is reused at depth (Halloween-safe).
- Rejections (`set-op-membership-nested`): `set <subtreeFlag> = true`, surfaced-inner-flag
  writes, insert-through into a subtree side.
- `schema.ts`: nested union body reports `is_updatable`/`is_deletable` = YES,
  `is_insertable_into` = NO; surfaced inner flags `is_updatable` = NO.

## Review findings

Adversarial pass over the implement diff (set-op.ts, schema.ts, property.spec.ts, docs) and the
files it touches / should touch (propagate.ts, set-operation-node.ts).

### MAJOR — `except` / `intersect` subtree fan-out silently corrupts base rows (FOUND + CONTAINED)

The fan-out recursion identifies a leaf's affected rows as "leaf rows whose data tuple ∈ the
up-front capture". Soundness rests on **a leaf's rows ⊆ the subtree's rows** — true for
`union` / `union all`, **false for `except` / `intersect`** (a leaf can hold a row the subtree
excludes). When an OUTER operand makes such an excluded row visible, it enters the capture and
the recursion deletes / mutates it in the inner leaves **even though its `inSub` membership
probe reads false** — silent base-table corruption. The implement diff's own doc comment and
`docs/view-updateability.md` both asserted the false invariant ("a leaf's rows ⊆ the subtree's";
"`intersect` / `except` subtree operands recurse the same way"), and the implement-stage `except`
test masked the bug (its `A = {1,2}` never overlapped the `B∩C` row `id=3`).

**Verified failing** (delete AND data-update, both `except` and `intersect`) with a row in
`A ∩ B ∩ C` made visible by the outer union — see the repro in the filed fix ticket.

**Disposition (contained inline; full fix filed):**
- `set-op.ts` — new `rejectNonUnionSubtreeFanout()` called at the top of `fanBranchDataUpdate` /
  `fanBranchDelete`; rejects an `except` / `intersect` subtree operand with a clean
  `set-op-membership-nested-except` diagnostic. Union / union all fan-out (the sound case +
  the bulk of the feature) is unchanged.
- `set-op.ts` — `isOperandWritable` now gates a compound operand on `isUnionLikeSubtree`, so the
  static `view_info` / `column_info` surfaces report the conservative all-`NO` shape for a view
  with an `except` / `intersect` subtree operand (agreeing with the dynamic reject; an honest
  under-claim — the leaf-flag writes that still work are not separately surfaced, consistent
  with the existing coarse whole-view gate).
- `property.spec.ts` — the implement-stage `except` test (which asserted the buggy success) is
  replaced by "except / intersect subtree fan-out is deferred", asserting: the reject, **base-row
  integrity** (B/C untouched), and static-surface agreement (view non-writable).
- `docs/view-updateability.md` — the false invariant removed; the deferral documented in all
  four places that described nested fan-out.
- **Filed `tickets/fix/set-op-membership-nested-except.md`** — implement correct membership-gated
  fan-out (thread the subtree-membership gating predicate down the recursion), then lift the
  guard and re-enable the static surfaces. Includes the repro, the design questions
  (predicate provenance, flag-less subtrees, `except` right-operand semantics), and the tests to
  add.

### MINOR — depth-3 nesting untested (FIXED inline)

The implement handoff flagged 3+ level nesting as structurally handled but tested only to 2
levels. Added "three-level union nesting: data + delete fan out to the deepest leaf"
(`A union (B union (C union D))`), exercising fan-out to the deepest leaf D and a row resident in
both A and D. Passes — the recursion descends past two levels correctly.

### Checked — no finding

- **Halloween safety at depth.** The recursion threads the OUTER `analysis` and reuses the ONE
  capture; no second capture is built. `buildMemberExists`'s `k.*` (outer data cols) vs
  `branch.*` (leaf data cols) positional alignment is guaranteed by `buildBranch`'s arity check
  (`dataColNames.length === dataColCount`) and the data-only `SetOperationNode` arity invariant.
  Sound for union (and now correctly fenced for except/intersect).
- **`dataColumnCount()` / `findSetOpNode`.** Recursive data arity is correct; `findSetOpNode`
  descends `getRelations()` and finds the outer set-op (root or under an outer-ORDER-BY
  `SortNode`). The `attrs.length - flags.length` over-count is genuinely fixed.
- **`propagate.ts` backstop (244–252).** Confirmed still a defensive backstop only — the supported
  fan-out paths recurse to LEAF branches before calling `propagate`, so no nested membership body
  reaches it. Left unchanged.
- **`set-op-membership-nested` rejects** (`set <subtreeFlag> = true`, surfaced-inner-flag write,
  insert-through into a subtree side, conflicting `set x, inSub=false`) — all verified clean and
  correctly named (NOT `unknown-view-column`, NOT the phase-1 message).
- **Key Soundness** under nested write — no flag enters a claimed key (existing test holds).
- **Binary (non-nested) set-op writes** — unaffected by the containment (`isOperandWritable`'s
  new gate fires only for compound operands; top-level binary `except` / `intersect` views stay
  writable, leaf operands unchanged).
