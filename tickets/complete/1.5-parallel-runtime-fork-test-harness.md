## Summary

Landed pure-test infrastructure plus a tiny env-gated production wrapper at the five `RuntimeContext` construction sites. The harness consists of:

- A pinned `EXPECTED_FORK_POLICY` table (compile-checked via `satisfies Record<keyof RuntimeContext, ForkPolicy>`) so adding a `RuntimeContext` field without declaring fork semantics is a TypeScript error.
- A static allowlist of files permitted to call `<rctx>.tableContexts.set/delete/clear` or `<rctx>.context.set/delete/clear` — current allowlists: `src/runtime/emit/recursive-cte.ts` for table contexts; `context-helpers.ts`, `emit/aggregate.ts`, `emit/hash-aggregate.ts`, `emit/window.ts` for row contexts.
- An opt-in `QUEREUS_FORK_STRICT=1` mode (Node-only, browser/RN/edge silently no-op via `typeof process !== 'undefined'` guard) that wraps every parent and fork map and throws `strict-fork: parent context mutated …` when a parent map is mutated while one of its forks is being driven. State is tracked per-parent-map (via two `WeakMap`s) so concurrent unrelated drivers do not interfere and forks freely mutate their own maps.
- `yarn test:fork-strict` script and `--fork-strict` flag in `test-runner.mjs`.
- A `### Parallel runtime fork contract` subsection in `docs/runtime.md` with the policy table and contract rules.

- Inspected the `Proxy` trap in `wrapTableContextsStrict`: uses `Reflect.get(target, prop, target)` to invoke `Map.prototype` getters/methods with the real Map as receiver (avoids the Proxy brand-check failure that would otherwise break `.size` and iteration). Non-mutating methods are `.bind(target)`-ed to keep `this` correct after iteration. `clear` is also guarded alongside `set`/`delete`.
- Verified `WeakMap` bookkeeping: `stateByWrappedMap` keys the per-map `ForkState`; `parentStateOfFork` carries the back-reference from each child map to its parent's state. `drive()` reads `forks[0]`'s back-reference and bumps the parent counter once per drive; the `finally` block decrements. Bump happens **after** the pre-aborted-signal short-circuit and the empty-factories return, so no counter leak on either short-circuit path (the `try` block isn't entered).
- Verified sub-fork semantics: a fork's own `StrictRowContextMap` carries its own `ForkState`, so a fork freely mutating its own maps does not trip the guard, but if that fork is itself sub-forked, the same guard activates relative to it.

### Findings — fixed inline

1. **`clear()` was missing from the static allowlist regex.** The strict-fork mode guard already checked `set`/`delete`/`clear`, but the static allowlist tests only matched `set`/`delete`. A new `tableContexts.clear()` or `<rctx>.context.clear()` site would have slipped past the cheap static tripwire. Tightened both regexes to include `clear`; renamed the test titles to reflect the wider coverage. Behavior unchanged on the current codebase (no `clear()` sites exist).

2. **`shared fields are aliased to parent` was trivially satisfied for undefined fields.** The test asserts `fork[k] === parent[k]` for every non-forked field, but `makeRuntimeContext()` leaves `db`, `stmt`, `tracer`, `activeConnection`, `contextTracker`, `planStack` at their default `undefined`/missing values — `undefined === undefined` passes vacuously and would have masked a future `fork()` bug that, say, dropped one of those fields. Strengthened by populating every shared field with a distinct non-undefined sentinel before forking, and added a guard assertion that the sentinel itself is non-undefined.

3. **Receiver-name regex convention was undocumented.** The row-context regex hardcodes `rctx|ctx|runtimeCtx|runtimeContext` as the receiver names; if a future contributor introduces a different identifier they could bypass the static check. The codebase is consistent today, but the convention was implicit. Added a code comment in the test calling out the convention and the action a future contributor must take (extend the alternation or rename).

### Findings — deferred (not in this pass)

4. **The pre-existing `packages/quereus/test/runtime/parallel-driver.spec.ts` constructs its `makeRuntimeContext()` with plain `new RowContextMap()` / `new Map()`.** Under `QUEREUS_FORK_STRICT=1` those parents are not wrapped, so the harness silently degrades to no-op for that spec's runs (parent is not in `stateByWrappedMap`, so `markForkOf` writes nothing, and `bumpParentForkCounter` returns `null` in `drive()`). The spec's existing assertions still pass under strict mode (verified — 12 passing), but they do not actually exercise the parent-immutability guard. **Disposition:** acceptable for now — the spec's purpose is primitive correctness, not contract enforcement, and the fork-contract spec covers strict-mode behavior end-to-end. If a future regression slips through, route this spec's factory through `createStrictRowContextMap` / `wrapTableContextsStrict` too.

5. **No sub-fork (fork-of-a-fork) tests.** The per-parent-map `ForkState` design supports them, and the manual trace through `fork()` + `markForkOf` confirms the bookkeeping cascades correctly, but there is no regression-test coverage. **Disposition:** defer until a consumer (#2 or later in the parallel-* track) actually exercises sub-forks — adding a synthetic test now is speculative and would over-fit the current internal API.

6. **DRY at the five construction sites.** Each site spells out `context: createStrictRowContextMap(), tableContexts: wrapTableContextsStrict(new Map())`. A helper returning both as a `{ context, tableContexts }` pair would shave a handful of lines. **Disposition:** declined — the current form is grep-able and readable, and the construction-site count is small.

### Categories explicitly checked and empty

- **Resource cleanup:** the strict-fork bump is paired with a drop inside `drive()`'s `finally` block; the bump itself is positioned after both early-return short-circuits so the counter cannot leak. No `WeakMap` is keyed by anything that outlives the wrapped maps (and `WeakMap` lifecycle is GC-driven anyway). Nothing to fix.
- **Error handling:** the `violation()` helper throws a `QuereusError` with `StatusCode.INTERNAL` and a pointer to `docs/runtime.md` § Parallel runtime fork contract; the message includes the parent name (`tableContexts` vs `context (RowContextMap)`) and the live-fork count. Both deliberate-violation tests assert the error message matches `/strict-fork/i`. Nothing to fix.
- **Cross-platform:** the `process.env` read is wrapped in `typeof process !== 'undefined'`, so the file remains browser/RN/edge-worker safe; in those environments the helpers return their inputs unmodified. Verified by reading the guard; the existing `env-compat.spec.ts` audit is the codebase-wide tripwire for cross-platform regressions.
- **Type safety:** no `any` introduced; the `satisfies` clause is the only meaningful type-level guarantee and was inspected. `as unknown as RuntimeContext[…]` casts in the test for shared-field sentinels are intentional — they only exist to construct opaque test fixtures.
- **Performance:** strict mode is the only added cost, and it is fully gated behind an env flag. Non-strict mode collapses every helper to either the input unchanged (for `wrapTableContextsStrict`) or a vanilla `new RowContextMap()` (for `createStrictRowContextMap`). The two `WeakMap` lookups inside `drive()` execute only when the parent was constructed via the strict factories.

## Riskiness

Low. The production-side hook is a single `typeof process !== 'undefined'`-guarded `process.env` read plus two factory-function calls at each of five construction points; non-strict mode is a pure pass-through. The full test suite is unchanged in non-strict mode (3313 → 3313 passing).
