## Review findings

### Verified against the code

- **The five-rejection count (MV-006).** Enumerated all eleven `cannotMaterialize()` throw sites.
  Eight are determinism rejects raised from different arms — one rejection *class*, not eight. The
  five distinct classes are exactly the ones MV-006 lists. The implementer's correction (docs were
  stale, code is right) is correct, and the sourceless reject is sound and guarded
  (`53.3-materialized-view-constraint-only-ddl.sqllogic:358,377`). MV-006 stands as written; no
  `fix/` ticket.
- **MV-003 + MV-004 together.** Confirmed both halves. `lookupCoveringConflicts`
  (`database-materialized-views.ts:976`) returns `[]` unless `plan.kind === 'inverse-projection'`,
  and `findRowTimeCoveringStructure` additionally declines any plan whose `chosenStrategy` is
  `'full-rebuild'` (and any stale MV). So a deferred full-rebuild backing is genuinely never read
  mid-statement, and deferring the floor cannot weaken per-row enforcement visibility.
- **MV-007.** `selectMaintenanceStrategy` returns `'full-rebuild'` on an empty sound set and an
  argmin otherwise. Stronger than the entry claims: every arm builder passes its **own singleton**
  sound set and throws `INTERNAL` if the gate answers with anything else, so the gate cannot return
  a strategy the arm declined. Confirmed.
- **MV-011.** Both consumers — `deriveBackingShapeUnguarded` and `buildFullRebuildPlan` — call
  `deriveCoarsenedBackingKey(root)` on the fully-optimized body root, and the function is
  deterministic over a given plan. The "agree by construction" argument holds at both call sites.
- **MV-024 (the entry the handoff called weakest).** The five-part adopt gate is **not** in
  `packages/quereus-store` as the handoff assumed — it is `tryAdoptPreExistingBacking` in
  `packages/quereus/src/schema/manager.ts:3069`. Read it. Four gates match the entry; the fifth
  (same-module sourcing) carries an extra condition the entry omitted: a maintained source must
  itself have been *adopted*, not refilled, this session. Fixed (below).

### Fixed in this pass (minor)

- **MV-024** — added `code:` pointer to `schema/manager.ts` — `tryAdoptPreExistingBacking` (the
  actual gate; `backingShapeMatches` is only gate 2), and stated the adopted-upstream condition.
  Body re-trimmed to fit the 120-word budget. An adopt-path guard does exist
  (`packages/quereus-store/test/mv-rehydrate-adopt.spec.ts`) but the register allows one `guard:`,
  so the refill-by-default guard was kept — refill is the safety half.
- **MV-019** — added the store-side `code:` pointer
  (`quereus-store/src/common/store-table.ts` — `enforceSecondaryUniqueForMaintenance`), which the
  handoff had not opened. It exists and mirrors the memory host. The guard is still a `.sqllogic`
  that runs against memory by default; `yarn test:store` covers the store path and was not run.
- **MV-006 ordinal collision** — MV-006 said "Adding a sixth is a breaking change" while
  `mv-maintenance.md` says the replicable gate *is* "a sixth, host-conditional reject". Two docs
  numbering the same thing differently. MV-006 now reads "adding an unconditional sixth", and names
  the host-conditional gates plural (there are two: non-replicable function, non-replicable
  collation — `findNonReplicableFunction` / `findNonReplicableCollation`).
- **`scripts/check-docs.mjs`** — the implementer parked a tripwire NOTE about area preambles being
  charged to the previous invariant's word budget. That was fixable, not conditional: an invariant
  block now ends at the next heading of **any** level, so `## <Area>` headings and their preambles
  are charged to nobody. Word counts can only shrink under this change, so it cannot mask an
  over-budget block; `docs:check` re-run confirms. The NOTE is gone, replaced by a statement of the
  boundary rule.
- **`planner/cost/index.ts`** — `MaintenanceStrategy`'s doc-comment claimed it "mirrors
  `MaintenancePlan['kind']`". It does not: `'prefix-delete'` and `'join-residual'` are plan kinds
  with no strategy of their own (they record `chosenStrategy: 'residual-recompute'`). Comment
  corrected and pointed at MV-007.

### Major — none filed

Nothing in the diff or the code it describes is wrong in a way that needs its own ticket. The one
candidate the handoff nominated for second-guessing — "is the sourceless-body reject itself the
bug?" — resolves in the code's favour: nothing can ever dirty a sourceless plan, so nothing could
maintain it, and rejecting at create is the only honest answer. MV-006 is right; no `fix/` ticket.
