# Identity-preserving reshape — completed

Part 4 of the maintained-table chain. A source `alter` that shifts a derivation's
output shape now reconciles the maintained table **in place** at `refresh`
(`reshapeBacking` → classify the column delta → ordered module `alterTable` ops →
data reconcile → one `table_modified`), preserving the table incarnation. Deltas
that can't be expressed in place (interleaving reorder, physical-PK definition
change, or a host module without `alterTable`) raise a **sited error** with the
table and rows untouched. The former drop+recreate `rebuildBackingTable` is
deleted.

The design as built is accurately described in the implement-stage handoff (now in
this file's history) and in `docs/materialized-views.md` (REFRESH § shifted-shape
+ backing-host incarnation note + Current-limitations refresh leg). The review did
not revise the design; it verified it and filed one correctness gap.

### Major (filed) — `fix/maintained-table-reshape-narrowing-attr-on-stale-data`

**Survivor attribute shifts (retype / recollate) are applied to the live
PRE-reconcile rows, not the reconciled body rows.** `reshapeBackingInPlace` runs
all `plan.ops` (including `alterColumn setDataType`/`setCollation`) **before**
`rebuildBacking` discards the old data. Memory's `alterColumn setDataType` iterates
and `validateAndParse`s every existing value (throws `MISMATCH`), and
`setCollation` does a strict secondary-index rebuild (throws `CONSTRAINT` on a
stale unique collision). So a narrowing that the freshly re-derived body satisfies
but the stale backing does not turns an *expressible* reshape into an error — and
because the offending data is re-encountered on every attempt, **a re-run does not
self-converge** (unlike the deferred NOT NULL tighten, which the implementer
correctly moved past the reconcile for exactly this reason). Low-likelihood
(passthrough type/collation narrowing on a non-empty backing) but a real
wrong-behaviour with a durable-store dimension. Fix is a design choice (defer
narrowing attr shifts like NOT NULL, or clear-then-reshape) so it warrants a ticket
rather than an inline patch. The ticket also folds in the handoff-flagged untested
paths it shares a fix surface with: **PK-column type-change re-key**, **store
reshape narrowing / in-transaction reshape**, and **mid-sequence-failure
convergence** (the recovery leg the 6.5 suite documents but only exercises via the
NOT NULL second-refresh fast path).

### Minor (no inline fix; documented)

- **Multi-op single-refresh end-to-end coverage.** drop + rename + add in one
  refresh is classifier-reasoned and indirectly exercised by the single-op tests,
  but has no single end-to-end assertion. Constructing one that genuinely forces a
  mid-sequence *drop* (vs the two-pointer collapsing drop+add into a positional
  rename) and that isn't pre-empted by source-column rename propagation is fragile;
  the value did not justify a brittle test. Folded into the fix ticket's test list
  rather than added here, to avoid a misleading green.

- **`describeBackingShapeMismatch` (positional) vs `classifyBackingReshape`
  (alignment) divergence** on a pathological multi-shift like `[a,b]→[b,c]`
  (positional → "pure name shift"; alignment → drop+add). Confirmed theoretical:
  the positional form is reached only by single-rename propagation, which never
  produces a multi-shift. The implementer's reasoning holds; no action.

### Empty categories

- **No security surface** — the change is engine-internal schema reconciliation; no
  new external input, auth, or serialization path.
- **No performance regression** — the in-place reshape replaces a strictly heavier
  drop+recreate; the fast path (shape unchanged) is untouched and still preserves
  `TableSchema` identity + warm caches.
- **No resource-cleanup leak** — no new host/connection allocation; the incarnation
  (and its `BackingHost`) is preserved rather than re-created.
