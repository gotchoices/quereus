# Complete: unified maintained-table schema model

Part 1 of the maintained-table chain. A materialized view is now ONE
`TableSchema` registered under the MV's own name, carrying
`derivation?: TableDerivation` (`schema/derivation.ts`). The `_mv_` prefix,
`Schema.materializedViews`, `MaterializedViewSchema`, `backingTableNameFor`,
and the tri-directional name disjointness are retired. All lifecycle verbs are
catalog flips on one record; canonical DDL renders on demand
(`generateMaterializedViewDDL(table)`); module identity lives on the table's
own `vtabModuleName`/`vtabArgs`. Behavior parity throughout, plus the
deliberate deltas listed in the implement handoff (schema() type
`materialized_view`, DROP TABLE drops the whole record, structural-ALTER
reject, self-reference reject, rehydrate `pendingDerivations` ordering gate).

### Found and fixed in this pass (minor → inline)

1. **Fail-unsafe DML dispatch through the schema path** (the handoff's
   "highest-risk seam", confirmed as a real corruption path). The three DML
   builders' maintained-table dispatch uses `stmt.table.schema ?? null`
   (current-schema default) while `buildTableReference` resolves through the
   schema path. Pre-unification a missed dispatch ended in "no such table"
   (fail-safe — the MV name was never a table); post-unification it resolves to
   a **direct write against the maintained table**. Reproduced:
   `create materialized view temp.mv …; pragma schema_path = 'main,temp';
   insert into mv values (…)` wrote the maintained table directly (row in mv,
   absent from source), silently diverging derived contents. **Fix**: backstop
   in insert/update/delete on the *resolved* `tableReference.tableSchema` —
   `isMaintainedTable` → route through the same `buildViewMutation` rewrite;
   plan-resolved `baseTable` check added to the single-source nested-MV reject;
   select-side mirror so a path-resolved maintained table still hits the stale
   re-validation guard. New regression spec
   `test/mv-cross-schema-dispatch.spec.ts` (insert/update/delete write-through
   + stale guard, 4 tests).
2. **`ALTER TABLE … SET/DROP TAGS` on a maintained table created
   non-persisting state**: the TABLE verb fires `table_modified`, not
   `materialized_view_modified`, so the tag edit never reached the persisted
   `create materialized view` catalog entry (memory-hosted: not persisted at
   all — tags silently lost on reopen). The declarative differ already emits
   `ALTER MATERIALIZED VIEW … SET TAGS`, so nothing relied on the TABLE verb.
   **Fix**: the maintained-table ALTER gate now allows only `RENAME TO`; tag
   actions reject with a steer to `ALTER MATERIALIZED VIEW`. Docs updated.
3. **Missing pins for the new ALTER behaviors**: sqllogic § 11 extended with
   structural-ALTER reject, the tag-verb steer, and the rename re-key
   (write-through + row-time maintenance live under the new name, catalog entry
   re-listed under the new name) — runs in both the memory and store suites.
4. **Docs**: write-boundary paragraph extended with the schema-path dispatch
   note; the "ALTER TABLE on the maintained table itself" section updated for
   the rename-only gate.

### Noted, deliberately not changed

- `applyMaterializedViewRewrite` fires `materialized_view_modified` whose
  `oldObject` shares the in-place-mutated derivation (old/new payloads are not
  snapshot-distinct). All current consumers key on names or `newObject` only;
  acceptable, but a future listener diffing old vs new derivation state would
  be misled.
- Renaming a *stale* maintained table re-registers maintenance
  (`registerMaterializedView` in the rename re-key); if the body no longer
  plans, the rename errors after the catalog mutation (MV stays stale under the
  new name — reads re-validate/error; fail-safe). If the body still plans,
  maintenance re-arms on a possibly-behind backing — no worse than before the
  rename (the backing stays behind by exactly the writes missed while stale;
  REFRESH owns recovery either way).
- Plan serialization embeds the derivation payload (body SQL, sourceTables) in
  maintained-table `TableReference` dumps — cosmetic verbosity the implement
  pass already adapted tests to; eliding it in `serializePlanTree` is left as a
  future cosmetic tweak.
- The store-host **two catalog entries** posture (table bundle + MV form) and
  the memory-host one-entry posture verified against the tests and docs § store
  host — consistent with the implement handoff's reading of the ticket.
