# Complete: lateral-TVF fan-out `'prefix-delete'` arm

## What shipped (verbatim summary)

The fourth `MaintenancePlan` arm, `'prefix-delete'`, for **single-source lateral-TVF
fan-out** bodies (`select t.id, …, f.value from t cross join lateral generate_series(1, t.x) f`).
Each base row of `t` fans out to N backing rows; backing PK is the composite product key
`(t.pk ∪ f.key)` with the base PK as leading prefix. Per source row-write: delete the base
row's whole fan-out slice **by base-PK prefix** and re-run the TVF fan-out **residual** for
that base row, upserting each recomputed row.

- **manager.ts** — re-added `{ kind: 'delete-by-prefix'; keyPrefix: SqlValue[] }` to
  `MaintenanceOp`; `applyMaintenanceToLayer` is now async (the new arm range-scans the
  primary btree over the prefix interval via `scanLayer`'s `equalityPrefix`, collect-then-
  deletes to avoid mutate-during-iterate, same secondary-index / change-tracking bookkeeping
  as the point `delete-key`). Exhaustive `never` default added.
- **database-materialized-views.ts** — `PrefixDeletePlan` added to the `MaintenancePlan`
  union + `applyMaintenancePlan` dispatch; `buildLateralTvfPrefixDeletePlan` routed **before**
  the generic join/TVF rejections; `applyPrefixDelete` mirrors `applyResidualRecompute`
  (affected base-key derivation OLD∪NEW → `delete-by-prefix` + N-row residual + upsert);
  `runResidual` generalized to `(scheduler, prefix, keyTuple)` and shared by both residual
  arms; `resolveTransitiveSourceCol` hoisted to module scope.

### Correctness — no blocking defect found
- **Async `delete-by-prefix` is sound.** Collect-then-delete avoids mutating the btree during
  iteration; the point ops stay synchronous within the pass so reads-own-writes ordering is
  preserved (no await separates an op's lookup from its record). Returns one `delete`
  `BackingRowChange` per matched row for the cascade. ✓
- **Key/prefix ordering is internally consistent.** `keyTuple` is in source-PK-definition
  order (bound `pk0..`, matching `injectKeyFilter(bindColumns=sourcePkCols)`); `prefix`
  (delete key) and `residualRowMatchesBasePrefix` both use **backing-PK** order. A composite
  PK whose backing base-prefix order differs from source-PK-def order stays consistent across
  delete / bind / filter. ✓
- **Per-change op ordering is correct.** INSERT → recompute+upsert (prefix-delete a no-op);
  DELETE → prefix-delete only; UPDATE → OLD then NEW affected entries (insertion-ordered
  dedup map), each delete-prefix-before-upserts; base-PK-moving UPDATE removes the old slice
  and rebuilds the new. Empty fan-out → 0 residual rows → delete-only → slice removed. ✓
- **Shape gates are thorough and correctly ordered:** routed before the generic join/TVF
  rejects (a lateral fan-out surfaces both); exactly-one-TVF, exactly-one-join,
  no-aggregate-over-fan-out, deterministic TVF + operands, advertised per-call key
  (`getType().keys.length>0`), base PK is the leading prefix (covers `order by f.value`
  reorder → user-facing `reject`, the correct disposition vs `INTERNAL` since it's
  user-reachable), non-empty TVF tail, tail does not reuse a base-PK column. ✓

### The collation question (implementer's flagged gap) — RESOLVED as reasoned-sound
`delete-by-prefix`'s `equalityPrefix` early-termination compares **binary**
(`scan-layer.ts:102`, `compareSqlValues(keyArr[i], plan.equalityPrefix[i])` — no collation
arg), while the backing btree orders the prefix by the column's declared collation and the
seek positions with the collation-aware tree comparator. Inspected `scan-layer.ts` in full.
This is sound for the prefix-delete arm because:
1. the backing base-PK column's collation is derived from the source PK column (MV backing
   columns inherit body-output column types incl. collation), so backing-order == source-order;
2. under a binary base-PK collation no collation-equal/binary-different base rows can
   interleave; under a non-binary collation `T`'s PK uniqueness collapses each collation class
   to a **single** binary value, so a base row's fan-out rows are binary-homogeneous and
   contiguous. Either way the binary early-term cannot break early or miss tail rows.
(Contrast `lookupCoveringConflicts`, which *does* gate off non-binary collation — its leading
columns are UC columns lacking the per-value uniqueness the base PK has here, so it genuinely
needs the gate while this arm legitimately does not.) The reasoning is sound; the only gap is
the **missing regression test** — filed as a minor follow-up (see below).

## Follow-up (out of scope)
A **fanning keyed join** (non-1:1 inner/cross join) is the natural next consumer of this
by-prefix machinery (shares the by-prefix delete + composite product key; the "fan-out
residual" is the join rather than a TVF). Noted in docs; not filed here (the 1:1 join is
covered by `materialized-view-rowtime-residual-join`; the fanning variant is its sibling).
