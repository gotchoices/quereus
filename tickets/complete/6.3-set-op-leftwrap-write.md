## What landed

A write through a parallel-sibling membership view (`(A∪B) union[inL,inR] (C∪D)`) previously
rejected with `the left branch uses 'select *'` because the mutation path saw the parser's wrapper
`select * from (<inner compound>) as values_N` for the LEFT operand. This change unwraps that pure
passthrough wrapper (sharing 6.2's `unwrapPassthroughSubquery` predicate) so the wrapped LEFT operand
becomes a first-class subtree operand — its data columns, `isNested`, and fan-out recursion all
derive from the inner compound, exactly as the always-direct right compound operand. The unambiguous
fan-out (data UPDATE / DELETE / `set <subtreeFlag> = false`) reaches the LEFT subtree's leaves at any
depth; ambiguous inserts into it stay deferred to `set-op-membership-nested`; the static surfaces
(`view_info` / `column_info`) walk both operands.

### Mechanism (all in `set-op.ts` unless noted)

- `unwrapBranchSelect(branchSelect)` — returns the inner compound for a pure `select * from
  (<compound>)` wrapper (via `unwrapPassthroughSubquery`), else the operand unchanged. No-op on a
  direct operand.
- `buildBranch` unwraps first (`effectiveSelect`), deriving `dataColNames`, `isNested`, and
  `branchView.selectAst` from the inner compound. The existing recursion (`fanBranchDataUpdate` /
  `fanBranchDelete` → `analyzeSetOpBranches`) then fans through to the leaves unchanged.
- `leftmostLeafSelect` + `branchColumnNames` — a nested branch's data-column NAMES come from its
  left-most leaf's projection (a set op preserves its left child's column ids at every depth). A
  LEFT-spine nest's own left leg is itself wrapped, so `branchColumnNames` descends to the real
  left-most leaf (unwrapping each layer) before reading names.
- Static AST helpers walk BOTH operands: `setOpHasSubtreeOperand` (→ `is_insertable_into = NO`),
  `surfacedInnerFlagNames` (→ surfaced left-inner flags report `is_updatable = NO` and reject
  cleanly), `isOperandWritable` (unwraps before classifying). No `schema.ts` change needed — those
  helpers cascade into `deriveViewInfo` / `column_info`.
- Docs: module docstring, `set-op-wrapper.ts` docstring (review fix, see below), and
  `docs/view-updateability.md` (new "Parenthesized LEFT subtree operand" subsection + static-surface
  paragraph).

### Correctness / logic — checked, sound
- **Unwrap symmetry.** Applying `unwrapBranchSelect` to BOTH operands in `buildBranch` is safe (no-op
  on a direct leaf / the right side's direct compound). A non-SELECT inner (`select * from
  (values…)`) stays the wrapper and is rejected downstream as a `select *` leg — verified against
  the `isPassthroughWrapper` predicate (`src.type === 'subquerySource'` gate).
- **`leftmostLeafSelect` descent.** Correct for the depth-3 left nest (peels every wrapper layer to
  the deepest left-most leaf); the nested branch's `dataColNames` is used only for the arity check,
  while each leaf recomputes its own names — `buildMemberExists` qualifies against the real leaf
  names. Right-spine behavior is unchanged (its left leg is a direct leaf; the descent reaches the
  same leaf a single `.columns` read used to).
- **Static/dynamic agreement.** The dynamic `analysis.surfacedInnerFlagNames` is plan-positional;
  the static AST helper now walks left-unwrapped-then-right. For the parallel-sibling P2 shape the
  layout is `[id, x][inA, inB (L inner)][inC, inD (R inner)][inLsub, inRsub (own)]` and both agree
  (asserted by the Key-Soundness + static-surface tests).
- **Gating threads through.** `except`/`intersect` LEFT subtree fan stays membership-gated on the
  outer boundary flag; flag-less non-union LEFT subtree stays deferred (`set-op-membership-nested-except`).
  Both covered by new tests and verified in code (`accumulateInnerGate` / `gateFlagForNonUnionSubtree`
  are side-agnostic).

### Findings & disposition

- **[minor — FIXED inline] Stale docstring.** `set-op-wrapper.ts:30-34` still claimed *"The write
  path … does NOT yet unwrap — a write through a left-wrapped compound view is currently rejected
  cleanly … `set-op-leftwrap-write` will reuse this same predicate"*. This ticket made the write path
  unwrap, so the comment was false. Rewrote it to state both paths now share the predicate and what
  the write-side unwrap delivers. (No code/behavior change; lint + full suite re-confirmed green
  after the edit.)

- **[major — pre-existing, NEW TICKET `set-op-subtree-leftleg-flag-surface` filed]
  `column_info` over-claims `is_updatable = YES` for a membership flag declared on the LEFT LEG of a
  subtree operand.** `collectSubtreeFlagNames` (behind the exported `surfacedInnerFlagNames`, unchanged
  by this ticket) descends only a subtree operand's RIGHT leg (`compound.select`), never its left leg
  (which lives in the SelectStmt core via `leftBranchSelect` and may be wrapped). The plan DOES surface
  those flags and the dynamic write correctly defers them, so the two surfaces disagree.
  **Empirically confirmed during review** (throwaway spec, since removed):
  for `select id,x from A union exists left as inL, exists right as inR ((B union[inP,inQ] C) union D)`,
  `column_info` reports `inP`/`inQ` = `YES` while `update … set inP = true` rejects with
  `set-op-membership-nested`. **Pre-existing** (`nestable-flagged-set-ops`; the repro is RIGHT-side and
  does not touch the new left unwrap) and **symmetric** to the left side this ticket adds. Out of scope
  for an inline fix (touches the read-side surfacing order); filed as `tickets/fix/`.
