## What shipped

Multi-source **INSERT** through a two-table key-preserving inner-join view, plus the
per-row **shared-surrogate mutation envelope** it requires. The shared join key is
not a view column, so an insert pre-evaluates it at the envelope: the
`ViewMutation` emitter materializes the augmented source once, appends the minted
(or directly-supplied) shared key per row, stashes the rows in `rctx.tableContexts`
under a unique descriptor, then drives the FK-parent-before-child base inserts —
each reading the identical rows back through an `EnvelopeScanNode`. One generation
per row, threaded across the fan-out. Each base side is re-planned through the
ordinary `buildInsertStmt` (new `preBuiltSource` seam), so every constraint /
conflict / FK / default rule is reused verbatim.

Full design and worked examples: `docs/view-updateability.md` § Inner Join
(Inserts), § Mutation Context. The implement-stage commit is `22868448`.

## Review findings

Reviewed the implement diff (`22868448`) with fresh eyes against SPP/DRY/modularity/
type-safety/resource-cleanup/error-handling, then probed adversarial shapes the
handoff flagged as lightly tested. Build, lint, and the full
`yarn workspace @quereus/quereus test` suite are green (**4164 passing, 9 pending,
0 failing**).

**Checked — clean:**
- **Envelope lifecycle / leak (`emitViewMutation`).** Seed evaluated before the
  source drain (observes pre-mutation state); descriptor entry set after
  materialize and removed in `finally`. `tableContexts` is `forked`, descriptor is
  a unique object per plan node — verified the fork-contract allowlist entry is
  justified. No leak.
- **Atomicity under mid-fan-out failure.** Probed a collision where the FK-parent
  base insert (side 0) succeeds and the FK-child insert then fails on a PK conflict;
  confirmed the minted parent row is rolled back — the statement is atomic, no
  orphan leaks.
- **Shared-key collision on the non-anchor side.** Seeded the non-anchor table so
  the next mint hits an existing key; confirmed it surfaces as a clean `constraint`
  error (base PK pass), not silent corruption — matching the handoff's stated limit.
- **`preBuiltSource` seam (`buildInsertStmt`).** Re-checked RETURNING (rejected
  upstream via `rejectReturning`), `ON CONFLICT` (`insert or replace` fans out to
  both base inserts — probed, correct), source arity (`assertSourceArity` rejects a
  wrong-width VALUES/SELECT), and SELECT-source inserts (probed end-to-end, mints
  correctly). `checkColumnsAssignable` runs on the injected source.
- **Routing / rejection shapes (`analyzeMultiSourceInsert`).** Probed composite-PK
  side (→ `no-default`), `select *` join body (→ reject), 3-table join (→ reject),
  directly-supplied key with different base-column names per side (→ threads
  correctly), empty source (→ 0 rows, no crash). All correct.
- **`withChildren` reconstruction** of the envelope (source + optional seed) round-
  trips through optimization; the property test (60 runs) exercises the real
  optimizer+runtime path including a previously-dangling child adopted by the minted
  parent.

**Found and fixed inline (minor):**
- **Silent drop of an over-specified shared key.** When a view exposed *both* sides
  of the equi-join key (`select a.k as ka, b.k as kb, …`) — or the same side's key
  twice — and the user supplied **divergent** values (`ka=5, kb=7`), the engine
  silently discarded one and forced `kb := ka`, landing `(5,5)`. That violated the
  substrate's pervasive "precise diagnostic over silent widening" stance. Added a
  guard in `analyzeMultiSourceInsert` that rejects an over-exposed shared key with
  `unsupported-join` ("the shared join key is exposed by more than one view
  column; supply it through a single view column"). Regression test added to
  `93.4-view-mutation.sqllogic` § Phase 2b (f); doc rejection list updated. The
  single-exposure directly-supplied path (the existing (c) test) is unaffected.

**Deferred / documented limits (no action — out of scope, accurately described in
the handoff + doc):**
- Inner join only; outer-join / optional-member insert rejected.
- `integer-auto` mint only — `declared-default`-expression and non-integer
  surrogates deferred (rejected `no-default` when not directly supplied). The mint
  is computed in the emitter (`seed + ordinal`); deterministic-given-context, so it
  replays, but a genuinely non-deterministic generator would need its per-row values
  recorded in a context structure first.
- Seed reads only the anchor side's key (`max(anchor.key)`); a non-anchor collision
  is caught by the base constraint pass (verified above), not pre-empted.
- `lens-multi-source-put-fanout` (`decomposition.ts`) is **not** wired to the
  envelope yet — still raises `unsupported-decomposition-insert`. The
  `ViewMutationNode.envelope` + `EnvelopeScanNode` surface is shaped to be consumed
  by it (plan-agnostic descriptor + materialized-rows-in-context). A follow-up.
- `default_for` not honored for multi-source insert; `> 2`-table / self-join /
  composite or expression join keys / cross-source values rejected (shared gates).
- No new bigint-precision risk introduced beyond the documented integer-surrogate
  limit (minted value is a JS number; round-trip verified for safe-int keys by the
  property test and probes).

**Empty categories:** No major findings (none filed as new fix/plan/backlog
tickets). No new test gaps left open for the supported shape — the probes above
became regression coverage or confirmed existing behavior.

## How to exercise

```sql
create table u_core (rid integer primary key, name text);
create table u_contact (rid integer primary key, email text);
create view appuser as select c.name as name, k.email as email
  from u_core c join u_contact k on k.rid = c.rid;
insert into appuser (name, email) values ('Ada','ada@x.io'), ('Lin','lin@x.io');
-- one minted rid per row, shared across both bases: u_core (1,'Ada'),(2,'Lin'); u_contact (1,'ada@x.io'),(2,'lin@x.io')
```

See `93.4-view-mutation.sqllogic` § Phase 2b (a)–(f) and the `multi-source inner
join` PutGet in `property.spec.ts`.
