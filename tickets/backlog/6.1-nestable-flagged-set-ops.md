## Problem

The membership-flag-as-appended-column model does not nest. Two distinct failures, both
verified empirically (project ts-node loader, in-memory tables A/B/C):

### 1. A flagged subtree operand fails to PLAN (column-count mismatch)

`A union[inA,inSub] (B union[inB,inC] C)` parses fine, but creating the view throws:

```
CREATE FAILED: right-nested both-levels
  → SET operation column count mismatch: left has 2, right has 4
```

Cause: the inner `(B union[inB,inC] C)` plans to a `SetOperationNode` whose attributes are
`[id, x, inB, inC]` (data columns + its two appended flag columns — `set-operation-node.ts`
`buildAttributes`). The outer `SetOperationNode` constructor validates
`left.columns.length === right.columns.length` (`set-operation-node.ts:46`) — 2 (A) vs 4
(inner subtree) → throw. Any inner flag inflates the operand's arity and breaks outer
alignment.

### 2. A subtree operand is not writable (recursion bottoms out unwritable)

A flag-less subtree operand *does* plan and read correctly — a subtree flag's read already
works generically (the probe is `tuple ∈ <subtree output>`):

```
CREATE/READ OK: select id,x from A union exists left as inA, exists right as inSub
                (select id,x from B union select id,x from C)
  → rows: {id:1,inA:true,inSub:false} {id:2,inA:true,inSub:true}
          {id:3,inA:false,inSub:true} {id:4,inA:false,inSub:true}
```

…but every **write** that touches the subtree branch throws, because the per-branch recursion
(`set-op.ts` → `propagate`) reaches the flag-less `(B union C)` subtree, which is not a
writable body:

```
update V set inSub = true  …  → FAILED: cannot write through view '__setop_right':
                                 view body operator 'SetOperation' is not updateable in phase 1
update V set inSub = false  …  → FAILED (same)
update V set x = 99 …  (data fan-out) → FAILED (same — fan-out hits the subtree branch)
delete from V where …          → FAILED (same)
update V set inA = false …  (leaf branch) → OK
```

So a set-op view with a subtree operand is **readable but inconsistently writable**: leaf-flag
writes succeed while subtree-flag writes and *all* data/delete fan-out throw at runtime — and
the static surface (`column_info`/`view_info`) still advertises the data columns
`is_updatable='YES'` (a catalog-honesty over-claim, cf.
`set-op-membership-static-surface-branch-writability`).

## Desired behavior

A `SetOperationNode` whose operand is itself a (possibly flagged) `SetOperationNode` is a
first-class, readable, **writable** body:

- **Column alignment compares DATA columns only.** The outer operand-arity check (and the
  union schema) must align on the operands' **data** column counts, not on data+flags. An
  inner subtree's flag columns must not count toward the outer alignment — decide whether
  inner flags are (a) hidden/consumed at the nesting boundary and re-derived as needed, or
  (b) surfaced as the outer view's columns under a defined column-projection rule. The
  membership model wants the inner flags to remain **readable columns of the outer view**
  (`inB`/`inC` are read row-by-row in the nested fixtures), so the alignment fix must surface
  them without letting them participate in the data-column set-identity/dedup or the outer
  arity check.
- **A subtree operand is recursively writable.** Writing a subtree flag (`set inSub = true`)
  recurses into the subtree's own `put` (insert/delete into `(B∪C)`, which itself fans out to
  its leaves); data/delete fan-out through a subtree operand recurses the same way. This
  requires the subtree to expose a writable decomposition even when it carries **no** flags of
  its own (a flag-less `union`/`intersect`/`except` subtree must become writable through the
  same per-branch fan-out the membership path uses — or the recursion must address the
  subtree's leaves directly).
- **Lineage/static surface.** The subtree-operand `set-op-branch` `existence` `UpdateSite`
  is already registered generically (`set-operation-node.ts` `membershipLineage`), and reads
  resolve correctly; the static `column_info`/`view_info` surface must agree with the now-real
  writability (and stop over-claiming when a subtree branch is non-writable).

## Acceptance (sketch — for the implement pass)

- `A union[inA,inSub] (B union[inB,inC] C)` creates, plans, and reads: `inA`/`inB`/`inC`/`inSub`
  agree row-by-row with `tuple ∈ <that operand's data relation>`, and the outer view keys
  stay on data columns only (no flag ever in a claimed key — Invariants 1–2 at all depths).
- `set inSub = true/false` over that view inserts into / deletes from the `(B∪C)` subtree
  (recursing to its leaves); data/delete fan-out reaches every member leaf through subtree
  operands. No `'SetOperation' is not updateable` runtime throw.
- The flag-less-subtree read fixture above keeps reading correctly (regression).

## Notes

- This is the structural prerequisite for `set-op-membership-nested` (blocked) — specifically
  its **subtree-flag writes** and **nested fan-out**. It is **independent of**
  `parenthesized-compound-set-op-legs`: the right-nested form here exercises nesting without
  needing a parenthesized left leg. (The *product-coordinate* model needs both foundations.)
- Folds in the symptom that `set-op-membership-static-surface-branch-writability` notes for
  the subtree case (static surface vs. runtime write disagreement).
- Sequenced as a foundation; promote to plan/ once the team decides the inner-flag
  column-exposure model (the load-bearing design call).
