## Why (verified, not theorized)

Foundations 6.0 (`parenthesized-compound-set-op-legs`) and 6.1 (`nestable-flagged-set-ops`)
landed, but the parent plan ticket's premise — that the remaining work is "just per-leaf
addressing" — is **false**. Empirically (ts-node loader, in-memory tables, 2026-06-08):

```
P1  (select id,x from A union select id,x from B)            -- flag-less inner
      union exists left as inLsub, exists right as inRsub
    (select id,x from A union select id,x from B)
  → PLANS + READS OK   (left/right both 2 data cols — no inner flags to miscount)

P2  (select id,x from A union exists left as inAL, exists right as inBL select id,x from B)
      union exists left as inXsub, exists right as inYsub
    (select id,x from A union exists left as inAR, exists right as inBR select id,x from B)
  → CREATE FAILED: SET operation column count mismatch: left has 4, right has 2
```

**Root cause.** The parser carries a parenthesized LEFT compound operand as the wrapper
`select * from (<inner compound>) as values_<offset>` (see `wrapAsSubquerySelect` /
`parseCompoundTail` in `parser.ts`; 6.0). `buildCompoundSelect` builds that wrapper into a
**ProjectNode (`select *`) over the inner `SetOperationNode`**, so the outer node's left child
is *not* a `SetOperationNode`. `dataArity(left)` in `set-operation-node.ts` therefore takes the
`else` branch (`left.getType().columns.length`) and counts the inner's surfaced flags as data
columns (2 data + 2 flags = 4). The RIGHT subtree is a direct `SetOperationNode`, so
`dataArity(right)` recurses correctly to 2 → the constructor's `leftData !== rightData` check
throws. **A flagged parallel-sibling view cannot even be planned, so none of the
product-coordinate / multi-target / contradiction work in the parent ticket is reachable.**

This ticket fixes the **read/plan** half: make the wrapped left subtree's DATA arity correct so
P2-shaped views plan and read with the **sum** surface (`[data] ++ [L flags] ++ [R flags] ++
[own flags]`, distinct flag names → distinct columns). Writes through the left wrapper are 6.3.
The **product** model (reused flag names → merged coordinate columns) is the blocked
`set-op-product-coordinate-model` and is explicitly out of scope here.

## Design

**Recommended locus: unwrap the wrapper AST at build time in `buildCompoundSelect`.** Before
building `leftCore` (and, defensively, the right operand), detect the pure-passthrough wrapper
shape and build the **inner compound directly**, so the operand plan IS the inner
`SetOperationNode` and the existing `dataArity` / `flagCount` recursion (which already handles a
`SetOperationNode` operand at any depth — `nestable-flagged-set-ops`) just works. This keeps the
fix in one place and leaves `dataArity` structurally simple (no fragile "recurse through an
arbitrary projection" logic).

Add a shared, exported pure predicate (it is reused verbatim by 6.3's write path, so define it
once — proposed home `planner/mutation/set-op-wrapper.ts` or alongside the parser wrapper):

```ts
/** The inner query-expr of a pure `select * from (<query-expr>) as values_N` wrapper, else undefined. */
function unwrapPassthroughSubquery(sel: AST.SelectStmt): AST.QueryExpr | undefined
```

True shape: `sel.columns` is exactly one `{type:'all'}` (no table-qualified star, no extra
columns); `sel.from` is exactly one `{type:'subquerySource'}`; and the wrapper carries **no**
`where` / `groupBy` / `having` / `distinct` / `all` / `limit` / `offset` / `schemaPath` of its
own (any of those make it a real projection, not a pure regrouping). Return
`sel.from[0].subquery`. Unwrap is **idempotent / recursive**: a doubly-wrapped left
(`((A∪B)) union …` nested deeper) unwraps until the inner is no longer a pure wrapper.

In `buildCompoundSelect`, apply the unwrap to `leftCore` before `buildSelectStmt`, and to a
`select`-typed right operand before its build, so a parenthesized compound operand on **either**
side becomes a direct `SetOperationNode` (or a direct leaf when the inner is a non-compound
`select`). A wrapper whose inner is a non-compound select unwraps to that select — harmless and
keeps arity at the leaf's column count.

Once the operands build directly, **no change to `set-operation-node.ts` should be needed** —
`buildAttributes` already produces `[data(2)] ++ [L flags(inAL,inBL)] ++ [R flags(inAR,inBR)] ++
[own flags(inXsub,inYsub)]` because it takes `left.getAttributes()` verbatim and slices the
right by `dataColumnCount()`. Verify this; if any residual wrapper survives (e.g. an
optimizer-inserted Retrieve), prefer fixing the build-time unwrap over special-casing
`dataArity`. If `dataArity`/`flagCount` MUST change, keep the change to "see through a pure
column-preserving passthrough to a `SetOperationNode`" and document why.

## Edge cases & interactions

- **Flag-less left wrapper (P1).** Already plans/reads; the unwrap must keep it byte-identical
  (no behavior change) — assert P1 still reads the same rows/flags after the change.
- **Distinct-name flagged parallel siblings (P2).** Now plans; read surface is 8 columns
  `id, x, inAL, inBL, inAR, inBR, inXsub, inYsub`; each inner flag reads `tuple ∈ <its own leaf>`
  and each outer subtree flag reads `tuple ∈ <that subtree>`. Pin row-by-row.
- **Reused-name flagged parallel siblings (P3, `inA`/`inB` at both levels).** Out of scope —
  produces duplicate column names. This ticket does NOT merge them; it is acceptable that P3
  reads duplicate `inA`/`inB` columns OR is rejected at create. **Pick one and pin it**
  (recommended: leave the duplicate-name surface as-is and add a test asserting current
  behavior, with a comment pointing at `set-op-product-coordinate-model`). Do not invent merge
  semantics here.
- **Depth ≥ 3 left nesting** (`(((A∪B)∪(C∪D))) union[…] (E∪F)`) — the recursive unwrap must peel
  every pure wrapper layer; arity stays at the left-most leaf's data count.
- **Right operand also parenthesized** (`(A∪B) union[…] ((C∪D))`) — the right unwrap path covers
  it; both operands become direct `SetOperationNode`s.
- **`except` / `intersect` inner** under the wrapper — arity/read are unaffected by operator;
  the read surface and key survival follow the existing per-operator rules. (Write gating stays
  6.3 / the `set-op-membership-nested-except` deferral.)
- **Non-passthrough derived table** (`select x from (A∪B) v` — projects a subset, or has a
  WHERE) — must NOT unwrap (the predicate's no-extra-clause / single-star guard prevents it);
  it stays an opaque relation. Add a negative assertion.
- **Key Soundness.** No flag (inner or outer, any depth) appears in a claimed key; the data-only
  key survival in `getType()` is unchanged. Assert against P2.
- **Regression scope.** `dataArity` is only consulted on `SetOperationNode` operands; the only
  shapes whose behavior changes are set-op operands that are pure compound wrappers (today: throw
  or flag-less). Full suite must stay green.
