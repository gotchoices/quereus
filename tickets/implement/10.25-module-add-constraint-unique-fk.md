# Module `ALTER TABLE ADD CONSTRAINT` for UNIQUE / FOREIGN KEY

## Why

`alter-table-drop-rename-constraint` (ticket 10.2, prereq) wired the declarative constraint
**lifecycle** (add / drop / rename by name) into the differ. DROP and RENAME work end-to-end for all
three classes (CHECK / UNIQUE / FOREIGN KEY). **ADD does not** for non-CHECK: `ADD CONSTRAINT CHECK`
is handled engine-side in `runtime/emit/add-constraint.ts` (`runAddCheck` mutates `checkConstraints`
directly), but `ADD CONSTRAINT UNIQUE` / `FOREIGN KEY` route through
`module.alterTable({ type: 'addConstraint', constraint })`, and **both built-in modules (memory +
store) throw `StatusCode.UNSUPPORTED`** for any constraint type.

Consequence (10.2 gap #1): the differ *emits* `ALTER TABLE … ADD <fragment>` for a declared-absent
UNIQUE/FK named constraint, so a **declarative add of a UNIQUE/FK to an already-existing table fails
at apply** with `UNSUPPORTED`. The migration batch rolls back atomically (no partial state), so it
is safe but a poor UX for a plausible action. It also **blocks ticket 10.3**
(`alter-constraint-body-change-drop-add`): a UNIQUE/FK body change is realized as drop+add and the
add half hits this same wall. (10.3 already lists this slug in its `prereq:` — keep the slug intact.)

## Goal

Implement `addConstraint` for UNIQUE and FOREIGN KEY in both built-in modules so the existing
`runAddConstraintViaModule` path succeeds, **re-validating existing rows** against the new constraint
and failing with `StatusCode.CONSTRAINT` (atomic — no schema mutation) when current data violates it.

## Design (resolved)

### 0. The constraint reaches the module as raw AST

`AddConstraintNode.constraint` is an `AST.TableConstraint` (see `planner/nodes/add-constraint-node.ts`),
and `SchemaChangeInfo.addConstraint` carries it verbatim (`vtab/module.ts`:
`{ type: 'addConstraint'; constraint: TableConstraint }`). For UNIQUE the AST carries
`con.columns: { name }[]`; for FK it carries `con.foreignKey: ForeignKeyClause` plus `con.columns`
(child cols). So each module must **resolve column names → indices** itself, exactly the way
`CREATE TABLE` does. `runAddCheck` already sets the precedent of building a constraint schema from
the AST in the emitter; we mirror that for UNIQUE/FK but inside the module (it owns the cached
`TableSchema` whose `columnIndexMap` we need, and whose enforcement reads the appended arrays).

`emit/add-constraint.ts` needs **no change** — it already dispatches CHECK→`runAddCheck` and
everything else→`runAddConstraintViaModule`, which calls `module.alterTable` and then
`schema.addTable` + `table_modified` notify. Confirm those two post-steps still fire identically to
the drop/rename paths.

### 1. Shared AST → schema builders + FK validator (`schema/constraint-builder.ts`, NEW)

`SchemaManager.extractUniqueConstraints` / `extractForeignKeys` already encode the canonical
table-level AST→schema mapping (auto-naming, default actions, deferred parent-column resolution,
child/parent column-count check). To stay DRY, **lift the per-`TableConstraint` body into exported
helpers** and have both the modules and `SchemaManager` call them:

```ts
// columnIndexMap is the CHILD table's map (lowercased name -> index)
export function buildUniqueConstraintSchema(
  con: AST.TableConstraint,            // con.type === 'unique', con.columns present
  columnIndexMap: ReadonlyMap<string, number>,
): UniqueConstraintSchema

export function buildForeignKeyConstraintSchema(
  con: AST.TableConstraint,            // con.type === 'foreignKey', con.foreignKey present
  columnIndexMap: ReadonlyMap<string, number>,
  childTableName: string,
  childSchemaName: string,
): ForeignKeyConstraintSchema
```

These must reproduce the existing logic exactly: UNIQUE → `{ name, columns(indices), defaultConflict:
con.onConflict, tags }`; FK → `{ name: con.name ?? '_fk_<table>_<cols>', columns(child indices),
referencedTable: fk.table, referencedSchema: childSchemaName, referencedColumns: [] (deferred),
referencedColumnNames: fk.columns, onDelete/onUpdate ?? 'restrict', deferred:
fk.initiallyDeferred ?? false, tags }`, including the **child/parent column-count mismatch** ERROR
(`fk.columns.length !== childColIndices.length`). Then refactor the **table-level** loops in
`manager.ts` (`extractUniqueConstraints` ~L1252, `extractForeignKeys` ~L1188) to call these. Leave
the column-level loops as-is (out of scope for ADD — `ALTER TABLE ADD CONSTRAINT` is always the
table-level form).

FK existing-row validation is engine-level and **identical across backends**, so factor it once:

```ts
// Throws QuereusError(CONSTRAINT) if any child row violates the new FK; no-op when
// `pragma foreign_keys` is off. Reads committed/base data via db.prepare.
export async function validateForeignKeyOverExistingRows(
  db: Database,
  childSchema: TableSchema,
  fk: ForeignKeyConstraintSchema,
): Promise<void>
```

Implementation: early-return when `!db.options.getBooleanOption('foreign_keys')`. Resolve the parent
via `db.schemaManager.findTable(fk.referencedTable, fk.referencedSchema)` (if absent, MATCH-SIMPLE
means only NULL-bearing child rows could be valid — match the child-side builder's behavior in
`planner/building/foreign-key-builder.ts` and treat any non-NULL-complete child row as a violation).
Resolve parent columns via `resolveReferencedColumns(fk, parentSchema)`. Build an orphan-detection
query with `quoteIdentifier` (exported from the barrel; precedent: `runtime/foreign-key-actions.ts`
builds `select … limit 1` strings) — **MATCH SIMPLE**: a child row with ANY NULL FK column is
allowed, so detect only fully-non-NULL orphans:

```sql
select 1 from <child>
where <fkCol_i> is not null  -- AND-chain over all FK columns
  and not exists (select 1 from <parent> where <parent.refCol_i> = <child.fkCol_i> -- AND-chain
                  )
limit 1
```

A returned row ⇒ orphan exists ⇒ throw `CONSTRAINT` (message naming the FK + child columns). Run
this from inside each module's FK arm **before** swapping the cached schema (so a throw leaves the
table unmodified). The module holds its schema-change latch during the call, but the validation only
*reads* (it goes through `connect()`/scan, which does not take the schema-change latch), so there is
no self-deadlock. For memory, `ensureSchemaChangeSafety()` has already drained in-flight transaction
layers to base, so the scan sees the rows that will be governed.

Export all three from `packages/quereus/src/index.ts`.

### 2. Memory module — `MemoryTableManager.addConstraint(constraint: AST.TableConstraint)`

New async method, **mirroring `createIndex` (manager.ts ~L1796) and `dropConstraint`/
`renameConstraint`** for the latch + `ensureSchemaChangeSafety()` + snapshot/restore + `'alter'`
schema-change-event scaffolding. Dispatch on `constraint.type`:

- **UNIQUE** — resolve via `buildUniqueConstraintSchema`. Construct the implicit covering index the
  same way `ensureUniqueConstraintIndexes` does: name = `uc.name ?? '_uc_<colNames join _>'`, columns
  = `uc.columns.map(i => ({ index: i, collation: columns[i].collation }))` (carry per-column
  collation so enforcement honors e.g. NOCASE), `predicate: uc.predicate`, `unique: true`. Build the
  `finalNewTableSchema` with the constraint appended to `uniqueConstraints` AND the index appended to
  `indexes`, `updateSchema(finalNewTableSchema)`, then `await this.baseLayer.addIndexToBase(indexSchema)`
  — which runs `populateNewIndex`, the existing path that **raises `CONSTRAINT` on the first duplicate
  among in-scope rows** (honoring partial predicate + per-column collation, NULLs distinct). On
  success register `implicitCoveringStructures.set(uc.name ?? indexName, { indexName, origin:
  'implicit-from-unique-constraint' })`. On throw, restore `originalManagerSchema` via
  `updateSchema` + `rebuildAllSecondaryIndexes()` (non-strict) and rethrow — exactly the createIndex
  catch.
  - Reuse-not-rebuild corner: if an **existing UNIQUE index** already covers the exact columns, skip
    the build (data is already validated by that index) and just append the constraint + register the
    covering structure. If only a **non-unique** index covers them, still build the unique covering
    index (it validates) — a non-unique index gives no uniqueness guarantee. (Mirror
    `ensureUniqueConstraintIndexes`' matching-index lookup but gate reuse on the match being unique.)

- **FOREIGN KEY** — resolve via `buildForeignKeyConstraintSchema`. Snapshot, append to `foreignKeys`,
  call `validateForeignKeyOverExistingRows(this.db, newSchema, fk)` (pragma-gated; throws CONSTRAINT
  on orphan → restore + rethrow), then commit the schema swap. No physical structure — FK enforcement
  is engine-side (synthesized EXISTS checks at plan time, `planner/building/foreign-key-builder.ts`),
  so the manager only needs the cached schema to carry the FK so introspection / the differ observe
  it and subsequent writes are checked.

Route both module entry points to it: `module.ts` `alterTable` `case 'addConstraint'` (currently
throws, ~L859) → `await manager.addConstraint(change.constraint)`; and `table.ts`
`MemoryTable.alterSchema` `case 'addConstraint'` (currently throws, ~L339) → same. Emit the same
`{ type: 'alter', objectType: 'table' }` schema-change event drop/rename emit.

### 3. Store module — `store-module.ts` `alterTable` `case 'addConstraint'`

Replace the UNSUPPORTED throw (~L774). Resolve `oldSchema` (already in scope). Dispatch on
`change.constraint.type`:

- **UNIQUE** — resolve via `buildUniqueConstraintSchema(con, oldSchema.columnIndexMap)`. Store
  enforces inline UNIQUE by full-scan over `uniqueConstraints` (no separate index store), so there is
  nothing physical to build — but we must **validate existing rows**: scan the data store and detect a
  duplicate key the same way `buildIndexEntries` already does (a `seen` Set keyed on a JSON signature
  of the indexed values; **NULLs distinct** — skip dup detection when any indexed value is NULL; honor
  the partial `predicate` via `compilePredicate`). On the first duplicate, throw `CONSTRAINT` (naming
  the columns) **before** `updateSchema`/`saveTableDDL`. Reuse the dup-scan shape from
  `buildIndexEntries` (factor a tiny private helper if it reduces duplication; do not write any index
  store). On success: `table.updateSchema({ ...oldSchema, uniqueConstraints: [...] })` +
  `await this.saveTableDDL(updatedSchema)` + `'alter'` event.

- **FOREIGN KEY** — resolve via `buildForeignKeyConstraintSchema`. Append to `foreignKeys`, call
  `validateForeignKeyOverExistingRows(db, updatedSchema-or-childSchema, fk)` (pragma-gated; throws
  before persistence on orphan), then `updateSchema` + `saveTableDDL` + `'alter'` event.

Match the existing store `dropConstraint`/`renameConstraint` arms for structure and the
`generateTableDDL`-backed persistence. Note: a UNIQUE add is persisted as inline UNIQUE in the
regenerated DDL — confirm `generateTableDDL` round-trips a `uniqueConstraints` entry that lacks
`derivedFromIndex` as a table-level `UNIQUE (...)` (it must, since DROP already relies on the
inverse).

### 4. Cross-cutting confirmations (verify; expected no-change)

- `emit/add-constraint.ts` — unchanged; `runAddConstraintViaModule` already does `module.alterTable`
  → `schema.addTable(updated)` → `table_modified` notify. Confirm cache invalidation fires as for
  drop/rename.
- `quereus-isolation/src/isolation-module.ts` — `alterTable` forwards to `underlying.alterTable`
  (~L633) and `translateOverlayRow` already no-ops `addConstraint` (grouped with drop/rename,
  ~L763) since the row shape is unchanged. Confirm an isolated memory/store table inherits the new
  behavior with no isolation-layer edit.

## Edge cases & interactions

- **UNIQUE add over a table with an existing duplicate** → `CONSTRAINT`; schema unchanged; enforcement
  NOT partially installed (memory: covering index not left in `indexes`/base; store: `uniqueConstraints`
  not persisted). Re-run the *same* add after removing the dup ⇒ succeeds.
- **UNIQUE add over conforming data** → enforces going forward (insert a dup post-add ⇒ CONSTRAINT);
  memory: covering index appears in `index_info` / `findIndexForConstraint`; idempotent re-declare
  ⇒ empty diff.
- **UNIQUE NULL semantics** — multiple NULLs allowed: existing rows with a NULL in any constrained
  column never count as duplicates (memory `populateNewIndex` and store dup-scan both skip NULL-bearing
  rows). Validate with two existing rows that share NULLs ⇒ add succeeds.
- **UNIQUE per-column collation** — existing rows unique under BINARY but colliding under NOCASE must
  be **rejected** when the column collation is NOCASE (the covering index keys by the column's
  collation; store: NOCASE is the store's fixed key collation — confirm the dup signature respects it,
  or document the store limitation if it cannot, mirroring the ALTER COLUMN SET COLLATE store note).
- **UNIQUE reusing a pre-existing index** — exact-column **unique** index present ⇒ skip rebuild, no
  double index; non-unique index present ⇒ still build-and-validate the covering index.
- **FK add, `pragma foreign_keys = true`, orphan child row** → `CONSTRAINT`; schema unchanged.
- **FK add, `pragma foreign_keys = false`** → add succeeds with NO validating scan; enforcement
  deferred to subsequent writes (and re-gated by the pragma at write time).
- **FK MATCH SIMPLE** — a child row with any NULL FK column is allowed even with no parent; only
  fully-non-NULL orphans abort. Validate a NULL-FK row + a satisfied row both survive.
- **FK parent table missing** — with pragma on, a non-NULL-complete child row cannot be satisfied ⇒
  abort (match the child-side builder's null-guards-only fallback); a child table with only NULL-FK
  rows ⇒ add succeeds.
- **Composite FK / cross-schema FK** — child/parent column-count mismatch ⇒ ERROR at resolve time;
  `referencedSchema` defaults to the child's schema (mirror `extractForeignKeys`). Exercise a
  cross-schema parent (see `41-fk-cross-schema.sqllogic` for the pattern).
- **Named vs unnamed** — only **named** constraints are differ-reachable (10.2 excludes auto-names),
  but the primitive must also accept the explicit unnamed `ALTER TABLE … ADD UNIQUE (...)` /
  `ADD FOREIGN KEY (...) REFERENCES …` form (auto-named `_uc_…` / `_fk_…`).
- **Declarative convergence (gap #1 closed)** — declaring a new named UNIQUE/FK on an existing table
  and `apply schema` now **converges**, and a second `apply` is a no-op (idempotent). This is the
  assertion 10.2 could not make.
- **Atomicity under the migration batch** — a failing add inside `runBatchedMigrationLoop` still rolls
  the whole overlay back (already true); additionally the *module itself* must not leave half-applied
  state on the single-statement path (covered by the snapshot/restore).
- **Read-only table** → `READONLY` (memory manager guards this already; keep the guard).
- **Concurrent / layered access (memory)** — take `MemoryTable.SchemaChange` latch + drain via
  `ensureSchemaChangeSafety()` before mutating, like every other manager schema op; re-point
  connections off the old base on success (the existing helpers already do this for index/constraint
  changes — reuse, don't reinvent).
- **Both modules under both harnesses** — the new `.sqllogic` must pass under `yarn test` (memory) and
  `yarn test:store` (store / LevelDB), since store-path ALTER + constraints is exactly what
  `test:store` exists to exercise.
