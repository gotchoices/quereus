# Skip RESTRICT on the trust-the-origin apply path

## Decision (already signed off — do not re-litigate)

On the external-row-change apply path with `applyForeignKeyActions: true`, the FK-actions
facet must **skip parent-side RESTRICT enforcement entirely** and propagate only the
non-RESTRICT actions (cascade / set-null / set-default). The origin already enforced
RESTRICT at its own commit; re-enforcing it on the receiver wedges the sync stream
(throw → no metadata commit → re-resolve → re-apply the identical batch → throw forever),
which contradicts `docs/sync.md` § Apply-time validation (trust-the-origin). A
replica-only RESTRICT invariant is, **by design, not enforced on apply**; a host that
needs the receiver notified expresses it as a global assertion (detect-and-notify).

## Why a Database-level flag, gated inside the assert functions

Skipping the seam's own top-level `assertTransitiveRestrictsForParentMutation` call is
**necessary but not sufficient**. `assertTransitiveRestrictsForParentMutation` is reached
during a single apply batch from **three** sites, all while the batch holds the exec
mutex (so no concurrent statement can observe the flag):

1. The seam's explicit top-level call — `database-external-changes.ts:211`.
2. **Nested cascade DML** — `executeForeignKeyActionsAndLens` issues cascade DELETE/UPDATE
   via `db._execWithinTransaction(...)`, which re-enters the DML executor; that executor
   fires the RESTRICT pre-check for the *cascaded* table's children at
   `dml-executor.ts:723` (REPLACE-eviction), `:797` (update), `:938` (delete). This is the
   depth-≥1 case the plan flagged.
3. **MV-maintenance-driven parent FK actions** — when `maintainMaterializedViews` is on,
   `applyParentFkActionsForMaintainedBacking` (`database-materialized-views.ts:1351`) runs
   the same assert over each maintained backing change.

Removing only site 1 leaves 2 and 3 throwing. Therefore: thread an **apply-mode flag** on
`Database`, set for the duration of the apply batch, and **early-return at the top of each
RESTRICT-asserting function** when the flag is set. One gate per function covers all three
sites uniformly, at **every cascade depth** — so this is a complete fix, not a partial-depth
one. Cascade propagation (`executeForeignKeyActions` / `executeLensForeignKeyActions`) is
untouched: it already skips RESTRICT (`foreign-key-actions.ts:78`) and only fires
cascade / set-null / set-default, so non-RESTRICT actions still flow at all depths.

The flag must **suppress only the RESTRICT *assertions*, never the cascade *actions***, and
must be restored in a `finally` so a non-RESTRICT throw mid-batch (e.g. a cascade DML
tripping a CHECK / NOT NULL) does not leave a later normal statement silently suppressed.

## Interface / shape

`Database` (mirror the existing `_setPendingCommitAssertionSink` convention, ~`database.ts:1899`):

```ts
private _fkRestrictSuppressed = false;

/** @internal Apply-mode RESTRICT suppression. While set, the parent-side FK
 *  RESTRICT pre-checks early-return (trust-the-origin apply path). Cascade /
 *  set-null / set-default propagation is unaffected. Returns the prior value so
 *  the caller restores it in a finally (supports nesting). */
public _setFkRestrictSuppressed(value: boolean): boolean {
  const prior = this._fkRestrictSuppressed;
  this._fkRestrictSuppressed = value;
  return prior;
}
public _isFkRestrictSuppressed(): boolean { return this._fkRestrictSuppressed; }
```

`foreign-key-actions.ts` — at the very top of each of the three exported RESTRICT asserts
(`assertTransitiveRestrictsForParentMutation`, `assertNoRestrictedChildrenForParentMutation`,
`assertLensRestrictsForParentMutation`), before the existing `foreign_keys`-pragma check:

```ts
// Trust-the-origin apply path: RESTRICT is enforced by the origin at its own commit,
// so the receiver skips it (gating cascade-action propagation is intentionally NOT here).
if (db._isFkRestrictSuppressed()) return;
```

(Gating all three is belt-and-suspenders: the lens assert is only reached *via* the
transitive one today, but gating it directly removes any reachability fragility.)

`database-external-changes.ts` — set the flag around the batch and drop the top-level call:

```ts
const releaseMutex = await db._acquireExecMutex();
const priorSuppress = applyForeignKeyActions ? db._setFkRestrictSuppressed(true) : false;
try {
  ...
  if (applyForeignKeyActions && change.op !== 'insert') {
    // Trust-the-origin: skip RESTRICT entirely (origin already enforced it);
    // propagate only cascade / set-null / set-default. The flag set above makes
    // every nested cascade DML and MV-maintenance FK pass skip RESTRICT too.
    await executeForeignKeyActionsAndLens(db, tableSchema, change.op, change.oldRow, change.newRow);
  }
  ...
} finally {
  if (applyForeignKeyActions) db._setFkRestrictSuppressed(priorSuppress);
  releaseMutex();
}
```

Note: the existing `assertTransitiveRestrictsForParentMutation(db, tableSchema, change.op, ...)`
seam call is **removed** (it would no-op under the flag anyway; removing it makes the intent
explicit and avoids a wasted walk). The flag must be set **before** the change loop so the
nested cascade DML in step 2/3 above observes it.

## Edge cases & interactions

- **Depth-≥1 cascade hitting a RESTRICT grandchild** (the core bug): origin-committed
  parent delete cascades to a child whose own children carry a RESTRICT FK. Must **not**
  throw. The cascade deletes the child; the grandchild's RESTRICT FK is left dangling —
  that is the trust-the-origin outcome (the origin's stream carries the consistent grand-
  child state separately). Verify the batch commits and no throw escapes.
- **Top-level replica-only RESTRICT**: parent delete trips the replica's RESTRICT on a
  surviving local child. Must **not** throw and must **not** cascade (RESTRICT is not an
  action) — the child row simply remains. Verify child survives + batch commits.
- **REPLACE-eviction RESTRICT** (`dml-executor.ts:723`): a cascade UPDATE that forces a
  non-PK UNIQUE eviction whose evicted row has RESTRICT children — covered by the uniform
  gating (same assert function). Note it; a dedicated test is optional if hard to construct.
- **Flag restored after a NON-RESTRICT mid-batch throw**: a cascade DML that legitimately
  throws (CHECK / NOT NULL / connection error) must still propagate AND restore the flag —
  the `finally` guarantees a subsequent normal statement re-enforces RESTRICT. This is the
  load-bearing correctness property; test it (or at minimum the restore-after-success case).
- **Post-batch normal DML enforces RESTRICT**: after an apply batch returns, a plain
  `delete from parent` against a RESTRICT child via `db.exec` must throw as before (flag
  cleared). Test this explicitly.
- **Nested / re-entrant suppression**: save-and-restore the prior value (not blind reset to
  `false`) so any future nesting restores correctly.
- **`applyForeignKeyActions` off (default)**: no FK actions fire and the flag is never set —
  unchanged behavior.
- **`foreign_keys` pragma off**: assert functions already early-return; flag is moot.
- **Global assertions unaffected**: the flag suppresses only FK RESTRICT, never the
  commit-time global-assertion sink (report mode). A replica that wants notification of a
  referential invariant uses a global assertion — document this in the test comments + docs.
- **Explicit caller transaction vs implicit**: the flag is set/restored entirely within the
  batch call (mutex held), so it never leaks across statements of an explicit caller txn.
- **`store-adapter.ts` / `admission.ts` need NO change**: the fix removes the throw at its
  source, so the consumer's whole-batch-error path is simply never triggered by a RESTRICT.
  Confirm by inspection that neither separately enforces RESTRICT.
