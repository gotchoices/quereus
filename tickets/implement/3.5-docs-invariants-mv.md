## Why materialized views are the right second area

The MV documentation makes an unusually strong, unusually *checkable* promise. Its
opening sentence is a guarantee, not a description: a materialized view is
"observably indistinguishable from the plain view it derives from, only faster."
Almost everything else in `mv-maintenance.md` and `mv-constraints.md` is a
consequence of that one promise. A register that captures the promise and its
consequences is the difference between a reader trusting the MV subsystem and a
reader having to read 28,000 words to decide.

It is also the area with the most existing test coverage to point `guard:` at —
`test/logic/51*.sqllogic`, `maintained-table-refresh-revalidation.spec.ts`,
`mv-coarsening-collision-telemetry.spec.ts`, `lens-fd-contribution.spec.ts` — so
fewer entries will need `guard: none`.

## Seed list

Read against the code; drop, merge, and add freely. Expect **15–25** `MV-*`
entries.

**The core promise**

- A materialized view is observationally equivalent to the plain view of the same
  body, including reads-own-writes within the writing transaction. There is one
  maintenance model (row-time) and no refresh-policy knob.
- Maintenance runs inside the writing statement's transaction: it commits and rolls
  back in lockstep with the source write. There is no post-commit window and no
  asynchronous drift.
- MV-over-MV cascade completes within the same transaction as the originating
  write.

**Coverage and rejection**

- Maintenance coverage is **total** — no body is rejected for its *shape*. The
  incremental arms handle common shapes; the full-rebuild floor handles everything
  else. Precisely four create-time rejections exist, all non-shape: a
  non-deterministic body; a bag body with no provable unique key; a body with no
  relational output; a full-rebuild-only body over a source past the configured
  size threshold. Adding a fifth rejection is a breaking change to this invariant.
- A backward (maintenance-direction) cost gate picks the strategy; every strategy
  it may pick is sound, so the gate can only be *slow*, never *wrong*.

**Identity and storage**

- A materialized view is exactly one schema object: an ordinary `TableSchema` under
  the view's own name, carrying a `TableDerivation`. One catalog name, one physical
  incarnation. `schema/derivation.ts`.
- The maintained relation is **always a set** — its logical key is the body's own
  key, so each body row maps to exactly one stored row.
- A maintained table occupies the ordinary table namespace and is read-only to
  direct DML.
- `bodyHash` is the identity of the definition; a definition change triggers a
  rebuild through the declarative-schema pipeline.

**The module boundary**

- The engine never reaches into the hosting module's internals. Every privileged
  operation — maintenance writes, the create/refresh fill, the enforcement scan —
  routes through the module-neutral backing-host capability. Every MV semantic
  holds regardless of which module hosts the table. `vtab/backing-host.ts`;
  `docs/mv-backing-host.md`. The one documented exception is the cross-module
  atomicity caveat — capture that as its own entry rather than as a hedge inside
  this one.

**Constraints and ingestion**

- Declared CHECK / FK / secondary-UNIQUE constraints on the maintained table are
  validated against *derived* rows at maintenance time.
  `core/derived-row-validator.ts`.
- A value-identical write is suppressed rather than applied — this is a semantic
  guarantee downstream change-feeds depend on, not an optimization.
- The external row-change ingestion seam's trust boundary: what the caller must
  guarantee, and what the engine will and will not re-validate.
  `docs/mv-ingestion.md`.
- Schema-change staleness: which DDL marks an MV stale, and that a rename
  propagates into the stored body rather than stranding it.
  `docs/mv-schema-change.md`.

## Edge cases & interactions

- **The four create-time rejections are load-bearing and enumerable.** Verify the
  list against the code before writing it down; the docs assert exactly four, and
  an invariant that miscounts is worse than none. If the code has five, the doc has
  drifted and that is a review finding (and possibly a `tickets/fix/` ticket) — say
  which.
- **"MV ≡ faster view" wants a real guard, and may not have one.** Look for a test
  that runs a query against a materialized view and the same body as a plain view
  and compares results. If one exists, it is the strongest `guard:` in the register.
  If none exists, this is the single best candidate for the one allowed
  `debt-` backlog ticket — a differential MV-vs-view harness in the shape of the
  existing optimizer differential-equivalence fuzz check. Do not build it here.
- **The cross-module atomicity caveat is a documented *limitation*, not an
  invariant.** It states that a two-module commit is not atomic. Record it as an
  invariant only if you can phrase it as something the code must uphold ("an MV
  hosted in a different module from its sources is …"); otherwise it stays prose in
  `mv-backing-host.md` and the register links to it. Resist the urge to file
  everything.
- **Overlap with the OPT area.** The MV read-side query rewrite and the coverage
  prover live in `docs/optimizer-rules.md` and are optimizer invariants
  (`OPT-*`), not MV ones — an MV that is never *used* by a rewrite is still
  correct. Do not double-file. Cross-reference by ID.
- **Overlap with `docs/lens.md`.** Covering structures are the substrate the lens
  layer builds on, and `lens.md` restates parts of the MV guarantee. `LENS-*` is
  reserved and stays empty; do not extract lens invariants here, and do not let an
  MV entry quietly become a lens entry.
- **Register size.** After both areas, `docs/invariants.md` should sit near
  5,000–6,000 words, comfortably under the 12,000-word cap. If it does not, entries
  are carrying rationale that belongs in the topic docs.
- **Back-link slug form.** `docs-invariants-optimizer` pinned how a `#mv-003`-style
  back-link resolves against a `### MV-003 — …` heading. Follow whatever it chose;
  do not invent a second convention.
- **Interaction with `docs-stability-tiers`** (separate plan ticket, proposes
  materialized views as **Beta**): a Beta feature having a register of guaranteed
  invariants is not a contradiction — the tier governs API/format stability, the
  register governs correctness. Say nothing about tiers here.
