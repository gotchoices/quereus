## Context

The predecessor ticket `planner-capability-brands` makes cross-class capability detection compiler-enforced (typed marker interfaces + brands) and establishes the canonical standard for node discrimination in the planner. This ticket writes that standard down where rule authors will read it, and adds a lint guard that keeps the corrected file honest.

**This must land after `planner-capability-brands`** — the doc describes the brand mechanism, and the lint tightening assumes `characteristics.ts` is already `as any`-free.

## The standard to document

Three questions, three mechanisms (see the predecessor ticket for the full rationale):

- **Concrete class identity → `instanceof`.** Type-sound, native narrowing, the dominant idiom. Correct/keep. Use it when a rule needs a *specific* class's API.
- **`nodeType` (`PlanNodeType`) → dispatch and serialization only.** Rule-manifest routing, plan formatter, EXPLAIN. Not a class-narrowing mechanism (not 1:1 with classes; no compiler narrowing).
- **Cross-class capability → branded marker interfaces + `CapabilityDetectors` guards in `characteristics.ts`.** Use it when a rule accepts *any* implementer of a capability (any join kind, any aggregate kind).
- **Physical characteristics → `PlanNode.physical` via `PlanNodeCharacteristics`.** Unchanged; the canonical source for readonly / ordering / FDs / determinism.

The intent distinction rule authors need at the keyboard: **need one class's specific API → `instanceof` that class; need "any node that can do X" → the capability guard.** Both are legitimate; they differ by intent, and that difference is not "drift."

## The doc problem to fix

`docs/optimizer-conventions.md` currently preaches "Characteristics Over Identity" and tells developers **"DON'T: Use `instanceof` checks"** (see its "For New Developers" and "Migration Patterns" sections). That guidance is the documented root cause of the drift the whole effort addresses — it steered authors away from the safe idiom toward the duck-typing that misfired. The framework README (`src/planner/framework/README.md`) simultaneously shows `if (!(node instanceof AggregateNode)) return null;` as the canonical rule shape — so the two docs already contradict each other, and reality (349 `instanceof` uses) follows the README, not the conventions doc.

Rewrite `optimizer-conventions.md` so it:

- States the three/four-mechanism standard above as the rule, with the "by intent" distinction.
- **Removes the blanket anti-`instanceof` guidance**; replaces it with "use `instanceof` for concrete class identity" and reserves the warning for the real anti-pattern: **duck-typed `as any` property-presence detectors** (`'foo' in node && typeof (node as any).foo === 'function'`).
- Documents the brand mechanism: to make a node detectable as a capability, declare `implements XCapable` and set the `is<Capability>Capable` brand; add the guard in `characteristics.ts`. The compiler enforces completeness.
- Notes the cross-bundle constraint (planner nodes are singletons in `@quereus/quereus`; plugins never receive them), so `instanceof` is safe here.
- Drops the now-inaccurate "Migration Patterns: From instanceof to Interface Checks" section (it advocated the reverse of the decided standard).

Reconcile the framework README example so it and the conventions doc agree (the `instanceof AggregateNode` rule shape is correct — keep it, and cross-reference the standard).

## The lint guard

Goal: prevent a new duck-typed `as any` node detector from being reintroduced into the canonical file, without a codebase-wide `any` purge (the broad planner `as any` audit is parked as backlog debt).

Minimum viable, concrete guard: a per-file override in `eslint.config.mjs` raising `@typescript-eslint/no-explicit-any` to `error` for `src/planner/framework/characteristics.ts` (the file is `as any`-free after the predecessor ticket, and its `eslint-disable` header is removed). Any reintroduced `as any` in the canonical detector file then fails lint.

A full custom "no duck-typed node detector" AST rule is **not** required here (it is hard to express precisely and is parked as backlog debt `debt-planner-duck-type-lint`). Do the file-scoped `no-explicit-any: error` override and document the convention; stop there.

## Edge cases & interactions

- The file-scoped `no-explicit-any: error` override must not accidentally widen to other planner files (they still carry legitimate `any` at `warn`). Scope the override `files` glob to exactly `src/planner/framework/characteristics.ts`.
- If, after the predecessor ticket, `characteristics.ts` still contains any unavoidable `as any`, the override will fail — that indicates the predecessor left a stray cast; fix the cast rather than lowering the rule.
- `docs/review.html` and other historical review artifacts mention the old patterns — leave point-in-time artifacts as-is (rewriting them falsifies history); only correct living guidance (`optimizer-conventions.md`, framework `README.md`).
