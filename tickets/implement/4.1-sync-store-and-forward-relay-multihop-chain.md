# Multi-hop relay chain (depth ≥ 3)

## Why

Current loop-freedom / convergence coverage in `store-and-forward-relay.spec.ts`
tops out at **depth 2** (the ping-pong spec exercises S → R1 → R2 → R1). The
convergence argument — original-HLC identity + a scalar per-peer watermark, with
**no** peer-membership oracle — holds for any depth, but a 3-or-more-hop chain to a
distant holder (S → R1 → R2 → R3 → H) is not explicitly tested. A regression that
broke convergence only at hop ≥ 3 (e.g. a re-stamp of the HLC on re-dispose, or
unbounded re-hold growth) would pass today's suite green.

## Design

Add a new `describe('multi-hop relay chain (depth >= 3)', ...)` block to the
existing `store-and-forward-relay.spec.ts`, reusing the file's **metadata-layer**
harness as-is (`makePeer`, `col`, `changeSet`, `pull`, `flatten`, `tablesIn`,
`maxSetHLC`, `RETIRED`). This is the same layer as the existing ping-pong /
3-peer specs — loop-freedom and convergence are CRDT-metadata properties; the
real-SQL materialization is covered separately by the e2e ticket. No new helpers
should be required; if one is, keep it local to the new block.

The chain:

```
  S            R1               R2               R3               H
  (straggler)  store-&-forward  store-&-forward  store-&-forward  holds `orders`
               no `orders`      no `orders`      no `orders`      (in basis)

  C = col(S, …, orders, …)
  applyChanges(S→R1)        pull(R1→R2)      pull(R2→R3)      pull(R3→H)
   R1 holds C forwardable    R2 re-disposes   R3 re-disposes   H APPLIES C
   (orig hlc+siteId)         → holds C        → holds C        → materialized
                              (one entry)      (one entry)      (columnVersions)
```

Settled facts (design is resolved — these are verified against the source):

- **Each non-holder re-disposes recursively.** `pull(from, to)` calls
  `to.manager.applyChanges(sets)`; a relay peer (`known` without `orders`,
  disposition `store-and-forward`) re-holds the forwarded change as exactly one
  forwardable entry (HLC-keyed idempotent), bumping its `forwarded` apply-time
  counter — exactly as the existing ping-pong R2 does.
- **Identity is the only thing carrying convergence.** C keeps S's `hlc` + `siteId`
  at every hop (no peer-membership oracle exists); the deterministic transaction id
  re-forms S's transaction at each peer. Assert at H that the materialized change's
  `hlc.siteId == S` and `compareHLC == 0` against the original.
- **From-zero pulls** (`since = undefined`) along the chain, so the
  scalar-watermark filter never suppresses C mid-chain (it only applies on the
  delta path). The holder H is the only peer with `orders` in `known`, so only H
  applies (materializes) rather than re-holding.
- Use **depth 3** relay peers (R1, R2, R3) — "three or more"; three is sufficient
  to exercise the >2 case without being arbitrary. Generate distinct siteIds per
  peer via `makePeer` (each gets a fresh `generateSiteId()`).

## Edge cases & interactions

- **Exactly once at the holder.** After the full chain, H's `orders`
  `columnVersions` carries C exactly once with S's origin HLC — not duplicated by
  the multiple hops, not re-stamped to any relay's siteId.
- **No unbounded growth.** Each of R1/R2/R3 holds **exactly one** forwardable entry
  for C at the end (`listForwardable().length === 1` per relay peer) — re-holds are
  HLC-keyed idempotent, so passing through N hops does not create N copies anywhere.
- **Chain quiescence.** After watermarks are advanced (re-`pull` each hop at the
  advanced `since`), no hop re-sends C — `tablesIn(sets)` excludes `RETIRED` on the
  second pass at every hop. The per-peer scalar watermark stops re-send after one
  exchange, at every depth.
- **Echo exclusion survives the chain.** A pull back toward the author
  (`Rk.getChangesSince(S)`) never re-offers C to S — the origin `siteId` is S at
  every hop, so echo exclusion holds regardless of how many relays C has traversed.
- **Back-relay idempotence (optional, mirrors ping-pong).** A reverse pull
  (`pull(R2, R1, undefined)`) offers C back to a peer that already holds it → still
  exactly one entry, no infinite re-hold. Confirms the depth-3 chain has the same
  loop-freedom the depth-2 ping-pong proves.
- **Holder relays onward.** After H applies C, `H.getChangesSince(neutral)` surfaces
  C from H's own change log with S's origin intact — a peer one hop past H would
  receive it correctly (the chain does not dead-end at the first holder).
