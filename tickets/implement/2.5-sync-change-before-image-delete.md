# Inline before-image on `RowDeletion`

Add an **optional** last-known row image to a synced deletion, mirroring the row
side of Lamina's before-image:

```ts
interface RowDeletion {
  // …existing: type, schema, table, pk, hlc…
  readonly priorRow?: Row;   // the row's last-known image at the origin (audit / undo)
}
```

Purely additive: producers may omit it, receivers ignore it when absent. Follows
the same **persist-so-it-reaches-the-wire** pattern established by
`sync-change-before-image-column` — read that ticket first; this one reuses its
serialization/round-trip approach for the delete path.

## Key design decision — persist `priorRow` in the `Tombstone`

`getChangesSince` re-resolves deletions from the `TombstoneStore` via
`resolveLogEntry` (`sync-manager-impl.ts:629`), so the before-image only reaches a
receiver if it is stored on the tombstone — exactly as the column prior is stored
on the cell version.

### Source of `priorRow` = the engine `oldRow`, not the column versions

At delete time `recordDataEvent` has `event.oldRow` in hand
(`sync-manager-impl.ts:387`) and is about to drop all column versions for the pk
(`deleteRowVersions`). Use `oldRow` directly — it is the column-ordered row image,
already on the event, **no extra read**. (Reassembling the row from per-column
versions would cost extra reads and add column-order mapping for no benefit.) When
`event.oldRow` is absent, omit `priorRow`.

## Extend `Tombstone` persistence

```ts
interface Tombstone {
  hlc: HLC;
  createdAt: number;
  priorRow?: Row;   // last-known row image before deletion
}
```

- `serializeTombstone`/`deserializeTombstone` (`tombstones.ts:25`): current format
  is a fixed 38 bytes (30 HLC + 8 `createdAt`). Add an absent-tolerant trailing
  encoding for `priorRow` — recommended: keep the 38-byte fixed head, then append
  a JSON array of the row's values, each through the existing column-version
  `encodeSqlValue`/`decodeSqlValue` helpers (re-export or import them) so
  `Uint8Array`/`bigint`/`null` cells round-trip. A tombstone with no `priorRow`
  serializes to the unchanged 38 bytes. Back-compat with on-disk data is **not**
  required (AGENTS.md).

## Plumbing

- `recordDataEvent` delete branch (`sync-manager-impl.ts:394`): pass
  `event.oldRow` into `setTombstoneBatch` so it lands on the persisted tombstone,
  and set `priorRow` on the inline `RowDeletion` pushed to `changes`.
- `setTombstoneBatch` / `setTombstone` (`tombstones.ts:72`/`:86`): accept an
  optional `priorRow` and persist it.
- `resolveLogEntry` delete branch (`sync-manager-impl.ts:636`): copy
  `tombstone.priorRow` onto the resolved `RowDeletion` (only when present).
- `collectAllChanges` tombstone branch (`sync-manager-impl.ts:688`): same copy.
- `commitDeleteMetadata` (`change-applicator.ts:494`): persist `change.priorRow`
  onto the tombstone so a receiver relaying via `getChangesSince` re-emits it.

## Edge cases & interactions

- **Delete without `oldRow`** (e.g. a relayed/synthesized delete event): omit
  `priorRow`; serialized tombstone stays the 38-byte form. Assert the wire
  `RowDeletion` has no `priorRow` key.
- **delete → reinsert → delete key reuse**: the existing tombstone dedup
  (`recordDataEvent` `sync-manager-impl.ts:401`, `commitDeleteMetadata`'s stale
  delete-entry cleanup) deletes the prior delete change-log entry. The surviving
  tombstone — and therefore the surviving `priorRow` — must be the **latest**
  delete's row image. Add a test covering the reuse sequence.
- **In-batch delete dedup** (`commitChangeMetadata` `deleteWinners` keepMaxHLC,
  `change-applicator.ts:433`): only the max-HLC delete's `priorRow` is written;
  losers are never persisted. Verify the winner's `priorRow` survives.
- **Storage cost**: every tombstone now optionally holds a full row image, retained
  until retention-horizon GC (`pruneTombstones`/`pruneExpired`). This is bounded
  but non-trivial for wide rows. It is acceptable for this pass (best-effort,
  additive); if a deployment wants to suppress it, a future opt-out config flag is
  the follow-up — file a `tickets/backlog/` note if you judge it worth gating, but
  do **not** add the flag here (scope creep).
- **Snapshot path**: snapshots reconstruct deletions/tombstones without history;
  deserialize of a `priorRow`-less tombstone must succeed (the absent-tolerant
  format covers this).
- **`Row` immutability**: store a value copy, not a live reference into the engine
  event, to avoid aliasing the engine's row buffer.
