## The finding (verified during planning — do NOT re-derive from scratch)

The plan ticket's design notes anticipated explicit routing work (map logical→basis
key columns, call `findIndexForConstraint`, thread the conflict action into a
bespoke lookup). **Verification proved none of that is needed.** The mechanism is
already complete by construction:

- The prover (`schema/lens-prover.ts` → `classifyKeyConstraint` →
  `findBasisCovering`) classifies a key `enforced-set-level{mode:'row-time'}`
  **only when** a matching **basis `UNIQUE` constraint** exists over the mapped
  basis columns *and* `_findRowTimeCoveringStructure` finds a non-stale row-time
  covering MV linked to that basis UC. So a row-time logical key is, by the
  classifier's own precondition, backed by a physically-enforced basis UC.
- A lens write re-plans against the **basis table by name** through the
  single-source spine (`rewriteViewInsert`/`rewriteViewUpdate` in
  `planner/mutation/single-source.ts`) — in basis-column terms (rename overrides
  handled by the column remap) and **preserving `onConflict` / `upsertClauses`**.
- The basis INSERT/UPDATE therefore hits the memory layer's
  `checkUniqueConstraints` (`vtab/memory/layer/manager.ts`), which for that basis
  UC resolves the covering structure (`findIndexForConstraint` prefers the
  row-time covering MV) and enforces via `checkUniqueViaMaterializedView` — an
  O(log n) existence lookup that **honors `ABORT` / `IGNORE` / `REPLACE`** (the
  "enforcement through a covering MV" path, `docs/materialized-views.md`).

Net: detection (O(log n)) and conflict resolution both fire for free. The
commit-time collector (`collectLensSetLevelConstraints`) already returns `[]` for
row-time (only `mode:'commit-time'` emits), and `rejectLensSetLevelConflictResolution`
already declines to reject row-time conflict resolution (it gates only on
`hasCommitTimeSetLevelObligation`). Both are correct as-is.

### Empirical confirmation (throwaway spec, run during planning, since deleted)

Memory-backed, full `apply schema` pipeline. All passed:

- **Detection**: `unique(email)` on basis `y.u` + covering MV; logical `x.u`
  `unique(email)`. A duplicate `insert into x.u` ABORTed (`UNIQUE constraint
  failed: u (email)`); the original survived.
- **`insert or replace`** through the lens evicted the conflicting row and landed
  the new one (`{id:5, email:'a@x'}` replaced `{id:1,…}`).
- **`insert or ignore`** through the lens skipped the new row; the original
  survived.
- **Rename override** (`select id, mail as email`): basis UC on `mail` + MV;
  detection (`UNIQUE constraint failed: u (mail)`) and replace both worked — the
  single-source rewrite maps the column, the basis UC enforces in basis terms.
- **Re-keyed PK** (logical PK `code` over basis `unique(code)` + MV, basis PK on
  `id`): detection and replace both worked (replace evicted the old `code='A'`
  row and landed the new one).

### Minor cosmetic note (decide, don't over-build)

The duplicate error through the lens names the **basis** relation/columns
(`UNIQUE constraint failed: u (email)` / `… (mail)`), not the logical constraint —
identical to physical-UNIQUE-through-covering-MV semantics. This is arguably
correct (the lens enforces *through* the basis structure). If logical-constraint
naming is judged worth it, file a small `tickets/backlog/` ticket
(`lens-set-level-error-logical-naming`) rather than expanding this one.

## Key tests (TDD — assert behavior, not just collector output)

Add to `packages/quereus/test/lens-enforcement.spec.ts`, ideally a new
`describe('lens enforcement: set-level (unique / PK) row-time at the write boundary')`
block sibling to the commit-time block. Mirror the commit-time suite's
deploy-through-`apply schema` style. Each scenario needs a basis `unique(...)` +
an explicit `create materialized view … order by <key cols>` so the key
classifies row-time.

- **row-time unique → collector emits nothing** — `collectLensSetLevelConstraints(slot)`
  returns `[]` (an existing test already asserts this; keep/extend it as the unit
  anchor for "row-time is not the count-subquery class").
- **detection aborts a duplicate** — two distinct keys ok; a third duplicating an
  existing key ABORTs; the original survives. Expect `/unique|constraint/i`.
- **`insert or replace` resolves through the lens** — replacing a duplicate evicts
  the prior row and lands the new one (assert the post-state by basis read *and*
  logical read; e.g. only the new `id` remains for that key).
- **`insert or ignore` resolves through the lens** — the duplicate is silently
  skipped; the original row is untouched; a fresh-key `or ignore` still lands.
- **explicit `or abort`** — ABORTs on a duplicate (consistent with plain insert).
- **NOT rejected** — `insert or replace` / `or ignore` / an `on conflict … do …`
  upsert against a **row-time** key must **not** raise the
  `lens-set-level-conflict-resolution` diagnostic (the negative of the commit-time
  rejection test at `lens-enforcement.spec.ts` ~line 625). This is the test that
  pins "row-time unlocks conflict resolution".
- **UPDATE that creates a duplicate ABORTs** — a key-changing update through the
  lens that collides ABORTs; one that keeps the key unique succeeds.
- **NULL-distinct holds** — multiple NULL-key rows are all accepted (the basis UC
  skips NULL columns and the covering MV is `where <col> is not null`).
- **rename override** — logical `email` → basis `mail`; detection + replace fire in
  basis terms (covering MV over `mail`). Confirms the single-source remap, not a
  lens collector, carries the mapping.
- **re-keyed PK** — logical PK on a non-basis-PK column backed by a basis
  `unique(<col>)` + MV; detection + replace work.
- **composition** — a logical table with a row-local `check` (rides
  `extraConstraints`) AND a row-time `unique` on the same write: the check ABORTs a
  bad row, the row-time key ABORTs a duplicate, a clean row lands. (Optional but
  cheap; pins "compose with row-local checks".)
- **composition caveat (document, maybe assert)** — a slot carrying *both* a
  commit-time key and a row-time key: `insert or replace` is still rejected by
  `rejectLensSetLevelConflictResolution` (it fires on *any* commit-time
  obligation). This is sound (the commit-time key genuinely cannot replace-resolve)
  and is the intended conservative behavior — note it in a test comment or the
  review handoff rather than "fixing" it.
