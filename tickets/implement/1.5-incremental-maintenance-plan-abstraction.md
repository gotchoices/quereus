## Purpose

This is the first implement follow-on from the substrate spike
(`incremental-maintenance-substrate-spike`). It makes one change of substance: the
internal type `RowTimeMaintenancePlan` is promoted into the `MaintenancePlan` tagged
union as its `'inverse-projection'` arm, with **zero change to semantics**. Every
field moves verbatim; every call site continues to reach exactly the same runtime
path. The two other arms added here — `'full-rebuild'` and a stub `'residual-recompute'`
shell — are **unreachable** until `incremental-maintenance-cost-gate` (1.6) and
`materialized-view-rowtime-general-bodies` (3) wire selection logic.

The second deliverable is the **maintenance-equivalence property harness**: a
`fast-check`-style oracle that asserts `read(MV_backing) == evaluate(body)` after
every source mutation and after rollback, for a zoo of eligible body shapes. This
harness is the regression net every subsequent ticket runs against; it must be
green against the shipped covering-index shapes before any new shape is admitted.

## What changes (and what does not)

### `RowTimeMaintenancePlan` → `MaintenancePlan` union

`core/database-materialized-views.ts` currently holds a private interface
`RowTimeMaintenancePlan` with fields `mv`, `sourceBase`, `backingSchema`,
`backingTableName`, `backingPkDefinition`, `projectionSourceCols`, and an optional
`predicate`. The spike's `MaintenancePlan` design translates those fields unchanged
into the `'inverse-projection'` arm:

```
type MaintenancePlan =
  | { kind: 'inverse-projection';          // ← RowTimeMaintenancePlan, verbatim
      mv: MaterializedViewSchema;
      sourceBase: string;
      backingSchema: string;
      backingTableName: string;
      backingPkDefinition: ReadonlyArray<{ index: number; desc?: boolean; collation?: string }>;
      projectionSourceCols: number[];
      predicate?: CompiledPredicate }
  | { kind: 'full-rebuild';                // always-correct escape; unreachable until cost gate
      mv: MaterializedViewSchema;
      sourceBase: string;
      backingSchema: string;
      backingTableName: string }
  | { kind: 'residual-recompute';          // stub; unreachable until general-bodies
      mv: MaterializedViewSchema;
      sourceBase: string;
      backingSchema: string;
      backingTableName: string;
      binding: BindingMode;
      degradeToRebuild: boolean };
```

The private Maps in `MaterializedViewManager` change their value type from
`RowTimeMaintenancePlan` to `MaintenancePlan`. `applyRowTimeChange` is renamed
`applyMaintenancePlan` and dispatches on `plan.kind`; for `'inverse-projection'`
the body is byte-for-byte the current implementation. The `'full-rebuild'` and
`'residual-recompute'` branches throw `StatusCode.INTERNAL` (unreachable guards) so
any accidental routing is loud.

`buildRowTimePlan` is renamed `buildMaintenancePlan` and continues to return only
`'inverse-projection'` plans — its shape allowlist is unchanged. The public entry
points `registerMaterializedView`, `unregisterMaterializedView`, and
`maintainRowTime` keep the same signatures modulo the internal type.

### `BindingMode` import

`binding-extractor.ts` already exports `BindingMode` (used by assertions and the
`DeltaExecutor`). The `'residual-recompute'` stub carries a `binding: BindingMode`
field to reserve the convergence point the spike names — the two engines sharing one
binding analysis. No new import cycles: `database-materialized-views.ts` already
imports from `planner/analysis/` indirectly; the direct import of `BindingMode` is
additive.

### `runtime/emit/dml-executor.ts`

The DML executor calls `maintainRowTime` on the `MaterializedViewManager`. Any
public method rename must be reflected here. No logic change.

### Docs forward-reference

`docs/materialized-views.md` and `docs/incremental-maintenance.md` receive additive
cross-references noting that `RowTimeMaintenancePlan` is now the `'inverse-projection'`
arm of `MaintenancePlan` and pointing at this ticket's harness as the correctness
oracle. No section is rewritten; no spike outcome is claimed.

## The property harness

A new spec at `packages/quereus/test/incremental/maintenance-equivalence.spec.ts`
(create the directory if absent) uses `fast-check` in the style of
`test/property.spec.ts` and the Key-Soundness Tier-1 harness:

```
for each (bodyShape, mutationSequence) pair drawn from arbitraries:
  1. CREATE TABLE source + CREATE MATERIALIZED VIEW mv AS <body>
  2. Apply a random sequence of insert/update/delete to source (within a transaction)
  3. Assert:  rows from (select * from mv)
           == rows from (select <body-columns> from source <body-where>)
     (as multisets, order-insensitive)
  4. ROLLBACK; assert again (maintenance never diverges after rollback)
  5. Repeat for numRuns iterations
```

Body shapes covered in this ticket (covering-index only — currently eligible):

- Single-column passthrough projection, no predicate.
- Multi-column passthrough projection with a partial `WHERE` predicate, including
  predicate-scope transition (a row moves in/out of the predicate's true region).
- Projection that reorders source columns (a column permutation, not identity).
- Key-changing `UPDATE` (the source PK itself changes value).

Shapes that are **not** yet covered (future tickets extend the harness):

- Aggregates, joins, lateral TVFs — `'residual-recompute'` arm, not yet wired.
- MV-over-MV — cascade ticket extends the harness to chain assertions.

The harness must run under `yarn test` (Mocha + ts-node/esm, not Vitest) without
separate configuration. It is the oracle the Z-set PoC (specified in the spike) also
runs against when that PoC executes.

## What does NOT change

- The shape allowlist in `buildMaintenancePlan` (formerly `buildRowTimePlan`). The
  gate remains identical: covering-index shape only. `'full-rebuild'` and
  `'residual-recompute'` arms are unreachable from the builder until the cost gate
  (1.6) and general-bodies (3) tickets land.
- `deltaPerRowFallbackRatio` in `delta-executor.ts` — that subsumption is the cost
  gate ticket (1.6), not this one.
- The synchronous in-transaction application policy. Row-time maintenance continues
  to fire at the DML write boundary. No commit-time path is introduced.
- `53-materialized-views-rowtime.sqllogic` must stay green. The rename is a pure
  internal refactor; no user-visible SQL surface changes.
- `materialized-view-diagnostics.spec.ts` must stay green. The create-time gate
  diagnostic wording is unchanged.

## Correctness guarantee

After this ticket the invariant encoded in the harness is:

> For every eligible row-time MV, `applyMaintenancePlan` with an `'inverse-projection'`
> plan produces the same backing-table state as the previous `applyRowTimeChange`
> would have on the same input. The harness asserts this for random mutations
> including rollback, predicate-scope transitions, and key-changing updates.

## Relationship to adjacent tickets

  selector that may route to `'full-rebuild'` instead of always returning
  `'inverse-projection'`. The `'full-rebuild'` arm added here is its target.

  arm's actual body on the abstraction this ticket introduces and extends the harness
  with aggregate/join body shapes.

- `optimizer-keyed-cross-product-join-keys` (2) is independent; it supplies the
  lateral-TVF backing PK to `keysOf` that the `'prefix-delete'` arm (not introduced
  here) will consume.
