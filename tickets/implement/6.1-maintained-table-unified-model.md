# Unified maintained-table schema model

Part 1 of the maintained-table chain (see `docs/materialized-views.md`
§ Current limitations "First-class derivation lifecycle" and
`docs/migration.md`). Design resolution from the plan stage:

> **A table is a stored relation; a derivation is an optional maintenance
> contract attached to it.** One `TableSchema`, one catalog name, one
> physical incarnation. Today's MV (`MaterializedViewSchema` in
> `Schema.materializedViews` + hidden `_mv_<name>` backing `TableSchema` in
> `Schema.tables`) becomes a single `TableSchema` named `<name>` carrying
> `derivation?: TableDerivation`.

The dual-registration alternative (first-class-named backing + separate MV
record) was considered and rejected: it keeps two catalog names for one
relation, retains the tri-directional name-disjointness rules, and still
needs every attach/detach/differ change on top — while for the migration
pattern the natural authoring collapses both names into one anyway.
Unification makes every later lifecycle verb (attach/detach, ticket
`maintained-table-attach-detach-verbs`) a pure catalog flip. There is no
persisted-catalog compatibility constraint (project rule: no backwards
compat yet).

**This ticket is behavior parity.** No new syntax, no attach/detach verbs,
no differ transition recognition — those are follow-on tickets. `create
materialized view` / `refresh materialized view` / `drop materialized view`
keep working and remain the canonical DDL render for now; what changes is
the catalog representation, the table's name (the MV name itself — no
`_mv_` prefix), and its user-facing visibility (one catalog entry, visible
as itself).

## The schema model

```ts
// schema/derivation.ts (new; or co-located in view.ts — avoid import cycles)
export interface TableDerivation {
	/** The parsed body AST — any relation-producing QueryExpr. */
	selectAst: AST.QueryExpr;
	/** Explicit output-column rename list from the MV-sugar form (`mv(a, b)`).
	 *  Retained verbatim for bodyHash parity and DDL round-trip; the table's
	 *  own columns are the authoritative output names. */
	columns?: ReadonlyArray<string>;
	insertDefaults?: ReadonlyArray<AST.ViewInsertDefault>;
	/** Same formula as today: computeBodyHash(viewDefinitionToCanonicalString(
	 *  columns, selectAst, insertDefaults)). Differ keying unchanged this ticket. */
	bodyHash: string;
	/** Was MaterializedViewSchema.primaryKey — the body's logical key. The
	 *  table's own primaryKeyDefinition stays the physical (order-by-seeded) key. */
	logicalKey: ReadonlyArray<{ index: number; desc: boolean }>;
	ordering?: ReadonlyArray<{ index: number; desc: boolean }>;
	coarsenedKey?: CoarsenedKeyInfo;
	sourceTables: ReadonlyArray<string>;
	stale?: boolean;            // runtime state, never serialized
	sourceScope?: ChangeScope;  // runtime state, never serialized
	covers?: { schemaName: string; tableName: string; constraintName?: string };
}
```

Fields that **die** rather than move: `name`/`schemaName` (the table's),
`backingTableName` (== table name), `backingModuleName`/`backingModuleArgs`
(the table itself lives in its module — `using store(...)` on the MV-sugar
create maps to the table's `vtabModuleName`/`vtabArgs`, with the same
text-PK collation reconciliation `create table … using store` performs;
`normalizeBackingModule` survives only as the differ's module-comparison
normalizer), `sql` (the table's `sql` carries the DDL), `tags` (the
table's), `origin` (the implicit-from-unique-constraint variant was never
registered; keep the vocabulary where the covering-structure docs need it).

Registration consequences:

- `Schema.materializedViews`, `addMaterializedView`, `getMaterializedView`,
  `removeMaterializedView`, `getAllMaterializedViews`, the tri-directional
  name-disjointness checks in `addTable`/`addView`, and
  `getMaterializedViewByBackingTable` all go away. Provide thin manager
  accessors (e.g. `getMaintainedTable(schema, name)` → table iff
  `derivation` present, `getAllMaintainedTables()`) so call-site churn
  stays mechanical. A facade kept for low-value call sites is acceptable;
  the storage model must not be.
- `backingTableNameFor` and every `_mv_` assumption dies: catalog hiding in
  `catalog.ts` (one visible entry now), the rehydrate adopt gate that
  pattern-matches `_mv_` sources (becomes "source table has a derivation ⇒
  must be in the adopted set"), the store stale-at-close marker payload
  (names are table names now).
- Query resolution: `select … from mv` resolves through the **ordinary
  table path** — the `mvSchema` FROM case in `building/select.ts` and the
  reference-rewrite to the backing collapse. Change-scope analysis still
  substitutes `derivation.sourceScope` for a derivation-bearing table
  reference (the table is still never user-change-logged).
- Write-through: the three DML builders' `getView(…) ?? getMaterializedView(…)`
  dispatch becomes `getView(…) ?? (table with derivation)` → the same
  `buildViewMutation` rewrite-to-source path. **The plain-table DML path
  must check `derivation` before treating the name as an ordinary table** —
  this dispatch-order is the highest-risk seam of the refactor.
- Maintenance manager: `MaterializedViewManager` keys (`rowTime`,
  `rowTimeBySource`) re-key from MV name to qualified table name; the
  MV-over-MV cascade looks up consumers by the maintained table's own name
  (no backing-name indirection); the synthetic cached-plan-invalidation
  `table_modified` event now simply names the table itself.
- Covering structures: `CoveringStructure`'s `'materialized-view'` variant
  carries the maintained `TableSchema` (+ derivation); the constraint's
  `coveringStructureName` forward pointer and
  `resolveCoveringStructureName` fallback re-key to the table name.
- Events: keep the `materialized_view_added/removed/modified/refreshed`
  event channel (re-keyed to the table name) so store catalog persistence
  keeps re-saving DDL on derivation changes; one catalog entry per
  maintained table (the create fires `table_added` for the same table —
  ensure the store listener persists exactly one bundle whose DDL is the
  `create materialized view` form this ticket).
- Rehydrate/adopt (`importMaterializedView`, `tryAdoptPreExistingBacking`,
  `adoptMaterializedView`, store `rehydrateCatalog`): gate 1 ("table exists
  at `_mv_<name>` in the declared module") becomes "a table exists at
  `<name>` in the entry's module" — trivially the entry's own pre-rehydrated
  table. Shape gate, source-module gate, adopted-upstream gate, and the
  clean-shutdown/stale-at-close gates carry over unchanged in meaning.

## Edge cases & interactions

- **Namespace collision**: `create materialized view mv` when a table or
  view `mv` exists ⇒ error through the ordinary table-name collision path
  (one namespace now). Pin both directions (`create table` over an existing
  maintained table too).
- **Write-through dispatch order**: DML naming a maintained table must hit
  the view-mutation rewrite, never the direct table write; DML naming a
  plain table is unaffected. RETURNING-through-MV and MV-over-MV
  write-through rejections keep their diagnostics.
- **Read path / rewrite suppression**: planning a derivation's own body
  (create fill, refresh, maintenance compile) must not resolve the body's
  source references through the read-side MV rewrite into the very table
  being populated (`withSuppressedMaterializedViewRewrite` re-keyed); a
  body that references the maintained table being created is now *lexically
  possible* (the name exists as a table mid-create only after registration)
  — the create path must register catalog state in an order that keeps
  self-reference impossible, or reject it.
- **MV-over-MV chains**: a consumer's body now names the producer table
  directly (no backing rewrite). Cascade dispatch (`rowTimeBySource` on the
  producer's name), depth guard, deferred-rebuild flush rounds, and the
  store rehydrate fixpoint (per-entry plan-failure ⇒ backing preserved as
  plain table, retried next round) must all hold. The "preserved as plain
  table" failure posture is now literally "table without derivation" —
  cleaner; assert it.
- **drop table X** where X is maintained ⇒ drops table + derivation
  (today's `DROP TABLE` redirect-to-`DROP MATERIALIZED VIEW` dies); `drop
  materialized view X` on a derivation-less table ⇒ error redirecting to
  `drop table`. `drop table` of a *source* leaves dependents stale
  (unchanged semantics).
- **Rename**: `alter table … rename to` on a maintained table is now an
  ordinary table rename (the MV-record re-key dies; consumers' bodies
  rewrite through the existing propagation); renames of a *source* propagate
  into derivation bodies exactly as today (`propagate*ToMaterializedViews`,
  `restoreUnaffectedMaterializedViews`, backing-column rename pass — all
  operating on the unified table). Staleness-discipline snapshots
  (pre-stale not cleared, statement-own staleness restored) carry over.
- **Refresh**: shape-match fast path (`replaceContents`) and the
  shape-shift rebuild keep today's behavior this ticket — including the
  drop+recreate on shape shift (identity-preserving reshape is ticket
  `maintained-table-identity-preserving-reshape`); but the drop+recreate
  must re-create under the *same table name* and keep one catalog entry.
- **Enforcement**: covering-MV UNIQUE resolution (`_findRowTimeCoveringStructure`
  → `_lookupCoveringConflicts` prefix scan), eviction reporting
  (`UpdateResult.evictedRows`), and the store-table fallback resolution all
  re-key; the per-row-immediate vs deferred-full-rebuild eligibility gate
  is untouched.
- **Watch / change scope**: `Database.watch` on a maintained table projects
  to its sources via `derivation.sourceScope`; a watch on a plain table is
  untouched; `getChangeScope()` parity.
- **Store rehydrate divergence healing**: crash-window refill, stale-at-close
  per-entry trust, marker consume-once, adopted-upstream composition — all
  re-keyed, semantics identical; `yarn test:store` (or at minimum the store
  package's own rehydrate suites) must pass.
- **Statement-cache invalidation**: a cached `select` plan on a maintained
  table now depends directly on that table; the manager's synthetic
  `table_modified` on a qualifying source change must still force stale
  re-validation through the build-time guard.
- **Tests referencing `_mv_`** (12 files: coarsened-backing-key,
  declarative-equivalence, maintenance-equivalence, change-scope.spec,
  materialized-view-diagnostics, materialized-view-refresh-reshape,
  plan/materialized-view-plan, plan/materialized-view-rewrite-plan,
  query-rewrite-equivalence, query-rewrite-join, query-rewrite,
  view-mv-ddl-persistence) — update expectations to the unified name;
  plan snapshots show a scan of `mv` itself now.
