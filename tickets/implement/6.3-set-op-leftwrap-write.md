description: Write-side foundation for parallel-sibling set-op membership views. Make the parenthesized LEFT subtree operand (`select * from (<compound>) as values_N`) a recursively-WRITABLE subtree operand in the mutation path, so data-column UPDATE, DELETE, and `set <subtreeFlag> = false` fan out through it exactly like the (already-writable) right subtree. Closes the deferred 6.0 backlog gap (`set-op-membership-nested` write branch over a `select * from (compound)` wrapper). The "sum" model; the "product" model is the blocked `set-op-product-coordinate-model`.
prereq: set-op-leftwrap-arity
files: packages/quereus/src/planner/mutation/set-op.ts, packages/quereus/src/func/builtins/schema.ts, packages/quereus/test/property.spec.ts, docs/view-updateability.md
----

## Why (verified, not theorized)

Even a flag-less parallel-sibling view that plans + reads fine (P1, see 6.2) cannot be written:

```
delete from P1 where id = 2
  → cannot write through view 'P1': the left branch uses 'select *';
    list its columns explicitly so each maps to a writable branch column
```

The mutation path (`set-op.ts`) works off the **view AST**, not the plan. `leftBranchSelect(sel)`
returns the parser wrapper `select * from (<inner compound>) as values_N`; `buildBranch` then
calls `branchColumnNames` → `tryBranchColumnNames` returns `null` for the `select *` leg →
reject. So the LEFT subtree of every parallel-sibling shape is unaddressable by the write path,
even though the right subtree (a direct compound) fans out correctly. This is exactly the gap the
6.0 review filed as `tickets/backlog/set-op-membership-nested` ("make the membership WRITE
branches writable when an operand is a `select * from (compound)` wrapper").

This ticket makes the wrapped left operand a first-class subtree operand for the **unambiguous
fan-out operations** (data UPDATE / DELETE / `set <subtreeFlag> = false`), reusing 6.1's
recursive `fanBranchDataUpdate` / `fanBranchDelete` and the ONE up-front Halloween-safe capture.
It does NOT add coordinate addressing, `set <subtreeFlag> = true`, or surfaced-inner-flag writes
— those are the product model (blocked `set-op-product-coordinate-model`) and continue to reject
cleanly via the existing `set-op-membership-nested` diagnostics.

## Design

Reuse the shared `unwrapPassthroughSubquery` helper introduced in 6.2 (`set-op-leftwrap-arity`).
The write path must treat a wrapped operand identically to a direct compound operand:

- **`buildBranch`** (and `analyzeSetOpBranches`): when `branchSelect` is a pure passthrough
  wrapper, set `branch.view.selectAst = unwrap(branchSelect)` (the inner compound) and derive
  `isNested` / `dataColNames` from the inner. Then the existing recursion (`fanBranchDataUpdate`
  / `fanBranchDelete` descending `analyzeSetOpBranches`) fans through the left subtree to its
  leaves with no further change — the inner branch view-likes (`__setop_left` / `__setop_right`)
  are built from the unwrapped inner exactly as the right subtree already is.
- **`surfacedInnerFlagNames` / `collectSubtreeFlagNames`**: today they walk only the RIGHT
  operand ("the left operand is never a compound in SQL"). That assumption is now false — a
  wrapped LEFT operand carries inner flags too. Walk BOTH operands (unwrapping the left wrapper),
  so the left subtree's inner flags (e.g. `inAL`, `inBL`) surface as readable-but-non-writable
  columns and a write to one rejects with the clean `set-op-membership-nested` diagnostic (NOT
  `unknown-view-column`).
- **`setOpHasSubtreeOperand`**: today checks only the right operand for `is_insertable_into = NO`.
  Extend to the (unwrapped) left operand so a view with a wrapped-left subtree also reports
  `is_insertable_into = NO`.
- **`isOperandWritable` / `isSetOpBodyWritable`** (static surface): unwrap the left operand
  before classifying, so `view_info` / `column_info` report YES/YES for data-update + delete and
  agree with the dynamic accept. The `except` / `intersect` gating (`isUnionLikeSubtree` +
  boundary-flag requirement, `set-op-membership-nested-except`) must apply to the unwrapped left
  subtree exactly as it does to the right.
- **`schema.ts`** (`is_updatable` / `is_deletable` / `is_insertable_into`): mirror the dynamic
  surface — a parallel-sibling view with writable subtrees reports updatable + deletable, not
  insertable; surfaced inner flags non-updatable.

## Edge cases & interactions

- **Flag-less parallel siblings (P1).** `delete from P1`, `update P1 set x=…` fan to all four
  leaves (both subtrees, both sides). The full-data-tuple `exists` correlation restricts each
  leaf op to its resident rows. Pin PutGet: row removed from / updated in every base it was in.
- **`set <outerSubtreeFlag> = false`** (`set inLsub = false`) — delete fan-out INTO the left
  subtree's leaves (the case that previously rejected on the wrapper). Must drop the row from the
  left subtree's resident leaves only.
- **`set <outerSubtreeFlag> = true`** (`set inLsub = true`) — still **rejected** (a multi-leaf
  subtree insert has no single target leaf; product addressing). Keep the `set-op-membership-nested`
  diagnostic; only now it must fire for the LEFT subtree too, not slip through.
- **Surfaced left-subtree inner-flag write** (`set inAL = true`) — **rejected** with
  `set-op-membership-nested` (product addressing), not `unknown-view-column`. This is the new
  collection-from-left requirement.
- **`except` / `intersect` left subtree** — fan-out is membership-gated on the captured boundary
  flag (`set-op-membership-nested-except`); a flag-less non-union left subtree stays deferred
  (rejected). The gating logic (`gateFlagForNonUnionSubtree` / `accumulateInnerGate`) must see the
  unwrapped left subtree's operator, not the wrapper.
- **Halloween safety at depth.** The single up-front capture is reused for the left subtree's
  leaves exactly as for the right; no second capture. `k.*` (outer data cols) vs `branch.*` (leaf
  data cols) positional alignment is guaranteed by the data-only arity invariant (now correct for
  the left, post-6.2) and `buildBranch`'s arity check.
- **Depth-3 left nesting** — data + delete fan to the deepest left-most leaf.
- **Contradiction guard** (`set inLsub = false` AND a data assignment that fans to that subtree)
  — the existing `conflicting-assignment` reject must fire for the left subtree too.
- **Binary (non-nested) and right-only-nested views** — unaffected; the unwrap is a no-op when no
  wrapper is present. Full suite green.
