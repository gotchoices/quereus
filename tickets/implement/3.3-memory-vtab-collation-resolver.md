# Memory vtab resolves collations against its own database

## Problem

`MemoryTable` and everything under `vtab/memory/` turns collation names into
comparison functions by calling the **process-global** `resolveCollation` from
`util/comparison.ts`, or by passing a name into the three-argument
`compareSqlValues(a, b, collationName)` (which does the same lookup internally).
That registry only ever holds `BINARY`, `NOCASE`, and `RTRIM`.

A collation registered with `db.registerCollation('REVERSE', …)` is therefore
invisible to memory-table primary keys, secondary indexes, index range seeks, and
plan-filter equality checks. They silently sort and seek by raw byte order.

Every one of these sites is reachable from a `MemoryTable`, and `MemoryTable
extends VirtualTable`, which has a `db: Database`. So the resolver is always
obtainable — it just is not threaded.

## Design

`3.2-collation-resolver-seam` provides `db.getCollationResolver(): CollationResolver`
(a `(name) => CollationFunction` that throws on an unregistered name).

**Thread the resolver down the layer stack; do not change `ScanPlan`'s shape.**

`ScanPlan` (`layer/scan-plan.ts`) stores collation *names* (`equalityPrefixCollations:
string[]`, `boundCollation?: string`). Plans are constructed, logged, and compared in
tests as plain data. Keep them as names — do not swap in function references.

Instead, carry the resolver on the objects that already flow to every comparison
site:

- **`Layer` interface** — add `readonly collationResolver: CollationResolver`.
  `BaseLayer` and `TransactionLayer` both receive `tableSchema` at construction;
  give them the resolver at the same time. `scanLayer(layer, plan)` and
  `matchesPlan(...)` then read `layer.collationResolver` instead of taking a new
  argument through the recursion.
- **`MemoryIndex`** — add a `collationResolver` constructor parameter, used in
  `createSingleColumnKeyFunctions` / `createCompositeColumnKeyFunctions`.
- **`createPrimaryKeyFunctions(schema, collationResolver)`** — add the parameter.
- **`LayerManager`** — add a `collationResolver` constructor parameter, supplied by
  `MemoryTable` as `this.db.getCollationResolver()`; forward to `BaseLayer`,
  `TransactionLayer`, `createPrimaryKeyFunctions`.

Where `plan-filter.ts` / `scan-layer.ts` currently call
`compareSqlValues(a, b, name)`, replace with `compareSqlValuesFast(a, b, fn)` and
resolve `fn` **once per scan**, not per row: hoist
`plan.equalityPrefixCollations.map(resolver)` and `resolver(plan.boundCollation)`
above the row loop. `matchesPlan` is called per row from a loop that already has
the layer — resolve in the caller and pass the resolved functions in, or memoize
on the plan object via a `WeakMap`. Prefer the explicit hoist; a per-row resolver
call would regress `test/performance-sentinels.spec.ts`.

`compareSqlValues(a, b)` with no collation argument stays as-is (BINARY); only the
name-passing calls change.

After this ticket, `vtab/memory/` imports nothing from the deprecated global
registry.

## Edge cases & interactions

- **`scanLayer` recursion** (`scan-layer.ts` ~50, ~70) recurses on the same layer
  with a narrowed plan; the resolver must be read from the layer each time, or the
  hoisted functions passed down. Do not resolve twice per recursion level.
- **`scanLayer` builds its own `createPrimaryKeyFunctions(seekSchema)`** (~45) for
  the seek key. That `seekSchema` may be an index schema, not the table schema —
  it still needs the layer's resolver.
- **Layer inheritance:** `TransactionLayer` inherits its parent's secondary BTrees
  (`new MemoryIndex(..., parentSecondaryTree)`). The child's comparator must be
  built from the *same* collation function as the parent's, or the inherited tree's
  ordering is inconsistent with the child's `compareKeys`. Since both resolve from
  the same database resolver, this holds — add a test that a transaction layer over
  a `REVERSE`-collated index yields correctly ordered rows.
- **`ALTER COLUMN … SET COLLATE`** forces `rebuildAllSecondaryIndexes` /
  `rebuildPrimaryTreeStrict` (see the `primaryKeyComparator` doc in `index.ts` ~39).
  Confirm those rebuild paths construct the new `MemoryIndex` with the resolver.
  Covered by `test/logic/41.7-alter-column-collate.sqllogic`.
- **Unknown collation now throws.** A `MemoryIndex` built for a column declaring an
  unregistered collation raises at construction (i.e. at `xConnect` / index
  creation) rather than silently byte-ordering. Verify
  `test/logic/102.1-unique-edge-cases.sqllogic` (which pins the `Unknown collation`
  DDL error) still passes and that the new error does not fire *earlier* than the
  DDL validation error, which would change the pinned message.
- **Descending indexes:** `descMultiplier` is applied after the collation; the
  resolver change must not touch that. `test/vtab/scan-layer-descending.spec.ts`
  guards it.
- **Range-seek bound comparison** (`plan.boundCollation`) must use the *index*
  column's collation, not the table column's, where they differ (a
  `CREATE INDEX … COLLATE NOCASE` over a `BINARY` column). `scan-plan.ts` ~348
  already picks `resolveColumnCollation(indexSchema, i)`. Preserve that; only the
  name→function step moves.
- **Direct-construction tests** (`test/vtab/memory-index-pk-value-identity.spec.ts`,
  `test/vtab/scan-layer-descending.spec.ts`, `test/performance-sentinels.spec.ts`)
  build `MemoryIndex` / `createPrimaryKeyFunctions` / `scanLayer` without a
  `Database`. They need a resolver. Give them a real `Database`'s resolver, or a
  tiny built-ins-only resolver helper — do **not** default the parameter to a
  global lookup, which would re-open the hole this ticket closes.
- **Empty table / empty index / null-only keys:** collation functions only see
  `string` arguments (`compareSameType` TEXT branch); nulls and numerics never reach
  them. Unchanged, but worth one test that a `REVERSE` collation is not called with
  `null`.
- **Two databases, same collation name, different comparators:** memory tables in
  each must sort per their own database. This is the headline regression test.

## Tests

New `test/vtab/memory-collation-per-database.spec.ts`:

- `db.registerCollation('REVERSE', (a,b) => a < b ? 1 : a > b ? -1 : 0)`, create a
  memory table with `t (k text collate REVERSE primary key)`, insert `'a','b','c'`,
  `select k from t` (primary-key order) → expects `c, b, a`. Today this returns
  `a, b, c` — that is the bug.
- Same with a secondary `create index … on t(v collate REVERSE)` and a query that
  drives an index scan; assert order.
- Range seek: `select … where k > 'b'` under `REVERSE` returns `a` (not `c`).
  Exercises `plan-filter` / `scan-layer` bound comparison.
- `UNIQUE` under `NOCASE` still rejects a case-variant duplicate (regression guard
  on the `manager.ts` unique paths).
- Two databases register `REVERSE` with opposite comparators; each memory table
  sorts by its own.
- A memory table whose column declares a collation registered on neither database
  raises `no such collation sequence`.
