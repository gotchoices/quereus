# Delete the process-global collation registry

## Problem

`util/comparison.ts` holds a module-level `Map<string, CollationFunction>` (~16)
plus `registerCollation` / `getCollation` / `resolveCollation` (~69–102), all
marked `@deprecated`. They are the process-wide collation registry: anything
registered there is visible to every open `Database`, and anything registered on a
`Database` is invisible to them.

By the time this ticket runs, the four prereq tickets have moved every reader onto
`Database.getCollationResolver()`. What remains is a loaded gun: the three-argument
`compareSqlValues(a, b, collationName)` overload still silently resolves against
the dead registry, so any *new* call site that passes a name gets `BINARY` for a
custom collation with no warning. Deleting the registry makes that a compile error
instead.

## Design

### Remove the registry

Delete from `util/comparison.ts`:

- the `collations` `Map` and its three built-in `.set(...)` calls,
- `registerCollation`,
- `getCollation`,
- `resolveCollation`.

`BINARY_COLLATION`, `NOCASE_COLLATION`, `RTRIM_COLLATION` stay — they are the
functions themselves, imported directly by `core/database.ts`,
`types/builtin-types.ts`, `types/temporal-types.ts`, `util/key-serializer.ts`, and
several emitters. So does `builtinCollationResolver` (added in
`3.2-collation-resolver-seam`), the switch-based built-ins-only lookup.
`normalizeCollationName` stays.

### Remove the name-taking comparison overloads

These three take a `collationName: string` and resolve it internally:

- `compareSqlValues(a, b, collationName = 'BINARY')` (~259)
- `compareWithOrderBy(a, b, direction, nullsOrdering, collationName)` (~397)
- `createOrderByComparator(direction, nullsOrdering, collationName)` (~425)

Drop the `collationName` parameter from all three. `compareSqlValues(a, b)` keeps
working as the BINARY two-argument form (~30 call sites rely on it and are correct
as-is). Any caller still passing a third argument becomes a type error — that is
the point, and it is how you verify the prereq tickets left nothing behind.

`compareWithOrderBy` and `createOrderByComparator` may have **no remaining callers**
once the name parameter is gone (their `*Fast` siblings are what emitters use).
Check with `find_references`; if a function is dead, delete it rather than leaving a
BINARY-only stub, and remove it from `src/index.ts`.

`compareRows(a, b)` (~499) calls `compareSqlValues(a[i], b[i])` — BINARY, unchanged.
Leave a `NOTE:` at `compareRows` stating it is deliberately BINARY-only (it backs
DISTINCT ordering in a BTree where the caller has already applied collation-aware
key normalization); collation-aware row comparison is `createCollationRowComparator`.

### Public API and docs

- `src/index.ts` (~75–77): drop the three exports. Ensure `CollationResolver`,
  `builtinCollationResolver`, `compareSqlValuesFast`, `createCollationRowComparator`,
  `createOrderByComparatorFast`, and the three `*_COLLATION` constants are exported —
  they are the replacement surface.
- `test/exports.spec.ts` (~23–25, ~104–106): remove the three assertions; add
  assertions for the replacement surface.
- `README.md` ~190 currently reads
  `**Collation Support** — \`registerCollation\`, \`getCollation\`, built-in collations …`.
  Rewrite: collations are registered per database with `db.registerCollation(name, func, options?)`;
  built-in `BINARY` / `NOCASE` / `RTRIM`; an unregistered collation name raises
  `no such collation sequence`.
- `docs/architecture.md`: if it names the collation seam, update it to say there is
  exactly one — `Database.getCollationResolver()`.

Backwards compatibility is explicitly not a concern for this repo (see `AGENTS.md`),
so no deprecation shim.

## Edge cases & interactions

- **Straggler call sites.** After deleting the overloads, `yarn build` must be clean
  across *all* packages — `quereus-store`, `quereus-isolation`, `quereus-sync`,
  `quoomb-cli`, `quoomb-web`, the plugins. Any remaining three-argument
  `compareSqlValues` in a package outside `packages/quereus` will surface only at
  full-monorepo build. Run `yarn build`, not just the quereus workspace.
- **`sample-plugins/` and `plugin-loader/`** may call the global `registerCollation`
  as the documented plugin API. `src/util/plugin-helper.ts` ~97 already routes plugin
  collations through `db.registerCollation`; grep the sample plugins and any plugin
  docs for the global import before deleting.
- **`test/utility-edge-cases.spec.ts`** exercises `compareSqlValues` heavily with two
  arguments — those stay. Check for any three-argument use.
- **`test/vtab/test-overclaim-module.ts`** uses the two-argument form; unaffected.
- **`quereus-sync/src/sync/store-adapter.ts` ~476** uses the two-argument form;
  unaffected.
- **Dead-code check:** once the `Map` is gone, `warnLog` in `comparison.ts` (~11) may
  have no remaining use. Remove the unused binding rather than leaving it — `yarn lint`
  will flag it.
- **`compareSqlValuesFast` must be publicly exported** before `quereus-store` /
  `quereus-isolation` can compile against it; `3.4` should already have done this —
  verify, do not assume.
