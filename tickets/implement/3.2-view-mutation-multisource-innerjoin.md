## Scope

This is the **multi-source acceptance case**: a view body that is a key-preserving equi-join of
two base tables, written through the substrate. The single-source path and the `ViewMutationNode`
+ emitter scaffolding land in `view-mutation-substrate-core`; this ticket makes the walk produce
more than one `BaseOp` and makes the emitter sequence/coordinate them.

## Propagation visitor — planned-body → ordered `BaseOp[]`

Walk the **planned** body (the logical operator tree the substrate already holds), NOT
`selectAst`. The structural reason: an inner view/CTE filter lives in the plan, not in the outer
AST — walking the plan is what lets the substrate generalize where the AST rewrite cannot. Read
`updateLineage` / `attributeDefaults` off each node's `PhysicalProperties` (do NOT re-derive
lineage; `view-mutation-physical-lineage` already threaded it via
`deriveProjectUpdateLineage` / `deriveFilterAttributeDefaults` / `deriveJoinUpdateLineage`).

One method per operator (mirrors `runtime/emit/`):

- **Projection** — updates rewrite assignments against base columns via the `UpdateSite` inverse
  chain (`scalar-invertibility.ts` `traceInvertibleColumn`); a computed/non-invertible target →
  `no-inverse` diagnostic. Inserts fill missing base columns in source order: value list
  (post-inverse) → constant-FD `attributeDefault` → FD reconstruction → EC propagation →
  `default_for` tag (the tag plumbing arrives in `view-mutation-tag-override-surface`; gate behind
  its absence) → base column default (incl. generated/context default) → `null`; a `not null`
  column with no value → `no-default`.
- **Selection (σ)** — conjoin the σ predicate into the propagated / identifying / existence
  predicate; provable contradiction → `predicate-contradiction`.
- **Inner Join** — route per-column updates to the owning side (two `BaseOp`s if a `set` touches
  both); inserts require both sides' `not null`-without-default values and emit FK-parent before
  FK-child; deletes follow the `delete_via` default (§ Inner Join — the FK-many side by default).
- **Set-ops / outer joins / aggregates / windows** — keep emitting the structured diagnostics
  for the still-out-of-scope shapes, now from the planned walk. Document precisely which shapes
  newly succeed (multi-source inner join) vs still reject.

`MutationRequest` shape (the tags map is populated by the tag-surface ticket; accept an empty map
here):
```typescript
interface MutationRequest {
  readonly op: 'insert' | 'update' | 'delete';
  readonly assignments?: ReadonlyMap<AttributeId, ScalarExpr>;  // update
  readonly insertValues?: …;                                    // insert existence inputs
  readonly userPredicate?: PredicateExpr;                       // where / values-derived
  readonly tags: ReadonlyMap<string, string>;                   // merged view+stmt tags (empty until 3.3)
}
```

**Identifying predicate construction** is the hard part of the planned walk: reconstruct the
row-identifying predicate (equality on traced base PKs) from `updateLineage` + the forward FD
keys (`keysOf`), since the planned body holds plan nodes, not AST predicates. The Design-A base
subtrees still need AST `where`/`values` to build through `buildUpdateStmt`/`buildDeleteStmt`/
`buildInsertStmt`, so this ticket must lower the per-base predicate back to AST (or extend the
base builders to accept a pre-built `ScalarPlanNode` predicate — decide and document; lowering to
AST keeps the base builders untouched and is the lower-risk path).

## Emitter — ordered execution, conflict, FK order, RETURNING, context

Extend `runtime/emit/view-mutation.ts` (the scaffold from 3.1):

- Execute base ops in list order within the statement transaction (§ Multi-Base-Table Mutations):
  FK-parent before FK-child where provable, unspecified within an FK class. Ordering is decided at
  plan time in `propagate()`; the emitter executes the given order.
- Compose conflict resolution across ops — the prevailing `or` mode applies per base op;
  same-row same-table fan-out merges when effects are identical, conflicts otherwise (§ Union All
  tail; relevant once a base op appears twice).
- Capture RETURNING rows projected through the **view's** column list, evaluated against
  post-mutation base values (computed-lineage columns re-evaluated). This is where
  `returning-through-view` flips from rejected to supported — update the diagnostic gate and
  `docs/view-updateability.md`.
- Thread the per-statement / per-row mutation-context envelope so a shared surrogate (join key)
  generated per row threads through every branch (§ Mutation Context worked example): the single
  captured per-row value is resolved at the envelope BEFORE propagation reaches the branches, so
  every branch references one already-captured binding (no "which branch generates first"
  question). Reuse the existing mutation-context machinery
  (`DmlExecutorNode.mutationContextValues` / `contextAttributes` / `contextDescriptor`; runtime
  `evaluateContextRow` in `runtime/emit/dml-executor.ts`).

## Round-trip law — dynamic multi-source PutGet/GetPut (acceptance gate)

Extend the `View Round-Trip Laws` block in `test/property.spec.ts` (currently single-source Tier A
over the `SHAPES` zoo, `numRuns: 50`, with a pure law core + negative self-test). Add a
multi-source equi-join view to the zoo and **dynamic** laws executed through the substrate:

- **PutGet** — apply a generated mutation through a multi-source view (key-preserving equi-join),
  read the view back, assert the read reflects exactly the mutation's writable-column effect; no
  rows escape the view predicate; computed-column writes rejected with `no-inverse`; a
  forward-claimed key is the tuple the backward walk bound.
- **GetPut** — read a row through the view, write the same values back keyed on the view's
  identifying predicate, assert the base diff is empty across ALL touched base tables.

A new operator's backward method is not "done" until PutGet / GetPut / lineage-agreement are green
over a planned tree surfacing it. Keep the existing single-source laws and the
`LIMIT`/`OFFSET`/`DISTINCT`-reject assertions green.

## Acceptance

- A multi-source view body (key-preserving equi-join) decomposes to an ordered base-op list and
  writes through correctly, with conflict / FK / RETURNING parity to hand-written base DML.
- `query_plan()` surfaces per-output-column lineage (already from `view-mutation-physical-lineage`;
  confirm intact across the join).
- PutGet / GetPut / lineage-agreement green over the planned multi-source tree.
- Set-op / outer-join / aggregate bodies still reject with their structured diagnostics.
