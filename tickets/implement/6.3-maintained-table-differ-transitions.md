# Differ recognition of attach/detach/re-attach

Part 3 of the maintained-table chain. With the unified model and the verbs
landed, `declare schema` transitions on a named maintained table must apply
as catalog flips + content reconcile â€” never as drop+recreate (which mints
a new incarnation, fatal to a replicated table's row metadata).

## Diff model

`materialized view` declared items **normalize at diff time into the table
category** (a declared table whose maintained clause carries the body, with
no declared columns â€” shape owned by the body). The separate
`materializedViewsToCreate/Drop` buckets and the standalone MV comparison
loop dissolve; comparison happens per table name in one category. Cross-
category drop+create for tableâ†”maintained therefore disappears
structurally. The differ stays **plan-free**: it never derives a body's
shape; shape errors surface at apply (the attach shape check), which is the
sited, authoritative check.

Per matched table name, compare as today (columns, constraints, PK, tags â€”
the existing `computeTableAlterDiff`) plus the derivation dimension
(declared maintained clause's `computeBodyHash` vs live
`derivation.bodyHash`):

| live \ declared | plain | maintained (same hash) | maintained (hash drift) |
|---|---|---|---|
| plain | table ops | **attach**: table ops, then `set maintained as` | same (attach is one op) |
| maintained | **detach**: `drop maintained`, then table ops | table ops only (column drift on a maintained table will error at apply's shape check â€” an authoring inconsistency, surfaced sited) | **re-attach**: same shape â‡’ single `set maintained as` (content reconcile); with column drift â‡’ `drop maintained` â†’ table ops â†’ `set maintained as` |

- A **module change** on any table (maintained or not) keeps today's
  posture for tables: there is no in-place module move; document that this
  remains destructive (drop+create) and out of scope here.
- **Rename reconciliation** carries over: an in-diff source table/column
  rename is inverse-applied before the hash comparison so a pure rename
  never churns a re-attach (today's `reconciledDeclaredViewDefinition`
  retargeted at the maintained clause).
- **Tags-only** change on a maintained table â‡’ `set tags`, never re-attach
  (tags stay out of the canonical definition).
- **Orphan** (live maintained table not declared at all) â‡’ `drop table`
  (parity with today's undeclared-MV drop â€” it is a table now).

**Apply order** in `generateMigrationDDL`: `drop maintained` ops run where
MV drops run today (early, before table alters/drops â€” a detach must
precede column ops on that table and may precede drops of its former
source); `set maintained as` ops run where MV creates run today (late,
after table creates/alters â€” sources and the target's final shape must
exist). The flip's cross-table pair (detach v2, attach v1) falls out of
this ordering naturally.

## Edge cases & interactions

- Round-trip: declare maintained â†’ apply â†’ declare same table without the
  clause â†’ apply â‡’ detach (table + rows intact, writable); reverse â‡’
  attach with zero row writes when content is already derivable.
- Idempotence: re-applying an unchanged schema â‡’ empty diff (no phantom
  re-attach), for both the table form and the MV-sugar declared item over
  the same live maintained table (sugar vs table form must compare equal â€”
  the hash and normalization must agree across authoring forms).
- Body change, same shape â‡’ exactly one `set maintained as`; content
  re-derives (a refresh, not a recreate); table incarnation and unrelated
  rows' identity preserved (assert via a vtab-connection-stable observation
  or store row metadata once available â€” at minimum assert no
  table_removed/table_added fires for the target).
- Body change with shape change â‡’ detach â†’ column ops â†’ attach; the new
  column's values arrive via the attach reconcile.
- Pure in-diff source rename (table and column) â‡’ no re-attach churn.
- Declared maintained over live plain table whose content diverges â‡’ attach
  reconciles per the verb's semantics (derived wins) â€” the differ emits the
  op; semantics live in the verb.
- A diff that detaches X and also drops X's former source in one apply â‡’
  ordering holds (detach first).
- MV-over-MV: re-attach of a producer while a consumer exists â‡’ consumer
  reconverges via the reported diff cascade; differ emits ops in
  source-before-consumer order for attaches.
- Logical-schema declares (`declare logical schema` / lens blocks) are
  untouched â€” maintained clauses are a basis-schema concern.

## The migration capstone test (engine-level)
