# Differ recognition of attach/detach/re-attach

Part 3 of the maintained-table chain. With the unified model and the verbs
landed, `declare schema` transitions on a named maintained table must apply
as catalog flips + content reconcile — never as drop+recreate (which mints
a new incarnation, fatal to a replicated table's row metadata).

## Diff model

`materialized view` declared items **normalize at diff time into the table
category** (a declared table whose maintained clause carries the body, with
no declared columns — shape owned by the body). The separate
`materializedViewsToCreate/Drop` buckets and the standalone MV comparison
loop dissolve; comparison happens per table name in one category. Cross-
category drop+create for table↔maintained therefore disappears
structurally. The differ stays **plan-free**: it never derives a body's
shape; shape errors surface at apply (the attach shape check), which is the
sited, authoritative check.

Per matched table name, compare as today (columns, constraints, PK, tags —
the existing `computeTableAlterDiff`) plus the derivation dimension
(declared maintained clause's `computeBodyHash` vs live
`derivation.bodyHash`):

| live \ declared | plain | maintained (same hash) | maintained (hash drift) |
|---|---|---|---|
| plain | table ops | **attach**: table ops, then `set maintained as` | same (attach is one op) |
| maintained | **detach**: `drop maintained`, then table ops | table ops only (column drift on a maintained table will error at apply's shape check — an authoring inconsistency, surfaced sited) | **re-attach**: same shape ⇒ single `set maintained as` (content reconcile); with column drift ⇒ `drop maintained` → table ops → `set maintained as` |

- A **module change** on any table (maintained or not) keeps today's
  posture for tables: there is no in-place module move; document that this
  remains destructive (drop+create) and out of scope here.
- **Rename reconciliation** carries over: an in-diff source table/column
  rename is inverse-applied before the hash comparison so a pure rename
  never churns a re-attach (today's `reconciledDeclaredViewDefinition`
  retargeted at the maintained clause).
- **Tags-only** change on a maintained table ⇒ `set tags`, never re-attach
  (tags stay out of the canonical definition).
- **Orphan** (live maintained table not declared at all) ⇒ `drop table`
  (parity with today's undeclared-MV drop — it is a table now).

**Apply order** in `generateMigrationDDL`: `drop maintained` ops run where
MV drops run today (early, before table alters/drops — a detach must
precede column ops on that table and may precede drops of its former
source); `set maintained as` ops run where MV creates run today (late,
after table creates/alters — sources and the target's final shape must
exist). The flip's cross-table pair (detach v2, attach v1) falls out of
this ordering naturally.

## Edge cases & interactions

- Round-trip: declare maintained → apply → declare same table without the
  clause → apply ⇒ detach (table + rows intact, writable); reverse ⇒
  attach with zero row writes when content is already derivable.
- Idempotence: re-applying an unchanged schema ⇒ empty diff (no phantom
  re-attach), for both the table form and the MV-sugar declared item over
  the same live maintained table (sugar vs table form must compare equal —
  the hash and normalization must agree across authoring forms).
- Body change, same shape ⇒ exactly one `set maintained as`; content
  re-derives (a refresh, not a recreate); table incarnation and unrelated
  rows' identity preserved (assert via a vtab-connection-stable observation
  or store row metadata once available — at minimum assert no
  table_removed/table_added fires for the target).
- Body change with shape change ⇒ detach → column ops → attach; the new
  column's values arrive via the attach reconcile.
- Pure in-diff source rename (table and column) ⇒ no re-attach churn.
- Declared maintained over live plain table whose content diverges ⇒ attach
  reconciles per the verb's semantics (derived wins) — the differ emits the
  op; semantics live in the verb.
- A diff that detaches X and also drops X's former source in one apply ⇒
  ordering holds (detach first).
- MV-over-MV: re-attach of a producer while a consumer exists ⇒ consumer
  reconverges via the reported diff cascade; differ emits ops in
  source-before-consumer order for attaches.
- Logical-schema declares (`declare logical schema` / lens blocks) are
  untouched — maintained clauses are a basis-schema concern.

## The migration capstone test (engine-level)

The `docs/migration.md` worked example, expand → flip → contract, single
database, all via `declare schema` + `apply schema` (memory module is fine;
`using store()` variant optional):

- **Expand**: declare `Contact_v1` (rows pre-seeded) + `Contact_v2`
  maintained from v1 (the `collate nocase` body) → apply ⇒ attach fills v2;
  writes through v1 propagate to v2; write-through against v2 lands in v1.
- **Flip**: declare v2 plain + v1 maintained from v2 with the inverse body
  → apply ⇒ detach v2, attach v1 (reconcile: zero writes — content already
  inverse-derivable); writes to v2 now propagate to v1.
- **Contract**: declare v2 only → apply ⇒ detach+drop v1 (drop of a
  maintained v1 = drop table; ordering: nothing references it).
- Assert data equivalence at every quiescent point and that v2's
  incarnation/rows survive all three applies untouched (no
  table_removed/table_added for v2 after expand).
