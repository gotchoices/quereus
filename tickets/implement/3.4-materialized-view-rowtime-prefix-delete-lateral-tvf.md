> **Adds the `'prefix-delete'` arm** to the `MaintenancePlan` union (the substrate's
> 1.5 union defines `'inverse-projection'`, `'full-rebuild'`, `'residual-recompute'`; this
> ticket adds `'prefix-delete'` as a fourth, reachable arm). It reuses the synchronous
> residual kernel built by `materialized-view-rowtime-residual-recompute` — the affected-key
> derivation, the `injectKeyFilter` residual compile, the per-statement batched flush, the
> cost gate — and differs only in that one base row **fans out** to many backing rows, so
> the delete is **by-prefix** rather than by-key.

## What this arm maintains

A **single-source lateral-TVF fan-out** body:

```sql
create materialized view mv as
  select t.id, t.x, f.*
  from t, lateral table_func(t.x) f;
```

Each base row `t` produces **N** MV rows (one per row the TVF emits for that `t`). The
backing PK is the **composite product key** `(t.pk ∪ f.key)` — the lateral right side is
built as a `JoinNode` (`building/select.ts`), so `optimizer-keyed-cross-product-join-keys`
(complete) advertises this composite through `keysOf` (see its § *Keyed cross/inner (and
lateral) product keys*). The base PK is the **leading prefix** of that composite key.

Maintenance per changed base row `t`:

- **delete the whole fan-out slice for `t`** — every backing row whose leading PK columns
  equal `t.pk` (a **by-prefix** delete);
- if the new `t` is in scope, **re-run the TVF fan-out residual** for just this `t` row and
  **upsert** each emitted row.

An UPDATE is delete-old-prefix + recompute-new (the base PK may itself change → the prefix
moves). A DELETE is delete-old-prefix with no recompute. An INSERT is recompute-and-upsert
(no prior prefix to delete, but deleting a non-existent prefix is a safe no-op).

## Why a distinct arm (vs `'residual-recompute'`)

The residual-recompute arm replaces a **point-keyed** slice (one group / one 1:1 join
row): delete one key, upsert ≤1 row. The lateral-TVF arm replaces a **prefix-keyed** slice
of unknown cardinality (a base row's whole fan-out), so:

1. it needs a **by-prefix delete** primitive the residual arm does not; and
2. the residual yields **N rows**, all sharing the base-PK prefix but distinguished by the
   TVF key tail.

Everything else — affected-key (here: base PK) derivation from the changed row, the
`injectKeyFilter` residual pinned to the base `TableReferenceNode`, the per-statement
batched accumulator, the cost gate, reads-own-writes execution — is the residual kernel
from the prereq, consumed unchanged.

## Re-add the by-prefix delete `MaintenanceOp`

The row-time-only consolidation removed the old `delete-by-prefix` op; re-introduce it on
the now-shared substrate:

```ts
export type MaintenanceOp =
  | { kind: 'delete-key';       key: BTreeKeyForPrimary }
  | { kind: 'upsert';           row: Row }
  | { kind: 'delete-by-prefix'; keyPrefix: BTreeKeyForPrimary };   // ← re-added
```

`applyMaintenanceToLayer` (`vtab/memory/layer/manager.ts`) gains a `'delete-by-prefix'`
arm: range-scan the backing primary tree over the half-open key interval whose leading
columns equal `keyPrefix` (the btree is ordered by the composite PK, base-PK columns
leading), and `recordDelete` each matched row on the pending layer — same bookkeeping the
existing `delete-key` arm uses, just over a prefix range instead of a point. Keep the
existing exhaustive `switch` exhaustive.

> **Prefix soundness depends on column order.** The base-PK columns must be the **leading**
> columns of the backing composite PK for a prefix range to select exactly one base row's
> fan-out. Confirm `deriveBackingShape` / `computeBackingPrimaryKey` order the composite so
> the base key leads (it should, since the product key is `leftKey ∪ rightKey-shifted`); if
> a body's `order by` would reorder it, either reject that shape or seed the base key first.
> Add an assertion at `buildMaintenancePlan` time that the base PK is a prefix of the
> backing PK, failing loud (`StatusCode.INTERNAL`) otherwise.

## Shape gate (extend `buildMaintenancePlan`)

Replace the blanket `TableFunctionCall` rejection with a soundness-checked admit for the
lateral-TVF fan-out shape:

- **single base source `T`** with a PK, joined to **one lateral TVF** whose arguments are
  per-row functions of `T` (so each base row drives an independent fan-out); no second base
  table, no nested TVF, no aggregate/DISTINCT/set-op/recursion over the result;
- the **TVF advertises a per-call key** so the composite product key is a real column key
  (not the all-columns `isSet` fallback) — consult `keysOf` on the optimized body. If the
  TVF exposes no per-call key, the fan-out rows are not individually addressable → reject
  `shape` (steer to a plain view), OR key the tail by all TVF output columns if that is a
  provable set (decide at implement time; prefer the advertised key, reject otherwise for
  v1 soundness);
- the base PK must be **projected** (so the prefix is recoverable from the backing row) and
  must be the **leading prefix** of the backing PK (see prefix-soundness note);
- a partial `WHERE` on `T` must be single-row-evaluable (same `compilePredicate` discipline
  as the other arms) — it gates whether the new base row fans out at all.

Keep recursion / set-ops / multi-base-source bodies rejected. A **fanning keyed join**
(non-1:1 inner/cross join, deferred from the residual-recompute ticket) is the natural
*next* consumer of this same by-prefix machinery — note it in docs as a follow-up but keep
it out of scope here unless it falls out trivially (it shares the by-prefix delete + product
key, differing only in that the "fan-out residual" is the join rather than a TVF).

## Cost gate

The lateral-TVF arm's per-write cost is `changeCardinality × (TVF fan-out cost + N
upserts)`. Feed it through `maintenanceCost('residual-recompute', …)` (the arm shares the
residual cost shape — a key-filtered re-execution) or add a `'prefix-delete'` cost branch
if the fan-out factor warrants its own formula. The synchronous degrade-vs-reject split
applies: a TVF whose fan-out makes per-write cost exceed the pathological full-scan
threshold must **reject at create** (it would make every base write pathological); a
cardinality spike on a bulk statement may `degradeToRebuild` for that statement.
