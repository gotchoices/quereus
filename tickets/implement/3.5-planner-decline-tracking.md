## Problem

`PassManager.applyPassRules` (`framework/pass.ts:557-593`):

```ts
while (changed) {
    changed = false;
    for (const rule of pass.rules) {
        if (rule.nodeType !== currentNode.nodeType) continue;
        if (context.tuning.disabledRules?.has(rule.id)) continue;
        if (hasRuleBeenApplied(currentNode.id, rule.id, context)) continue;   // only skips APPLIED rules
        const result = rule.fn(currentNode, context);
        if (result && result !== currentNode) {
            markRuleApplied(currentNode.id, rule.id, context);                 // marks ONLY on transform
            this.inheritVisitedRules(currentNode.id, result.id, context);
            ...
            currentNode = result; changed = true;
        }
    }
}
```

A rule that **declines** (returns `null` or the same node) is never recorded. So after any
single transform, the `while` loop re-scans the full rule battery for that node type — every
decliner runs again. For a `FilterNode` that is the full ~9-rule battery, including the
~1300-line query-rewrite matcher, re-fired even though nothing about those rules' inputs
changed.

## The subtle part (read before coding)

The naive fix — "record declines in a per-node-id set and skip them" — **saves almost
nothing**, because a transform mints a *new* node id (`rule.fn` returns a fresh `PlanNode`),
and the next `while` iteration scans against that new id, where no decline was recorded.

The existing code already handles the symmetric case for *applied* rules: `inheritVisitedRules`
(pass.ts:595-608) copies the applied-rule set from the old node to the transformed node, so an
applied rule is not re-offered on the re-mint. The correct fix unifies decline tracking onto
that same mechanism:

- Record a rule as **visited** on a node whether it transformed *or declined*.
- Inherit the full visited set (applied ∪ declined) across a transform, exactly as
  `inheritVisitedRules` does today for the applied set.

Net effect: each rule gets **at most one shot per node lineage per pass**.

### Correctness question that MUST be answered by tests

Inheriting declines across a re-mint means: if rule B transforms A→A′, a rule X that declined
on A will **not** be re-offered on A′ — even though A′ differs and X might now apply. This is
the *same* soundness assumption the code already makes for applied rules (an applied rule is
not re-tried on A′ either). The pass framework relies on multi-pass re-traversal and rule
fixpoint/monotonicity to converge, so this is believed safe — but it is a real behavior change
and the whole point of splitting this ticket out.

Verify explicitly:
1. A transform that changes a node still lets **previously-untouched** rules fire on the new
   node (only *visited* rules are suppressed, and `withChildren`-minted children that were
   never visited must still get their full battery).
2. No existing plan/optimizer test changes output. If one does, understand *why* before
   touching it — a changed plan here is a signal, not a nit to silence.
3. The `maxRulesFired` non-convergence guard (pass.ts:579-584) still trips on a genuinely
   non-converging rule (i.e. the decline-suppression must not mask a rule that legitimately
   should keep firing on genuinely-new nodes).

## Where the state lives

`context.visitedRules: Map<string, Set<string>>` (context.ts:42) already exists and is keyed
by node id. Options:
- Reuse `visitedRules` for both applied and declined (simplest; `hasRuleBeenApplied` already
  reads it). Rename or re-document so "visited" clearly means "offered, regardless of
  outcome". `markRuleApplied` currently means "record as visited" — it can be reused as-is for
  the decline case since it just adds to the set.
- Keep a separate `declinedRules` map if you want to preserve the applied/declined distinction
  for diagnostics — but then `inheritVisitedRules` must copy both, and the skip check must
  consult both. Prefer the single-set approach unless a diagnostic consumer needs the split
  (none found in this ticket's research).

Note `context.visitedRules` is copied verbatim by `OptimizationContext.copyTrackingState`
(context.ts:124-134) across `withPhase` / `withContext` — folding declines into the same map
means they propagate across phases identically. Confirm that is intended (it matches how
applied rules already behave).
