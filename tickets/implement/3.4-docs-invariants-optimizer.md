## What a register buys

After the splits, the optimizer's documentation is seven files under the
12,000-word cap. That makes each *readable*, but a reviewer who wants to answer
"does the code still do what we said it does" still has to read all seven.

`docs/invariants.md` is the answer: a single file holding **only** statements the
code must satisfy, each numbered, each ≤ 120 words, each naming the file and
symbol that implements it and the test that guards it. It is meant to be read
end-to-end against the implementation in one sitting, a few times a year. The
topic docs keep the exposition and link *into* it.

`scripts/check-docs.js` (from `docs-doc-conventions-and-checker`) already validates
the block format and — crucially — that every `code:` path exists and every named
symbol still appears in it. That is the drift tripwire: when someone deletes or
renames `validateSideEffectMode`, the build fails and forces a doc read. The
checker never verifies an invariant *holds*; that is what `guard:` is for.

This ticket creates the file and populates the `OPT-*` area.
`docs-invariants-mv` adds `MV-*` afterwards. `RT-` / `SCH-` / `SYNC-` / `LENS-`
are reserved and stay empty.

## What qualifies as an invariant

All four must hold:

1. It states a property the **code** must uphold — not a description of what the
   code happens to do, and not advice.
2. Violating it is a **bug**, not a missed optimization. "The optimizer eliminates
   FK→PK joins whose non-preserved side is unused" is a capability, not an
   invariant. "A rule that drops a subtree must consult `subtreeHasSideEffects`" is
   an invariant — violating it silently drops a write.
3. It can be stated in **≤ 120 words** without the surrounding section for context.
4. It has a **concrete code site**: file, and where meaningful, symbol.

Deliberately *not* invariants: cost-model constants and their particular values;
the rule catalog; performance advice; worked examples; "how to add a rule";
anything in `## Rejected alternatives`.

Expect roughly **20–30** `OPT-*` entries. If you are past 40 you are transcribing;
if you are under 12 you are skimming. Number them `OPT-001` upward with gaps of
your choosing — a retired invariant's number is never reused.

## Seed list

Verified against the code at plan time. It is a starting point, not the answer —
read the docs and add what is missing, and drop any of these that turns out not to
survive contact with the implementation. Every entry needs a `guard:` line, and
`guard: none — <reason>` is a legitimate, expected answer for many of them.

**Rule framework**

- Every rule reaching `PassManager.addRuleToPass` declares `sideEffectMode:
  'safe' | 'aware'`; registration throws otherwise.
  `planner/framework/registry.ts` — `validateSideEffectMode`;
  `planner/framework/pass.ts` — `addRuleToPass`. Self-guarding (throws).
- An `'aware'` rule — one that moves, duplicates, drops, or merges a subtree —
  consults `PlanNodeCharacteristics.hasSideEffects` / `subtreeHasSideEffects` and
  refuses or weakens when a participating subtree carries a write.
  `planner/framework/characteristics.ts`.
- A pass with a custom `execute` bypasses `sideEffectMode` validation; its
  soundness argument must live in a comment at the pass. `pass.ts` —
  `createMaterializationPass` is the one live instance, and its argument is that
  `CacheNode` is a run-once fence.
- Rules dispatch on characteristics and physical properties, never on `instanceof`
  or `nodeType` equality. `planner/framework/characteristics.ts`;
  `docs/optimizer-conventions.md` is the exposition.

**Functional dependencies** (all in `planner/util/fd-utils.ts` unless noted)

- Uniqueness is read through `keysOf` / `isUnique` / `isUniqueDeterminant` and
  nowhere else. Coverage (`closureCoversAll`) is a value claim and proves nothing
  about row-uniqueness over a bag. Guard: `test/property.spec.ts` § Key Soundness
  (Tier 1 top-node + Tier 2 every-node soundness assertions).
- All closure helpers skip guarded FDs. A conditional uniqueness claim never
  proves a key for an unrelated subtree.
- A guard is discharged only at the producing `Filter.computePhysical`, via
  `predicateImpliesGuard` + `stripGuard` — never at a consumer.
  `planner/nodes/filter.ts` — `activateGuardedFds`.
- A fanning join downgrades the non-preserved side's `'unique'` FDs to
  `'determination'`, guarded FDs included. `planner/nodes/join-utils.ts` —
  `downgradeUniqueFds`, `propagateJoinFds`.
- Outer joins drop the null-padded side's FDs, ECs, bindings, and domains.
  `join-utils.ts` — `propagateJoinFds`.
- `fdsEqual` is structural and ignores `kind`, `source`, and `valueEquality`. When
  `addFd` merges equal-determinant entries, `'unique'` wins.
- `addFd` — never `Array.push` — is the only accumulation path; it performs
  subsumption and enforces `MAX_FDS_PER_NODE`, preferring to evict
  `'determination'` over `'unique'`.
- FD / EC / binding / domain column indices are **output-column indices** on the
  carrying node. Crossing a Project / Returning / Aggregate / join boundary goes
  through `projectFds` / `shiftFds` and their mirrors.
- An equality-derived fact is extracted only when the conjunct's effective
  comparison collation is value-discriminating. `planner/nodes/filter.ts` and the
  CHECK-extraction path. Guard: the collation-blind extraction tests.
- `source` provenance is informational; no rule branches on it.
- "All output columns form a key" lives on `RelationType.isSet`, never in `fds`.

**Plan structure**

- An attribute ID is *originated* once, at the node that mints it, and may be
  forwarded verbatim by any number of ancestors. The invariant is "originated
  once", not "appears once". `docs/optimizer.md` § Attribute provenance.
- Plan nodes are immutable; a transform re-mints rather than mutates, and
  `PassManager.inheritVisitedRules` carries visited-rule state across the re-mint.
- Cost is in abstract units, lower is better, and a transform that changes a
  node's shape recomputes cost rather than inheriting it. Cross-reference
  `tickets/complete/1-planner-cost-model-double-count.md` — that ticket fixed a
  double-count; **do not restate its analysis**, cite it. `planner/cost/index.ts`.
- The physicalization invariant for `Retrieve` (every node beneath a `Retrieve`
  is either supported by the module or lives above the boundary as a residual).
  `docs/optimizer-retrieve.md`.

**Explicitly out of scope, cross-reference only**

- Runtime context shadowing — owned by `tickets/plan/3-runtime-context-shadow-assertion.md`.
  If that ticket lands an assertion, its invariant belongs in an `RT-*` entry filed
  by *that* work, not here. Add nothing about it.

## `guard:` and what to do about `none`

Three legitimate values:

- **A test.** `test/property.spec.ts` § Key Soundness backstops the FD-soundness
  family. `test/property.spec.ts` § View Round-Trip Laws backstops the backward
  direction. Point at the file and the describe-block name.
- **A runtime assertion.** `validateSideEffectMode` throws. That *is* the guard;
  say so.
- **`none — <reason>`.** Most structural invariants have no mechanical guard, and a
  bare `none` is not allowed — the reason is the useful part ("no cheap mechanical
  check; violation shows up as a wrong plan, not a crash").

**Cap on follow-on work:** you may file **at most one** `tickets/backlog/debt-…`
ticket, listing together every `guard: none` invariant for which a cheap
mechanical check looks worthwhile, with a sentence each. Do not file thirty. Do
not implement guards here — this is a documentation ticket, and an untested new
assertion in the optimizer's hot path is exactly the kind of thing that should go
through its own plan.

## Back-links

Each invariant carries a `doc:` line pointing at the topic-doc section that
explains it. Each topic-doc section that an invariant summarizes gains a
`> **Invariant:** [OPT-014](invariants.md#opt-014)` blockquote near its heading —
one line, no restatement. The register is the normative text; the topic doc is the
explanation. When they disagree, the register wins, and `docs/doc-conventions.md`
should say so if it does not already.

Link `docs/invariants.md` from `docs/architecture.md` — one line, near the
`doc-conventions.md` link.

## Edge cases & interactions

- **The register must not become the next mega-doc.** The 12,000-word cap applies
  to it like any other doc. 30 OPT entries × ~90 words ≈ 2,700 words; MV adds a
  similar amount. If yours is heading past 6,000 words with two areas populated,
  entries are carrying rationale they should have left in the topic doc.
- **An invariant that turns out to be false.** Reading these against the code is
  the point, and it is likely you will find at least one prose claim the
  implementation does not honor. **Do not silently soften the statement to match
  the code.** Either the doc is wrong (fix the prose, note it in the handoff) or
  the code is wrong (file a `tickets/fix/bug-…` ticket and record the invariant
  with a `guard: none — <reason>` plus a pointer to the bug ticket). Which it is,
  is a judgement call you must make and show your work on.
- **Symbol-substring checking is coarse.** `code: … — \`addFd\`` passes because the
  string `addFd` appears in `fd-utils.ts` — including inside a comment. That is
  accepted: the check exists to catch deletion and rename, not to prove call
  sites. Don't over-engineer it.
- **Duplicate slugs.** Heading `### OPT-014 — …` slugifies to `opt-014--…`, and
  back-links use the short `#opt-014` form — which will **not** match. Either write
  the back-link with the full slug, or give each invariant an explicit anchor. Pin
  the chosen form with a checker assertion; get this wrong and every back-link in
  the repo is dead in a way the checker catches loudly on the first run (good) or
  the first reviewer catches quietly (bad).
- **Ascending IDs, gaps allowed.** The checker enforces ascending order within an
  area. Leaving `OPT-005` unused because you deleted a draft is fine and expected.
- **Interaction with `docs-invariants-mv`.** It appends `MV-*` to the same file and
  depends on this one landing first (it is `prereq`-chained). Leave the `## MV —
  Materialized views` heading in place with a one-line "populated by
  `docs-invariants-mv`" note, so that ticket has an obvious insertion point and the
  checker's empty-area handling is exercised now rather than later.
- **`docs/optimizer-conventions.md` overlaps.** Its eight numbered "key conventions
  for FD-aware rules" are, several of them, invariants. Do not delete that document
  — it is the *how to write a rule* guide and has a different audience. Replace the
  duplicated conventions with `> **Invariant:** [OPT-0nn](invariants.md#…)`
  back-links, keeping the code examples that make the guide useful.
