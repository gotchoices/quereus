# Unify SQL value equality on one function

## Problem

`util/comparison.ts` exports two ways to ask "are these two SQL values the same
value?":

- `compareSqlValuesFast(a, b, BINARY_COLLATION) === 0` — SQLite storage-class
  semantics: `5n` (bigint) and `5` (number) are the same NUMERIC value; blobs
  compare bytewise; JSON objects compare by canonical string.
- `sqlValuesEqual(a, b)` — special-cases blobs and JSON objects the same way, then
  falls through to raw JS `===`. So `sqlValuesEqual(5n, 5)` is `false`.

Every caller of `sqlValuesEqual` is asking "did this value change?" — foreign-key
parent-column change detection, constraint-check old/new diffing, DML
value-identical-upsert suppression, planner mutation cell comparison. For all of
them, `5n` and `5` are the same value and the `===` divergence is a latent bug: a
storage layer that hands back a bigint where the proposed row carries a number
would report a spurious change. `dml-executor.ts` (~331) already carries a comment
naming `sqlValuesEqual`'s raw-JS-equality as the thing to fix.

`rowsValueIdentical` (same file, ~278) already does the right thing per row and is
documented as the normative value-identity contract. `sqlValuesEqual` is its
divergent, undocumented single-value twin.

## Design

Delete `sqlValuesEqual`. Add its replacement next to `rowsValueIdentical`:

```ts
/**
 * Byte-faithful single-value identity: {@link compareSqlValuesFast} under
 * BINARY. Numeric-storage-class tolerant (`5n` equals `5`), bytewise for blobs,
 * canonical-JSON for object-class values, byte-exact for text. This is the
 * scalar form of {@link rowsValueIdentical} and shares its contract.
 */
export function sqlValueIdentical(a: SqlValue, b: SqlValue): boolean {
	return compareSqlValuesFast(a, b, BINARY_COLLATION) === 0;
}
```

Rewrite `rowsValueIdentical` as a loop over `sqlValueIdentical` so the two cannot
drift again. Update every `sqlValuesEqual` call site to `sqlValueIdentical`.
Update the public export list in `src/index.ts`.

Note: `compareSqlValuesFast`'s OBJECT-class path routes through
`objectCanonicalString`, the same `WeakMap`-backed canonical cache
`sqlValuesEqual` used, so the JSON-object behavior is unchanged (no perf
regression, no semantic change).

At `dml-executor.ts` ~331, the "Representation-sensitive keys" comment names
`sqlValuesEqual` and its `===` behavior. Update the comment: the bigint/number
half of that concern is now gone; the affinity-coercion and enforcement-collation
halves remain open. Do **not** try to fix affinity/collation here — that is a
separate concern already documented at that site.

## Edge cases & interactions

- `sqlValueIdentical(5n, 5)` → `true` (was `false`). This is the intended
  behavior change. Confirm no caller depends on the old answer: all four caller
  families are "did the value change?" checks where a numeric-equal value has not
  changed.
- `sqlValueIdentical(true, 1)` → `true` (boolean coerces to NUMERIC in
  `getStorageClass`). Previously `false`. Same reasoning; verify FK
  change-detection and `constraint-check` tests still pass.
- `sqlValueIdentical(null, null)` → `true`; `sqlValueIdentical(null, 0)` → `false`.
  Preserved.
- `sqlValueIdentical('1', 1)` → `false` (TEXT vs NUMERIC storage class). Preserved.
- Blobs: differing length → `false`; empty vs empty → `true`. Preserved.
- JSON objects: key-reorder-equal objects → `true`. Preserved.
- `test/util/json-canonical.spec.ts` (~95) asserts that hash-key equality and
  `sqlValuesEqual` agree. Retarget it at `sqlValueIdentical` and check the
  assertion still holds for the fixture values — if the fixture includes a
  bigint/number pair the hash key would now disagree, extend or narrow the
  fixture rather than loosening the assertion.
- `test/utility-edge-cases.spec.ts` (~424) has a `sqlValuesEqual` describe block.
  Rename it and **add** cases for `5n` vs `5`, `true` vs `1`, and `1` vs `'1'`.
