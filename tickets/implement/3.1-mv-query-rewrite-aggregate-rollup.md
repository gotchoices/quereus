## Goal

The headline payoff on the MV rewrite roadmap. Recognize when a
`group by g1,…,gk agg(...)` query is answered by a grouped MV and rewrite to read
the MV's backing table instead of re-scanning + re-aggregating the base.

```sql
create materialized view daily as
  select d, sum(amt) total from sales group by d;

select d, sum(amt) from sales group by d where d >= '2026-01-01';  -- → scan daily, residual filter on d
select sum(amt) from sales;                                        -- → re-aggregate over daily (no group key)
```

Two sub-cases, both gated by the same soundness contract as the foundation (a
false Match returns wrong rows; forgo on doubt):

- **Exact-key match** — the query's GROUP BY key equals the MV's group key (as a
  set of source columns) and the query's aggregates are a subset of the MV's
  stored aggregates. The MV's backing rows *are* the answer: scan directly, with
  an optional residual Filter on the group-key columns (a `where` on a group
  column, as in the `d >= ...` example) and a residual Project selecting/reordering
  the requested output columns. No re-aggregation.

- **Superset-key (rollup) match** — the query groups on a *subset* of the MV's
  group key (the MV is finer-grained). Re-aggregate the MV's backing rows down to
  the query's coarser key. Sound **only for aggregates that compose under
  re-aggregation**:
  - `sum(x)` → `sum(mv.sum_x)`; `count(*)` / `count(x)` → `sum(mv.cnt)`;
    `min`/`max` → `min`/`max` of the partials; `sum`-derived only.
  - `avg(x)` is **not** directly re-aggregatable from a stored `avg` — it needs a
    stored `sum` **and** `count` to recombine. v1: rewrite `avg` only when the MV
    stores both the matching `sum(x)` and a `count` (recombine as
    `sum(sum_x)/sum(cnt)`); otherwise forgo.
  - `count(distinct x)`, `group_concat`, and any non-decomposable / order-sensitive
    aggregate ⇒ **forgo** (a false Match here is the classic rollup correctness
    trap). Maintain an explicit allowlist of decomposable aggregates; default-deny.
  - The global-scalar query (`select sum(amt) from sales`, no GROUP BY) is the
    degenerate superset case: re-aggregate every backing row into one group.

## Matcher extension

Add to `query-rewrite-matcher.ts` an aggregate arm that fires when the **fragment
root is an aggregate** (`StreamAggregate` / `HashAggregate` over the same single
source `T`). Reuse the foundation's source gates, shape walk, and predicate
entailment for the pre-aggregation WHERE.

New obligations:
- **Group-key alignment.** Map the fragment's GROUP BY expressions and the MV's
  GROUP BY to **bare source-column indices** (a computed group key on either side
  ⇒ forgo, mirroring the row-time aggregate eligibility gate which requires bare
  group columns). Query key ⊆ MV key ⇒ rollup; query key == MV key ⇒ exact.
  Query key ⊄ MV key ⇒ NotMatch.
- **Aggregate decomposition.** For each fragment aggregate, find the stored MV
  aggregate(s) that reconstruct it (the allowlist above), recording the recombine
  recipe (passthrough for exact-key, or the `sum`/`min`/`max`/avg-recombine for
  rollup). Any aggregate without a recipe ⇒ NotMatch.
- **Predicate alignment.** A query WHERE on a *group-key* column becomes a residual
  Filter on the backing rows **before** re-aggregation (it partitions whole
  groups, so it commutes with the rollup). A query WHERE on a *non-group, non-stored*
  column cannot be applied post-materialization ⇒ NotMatch (the MV already
  aggregated those rows away). A query `having` clause maps to a residual Filter on
  the re-aggregated output.

Extend `RewriteMatch` with an optional aggregate-rollup descriptor:
```ts
readonly rollup?: {
  readonly groupKeyBackingCols: readonly number[]; // query group key in backing-col indices ([] for global)
  readonly aggregates: readonly AggregateRecipe[];  // how to reconstruct each fragment aggregate
  readonly exact: boolean;                          // true ⇒ scan directly, no re-aggregation
};
```

The `proveEffectiveKeyUnique` primitive in `coverage-prover.ts` is the right
witness that the MV's backing relation is one row per MV group key (so the
exact-key direct scan is sound and the rollup re-aggregates a set, not a bag).

## Rule + emission

In `rule-materialized-view-rewrite.ts`, when the match carries a `rollup`:
- **Exact + no residual** → backing `TableReference` + residual Project (reorder
  to fragment output) preserving fragment output attribute ids.
- **Exact + group-key WHERE** → add a residual Filter on the group-key columns
  below the Project.
- **Rollup** → backing scan → optional residual Filter on group-key columns → a
  **new StreamAggregate/HashAggregate** grouping on `groupKeyBackingCols` whose
  aggregate expressions are the recombine recipes (`sum(mv.sum_x)`, etc.) → optional
  `having` residual Filter → residual Project. The synthesized aggregate must emit
  the fragment's output attribute ids.

**Cost gate** (`cost/index.ts`): exact-key is `seqScanCost(mvGroups)` (tiny);
rollup adds `streamAggregateCost`/`hashAggregateCost(mvGroups, queryGroups)`. Both
compared to recompute-over-base (`seqScanCost(baseRows) + aggregateCost(...)`).
The MV path nearly always wins because `mvGroups ≪ baseRows`, but a tiny base or a
highly selective base predicate may not — keep the gate honest.

## Tests (TDD)

Unit (matcher): exact-key match (with and without group-key WHERE); rollup `sum`
and `count`; rollup `avg` recombine from stored sum+count; global-scalar rollup;
NotMatch for `count(distinct)` / `group_concat`; NotMatch for a WHERE on a
non-group column; NotMatch for a computed group key.

Golden plans: `select d, sum(amt) from sales group by d` → `_mv_daily` scan;
`select sum(amt) from sales` → backing scan + re-aggregate node; a near-miss
(coarser MV than query, i.e. query key ⊄ MV key) → base recompute.

Equivalence harness (extends the foundation harness with aggregate shapes):
`rewritten == unrewritten` over random base seeds for each rollup shape —
**critically including empty groups, all-NULL groups, and `sum` over zero rows**
(must yield identical NULL/0 semantics rewritten vs not). Run rule-enabled vs
rule-disabled.
