## Why prereq on both core and multisource

`default_for` (omitted-insert default) and the validation/merge plumbing depend only on the
substrate core. But `target` / `exclude` / `delete_via` consume the **multi-source candidate
base set** that `view-mutation-multisource-innerjoin` produces — narrowing a one-element base-op
list is a no-op. Sequence after both so the routing tags have something to route.

## Statement-level tag parsing (decision needed)

Today a DML statement's `with (...)` clause parses into `stmt.contextValues`
(`AST.ContextAssignment[]`, name=value pairs) — the mutation-context surface — NOT a distinct
tags field (`InsertStmt`/`UpdateStmt`/`DeleteStmt` have no `tags`; only `CreateViewStmt.tags`
exists, threaded to `ViewSchema.tags`). The doc syntax
`insert into v with ("quereus.update.default_for.created" = epoch_ms('now'))` therefore currently
lands as a context assignment whose name is the reserved key. **Decide and document one of:**
- (a) Filter reserved `quereus.*` keys out of `contextValues` at build time and route them to the
  tag surface (smallest parser change; risk: a real mutation-context var named `quereus.*` — but
  that is already reserved, so safe), OR
- (b) Add a distinct statement-level tags clause to the parser/AST (cleaner separation; larger
  parser change).
Option (a) is the lower-risk path and is recommended unless the parser already distinguishes them.

## Collection, validation, merge

- View-level tags: `ViewSchema.tags` validated at `TagSite 'view-ddl'`.
- Statement-level tags: the separated reserved keys validated at `TagSite 'dml-stmt'`.
- Run `validateReservedTags(tags, site)`; for each returned `TagDiagnostic`, raise errors (and
  surface warnings) as **sited** diagnostics with statement/DDL location. An unknown or mis-sited
  reserved key is a hard error (the registry returns the issue; raise it with location). This
  realizes the acceptance criterion "unknown/mis-sited reserved key → sited diagnostic".
- Merge statement-level over view-level for the statement's duration (statement wins on key
  collision), per § Tags ("Statement-level tags override view-level tags").
- Read typed values via `getReservedTag` / `getReservedTagByTemplate` (already exported) — never
  re-parse the namespace.

## Propagation consumption (in `propagate.ts`)

Feed the merged validated tags into `MutationRequest.tags`; the visitor consumes them:
- `target` / `exclude` — narrow the candidate base set; never broaden past what predicates allow
  (tags compose AFTER predicate dispatch). An unknown branch/table name → `tag-target-not-found`.
- `default_for.<col>` — supply an omitted insert value (slots into the Projection insert
  default-fill chain from `view-mutation-multisource-innerjoin`, ahead of the base column default).
- `delete_via` — pick the deletion side for an ambiguous multi-side delete.
- `policy=strict` — reject any residual ambiguity with a structured diagnostic (vs the default
  `lenient` predicate-honest fan-out).
