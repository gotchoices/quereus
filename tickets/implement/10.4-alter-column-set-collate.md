# ALTER TABLE … ALTER COLUMN … SET COLLATE

## Why

A column's collation lives on `ColumnSchema.collation` (a string, default `'BINARY'`), is set at
definition time via the `COLLATE` column constraint, and is readable from SQL via
`table_info().collation`. But it is **absent from the declarative-diff catalog projection**
(`CatalogTable.columns` carries only `{ name, type, notNull, primaryKey, defaultValue, tags }`), so
the differ is **blind** to a collation change, and there is **no `ALTER COLUMN … SET COLLATE`**
primitive. A column drop+recreate is the only current path — and that throws away the column's data
for what should be a cheap metadata change. This ticket closes the gap as a sibling of the existing
`ALTER COLUMN SET DATA TYPE` / `SET DEFAULT` / `SET NOT NULL` verbs.

(Unlike tags, collation is **semantic** — it changes comparison/ordering — so it participates in the
schema hash exactly as it does at create time. Do **not** treat it like a tag in the hasher.)

## Design

```sql
alter table t alter column c set collate nocase;
```

- **AST**: extend the `alterColumn` action (parser/ast.ts) with `setCollation?: string`.
- **Parser** (`alterColumnAction`): add `SET COLLATE <name>` alongside the existing
  `SET DATA TYPE` / `SET DEFAULT` / `SET NOT NULL` / `DROP …` subcommands. (`COLLATE`/`COLLATION`
  keyword — match the column-constraint parse at parser.ts:3901.)
- **Node**: add `setCollation` to `AlterTableNode`'s `alterColumn` action.
- **Build**: thread it in `building/alter-table.ts`.
- **Emit** (`runAlterColumn`): validate the collation name against the column's
  `logicalType.supportedCollations` (reuse the create-time validation in `schema/table.ts:~193`,
  which already throws `Unknown collation '…' for type '…'`). Route the change through
  `module.alterTable` so a backing module can re-key / re-sort any structure that orders by this
  column (see SchemaChangeInfo below). Update `ColumnSchema.collation` and re-register the schema;
  fire `table_modified`.
- **Module** (`vtab/module.ts`): add `setCollation?: string` to the `alterColumn` member of
  `SchemaChangeInfo`. The **memory** module updates the column collation and rebuilds any index /
  unique / PK BTree that orders by the column under the new collation (re-validating uniqueness — a
  set unique under `BINARY` may collide under `NOCASE`). The **store** module mirrors and
  re-persists DDL.
- **Catalog** (`catalog.ts`): add `collation` to the `CatalogTable.columns` projection (default
  `'BINARY'`).
- **Differ** (`computeColumnAttributeChange`): extract the declared column's collation (its `COLLATE`
  constraint, default `'BINARY'`), compare case-insensitively to actual; on drift set
  `ColumnAttributeChange.collation`. `generateMigrationDDL` emits `ALTER TABLE <t> ALTER COLUMN <c>
  SET COLLATE <name>` — order it right after `SET DATA TYPE` (both are comparison-domain changes),
  before `SET DEFAULT` / `SET NOT NULL`.
- **Stringify** (`alterColumnToString`): add the `setCollation` arm.

## Edge cases & interactions

- **Unknown / unsupported collation** for the column's logical type → reject with the same error
  shape as create time (validate against `logicalType.supportedCollations`).
- **Column participates in PK / UNIQUE / index**: changing collation changes ordering and
  uniqueness semantics — the module must re-sort the structure and re-validate; a previously-unique
  value set that collides under the new collation fails with `CONSTRAINT`. Cover both the succeeding
  and the colliding case.
- **Schema hash DOES change** on a collation ALTER (collation is real schema, included in
  `computeSchemaHash` via the column AST). Assert the hash moves — the inverse of the tag tickets'
  "hash unchanged" assertion.
- **Default `'BINARY'`**: setting a column back to `BINARY` clears any non-default collation; the
  catalog/differ treat absent and `'BINARY'` as equal (no spurious diff).
- **`committed.*` snapshot / savepoints**: collation is schema-level; the committed pseudo-schema
  view is unaffected mid-transaction.
- **PK column collation** is also tracked on `PrimaryKeyColumnDefinition.collation` (see
  `schema-equivalence.ts`); confirm a `SET COLLATE` on a PK column keeps the PK-def collation in
  sync (or document that PK-column collation changes are out of scope and must go via
  `ALTER PRIMARY KEY`).
- **Store parity**: run the collation `.sqllogic` under `yarn test:store`.
