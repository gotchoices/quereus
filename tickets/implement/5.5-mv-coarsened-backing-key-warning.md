# Coarsened backing key + key-coarsening warning for materialized views

## Current behavior (verified in the fix stage, ticket `collation-weakening-key-claims`)

Over `create table t (b text primary key)`:

- `create materialized view m as select b collate nocase as b from t` **cannot
  be created at all**: `keysOf` over the body root is empty (the collate
  projection drops out of the key-propagation map ŌĆö correctly, since `{b}`
  under NOCASE is not a key of that output), so `buildFullRebuildPlan` throws
  `cannotMaterialize(... 'no provable unique key ŌĆö it is a bag')`.
- With colliding source data (`'Bob'`,`'bob'`) the create fails *earlier*, at
  fill: `deriveBackingShape`'s all-columns fallback PK is `{b}` with the output
  NOCASE collation, and the backing host's duplicate guard raises the
  "must be a set" diagnostic. This is the loud at-deploy rejection
  migration.md specifies, and it works today.
- Steady-state maintenance after a clean fill is **unreachable** today (the
  create never succeeds), so the in-window LWW-merge path has never run.

So the engine is conservative-sound here, but the parallel-migration pattern
(`docs/migration.md`: `Contact_v2` derived from `Contact_v1` with a collation
change on the key) is blocked.

## Design

**Coarsened key recognition is an MV-create policy, not a planner key fact.**
The output column set is *not* a key of the body relation (two source rows can
collide under the output collation), so it must never enter `keysOf` /
`RelationType.keys`. It is the *intended backing identity*: create-fill rejects
collisions loudly; steady-state upsert merges last-writer-wins.

At MV create (both `deriveBackingShape` and `buildFullRebuildPlan` ŌĆö they must
agree; consider deriving once and threading), when `keysOf(root)` is empty:

1. Trace each output column to a base attribute through **value-preserving
   passthrough chains only** (`ColumnReferenceNode` / `CollateNode` / no-op
   `CastNode` ŌĆö the `traceInvertibleColumn` passthrough subset with no
   `inverse` steps). The existing `updateLineage` surface already carries this.
2. If some source key K (of the single source table) is fully covered by such
   outputs, the corresponding output set K' is the **coarsened backing key**:
   use it as the backing PK (its per-column collation is the *output*
   collation ŌĆö `buildBackingTableSchema` already stamps `pkDefinition`
   collation from the output column type, and the fill duplicate guard is
   already loud).
3. Determine whether K' functionally determines the source primary key under
   the output collations ŌĆö i.e. whether any output key column's collation is
   strictly coarser than the source key column's enforcement collation
   (BINARY ŌåÆ NOCASE/RTRIM coarsens; equal collations do not). When coarsened,
   **emit the key-coarsening warning** rather than rejecting:

   > materialized view '<name>': backing key (<cols>) is coarser than the
   > source primary key (collation <src> ŌåÆ <out>); colliding source rows will
   > last-write-win until they are merged

   Channel: prefer the lens advisory surface (`lens-ack.ts` /
   declared-schema-manager reports ŌĆö acknowledgeable, anti-fatigue governed,
   surfaced via the explain/report builtins) if MV create can reach it
   cleanly; otherwise a structured logger warning plus a documented tag on the
   MV record. Decide in implementation; the doc contract is "warn, don't
   reject".
4. When no coarsened key is derivable either, keep today's
   `cannotMaterialize` bag rejection unchanged.

**Maintenance semantics** (document, don't engineer around):

- Row-time upsert keyed on K' merges colliding source rows last-writer-wins ŌĆö
  this is the *intended* in-window behavior per migration.md (deterministic
  oscillation until source rows are merged).
- A source delete of one colliding row deletes the shared derived row even
  while the sibling source row lives; the next edit of the sibling (or a
  REFRESH / full rebuild) re-asserts it. Document this anomaly in
  materialized-views.md alongside the warning; full-rebuild paths recover it,
  which is the correctness backstop.
- The mandatory row-time eligibility gate must accept the coarsened key for
  these bodies (the body is a passthrough projection ŌĆö the simplest
  maintainable shape).

**Coverage-prover caution:** once these MVs exist, `proveCoverage` /
`linkCoveredUniqueConstraints` could match a NOCASE-keyed backing against a
BINARY UNIQUE constraint. The link is informational today, but add a collation
match (constraint column collation vs backing key collation) to the prover so
the link is never established across a collation mismatch.

**Interaction with `collation-blind-equality-fact-extraction`:** none ŌĆö
that ticket gates Filter/guard equality facts; this one adds a create-time
policy below `keysOf`. No prereq either way. (If the group-by key completeness
backlog ticket ever lands, `... group by b collate nocase` bodies gain a real
key and bypass this path entirely; the coarsened path remains necessary for
multi-column passthrough bodies, which cannot be grouped.)

Runtime collision telemetry (the operational complement migration.md names) is
parked in backlog: `mv-collation-collision-telemetry`.
