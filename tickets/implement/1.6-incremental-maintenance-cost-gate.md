## Purpose

After `incremental-maintenance-plan-abstraction` introduces the `MaintenancePlan` strategy
union, the MV create-time gate is still a hard **shape allowlist**: every body that is not
the `'inverse-projection'` covering-index shape is rejected unconditionally. This ticket
replaces that allowlist gate with a **cost-gated choice among sound strategies**, delivers
the missing backward (maintenance-direction) cost surface, and subsumes the ad-hoc
`deltaPerRowFallbackRatio` heuristic.

The engine has a rich forward optimizer cost model (`planner/cost/index.ts` formulas,
`StatsProvider.tableRows` / `distinctValues` in `planner/stats/index.ts`) but **nothing
on the maintenance direction**. Today eligibility is a hard shape allowlist
(`buildRowTimePlan` rejects anything but the covering-index shape); each follow-on ticket
would widen that allowlist by hand without a shared cost judgment. This ticket is the
shared cost judgment.

The cost gate is a prerequisite for `materialized-view-rowtime-general-bodies`, which adds
new reachable arms; without a cost gate those arms would need bespoke affordability logic
that recreates the gap the spike named.

## The `maintenanceCost` surface

A small pure function added to `planner/cost/index.ts` alongside the existing forward
cost helpers:

```ts
export interface MaintenanceSourceStats {
  tableRows: number;
  distinctGroupsEstimate?: number;   // from StatsProvider.distinctValues when available
  forwardBodyCost: number;           // pre-computed by forward optimizer at create time
}

export function maintenanceCost(
  strategy: MaintenanceStrategy,
  changeCardinality: number,         // distinct changed rows/groups this statement
  stats: MaintenanceSourceStats,
): number;
```

`changeCardinality` is known at the DML boundary for runtime evaluation and estimated at
create time (e.g. `tableRows × 0.01` for typical OLTP) for plan-time decisions.

### Cost formulas per arm

- `'inverse-projection'`: `changeCardinality × (INDEX_SEEK_PER_ROW + PROJECT_PER_ROW)`.
  The O(1)-per-row write cost; the arm is always cheapest for its eligible shapes and is
  never demoted to `'full-rebuild'`.

- `'residual-recompute'`: `changeCardinality × residualCostPerGroup`, where
  `residualCostPerGroup` is the forward cost of the key-filtered residual body estimated
  against `stats.tableRows / stats.distinctGroupsEstimate`. **When stats are absent**,
  falls back to `changeCardinality × stats.forwardBodyCost × deltaPerRowFallbackRatio` —
  byte-for-byte the existing `deltaPerRowFallbackRatio` heuristic so behavior is unchanged
  on the no-stats path.

- `'full-rebuild'`: `stats.forwardBodyCost`. The always-correct floor.

### Decision rule at create time

Inside `buildRowTimePlan` (or its replacement), after the soundness check determines which
strategies are structurally valid for the body shape:

```
chosen = argmin over sound_strategies of maintenanceCost(strategy, estimatedChangeCardinality, stats)
```

The chosen strategy is stored in the `MaintenancePlan` record and used for every subsequent
write. The cost gate is a **planning-time decision**; it is not re-evaluated per write.

### Runtime demotion (`degradeToRebuild`)

At the DML boundary (`maintainRowTime`), if the actual `changeCardinality` of the current
statement causes `maintenanceCost('residual-recompute', actual, stats) >
maintenanceCost('full-rebuild', ...)`, the driver sets `degradeToRebuild = true` on the
arm and executes a full body re-evaluation for that statement only. The MV retains its
stored strategy; subsequent writes with lower cardinality revert to `'residual-recompute'`.
This per-write escape hatch prevents a single bulk statement from producing stale data
without invalidating the MV entirely.

## Synchronous policy: degrade-vs-reject classification

Under the synchronous (row-time, in-transaction) policy, a per-write `'full-rebuild'` for
a large source is pathological — it turns every DML statement into a full body scan. The
cost gate must encode which shapes may degrade and which must be rejected at create time:

**Must reject at create** (not eligible for row-time):
- Bodies whose only sound strategy is `'full-rebuild'` and whose `forwardBodyCost`
  exceeds a configured threshold (`TABLE_ROWS_THRESHOLD × SEQ_SCAN_PER_ROW`, i.e. the
  cost of a full scan of the estimated source). These would make every write on the source
  table do a full scan of it.
- Non-deterministic projections and bag bodies — unchanged hard rejections (soundness,
  not cost).

**May degrade to rebuild per write** (MV retained, single statement demoted):
- Bodies where `'residual-recompute'` is sound but runtime `changeCardinality` spikes
  above the crossover point. The `degradeToRebuild` flag handles this.

**`'inverse-projection'`**: never degrades. The covering-index gate is a soundness
precondition; its cost is always below `'full-rebuild'` for eligible shapes.

## Subsumption of `deltaPerRowFallbackRatio`

`DeltaExecutor`'s `deltaPerRowFallbackRatio` (currently `0.5`, threaded through
`OptimizerTuning` into `AssertionEvaluator` and `WatcherManager`) is a **detection-kernel
heuristic**: when changed binding tuples exceed `0.5 × tableRows`, the kernel demotes that
relation to global re-evaluation. It governs assertions and watchers, not MV maintenance.

This ticket does **not** remove `deltaPerRowFallbackRatio` or change the
assertion/watcher path. What changes:

- MV maintenance stops using an implicit ratio heuristic. Its affordability decision is
  now `maintenanceCost(...)`, with `deltaPerRowFallbackRatio`'s value repurposed as the
  **no-stats fallback multiplier** in the `'residual-recompute'` formula (see above). With
  stats present the formula replaces the ratio; without stats it reproduces identical
  behavior.
- `OptimizerTuning` and `docs/optimizer.md` are updated to clarify that
  `deltaPerRowFallbackRatio` governs the detection kernel only, and that MV maintenance
  cost uses `maintenanceCost(...)` with this value as the stats-absent fallback.

## Key tests

- **No-stats path**: create an MV with no analyzer stats; verify `maintenanceCost` uses
  the ratio fallback and produces output identical to the old `deltaPerRowFallbackRatio`
  threshold behavior.
- **With-stats crossover**: inject `MaintenanceSourceStats` values that make `'full-rebuild'`
  cheaper than `'residual-recompute'`; verify the gate selects `'full-rebuild'` (and, if
  `forwardBodyCost` exceeds the pathological threshold, `buildRowTimePlan` rejects at
  create).
- **Runtime demotion**: a statement whose actual `changeCardinality` crosses the crossover
  triggers `degradeToRebuild = true`; subsequent low-cardinality statements revert to
  `'residual-recompute'`.
- **Regression**: `53-materialized-views-rowtime.sqllogic` stays green — the covering-index
  MV is still `'inverse-projection'` and the cost gate never changes its strategy.
- **Property harness**: the maintenance-equivalence property harness shipped by
  `incremental-maintenance-plan-abstraction` stays green across all cost-gate paths,
  including the `degradeToRebuild` path.
- **Assertion/watcher neutrality**: delta dispatch with no stats present produces
  byte-identical decisions to the pre-change `deltaPerRowFallbackRatio` path.

## Out of scope

- Adding new reachable `MaintenancePlan` arms (`materialized-view-rowtime-general-bodies`).
- Removing or restructuring `deltaPerRowFallbackRatio` from `DeltaExecutor` /
  `OptimizerTuning` — assertions and watchers own that path.
- A persistent maintenance-statistics collector — `StatsProvider` heuristics are the v1
  input, consistent with the forward optimizer.
- Commit-time (post-transaction) maintenance policy.
