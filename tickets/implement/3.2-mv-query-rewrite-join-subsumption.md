## Goal

Recognize when a query joining `T` to a lookup table is covered by an MV whose
body is the **same 1:1 row-preserving inner/cross join** (the row-time
`'join-residual'` arm, eligibility shape 4 in `docs/materialized-views.md`), and
rewrite to scan the MV's backing table — eliminating the join at read time.

```sql
create materialized view enriched as
  select o.id, o.customer_id, o.amt, c.name
  from orders o join customers c on o.customer_id = c.id;   -- 1:1 (FK→PK, NOT NULL)

select o.id, o.amt, c.name
from orders o join customers c on o.customer_id = c.id
where o.amt > 100;                                          -- → scan _mv_enriched, residual filter + project
```

## Why this reuses existing work

The hard soundness question — "does this join contribute exactly one row per
governed `T` row?" — is *already solved* by `coverage-prover.ts`'s shared
`proveOneToOneJoin` (no-row-loss descent + `proveJoinNoFanout`). A 1:1 join's
output relation is in bijection with `T`'s governed rows, so two 1:1 joins over
the *same tables, same equi-pairs, same join type* produce the same row set. The
matcher's job is therefore: prove both the **fragment's** join and the **MV
body's** join are 1:1 over the same `(T, lookup, equi-pairs)`, then fall back to
the foundation's projection/filter subsumption over the joined output relation.

## Matcher extension

Add to `query-rewrite-matcher.ts` a join arm that fires when the **fragment root's
shape walk descends through a binary join** (reuse `walkToConstrainedBase` /
`proveOneToOneJoin` from `coverage-prover.ts` — they are already exported and
shared). Obligations:

- **Both sides 1:1 over the same join.** Run `proveOneToOneJoin` on both the
  fragment and the MV body. Require: same driving table `T`, same lookup table,
  same join type (inner/cross — outer is deferred, matching the row-time arm),
  and **equi-pair equivalence** (the fragment's equi-pairs equal the MV's, as a
  set, in attribute→source-column terms). A mismatch on any ⇒ NotMatch (`shape`).
- **Predicate subsumption over the joined relation.** Once both are proven 1:1,
  the joined output relations are equal, so the foundation's predicate-entailment
  + residual-filter logic applies directly to the *post-join* WHERE. A WHERE term
  on a lookup-side column is fine here (unlike the row-time partial-WHERE
  restriction) because we are only *reading* the already-materialized join, not
  maintaining it — entailment is over the stored joined rows.
- **Projection coverage** over the joined output (the MV must store every fragment
  output column, including lookup-side columns), via the stable-attribute-id map.

The join arm produces the same `RewriteMatch` the foundation emits (residual
clauses + output column map); no new descriptor field is needed — the
replacement is still backing scan → residual Filter → residual Project. The only
new logic is the **join-equivalence proof** that licenses treating the fragment's
join and the MV's join as the same relation.

## Rule + cost gate

Reuse the foundation's emission unchanged (the match shape is identical). Cost
gate: the recompute side now includes the join cost
(`hashJoinCost`/`nestedLoopJoinCost` + scans), so the MV scan wins more decisively
than the projection/filter case. Same cheapest-wins + stable-name-tiebreak.

## Tests (TDD)

Unit (matcher): positive — fragment join == MV join, both 1:1 (FK→PK NOT NULL),
fragment WHERE ⊇ MV WHERE (incl. a lookup-column residual). NotMatch — different
join type; different equi-pairs; a *fanning* (non-1:1) join on either side
(rejected by `proveJoinNoFanout`); MV omits a needed lookup column.

Golden plans: a 1:1-join query → `_mv_enriched` backing scan (no join in the
plan); a fanning-join near-miss → base recompute (join survives).

Equivalence harness (extends the foundation harness with join shapes):
`rewritten == unrewritten` over random seeds, including the FK→PK boundary
(every `T` row has its parent), NULL lookup columns, and the empty-lookup case.
Run rule-enabled vs rule-disabled.
