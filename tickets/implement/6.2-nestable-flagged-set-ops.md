## Goal

With the read half (`nestable-flagged-set-ops-read`) a nested set-op body plans and reads, but
every **write** that reaches a subtree operand throws (`set-op-membership-write` recurses each
branch through `propagate`, which rejects a `SetOperationNode` body — `propagate.ts:244-252` /
`classifyViewBody`):

```
update V set inSub = true   → cannot write through view '__setop_right': 'SetOperation' not updateable in phase 1
update V set x = 99         → (same — data fan-out hits the subtree branch)
delete from V where …       → (same)
update V set inA = false    → OK (leaf branch)
```

Make a subtree operand **recursively writable** for the unambiguous fan-out operations, and
reject the ambiguous insert-into-a-subtree operations cleanly (their product-coordinate "which
leaf" model is the backlog `set-op-membership-nested`). Independently valuable: makes any
nested set-op view writable (data/delete), not only flagged ones.

## What this ticket delivers vs. defers (decided — do not re-open)

**Delivers** (correct behavior, no throw):

- `update V set <dataCol> = v [where …]` — data fan-out recurses through a subtree operand to
  every member leaf (A, B, C).
- `delete from V where …` — delete fan-out recurses to every member leaf.
- `update V set <subtreeFlag> = false [where …]` — drops the row from the subtree (a delete
  fan-out into the subtree's leaves).
- `update V set <leafFlag> = true/false` — unchanged leaf path (already works).
- A **flag-less set-op subtree operand** (`A union[inA,inSub] (B union C)`) becomes writable for
  data/delete through the same recursion.

**Defers** to `set-op-membership-nested` (reject with a clear, correct diagnostic — NOT the
misleading "SetOperation is not updateable in phase 1", and NOT `unknown-view-column`):

- `update V set <subtreeFlag> = true` — insert into a multi-leaf subtree ("which leaf?" =
  product-coordinate addressing).
- `update V set <surfacedInnerFlag> = …` (writing `inB`/`inC` through the outer view) — nested
  per-leaf addressing.
- **Insert-through** whose active routing flag is a subtree side (`insert into V(…, inSub, …)
  values (…, true)`) — can't route a VALUES row into a multi-leaf subtree.

Rationale for the insert/inner-flag deferral: inserting "make this row a member of B∪C" has no
single deterministic leaf, and writing a surfaced inner flag means addressing a branch *inside*
the operand — both are exactly the product-coordinate / multi-target model the backlog ticket
owns. The fan-out (update/delete) and `=false` paths are unambiguous (touch every member leaf),
so they ship here as the structural foundation that backlog recurses on.

## Mechanism — recursive fan-out reusing the one capture

`set-op-membership-write` already captures the affected view rows + their probe flags **once**
into the context-backed `__vmupd_keys` relation (Halloween-safe) before any branch op fires,
then fans each branch out as a `BaseOp` lowered through `propagate` against a synthetic branch
view-like, correlated by a NULL-safe full-data-tuple `exists` against `__vmupd_keys`
(`buildMemberExists`). The nested recursion **reuses that single capture** — it does NOT build a
second one:

- A subtree branch's leaves share the outer's data columns (nesting preserves data columns), so
  "delete/update the leaf rows whose data tuple ∈ `__vmupd_keys`" is the same frozen-capture
  correlation, just rebuilt against each inner branch's column names. The outer's member-exists
  (`exists k: k.data = __setop_right.data`) and a re-derived inner member-exists (`exists k:
  k.data = B.data`) select the same rows for the leaf, because the capture is frozen and on data
  tuples (a leaf's rows ⊆ the subtree's rows). So the recursion needs no new capture and returns
  ordinary `BaseOp[]` — which is exactly what the `propagate`/`BaseOp[]` model can carry. This is
  why the fan-out (no insert) recursion is tractable while insert-into-subtree is not.

**Recurse explicitly in `set-op.ts`, not via `propagate`.** Detect a nested operand at branch
build time and route its per-branch op into a recursive fan-out helper; keep leaf branches on
`propagate`. This preserves `propagate`'s top-level reject semantics (a genuine top-level
flag-less `A union B` write still rejects — no membership column) while the recursion lives in
the set-op decomposer that owns the capture.

### `analyzeSetOpView` — fix data-column count for surfaced inner flags

Now that the planned root surfaces inner flags, `dataColCount = attrs.length - flags.length`
(`set-op.ts:233`) over-counts (it would treat `inB`,`inC` as data columns). Compute it from the
planned `SetOperationNode.dataColumnCount()` (recursive, exposed by the read half) instead:
`dataColCount = root.dataColumnCount()`; `dataColNames = viewColNames.slice(0, dataColCount)`.
Also expose the **surfaced inner-flag names** (the view columns that are neither data nor own
flags) so `buildUpdate` can classify a write to one and reject it with the nested diagnostic
rather than `unknown-view-column`.

### Branch nested-detection + recursion

In `buildBranch` mark `isNested = !!branchSelect.compound && branchSelect.compound.op !==
'diff'`. A nested branch needs a membership-free branch analysis (a flag-less subtree has no
`compound.existence`, which the membership-requiring `analyzeSetOpView` rejects) — factor a
light `analyzeSetOpBranches(ctx, branchView)` returning the inner branches + dataColNames +
`dataColumnCount`, without the membership gate. Then:

- `buildBranchDataUpdate`: if `branch.isNested`, recurse — for each inner branch call
  `buildBranchDataUpdate` again (fanning the SAME positional `dataAssignments`, each re-mapped to
  that inner branch's `dataColNames`, correlated by a freshly-built `buildMemberExists` against
  `__vmupd_keys`); else today's leaf path (`propagate`).
- `buildBranchMembershipDelete` / `buildDelete`'s per-branch loop: same nested recursion (delete
  fan-out to inner branches), else `propagate`.
- `buildBranchMembershipInsert`: if `branch.isNested`, **reject** (deferred — insert into a
  multi-leaf subtree); else today's leaf insert path. (`set <leafFlag> = true` still works.)

`buildUpdate` flag/data classification: a SET target that names a **surfaced inner flag** →
reject (deferred — surfaced-inner-flag write); a target that is an **own subtree flag** set to
`true` → reject at `buildBranchMembershipInsert` as above (the `false` flip routes to the delete
fan-out, which works).

`buildInsertThrough`: when an active routing side is nested (`branch.isNested`), reject
(deferred — insert-through into a subtree side).

All rejections use `raiseMutationDiagnostic({ reason: 'unsupported-set-op', … })` with a message
naming `set-op-membership-nested` and the specific unsupported shape.

### `propagate` backstop

Leave the existing nested-set-op-membership reject in `propagate` (`propagate.ts:244-252`) as a
defensive backstop — with explicit recursion in `set-op.ts`, no nested branch should reach it,
but a direct/unexpected `propagate` on such a body should still fail with a pointer to
`set-op-membership-nested` rather than mis-route into the single-source rewrite. (The
single-source path's `classifyViewBody` reject for a flag-less `SetOperation` body is likewise
the correct backstop for a genuine top-level flag-less set-op write.)

## Static surfaces (`func/builtins/schema.ts`, `mutation/set-op.ts`)

`isSetOpBranchWritable` currently passes a nested-compound operand as a one-level, deliberate
over-claim (`set-op.ts:96-122`). Tighten it to **recurse**: an operand is branch-writable iff it
is a plain-leg leaf (matching arity) OR a (recursively) branch-writable set-op body. So a nested
view reports `is_updatable = YES` / `is_deletable = YES` (data + delete fan-out genuinely work),
agreeing with the dynamic accept.

Inserts into a subtree are deferred, so gate `is_insertable_into` to `NO` when **any** operand is
a subtree (a `compound` operand). Conservative under-claim (honest), small and well-defined; do
NOT attempt per-VALUES-row leaf routing here (that is the backlog ticket).

`column_info`: data columns and own **leaf** flags report `YES` via the existing generic
`updateLineage` path; **surfaced inner flags** (`inB`,`inC`) report `NO` naturally — the outer
node's `updateLineage` carries no site for their (inner) attribute ids, so `baseSiteOf` is
absent. An own **subtree flag** (`inSub`) carries an `existence` site → the generic path reports
`is_updatable = YES`; this is a documented partial (the `= false` delete works; the `= true`
insert is the deferred case) — note it in the v1-limitations doc paragraph rather than adding
deep per-operand column_info coupling. Do not over-engineer column_info beyond this.

## Edge cases & interactions

- `update V set x = 99 where id = k` over `A union[inA,inSub] (B∪[inB,inC]C)` updates the row in
  every member leaf (A and/or B and/or C); a non-member leaf matches no row (the full-tuple
  `exists` correlation), so no spurious writes. PutGet: the view re-reads the new value and
  never escapes the user predicate. No `'SetOperation' not updateable` throw.
- `delete from V where …` removes the row from every member leaf at every depth.
- `set inSub = false where …` deletes the row from B and C (recursing the subtree's delete
  fan-out); the row leaves the view. `set inSub = true` rejects cleanly (→ nested).
- **Flag-less subtree operand** (`A union[inA,inSub] (B union C)`): data fan-out + delete now
  succeed (the subtree is writable through the recursion); membership flips on `inSub` behave as
  above (`false` deletes, `true` defers).
- `intersect` / `except` **subtree operand** (not just `union`) as the inner node: delete/data
  fan-out recurses through it; `set inSub = false` deletes from the subtree's resident leaves.
- **Halloween-safety preserved at depth**: every leaf op (any depth) correlates on the ONE
  frozen `__vmupd_keys` capture; a leaf insert/delete cannot perturb the affected set out from
  under a sibling/deeper op (no second capture is introduced). Verify the existing
  `view-mutation.ts` drain path is untouched.
- **Data fan-out value referencing a data column** (`set x = x + 1`): the v1 caveat (refs
  resolve against branch columns when leg names match) recurses — at each level the value is
  cloned and re-mapped to that branch's columns; matching leg names round-trip, a renamed
  referenced column stays the documented limitation. Literal values are unaffected.
- **Surfaced-inner-flag write** (`set inB = true`) rejects with a nested-deferral message (NOT
  `unknown-view-column` — `inB` IS a view column now).
- **Insert-through** with a leaf-only active side still works; with a subtree active side rejects
  cleanly.
- **Conflicting assignment** parity recurses: `set x = 5, inSub = false` (data write + branch
  removal targeting the same subtree) keeps rejecting `conflicting-assignment` (the existing
  `anyFalseFlip && dataAssignments.length > 0` guard).
- **Static vs dynamic agreement**: `view_info`/`column_info` for a nested view report YES exactly
  where the dynamic write succeeds (update/delete), NO where it defers (insert-through subtree
  side), with the documented `inSub = true` partial.
- **RETURNING / bound parameters**: RETURNING through a set-op write stays rejected at every
  depth; bound params in the WHERE resolve (the capture scope fix from `set-op-membership-write`
  is inherited — the recursion adds no new scope).

## Tests (`test/property.spec.ts` § Set-operation membership writes)

Add a nested schema (`A`/`B`/`C`) and cases:

- **Data fan-out through a subtree** — seed A/B/C with shared + disjoint rows; `update Vn set x
  = 99 where id = k` for an A-only, a (B∩C), and a B-only row; assert each member leaf updated
  and the view re-reads the new `x` (PutGet), no throw.
- **Delete fan-out through a subtree** — `delete from Vn where id = k`; assert removed from every
  member leaf.
- **`set inSub = false`** — drops the row from B and C; the view no longer shows it; `inSub`
  reads false / row gone.
- **Flag-less subtree writable** — `A union[inA,inSub] (B union C)`: data fan-out + delete
  succeed (regression that the prior throw is gone).
- **Deferred writes reject cleanly** — `set inSub = true`, `set inB = true`, and an
  insert-through routing to the subtree side each throw a diagnostic whose message names
  `set-op-membership-nested` (assert the throw + that it is NOT the old phase-1 message and NOT
  `unknown-view-column`).
- **Static surface honesty** — `view_info('Vn')`: `is_updatable=YES`, `is_deletable=YES`,
  `is_insertable_into=NO`; `column_info('Vn')`: data cols + `inA` (leaf flag) YES, `inB`/`inC`
  (surfaced inner) NO; cross-check the dynamic update/delete succeed and the deferred forms
  reject.
- **Key Soundness under write** — after a fan-out write the nested view stays keyed on data
  columns; no flag appears in a claimed key (Inv. 1–2 at depth).
