> **Split out of `3.2-materialized-view-rowtime-residual-recompute`.** That ticket
> shipped the shape-agnostic residual kernel + the single-source **aggregate** shape
> (`'group'`/`'gk'` binding). This ticket adds the **1:1 join** shape (`'row'`/`'pk'`
> binding) on the same kernel. It was separated because the join arm carries two hard,
> self-contained sub-problems (the lookup-side change + the coverage-prover refactor)
> that don't belong in the aggregate pass.

## What this arm maintains

A **1:1 row-preserving inner/cross join** body where `T` contributes **exactly one** MV
row per governed `T` row, proven by the coverage prover's existing 1:1 decomposition
(no-row-loss via row-preservation **or** NOT-NULL FK→PK referential integrity; no-fan-out
via `proveJoinNoFanout` / `isUnique(T.pk)` at the join frame). Each changed `T` row maps
to one backing row keyed by `T`'s PK; maintenance recomputes that one joined row (join
fan-in over the live lookup side) and replaces it. **Fanning (non-1:1) joins are out of
scope** — reject them as `shape` (the existing fan-out gate already does); they need the
by-prefix delete of `materialized-view-rowtime-prefix-delete-lateral-tvf`.

## The residual kernel already exists — reuse it

`materialized-view-rowtime-residual-recompute` left `database-materialized-views.ts`
with a **shape-agnostic** residual kernel that this ticket should reuse directly:

- `compileResidual(analyzed, relKey, bindColumns, paramPrefix, reject)` — injects the
  key filter on `T` (pass `'pk'` here) and caches the `Scheduler`.
- `runResidual(plan, keyTuple)` — runs the cached residual bound to one key tuple against
  live mid-transaction source state (reads-own-writes), returns the result rows.
- `applyResidualRecompute(plan, change, cache)` — derives affected keys (OLD ∪ NEW,
  deduped), deletes the old backing slice, runs the residual, upserts only rows whose
  backing key matches (`residualRowMatchesKey`). **Already generic over `'pk'`/`'gk'`** —
  for a `'row'` binding the affected key is `T`'s PK and the residual returns ≤1 joined
  row.

`ResidualRecomputePlan` already carries `bindParamPrefix: 'gk' | 'pk'`, `bindColumns`,
`backingPkDefinition`, `backingPkSourceCols`, and `binding: BindingMode`. So the join arm
is mostly a new **builder** (`buildJoinResidualPlan`) that admits the join, sets
`bindParamPrefix: 'pk'`, `bindColumns = T.pk` (in `T`'s output-column space), and
`backingPkSourceCols` mapping each backing-PK column to `T`'s PK source columns.

### The lookup-side (P) change problem — the hard part
- The MV's `sourceTables` includes **both** `T` and `P`, so `rowTimeBySource` fires
  maintenance on a write to `P` too. But the cached residual is keyed on `T`'s PK — a `P`
  change gives no single `T` key (one `P` row joins many `T` rows). Options to evaluate:
  - a **second residual keyed on `P`'s key** that finds the affected `T` rows (a
    `T.fk = :pk0` reverse lookup), recomputes each, and upserts — i.e. the plan carries a
    per-source binding (`T` → row/PK, `P` → reverse-lookup), OR
  - a **bounded global recompute over the matched `T` slice** for a `P` change.
  The ticket's test plan requires lookup-side mutations to be covered, so this must be
  solved, not deferred. Document the chosen mechanism.
