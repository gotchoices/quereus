## Why this one first

`docs/optimizer.md` is 2,676 lines / ~38,000 words. A single heading —
`### Functional Dependency Tracking` — is **10,154 words of that, 27% of the
file.** It is the most-referenced section in the repo (four inbound anchor links
plus source comments) and the substrate that view updateability, lenses,
materialized views, and half the optimizer rules all reason on top of.

It is also **misfiled**: it sits at `###` depth underneath
`## Visited Tracking Architecture`, a 1,200-word section about DAG traversal
bookkeeping that has nothing to do with functional dependencies. So do eight of
its neighbours. The outline of `optimizer.md` has collapsed; the table of contents
lies about what the document contains.

Extracting this one block is the largest single reduction available, has the
smallest blast radius of any split in this chain, and proves the checker from
`docs-doc-conventions-and-checker` on a real move before the wider split
(`docs-optimizer-split-satellites`) leans on it. Do it alone.

## Edge cases & interactions

- **Slug collisions after promotion.** `optimizer-fd.md` will contain headings
  named `Overview`, `Registration`, or similar that were unique at `####` depth
  inside one section but may now collide with each other at `##`. The checker's
  `-1` disambiguation will silently make links resolve to the *wrong* heading.
  Scan the promoted outline for duplicate names and rename rather than rely on
  positional suffixes.
- **The `### Attribute provenance` boundary.** It immediately precedes
  `### Functional Dependency Tracking`. Cut at the heading, not by line number —
  line offsets in this ticket are as-of-writing and other tickets in this run may
  have shifted them.
- **`docs/review.html` mentions FD sections.** It is a frozen review artifact.
  Exempt from the checker; do not update it.
- **The doc claims a code contract that is real.** `fd-utils.ts` exports
  `computeClosure` / `determines` / `closureCoversAll` / `isUniqueDeterminant` /
  `hasAnyKey` / `hasSingletonFd` / `deriveKeysFromFds` / `keysOf` / `isUnique` /
  `addFd` / `shiftFds` / `projectFds` / `stripGuard`. As you move each claim, spot-check
  the symbol still exists. If a claim names a symbol that is gone, that is genuine
  drift: fix the prose, and note it in the review handoff. Do not open a new ticket
  for a one-word correction.
- **`fdsEqual` is documented as ignoring `kind`, `source`, and `valueEquality`.**
  That is a normative statement with a real code site. It becomes an invariant in
  `docs-invariants-optimizer`. Do not extract invariants here — just don't lose it.
- **Ratchet.** `optimizer.md` drops by ~10,150 words; `optimizer-fd.md` is new and
  under cap. Run `node scripts/check-docs.js --update-ratchet` to lower
  `docs/optimizer.md`'s entry, and confirm the tool refuses to add an entry for a
  doc that is under cap.
- **Concurrent edits to `optimizer.md`.** Sibling ticket `docs-mv-split` touches
  `materialized-views.md`, which links *into* `optimizer.md` twice
  (`#materialized-view-query-rewrite-read-side`, `#coverage-proving`,
  `#effective-key-proving-body-proves-it`). Those anchors are not in the FD subtree
  and must survive this move untouched. Verify with the checker after both land.
