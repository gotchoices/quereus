# Index canonical body: include per-column collation

Follow-on to `schema-differ-index-body-drift` (landed). That ticket deliberately
**excluded** per-column collation from `createIndexBodyToCanonicalString` because
the declared side carries only as-written `COLLATE`, while the actual
`IndexSchema` always stores a *resolved, explicit, normalized* collation — so a
naive include would churn an inherited/default-`BINARY` collation on every diff.

This ticket includes collation correctly by resolving the **declared** side's
effective collation the same way the engine does at create/import time, then
rendering both sides identically (BINARY elided, non-BINARY explicit).

## Background: how collation resolves today

Both live create (`buildIndexSchema`, manager.ts:2056-2078) and catalog import
(`importIndex`, manager.ts:2457-2472) resolve each index column's collation as:

```
normalizeCollationName( per-column COLLATE  ||  table column's collation  ||  'BINARY' )
```

and store it on `IndexColumnSchema.collation` (always set, always normalized
uppercase — e.g. `'BINARY'`, `'NOCASE'`). The table column's collation comes from
a `collate` *column constraint* on the `ColumnDef` (default `'BINARY'`, see
`columnDefToSchema`, table.ts).

So on the **actual** side, `indexSchema.columns[i].collation` is the fully
resolved effective collation. On the **declared** side (a raw parsed
`CreateIndexStmt`), only an *explicit* per-column `COLLATE` is present — inherited
collation is not, because it lives on the declared table's `ColumnDef`.

The parser folds `col COLLATE x` into a `collate` expression wrapping a column ref
(name on `col.expr.expr.name`, collation on `col.expr.collation`); a plain column
carries `col.name` + optional `col.collation`. `indexedColumnBareName` already
unwraps both forms for the name.

## Design

Make the canonical renderer render an effective per-column collation that **both
sides pre-resolve identically**:

1. **Renderer** (`createIndexBodyToCanonicalString` / `canonicalIndexedColumnsToString`,
   ast-stringify.ts): render each named column as
   `quoteIdentifier(lowerName) [collate <c>] [desc]` — emitting `collate <c>` only
   when the column carries an effective collation that is **not** `BINARY`
   (compare case-insensitively; `BINARY`/absent both elide). Render order matches
   the persistence renderer `indexedColumnsToString`: name, then collate, then
   desc. The renderer reads the effective collation off `IndexedColumn.collation`
   (plain form). The collate-folded form's collation lives on `col.expr.collation`
   — but callers normalize to plain form before rendering (below), so the renderer
   only needs the plain-form read; keep a defensive fallback that reads the folded
   form's collation so a future raw caller doesn't silently drop collation. A
   genuine expression-index column (no resolvable name) still falls through to
   `expressionToString(col.expr)` unchanged.

2. **Actual side** (`indexToCanonicalDDL`, ddl-generator.ts): set each lifted
   `IndexedColumn.collation` to the resolved column collation. Prefer
   `indexSchema.columns[i].collation`; if (defensively) unset, fall back to
   `tableSchema.columns[col.index].collation`; normalize via
   `normalizeCollationName(... ?? 'BINARY')`. The lift already produces plain-form
   columns (`{ name, direction }`), so just add `collation`.

3. **Declared side** (schema-differ.ts index loop in `computeSchemaDiff`): replace
   the direct `createIndexBodyToCanonicalString(declaredIndex.indexStmt)` call with
   a new helper `declaredIndexCanonicalBody(indexStmt, declaredTable)` that builds a
   normalized plain-form column list, resolving each column's effective collation as
   `normalizeCollationName(explicitIndexColumnCollation(col) || declaredColumnCollation(declaredTable, bareName) || 'BINARY')`,
   then calls the renderer.
   - `explicitIndexColumnCollation(col)` reads `col.collation` or, for the
     collate-folded form, `col.expr.collation`.
   - `declaredColumnCollation(declaredTable, columnName)` reads the matching
     `ColumnDef`'s `collate` column constraint (`constraints.find(c => c.type ===
     'collate').collation`), returning `'BINARY'` when none / column not found.
   - `declaredTable` is looked up from the existing `declaredTables` map by
     `indexStmt.table.name.toLowerCase()`. If absent (index on a non-declared
     table — not a coherent name-match scenario), pass `undefined` and resolution
     falls back to explicit-or-BINARY.

Because both sides now feed the renderer an already-resolved, normalized collation,
an unchanged inherited/BINARY collation renders identically on both sides (no
churn), and a genuine collation change renders differently (drop+recreate).

### Table-column collation changing in the same diff (resolved decision)

When a declared table column's collation changes in the same diff (e.g. `name
text` → `name text collate nocase`) and a same-named index inherits it, the
**index follows the declared column and recreates**: the declared side resolves
the index column's collation from the *new* (declared) `ColumnDef`, so its
canonical body diverges from the actual (old-collation) body → drop+recreate.
This is correct — a column collation change requires the index's key ordering to
be rebuilt. The differ does **not** attempt to suppress the index recreate on the
theory that the column's own `SET COLLATE` (emitted as a `columnsToAlter`
`collation` change) might rebuild dependent indexes; an explicit recreate is the
safe, backend-agnostic choice. The potential redundancy (column `SET COLLATE`
rebuild + explicit index drop/recreate) is accepted and noted; do not optimize it
here. **Apply ordering** must run the column `SET COLLATE` before the index
recreate so the rebuilt index uses the new collation — verify the existing apply
order (column alters before index creates) already guarantees this; if not, flag
it (do not silently rely on it).

## Edge cases & interactions

- **Inherited BINARY, unchanged** — `index ix on t(name)` with `name text` (no
  COLLATE): both sides resolve `BINARY`, elided → **no churn**. (Most common case;
  this is the regression the exclusion originally protected against.)
- **Explicit index COLLATE added** — `index ix on t(email)` →
  `index ix on t(email collate nocase)`: declared resolves NOCASE, actual still
  BINARY → **drop+recreate**.
- **Inherited non-BINARY, unchanged** — `email text collate nocase` +
  `index ix on t(email)` (no explicit COLLATE on the index): both sides resolve
  NOCASE → **no churn**. Guards that the declared side reads the table-column
  collation, not just the index column.
- **Column collation change drives index recreate** — `name text` →
  `name text collate nocase` with `index ix on t(name)`: index **drops+recreates**
  (declared inherits NOCASE; actual stored BINARY) AND the column emits a
  `columnsToAlter` collation change. Assert both, and that the recreate DDL is
  present.
- **Explicit index COLLATE that equals the column default** — index explicitly
  `collate binary` on a BINARY column: both elide → **no churn** (normalization
  makes `binary`/`BINARY`/absent equivalent).
- **Collate-folded vs plain form symmetry** — `email collate nocase` (folded by
  the parser) must resolve the same as a hypothetical plain `col.collation`; the
  bare-name + collation extraction must cover both forms.
- **Composite index, one column's collation changes** — only that column's render
  differs → recreate; verify a composite whose collations are all unchanged does
  not churn.
- **Desc + collate ordering** — `name collate nocase desc` renders name→collate→desc
  on both sides; a desc index re-declared verbatim with inherited collation does
  not churn (extends the existing desc-symmetry guard).
- **require-hint policy** — a collation-driven recreate is a body change (counts in
  `indexBodyRecreates`), so it must stay excluded from the unhinted-rename guard
  exactly as other body changes are. Verify the existing `indexBodyRecreates`
  bookkeeping still covers it (no code change expected, but assert under
  `policy: 'require-hint'`).
- **Exposed implicit covering indexes** are already filtered out of the index loop
  (`!i.implicit`); collation inclusion must not change that — they never reach this
  path.

## Tests (add under the "declarative differ stability" describe in index-ddl-roundtrip.spec.ts)

Use the existing `diffIndexEdit(baseline, modified, policy?)` helper and `TABLE`
fixture. Expected outputs in parentheses.

- inherited BINARY unchanged → no drop, no create (the common no-churn case).
- adding an explicit index `collate nocase` → `indexesToDrop = ['ix']`, one recreate
  whose DDL matches `/collate nocase/i`.
- inherited NOCASE unchanged (`email text collate nocase`, index without explicit
  COLLATE, re-declared verbatim) → no drop, no create.
- changing the *column's* collation under a stable-named index → index drops+recreates
  AND `tablesToAlter` carries a `columnsToAlter` collation change for that column.
- explicit `collate binary` on a BINARY column → no churn.
- composite index, all collations unchanged → no churn; one column's collation added
  → recreate.
- `policy: 'require-hint'`: a pure collation body-change recreate does **not** throw
  (it is a recreate, not an unhinted rename).
