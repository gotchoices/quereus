# Surface inbound assertion violations to the host; converge instead of diverging

## Why

Today, when an inbound CRDT batch's column-LWW-merged row state violates a
**local** commit-time global assertion, the store adapter's seam call throws,
CRDT metadata is left uncommitted, and the next sync retry re-applies
value-identically → the storage write is suppressed → the seam batch is empty →
the assertion is never re-evaluated, metadata commits, and the **derived effects
for the violating row are never re-driven**. Net effect: an incremental
materialized view (and any `Database.watch` subscriber) silently diverges from
the base table for that row until a manual MV refresh. (Full analysis: plan
ticket `3-sync-seam-throw-retry-mv-divergence`.)

The resolved posture is **detect-and-notify** (trust-the-origin means the data
must land regardless; a local assertion can only usefully *notify*, not block):

- The engine seam gains a **report mode** (prereq ticket
  `sync-seam-assertion-report-mode`): the violation is **collected and returned**
  instead of thrown, and the batch **commits** — MV maintenance and watch capture
  for the violating row land on the **first** attempt, so there is no divergence
  and no retry suppression. CRDT metadata commits normally.
- This ticket is the **sync half**: drive the seam in report mode, surface any
  returned violations to the host as a typed, host-visible **sync event**, and
  rewrite the test that currently pins the divergence as expected behavior.

## Design

### Adapter drives report mode and returns violations

`createStoreAdapter` (`store-adapter.ts`) passes
`assertionFailureMode: 'report'` on the **incremental** seam call only
(`db.ingestExternalRowChanges(seamBatch, { applyForeignKeyActions, assertionFailureMode: 'report' })`).
Bootstrap flushes skip the seam entirely and bootstrapFinalize does not evaluate
assertions, so neither changes. The adapter copies the seam result's
`assertionViolations` into its `ApplyToStoreResult`:

```ts
// protocol.ts — ApplyToStoreResult gains:
readonly assertionViolations?: AssertionViolation[];   // from @quereus/quereus
```

Importing `AssertionViolation` from `@quereus/quereus` (exported by the prereq
ticket). The adapter populates it from the single end-of-invocation seam call's
return; empty/omitted when there are none.

The adapter no longer throws on the headline assertion case, so the whole
"throw → uncommitted metadata → retry → suppression" chain disappears. (Per-change
storage failures collected in `result.errors` are unchanged — that is the
separate `sync-apply-per-change-errors-ignored` concern.)

### Host-visible event (sync event emitter)

`events.ts` gains a typed event alongside `onUnknownTable` (the existing
host-visible-divergence precedent):

```ts
export interface AssertionViolationEvent {
  /** Name of the violated local assertion. */
  readonly assertion: string;
  /** Sample rows from the assertion's violation query (diagnostic; capped). */
  readonly samples: SqlValue[][];
}
// SyncEventEmitter: onAssertionViolation(listener): Unsubscribe
// SyncEventEmitterImpl: listener set + emitAssertionViolation(event)
```

Keep it minimal and assertion-scoped (an assertion may span several tables, so
no single schema/table is meaningful). The host decides policy (alert, audit,
trigger an out-of-band reconciliation). No MV refresh is scheduled — report mode
already kept the MV consistent with the base table.

### Emit point: `admission.ts`

`applyDataToStore(ctx, ...)` already owns the post-apply decision (it calls the
adapter and `throwIfApplyErrors`). After a **successful** apply (no throw, no
per-change errors), if `result.assertionViolations?.length`, emit one
`ctx.syncEvents.emitAssertionViolation(...)` per violation. This keeps the
adapter a pure `ApplyToStoreCallback` and routes host events through the sync
event emitter where `onUnknownTable` / `onSyncStateChange` live. Crucially: a
reported violation does **not** abort the apply — convergence completes,
metadata commits, the event is informational.

## Edge cases & interactions

- **No regression to per-change errors** — `result.errors` non-empty still
  throws via `throwIfApplyErrors` with no metadata committed; that path is
  orthogonal and must not swallow or be swallowed by the violation path. Order:
  surface violations only on the success branch (after `throwIfApplyErrors`
  passes).
- **Multiple violations in one apply** — emit one event each (the seam returns
  one entry per violated assertion).
- **Bootstrap / finalize** — unaffected (seam skipped / no assertion eval);
  assert no violation event fires on a bootstrap path.
- **No assertions configured** — seam returns `[]`; no event; identical to today
  minus the throw.
- **Convergence-on-first-attempt** — the rewritten test must prove the MV/watch
  converge on the FIRST apply (no retry needed) and CRDT metadata commits
  (`getChangesSince` relays the change) on that first attempt.
- **Idempotent re-apply** — a second apply of the same change set is a
  value-identical no-op (empty seam batch) → no second violation event, no
  double-relay. Pin this so the event does not re-fire on benign retries.
- **Watch subscriber sees the change** — a `Database.watch` hit on the violating
  row now fires (it did not before); cover it.
- **FK-actions opt-in interplay** — report mode is independent of
  `applyForeignKeyActions`; a deployment with both on still detects-and-notifies
  on assertion violation while cascades apply in-batch.

## Tests (`store-adapter-seam.spec.ts`)

Rewrite the `seam-throw propagation through the sync layer` describe block's
assertion test (`'assertion failure propagates, leaves CRDT metadata
uncommitted; retry converges'`). The new expectations, using the same
`SyncManagerImpl` + `non_negative` assertion + `-5` change set:

- `applyChanges` **resolves** (no throw); `result.applied > 0` on the **first**
  attempt.
- The violating row persists in the base table (`select v from t` → `[-5]`).
- The MV (add a covering MV over `t`) **also** shows `-5` — base/MV converged on
  the first attempt (the regression the old behavior failed).
- A `Database.watch` subscription on the row fired.
- CRDT metadata committed on the first attempt (`getChangesSince(other)` relays
  the change) — no second attempt required.
- An `onAssertionViolation` listener received one event naming `non_negative`
  with non-empty `samples`.
- A second identical `applyChanges` is a benign no-op: no new violation event,
  no double-relay.

Keep the sibling `'no CRDT echo: applying remote changes records nothing as
local'` test as-is (no assertions involved). Update the describe block's name/
doc since it no longer demonstrates a throw.
