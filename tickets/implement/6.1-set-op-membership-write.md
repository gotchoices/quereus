## Why

The read half (`set-op-membership-read`) reifies set-op branch provenance as first-class
`existence`-sited boolean columns and computes, per row, a runtime membership probe
(`inA ≡ tuple ∈ A`). This half delivers the payoff: the membership column **is** the branch
presence, and *writing* it drives the branch's existence — the explicit, per-row control surface
for set-op write-through. It replaces the never-built `quereus.update.*` routing-tag dispatch for
set-ops (`union-branch` / `delete_via`, which are declared-but-dead today — no parser syntax, no
consumer; see `set-op-membership-read` § Substrate state).

**This is the first set-op view writability in the engine.** `propagate.ts` rejects
`SetOperation` as `unsupported-set-op` today. This ticket adds a set-op decomposition path. The
membership write is *explicit* (the flag directly names the branch), so it does **not** require the
full predicate-honest branch-consistency dispatch the docs describe (§ Union All / Intersect /
Except) — that static dispatch stays unbuilt; the runtime membership probe is the branch oracle
instead.

## The membership write is a per-row conditional materialization (reused substrate)

For `(A) union exists left as inA, exists right as inB (B)`, a membership write is a **per-row
insert-or-delete branch** keyed on the pre-mutation membership probe (the read half's bit):

| write | row currently in that branch | row currently absent from that branch |
|---|---|---|
| `set inB = true`  | no-op | **insert** into B |
| `set inB = false` | **delete** the matching B row | no-op |

This is the **same per-row conditional substrate** `view-write-optional-member-transitions` builds
(matched/absent partition → matched-op + absent-op) and `outer-join-existence-column` wires for the
join existence write. The set-op version partitions by the **membership probe** (in-branch subset /
not-in-branch subset) instead of by join match. **Reuse, do not fork** the partition-and-branch
core (`view-mutation-node.ts` / `view-mutation.ts`); the `outer-join-existence-column` write ticket
is required to keep its existence-write routing **component-generic** (no hard-coded join side) so
this ticket extends it for the `set-op-branch` `RelationalComponentRef`.

## A branch is itself a view body — route per-branch via recursive `propagate`

The load-bearing structural move: each set-op operand is a relational sub-plan (a base table, a
projection-filter, possibly another view). Inserting/deleting a row **into a branch** is exactly a
recursive view-mutation on that branch's sub-plan. So the set-op write path **recurses through
`propagate` on the branch's relational sub-plan** (the recursive view-update fan-out), reusing the
existing single-source / multi-source spines per branch:

- `set inB = true` ⇒ build an `insert` `MutationRequest` against branch B's sub-plan, with the
  inserted values = the result row's data columns (plus any same-statement `set <dataCol> = …`),
  and recurse `propagate(ctx, <B-as-view-like>, insertReq)`. B's own σ predicate (the `guard`
  carried on the `existence` site) is honored by B's spine.
- `set inB = false` ⇒ build a `delete` against B's sub-plan identifying the matching row, recurse.
- A branch that is a base table bottoms out in a single base op; a branch that is itself a
  `SetOperationNode` recurses again (the **nested** ticket's subtree write — naturally falls out of
  this recursion, but the per-leaf product-coordinate addressing is deferred there).

New module `planner/mutation/set-op.ts` (sibling of `multi-source.ts` / `decomposition.ts`): given
a planned set-op view body + a `MutationRequest`, classify and emit the per-branch `BaseOp[]` via
recursion. `propagate.ts` dispatches to it when the body root is a `SetOperationNode` (replacing the
`unsupported-set-op` rejection for the membership-driven and probe-driven cases; a set-op write that
is neither — see § Unspecified case — still rejects with a clear diagnostic).

## Per-operator write semantics

- **`union` / `union all`** — `inA` / `inB` independent. `set inA = true` ⇒ insert into A;
  `= false` ⇒ delete from A; symmetrically for B. **Both false** ⇒ the row leaves the view entirely
  (deleted from every branch it was in).
- **`except`** (`A except B`) — a visible row is always `inLeft = true, inRight = false`.
  `set inRight = true` ⇒ insert into B (pushing the row **out** of the view — the explicit form of
  the dead `delete_via='right_insert'` tag). `set inLeft = false` ⇒ delete from A (the row leaves
  the view). `set inRight = false` / `set inLeft = true` over a visible row ⇒ no-op (already so).
- **`intersect`** — reads are trivially all-true, so membership columns are **write-useful only**:
  `set inB = false` ⇒ delete from B (dropping the row from the intersect — the explicit form of a
  single-side delete). `set inB = true` over a visible row ⇒ no-op.

## Data-column writes & deletes fan out via the runtime probe

The membership probe doubles as the branch oracle for non-flag writes (the predicate-honest reading,
realized at runtime instead of by static predicate analysis):

- `delete from U where <pred>` ⇒ for each matched row, delete from **every branch it is currently a
  member of** (the probe set). For `except` that is the left only; for `intersect` every branch; for
  `union` whichever branches contain it.
- `update U set <dataCol> = v where <pred>` ⇒ update the row in **every branch it is a member of**
  (keeping the branches aligned). Composes with a same-statement membership write (below).
- This reuses the read-half probe partition; no new branch-consistency analysis is needed.

## Composition & rejection rules

- **Membership write + same-side data write.** `set x = 5, inB = true` over a row absent from B ⇒
  insert B with `x = 5` (the same-statement data assignments flow into the insert-branch
  projection). `set x = 5, inB = false` ⇒ **reject** `conflicting-assignment`-style (cannot both
  delete the branch and write its column).
- **No-default.** A `set inB = true` insert of a B row whose `not null` column has no supplied value
  and no default fails with the existing `no-default` / `null-extended-create-conflict` diagnostic —
  not a silent drop (identical to the transitions / join-existence path).
- **Boolean literal only (v1).** The assigned membership value must be a boolean literal /
  boolean-typed constant; `true` ⇒ insert-branch, `false` ⇒ delete-branch. A non-literal boolean is
  a per-row branch on the *written* value — defer; document if it complicates the partition (matches
  `outer-join-existence-column` § Out of scope).
- **Unspecified case.** A set-op view write that names **no** membership column and is not a plain
  data-column fan-out / delete (e.g. an ambiguous multi-branch insert with no flags) rejects with a
  clear diagnostic. The `strict`-vs-`lenient` *policy* knob for the unspecified case is decided by
  `remove-update-routing-tag-surface`; this ticket implements the **lenient/predicate-honest**
  default (fan out to consistent/member branches) and rejects only genuine ambiguity.

## Insert through the view

`insert into U (id, x, inA, inB) values (9, 'q', true, false)` ⇒ insert into the branches whose flag
is `true` (here A only). `inA = inB = true` ⇒ insert into both. The membership column is itself never
stored — on INSERT it is consumed as a routing directive. A flag column **omitted** from an insert:
default to the predicate-honest reading (insert into every branch whose σ predicate is consistent
with the inserted values) — or, simpler and v1-acceptable, require the flags be supplied for an
ambiguous multi-branch insert and reject otherwise with a clear message. Pick the simpler gate first;
document the choice.

## Static surfaces (`func/builtins/schema.ts`)

- `column_info(view)` — flip the membership column to `is_updatable = 'YES'` (writable through its
  insert/delete effect), `base_table` / `base_column` still `null` (writable through an *effect*,
  not a base mapping).
- `view_info(view)` — a set-op view with membership columns is now `is_insertable_into` /
  `is_updatable` / `is_deletable` = the per-branch reachability the new path supports; keep the
  surface agreeing with the dynamic `propagate()` truth (cross-checked by test).

## Tests (acceptance gate: `test/property.spec.ts` § View Round-Trip Laws → a new set-op family)

Build on the `set-op-membership-read` fixture (`U = (A) union exists left as inA, exists right as inB (B)`):

- **`inB` false→true (insert branch).** `update U set inB = true where id = K` over a row in A only:
  a `B` row appears (data columns from the result row); PutGet shows `inB` now reads `true`. With an
  undefaulted `not null` B column → `no-default` / `null-extended-create-conflict`.
- **`inB` true→false (delete branch).** Over a row in B: the `B` row disappears, `A` untouched; the
  row stays visible if still `inA`. **Both false** (`set inA = false, inB = false`) over a row in
  both ⇒ row leaves the view.
- **`except`.** `Ux = (A) except exists left as inL, exists right as inR (B)`:
  `set inR = true` inserts into B and the row vanishes from `Ux`; `set inL = false` deletes from A.
- **`intersect`.** `Ui = (A) intersect exists left as inL, exists right as inR (B)`:
  `set inR = false` deletes from B and the row drops from `Ui`.
- **Composition.** `set x = 5, inB = true` over a row absent from B inserts B with `x = 5`;
  `set x = 5, inB = false` rejects (contradiction).
- **Data fan-out / delete via probe.** `update U set x = 7 where id = K` updates the row in every
  member branch; `delete from U where id = K` deletes it from every member branch. Verify against a
  row in A only, B only, and both.
- **Insert-through.** `insert into U (id, x, inA, inB) values (…, true, false)` inserts only A;
  `(…, true, true)` inserts both; reads back the membership accordingly.
- **GetPut.** Writing the read-back membership value back is a base no-op (true→true, false→false
  both leave the base diff empty).
- **Key Soundness** unchanged from the read half; negative self-tests stay red on an injected branch
  divergence.

Run `yarn workspace @quereus/quereus test 2>&1 | tee /tmp/t.log; tail -n 80 /tmp/t.log`
and `yarn workspace @quereus/quereus lint`.

## Out of scope (keep deferring)

- **Nested / subtree flags, product coordinate system, multi-target fan-out, `predicate-contradiction`
  rejection** → `set-op-membership-nested`. (The recursion in § A branch is itself a view body
  *enables* a subtree write to bottom out, but the per-leaf product-coordinate **addressing** /
  co-satisfiability checks are the nested ticket.)
- Non-literal boolean membership writes; the `strict` policy knob (→ `remove-update-routing-tag-surface`).
- Flat n-way shorthand, `union all` count-variant, projection-position sugar (backlog).
