# Identity-preserving reshape

Part 4 of the maintained-table chain (prereq: the unified model; independent
of the verbs/differ tickets). Today a source `alter` that shifts a
derivation's output shape makes `refresh` **drop + recreate** the table
(`rebuildBackingTable`) — a new incarnation, fatal to a replicated basis
table's row metadata and the one remaining identity-destroying verb after
the rest of the chain lands. (The differ's body-change path stops using
drop+recreate in `maintained-table-differ-transitions`; this ticket owns
the refresh path, which also serves stale-recovery after source alters.)

## Resolved design

On a refresh whose re-derived shape mismatches the live table, compute a
**column-level delta** old→new and classify:

- **Expressible in place** — any combination of: appended (trailing) new
  columns, dropped columns, renamed columns (the value-preserving
  passthrough trace `renameShiftedBackingColumns` already uses), and
  per-column attribute changes (type / collation / not-null), where the
  **surviving columns' relative order is preserved**. Apply via the
  module's `alterTable` (`addColumn` — backfill values arrive via the
  subsequent data reconcile, so a NULL-default add is fine; `dropColumn`;
  `renameColumn`; `alterColumn`), then run the data refresh (the keyed-diff
  reconcile against the reshaped table). Same incarnation throughout: no
  `table_removed`/`table_added`, the backing-host instance stays owned,
  cached plans invalidate via `table_modified` only.
- **Not expressible** — an *interleaving* column reorder (e.g. a `select *`
  body whose new source column lands mid-output: positional backing↔body
  alignment cannot be preserved by append-only adds), or a **physical-PK
  definition change** (column set, order, direction, or collation of the
  key): **sited error** — "the derivation's output shape changed
  incompatibly with table <name>; alter the table to the new shape and
  re-attach, or drop and recreate" — leaving the table and its rows
  untouched and the derivation stale (recoverable exactly as the message
  says). `alterPrimaryKey` exists on the module surface, but a maintained
  table's PK is its replicated row identity; silently re-keying it is the
  same fatality drop+recreate was. Documented tradeoff: today's silent
  rebuild *convenience* for these cases becomes an actionable error —
  predictability and the frozen-basis invariant win; the remedy is
  explicit and the differ ticket's detach→alter→attach sequence covers the
  declarative route.
- The **explicit-column-list stance is retained**: a sugar-created MV with
  a declared column list whose body output count shifts errors as today
  (the list is a declared interface).
- A module **without** `alterTable` ⇒ the same sited error (parity with the
  rename-propagation posture: never drop a durable module's table to
  reshape it).

The shape-match fast path (data-only reconcile, today's `rebuildBacking`)
is untouched. After this ticket `rebuildBackingTable` (drop+recreate) has
no remaining callers — delete it.

## Edge cases & interactions

- Trailing source-column add reaching the output (`select *` body, column
  appended at source end) ⇒ in-place add + reconcile fills values; assert
  no table_removed/table_added and that a consumer maintained table over
  this one is NOT incarnation-cascaded (it goes stale via table_modified
  and recovers by refresh, as docs describe for source alters).
- Output column dropped (source column dropped; body no longer projects
  it) ⇒ in-place dropColumn + reconcile.
- Type/collation attribute shift (source `alter column set data type` /
  `set collate` flowing through a passthrough projection) ⇒ in-place
  alterColumn; collation change touching a **PK column** is a PK-definition
  change ⇒ error.
- Interleaving `select *` reorder ⇒ sited error; table and rows intact;
  subsequent detach→alter→attach (or drop+recreate) recovers; pin the
  message.
- Refresh-in-transaction parity (the existing store `replaceContents`
  commit-first caveat) still holds on the reshape path.
- Reshape failure midway (e.g. alterColumn MISMATCH on narrowing) ⇒ the
  refresh fails; whatever module ops already applied must leave a
  recoverable state — prefer ordering ops so data-lossless ops (adds,
  renames) precede lossy-capable ones, and document that a failed reshape
  leaves the table stale with a re-runnable refresh.
- Memory and store modules both exercise the in-place path (store's
  alterTable support level determines whether store gets the error posture
  — verify and document which).
- Rename propagation's backing-column rename pass and
  `restoreUnaffectedMaterializedViews`' pure-name-shift carry already do
  in-place work — ensure the new delta classifier is shared with
  `describeBackingShapeMismatch` rather than a second shape-diff
  implementation (stay DRY).
