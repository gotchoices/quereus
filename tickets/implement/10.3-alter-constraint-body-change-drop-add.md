# Constraint body change → DROP + ADD CONSTRAINT in the declarative differ

## Why

`alter-table-drop-rename-constraint` (prereq) adds the `DROP CONSTRAINT` / `RENAME CONSTRAINT`
primitives and wires **name-level** add / drop / rename of named constraints into the declarative
differ. What it cannot yet see is a constraint whose **name is unchanged but whose body changed** —
an edited CHECK expression, a changed FK `ON DELETE/UPDATE` action or referenced table/columns, a
changed UNIQUE column set / partial predicate / `ON CONFLICT`. The catalog projection
(`CatalogTable.namedConstraints`) carries only `{ name, tags }`, so the differ has nothing to
compare bodies against and a body edit silently no-ops at `apply schema`.

Per product direction, a constraint body change is realized as **drop the old constraint + add the
new one** (drop+recreate) — there is no in-place "redefine" primitive, and re-creation is the
honest way to re-validate existing rows against the new rule. This ticket gives the differ the
information to detect the change and reuses the `constraintsToDrop` / `constraintsToAdd` buckets +
DDL emission from the prereq.

## Design

### Carry a comparable body in the catalog

Widen each `CatalogTable.namedConstraints` entry from `{ name, tags? }` to
`{ name, tags?, definition: string }`, where `definition` is a **canonical DDL fragment** for the
constraint (e.g. `unique (email) on conflict abort`, `check (qty > 0)`,
`foreign key (parent_id) references parent (id) on delete cascade`). A canonical string fragment is
the simplest order/format-stable comparison key and reuses the existing stringify machinery, so the
comparison is `actualFragment !== declaredFragment`.

- **Actual side**: factor a `constraintToCanonicalDDL(schemaConstraint, tableSchema)` helper out of
  `ddl-generator.ts` (which already serializes constraints inside `generateTableDDL`) covering
  check / unique / fk. Populate `definition` in `tableSchemaToCatalog` (catalog.ts).
- **Declared side**: render the declared AST `TableConstraint` with `tableConstraintsToString([c])`
  (ast-stringify.ts). Use the **same** normalization both sides (house-style lowercase keywords,
  identifier quoting, as-written column order, conflict + FK actions, partial predicate).

**Exclude tags from `definition`.** A constraint tag change is handled by
`alter-table-tag-mutation`'s `ALTER CONSTRAINT … SET TAGS`; it must **not** masquerade as a body
change and force a needless drop+recreate. The canonical fragment omits the `WITH TAGS (...)`
clause; `tags` stays a separate field compared separately (the tag-diff from ticket #10).

### Detect in `computeTableAlterDiff`

For each declared named constraint that **name-matches** an actual one (i.e. not a create, drop, or
rename target), compare `definition`. On drift, push the constraint to both `constraintsToDrop` (old
name) and `constraintsToAdd` (declared fragment) — the same buckets the prereq emits, so
`generateMigrationDDL` needs no new code, only population. DROP already orders before ADD within the
per-table block (established by the prereq).

Reuse the prereq's **declared-side gathering** fix (table-level **and** column-level named
constraints) so an inline `qty int constraint chk_qty check (qty > 0)` body edit is detected.

### Precedence: name + body both changed

If a constraint is matched by a rename hint (`quereus.id` / `quereus.previous_name`) **and** its
body differs, prefer **drop-old + add-new** (under the declared name/def) and **suppress the
RENAME** for that constraint — a rename-then-alter is two ops where a drop+recreate is one coherent
one, and the new body must be re-validated regardless. Document this precedence; test it.

## Edge cases & interactions

- **Tag-only change must NOT drop+recreate** — the single most important correctness property here.
  `definition` excludes tags; assert that a constraint whose only change is its tags produces a
  `SET TAGS` (ticket #10) and **no** drop/add.
- **UNIQUE body change** (column set / partial predicate / conflict action): drop tears down the
  covering structure (auto-index), add rebuilds it and re-validates existing rows — fails with
  `CONSTRAINT` if current rows violate the new uniqueness. Surface clearly; test the violating case.
- **FK body change** (referenced table/columns or `ON DELETE/UPDATE` action): re-add re-validates
  child rows against parent existence; respects `pragma foreign_keys`.
- **CHECK body change**: re-add re-validates existing rows against the new predicate; fails if any
  violate.
- **Canonicalization fidelity**: the declared and actual fragments must agree on semantically-equal
  forms (e.g. `references parent` vs `references parent (id)` when `id` is the PK; default conflict
  `abort` emitted vs elided). Mismatched canonicalization would cause spurious drop+recreate churn —
  lean on the existing declarative-equivalence round-trip to pin format parity, and add targeted
  "no spurious diff" assertions (declare == actual ⇒ empty diff, idempotent re-apply).
- **Unnamed constraints**: body-change detection keys off names. Unnamed-constraint body changes
  are not individually addressable and are out of scope (note the limitation; they may already
  ride table-level handling or be missed — do not silently claim coverage).
- **Store vs memory parity**: the drop+add executes through the prereq's module paths; run the
  body-change `.sqllogic` under `yarn test:store`.
