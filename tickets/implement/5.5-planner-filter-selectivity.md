## Problem

`FilterNode.estimatedRows` (`packages/quereus/src/planner/nodes/filter.ts:61-68`) hardcodes selectivity `0.5` — every WHERE clause is assumed to keep half its input rows. Meanwhile a real statistics layer already exists and is already wired into the optimizer:

- `StatsProvider.selectivity(table, predicate)` (`packages/quereus/src/planner/stats/index.ts`) with a live `CatalogStatsProvider` implementation (`packages/quereus/src/planner/stats/catalog-stats.ts`) that does histogram-interpolated range selectivity, `1/ndv` equality, null-fraction for `IS NULL`, etc., and falls back to `NaiveStatsProvider` per-node-type heuristics when no catalog stats exist.
- The provider is reachable as `context.stats` on the `OptContext` handed to every optimizer rule (`packages/quereus/src/planner/optimizer.ts:1079` creates it; rules like `rule-materialized-view-rewrite` and `rule-grow-retrieve` already call `context.stats.tableRows(...)`).

The gap: `FilterNode.computePhysical` / `estimatedRows` are **node-level accessors with no `OptContext` parameter**, so a Filter can never consult `context.stats` from inside itself. We must NOT widen `computePhysical` to take a context (that would thread `OptContext` through every node in the tree).

## Chosen design

Route selectivity estimation through a **Physical-pass rule** that already holds `OptContext`, and have it stamp the estimate onto the FilterNode via a new optional field.

### 1. FilterNode gains an optional `selectivity` field

`FilterNode` constructor takes a new optional `readonly selectivity?: number` (0..1). Wire it through:
- `estimatedRows`: use `this.selectivity ?? DEFAULT_FILTER_SELECTIVITY` where `DEFAULT_FILTER_SELECTIVITY = 0.5`. Keep the existing `Math.max(1, floor(rows * sel))` / `0`-on-empty shape. Add a `NOTE:` documenting that `0.5` is the *last-resort* default used only before the physical-pass rule runs or when the source table can't be identified — the real heuristic path is the stats provider (see rule below).
- `computePhysical`: already derives `rows` from `this.estimatedRows`, so it inherits the new selectivity automatically. The covered-unique-key branch that forces `rows = 1` stays as-is and must still win (a provably ≤1-row filter overrides any stats fraction).
- `withChildren` / `withPredicate`: the existing `return this` fast-path (unchanged source AND predicate) already preserves `selectivity`. When a NEW instance is minted, pass `this.selectivity` through ONLY when the predicate is unchanged (`newPredicate === this.predicate`); drop it (undefined → recompute default) when the predicate changes, since the estimate is predicate-specific. Document this in a comment.

### 2. New rule `rule-filter-selectivity` in the Physical pass

Create `packages/quereus/src/planner/rules/predicate/rule-filter-selectivity.ts`. Signature per `RuleFn` (`(node: PlanNode, context: OptContext) => PlanNode | null`). Logic:

```
if node.selectivity !== undefined → return null   // already stamped; idempotent
tableSchema = extractTableSchema(node.source)      // key-utils.ts, walks Filter/Project/Sort/Retrieve/TableReference
if !tableSchema → return null                      // multi-table (join) or non-table source: leave default
sel = context.stats.selectivity(tableSchema, node.predicate)
if sel === undefined → return null                 // provider declined: leave default
return new FilterNode(node.scope, node.source, node.predicate, undefined, clamp01(sel))
```

Reuse the existing `extractTableSchema` helper (`packages/quereus/src/planner/util/key-utils.ts:493`) — do not write a new walker. Clamp `sel` to `[0,1]` defensively.

Register in `RULE_MANIFEST` (`optimizer.ts`) under `pass: PassId.Physical`, `nodeType: PlanNodeType.Filter`, `phase: 'impl'`, `sideEffectMode: 'safe'` (annotation-only: rebuilds the same Filter with an added estimate; source and predicate preserved verbatim). Place it among the Physical-pass entries (near `select-access-path`). The Physical pass is bottom-up and runs AFTER the Structural pass, so by the time this fires, predicate-pushdown / grow-retrieve have already put the Filter in its final position over its final source — `extractTableSchema` sees the settled tree.

### Convergence note

The pass engine does not re-offer a rule its own output (see `pass.ts` convergence contract — documented by the prereq ticket `planner-optimizer-cleanups`). This rule fires at most once per Filter: it produces a new node whose id inherits the applied-rule set, so `hasRuleBeenApplied` suppresses a second fire; and the `selectivity !== undefined` guard makes it decline on any already-stamped node anyway. No self-loop needed.

## Expected behavior / tests

- `select * from t where indexed_col = 5` over a table whose catalog statistics carry a histogram/ndv for `indexed_col` → the FilterNode's `physical.estimatedRows` reflects `1/ndv` (or histogram-derived), NOT `floor(rows * 0.5)`.
- Same query over a stats-less table → falls back gracefully to `NaiveStatsProvider` heuristics via the same `context.stats.selectivity` call (e.g. equality ≈ 0.1), and if the provider returns `undefined`, the node keeps the documented `0.5` default. Never a crash, never a silent wrong number.
- Existing golden plans: memory-vtab tables generally have no catalog stats, so `selectivity` flows through `NaiveStatsProvider` — this WILL change some `estimatedRows` values in EXPLAIN/plan output away from the old flat-0.5. **Expect golden-plan / EXPLAIN snapshot churn** and update the affected fixtures (they live under `packages/quereus/test/plan/` and `test/optimizer/`). Verify each changed number is a defensible stats-derived estimate before blessing.
- Add a focused unit/spec test that seeds a `TableSchema.statistics` with a known ndv (mirror `test/optimizer/statistics.spec.ts` seeding), plans a single-equality filter, and asserts the FilterNode's physical `estimatedRows` matches `rowCount / ndv` (floored, min 1) rather than `rowCount / 2`.
- Add a test asserting the covered-unique-key path still forces `estimatedRows === 1` even when a stats fraction would give more (the `rows = 1` branch must win).

## Edge cases & interactions

- **Filter over a join / multi-table source:** `extractTableSchema` returns `undefined` (relations.length ≠ 1) → rule declines → default 0.5 retained. Correct for now; join/multi-column selectivity is out of scope (parked in backlog `feat-conjunction-and-join-selectivity`).
- **Conjunctive predicates (`a = 1 and b = 2`):** the current `CatalogStatsProvider` does not decompose an `AND` — `extractColumnFromPredicate` finds no single column child, so it falls back to `NaiveStatsProvider`, which returns `0.1` for any `BinaryOp` (including the AND root). This is a coarse estimate, not a regression vs. the old flat 0.5, but it IS crude. Do NOT try to fix conjunction decomposition here — it is parked in backlog `feat-conjunction-and-join-selectivity`. Add a `NOTE:` in the rule pointing at that limitation so a future reader isn't surprised.
- **`selectivity` = 0 (provider says nothing matches):** `estimatedRows` must not go negative or NaN; `Math.max(1, floor(rows*0))` yields 1 today — decide whether 0-selectivity should yield 0 rows or the min-1 floor. Keep min-1 for consistency with the empty-source path unless a test says otherwise; document the choice.
- **Covered-unique-key override:** when equality conjuncts cover a unique key, `computePhysical` already sets `rows = 1`. The stamped `selectivity` must not override that — verify ordering in `computePhysical` (the `rows = 1` assignment happens after the initial `rows` from `estimatedRows`, so it already wins; add a test to lock it).
- **`withChildren` after stamping:** if a later pass rewrites the Filter's source via `withChildren` with a *different* source but same predicate, per the design the new instance keeps `selectivity` (predicate unchanged). That estimate is now slightly stale (computed against the old source's table). Physical is the last rule-bearing pass over Filters, so in practice nothing re-sources a stamped Filter — record this as a tripwire `NOTE:` at the `withChildren` site rather than a ticket.
- **`estimatedRows` read before the Physical pass** (e.g. during Structural cost comparisons): `selectivity` is still undefined then, so the default 0.5 is used — same as today. No behavior change pre-physical.
- **`sideEffectMode: 'safe'` justification:** the rule reads stats and rebuilds the identical Filter (same scope, source, predicate, same output attribute ids) with only an added estimate — no side-effect reordering. Confirm the side-effect audit test accepts `safe`.
