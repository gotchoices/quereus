# Core callers resolve collations against the database

## Problem

Two remaining subsystems inside `packages/quereus` pass collation *names* to the
three-argument `compareSqlValues(a, b, collationName)`, which resolves them against
the **process-global** registry holding only `BINARY`, `NOCASE`, and `RTRIM`. A
collation registered with `db.registerCollation(...)` is invisible there.

### Materialized-view maintenance

`database-materialized-views.ts` and `-apply.ts` compare rows and keys under the
source column's declared collation when deciding whether a watched column changed
(~538), whether a `UNIQUE` conflict is a self-conflict (~1026, ~1037), whether a
residual-plan row matches a seek key (~402, ~644), and whether two backing primary
keys are equal (~630). Under a custom collation all of these silently degrade to
byte comparison, so a maintenance apply can miss a row it should have updated or
delete the wrong one.

### Planner contradiction checker

`checkSatisfiability` (`sat-checker.ts` ~85) decides whether a set of predicate
conjuncts is provably unsatisfiable, so the optimizer can replace a `Filter` with
an empty relation. It takes an optional `getCollation(col) => name` and compares
literals under that name (~97). If it resolves a custom collation to `BINARY`, it
can conclude `'unsat'` for a predicate that *is* satisfiable under the real
collation — the optimizer then deletes rows from the result. This is the most
severe instance of the bug in this family: silent wrong results, not just wrong
order.

## Design

`3.2-collation-resolver-seam` provides `db.getCollationResolver(): CollationResolver`
(throws on an unregistered name) and `builtinCollationResolver(name)` (returns
`undefined` for anything outside the three built-ins).

### Materialized views

Add `getCollationResolver(): CollationResolver` to the
`MaterializedViewManagerContext` interface (`database-materialized-views-plans.ts`
~31) — it is already the narrow database-capability surface these modules see.
`Database` supplies it.

The free functions in `-apply.ts` (`residualRowMatchesKey`, `backingPkEqual`,
`residualRowMatchesBasePrefix`) take a `plan` object whose descriptors carry
collation *names* (`d.collation`). Do **not** change the plan shape. Instead:

- Where a plan is *built* (in `-plans.ts` / `-apply.ts` plan construction, which has
  `ctx`), resolve each descriptor's collation name once and store the resolved
  function alongside the name on the descriptor (`collationFn`). The name stays for
  logging and equality of plans; the function is the thing the row loop uses.
- The three free functions then use `d.collationFn` with `compareSqlValuesFast`.

If a plan descriptor turns out to be constructed somewhere without `ctx` reach,
fall back to passing the resolver as an explicit parameter to the three functions —
but prefer the resolve-at-plan-build approach, because these functions run per row.

### Satisfiability checker

Change the optional fifth parameter from a name-getter to a pair, keeping the
name-getter and adding the resolver:

```ts
export function checkSatisfiability(
	conjuncts: ReadonlyArray<ScalarPlanNode>,
	domains: ReadonlyArray<DomainConstraint>,
	bindings: ReadonlyArray<ConstantBinding>,
	attrIndex: (attrId: number) => number | undefined,
	getCollation?: (col: number) => string | undefined,
	collationResolver?: CollationResolver,
): SatResult
```

Resolution rule inside `checkSatisfiability`:

1. `BINARY` → `BINARY_COLLATION`, no lookup.
2. `collationResolver` supplied → use it (throws on an unregistered name, which
   would be an internal bug at this point since the name came from a column type).
3. No resolver → `builtinCollationResolver(name)`; on `undefined`, **return
   `'unknown'`** for the whole check.

Step 3 is the key soundness fix: `'unknown'` is an already-supported return value
meaning "cannot prove unsatisfiable", and it makes the optimizer keep the filter.
Assuming `BINARY` — the current behavior — is unsound. Resolve the collations once,
before the conjunct loop, and bail early if any is unresolvable.

Both call sites should pass a real resolver where one is reachable:

- `rules/predicate/rule-filter-contradiction.ts` ~64 — the optimizer rule context
  reaches the database; pass `db.getCollationResolver()`.
- `planner/mutation/set-op.ts` ~1855 — the `leg` object is built at ~1815 from
  `legAttrs`; add `collationResolver` to that object alongside the existing
  `getCollation` field, sourced from the same context that supplies `legAttrs`.

If a call site genuinely cannot reach a `Database`, leaving the resolver
`undefined` is now *safe* (degrades to `'unknown'`) rather than *wrong* — document
which site, if any, ends up in that state.

## Edge cases & interactions

- **`'unknown'` is not `'unsat'`.** Confirm `rule-filter-contradiction.ts` treats
  `'unknown'` as "do not rewrite". Read the rule before changing the checker.
- **Numeric comparisons are collation-independent.** An unresolvable collation on a
  column whose accumulated domain is purely numeric would still bail to `'unknown'`
  under the rule above. That is conservative and acceptable; do not add a
  storage-class carve-out.
- **Self-conflict detection** (`database-materialized-views.ts` ~1026/~1037): the
  comparison must use the *source* column's collation (`w.sourceCollation`,
  `pkDef[i]?.collation`), which may differ from the backing column's. Preserve the
  existing name selection; only the name→function step moves.
- **Missing collation on a descriptor** (`d.collation` is `undefined`) means
  BINARY. Keep that; do not call the resolver with `undefined`.
- **Plan reuse across a `registerCollation` call.** Resolving at plan-build time
  freezes the function into the plan. A maintenance plan built before a collation is
  re-registered keeps the old comparator. Materialized-view plans are already
  invalidated on schema change; a bare `registerCollation` does not invalidate them.
  This is the same pre-existing exposure as index comparators — leave a `NOTE:`
  comment at the plan-build resolve site, do not file a ticket.
- **Covering-materialized-view collision telemetry** (`mv-collation-collision-telemetry`,
  already complete) fires when a coarser collation admits a collision. Make sure a
  custom collation now genuinely resolving does not double-report; run that suite.
- **Cross-check with `uniqueEnforcementCollations`.** The materialized-view row-time
  eligibility gate and the store/isolation re-validators must agree on the per-column
  enforcement collation (see the completed `unify-unique-enforcement-collation-resolver`
  ticket). All three now resolve through the same database resolver — assert that in
  a test rather than assuming it.

## Tests

- Register a custom collation; create a materialized view over a source column
  declaring it; update the source in a way that is a no-op *under the collation* but
  a byte change, and vice-versa; assert the maintenance apply does the right thing
  in both directions.
- `sat-checker`: a predicate like `x = 'a' and x = 'A'` over a column collated
  `NOCASE` is **satisfiable** — `checkSatisfiability` must not return `'unsat'`.
  Same predicate under `BINARY` is `'unsat'`. Add both.
- `sat-checker` with no resolver and a custom collation name returns `'unknown'`
  (unit test, direct call).
- End-to-end: `select * from t where x = 'a' and x = 'A'` on a `REVERSE`-collated
  column returns the rows it should, i.e. the optimizer does not eliminate the
  filter. Today, verify whether this already returns wrong rows — if so, note it as
  the regression this ticket fixes.
- Existing suites: the materialized-view maintenance suites and
  `test/optimizer/` filter-contradiction tests stay green.
