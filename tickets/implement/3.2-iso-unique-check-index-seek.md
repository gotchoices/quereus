## Problem

`IsolatedTable.findMergedUniqueConflict` (`isolated-table.ts:1317`) checks a non-primary-key
UNIQUE constraint by **scanning the whole underlying table**:

```ts
for await (const underlyingRow of this.underlyingTable.query(this.createFullScanFilterInfo())) {
	const pk = pkIndices.map(i => underlyingRow[i]);
	if (selfPks.some(self => this.keysEqual(pk, self))) continue;
	const overlayRow = await this.getOverlayRow(overlay, pk);   // one point lookup PER ROW
	...
}
```

Every constrained insert or update pays O(rows) underlying reads *plus* O(rows) overlay point
lookups. `checkMergedUniqueConstraints` calls it once per UNIQUE constraint on the table.

The scan exists for a real reason, and any fix must preserve it. The check is against the
**merged view** — the underlying rows, with this connection's uncommitted overlay rows
superimposed. Consider a table with `unique(email)` where the committed row `#7` has
`email = 'a@x'`, and earlier in this transaction the connection updated `#7` to
`email = 'b@x'` (staged in the overlay, not yet flushed). Now the connection inserts a new
row with `email = 'b@x'`.

- The correct answer is **conflict**: in the merged view, row `#7` already holds `'b@x'`.
- A naive equality seek of the underlying on `email = 'b@x'` finds **nothing** — the
  underlying still holds `'a@x'` for row `#7`.
- Today's full scan finds it: it visits row `#7`, sees a live overlay entry for its primary
  key, evaluates the constraint against the *overlay* row (`mergedRow`, line 1347), and
  reports the conflict.

So an index seek on the underlying is only half the merged view. It must be paired with a scan
of the other half.

## Design — two-phase merged check

The overlay holds only this connection's uncommitted rows: it is small, bounded by the
transaction's write set, and already in memory. The underlying is large. Split the merged view
along exactly that line, so each half is searched the way it is cheap to search.

```
merged view  =  (overlay rows)  ∪  (underlying rows with no overlay entry)
```

Both halves are searched for a row that (a) is not one of `selfPks`, (b) matches `newRow` on
the constraint's columns under the enforcement collations, and (c) satisfies the partial
predicate if there is one.

**Phase 1 — scan the overlay.** Iterate the overlay's rows (a full scan of the overlay, which
is what `mergedSecondaryIndexQuery` already does for its modified-PK set). Skip tombstones and
`selfPks`. A live overlay row that matches is a conflict; its merged row *is* the overlay row,
stripped of the tombstone column.

**Phase 2 — seek the underlying.** Look up underlying rows whose constrained columns equal
`newRow`'s values. Skip any candidate whose primary key already has an overlay entry — Phase 1
owns those rows, whatever the underlying still says about them. A surviving candidate is a
conflict; its merged row is the underlying row.

Phase 2's lookup is an index seek when it safely can be, and the existing full scan otherwise.
Phase 1 is unchanged in cost. Together they cover the merged view exactly once, with no row
visited twice, and the per-row `getOverlayRow` call disappears from the underlying scan
(Phase 2 needs it only for candidates the seek actually returned).

### When Phase 2 may seek

`UniqueConstraintSchema.derivedFromIndex` (`packages/quereus/src/schema/table.ts:689`) names
the index backing the constraint, when the constraint was synthesized from a
`create unique index`. Seek only when **all** of these hold:

- `uc.derivedFromIndex` is set and resolves to an entry in `this.tableSchema.indexes`;
- every one of that index's key columns has an **effective enforcement collation of BINARY**
  (per `resolveUniqueEnforcementCollations`, already computed in this function).

Otherwise fall back to the current full scan of the underlying.

The collation gate is not optional. `resolveUniqueEnforcementCollations` compares under the
column's declared `COLLATE`, but the store's physical index key bytes come from a separate
encoder registry that does not consult the database's collation registry — the defect tracked
by `backlog/debt-store-index-keys-use-column-collation` and referenced in the comment at
`isolated-table.ts:734`. Seeking a `NOCASE` index for `'B@X'` would miss the committed
`'b@x'` that the current full scan catches, turning a performance fix into a lost UNIQUE
violation. Gate on BINARY now; widen the gate when the encoder defect is fixed, and leave a
`NOTE:` at the gate saying so.

### Building the seek

With `iso-index-descriptor-isolation-consumer` landed, the isolation layer can construct a
valid typed `FilterInfo` rather than a bare string:

```ts
const index = this.tableSchema!.indexes!.find(i => i.name === uc.derivedFromIndex)!;
const descriptor: IndexDescriptor = {
	name: index.name,
	role: 'secondary',
	keyColumns: index.columns.map(c => ({ columnIndex: c.index, desc: !!c.desc, collation: c.collation })),
	unique: true,
};
// Values in INDEX key order, which is not necessarily `uc.columns` order.
const values = index.columns.map(c => newRow[c.index]);
const fi = makeIndexEqSeekFilterInfo(descriptor, index.columns.map(c => c.index), values);
for await (const candidate of this.underlyingTable.query!(fi)) { ... }
```

`makeIndexEqSeekFilterInfo` populates `constraints` with `EQ` entries as well as `idxStr` /
`accessPath`, so a module that ignores the index hint still applies the equalities as a
residual filter rather than returning the whole table.

Note the index key order versus `uc.columns` order: `uc.columns` mirrors the index's columns
today, but read the values off `index.columns[i].index` rather than assuming the two arrays
agree.

### Conflict precedence

Today the first conflicting row in *underlying scan order* wins and is reported in the
`UNIQUE constraint failed` message / `existingRow` / REPLACE eviction. Phase 1 now runs first,
so an overlay conflict is reported in preference to an underlying one. Under a satisfied
UNIQUE constraint at most one conflicting row can exist in the merged view, so this only
changes which row is named when the constraint was *already* violated before the statement.
Assert the single-conflict case in tests and note the tie-break change.

## Edge cases & interactions

- **Overlay-side conflict (the case a naive seek loses).** Committed `#7 email='a@x'`;
  transaction updates `#7` to `'b@x'`; then inserts `email='b@x'`. Must conflict. This is the
  spec that fails if Phase 1 is dropped.
- **Overlay-side *resolution*.** Committed `#7 email='a@x'`; transaction updates `#7` to
  `'z@x'`; then inserts `email='a@x'`. Must **not** conflict — the underlying still holds
  `'a@x'` at `#7`, but `#7` has an overlay entry, so Phase 2 skips it. This is the existing
  regression `isolation-merged-unique-stale-underlying-false-positive`; keep its spec green.
- **Tombstoned conflict.** Committed `#7 email='a@x'`; transaction deletes `#7`; then inserts
  `email='a@x'`. Must not conflict — Phase 1 skips tombstones, Phase 2 skips `#7` because it
  has an overlay entry.
- **Tombstone revival.** Delete `#7`, then re-insert `#7` with a colliding non-PK unique value;
  the revival path (`isolated-table.ts:897`) calls `checkMergedUniqueConstraints` with
  `selfPks = [pk]`. Both phases must honour `selfPks`.
- **PK-changing update.** `selfPks = [targetPK, newPK]`; both must be excluded in both phases.
- **`OR REPLACE` eviction.** `checkMergedUniqueConstraints` tombstones `conflict.pk` and pushes
  `conflict.row` onto `evicted`. When the conflict came from Phase 1, `conflict.row` is an
  overlay row (already stripped) — `insertTombstoneForPK` on a PK that already has a live
  overlay row takes its `update` branch. When it came from Phase 2 it is an underlying row.
  Both must produce the same `evictedRows` shape.
- **Multiple UNIQUE constraints, one statement.** `checkMergedUniqueConstraints` loops; a
  REPLACE eviction in constraint 1 writes a tombstone into the overlay that constraint 2's
  Phase 1 scan then sees. Confirm the eviction cannot be re-reported as a conflict for the
  next constraint (it is a tombstone, so Phase 1 skips it — assert it).
- **Partial UNIQUE (`create unique index … where …`).** The predicate is evaluated against the
  merged row in both phases. If the store's index physically excludes out-of-scope rows, the
  seek returns fewer candidates — still correct, since those rows are skipped anyway. Do not
  rely on either behaviour; run the predicate in both phases.
- **NULL in a constrained column.** `checkMergedUniqueConstraints` already skips a `newRow`
  with any NULL in `uc.columns`, and `findMergedUniqueConflict` skips candidates with a NULL.
  The seek must not be built for a NULL key (unreachable given the outer guard — assert it).
- **Non-BINARY collation ⇒ no seek.** `unique(email collate nocase)` must keep full-scanning
  and keep detecting `'B@X'` vs committed `'b@x'`. This spec is the guard on the collation gate.
- **Table-level `unique(a, b)` with no backing index** (`derivedFromIndex` unset) ⇒ full scan,
  unchanged behaviour.
- **`drop index` mid-transaction.** Retires the `UniqueConstraintSchema`; a fresh constraint
  object arrives with a new schema. `derivedFromIndex` may name an index no longer in
  `tableSchema.indexes` — the resolve-or-fall-back keeps that safe, but assert it rather than
  `!`-asserting the lookup.
- **Bigint and composite keys.** A composite unique index over `(bigint, text)`; the seek's
  `values` must be read in index-key order.
- **Underlying without `query`.** `findMergedUniqueConflict` currently returns `null` early.
  Preserve — but note Phase 1 can still find an overlay conflict, so the early return should
  now skip only Phase 2.

## Expected tests

Extend `packages/quereus-isolation/test/isolation-layer.spec.ts`:

- the four merged-view scenarios above (overlay conflict, overlay resolution, tombstoned
  conflict, tombstone revival) — each must pass both with and without a backing index, so
  parameterise over `unique(email)` (table-level, no index) and
  `create unique index ux on t(email)`;
- `unique(email collate nocase)` detects a case-differing committed conflict (proves the
  collation gate declines the seek);
- `OR REPLACE` eviction reports the same `evictedRows` whether the conflict was overlay-side
  or underlying-side;
- two UNIQUE constraints where the first REPLACE-evicts a row the second would otherwise see;
- a counting assertion that the seek path actually seeks: wrap the underlying module and count
  rows yielded from `query()` during the constrained insert, asserting it is bounded (e.g. ≤ 2)
  on a table of 100 rows, and unbounded (100) on the `collate nocase` variant. This is the only
  test that proves the optimisation happened rather than merely stayed correct.
