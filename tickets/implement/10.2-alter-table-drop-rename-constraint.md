# ALTER TABLE … DROP CONSTRAINT / RENAME CONSTRAINT + declarative wiring

## Why

Named table-level constraints (CHECK / UNIQUE / FOREIGN KEY) can be **added** after creation —
`ALTER TABLE … ADD CONSTRAINT` is parsed, planned (`AddConstraintNode`), and emitted — but there is
no way to **drop** or **rename** one. The schema catalog surfaces named constraints
(`CatalogTable.namedConstraints`), and the declarative differ even **detects** constraint renames
(`computeTableAlterDiff` populates `TableAlterDiff.constraintsToRename`) — but:

1. There is no `ALTER TABLE … DROP CONSTRAINT <name>` primitive (parser stops at `DROP COLUMN`).
2. There is no `ALTER TABLE … RENAME CONSTRAINT <old> TO <new>` primitive.
3. **`generateMigrationDDL` never reads `constraintsToRename`** — the detection is dead code, so a
   declared constraint rename is silently dropped on the floor (no DDL, no error). The differ's own
   comment admits "the engine has no ALTER for those today."

This ticket adds the two missing primitives and finishes wiring the constraint lifecycle
(add / drop / rename **by name**) through the declarative diff engine, so a `declare schema` that
adds, drops, or renames a named constraint actually converges at `apply schema`.

Constraint **body** change (edited CHECK expr / changed FK action / changed UNIQUE column set) is
handled by the follow-on ticket `alter-constraint-body-change-drop-add` (prereq: this), which
widens the catalog to carry a comparable body and **reuses the DROP+ADD buckets established here**.
v1 in *this* ticket is name-level lifecycle (add / drop / rename **by name**).

## Design

### Primitives

```sql
alter table t drop constraint uq_email;
alter table t rename constraint uq_email to uq_user_email;
```

**AST** (`AlterTableAction`): add
`{ type: 'dropConstraint'; name: string }` and
`{ type: 'renameConstraint'; oldName: string; newName: string }`.

**Parser** (`alterTableStatement`):
- DROP branch currently does `DROP [COLUMN] <name>` only. Branch on `CONSTRAINT`:
  `DROP CONSTRAINT <name>` → `dropConstraint`; else the existing `DROP [COLUMN]` path.
- RENAME branch currently does `RENAME COLUMN` / `RENAME TO`. Add `RENAME CONSTRAINT <old> TO
  <new>` → `renameConstraint`.

**Plan + emit**: handle both in `AlterTableNode` (mirror the lightweight `dropColumn` /
`renameColumn` arms — these are schema-catalog operations, not the separate-node treatment ADD got).
Add `runDropConstraint` / `runRenameConstraint` to `runtime/emit/alter-table.ts`.

### Constraint resolution + catalog mutation

A name may live in any of `checkConstraints` / `uniqueConstraints` / `foreignKeys`. Resolve in a
fixed order (check → unique → fk, first match); reject if the same name exists in two arrays
(ambiguous) and `NOTFOUND` if in none. Drop/rename rebuilds the matched array entry and re-registers
the `TableSchema`, firing `table_modified`.

**DROP CONSTRAINT teardown by class:**
- **CHECK** — drop the `RowConstraintSchema`. If the CHECK was subquery-deferred / assertion-style,
  ensure no commit-time enforcement lingers (it is derived from the table's constraint list, so
  removing it suffices — verify).
- **UNIQUE** — drop the `UniqueConstraintSchema`. A UNIQUE constraint may own a **covering
  structure** (auto-built secondary index, `coveringStructureName`); tearing the constraint down
  must also tear down / orphan that backing index appropriately (mirror how `dropTable` / the
  memory manager handles `ensureUniqueConstraintIndexes`). Do **not** drop a user's explicit
  `CREATE UNIQUE INDEX`-derived constraint silently in a way that leaves the index dangling — define
  and test the interaction (`derivedFromIndex`).
- **FOREIGN KEY** — drop the `ForeignKeyConstraintSchema`; FK enforcement reads the table's
  `foreignKeys`, so removal disables it. No child-row scan needed to drop the *constraint* (unlike
  dropping a parent table).

### Module round-trip

Constraint metadata is schema state that persistent modules serialize, so this **does** go through
`module.alterTable` (unlike tags). Extend `SchemaChangeInfo` (`vtab/module.ts`) with:

```ts
| { type: 'dropConstraint'; constraintName: string }
| { type: 'renameConstraint'; oldName: string; newName: string }
```

Implement both in the **memory** module's `alterTable` (schema-catalog rewrite; no row migration
needed — constraints don't change row shape, though dropping a UNIQUE may drop its backing index)
and in the **store** module (`quereus-store/src/common/store-module.ts`) so DDL is re-persisted.
A module that doesn't implement `alterTable` already rejects data-affecting alters; keep that path.

### Declarative differ — finish the lifecycle

`computeTableAlterDiff` already builds `constraintsToRename` (via `resolveRenames` over
`namedConstraints`). Now:

- **Emit renames**: `generateMigrationDDL` must read `alter.constraintsToRename` and emit
  `ALTER TABLE <t> RENAME CONSTRAINT <old> TO <new>` (this fixes the existing silent-drop bug).
- **Detect + emit drops**: a named constraint present in actual `namedConstraints` but absent from
  declared (and not consumed by a rename) → `DROP CONSTRAINT`. Add a `constraintsToDrop: string[]`
  field to `TableAlterDiff`.
- **Detect + emit adds**: a declared named constraint absent from actual (and not a rename target)
  → `ADD CONSTRAINT` using the declared `TableConstraint` AST (the `ADD CONSTRAINT` primitive
  already exists). Add `constraintsToAdd` (full DDL fragment via the existing constraint stringifier).
- **Declared-side gathering**: the existing rename code reads named constraints from table-level
  `declaredTable.tableStmt.constraints` only. Add/drop (and rename) detection must **also** gather
  **column-level named constraints** (`columns[].constraints` carrying a `name`, e.g.
  `qty int constraint chk_qty check (qty > 0)`), because actual `namedConstraints` already includes
  both. Normalize both into one declared map keyed by lowercased name.

Ordering in `generateMigrationDDL`'s per-table block: RENAME CONSTRAINT and DROP CONSTRAINT before
ADD CONSTRAINT (free a name / remove a conflicting constraint before re-adding), and DROP/ADD
CONSTRAINT after column adds but with PK changes considered (a UNIQUE drop must not strand a PK
dependency — sequence relative to `alterPrimaryKey`). Update the migration-order doc.

`require-hint` policy: a constraint that appears as both an add and a drop with no `quereus.id` /
`quereus.previous_name` hint should follow the same `enforceRequireHint` treatment already applied
to tables/columns (extend the per-table guard).

### Stringify

Add `dropConstraint` / `renameConstraint` arms to `alterTableToString` for emit-roundtrip.

## Edge cases & interactions

- **Dead `constraintsToRename` regression**: add a declarative test that a renamed named constraint
  actually emits `RENAME CONSTRAINT` and converges — this is a real latent bug, not just a missing
  feature.
- **UNIQUE drop ↔ covering structure**: dropping a UNIQUE constraint must release its auto-index;
  dropping a UNIQUE that was *derived from* an explicit `CREATE UNIQUE INDEX` must define behavior
  (does the index survive? — likely the index is the user's and the constraint is the synthesized
  side; document and test).
- **FK drop ↔ enforcement**: after `DROP CONSTRAINT` on an FK, previously-rejected child writes
  succeed; confirm `pragma foreign_keys` interplay.
- **CHECK drop ↔ deferred/assertion-hoisted premises**: a dropped CHECK must no longer contribute
  FDs / domain constraints to the optimizer at the table reference (it's recomputed from the
  constraint list, so removal flows through — verify a query that folded to `EmptyRelation` on the
  CHECK no longer does).
- **Rename collision**: `RENAME CONSTRAINT old TO new` fails if `new` already names a constraint, or
  `old` doesn't exist.
- **Cross-class name ambiguity**: same name in two of check/unique/fk arrays → reject with a clear
  error rather than dropping/renaming an arbitrary one.
- **Rename propagation**: unlike column rename, a constraint name isn't referenced elsewhere in
  SQL — no AST-rewrite propagation needed (confirm; constraint names are local to the table).
- **Declarative add of an FK to a not-yet-created table**: order-independence — declared schemas may
  reference later-declared tables. Confirm `ADD CONSTRAINT` FK emission lands after both tables
  exist (creates precede the alter block already; verify FK-to-new-table works in one apply).
- **Store vs memory parity**: run the constraint-lifecycle .sqllogic under `yarn test:store` too —
  the store module's `alterTable` handler is a separate implementation and is the usual source of
  catalog-persistence drift.
