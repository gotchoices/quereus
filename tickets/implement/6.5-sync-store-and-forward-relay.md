# Store-and-forward, part 2: outbound relay

Sibling of `sync-store-and-forward-hold` (prereq). The hold ticket made
`store-and-forward` durably hold a straggler's out-of-basis changes **marked
forwardable** and telemeter them. This ticket makes those forwardable changes
actually reach peers that still hold the table, by folding them into this peer's
outbound delta sync — **with no new transport surface**: forwardable changes ride
the existing `getChangesSince` → `ChangeSet[]` return that the client and
coordinator already serialize, send, and watermark.

## The load-bearing design decision (resolved — do not re-open)

A forwarded change keeps its **original `hlc` + `siteId`** (it is the straggler's
fact, not a new local one). The consumer of `getChangesSince` advances its
per-peer `lastSyncHLC` to `max(returned ChangeSet.hlc)` (see
`sync-client.ts` `pushLocalChanges`/`handleRequestChanges` and
`coordinator-service.ts`). Therefore forwardable changes **must be filtered
`HLC > sinceHLC`** before being returned, exactly like change-log changes:

- **Why the filter is required.** If a forwarded change with an HLC ≤ the peer's
  watermark were returned, a round that returned *only* forwarded changes would
  set `max(ChangeSet.hlc)` to an old HLC and **regress** the peer's watermark,
  causing a re-scan/re-deliver flood. Filtering `HLC > sinceHLC` makes the merged
  response a safe contiguous prefix (everything ≤ the batch's max HLC is present),
  so advancing the watermark to the max loses nothing — identical to the change-log
  contract.
- **The accepted limitation.** A straggler whose changes predate the holder's
  current sync recency (`HLC ≤ lastSyncHLC`) is **not** relayed via this delta
  path. This is **not a new defect** — it is the same scalar-watermark limitation
  the base delta-sync layer already has (a causally-old change learned late from a
  peer is not re-propagated via delta; it is delivered by direct sync / snapshot
  when `canDeltaSync` is false). `store-and-forward` targets the **transitional
  uneven-retirement window**, where the straggler's writes are recent enough to
  exceed holder watermarks; that is the case it serves. Document this explicitly;
  do **not** build a separate forward channel or per-peer forward cursor (a
  rejected heavier alternative — store-and-forward is an optional optimization, not
  a correctness requirement; quarantine already prevents write loss).

## What this ticket does

### Merge forwardable changes into outbound extraction
In `getChangesSince` (`sync-manager-impl.ts`), collect forwardable changes
alongside the existing `collectChangesSince`/`collectAllChanges` + schema
migrations, then pass the **union** to `buildTransactionChangeSets`:

- New private `collectForwardableChanges(peerSiteId, sinceHLC?): Promise<Change[]>`:
  - Source = `quarantine.listForwardable()` (from the prereq) → map to the held
    `Change` (`entry.change`).
  - **Echo exclusion**: drop entries whose `change.hlc.siteId === peerSiteId`
    (same origin-site echo rule `collectChangesSince` uses — the change's HLC
    siteId is the straggler origin).
  - **Watermark filter**: when `sinceHLC` is defined, keep only
    `compareHLC(change.hlc, sinceHLC) > 0`. When `sinceHLC` is undefined
    (from-zero / `collectAllChanges` path), keep all (no lower bound).
  - This is a **full** horizon-bounded scan (no early-exit), paralleling
    `collectAllChanges`. Document the footprint (bounded by the forwardable set,
    which is bounded by the retention horizon; zero in the no-straggler case).
- Merge: `const changes = [...dataChanges, ...forwardable];` before
  `buildTransactionChangeSets`. The grouper sorts by HLC and bounds at
  `batchSize` over the union, so forwardable changes are grouped into ChangeSets
  by their **original transaction id** (`deterministicTxnId` over the straggler's
  base HLC) — a forwarded change re-forms the straggler's transaction, and if the
  straggler wrote both a live and a (now-retired-here) table in one txn, the
  applied part (from the change log) and the forwarded part rejoin into one
  ChangeSet. Correct by construction.

### Bound interaction (verify, don't re-architect)
`collectChangesSince` keeps its scan-time early-exit. The contiguous-prefix
safety still holds: change-log entries are scanned in HLC order up to its cut C;
forwardable entries are fully scanned (all `> sinceHLC`); `buildTransactionChangeSets`
bounds the union at `batchSize` at a transaction boundary M ≤ C. Everything ≤ M
from **both** sources is present (change-log up to C ⊇ ≤ M; forwardable fully
scanned ⊇ ≤ M), so the returned prefix is contiguous and the watermark advance to
M is safe; forwardable entries with HLC in (M, …] are deferred and re-collected
next round (still `> sinceHLC`). Add a comment to this effect; no code beyond the
merge is needed for the bound.

### Snapshot / bootstrap carve-out (resolved)
Forwardable entries are **delta-only**; they are **not** included in snapshot
paths (`getSnapshot` / `streamSnapshot` / `applySnapshot*`). Rationale: a snapshot
transfers *this peer's own basis*, and a forwarded change is for a table this peer
does **not** have — it has no place in this peer's basis snapshot. A holder peer
receives the straggler's changes via the delta relay above, not via our snapshot.
This is consistent with the existing carve-out that snapshot paths are out of
scope for unknown-table handling (`docs/sync.md`). Confirm the snapshot collectors
do not touch `qt:`/forwardable entries (they scan `cv:`/`tb:`/`sm:` only) and add
a one-line doc note.

### Optional: per-relay telemetry
Optionally add a `relayed` cumulative counter bumped in `collectForwardableChanges`
by the number of forwardable changes actually emitted (distinct from the hold
ticket's apply-time `forwarded` count, since one held entry relays repeatedly until
GC). If added, surface it in `getUnknownTableStats()` and document. Skip if it
complicates the diff disproportionately — note the decision either way.

## Edge cases & interactions to cover (tests)

- **3-peer convergence (the headline test): straggler → relay → holder.**
  - Peer R retires table T (drop_table on R, or R's basis oracle returns nothing
    for T) with `unknownTableDisposition: 'store-and-forward'`. Peer H still holds
    T. Straggler S sends a change C for T (HLC recent enough to exceed H's
    watermark with R).
  - R applies S's batch → C is diverted, held forwardable (assert via
    `listForwardable`).
  - H pulls from R via `getChangesSince(H, sinceHLC_H)` → the returned ChangeSets
    include C with its **original hlc + siteId (S)**. H applies it normally (T is
    in H's basis) → assert C is materialized on H with S's HLC, and H's change log
    now carries it (so H would itself relay/serve it).
  - Re-pull from R after H's watermark advanced past C → C is **not** re-sent
    (watermark filter). Assert no duplicate relay.
- **Ping-pong loop-freedom between two non-holders.** R1 and R2 both
  `store-and-forward`, neither holds T. S→R1. R1 forwards C to R2; R2 re-disposes
  (re-holds forwardable, HLC-keyed idempotent — assert exactly one entry) and would
  forward C back to R1; R1 already holds C (idempotent no-op) and the per-peer
  watermark (R1↔R2) stops re-send after one exchange. Assert the system quiesces:
  no unbounded growth, exactly one forwardable entry per peer for C, no infinite
  re-send. Convergence rests on original-HLC identity + watermark, **not** on any
  per-table peer-membership oracle (none exists).
- **Echo exclusion.** A holder that pulls is also the straggler origin
  (`change.hlc.siteId === peerSiteId`): C is excluded from that peer's
  `getChangesSince` — never echoed back to its own author.
- **Below-watermark straggler is not relayed (documented limitation).** A
  forwarded change with `HLC ≤ sinceHLC` is filtered out; assert it does not appear
  and the watermark does not regress. (This is the accepted scalar-watermark
  limitation, asserted as intended behavior, not a bug.)
- **Ordering.** Forwarded changes interleave with locally-originated changes in
  global HLC order in the response (assert `buildTransactionChangeSets` output is
  HLC-consistent across the merged set — no forwarded change appears out of HLC
  order relative to change-log changes).
- **From-zero path.** A peer pulling with no `sinceHLC` (`collectAllChanges`)
  receives all forwardable changes (origin ≠ peer), grouped by original txn.
- **Recursive re-dispose at the receiver.** A holder that *also* lacks T
  re-disposes the forwarded change per its own config (re-quarantine / re-forward),
  converging because every hop preserves the original HLC. (Covered by the
  ping-pong test; assert the receiver's disposition path runs on the forwarded
  change.)
- **GC vs in-flight relay (document, light test).** A forwardable entry pruned at
  the horizon while a slow peer still needs it is **acceptable** — that peer was
  already past the delivery guarantee. Assert prune removes it and a subsequent
  `getChangesSince` no longer relays it; document the tradeoff.
