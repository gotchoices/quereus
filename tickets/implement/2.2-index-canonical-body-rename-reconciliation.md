# Index canonical body: concurrent column-rename reconciliation

Follow-on to `index-canonical-body-collation` (prereq). The drift detector compares
a declared index body against the actual catalog body. The **actual** body renders
the *pre-rename* column names (a column rename has not landed at diff time), while
the **declared** body renders the *new* names. So a same-named index over a column
renamed in the same diff renders new-name (declared) vs old-name (actual) and
produces an unnecessary drop+recreate — churn, not corruption.

Constraints already solve the analogous problem via `reconciledDeclaredBody`
(schema-differ.ts): inverse-apply the in-diff column renames to the declared body
before comparing, so a pure rename matches (no churn) while a genuine body edit
still differs (recreate). Indexes need the same reconcile on their column list.

## Design

The index body's canonical render **excludes** the `on <table>` reference, so a
*table* rename alone never churns an index body — only **column** renames matter.
This is strictly simpler than the constraint FK case (no parent-table / parent-column
reconcile needed).

Extend `declaredIndexCanonicalBody` (added by the prereq) to accept the relevant
table's column renames and inverse-apply them to each column's resolved bare name:

```
declaredIndexCanonicalBody(indexStmt, declaredTable, colRenames)
  for each index column:
    bareName  = indexedColumnBareName(col)            // declared (NEW) name
    collation = resolve from bareName + declaredTable  // resolved on the NEW name (prereq logic)
    oldName   = colRenames.find(r => r.newName ~= bareName)?.oldName ?? bareName
    emit { name: oldName, direction: col.direction, collation }
```

**Ordering is load-bearing**: resolve collation from the *new* (declared) column
name FIRST (the declared table's `ColumnDef` is keyed by the new name), THEN
inverse-rename the emitted name to its old form. Reversing this would look up the
declared column collation under the old name and miss it.

The index loop in `computeSchemaDiff` already has the `columnRenamesByTable` map
(built in the existing pre-pass, keyed by declared/new table name lowercased) and
`declaredTables` in scope. For each declared index, look up the renames with
`columnRenamesByTable.get(indexStmt.table.name.toLowerCase()) ?? []` and pass them
to `declaredIndexCanonicalBody`. The index's `stmt.table.name` is the table's
*declared* (post-rename) name, which is exactly how `columnRenamesByTable` is keyed
— so a table renamed in the same diff still resolves its column renames correctly.

Reuse the existing case-insensitive inverse-rename shape (`inverseRenameConstraintColumns`
maps new→old by `cr.newName.toLowerCase() === name`). The index column list is the
collate-folded/plain `IndexedColumn[]`, not the constraint `{ name }[]`, so the
unwrap-to-bare-name step (already present from the prereq's normalization to plain
form) handles both forms before the rename lookup.

A genuine body edit layered on a rename still differs after reconciliation (the
edit is not undone by the inverse-rename), so it still drops+recreates — preserving
the rename-vs-body precedence the constraint path established.

## Edge cases & interactions

- **Pure column rename under a stable-named index** — `index ix on t(email)` with
  `email` renamed to `email_addr` (via `quereus.previous_name`/`quereus.id` on the
  column): expect the **column rename only** (`tablesToAlter[].columnsToRename`), and
  **no** index drop/recreate. This is the headline case.
- **Body edit + rename together** — rename `email`→`email_addr` AND add a second
  index column / flip direction / add COLLATE: index still **drops+recreates** (body
  differs after reconciliation) while the column rename is also emitted.
- **Composite index, one renamed + one stable column** — only the renamed column is
  inverse-mapped; body matches after reconcile → no churn.
- **Collation + rename interaction** — a renamed column whose collation is unchanged
  must not churn (collation resolved on the new name, then name mapped back to old);
  a renamed column whose collation *also* changed must recreate. Covers the
  ordering requirement above.
- **Renamed index AND renamed column together** — an index matched via its own
  rename hint (`indexRenames.pairs`) whose referenced column also renamed: the body
  reconcile still applies (drop targets the actual index name, recreate carries the
  declared name only if the body genuinely changed). A pure column rename under a
  pure index rename → the index rename op only, no body recreate.
- **Table rename, columns stable** — index body excludes the table name, so no
  churn; the column-rename lookup keyed by the new table name returns `[]`. Confirm
  no spurious recreate.
- **partial WHERE referencing a renamed column** — KNOWN LIMITATION to document
  (not necessarily fix here): the canonical body's WHERE predicate
  (`lowerExprIdentifiers(stmt.where)`) renders the NEW column name on the declared
  side vs OLD on the actual side, so a partial index whose predicate references a
  renamed column would still churn. The constraint CHECK path reconciles this via
  `renameColumnInCheckExpression`; decide whether to extend the index reconcile to
  inverse-rename the WHERE predicate too. **Recommended: do it** — the same
  `renameColumnInCheckExpression(cloneExpr(where), table, newName, oldName, schema)`
  inverse-rewrite the constraint CHECK path uses applies directly, keeping indexes
  symmetric with constraints. If deferred instead, add an explicit
  `tickets/backlog/` ticket and a failing-case note; do not leave it silently
  unhandled.
- **require-hint policy** — a reconciled pure column rename produces NO index
  drop+create, so it must not trip the `index` unhinted-rename guard; verify under
  `policy: 'require-hint'` that a column-rename-only diff does not throw.

## Tests (add under the "declarative differ stability" describe in index-ddl-roundtrip.spec.ts)

The existing `diffIndexEdit` applies a baseline then diffs a modified declaration.
A column rename needs the `quereus.previous_name` (or `quereus.id`) hint on the
declared column in the *modified* schema (baseline column under the old name,
modified column under the new name with the hint). Expected outputs in parentheses.

- column renamed, index references it, body otherwise identical →
  `indexesToDrop = []`, `indexesToCreate = []`, and the matched table's
  `columnsToRename` carries `{ oldName, newName }`.
- column renamed AND index body edited (e.g. direction flip) → index recreates AND
  column rename emitted.
- composite index, one column renamed, one stable → no index churn.
- renamed column whose inherited collation is unchanged → no index churn;
  renamed column whose collation also changed → recreate (ordering guard).
- partial-WHERE index over a renamed column → no churn (if the WHERE reconcile is
  implemented per the recommendation) — otherwise a documented failing/skipped case
  plus a backlog ticket.
- `policy: 'require-hint'`: a column-rename-only diff does not throw on the index
  guard.
