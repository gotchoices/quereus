# Store and isolation table layers resolve collations against their database

## Problem

`StoreTable` (persistent key-value backed tables) and `IsolatedTable` (the
read-your-own-writes transaction overlay) both call the three-argument
`compareSqlValues(a, b, collationName)` from `@quereus/quereus`. That overload
resolves the name against a **process-global** registry holding only `BINARY`,
`NOCASE`, and `RTRIM`. A collation registered with `db.registerCollation(...)` is
invisible there, so these layers silently fall back to raw byte comparison for:

- `store-table.ts` ~1204 — a key comparator.
- `store-table.ts` ~1912 / ~1950 / ~2012 — `UNIQUE` conflict detection against
  candidate and live rows.
- `isolated-table.ts` ~702 — row ordering; ~1124 — key equality; ~1273 — merged-row
  comparison.

Both classes extend `VirtualTable`, which carries `db: Database`, and both already
reach engine internals via `this.db as DatabaseInternal`. The resolver is one call
away.

## Design

`3.2-collation-resolver-seam` adds `getCollationResolver(): CollationResolver` to
`Database` and to the `DatabaseInternal` interface. `CollationResolver` is
`(name: string) => CollationFunction`, throwing on an unregistered name.

At each site:

1. Obtain the resolver once — `(this.db as DatabaseInternal).getCollationResolver()`.
   Cache it on the table instance (`private readonly collationResolver`), set in the
   constructor. It is a stable bound closure reading the live registry, so caching
   the *resolver* is safe; do not cache resolved *functions* across a
   `registerCollation` call.
2. Resolve each collation **name → function once per comparator / per constraint
   check**, above any row loop.
3. Replace `compareSqlValues(a, b, name)` with `compareSqlValuesFast(a, b, fn)`.

Leave the name-less calls (`store-table.ts` ~1661/~1667, `isolated-table.ts` ~725,
`quereus-sync/src/sync/store-adapter.ts` ~476) alone — they are BINARY by intent.

Note both files already import the shared `uniqueEnforcementCollations(schema, uc)`
helper (see the completed `unify-unique-enforcement-collation-resolver` ticket),
which returns per-column collation *names*. Those names are exactly what now feed
the resolver. Resolve them once per `UNIQUE` check, not once per candidate row —
`~1912`/`~2012` sit inside candidate loops.

## Edge cases & interactions

- **Cross-package build order.** `getCollationResolver` must be on the exported
  `DatabaseInternal` interface (not just the `Database` class) or the store and
  isolation packages will not type-check. `yarn build` runs packages sequentially
  and will catch this.
- **Unknown collation now throws.** A persisted table whose DDL names a custom
  collation the embedder has not re-registered on this connection now raises
  `no such collation sequence: X` on first comparison instead of silently
  byte-ordering. That is the intended behavior (see `3.2`). Check
  `packages/quereus-store/test/` for any fixture that opens a store with a custom
  collation without registering it.
- **Resolver acquisition timing.** `StoreTable` / `IsolatedTable` constructors run
  during `xConnect`. Resolving *names* there would be too early (a collation may be
  registered after connect); grabbing the *resolver closure* there is fine. Do not
  resolve names in the constructor.
- **`UNIQUE` under a custom collation.** With the resolver in place, a custom
  collation genuinely participates in conflict detection. Add a store test:
  register `NOACCENT` (or a simple `REVERSE`), declare a `UNIQUE` column with it,
  insert two values that collide under it, expect a constraint violation. Today the
  second insert succeeds.
- **`UNIQUE` under `NOCASE`** must keep rejecting case-variant duplicates — a
  regression guard, since `NOCASE` resolves in *both* registries and would mask a
  wiring mistake.
- **Isolation overlay vs base table disagreement.** The overlay's key comparator
  (~1124) and the base table's must use the same collation function, or a staged row
  fails to shadow the base row it replaces. Both now come from the same database
  resolver. Add a test: stage an update to a `REVERSE`-collated-PK row inside a
  transaction, read it back before commit, expect the staged value once (not the
  base row too).
- **Savepoint / rollback paths** re-derive comparators. Make sure any comparator
  rebuilt after `xRollbackTo` uses the cached resolver, not a fresh global lookup.
- **Empty collation-name array** (a `UNIQUE` over zero columns cannot exist, but a
  `pkDefinition` slice can be empty in `isolated-table.ts` ~1273): `.map(resolver)`
  over an empty array is fine; assert no throw.
- **Hot path:** ~1912 and ~2012 are inside candidate loops that already run per
  conflicting row. Hoisting the resolve is a small win, not a regression risk — but
  a per-row `resolver(name)` call *would* be, because the resolver throws on miss
  and cannot be inlined.
