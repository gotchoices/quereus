## Why

`docs/lens.md` § The Default Mapper specifies the optional/EAV write dual: a null→non-null update **materializes** a member row, a non-null→all-null update **deletes** it. Today `decomposition.ts` `routeAssignment` rejects an optional-member / EAV target with `unsupported-decomposition-update`. Consumer 1 (`view-write-outer-join-nonpreserved-update`, now in review) shipped the outer-join dual: per affected row, **matched → base UPDATE**, **absent → INSERT the missing component**, realized over the existing capture substrate as plain AST base ops (no new ViewMutationNode/emitter). This ticket applies the same shape to the decomposition fan-out, plus the third **all-null → DELETE** branch.

## Key realization from consumer 1 (reuse this)

Consumer 1 did **not** need a new `ViewMutationNode` conditional-materialization side structure or emitter change. The matched-UPDATE rode existing per-side routing (rows the partition excludes are naturally no-ops), and the materialize-INSERT / delete branches were expressible as ordinary AST `BaseOp`s over the captured/identifying partition. The decomposition path already keys every member op off an **anchor subquery** (`<memberKey> in (select <anchorKey> from <anchor> where <pred>)`), so the same trick applies — these stay AST `BaseOp`s, no plan-node substrate. Do **not** fork a parallel conditional-emit node.

## Target behavior

`decomposition.ts` `routeAssignment` / `decomposeUpdate` currently route each assignment to a single mandatory, non-EAV member and reject optional / EAV / shared-key / computed targets. Replace the optional-member and EAV rejects with the per-row branch:

### Optional columnar member (e.g. the Car ⟕ CarSpeed decomposition, `docs/lens.md`)

`update Car set topSpeed = 200 where id = 7`, `topSpeed` on the optional `CarSpeed` member:
- **matched** (a CarSpeed row exists for the anchor key) → `update CarSpeed set topSpeed = 200 where <memberKey> in (<anchor subquery>)` (the existing member-update path).
- **null-extended** (no CarSpeed row) → `insert into CarSpeed (<memberKey>, topSpeed) select <anchorKey>, 200 from <anchor> where <pred> and <anchorKey> not in (select <memberKey> from CarSpeed)` — the anchor key threads the member key (EC), the assigned value fills the column, other member columns take base defaults.
- **all-null** (every optional column of the member assigned null — for a single-optional-column member, `set topSpeed = null`) → `delete from CarSpeed where <memberKey> in (<anchor subquery>)` (the component row becomes empty). Gate strictly on "all of the member's optional columns assigned null in this statement"; a partial null leaves a row with nulls (an update, not a delete).

### EAV pivot member

A per-attribute triple is **materialized** (value non-null) or **deleted** (value null) — the EAV analogue. Reuse the EAV triple insert/delete fan-out the decomposition insert/delete already builds (`emitMemberInsert` EAV branch, `memberDeleteOp` EAV branch): `update E set p = 99 where id = 1` upserts the `(entity, 'p', 99)` triple; `set p = null` deletes it.

## Out of scope (keep rejecting)

- Composite shared keys (`unsupported-decomposition-key`).
- A predicate touching a non-anchor member / EAV pivot / subquery (`unsupported-decomposition-predicate`) — unchanged; the materialization keys off the anchor-resolvable predicate exactly as the existing member update/delete do.
- Shared-key (identity) column writes (`unsupported-decomposition-update`), computed-mapping writes (`no-inverse`), cross-member value reads (`cross-source-assignment`).
