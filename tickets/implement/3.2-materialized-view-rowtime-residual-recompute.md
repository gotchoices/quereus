> **Builds on the shared maintenance substrate.** This ticket fills in the body of
> the `'residual-recompute'` arm that `incremental-maintenance-plan-abstraction` (1.5)
> stubbed and `incremental-maintenance-cost-gate` (1.6) made cost-selectable. It does
> **not** re-derive the cost gate, the strategy union, the synchronous DML-boundary
> hook, or the equivalence harness — those are the substrate's. It implements two new
> **reachable** body shapes on that abstraction and extends the harness to cover them.
> The lateral-TVF fan-out (`'prefix-delete'`) arm is a **separate** downstream ticket
> (`materialized-view-rowtime-prefix-delete-lateral-tvf`) that reuses the residual
> primitive built here.

## What this arm maintains

Two body shapes whose per-write backing delta is a **bounded, key-filtered
re-execution** of the body (not a pure projection, but still bounded by the change):

1. **Single-source aggregate** — `select g1, …, agg(...) from T [where P] group by g1, …`
   over **bare** group-by columns. Each changed source row belongs to exactly one
   group (its `(g1,…)` tuple); maintaining the MV means **recomputing that group** and
   replacing its backing row. A key-changing update touches two groups (OLD + NEW).

2. **1:1 row-preserving inner/cross join** — a body where `T` contributes **exactly
   one** MV row per governed `T` row, proven by the coverage prover's existing 1:1
   decomposition (row-preservation **or** NOT-NULL FK→PK referential integrity for the
   no-row-loss obligation; `proveJoinNoFanout` / `isUnique(T.pk)` at the join frame for
   the no-fan-out obligation). Each changed `T` row maps to one backing row keyed by
   `T`'s PK; maintaining the MV means **recomputing that one joined row** (join fan-in
   over the live lookup side) and replacing it.

Both reduce to the same kernel: *derive the affected binding key from the changed row →
run a key-filtered residual against live source state → delete the old backing slice →
upsert the recomputed slice.* Fanning (non-1:1) joins are **out of scope** — they need
the by-prefix delete of the downstream prefix-delete ticket; reject them here as `shape`
(the existing fan-out gate already does, via `proveJoinNoFanout`).

## The residual kernel (the hard part the consolidation removed)

The row-time-only consolidation deleted the old commit-time residual machinery. This
ticket rebuilds it **synchronously, in-transaction**, on the shared abstraction. The
moving parts:

### Affected-key derivation — reuse `BindingMode`, no new analysis

`extractBindings(analyzedBody)` (`planner/analysis/binding-extractor.ts`) already
classifies each `TableReferenceNode` and yields exactly the columns this arm needs:

- **aggregate** → `{ kind: 'group'; groupColumns }` — the minimal GROUP BY columns in
  the source reference's output-column space. The affected group key is
  `groupColumns.map(c => changedRow[c])`.
- **1:1 join** → `{ kind: 'row'; keyColumns }` — `T`'s covered unique key (PK preferred).
  The affected row key is `keyColumns.map(c => changedRow[c])`.

The substrate's `'residual-recompute'` arm already reserves a `binding: BindingMode`
field; populate it at `buildMaintenancePlan` time from `extractBindings`. A `'global'`
classification (no derivable key) means the body is **not** residual-eligible — fall
back to the cost gate's `'full-rebuild'` / reject path, never silently re-run globally
per write.

### Key-filtered residual — reuse `injectKeyFilter`

`injectKeyFilter(block, targetRelationKey, keyColumns, 'gk' | 'pk')`
(`planner/analysis/key-filter.ts`) is the **already-shared** primitive — its module doc
names `database-materialized-views.ts` as an intended consumer. It rewrites the body
plan to pin the source `TableReferenceNode` to `pk0…`/`gk0…` parameters. The MV manager:

1. compiles the residual **once** at registration (cache it on the plan, like the
   assertion path caches `residualsByRelation`), parameterized by the binding key;
2. at maintenance time, binds the affected key tuple(s) and **executes the residual
   through the current transaction/connection** so it reads the source's *pending*
   layer (reads-own-writes) — the synchronous analogue of
   `database-assertions.ts:executeResidualPerTuple`, which runs the same shape at
   commit. Use the same emit/scheduler path the assertion evaluator uses
   (`emitPlanNode` → `Scheduler`), with a `RuntimeContext` bound to the live
   transaction so the source read sees this statement's writes.

For an aggregate the residual yields **one** recomputed group row (or zero if the group
emptied); for a 1:1 join it yields **one** recomputed joined row (or zero if a predicate
or the FK match now excludes it).

### Delete-then-upsert into the backing

Reuse the existing `applyMaintenanceToLayer(connection, ops)` path with `delete-key` +
`upsert` `MaintenanceOp`s — **no new op kind needed** for this arm (both shapes replace a
bounded, point-keyed slice; only the downstream lateral-TVF arm needs by-prefix delete):

- delete the OLD backing key (the affected group/row key computed from the OLD image);
- run the residual against current state; upsert each result row (0 or 1 rows here);
- for an UPDATE that moves the binding key (group-key change / PK change), the OLD and
  NEW keys differ → delete OLD slice, recompute+upsert BOTH the OLD-key group (it may
  still be non-empty from other rows) and the NEW-key group. Derive both keys and run
  the residual for each.

> **Emptied-group correctness.** When the last row leaves a group, the residual for that
> group key returns **zero** rows, so the delete-OLD-key with no upsert removes the
> backing row. This is why the arm always deletes the old key *before* re-running, even
> for inserts/updates — never "upsert only."

### Per-statement batching (why this prereqs the batching ticket)

`materialized-view-rowtime-per-statement-batching` makes maintenance accumulate per
statement and flush once. Residual recompute **must** dedup affected keys per statement:
a bulk `update t set v = v+1` touching 10k rows across 50 groups should run the group
residual **50 times, once per distinct affected group key** — not 10k times. Accumulate
the set of distinct affected binding keys (OLD ∪ NEW) over the statement, then run one
residual per distinct key at the statement-flush boundary. This is both the affordability
story and the natural place for the cost gate's runtime `degradeToRebuild` check to fire
(if the distinct-affected-key count crosses the crossover, do one `'full-rebuild'` for
the statement instead of N residuals).

> Correctness holds *without* batching too (each per-row recompute reads current state,
> and the last write to a group yields the right final value), so this prereq is for
> affordability + clean dedup, not soundness. But building on the batched accumulator
> avoids a rework, hence the hard prereq.

## Shape gate (extend `buildMaintenancePlan`)

Today `buildMaintenancePlan` rejects aggregates and joins outright. Replace those two
blanket rejections with **soundness-checked admits**, then let the cost gate (1.6) pick
the strategy:

- **Aggregate admit.** Single source `T`; body is `Filter? → Aggregate(group by bare
  cols) → Project? → Sort?`; every GROUP BY expression is a **bare source column**
  (reuse the `collectProducingExprs` / `ColumnReferenceNode` provenance already in this
  file — a computed group key is rejected for v1, as the binding's group columns must be
  source-column indices); aggregate functions may be any deterministic aggregate. The
  partial `WHERE`, if present, must be single-row-evaluable (`compilePredicate`) so the
  predicate-scope transition is handled the same way the inverse-projection arm handles
  it. Bare `count(*)`/`sum`/etc. are fine; `group by` is **required** (a scalar aggregate
  with no GROUP BY is one global row — admit as a single-group degenerate case keyed by
  the empty key, or reject as `shape` for v1; pick the empty-key admit only if the
  residual + harness cover it cleanly, else defer with a clear diagnostic).
- **1:1 join admit.** Reuse the coverage prover's decomposition directly — factor the
  reusable predicates out of `coverage-prover.ts` (`proveJoinNoFanout`,
  `innerJoinRetainsConstrainedTable`, the row-preservation side/type check) into a shared
  helper both the prover and this gate call, rather than duplicating the join soundness
  logic. The gate admits a join body iff it is provably 1:1 on `T`; `T`'s PK is the
  backing key. A fanning join is rejected `shape` (→ prefix-delete ticket).

Keep all the *unchanged* rejections (DISTINCT, set ops, recursive CTE, LIMIT/OFFSET,
non-deterministic projection, MV-over-MV) exactly as they are. Recursion and set ops stay
out of the row-time model entirely.

## Backing PK derivation

- **aggregate**: the backing PK is the group-by columns (already what
  `deriveBackingShape` / `keysOf` yields for a `group by` body — `propagateAggregateFds`
  emits the group-key FD). Confirm `MaterializedViewSchema.primaryKey` resolves to the
  group key, not the all-columns fallback.
- **1:1 join**: the backing PK is `T`'s PK, surfaced through the composite product key
  that `optimizer-keyed-cross-product-join-keys` (complete) now advertises via `keysOf`.
  For a 1:1 join the product key reduces to `T`'s key (the lookup side contributes ≤1
  row); verify `keysOf` surfaces a real column key (not the all-columns `isSet`
  fallback) so the backing table is properly keyed.

## Soundness notes (carry into code comments)

- The residual reads **live mid-transaction source state**, so the recomputed slice is
  exactly what `select <body>` would return at that point — the harness oracle
  (`read(MV_backing) == evaluate(body)`) is the regression net.
- The arm never re-runs the body **globally** per write: a body whose binding is
  `'global'` is not residual-eligible and routes to the cost gate's rebuild/reject.
- Determinism: a non-deterministic body (volatile UDF, `random()`/`now()`) cannot be
  residual-maintained reproducibly — keep the existing determinism rejection.

## Use case (unchanged from plan)

A logical-schema (lens) `unique`/PK whose declared covering MV is provable only through
an aggregate or join body needs row-time maintenance of that body to drive row-time
conflict resolution. Until this lands, such a constraint relies on detection-only
commit-time enforcement (the `lens.no-backing-index` advisory path in `docs/lens.md`).
This ticket does **not** wire the covering enforcement for aggregate/join MVs (the
`proveEffectiveKeyUnique` / lens path owns that) — it makes the backing table row-time
*consistent* so that enforcement becomes possible.

  groups, and updates that empty a group); assert `rows(select * from mv)` multiset-equals

  rollback**. Add a partial-`WHERE` variant (predicate-scope transition).

  FK `t.fk → p.id`; random mutations on `t` (and on `p` — a lookup-side change recomputes
  every `T` row whose key matches, via the lookup side being a dependency of the residual)
  with the same multiset + rollback assertion.

  contributing source row is deleted.
