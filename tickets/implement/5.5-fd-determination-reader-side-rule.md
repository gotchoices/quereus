## Context

Phase 2 of the approved direction B (decision record in plan ticket
`fd-determination-reader-side-preservation`, 2026-06-10). Phase 1
(`fd-kind-provenance-field`) made `FunctionalDependency.kind` trustworthy at
every producer, transform, and fan-out site. This ticket makes the readers
consume it, then deletes every producer-side drop gate the readers subsume.

## Two CONFIRMED live wrong-results bugs this fixes

Both verified failing at HEAD during planning (mocha repros run on 2026-06-10);
both are reader-side over-claims the producer-gate family structurally cannot
catch. Each becomes a permanent regression test:

1. **Constant-pin phantom singleton.** `∅→col` FDs from `where a = 1 and b = 2`
   survive projection (the empty-determinant exception in `projectFds`), so
   `hasSingletonFd` reads pure closure coverage as "≤1 row" and
   `rule-distinct-elimination` drops a required DISTINCT:
   ```sql
   create table t (a integer, b integer, c integer primary key);
   insert into t values (1, 2, 1), (1, 2, 2), (3, 4, 3);
   select distinct a, b from t where a = 1 and b = 2;  -- HEAD: 2 rows; correct: 1
   ```
2. **Partial-unique guarded key FD through a fanning join.** The guarded
   `{c}→others` 'unique' FD crosses a fanning join untouched (`dropSideKeyFds`
   keeps guarded FDs), activates at a filter above (multi-dependent shape passes
   the single↔single activation gate), and a projection then derives phantom key
   `{c}` over duplicated rows:
   ```sql
   create table t (id integer primary key, c integer, region text, p integer);
   create unique index t_c_partial on t (c) where p = 1;
   insert into t values (1, 100, 'east', 1), (2, 200, 'west', 1);
   create table u (uid integer primary key, k integer);
   insert into u values (10, 1), (11, 1);
   select distinct c, region from (
     select t.c as c, t.region as region, t.p as p from t join u on u.k = t.id
   ) where p = 1;  -- HEAD: 2 duplicate rows; correct: 1
   ```
   Fix = phase 1's fan-out kind-downgrade (guarded FDs included) + this ticket's
   kind-aware readers. Related: `returning-node.ts` emits its injective bi-FD
   with NO endpoint gate at all (project-node's gate was never mirrored) — a
   third latent producer, mooted by the reader rule.

## The reader rule

Core primitive in fd-utils.ts (name as you see fit; suggestion below):

```ts
/**
 * True iff `attrs` is provably row-unique on the relation: its FD closure
 * covers every column AND uniqueness is reachable —
 *   - the relation is a set (two rows agreeing on attrs would agree on all
 *     columns = a duplicate, impossible in a set), or
 *   - some unguarded kind:'unique' FD has determinants ⊆ closure(attrs)
 *     (rows agreeing on attrs agree on that unique determinant set; ≤1 row
 *     per its tuple ⇒ ≤1 row per attrs-tuple).
 * Coverage alone (a determination-only closure path over a bag) proves nothing.
 */
export function isUniqueDeterminant(
  attrs: ReadonlySet<number>,
  fds: ReadonlyArray<FunctionalDependency> | undefined,
  columnCount: number,
  isSet: boolean,
): boolean
```

Split, don't overload: keep the pure-coverage predicate and **rename it**
`closureCoversAll` (was `isSuperkey` — the old name reads as a uniqueness claim
and invites the exact misuse this family of tickets fixed). The rename is the
guard rail: every call site must re-pick coverage vs uniqueness explicitly.

### Reader rewrites (all in fd-utils.ts unless noted)

- `deriveKeysFromFds(fds, columnCount, isSet)`: candidate `K` is a key iff
  `isUniqueDeterminant(K, …)`. Keep the `det.length >= columnCount` tautology
  bound and `minimalCover` (same closure ⇒ same reachability).
- `hasAnyKey(fds, columnCount, isSet)`, `hasSingletonFd(fds, columnCount, isSet)`
  (the latter ≡ `isUniqueDeterminant(∅, …)`): thread `isSet`.
- `isAssertedKey(attrs, fds, columnCount, isSet)`: body becomes
  `isUniqueDeterminant(attrs, …)` — strictly more general (closure composes
  multi-FD paths the old per-FD anchoring missed) and now genuinely a
  uniqueness predicate for the sort/window strict-monotonic checks.
- `isUnique` (KeyRel): closure branch becomes
  `isUniqueDeterminant(colSet, rel.physical?.fds, columnCount, type.isSet)`.
  The proper-subset guard can be dropped: an all-columns probe on a bag fails
  the rule on its own (no unique FD ⇒ false; if a unique FD exists the relation
  cannot hold duplicates, so `true` is correct).
- `keysOf`: pass `type.isSet` through; step 4 (all-columns fallback for sets)
  stays. The `isSet` branch in the rule is what restores derived keys above
  DISTINCT / GROUP BY (e.g. DistinctNode passes source FDs through verbatim and
  declares `isSet: true` — a determination-covering `{a}` is now a genuine key).
- Callers needing `isSet` all have a node/type in hand: `characteristics.ts`
  (`guaranteesUniqueRows`, `hasUniqueKeys` → `node.getType().isSet`), `sort.ts:90`,
  `window-node.ts:265`, `rule-join-greedy-commute.ts:15`. The `key-utils.ts:424/427`
  fallback branch (logical type unavailable) passes `isSet: false` — conservative.

### Gate removals (the point of direction B)

Every gate below is subsumed by the reader rule; leaving any in place keeps
dropping the determinations whose preservation is this ticket's payoff:

- DELETE `foldSingleSingleGated` (fd-utils.ts:601) and replace its three call
  sites with plain folds: `reference.ts:156/:179` (CHECK + assertion-hoist; the
  `realKeyFds` probe snapshot at :132 dies too; keep the guarded-FD pass-through
  behavior — guarded producer FDs still just `addFd` in untouched) and
  `filter.ts:130` (predicate FDs fold unconditionally — they are 'determination'
  by phase 1).
- `filter.ts activateGuardedFds`: remove the single↔single endpoint gate
  (:281–:308) — activation is now `stripGuard` (kind-preserving) for every
  entailed FD. KEEP the `valueEquality`-keyed EC lift exactly as is (it is the
  EC optimization, orthogonal to key soundness; phase 1 made the marker durable
  through shift/project as a bonus).
- `join-utils.ts`: equi-pair endpoint gate (:296–:308) — emit both directions
  unconditionally as 'determination'. Replace `dropSideKeyFds` with phase 1's
  kind-downgrade only (stop dropping; the FDs stay as determinations). Delete
  the now-unused `equiKeyFds` probe.
- `project-node.ts:303–:311` / `returning-node.ts:257–:262`: drop the endpoint
  gate (project) — emit injective pairs unconditionally as 'determination'
  (returning-node already does; now it's sound).
- Audit remaining `closureCoversAll` (ex-isSuperkey) call sites during the
  rename — each must consciously be coverage semantics. Expected outcome:
  closure-internal uses (deriveKeysFromFds via the new primitive), and the
  key-utils fallback (switched to the uniqueness primitive with isSet:false).

## Edge cases & interactions

- **Controls must not regress**: every "endpoint IS a genuine key ⇒ DISTINCT
  still eliminated" control in the existing specs (the reader rule must derive
  at least everything the gates allowed: a 'unique' FD U with U ⊆ closure(K)
  generalizes the old endpoint-superkey test).
- **isSet=false with genuine 'unique' FDs**: LEFT-join output is always
  `isSet:false` yet carries preserved-key 'unique' FDs — keys must still derive
  via the unique-FD branch, not the isSet branch.
- **Set + determination-only closure**: genuinely a key (the isSet branch) —
  this is gain (c); also covers constant-pinning ALL columns of a set (now a
  sound ≤1-row derivation where the old `hasSingletonFd` was accidentally right).
- **Bag + determination-only closure**: never a key — bugs 1/2 and all four
  regression families.
- **Guarded FDs**: skipped by `computeClosure` and by the unique-witness scan
  (only UNguarded 'unique' FDs can witness) — re-verify after gate removal that
  no guarded FD reaches closure consumers pre-activation.
- **Activation above a NON-fanning join**: partial-unique FD stays 'unique'
  (phase 1 downgrade only fires on non-preserved sides) ⇒ DISTINCT elimination
  still works there — add a control for bug 2's repro with a 1:1 join.
- **Aggregate/Distinct stack**: group-key 'unique' FD + isSet — keysOf above an
  AggregateNode unchanged or better.
- **Empty-determinant FDs after projection** (bug 1's vector): pinned-bag,
  pinned-set, and partially-pinned variants.
- **Zero-column relations** (`columnCount === 0` early-outs in the rewritten
  helpers — preserve current semantics).
- **FD cap pressure**: a relation with many CHECK determinations must not evict
  the unique witnesses (phase 1's cap bias; add a stress test only if cheap).
- **`analyzeJoinKeyCoverage`** consumes `keysOf`/`isUnique` — verify preserved
  keys (and therefore 'unique' minting in `withKeyFds`) still match: derived
  keys feeding `preservedKeys` are now uniqueness-true by construction.
