## Shape of the problem

`docs/materialized-views.md` is 826 lines but **28,575 words** — the prose is
unwrapped, so line count badly understates it. Unlike `optimizer.md`, its outline
is intact: eleven `##` sections, each internally coherent. The problem is purely
volume. Nobody audits a 28,000-word file against the implementation, and the file
covers subjects with genuinely different audiences:

- a user asking *what a materialized view is and how to declare one*,
- an engine developer changing *how maintenance keeps it fresh*,
- a **module author** implementing the backing-host capability,
- a sync/replication developer using the external row-change ingestion seam.

Today all four read the same file. The split follows the audience boundary.

## Target layout

Measured `##`/`###` word counts, as of plan time — re-measure. Every target lands
under the 12,000-word cap, so **no new doc needs a ratchet entry**.

**`docs/materialized-views.md` — the overview (~9,700 words).** Keeps: the opening
("a transparent materialization cache… observably indistinguishable from the plain
view it derives from"); `## Why one model`; `## Substrate: a maintained table`
*minus* the backing-host subsection (so: the intro + `### Primary key inference`,
~1,800 words); `## DDL statements` (all four verbs + `REFRESH`, ~3,800);
`## Query resolution` (~1,545); `## Write boundary (write-through)` (~652);
`## Change-scope projection` (~120); `## Declarative-schema integration` (~1,091);
`## Current limitations` (~469). Plus a table of the five satellites.

**`docs/mv-maintenance.md` (~5,800 words).** `## Maintenance strategy` (the
backward cost gate, the arms, the full-rebuild floor) + `## Maintenance (row-time,
per-statement)` (the five strategy write-ups, value-identical write suppression,
MV-over-MV cascade, "Synchronous, transactional, per-statement").

**`docs/mv-constraints.md` (~4,900 words).** `## Derived-row constraint validation`
(CHECK / FK / secondary UNIQUE, parent-side referential enforcement) +
`## Covering structures` (implicit + explicit covering structures, the coverage
prover, enforcement through a covering MV). These belong together: covering
structures exist to *enforce* constraints, and `lens.md` cites both from the same
paragraphs.

**`docs/mv-ingestion.md` (~2,620 words).** `## External row-change ingestion` —
facets, trust boundary, transaction & visibility contract, DML replay vs. the
ingestion seam, converging all materialized views, relationship to
`Database.notifyExternalChange`. The sync/replication audience.

**`docs/mv-schema-change.md` (~2,425 words).** `## Schema-change staleness` —
staleness marking, rename propagation ("MV ≡ faster view"), `ALTER TABLE` on the
maintained table itself.

**`docs/mv-backing-host.md` (~2,850 words).** `### Backing-host capability` and the
cross-module atomicity note. This is the **module-author contract** — the
module-neutral surface every privileged MV operation routes through — and
`docs/module-authoring.md` and `packages/quereus-store/README.md` already link
straight into it. It has no business being a subsection of "Substrate."

## Rules of the move

Identical to `docs-optimizer-split-satellites`, restated so this ticket stands alone:

- Cut at headings, never line numbers.
- Promote depth by one per satellite; heading slugs are depth-independent, so only
  the *filename* in each inbound link changes.
- Every removed `##` leaves a one-to-three sentence stub + link in the overview at
  its original position.
- Apply the three-vocabulary sort from `docs/doc-conventions.md`: delete narrative
  history; condense why-nots into `## Rejected alternatives` in the receiving
  satellite; push future/planned prose to `docs/todo.md` or a backlog ticket with
  a one-liner left under `## Current limitations`.
- **`## Why one model` is rationale and it stays, whole.** It is the clearest
  statement in the repo of why there is no refresh-policy knob, and deleting it
  would guarantee someone re-proposes asynchronous maintenance. Rationale is not
  history.
- Do not extract invariants — that is `docs-invariants-mv`. Do not change engine
  behavior. A wrong claim you find is a review finding, not a scope extension.

## Edge cases & interactions

- **`####`-depth anchors were never enumerated at plan time.** Five inbound links
  target them: `#coarsened-backing-keys`, `#cross-module-atomicity`,
  `#the-store-host-using-store`, `#aggregate-rollup-indexed-view-matching`,
  `#join-subsumption`. All five currently resolve (the checker proved it), so each
  exists somewhere. Locate each before deciding its satellite. `#the-store-host-using-store`
  in particular is likely a `####` under the backing-host capability and is the
  only inbound link from outside `docs/` — breaking it breaks the
  `quereus-store` README.
- **Slug collisions after promotion.** The five maintenance-arm headings are
  quoted strategy names (`### 'inverse-projection' (covering-index shape)`); their
  slugs contain the quotes stripped. Confirm the promoted set has no duplicate
  heading text, and never rely on the checker's `-1` disambiguation — it makes a
  wrong link resolve *successfully*.
- **`## Covering structures` is cited from three different docs for three
  different reasons** (`schema.md` for the auto-index reframing, `lens.md` for
  basis-layer covering MVs, `optimizer.md` for the coverage prover). Moving it into
  `mv-constraints.md` is a judgement call: the alternative home is `lens.md`, which
  is itself 17,932 words and over cap. Constraints wins because enforcement is what
  covering structures are *for*. Record this in the satellite's opening paragraph so
  the next reader does not re-litigate it.
- **`## Substrate` loses its largest subsection.** After backing-host leaves, check
  the surviving intro still explains the `TableSchema` + `TableDerivation`
  one-object model without forward-referencing text that has moved, and that the
  ASCII diagram survives intact.
- **Overview must stay under 12,000 words** with no ratchet entry. It should land
  near 9,700. If it does not, `## DDL statements` (3,792) is the next candidate to
  leave — but prefer trimming history first.
- **Five new files in one commit.** Work satellite-by-satellite (cut → stub →
  repoint → `yarn docs:check`) so each is green before the next begins.
- **Interaction with `docs-optimizer-split-satellites`** (sibling; no ordering
  constraint): both edit the four links between `optimizer.md` and
  `materialized-views.md` (`#materialized-view-query-rewrite-read-side`,
  `#coverage-proving`, `#effective-key-proving-body-proves-it`, and this doc's
  `#covering-structures` back-link). Whichever lands second re-runs
  `yarn docs:check` and repairs the crossing. Neither may assume the other's state.
- **Interaction with `docs-stability-tiers`** (separate plan ticket): materialized
  views are proposed as **Beta**. Five new docs are five new places a tier banner
  goes. Keep each satellite's header conventional (h1, one-line abstract, then
  content) so a banner has an obvious slot. Do not add banners here.
