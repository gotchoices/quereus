# Attach/detach verbs and the maintained-table syntax

Part 2 of the maintained-table chain (prereq: the unified
table+derivation schema model). Adds the authoring surface and the two
lifecycle verbs the migration pattern's flip/contract phases stand on
(`docs/migration.md` § The pattern), and switches the canonical DDL to the
table form.

## Syntax

```sql
-- Declared-shape form: the table layout is authored (the frozen-basis
-- invariant); the body must derive exactly that shape.
create table X (
  handle text collate nocase primary key,
  email text
) using store()
  maintained as select handle collate nocase as handle, email from Contact_v1
  [insert defaults (col = expr, …)]
  [with tags (…)];

alter table X set maintained as <body> [insert defaults (…)];  -- attach / re-attach
alter table X drop maintained;                                  -- detach
```

- `maintained` is a **contextual** keyword (no new reserved word), lowercase
  per project convention. Clause order on create:
  `(columns) → using → maintained as … → insert defaults → with tags`.
- AST: `CreateTableStmt.maintained?: { select: AST.QueryExpr;
  insertDefaults?: AST.ViewInsertDefault[] }`; `AlterTableAction` gains
  `{ type: 'setMaintained', select, insertDefaults? }` and
  `{ type: 'dropMaintained' }`.
- `alter table … set maintained` has **no** `using` clause — the module is
  the table's identity and never changes via attach.
- **Sugar:** `create materialized view mv [(cols)] [using m(…)] as <body>`
  normalizes to the table form with the shape *derived* from the body
  (column list = renames). `drop materialized view X` requires a derivation
  and then behaves as `drop table X`. `refresh materialized view X` works
  on any maintained table. The declarative `materialized view` item stays
  accepted (differ normalization is ticket
  `maintained-table-differ-transitions`); `declare schema` table items may
  carry the maintained clause through the existing CreateTableStmt reuse.
- **Canonical DDL** (ddl-generator, catalog persistence, schema export)
  becomes the `create table … maintained as` form for every maintained
  table regardless of authoring form — one comparison surface, stable
  round-trip. `bodyHash` formula is unchanged (the derivation's recorded
  `columns` list participates as today); revisit only if the differ ticket
  needs it.

## Attach semantics — verify-by-diff (resolved at plan stage)

Attaching to a table that already has rows (the upgrading-peer case)
**re-derives and reconciles**; it never trusts blindly and never refills
wholesale:

- Plan the body (rewrite-suppressed), run the create-time gates
  (determinism, keyed-or-coarsened body, relational output, full-rebuild
  size threshold — identical to create), derive the backing shape.
- **Shape mismatch is an error** — sited, naming the first mismatching
  column/PK component — with no catalog or data mutation.
- Evaluate the body and apply a **keyed diff** against the table's current
  effective contents — the same collation-aware pairing + byte-faithful
  `rowsValueIdentical` skip discipline as the full-rebuild floor's
  `'replace-all'` op: identical derivable content ⇒ **zero row writes**;
  divergence ⇒ **derived content wins** (upsert changed, delete extra),
  reporting only the genuine `BackingRowChange`s (which cascade to consumer
  maintained tables). Duplicate derived keys ⇒ the loud "must be a set"
  reject, attach aborts whole.
- On success: stamp `derivation`, register row-time maintenance, stamp
  covering links (coverage prover), fire the modified/added event so store
  catalogs re-persist the DDL.
- Rationale (documented tradeoff): blind trust stays the *rehydrate* fast
  path's domain, where clean-shutdown attestation gates it; the verb has no
  such attestation, attach is a rare lifecycle event, and one body
  evaluation deterministically heals any lag. The diff (not a wholesale
  replace) is what makes attach non-destructive to row identity and keeps
  reported changes minimal — the property the future sync change-log
  opt-in (`sync-derivation-changelog-optin`, backlog) depends on.
- `create table … maintained as` is attach-to-empty: same path, diff
  against empty = the fill. Keeping `replaceContents` as a fast path for
  the genuinely-empty case is fine if observably equivalent.
- Re-attach (`set maintained` over an existing derivation) = replace the
  body atomically: gates + shape check against the *existing* table shape,
  then the same reconcile. This is the differ's body-change primitive.

**Cycle check.** Create-MV could never form a cycle (consumer created after
producer); attach can (`alter table A set maintained as select … from B`
where B's derivation reads A) — including self-reference (`… from A`).
Reject at attach with a sited diagnostic by walking the
sourceTables→derivation edges. The cascade's depth guard stays as
defense-in-depth.

## Detach semantics

`alter table X drop maintained` — nothing physical changes:

- Remove `derivation`, release the row-time plan, clear staleness state
  (meaningless without a derivation), un-stamp covering-structure links
  (the constraint's `coveringStructureName` forward pointer — UNIQUE
  enforcement falls back to the auto-index), fire events so cached
  write-through plans invalidate and store catalogs re-persist the DDL
  (now without the clause).
- The table keeps its rows, becomes ordinary and user-writable. READONLY
  shedding is structural: the (backlog) enforcement seam keys off
  `derivation` presence, never a name pattern — see
  `backing-tables-readonly-enforcement`, which gained a prereq on the
  unified model.
- Consumers whose bodies read X are unaffected (X exists with the same
  data; subsequent user writes to X drive their maintenance exactly like
  any source writes).
- Detaching a *stale* maintained table is allowed (the flag leaves with the
  derivation).

## Edge cases & interactions

- attach-to-populated, identical content ⇒ zero row writes and zero
  reported changes (pin via change events / a consumer maintained table
  that must see no dispatch).
- attach-to-populated, divergent ⇒ derived wins; pin both the data outcome
  and the minimal reported diff (extra row deleted, lagged row updated,
  missing row inserted — one case each).
- attach with duplicate derived keys (coarsened-key collision present in
  source) ⇒ loud reject, table and catalog untouched, error names the key.
- attach shape mismatch (column name, type, not-null, collation, physical
  PK order/direction) ⇒ sited error, no mutation; `create table …
  maintained as` with mismatched declared shape errors before any catalog
  registration (all-or-nothing, including `if not exists` interplay —
  existing table + `if not exists` skips entirely, never half-attaches).
- self-reference and two-table cycle via attach ⇒ rejected with the cycle
  diagnostic; re-attach replacing a body with a non-cyclic one over the
  same table works.
- attach/detach inside an explicit transaction: catalog flip + reconcile
  writes commit/roll back in lockstep (DDL-in-transaction posture matches
  today's create-MV; store `replaceContents`' commit-first caveat applies
  only if the fill fast path is used).
- detach → user write → re-attach: reconcile records exactly the
  user-written drift; maintenance resumes (the round-trip test below).
- detach of a covering MV that was answering a UNIQUE constraint ⇒
  enforcement falls back to the auto-index, conflict resolution still
  correct (pin with an `insert or replace` after detach).
- coarsened-key attach (the migration shape) ⇒ key-coarsening warning fires
  on attach exactly as on create; `coarsenedKey` stamped.
- attach where the body is full-rebuild-only over a source past the size
  threshold ⇒ rejected (same as create).
- write-through DML against the maintained table immediately after attach
  and immediately after detach (dispatch flips with the catalog flip;
  cached statement invalidation).
- `insert defaults` clause round-trips through the new canonical form;
  MV-sugar with explicit column list round-trips (renamed columns become
  the table's declared column names).
- store-backed maintained table: attach/detach re-persists the catalog
  bundle; rehydrate of a detached (plain) table and of a re-attached one
  lands in the right state; the adopt fast path consumes the new canonical
  DDL form.

## Key tests

- attach-to-empty ≡ create-MV (same fill semantics, same maintenance
  behavior afterward — extend the maintenance-equivalence harness with an
  attached-not-created variant over a couple of body shapes).
- The detach round-trip: maintained → detach (rows intact, writable,
  maintenance stopped) → re-attach (reconciles, maintenance resumes).
- The edge cases above, one test each (sqllogic where expressible;
  spec tests for event/diff assertions).
- DDL round-trip: schema → canonical table-form DDL → parse → schema, for
  sugar-created and table-form-created maintained tables, with using /
  insert defaults / tags variants.
