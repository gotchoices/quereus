## Problem

`IsolatedTable` merges a per-connection overlay against the underlying table. To merge two
sorted streams it must know the sort order the underlying scan emits — primary-key order, or
`(secondaryIndexKey…, pk…)` order. It learns that by parsing `FilterInfo.idxStr`:

`packages/quereus-isolation/src/isolated-table.ts:548` — `parseIndexFromFilterInfo` splits the
string on `;` and `=`, pulls `idx=name(n)`, tests the name against
`PK_INDEX_NAME_RE = /^_primary_\d*$/`, and looks the rest up in `tableSchema.indexes` by name.

Three ways that goes wrong:

1. **Unparseable string ⇒ merge by primary key.** Any `idxStr` shape the splitter does not
   recognise returns `{ type: 'primary' }`. If the scan was really walking a secondary index,
   the merge interleaves overlay rows at positions computed under the wrong comparator. The
   query returns rows out of order — and the planner may have consumed an `ORDER BY` on the
   strength of that index, so nothing downstream re-sorts them.
2. **Name not in the schema ⇒ empty sort key.** `getIndexColumnIndices` returns `[]` for an
   unknown name. `buildMergeConfig` then builds a sort key of zero index columns, so
   `compareSortKey` compares only the primary key — against a stream physically ordered by the
   secondary index. Same corruption, reached from the other direction.
3. **The regexes are guesses about a foreign module's naming.** `lamina-quereus` aliases the
   primary-key index as `_primary_`, `_primary_1`, `_primary_2`, …; the two regexes hardcode
   "numeric suffix" as the alias shape. An alias of `_primary_a` classifies as a secondary
   index, hits case 2, and corrupts scan order.

`prereq: iso-index-descriptor-engine-seam` lands `FilterInfo.accessPath` — a typed
`AccessPath` union that says exactly which index the scan walks, whether it is the primary
key (regardless of what it is *named*), and what its full key columns are. This ticket makes
the isolation layer read that and delete the string parsing.

## Design

### Replace `parseIndexFromFilterInfo`

```ts
/**
 * Determine the order the underlying scan emits, from the planner's typed access path.
 *
 * A `fullScan` merges by primary key: every underlying module the isolation layer wraps
 * emits an unbounded scan in primary-key order. That is a contract on the underlying, not
 * an inference from the string — see docs/design-isolation-layer.md.
 */
private resolveScanIndex(filterInfo: FilterInfo): IndexScanInfo {
	const path = filterInfo.accessPath;
	if (!path) {
		throw new QuereusError(
			`IsolatedTable '${this.tableName}': FilterInfo carries no accessPath, so the ` +
			`underlying scan's sort order is unknown and the overlay cannot be merged. ` +
			`Build FilterInfo with the engine's makeFullScanFilterInfo/makeIndexEqSeekFilterInfo helpers.`,
			StatusCode.INTERNAL);
	}
	switch (path.kind) {
		case 'fullScan':
		case 'empty':
			return { type: 'primary' };
		case 'index':
			return path.index.role === 'primary'
				? { type: 'primary' }
				: {
					type: 'secondary',
					indexName: path.index.name,
					columnIndices: path.index.keyColumns.map(c => c.columnIndex),
				};
		case 'unresolvedIndex':
			throw new QuereusError(
				`IsolatedTable '${this.tableName}': the underlying module chose index ` +
				`'${path.indexName}', which the engine could not resolve against the table schema. ` +
				`A module that names an index anything other than '_primary_' or a schema index ` +
				`must return an 'indexDescriptor' from getBestAccessPlan (see docs/module-authoring.md).`,
				StatusCode.INTERNAL);
	}
}
```

Delete `PK_INDEX_NAME_RE`, `SUFFIXED_PK_IDXSTR_RE`, and `getIndexColumnIndices` (now dead).

**Why `unresolvedIndex` throws rather than falling back to a full scan.** The plan ticket
proposed "explicit, logged full-scan fallback". That is not available here: the merge is not
choosing a scan, it is choosing a *comparator* for a stream the underlying has already
decided to emit in index order. Downgrading to a primary-key comparator is exactly the silent
corruption this ticket exists to remove. Rewriting the underlying's own `FilterInfo` to a full
scan would be sound in isolation but discards an `orderByConsumed` the planner already
committed to upstream. So: fail loudly, with a message that names the module's obligation.
The engine already logs a warning at plan time when it emits `unresolvedIndex`, so the
diagnosis is available before the query runs.

### Replace `adaptFilterInfoForOverlay`

The overlay is a `MemoryTable` that always names its primary-key index `_primary_`. When the
underlying's plan walks the primary key under an alias, the overlay must be queried with the
canonical name — otherwise it tries to resolve a secondary index that does not exist.

```ts
private adaptFilterInfoForOverlay(filterInfo: FilterInfo): FilterInfo {
	const path = filterInfo.accessPath;
	if (path?.kind === 'index' && path.index.role === 'primary' && path.index.name !== '_primary_') {
		return retargetFilterInfoIndex(filterInfo, '_primary_');
	}
	return filterInfo;
}
```

`retargetFilterInfoIndex` (engine, from the prereq) rewrites `idxStr`,
`indexInfoOutput.idxStr`, and `accessPath.index.name` together, preserving `plan=`, the
`(n)` group, and every parameter it does not understand. The isolation layer no longer does
string surgery.

### `packages/quereus-isolation/src/filter-info.ts`

`makeFullScanFilterInfo` and `makePkPointLookupFilter` move to the engine in the prereq.
Delete the local literals; keep this file only if a thin `makePkPointLookupFilter(pkIndices, pk)`
wrapper (which needs `this.tableSchema` to build the PK descriptor) is still convenient —
otherwise call `makeIndexEqSeekFilterInfo(primaryKeyDescriptor(schema), pkIndices, pk)`
directly from `IsolatedTable.buildPKPointLookupFilter`.

Both helpers must now populate `accessPath`, or `resolveScanIndex` will throw on the
isolation layer's own internal queries (`getOverlayRow`, `getUnderlyingRow`, the commit flush,
and the ALTER/DROP INDEX overlay migrations in `isolation-module.ts` all go through them).

## Edge cases & interactions

- **Every internal `FilterInfo` the isolation package builds must carry `accessPath`.** Grep
  `indexInfoOutput:` across `packages/quereus-isolation/src` and the specs; the module's
  overlay-migration and commit-flush scans construct their own. Missing one turns a working
  commit into an INTERNAL error.
- **`analyze` over an isolated table.** `planner/stats/analyze.ts` hand-builds a full-scan
  `FilterInfo` and calls `table.query()`. If it runs against an isolated table with a dirty
  overlay it reaches `resolveScanIndex`. The prereq points it at `makeFullScanFilterInfo()`;
  verify with a test that `analyze` on an isolated table inside an open transaction succeeds.
- **`readCommitted` / no-overlay fast path.** `query()` returns `underlyingTable.query(filterInfo)`
  before `resolveScanIndex` is reached, so a missing `accessPath` must NOT break a clean read.
  Test: a hand-built `accessPath`-less FilterInfo reads fine with no overlay, throws with one.
- **Secondary index whose name is `_primary_extra`.** Must merge as a secondary index. The old
  `PK_INDEX_NAME_RE` was careful about this; the descriptor makes it structural. Keep the test.
- **Aliased primary key, both directions.** With a module aliasing to `_primary_1` and
  supplying a descriptor: merges by primary key, and the overlay query receives `_primary_`.
  With the alias but *no* descriptor: `query()` on a dirty overlay throws INTERNAL naming the
  index. Both need specs (the existing spec at `isolation-layer.spec.ts:2925` already stubs a
  suffixed-PK module — retarget it).
- **`plan: 'scan'` on a secondary index (ordered walk, no bounds).** Reaches
  `mergedSecondaryIndexQuery` exactly as an `eqSeek` does. The descriptor's `keyColumns` are
  the *full* index key even when only a prefix was seeked — that is what the sort key needs.
  A prefix-range seek (`plan=7`) still emits in full-index-key order.
- **`getIndexComparator(indexName)`.** Fed `path.index.name` — the module's own name, which is
  what it keys its comparators by. Do not pass a canonicalised name.
- **DESC index columns.** `IndexKeyColumn.desc` is now available but `buildCompareSortKey`
  still relies on `underlyingTable.getIndexComparator` to fold direction in, falling back to
  a direction-blind `compareSqlValues`. Do not change that here; it is a separate defect. If
  the fallback is exercised in a spec, note it.
- **Empty-plan merge.** `{ kind: 'empty' }` maps to `{ type: 'primary' }`, so a dirty overlay
  merged against an empty underlying stream yields the overlay's rows. Confirm the runtime
  never calls `query()` on an `EmptyResultNode`'s FilterInfo; if it does, an impossible
  predicate would surface overlay rows. Add a spec either way.
- **Multi-table commit / poisoned overlay.** Untouched, but the commit flush's full-scan
  FilterInfo runs through the new path — cover a commit of two tables in one transaction.

## Expected tests

Extend `packages/quereus-isolation/test/isolation-layer.spec.ts` (144 specs today — the plan
ticket's "~4 specs" claim is stale, so no separate test-broadening ticket is warranted):

- secondary-index scan with a dirty overlay returns rows in `(indexKey, pk)` order, including
  an overlay row whose indexed column was *updated* off the scanned value (tombstone revival
  + index-key change);
- an index literally named `_primary_extra` merges as a secondary index;
- aliased-PK module **with** `indexDescriptor` → PK merge, overlay queried as `_primary_`;
- aliased-PK module **without** `indexDescriptor` → `QuereusError` INTERNAL naming the index,
  raised on the dirty-overlay read, not on the clean read;
- `FilterInfo` with no `accessPath` → clean read OK, dirty read throws;
- `analyze` on an isolated table inside an open transaction;
- two-table commit still flushes both overlays.
