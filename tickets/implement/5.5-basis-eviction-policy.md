# Basis-table eviction policy & network-wide retirement signal (dynamic half)

Builds on `basis-lifecycle-classification` (the static three-state record). Adds
(1) the **dynamic** "is any peer still writing this table directly?" signal, and
(2) the **default eviction policy** that reclaims a legacy table's local storage
once it has been quiet for the retention horizon — with an override knob. This
realizes `docs/migration.md` § 4 Contract: *drop the legacy table no sooner than
`retentionHorizonMs` after the last directly-mapped write.*

## Dynamic signal: last directly-mapped write

The doc's question is "when did a change to this table last originate at a peer
that maps it directly?" A peer does not know remote peers' lens deployments, so
this is computed as a **conservative observation, not an oracle**:

> Once the local classification has this table in `derivation-source-only` /
> `unreferenced` / `detached` (i.e. the *local* peer no longer maps it
> directly), any **inbound** change to it from a **remote** site is presumed to
> be a direct write from a peer that still maps it. Each such change bumps
> `lastDirectlyMappedWriteAt = max(current, change wall-time)`.

This needs no remote-deployment knowledge: while *some* peer still writes the
legacy table directly, foreign writes keep arriving and keep resetting the
clock; when they cease network-wide for a full horizon, retirement is safe. It
uses only data the sync layer already has — inbound `Change` site ids + HLC
wall-times + the local lifecycle record.

Implementation: in `change-applicator.ts`, after a batch's data changes are
admitted, for each touched table that has a lifecycle record **not** in
`directly-mapped` state and whose change originated from a remote site, bump
`lastDirectlyMappedWriteAt` from the change's HLC wall-time. Keep it cheap: skip
entirely for tables with no record or in directly-mapped state (the common
case — zero overhead pre-migration). Batch the writes (one KV update per table
per apply, not per change).

Distinguishing a *direct* write from a *derivation/maintenance* write on the
legacy table: maintenance writes are value-derived and only enter the change log
when `quereus.sync.replicate=true`; for the conservative goal, treating any
remote write to a legacy table as "still in use" is the safe over-estimate
(errs toward retaining storage longer). Note this explicitly — under-counting
would be the dangerous direction (premature drop); over-counting only delays
reclamation.

## Eviction policy

Effective retention per table:

```
effectiveHorizonMs(record) =
  record.evictPolicy === 'never'      → ∞ (never evict)
  record.evictPolicy === 'immediate'  → 0
  typeof record.evictPolicy === number → that value
  otherwise                            → config.basisEviction.horizonMs
                                          ?? config.retentionHorizonMs

quietSince(record) = max(record.unmappedSince ?? detachedAt,
                         record.lastDirectlyMappedWriteAt ?? 0)

evictable(record, now) =
  record.state === 'detached'              // out of basis; storage lingering
  && record.evictPolicy !== 'never'
  && now - quietSince(record) >= effectiveHorizonMs(record)
```

- **Default eviction targets `detached` tables only** — a table the app removed
  from its basis schema (the § 4 Contract DDL), whose physical storage `lens.md`
  intentionally retains on detach. An **in-basis** `unreferenced` table is a
  *signal* (surfaced by the prereq), **never auto-dropped** — dropping it would
  diverge from the basis the app still declares and a re-map would resurrect it.
  Document this boundary; it is the load-bearing safety rule.
- `immediate` still requires `detached` (zero horizon, not "drop in-basis
  tables"). `never` opts a table out entirely.
- The retention horizon precedent is `sync-unknown-table-disposition`'s
  `retentionHorizonMs` (also used for tombstone + quarantine GC). A peer offline
  longer than the horizon was already outside the delivery guarantee, so its
  late writes to a reclaimed table fall under the existing unknown-table
  disposition (quarantine/ignore) — the two policies compose cleanly.

### The knob

Add to `SyncConfig` (`protocol.ts`) + `DEFAULT_SYNC_CONFIG`:

```ts
interface BasisEvictionConfig {
  /** 'horizon' (default) | 'never' | 'immediate' — global default mode. */
  mode: 'horizon' | 'never' | 'immediate';
  /** Override horizon for 'horizon' mode; defaults to retentionHorizonMs. */
  horizonMs?: number;
}
// SyncConfig.basisEviction?: BasisEvictionConfig   (default { mode: 'horizon' })
```

Per-table override via the reserved tag `quereus.sync.evict =
never | immediate | <ms>` on the basis table (sits alongside the existing
`quereus.sync.replicate` tag pattern). **Capture the tag into the lifecycle
record at observation time** (during `recordLensDeployment`, populate
`record.evictPolicy`), because once detached the table schema — and its tag — is
gone. Per-table override wins over the global mode.

### The sweep & the drop

- `evictExpiredBasisTables(now = Date.now()): Promise<number>` on `SyncManager`,
  host-driven exactly like `pruneTombstones` / `pruneQuarantine` (the host
  schedules it; do **not** add a timer inside the library). It iterates
  lifecycle records, selects `evictable`, and for each: re-checks the record is
  still `detached` (guard against a concurrent re-deploy that re-mapped it),
  calls `dropLocalTable(schema, table)`, emits `onBasisTableEvicted`, and removes
  the lifecycle record (or marks it `evicted` — pick one; removing is simpler and
  a later re-create starts fresh).
- `dropLocalTable` is a new callback (like `applyToStore` / `getTableSchema`),
  wired in the worker to a **store-side reclaim-by-name helper**. The store's
  `module.destroy` path and `provider.deleteTableStores(schema, table,
  indexNames)` already reclaim storage, but they need the index-name list, which
  is gone after detach. Resolve one of:
  - capture the index-name list into the lifecycle record at detach time
    (extend the prereq's record — coordinate the field), **or**
  - add `StoreModule.reclaimDetachedTable(schema, table)` that reclaims the data
    store + prefix-discovers index stores it still tracks for that key.
  Prefer the explicit captured-index-list approach (no prefix scanning — the
  store code already warns prefix scans can clobber sibling `*_idx_*` tables).

## Edge cases & interactions

- **Re-deploy resurrects mapping**: a `detached` table re-added to basis and
  re-mapped before the sweep runs must flip back to directly-mapped (prereq
  handles the reclassification) and the sweep's re-check must skip it. Test the
  race: schedule eligibility, then re-deploy, then sweep → no drop.
- **Foreign write resets the clock**: a remote direct write arriving during the
  horizon bumps `lastDirectlyMappedWriteAt`, pushing eligibility out. Test that
  a table is *not* evicted while foreign writes keep arriving, and *is* once they
  stop for a horizon.
- **Drop partial failure**: `dropLocalTable` throws → leave the record, log,
  retry next sweep (idempotent). Never leave a half-reclaimed table with the
  record removed.
- **`never` / `immediate` overrides**: exercise both via the reserved tag and the
  global mode; per-table tag wins.
- **Storage already gone**: if the app issued a real `drop table` (module
  `destroy` ran), `dropLocalTable` may find nothing — treat "already absent" as
  success (idempotent), still emit the event / clear the record.
- **In-basis unreferenced is never dropped**: assert a long-unreferenced but
  still-in-basis table is *not* evicted by the default or `immediate` policy.
- **Relay-only coordinator** (no `db`, no store): no deployments fire, no
  lifecycle records, sweep is a no-op. Verify it does not throw when wired
  without `dropLocalTable`.
- **Composition with unknown-table disposition**: after eviction, a straggler's
  inbound write to the reclaimed table is out-of-basis and follows
  `unknownTableDisposition` (quarantine/ignore) — no new handling needed; add a
  test asserting the two interact (evict → straggler write → quarantined).

## Key tests (TDD)

- Detach a table, advance clock past horizon, sweep → `dropLocalTable` called,
  `onBasisTableEvicted` fired, record cleared.
- Within horizon → not evicted.
- Foreign remote write bumps the clock and defers eviction; local-only /
  directly-mapped writes do not.
- `evictPolicy = 'never'` → never; `'immediate'` → evicted on first sweep after
  detach; custom ms honored; per-table tag overrides global mode.
- Re-deploy before sweep cancels eviction.
- `dropLocalTable` throws → record retained, retried.
- In-basis unreferenced table never auto-dropped.
- Evicted-then-straggler-write → quarantined per disposition.
