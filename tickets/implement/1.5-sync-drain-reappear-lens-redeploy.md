# Low-latency scoped drain when a lens redeploy re-maps a basis table back

## Context

`sync-drain-reappear-inbound-ddl` (prereq) added the `drainOnReappear` config flag (default
true) and the exported advisory helper `drainReappearedTables(ctx, tables)`, hooking the
inbound-`create_table` reappearance path inside `applyChanges`. This ticket hooks the second
reappearance path named in the original plan: a local **`apply schema` lens redeploy**
re-maps a basis table back into the basis.

`SyncManagerImpl.recordLensDeployment` (`sync-manager-impl.ts`) is the single place lens
deployments are recorded. It already classifies every affected basis-relation key per deploy,
computing `prior` (the stored record), `state` (new aggregate lifecycle state), `inBasis`,
and the `wasDetached`/`isDetached` transition. A table transitioning out of `detached` back
into an in-basis state is exactly "a retired table reappeared": its held out-of-basis changes
should drain now, not on the next periodic sweep.

## Design

### Candidate detection

Inside the existing per-key loop in `recordLensDeployment`, collect a `reappeared` list of
`{ schema, table }` for every key whose transition is **detached → present this deploy**:

```ts
if (prior?.state === 'detached' && state !== 'detached') {
  reappeared.push({ schema: record.schema, table: record.table });
}
```

Use `prior?.state === 'detached'` (a previously-tracked, previously-detached table coming
back) rather than the broader "`inBasis` newly true": held entries only ever exist for tables
that were out of basis, and the precise transition avoids spurious work for brand-new tables.
The oracle gate inside `drainHeldChanges` makes any over-trigger harmless regardless, so this
condition is a cheapness optimization, not a correctness gate.

### Mechanism — SEPARATE post-commit drain, advisory

`recordLensDeployment` ends with `if (wroteAny) await batch.write();` then emits the pending
`onBasisTableLifecycle` events AFTER the records are durable. Run the drain in the same "after
durable" position — a separate apply unit once the lifecycle batch has committed:

```ts
// ... after the pendingEvents emit loop:
await drainReappearedTables(this, reappeared);
```

`drainReappearedTables` is already advisory (early-returns on `!drainOnReappear` / empty;
logs + swallows per-table failures). `recordLensDeployment` is itself "advisory bookkeeping:
a throwing call must never abort the deploy" (the store-module forwarder wraps it in
try/catch), and the helper preserves that — a drain failure cannot abort the deploy.

This is symmetric with the inbound-DDL path: the re-creating change (here, the lens redeploy's
basis re-map) is fully committed before the held changes LWW-resolve against the now-present
table.

### No new flag, no new helper

Reuses `drainOnReappear` and `drainReappearedTables` from the prereq. `recordLensDeployment`
calls `drainHeldChanges` indirectly through the helper, which already calls
`drainHeldChangesImpl(this, schema, table)`-equivalent (`drainHeldChanges(ctx, ...)` in
`change-applicator.ts`). No worker changes — `onHeldChangesDrained` / `onRemoteChange` already
flow to the UI.

## Edge cases & interactions

- **Advisory.** A drain throw is swallowed by the helper; the deploy's lifecycle records are
  already durable and its `onBasisTableLifecycle` events already emitted. Held entries stay
  held; periodic sweep re-drains. Verify the deploy still completes when the drain throws.
- **Order: drain after lifecycle commit + events.** The basis re-map and lifecycle records
  are durable before the drain runs, so the oracle (`getTableColumnNames`) sees the
  re-attached table and the drain resolves against the present basis.
- **No transition / idempotent re-deploy.** A redeploy that maps the same tables (no
  `detached → present` transition) collects an empty `reappeared` list ⇒ no drain. A
  brand-new table (no `prior`) is excluded by `prior?.state === 'detached'`.
- **Re-map but still no held entries.** `quarantine.list(schema, table)` returns `[]` ⇒ 0;
  cheap scoped scan per reappeared table only.
- **Re-map but table still not resolvable by oracle.** Defensive: if the basis schema can't
  be resolved (the `basisSchema` not-found branch already warns), the oracle returns
  `undefined` and the drain is a no-op — entries stay held for the sweep.
- **`drainOnReappear: false`.** No drain fires from the redeploy path; defers to the periodic
  sweep. Covered by the helper's early return.
- **Concurrency / idempotency.** Same as the inbound path: a race with the periodic sweep or
  with an `applyChanges`-triggered drain re-resolves the same held changes (LWW-idempotent),
  second delete is a no-op. No new hazard.
- **Multiple tables reappear in one deploy.** Each drains independently (the helper iterates);
  one table's drain failure does not block the others.

## Key tests (`basis-lifecycle-recorder.spec.ts`, reusing the drain harness pattern)

These need both a basis oracle (`getTableColumnNames`) AND seeded quarantine entries — mirror
the seeding/oracle setup from `unknown-table-disposition.spec.ts`'s drain block.

- **Detached → re-mapped triggers drain.** Record a deploy that detaches a table (prior state
  becomes `detached`); seed held entries for it; record a later deploy that re-maps it; assert
  the held entries drained, `onHeldChangesDrained` fired once with `applied + skipped ===
  drained`, WITHOUT an explicit `drainHeldChanges`.
- **Idempotent re-deploy does not drain.** A deploy with no `detached → present` transition
  fires no drained event; held entries (if any seeded for a still-detached table) survive.
- **Brand-new table does not drain.** First-ever classification (no `prior`) with held
  entries elsewhere fires no drain for the new table.
- **`drainOnReappear: false` defers to the sweep.** Detach + seed + re-map with the flag off;
  held entries survive the deploy; a later explicit `drainHeldChanges()` clears them.
- **Drain failure is swallowed.** With the harness failure hook armed for the re-mapped
  table, the deploy completes (lifecycle records written, `onBasisTableLifecycle` emitted),
  no drained event, entries stay held; a later drain succeeds without double-apply.
