description: When a query joins two tables by scanning the inner table once for every row of the outer table, the engine reconnects to that inner table from scratch on every pass; connect once per execution and reuse it instead.
prereq: runtime-amortize-prepared-statement-setup

A nested-loop join whose right (inner) side is **not** wrapped in a cache/materialize
node re-scans the inner relation once per outer row (`runtime/emit/join.ts`
`driveFromLeft`: `for await (rightRow of rightCallback(rctx))` inside the outer
`for await (leftRow of leftSource)`). `rightCallback` is an `emitCall` sub-program that
re-runs the inner sub-tree — including the scan leaf — on each outer row. The scan leaf
(`emitSeqScan`, `runtime/emit/scan.ts`) does `module.connect(...)` at generator start
(:76) and `disconnectVTable(...)` in its `finally` (:127). Result: one connect +
disconnect on the inner table **per outer row**.

Goal: connect the inner table's virtual-table instance once per statement execution,
reuse it across every re-scan, and disconnect once when execution ends.

## Design

Introduce a per-execution cache of connected `VirtualTable` instances, owned by the
`RuntimeContext`, keyed by **scan plan-node identity** (not by table name), torn down
at statement teardown.

### Why key by plan node, not by table

Keying by `(table, readCommitted)` is unsafe: a self-join scans the same table on both
sides, and the outer scan's cursor is still live while the inner re-scans run — two
**concurrent** `query()` calls on one shared instance, which a module may not support
(per-instance cursor state). Today each scan invocation gets its own fresh instance, so
concurrent scans are isolated.

A single scan **plan node** is never concurrently self-live: its single consumer pulls
one row at a time, and the NLJ inner re-creates the inner cursor sequentially (each
inner re-scan fully drains before the next outer row). So caching keyed by the scan
plan node gives per-scan-site reuse for the sequential re-scans we want, while two
distinct scan sites over the same table (the self-join case) still get distinct
instances and stay isolated. Give each `emitSeqScan` a stable per-node key (e.g. a
unique `Symbol()` minted in the emitter closure, or the plan node reference itself) and
use it as the map key.

### Lifecycle

- Add an optional cache to `RuntimeContext` (`runtime/types.ts`), e.g.
  `scanConnections?: Map<symbol, VirtualTable>`, created in
  `Statement._iterateRowsRawInternal` alongside the other runtime-context fields.
- In `emitSeqScan`'s `run`: look up this scan's key. Hit -> reuse the instance and skip
  connect. Miss -> `module.connect(...)` (unchanged args, including the `_readCommitted`
  option) and store it. Either way, **do not** disconnect in the scan's `finally`; still
  `rowSlot.close()` there (the row slot is per-invocation and must close each pass).
- Teardown: in `_iterateRowsRawInternal`'s `finally`, disconnect every cached instance
  (via `disconnectVTable`) and clear the map. That `finally` runs after the consumer
  finishes draining (normal completion, `break`, error, or abort), so instances stay
  live for the whole streaming window and are released exactly once.

### Semantics note (why reuse is safe here)

The memory vtab reads live-at-`query()`-time (each `query()` reads the current
layer/tree), so a reused instance's later `query()` observes the same state a fresh
connect would — **no visibility change**. For `_readCommitted` instances that snapshot
at connect, holding one snapshot for the whole statement execution is at least as
correct as re-snapshotting per outer row (a committed snapshot should be stable within
one execution). The scan read path connects `module.connect` **directly** and never
registers a `VirtualTableConnection`, so this work does not touch the
`adoptConnection` / connection-registration path added by
`runtime-adopt-connection-hook` (already complete). Routing the read scan through
`getVTable`/`adoptConnection` is a larger, separate change — out of scope here; note it
as a backlog candidate if the read path ever needs registered-connection reuse.

## Edge cases & interactions

- **Self-join / same table on both NLJ sides.** Outer scan and inner scan are distinct
  plan nodes -> distinct cached instances -> no concurrent-cursor sharing. Add a
  self-join logic test and assert correct rows.
- **Cached/materialized inner.** When the optimizer wraps the right side in a cache
  node the inner is materialized once and never re-scans; the scan runs once, caches its
  instance, and it is disconnected at teardown. Confirm no double-disconnect and no
  behavior change for the cached path.
- **Repeated scans of one site are sequential, never concurrent.** Verify the inner
  cursor is fully drained each pass before the next `query()` on the reused instance
  (it is, by NLJ structure) — a module relying on one-cursor-per-instance must not see
  overlapping iterations.
- **Abort / error mid-scan.** An `AbortError` or query error must still unwind and
  disconnect all cached instances exactly once via the teardown `finally`. Preserve the
  existing `AbortError` identity handling in the scan's catch. Verify no instance leak
  and no double-disconnect on the error path.
- **IndexSeek dynamic args.** The seek keys differ per re-scan (per outer row), but they
  flow through `query(effectiveFilterInfo)` each pass, not through `connect`. Reusing
  the connected instance while calling `query` with fresh args per pass is the intended
  shape — confirm correlated IndexSeek joins still return correct rows.
- **DML with a correlated read of a table being mutated in the same statement.** If any
  path connects fresh per row to observe intra-statement writes, reuse could change what
  it sees. Add a read-after-write-within-one-statement test; if the memory vtab shows a
  difference (it should not — live-at-query-time), gate reuse behind a module capability
  rather than shipping a silent semantic change.
- **`disconnectVTable` on an instance that was reused N times.** Called once at
  teardown; confirm idempotent-enough (it only calls `vtab.disconnect()` if present and
  swallows/logs disconnect errors). No per-pass disconnect remains.
- **Non-scan capturing instructions.** Only the scan leaf owns connect/disconnect; CTE /
  envelope / TVF emitters do not connect vtabs, so they are unaffected.
