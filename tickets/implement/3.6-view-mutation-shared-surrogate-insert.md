## Why this is its own ticket

`view-mutation-multisource-innerjoin` (3.2) shipped multi-source **update / delete** but
**deferred insert**, because insert through a key-hiding join is blocked on a genuinely new
runtime surface that update/delete do not need:

- Update/delete address **existing** rows by a subquery over the join — no shared key is minted.
- Insert must **create** rows in two base tables that share a join key the view does not expose.
  The key has to come from somewhere; the design's answer (§ Mutation Context) is a **per-row
  generated default on the anchor**, captured once and threaded.

The implement-stage investigation for 3.2 confirmed: mutation context today is **per
`DmlExecutorNode`, evaluated once per statement** (`runtime/emit/dml-executor.ts`
`evaluateContextRow`, called before the row loop). There is **no** mechanism for a value resolved
**per row** and **shared across sibling base ops**. That envelope is what this ticket builds.

## Required behavior

### The shared-surrogate envelope (load-bearing)
A per-row generated value that also serves as a join key shared across base tables is resolved
**once per produced logical row at the envelope, before propagation reaches the branches**, and
the single captured value is referenced by every branch. There is no "which branch generates
first" question. Two cadences (per `docs/view-updateability.md` § Mutation Context):

- **per-statement** (a captured `now`, a bound param): resolved once, stamped on every row and
  every base op — already rides the existing statement-level `mutationContextValues` /
  `evaluateContextRow` seam.
- **per-row** (a sequence / surrogate allocator like `next_rid()`): resolved once **per produced
  row**, distinct per row, identical across that row's base ops. This is the new surface.

Decide and document the mechanism (two candidate designs surfaced in the 3.2 investigation):
1. **Pre-evaluate at the envelope**: the `ViewMutationNode` (or a wrapping context node) evaluates
   the shared generator per row and binds the captured value into each base insert's value list /
   context, so neither base op re-evaluates the default.
2. **Capture-and-thread via RETURNING**: the anchor (FK-parent) insert generates the key via its
   own default and surfaces it (RETURNING-style), and the child insert consumes it. (Couples to
   `view-mutation-returning-through-view`; pick deliberately.)
The captured per-row values must be recorded so replay re-emits byte-identical base rows
(deterministic-given-context).

### Worked example (must pass as sqllogic — § Mutation Context)
```sql
create table u_core    (rid int primary key default next_rid(), name text) using mem();
create table u_contact (rid int primary key, email text) using mem();
create view User as
  select c.name, k.email from u_core c left join u_contact k on k.rid = c.rid;
insert into User (name, email) values ('Ada', 'ada@x.io'), ('Lin', 'lin@x.io');
-- one next_rid() per row, shared across both base inserts of that row:
--   u_core(rid=1001,name='Ada') + u_contact(rid=1001,email='ada@x.io'), then the 1002 pair.
```
Assert exactly one generation shared across both base inserts per row, distinct across rows.
(Note this example is a LEFT join — coordinate with the outer-join insert semantics in
§ Outer Joins, or restrict v1 to the inner-join + optional-member shape and document.)

### Fan-out + defaults
- Emit one base insert per side; FK-parent before FK-child (3.2 already orders `BaseOp[]`).
- Per-side omitted-column fill follows § Projection: value list (post-inverse) → constant-FD
  default → FD reconstruction → EC propagation → base column default (incl. generated/context
  default) → null; a `not null` column with no value → `no-default` diagnostic.
- The join-key EC supplies the shared key to the side that does not generate it.

## Acceptance
- Multi-source inner-join insert writes through with FK / conflict / constraint parity to
  hand-written base DML; the shared-surrogate worked example passes with one generation per row.
- `unsupported-multisource-insert` no longer raised for the supported shape.
- Dynamic PutGet over the planned join tree covers insert (extend the `multi-source inner join`
  block in `test/property.spec.ts`).
- `lens-multi-source-put-fanout` can consume the per-row/per-statement envelope (coordinate the
  surface so it threads, not re-invents).
