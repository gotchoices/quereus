description: **RETURNING through a view** (single- and multi-source). `insert` / `update` / `delete` through a view with a `returning` clause returns rows projected through the **view's** column list (not the base tables'), evaluated against the **post-mutation** base state — computed-lineage columns re-evaluated. Today RETURNING-through-view is rejected everywhere with the `returning-through-view` diagnostic (single-source `single-source.ts`, multi-source `multi-source.ts`). This ticket flips it to supported: promote `ViewMutationNode` to a **relational** node, wire its reserved `returning` slot through `getChildren`/`withChildren`/`getRelations`/`getType`, capture the projected rows in the emitter, and revisit the `block.ts` result-shadowing exclusion so a RETURNING view mutation surfaces its rows (a non-RETURNING one still yields nothing). Design source: `docs/view-updateability.md` § `returning` Clauses.

## Current state

- `ViewMutationNode` (`planner/nodes/view-mutation-node.ts`) is a **scalar** side-effect node
  (`getType()` → `scalar`, like `SinkNode`) with a **reserved but unused** `returning?:
  RelationalPlanNode` constructor slot. The emitter (`runtime/emit/view-mutation.ts`) drives the
  base ops sequentially and yields nothing.
- `block.ts` result selection (`emitBlock`) explicitly **excludes** `ViewMutation` (alongside
  `Sink`) from being a block's result value — correct while view mutations are void, but it must
  be revisited so a RETURNING view mutation is *not* excluded.
- RETURNING is rejected at decomposition time: `single-source.ts rejectReturning` and
  `multi-source.ts rejectReturning` both raise `returning-through-view` when `stmt.returning` is
  non-empty.

## Required behavior

- The returned rows are projected through the **view's** column list. The engine evaluates the
  view body against the **post-mutation** base state to produce them — equivalently, against the
  captured per-op results, since the view's lineage maps base rows back to view rows. Computed
  (`computed`-lineage) view columns are evaluated against the post-mutation base values.
- Works for single-source (the common case) and for the multi-source inner-join update/delete that
  3.2 shipped. (Multi-source insert RETURNING coordinates with `view-mutation-shared-surrogate-insert`
  — the post-insert key must be resolved; if that ticket has not landed, restrict multi-source
  insert RETURNING with a clear diagnostic.)
- A view mutation **without** RETURNING still yields nothing and is still excluded from block
  result selection (no regression to the 3.1/3.2 void behavior).

## Design sketch (adapt to the actual substrate surface)

- Promote `ViewMutationNode` to relational when a `returning` projection is present: `getType()`
  returns the view's row type; `getRelations()`/`getChildren()`/`withChildren()` thread the
  `returning` child; keep the scalar/void shape when absent (or always-relational with an empty
  output — decide and document).
- Build the `returning` projection in `view-mutation-builder.ts` from the view's column list over a
  post-mutation read of the affected rows (reuse the base ops' RETURNING / change capture where
  possible rather than re-querying the whole view).
- Emitter: collect the projected rows and yield them; the sequential base-op driver already exists.
- `block.ts`: only exclude a `ViewMutation` from result selection when it is void (no `returning`).

## Acceptance
- `insert`/`update`/`delete ... returning <view cols>` through a single-source view returns the
  view-projected post-mutation rows (incl. a computed view column).
- The same through a multi-source inner-join view (update/delete) returns view-projected rows.
- `returning-through-view` no longer raised for supported shapes; non-RETURNING view mutations
  unchanged.
- sqllogic cases in `93.4-view-mutation.sqllogic`; docs § `returning` Clauses flipped to shipped.
