description: Widen the grammar so a parenthesized query expression `( <query-expr> )` is a relational operand wherever a set-operation leg, view body, or top-level query is expected — making `(A union B) union (C union D)` (and `(select 1) union (select 2)`) parseable. Unify the left/right compound-leg paths onto one operand production and one left-associative tail parser. No new AST node; round-trips for free.
files: packages/quereus/src/parser/parser.ts, packages/quereus/src/parser/ast.ts, packages/quereus/src/emit/ast-stringify.ts, packages/quereus/test/emit-roundtrip.spec.ts, packages/quereus/test/property.spec.ts, docs/sql.md
----

## Background

A set operation's **right** leg already accepts a leading `(` (the `usedParen` path in
`selectStatement` and `continueSelectAfterFrom`), but the **left** leg must be a bare query
keyword: both `parseQueryExpr` (`parser.ts:230`) and `statement()` (`parser.ts:322`) switch on
the first token's keyword and a leading `(` falls through to the default error branch. Net: the
grammar can express `A union (B union C)` (right-nested) but not `(A union B) union (C union D)`
(parallel siblings) nor `(select 1) union (select 2)`.

The widening is **approved** (SQLite parity). This ticket lands the parser change plus the
round-trip / behavioural tests. The design below is settled — implement it as written.

## Target grammar

```
query-expr  := query-term { compound-op [membership] query-term }      // LEFT-associative
query-term  := [WITH …] ( SELECT… | VALUES… | INSERT|UPDATE|DELETE… ) | '(' query-expr ')'
compound-op := union [all] | intersect | except | diff
```

i.e. a *query-term* gains a new alternative — a parenthesized query expression — and both legs
of a compound draw from the same `query-term` production.

## Representation — subquery wrapper, no new AST node

The AST keeps `SelectStmt.compound = { op, select: QueryExpr, existence? }` (a right-leaning
linked list). The left operand of a compound is the `SelectStmt`'s own fields; the right operand
is `compound.select` (already a full `QueryExpr`, so it can itself be a compound — that is why
right-nesting already works).

A parenthesized operand parses to its **inner `QueryExpr`**. Two cases:

- **Pure grouping** (no compound operator follows the `)`): return the inner `QueryExpr`
  unchanged. So `(select 1)` ≡ `select 1`; `((A union B))` ≡ `A union B`. No wrapper.
- **Left operand of an outer compound:** if the accumulator it must attach to already carries a
  `.compound` (or is not a plain `SelectStmt`), wrap it as a synthetic
  `select * from ( <inner> ) as <alias>` and hang the outer `.compound` on the wrapper. This is
  the **exact pattern already used for VALUES** in `valuesStatementWithOptionalCompound`
  (`parser.ts:2324`, `syntheticAlias = values_${startOffset}`) — reuse it, do not invent a node.

A right operand needs **no** wrapper — it drops straight into the `compound.select` slot, which is
a `QueryExpr` and may itself be a (parenthesized) compound.

Worked example — `(A union B) union (C union D)`:
- left `(A union B)` → inner is `A` with `.compound={union, B}` → accumulator already has a
  compound → wrap once → `select * from (A union B)`.
- outer `union`, right `(C union D)` → inner `C` with `.compound={union, D}` → used directly.
- result AST: `SS_from(A∪B){ compound:{union, select: C{compound:{union, D}}} }`.
- `buildCompoundSelect` (`planner/building/select-compound.ts:33`) strips the outer compound for
  the left core (`select * from (A∪B)` → `SetOp(union, A, B)`) and builds the right (`C∪D` →
  `SetOp(union, C, D)`), yielding the balanced **left-leaning** plan
  `SetOp(union, SetOp(A,B), SetOp(C,D))`. Exactly one wrapper, only where structurally required.

## Associativity

Implement a single **left-associative tail parser** `parseCompoundTail(leftSel, withClause,
isCompoundSubquery)` that all three compound sites delegate to:

```
while (a compound operator follows) {
  { op, existence } = parseCompoundOperator()            // decode keyword + setOpMembershipClauses
  if (next token is '(') {
     right = parseCompoundOperand(withClause)             // single parenthesized operand (bounded by parens)
     if (leftSel.compound) leftSel = wrapAsSubquerySelect(leftSel)   // accumulator slot taken → wrap
     leftSel.compound = existence ? {op, select: right, existence} : {op, select: right}
     continue                                             // loop continues — left-associative
  } else {
     right = parseBareOperand(withClause)                 // SELECT/VALUES/DML, isCompoundSubquery=true → GREEDY
     leftSel.compound = existence ? {op, select: right, existence} : {op, select: right}
     break                                                // bare operand consumed the remaining chain itself
  }
}
```

This is the crux that **preserves every existing shape** while adding the new capability:

- **All-bare chain** `A union B union C` → first iteration takes the bare branch; the bare
  operand (`selectStatement(…, isCompoundSubquery=true)` → itself calls `parseCompoundTail`)
  greedily swallows `union C`, then we `break`. Result is the current **right-leaning**
  `A{union, B{union, C}}` — byte-identical to today. Zero blast radius on the builder, the
  set-op membership write path (`planner/mutation/set-op.ts` `leftBranchSelect`/`rightBranchSelect`),
  and existing tests.
- **Parenthesized operands** chain **left-associatively** via the loop, wrapping only when the
  accumulator already has a compound. `(A) union (B) union (C)` → `SS_from(A∪B){union, C}`.
- **Mixes** resolve predictably: a parenthesized operand iterates the loop; the first **bare**
  operand greedily consumes the rest and breaks. `(A) union B union C` → `A{union, B∪C}`;
  `(A) union (B) union C` → `SS_from(A∪B){union, C}`. The user's explicit parens are honoured;
  unparenthesized runs keep today's behaviour.

> Note (pre-existing, out of scope): unparenthesized mixed-operator chains
> (`A except B union C`) remain right-leaning, i.e. `A except (B union C)` rather than SQL's
> left-associative `(A except B) union C`. This ticket does not change that — parentheses are the
> escape hatch and now work. If we later want SQL-standard left-assoc for bare mixed chains, file
> a separate ticket (it changes the AST shape of existing chains and must re-validate the set-op
> membership write path).

## Operand & helper production

Add `parseCompoundOperand(withClause, requireReturning)` — the single `query-term` reader used by
the tail parser and by the new entry points:

```
if (check LPAREN) {
   advance();                                            // (
   inner = parseQueryExpr(withClause, requireReturning); // full recursion: WITH, compound, nested parens,
                                                          // and the inner's OWN trailing ORDER BY/LIMIT
   consume RPAREN;
   return inner;
}
// bare keyword-led single operand — isCompoundSubquery=true so it does NOT eat the OUTER's order/limit
start = peek();
if (check SELECT) { advance(); return selectStatement(start, withClause, /*isCompoundSubquery*/ true); }
if (check VALUES) { advance(); return valuesStatementWithOptionalCompound(start, withClause, true); }
if (check WITH|INSERT|UPDATE|DELETE) return parseQueryExpr(undefined, /*requireReturning*/ true);
throw error("Expected a query (SELECT/VALUES/DML) or '(' …");
```

Replace the duplicated `usedParen` blocks in `selectStatement` (`~parser.ts:664-697`) and
`continueSelectAfterFrom` (`~parser.ts:2376-2402`) with calls through this operand reader inside
`parseCompoundTail`. **DRY extractions** (AGENTS.md "Stay DRY"):

- `parseCompoundOperator(): { op, existence }` — the op-keyword decode (`union`/`union all`/
  `intersect`/`except`/`diff`) plus the existing `setOpMembershipClauses(op)` call
  (`parser.ts:1232`). Currently copy-pasted at both compound sites.
- `parseTrailingOrderLimit(sel)` — the ORDER BY / LIMIT / OFFSET tail currently duplicated in
  `selectStatement` (`~parser.ts:700-751`) and `continueSelectAfterFrom` (`~parser.ts:2406-2434`).
- `wrapAsSubquerySelect(inner, startToken): SelectStmt` — the `select * from (inner) as
  values_<offset>` wrapper, generalized from `valuesStatementWithOptionalCompound`.

A note on the right-leg via parseCompoundOperand: a parenthesized right leg now recurses through
`parseQueryExpr` (`isCompoundSubquery=false` *inside* the parens), so a parenthesized leg may
carry its **own** ORDER BY / LIMIT (`A union (select 2 order by x limit 1)`) — previously the
`usedParen` path used `isCompoundSubquery=true` and rejected that. This is a strict improvement;
add a test pinning it. Bare right legs keep `isCompoundSubquery=true` (their order/limit bind to
the outer compound, unchanged).

## Entry points

- **`parseQueryExpr` (`parser.ts:230`)** — before the keyword switch, after the inner-WITH
  consumption, add: `if (check LPAREN) stmt = parseCompoundOperand(resolutionContext,
  requireReturning)` then run the existing chain via `parseCompoundTail` so a parenthesized first
  operand can be followed by `union …`. (The keyword branches already terminate in
  `selectStatement`/`valuesStatementWithOptionalCompound`, which themselves use
  `parseCompoundTail`.) Because **view body** (`createViewStatement`, `parser.ts:2671`), **CTE
  body** (`commonTableExpression`, `parser.ts:312`), **FROM subquery**, and **scalar/IN/EXISTS
  subquery** all funnel through `parseQueryExpr`, they inherit the capability for free.
- **`statement()` (`parser.ts:322`)** — intercept a leading `(` *before* the keyword switch
  (after the top-level WITH is consumed): `if (check LPAREN) stmt = parseQueryExpr(withClause)`.
  The existing post-switch WITH-attach handles `with … (select …) union …`. This delivers the
  desired-behaviour item "and a top-level query" (`(select 1) union (select 2);` — SQLite parity).

## Stringify — no change required

`subquerySource` already emits `( <subquery> ) as <alias>` (`ast-stringify.ts:650-660`) and the
compound stringifier already dispatches `compound.select` through `astToString`
(`ast-stringify.ts:577-590`). Therefore:
- a wrapped left compound re-emits as `select * from ( … ) …` (parens carried structurally), and
- a simple unwrapped `(select 1) union (select 2)` re-emits as `select 1 union select 2`
  (identical AST to the plain compound — the redundant parens are intentionally dropped).

Both satisfy the round-trip **idempotence** contract (`parse → stringify → parse → stringify`
stable; `emit-roundtrip.spec.ts:7`). Confirm no stringify edit is needed by the round-trip tests
below; only touch `ast-stringify.ts` if a case proves otherwise.

## Edge cases & interactions

- **Simple parens** `(select 1) union (select 2)` → unwrapped `select 1 union select 2` AST.
- **Parallel siblings** `(A union B) union (C union D)` → one left wrapper; balanced
  `SetOp(union, SetOp(A,B), SetOp(C,D))`.
- **n-ary parenthesized chain** `(A) union (B) union (C)` → left-leaning, one wrapper.
- **Deep nesting** `((A union B)) union C`, `((A∪B) union C) intersect D`.
- **Mixed operators** across a parenthesized chain (`(A) intersect (B) except (C)`); membership
  attaches to the **outer** op (`(A∪B) union exists left as inX, exists right as inY (A∪B)`); an
  inner parenthesized compound carries its **own** `compound.existence`.
- **Parenthesized leg as a view body** (`create view v as (select 1) union (select 2)`) — the
  primary path for `set-op-membership-nested`. Confirm `planViewBody`/`buildSelectStmt` accept the
  wrapper (it is an ordinary `select * from (subquery)`).
- **Inner `WITH`** inside the parens: `(with t as (…) select * from t) union select …`. The inner
  WITH is owned by the inner statement; an outer top-level WITH still resolves into both operands.
- **`VALUES` legs and DML-with-`RETURNING` legs inside parens** — flow through the same
  `parseQueryExpr` recursion; `requireReturning` is propagated, so `(insert … )` without RETURNING
  in a leg position is still rejected, and `(insert … returning *)` is accepted.
- **Parenthesized leg with its own ORDER BY/LIMIT** (`A union (select 2 order by 1 limit 1)`) —
  now binds inside the parens (improvement). Bare `A union select 2 order by 1` still binds
  ORDER BY to the outer compound.
- **Empty parens / malformed** `()`, `(union select 1)` → the inner `parseQueryExpr` raises the
  existing "Expected SELECT, VALUES, …" error; ensure the message is sensible and `loc` points
  inside the parens.
- **Lens override body** (`declareLensStatement`, `parser.ts:3361`) rejects compound bodies via
  `if (body.compound || body.union)`. A wrapped parenthesized compound `(A∪B)` used as a lens body
  becomes `select * from (A∪B)` whose **top level has no compound**, so it would slip past that
  guard. Decide: either (a) leave as-is — `select * from (compound)` is a legitimate single-SELECT
  body the lens merger can handle, or (b) tighten the lens guard to also reject a single
  subquery-source whose subquery is a compound. Default to **(a)** (do not expand lens scope here);
  add a test asserting the lens path does not crash, and note (b) as a follow-up only if it throws.
- **Set-op membership write (forward interaction, not in scope):** the wrapper makes the write
  branches (`leftBranchSelect`) a `select * from (compound)` rather than a bare compound. Making
  those **writable** is the backlog ticket `set-op-membership-nested`; this ticket only enables
  the parse. Add a *parse + read-plan* test for `(A∪B) union exists … (C∪D)` and confirm the
  existing single-level membership write tests are untouched.

## Tests

Key cases (extend `emit-roundtrip.spec.ts` "compound" describe and `property.spec.ts`):

- **Round-trip / idempotence** (`roundTripStmt`):
  - `(select 1) union (select 2)` → stable (re-emits as `select 1 union select 2`).
  - `select a from t1 union select b from t2` (regression — unchanged shape).
  - `(select a from t1 union select b from t2) union (select c from t3 union select d from t4)`
    → stable; second emit equals first.
  - `(select 1) intersect (select 2) except (select 3)` → stable.
  - parenthesized leg view body via `create view` round-trip.
  - membership on the outer op: `(select id,x from a union select id,x from b) union exists left
    as inL, exists right as inR (select id,x from a union select id,x from b)` → stable.
- **Parse-shape assertions** (`new Parser().parse(...)`):
  - `(select 1) union (select 2)` parses (no throw) and yields a single-level compound (no wrapper
    in the simple case).
  - `(A union B) union (C union D)` yields a left wrapper on the left operand and a nested compound
    on the right; building it produces `SetOp(union, SetOp, SetOp)` (assert via `_buildPlan` /
    `findSetOp`, mirroring `property.spec.ts:2186`).
  - `select 1 union (select 2 union select 3)` still parses (regression for the right-nested case).
  - `(select 1)` ≡ `select 1` (pure grouping unwrap).
  - `A union (select 2 order by 1 limit 1)` — inner order/limit bind inside the parens.
- **Negative:** `()` and `(union select 1)` throw with a clear message; `(insert into t values
  (1))` (no RETURNING) in a leg position throws the RETURNING-required error.
- **Remove the stale limitation comment** at `property.spec.ts:2173`-area documenting that
  parenthesized left legs are unparseable, and replace with a positive test.

  AGENTS.md. Spot-check any `.sqllogic` set-op fixtures still pass.
