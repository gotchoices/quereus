## What this is

`docs/view-updateability.md` is a **design doc**, so the split (`docs-vu-split`) is only
half the job: the normative statements — the things the *code* must satisfy, where a
violation is a bug — belong in `docs/invariants.md`, the single register read end-to-end
against the implementation a few times a year. This ticket adds the `VU` area. It mirrors
the completed `docs-invariants-mv` and `docs-invariants-optimizer` tickets exactly; copy
the shape of an existing `MV-*` entry rather than working from a restated grammar.

View updateability is the **strongest invariant candidate of any doc in the repo**: its
round-trip laws already have a property-test guard (`test/property.spec.ts § View Round-Trip
Laws`), so the `guard:` lines point at real, passing tests rather than `none — <reason>`.

## Candidate invariants

These are the load-bearing normative claims the doc makes. Finalize each against the code
(the register's rule: states a property the code upholds, violating it is a bug, fits in
120 words without its topic doc, names a concrete code site, carries exactly one `guard:`).
Numbers ascend within the area; gaps are fine. Merge or drop any that turn out to be
rationale wearing an invariant's clothes.

- **View-targeted DML rewrites to base tables and re-plans.** An `insert`/`update`/`delete`
  on a view/CTE/subquery-in-`from` is rewritten to target the underlying base table(s) and
  re-planned through the ordinary base-table builder — there is no view-level updateability
  flag, no `instead of` trigger, no `with check option`. code:
  `planner/building/view-mutation-builder.ts`. doc: overview § Philosophy / Update Site Model.
- **Updateable iff a deterministic decomposition exists at plan time.** Otherwise a
  structured diagnostic (`no-inverse` / `predicate-contradiction` / `recursive-cte` / …)
  names the obstruction; the write is never silently dropped or silently widened. code:
  `planner/mutation/propagate.ts`, `planner/mutation/mutation-diagnostic.ts`. doc:
  overview § Mutation Propagation, § Diagnostics.
- **PutGet — a write through a view never escapes the view predicate.** Base rows outside
  the predicate are untouched; the post-state view image agrees with the base. guard:
  `test/property.spec.ts § View Round-Trip Laws`. doc: `vu-roundtrip.md`.
- **GetPut — read-a-row-and-write-it-back is a no-op on the base.** Writing the same values
  back via the identifying predicate leaves the base unchanged. guard: same. doc:
  `vu-roundtrip.md`.
- **Forward/backward lineage agreement.** Every forward key (`keysOf` / `isUnique`) is
  `base`-writable and reconstructs the base identifying predicate; a base PK that survives
  projection is advertised as a forward key. code:
  `planner/analysis/update-lineage.ts`. guard: `test/property.spec.ts § View Round-Trip Laws`.
  doc: `vu-roundtrip.md`.
- **A computed / non-invertible column is read-only.** A write reds with the `no-inverse`
  diagnostic (never silently dropped) — *unless* the result column carries an authored
  inverse (`with inverse`), which upgrades it to writable. code:
  `planner/analysis/scalar-invertibility.ts`, `planner/analysis/authored-inverse.ts`. doc:
  `vu-inverses.md`.
- **Evaluate-once-and-thread for a shared/surrogate key.** The anchor key column's default
  is evaluated once per produced row at the mutation envelope and threaded into every
  member of the decomposition's fan-out via the equivalence class, so all members agree on
  identity. code: `planner/mutation/decomposition.ts`, `planner/mutation/multi-source.ts`.
  doc: `vu-mutation-context.md`.
- **LIMIT/OFFSET/DISTINCT bodies reject rather than widen.** A view body outside the
  supported fragment reds a diagnostic rather than mutating a widened row set. guard:
  `test/property.spec.ts § View Round-Trip Laws` (asserts these reject). doc: `vu-operators.md`.

## Edge cases & interactions

- **Guard must name a section that exists.** `test/property.spec.ts § View Round-Trip Laws`
  is the section title in the spec file; if the extraction renames it, keep the `guard:`
  marker in sync. The checker validates the *file* path and, where the line ends in a
  `` `symbol` ``, that the symbol appears — not the `§` marker — so also confirm by hand
  the section still exists.
- **120-word ceiling.** The round-trip laws are stated together in the doc; split them into
  one invariant each (PutGet, GetPut, lineage-agreement) rather than one 300-word entry —
  "an invariant you cannot state in 120 words is two invariants."
- **Ascending IDs, no reuse.** If you drop a candidate mid-drafting, leave its number a gap
  rather than renumbering — the register's rule.
- **Rationale is not an invariant.** "The FD framework subsumes the dialect-specific rules
  of other engines" is rationale — it stays in the doc, not the register. Only claims a
  single code read can falsify earn an entry.
- **Prereq is the split, not the src-repoint.** This ticket depends only on `docs-vu-split`
  (the satellites must exist to link into and to name in `doc:` lines). It is independent of
  `docs-vu-repoint-src`; the two can land in either order.
- **Self-guarding vs test-guarded.** The rewrite-to-base and diagnostic invariants may have
  no property test — use `guard: none — <reason>` (a bare `guard: none` fails the checker),
  or point at the relevant `test/logic/*.sqllogic` case if one exercises the rejection.
