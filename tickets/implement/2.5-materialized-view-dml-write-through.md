## What it does

DML whose target *names* a materialized view is rewritten to target the MV's
source table `T` and re-planned through the ordinary base-table builder — the
identical AST-level rewrite plain-view mutation already performs
(`building/view-mutation.ts`, `rewriteViewInsert` / `rewriteViewUpdate` /
`rewriteViewDelete`). The rewritten write hits `T`, which fires the existing
row-time maintenance hook (`maintainRowTimeStructures`, called from
`runtime/emit/dml-executor.ts` at the six `_recordInsert/_recordUpdate/_recordDelete`
sites). The backing table is brought into sync **inside the same statement /
transaction**, so a subsequent `select … from m` sees the write (reads-own-writes)
and a rollback reverts source + backing in lockstep.

This is the **write direction** of the "MV ≡ faster view" contract the row-time
consolidation established in the read direction. It is distinct from the
already-shipped `materialized-view-rowtime-write-through` ticket, which is *source*
→ backing maintenance; this ticket is *MV-name DML* → source.

## Why this is now small

The `materialized-view-rowtime-only-consolidation` work (shipped, in `complete/`)
narrows every MV at the create gate (`database-materialized-views.ts`
`buildRowTimePlan`) to: a single keyed source `T`, a row-preserving linear body
(`TableReference → optional Filter → Project → optional Sort`), projecting every
source PK column, optional single-row equality `WHERE`. That is a **strict subset**
of what the view-mutation classifier (`planner/mutation/propagate.ts`
`classifyViewBody`) accepts as a single-source projection-and-filter. So an MV body
that *exists* is already guaranteed updateable by the Phase-1 rewrite — the rewrite
needs no new shape support.

The `MaterializedViewSchema` carries the body AST (`selectAst`) and optional
explicit `columns` — the only fields `analyzeView` reads off a `ViewSchema`
(alongside `name` / `schemaName`). So the rewrite functions can operate on an MV
schema structurally with a one-line type widening.

## Reuse map (almost everything is already wired)

- **DML rewrite + body gating:** `view-mutation.ts` → `classifyViewBody`
  (`propagate.ts`). Unchanged.
- **Omitted-column defaults:** the existing constant-FD path in `rewriteViewInsert`
  (`extractFilterConstants` / append) supplies a base column pinned by the body's
  `where` (`create materialized view m as select id, x from t where color = 'green'`;
  `insert into m (id, x) …` defaults `color = 'green'`), plus base-column defaults —
  identical to the view path.
- **Predicate-scope transition on update:** an update that carries a row out of the
  body `where` succeeds in `T` and leaves the MV; the row-time maintenance *update
  arm* already handles the predicate-scope transition (delete old image / upsert new
  image), so the backing follows with **zero** extra code.
- **Delete identity:** `rewriteViewDelete` builds the base predicate from the
  projected columns; the consolidation guarantees the projection includes `T`'s PK,
  so the base delete is exact and maintenance removes the right backing row.

## Per-column writeability (inherited verbatim from view updateability)

`deriveViewColumns` (`planner/analysis/update-lineage.ts`) +
`classifyProjectionExpr` (`planner/analysis/scalar-invertibility.ts`) classify each
projected column:

- **passthrough / rename** (`select id, x as y from t`) ⇒ `base` lineage ⇒
  writeable (routes the assignment/value to the base column).
- **any expression column** (`select id, x + 1 as y from t`) ⇒ `computed` lineage ⇒
  **read-only**: a write to it raises the existing `no-inverse` diagnostic
  (`requireBaseColumn`), reads are unaffected.

> **Scope correction vs the source plan ticket.** The source plan asserted that
> view updateability "already handles invertible expression columns through a
> scalar-invertibility profile (invertible ⇒ writeable)" and referenced a
> `func/invertibility.ts`. Neither is true against shipped code: the only file is
> `planner/analysis/scalar-invertibility.ts`, and Phase-1 treats **only a bare
> `ColumnExpr` as invertible** — every other expression is `computed`/read-only,
> with no inverse-application machinery. Writing `y` of `select id, x + 1 as y`
> through an inverse `y → x = y-1` is **not shipped** and is **out of scope** here:
> it is a view-updateability enhancement (Phases 2–7 in `docs/view-updateability.md`
> § Status) that applies to plain views too, not an MV-specific feature. MV
> write-through inherits whatever view updateability supports — today, computed
> columns are read-only. No regression: reads still work.

## RETURNING through an MV — rejected (inherited)

`view-mutation.ts` `rejectReturning` rejects RETURNING through any view today
(view-updateability "phase 6", not shipped). MV write-through inherits this: a
`… returning …` against an MV raises the `returning-through-view` diagnostic. When
RETURNING-through-views lands, the MV path picks it up for free. **Not delivered in
this ticket** — the source plan's "RETURNING through the MV column list returns
post-mutation rows" depends on the unshipped view RETURNING phase.

## MV-over-MV — already prevented at create (no new code)

The consolidation's create gate rejects an MV whose source is another MV's backing
table (`buildRowTimePlan` → "its source … is itself a materialized view's backing
table"; covered by `53 §8` and `materialized-view-diagnostics.spec.ts`). So an
MV-over-MV cannot exist, and write-through can never encounter one — **no separate
write-through rejection is needed.** (An MV over a plain *view* would be caught by
`analyzeView`'s existing `getView`-on-source nested-view check.) The source plan's
"reject with a clear source-is-an-MV diagnostic" is therefore satisfied by the
existing create-time rejection; do not add a redundant runtime check.

## The row-time composition (the point)

Because the MV is row-time maintained, write-through and maintenance compose with
**zero** extra maintenance code: MV-name write → source write → existing
per-statement maintenance flush → reads-own-writes via the shared backing
connection → rollback in lockstep. A write-through to an MV is observably identical
to writing the source and reading the MV.

## Non-goals

- Invertible-expression-column write-through (view-updateability Phase 2+; not
  MV-specific).
- RETURNING-through-MV (gated on the view RETURNING phase).
- MV-over-MV write-through (cascade deferred to
  `materialized-view-rowtime-mv-over-mv-cascade`; create is already rejected).
- No `with check option`, no `instead of` triggers — same stance as view
  updateability (the body `where` is a read-time filter, not a write-time invariant).
- This ticket does **not** touch the create-time eligibility gate (owned by the
  consolidation); it relies on the single-source shape that gate guarantees.

## Key tests (TDD targets)

New `test/logic/53.1-materialized-view-write-through.sqllogic`, modeled on the
plain-view `test/logic/93.4-view-mutation.sqllogic`:

- **Round-trip + reads-own-writes + rollback.** `insert` / `update` / `delete` into
  `create materialized view m as select id, x from t`; a same-transaction
  `select … from m` reflects the write; a `rollback` (explicit txn) reverts both `t`
  and `m`.
- **Filtered MV** (`… where color = 'green'`): `insert into m (id, x) …` omitting
  `color` defaults it via the constant-FD path; an `update` moving a row out of the
  `where` scope removes it from `m` (maintenance predicate-scope transition); a row
  whose insert provably contradicts the predicate is rejected
  (`predicate-contradiction`).
- **Delete identity:** delete by a non-PK projected column removes exactly the
  matching backing row(s).
- **RETURNING rejected:** `delete from m … returning *` raises the
  `returning-through-view` diagnostic (locks the inherited behavior).
- **Expression column read-only** (needs `materialized-view-rowtime-expression-projections`
  so `create materialized view m as select id, x + 1 as y from t` is creatable):
  a write to `y` raises `no-inverse`; `select y from m` reads fine; a write to the
  passthrough `id`/`x` still propagates.
- **Bulk multi-row** write-through (`insert into m select …` and a multi-row
  `update`/`delete`) produces correct backing contents under the per-statement
  maintenance batching.

Extend `test/logic/54-covering-mv-enforcement.sqllogic`:

- Write-through to a **row-time covering MV** that enforces a UNIQUE constraint
  resolves conflicts correctly (backing stays consistent within the statement; an
  `or replace` / `on conflict` against the MV resolves against the live backing).

MV-over-MV create-rejection is already covered (`53 §8`); cross-reference rather than
duplicate.
