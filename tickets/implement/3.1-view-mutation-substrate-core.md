## Why this is its own ticket

The parent (`3-view-mutation-substrate-orchestrator`) bundled the substrate, multi-source
joins, the tag-override surface, and the dynamic property harness. That is multi-pass work
on the engine's most delicate subsystem, and "retire the AST rewrite" is an all-or-nothing
change that must land with a green tree. This ticket isolates **the substrate + single-source
parity + rewrite deletion** so it can be proven green before the additive multi-source and
tag work rides on top. The `quereus.update.*` registry↔doc `dml-stmt` site reconciliation
that the parent also called for **already landed** in the parent's implement pass
(`schema/reserved-tags.ts` + `docs/view-updateability.md` tag table + 5 new tests in
`test/schema/reserved-tags.spec.ts`, all green) — do not redo it.

## Decision context (settled — do not re-litigate)

The AST rewrite is **retired**. The substrate is the single propagation path for all view
mutations; the single-source case is its trivial one-base-op path. A permanent two-tier
dispatcher (AST fast path + substrate) is exactly the two-codepaths-for-one-semantics debt
the `bx` derived-dual discipline exists to eliminate.

## Concrete architecture map (from the parent's exploration — saves rediscovery)

**Builder call sites that dispatch to the rewrite today** (all three follow one pattern):
- `building/insert.ts:385-389` — `const insertView = ctx.schemaManager.getView(schema, name) ?? ctx.schemaManager.getMaterializedView(schema, name); if (insertView) return buildInsertStmt(ctx, rewriteViewInsert(ctx, stmt, insertView));`
- `building/update.ts:43-47` — same shape with `rewriteViewUpdate` / `buildUpdateStmt`.
- `building/delete.ts:43-47` — same shape with `rewriteViewDelete` / `buildDeleteStmt`.

The dual `getView ?? getMaterializedView` lookup means **MV write-through rides this exact
dispatch** — rerouting these three sites through the substrate migrates MV write-through for
free (the row-time maintenance hook in `runtime/emit/dml-executor.ts` is orthogonal and fires
on the base write regardless of how the plan was produced; no MV-specific change needed).

**Why the rewrite is powerful (and what parity must preserve):** `rewriteView*` returns a
base-table AST DML statement that is re-fed through `buildInsertStmt`/`buildUpdateStmt`/
`buildDeleteStmt`, so it inherits the ENTIRE base-table builder for free — row-expansion +
defaults + generated columns (`insert.ts:41-225`), constraint checks (`ConstraintCheckNode`),
child/parent FK checks, NOT-NULL default evaluators, UPSERT clause plans, mutation-context
attributes/descriptors, RETURNING projection (`ReturningNode`), and the `SinkNode` wrapper.
Any from-scratch substrate that does NOT reuse this loses parity.

**DmlExecutorNode** (`planner/nodes/dml-executor-node.ts:39-144`): constructor is
`(scope, source: RelationalPlanNode, table: TableReferenceNode, operation: 'insert'|'update'|'delete', onConflict?, mutationContextValues?, contextAttributes?, contextDescriptor?, upsertClauses?)`.
`nodeType = PlanNodeType.UpdateExecutor`. `getRelations() → [source, table]`, `getChildren() → [source]`,
`computePhysical() → { readonly:false, idempotent:false, deterministic:false }`.

**Emit registration** (`runtime/register.ts:146-149`): `registerEmitter(PlanNodeType.Insert, emitInsert); …Update…; UpdateExecutor → emitDmlExecutor; Delete → emitDelete`. Dispatch is
`runtime/emitters.ts:emitPlanNode()` keyed on `plan.nodeType`. PlanNodeType enum lives at
`planner/nodes/plan-node-type.ts` (DML entries: `Insert`, `Update`, `UpdateExecutor`, `Delete`).

**Backward-lineage surface already landed** (`planner/analysis/update-lineage.ts`):
`deriveProjectUpdateLineage` / `deriveFilterAttributeDefaults` / `deriveJoinUpdateLineage`
thread `PhysicalProperties.updateLineage` / `attributeDefaults`; `viewColumnsFromUpdateLineage`
re-expresses the writable set off a planned node; `deriveViewColumns` is the AST-driven Phase-1
reader the rewrite consumes today. `identityBaseColumn` bridges the two.

## Recommended approach: Design A (reuse-maximizing)

Two designs were considered:
- **Design A (recommended):** `ViewMutationNode` holds the ordered list of **fully-built
  base-table plan subtrees** (each produced by `buildInsertStmt`/`buildUpdateStmt`/
  `buildDeleteStmt` from a per-base AST statement carried on a `BaseOp`). For single-source,
  the list has exactly one entry == today's rewrite output, so parity is **structural** — the
  wrapped subtree is byte-identical to what the rewrite re-plans today. The emitter sequences
  children; for one child it degenerates to a passthrough of that child's rows.
- **Design B (the parent's literal phrasing):** `ViewMutationNode` holds bare `DmlExecutorNode`s
  and the emitter re-implements constraint/FK/RETURNING/context coordination. Maximum control,
  but **no free parity** and a large reimplementation. Rejected for v1; revisit only if Design A
  cannot express multi-source ordering cleanly (it can — see 3.2).

`propagate(ctx, view, req): BaseOp[]` — for the single-source spine reuse the existing
`analyzeView` + `rewriteViewInsert/Update/Delete` logic (relocate it from `building/view-mutation.ts`
into `mutation/propagate.ts` or a sibling `mutation/single-source.ts`) to produce exactly one
`BaseOp` whose `.statement` is the rewritten base-table AST DML. `BaseOp` shape (extend in 3.2):
```typescript
interface BaseOp {
  readonly table: TableReferenceNode;        // resolved base table
  readonly op: 'insert' | 'update' | 'delete';
  readonly statement: AST.InsertStmt | AST.UpdateStmt | AST.DeleteStmt; // base-table DML to build
}
```
Then a small `building/view-mutation-builder.ts` (or inline helper) builds each `BaseOp.statement`
via the matching base builder and wraps the results in a `ViewMutationNode`.

## Risk callouts (must verify green, not assume)

- **Plan-snapshot tests** (`test/plan/**`): wrapping the result in a `ViewMutationNode` changes the
  top node from `SinkNode`/`ReturningNode` to `ViewMutationNode`. Grep `test/plan` and any
  `*.plan` fixtures for view-mutation expectations; update snapshots deliberately, never silently.
  If snapshot churn is large, consider making `ViewMutationNode` delegate `toString`/EXPLAIN to a
  single child for the one-BaseOp case, or have the single-source builder return the child subtree
  directly while still constructing through `propagate()` (decide and document).
- **Optimizer rules** traverse via `getRelations()`/`getChildren()`/`withChildren()`. Make
  `ViewMutationNode` a well-behaved `RelationalPlanNode` that exposes its children so existing
  rules (and the runtime scheduler) see the base DML subtrees. Confirm no rule assumes a DML top
  node is `UpdateExecutor`/`Sink`.
- **`getChangeScope()` / `Database.watch` / binding-extractor** must still see the base-table
  write. They walk the plan tree; because Design A keeps the real base DML subtrees as children,
  they should see the base writes — but add/confirm a change-scope test for a view mutation.
- **RETURNING-through-view** stays **rejected** (`returning-through-view` diagnostic) in this
  ticket — it lands in 3.2. Keep the existing rejection path working through the substrate.

## Parity gate (explicit test, add before deleting `building/view-mutation.ts`)

A single-source projection-filter view write constructs a `ViewMutationNode` over exactly one
`BaseOp`, and the built base subtree has conflict / FK / RETURNING-rejection / mutation-context
parity with the retired rewrite. Add this as a planner-level assertion (e.g. in
`test/property.spec.ts` near the `View Round-Trip Laws` block, or a new `test/planner/` spec):
plan a `create view v as select … from t where …; insert/update/delete … v`, assert the top is a
`ViewMutationNode`, its base-op list length is 1, and the wrapped `DmlExecutorNode.table` is `t`.
