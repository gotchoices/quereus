## Goal

Today an inner flagged `SetOperationNode` cannot be an operand of an outer one — the outer
constructor's column-count check (`set-operation-node.ts:47`) compares **total** columns
(data + appended flags), so `A union[inA,inSub] (B union[inB,inC] C)` throws at CREATE:

```
SET operation column count mismatch: left has 2, right has 4
```

because the inner subtree's flags `inB`,`inC` inflate its arity to 4 vs A's 2.

Resolve per **model (b)**: alignment, the union schema, dedup, and set identity are all on
**DATA columns only**; an inner operand's flag columns are appended/surfaced as readable
columns of the outer view but never enter the data-column set. This is the exact extension of
the *existing* single-level invariant (`set-operation-node.ts:64-77,95-106`) — flags already
sit after the data columns and out of the union schema/dedup — so the change generalizes the
invariant rather than special-casing it.

## Data arity is recursive

A `SetOperationNode`'s **data columns** are its left operand's data columns (flags are always
appended after), so data arity bottoms out at the left-most non-set-op leaf. Define a helper
(module fn or method) `dataArity(node)`:

```
dataArity(node) = node instanceof SetOperationNode ? dataArity(node.left)
                                                    : node.getType().columns.length
```

`SetOperationNode.dataColumnCount()` becomes `dataArity(this.left)` (replacing the current
`this.left.getType().columns.length`, which over-counts when `left` is itself a flagged
set-op). Expose `dataColumnCount()` (make it public or add a public getter) — the runtime
emitter and the write half (`nestable-flagged-set-ops`) both need it.

## Output layout (the defined projection rule)

For `N = L op R` with own membership `M`, N's output attributes are:

```
[ data columns ]  ++  [ L's flag attrs ]  ++  [ R's flag attrs ]  ++  [ M's own flag attrs ]
```

- **data columns**: the first `dataArity` attributes, taken verbatim from `L.getAttributes()`
  (preserves data attribute ids so an ORDER BY / enclosing view still resolves — the current
  `buildAttributes` already relies on preserving the left child's data attrs).
- **L's / R's flag attrs**: each operand's attributes *beyond* its own `dataArity` (empty when
  the operand is an unflagged leaf or a flag-less set-op). Their attribute ids are the inner
  specs' ids — threaded verbatim, so a row's surfaced inner flag keeps the inner node's id.
- **M's own flag attrs**: the appended `{true,false}` NOT NULL boolean columns with their
  pre-minted stable ids, exactly as today.

For the ticket fixture (`L=A` unflagged, `R=(B∪[inB,inC]C)`, `M=[inA(left),inSub(right)]`):
output = `[id, x] ++ [] ++ [inB, inC] ++ [inA, inSub]`, i.e. `id, x, inB, inC, inA, inSub`.

`getType().columns` mirrors the same layout (data ColumnDefs then every surfaced flag
ColumnDef). `getType().keys` is unchanged (`(intersect|except) ? leftType.keys : []`) — keys
are data-column ColRefs and **never** include any flag at any depth (Key-Soundness Inv. 1–2).

A node now has surfaced flags when `hasMembershipColumns OR L has flags OR R has flags`; add a
`hasSurfacedFlags` predicate. (A flag-less outer over a flagged operand — `A union (B∪[inB]C)`
— still surfaces `inB`.)

## computePhysical — own-flag column indices shift

The own membership flags now start at `ownFlagBase = dataColumnCount() + leftFlagCount +
rightFlagCount`, not `dataColumnCount()`. Update the index arithmetic in:

- `membershipDomains` — `column: ownFlagBase + i`.
- `membershipConstantBindings` — `col = ownFlagBase + i`.
- `membershipFds` — `totalCols = ownFlagBase + membership.length`; `allDataCols` stays
  `0..dataColumnCount()-1`. `superkeyToFd(allDataCols, totalCols)` then yields `key → {every
  surfaced flag}` (key determines inner flags too — correct).
- `membershipLineage` — keyed by `spec.attrId`, no index shift; unchanged.

Merging the inner operands' FDs/domains/lineage onto the outer is **not** required for the read
half (the write half handles inner-flag lineage surfacing); the outer simply must not mis-place
its own flags. Inner flags' physical surface stays on the inner node.

## Runtime read half (`runtime/emit/set-operation.ts`)

`dataColCount` must be `plan.dataColumnCount()` (recursive), not
`plan.left.getType().columns.length` (which over-counts a flagged left). `createOutputRow` /
`dataComparator` keep operating on the first `dataColCount` columns.

The membership runner (`runWithMembership`) must surface inner flags. Rework it to a
`runWithSurfacedFlags` selected whenever `plan.hasSurfacedFlags` (so a flag-less outer over a
flagged operand still surfaces). It must:

- buffer each operand's **full** row (not data-stripped) so the operand's flag columns survive;
- key the probe set (`leftSet` / `rightSet`) on the DATA columns (first `dataColCount`),
  storing the full operand row;
- produce each output row as `[data] ++ [L flags] ++ [R flags] ++ [own flags]`, where:
  - **L flags**: if the data tuple is in `leftSet`, the stored left row's flag slice
    (`row.slice(dataArity(left), dataArity(left)+leftFlagCount)`); else `false × leftFlagCount`.
  - **R flags**: symmetric over `rightSet`.
  - **own flags**: the existing per-spec probe `data ∈ <branch set>` → boolean.

The "default false when the output row is absent from an operand" rule is **sound**: an output
row not present in operand R is, by construction, in none of R's nested branches, so every R
flag (`tuple ∈ <nested branch>`) is false (verified for all four outer ops in the design pass —
union: an A-only row is in no nested branch of R; intersect/except: every visible row is/ isn't
in R uniformly). So a surfaced inner flag equals `tuple ∈ <that inner branch's data relation>`
row-by-row at every depth.

The plain streaming runners (`runUnionAll` etc.) are used only when **no** surfaced flags exist
(neither own nor inner) — unchanged.

## Scope (`building/select-compound.ts`)

`createSetOperationScope` iterates `setNode.getType().columns` against `setNode.getAttributes()`
positionally — both now include the surfaced inner flags in the same order, so an enclosing
view / ORDER BY can reference `inB`/`inC`. No change expected, but verify the alignment holds
(columns.length === attributes.length at every depth).

## Edge cases & interactions

- `A union[inA,inSub] (B union[inB,inC] C)` CREATEs, plans, and reads: `inA/inB/inC/inSub`
  each agree row-by-row with `tuple ∈ <that operand's data relation>`; data columns/keys stay
  on `[id,x]` only (no flag in any claimed key at any depth — Inv. 1–2).
- `intersect` / `except` as the **outer** op over a flagged subtree operand (not only `union`):
  every visible row's surfaced inner flags agree with the probe (intersect ⇒ row ∈ R ⇒ inner
  flags from R's stored row; except ⇒ row ∉ R ⇒ inner flags all false).
- `intersect` / `except` as the **inner** subtree op (the inner node computes its own flags
  per its operator; the outer just surfaces them).
- **Flag-less outer over a flagged operand** (`A union (B∪[inB,inC]C)`, no outer flags) still
  surfaces `inB,inC` (selected via `hasSurfacedFlags`, not `hasMembershipColumns`).
- **Flag-less subtree operand** read regression (`A union[inA,inSub] (B union C)`): `inSub ≡
  tuple ∈ (B∪C)`, no inner flags surfaced — the one nested behavior that already works stays
  green.
- **NULL data tuple** present in nested branches: the dedup/probe comparator treats `NULL =
  NULL` as equal at every depth, so surfaced inner flags stay clean `{true,false}` NOT NULL.
- **union all** outer/inner: bag multiplicity preserved; surfaced inner flags read off the
  buffered full operand rows (no dedup on the surfacing).
- Attribute-id stability across `withChildren` rebuilds: own specs already carry pre-minted
  ids; inner flag ids ride the operand's attributes — confirm a rebuilt nested node keeps every
  flag id.
- ORDER BY / enclosing-view reference to a surfaced inner flag (`select … from V order by inB`)
  resolves through `createSetOperationScope`.

## Tests (`test/property.spec.ts` § Set-operation membership columns)

Add nested read cases alongside the existing binary ones (reuse the `A`/`B` schema helper plus
a `C` table; `findSetOp` / `readRows` helpers already exist):

- **Nested read agreement** — `create view Vn as select id,x from A union exists left as inA,
  exists right as inSub (select id,x from B union exists left as inB, exists right as inC select
  id,x from C)`. Assert `inA/inB/inC/inSub` row-by-row equal membership in A / B / C / (B∪C),
  including an A-only row (inB=inC=inSub=false) and a (B∪C)-only row (inA=false).
- **Outer intersect / except over a flagged subtree** — same operand, outer op = `intersect`
  then `except`; assert surfaced inner flags agree with the probe.
- **Flag-less subtree read regression** — `A union[inA,inSub] (B union C)` reads `inSub ≡ tuple
  ∈ (B∪C)` with no inner flags (keep the behavior the backlog ticket noted already works).
- **Key Soundness at depth** — `findSetOp` on the nested body: `getType().columns` has the
  surfaced layout (`[id,x,inB,inC,inA,inSub]`); no flag column index appears in any
  `getType().keys`; `key → flag` FD present over the distinct nesting; each flag NOT NULL with a
  `{true,false}` enum domain at its shifted index.
- **Flag-less outer surfaces inner flags** — `A union (B union[inB,inC] C)` (no outer flags)
  still exposes `inB,inC` and reads them correctly.
