> Builds on the shared maintenance substrate: the `MaintenancePlan` union and
> `applyMaintenancePlan` dispatch (`incremental-maintenance-plan-abstraction`, 1.5) and the
> per-statement backing-connection cache (`materialized-view-rowtime-per-statement-batching`,
> 3). It is **arm-agnostic** — it does not implement or require any specific arm
> (`'inverse-projection'` / `'residual-recompute'` / `'prefix-delete'`); it routes whatever
> backing delta an arm produces back through `maintainRowTime`, so a chain may mix arms once
> the general-bodies tickets land. The covering-index (`'inverse-projection'`) chain is the
> shippable, testable case at landing time.

## Problem

The row-time-only consolidation rejects an MV whose single source resolves to another MV's
backing table:

```sql
create materialized view mv1 as select id, x from t where x > 0;
create materialized view mv2 as select id, x from mv1;   -- rejected today
```

A write to `t` maintains `mv1`'s backing `_mv_mv1` via `applyMaintenanceToLayer`. That
privileged write does **not** re-enter the DML-executor row-time hook (it is not user DML),
so `mv2` — whose source is `_mv_mv1` — is never maintained and silently serves stale rows.
The consolidation's `buildMaintenancePlan` therefore rejects the shape outright. This ticket
makes the cascade real and lifts the rejection.

## Key insight: the dependency edge already exists

When a body reads `mv1`, `building/select.ts` rewrites the reference to mv1's **backing
table** (`backingTableNameFor('mv1')` = `_mv_mv1`). So `mv2`'s single `TableReferenceNode`
points at `main._mv_mv1`, and `buildMaintenancePlan` computes `mv2.sourceBase === 'main._mv_mv1'`
— exactly mv1's backing base. The existing `rowTimeBySource` map (`source base → set of MV
keys reading it`) thus **already records the producer→consumer edge** the moment `mv2`
registers. No new dependency structure is needed; we only need to:

1. stop rejecting the shape at create, and
2. drive `rowTimeBySource[backingBase]` from each backing write.

Eligibility itself already works against a backing source: a backing table is an ordinary
keyed `memory` table with a `primaryKeyDefinition`, so the existing single-keyed-source +
passthrough-projection-covering-source-PK checks in `buildMaintenancePlan` evaluate against
the backing schema unchanged. For `mv2 as select id, x from mv1`, the source PK is mv1's
backing PK (`id`), which `mv2` projects — eligible.

## Approach

### 1. Capture each backing write's effective per-row delta (uniform across arms)

All arms funnel their backing mutation through
`MemoryTableManager.applyMaintenanceToLayer(connection, ops)` (`vtab/memory/layer/manager.ts`),
which already calls `lookupEffectiveRow` per op to find the row being replaced/removed. Make
it **return** the effective changes it applied so the cascade driver can route them onward —
the same `{ op, oldRow?, newRow? }` shape `maintainRowTime` already consumes:

```ts
export interface BackingRowChange {
  op: 'insert' | 'update' | 'delete';
  oldRow?: Row;
  newRow?: Row;
}

// applyMaintenanceToLayer(connection, ops): BackingRowChange[]
//   delete-key:        existing found → { op:'delete', oldRow: existing };  absent → (nothing)
//   upsert:            existing found → { op:'update', oldRow: existing, newRow: op.row }
//                      absent         → { op:'insert',                    newRow: op.row }
//   delete-by-prefix:  per matched row → { op:'delete', oldRow: matched }   (future arm)
```

This is the one structural change in `manager.ts`. It localizes delta capture to the single
place every arm already passes through — the producer plan needs no source re-read, because
the layer already knows the before image. Keep the existing exhaustive `switch` exhaustive.

### 2. Drive dependents from each backing write, recursively

`applyMaintenancePlan` returns the `BackingRowChange[]` from `applyMaintenanceToLayer` (it is
currently `void`). `maintainRowTime` then feeds each onward to the dependents of the plan's
**backing** base:

```ts
async maintainRowTime(sourceBase, change, mctx): Promise<void> {
  const keys = this.rowTimeBySource.get(sourceBase.toLowerCase());
  if (!keys || keys.size === 0) return;                 // O(1) negative fast path (unchanged)
  for (const key of keys) {
    const plan = this.rowTime.get(key);
    if (!plan) continue;
    const backingChanges = await this.applyMaintenancePlan(plan, change, mctx);
    if (backingChanges.length === 0) continue;
    const backingBase = `${plan.backingSchema}.${plan.backingTableName}`.toLowerCase();
    if (!this.rowTimeBySource.has(backingBase)) continue;   // no dependents → leaf, done
    for (const bc of backingChanges) {
      await this.maintainRowTime(backingBase, bc, descend(mctx));   // cascade
    }
  }
}
```

`mctx` is the per-statement maintenance context the batching ticket threads in (the cached
backing-connection map + statement scope); the cascade passes it through `descend()` so every
level resolves the **same** cached connection per backing table. `descend()` also bumps a
cascade-depth counter for the guard below.

Because a downstream MV must already exist when an upstream MV is created, the dependency
graph is **acyclic**; synchronous depth-first recursion is therefore DAG-ordered: a producer's
backing is fully written before its consumers run. Everything happens on the live transaction
within the current statement, so the whole chain commits or rolls back atomically. The leaf
fast path (`!rowTimeBySource.has(backingBase)`) keeps a non-chained MV at exactly today's cost
(one map lookup, no recursion).

### 3. Atomicity and intra-statement visibility fall out

- **Atomic rollback across the chain.** A depth-≥2 backing connection registers lazily on its
  first cascade write; `Database.registerConnection` replays the active savepoint depth onto
  it (including the statement-atomicity savepoint opened before the row loop in
  `runWithStatementSavepoints`), so a statement/transaction rollback reverts every level's
  backing layer in lockstep. No new transactional plumbing.
- **Reads-own-writes for downstream enforcement (semantic edge B).** Cascade writes go through
  the same per-statement cached connection a `select`/enforcement scan resolves to. So when a
  later same-statement source row's enforcement scans a downstream covering MV's backing
  (`lookupCoveringConflicts`), it observes all prior rows the cascade already wrote this
  statement. This is the per-statement-batching invariant, preserved automatically by routing
  cascade through the cached connection — **do not** introduce deferred/end-of-statement flush
  on the cascade path that could hide an earlier row from a later enforcement read. With the
  batching ticket's v1 (per-row apply on the cached connection), each level applies
  synchronously, so flush order is trivially correct; if op-coalescing is later layered on,
  cascade must flush a level before recursing into its dependents.

### 4. Lift the rejection + add guards

In `buildMaintenancePlan`, delete the `getMaterializedViewByBackingTable(...)` rejection block
(the `its source '…' is itself a materialized view's backing table …` arm). The remaining
eligibility checks already validate the backing source correctly.

Add a **cascade-depth guard** (defense-in-depth — cycles are structurally impossible, so this
should never fire): if recursion depth exceeds a small bound (e.g. the count of registered
row-time MVs, an absolute upper bound on a valid chain length), throw `StatusCode.INTERNAL`
naming the backing base, rather than overflowing the stack. Optionally, at create time, walk
the source-MV chain (`getMaterializedViewByBackingTable` upward from the new MV's source) to
assert no cycle and bound depth — loud `INTERNAL` if violated. Keep all unchanged rejections
(aggregate/join/DISTINCT/set-op/recursive-CTE/TVF/LIMIT-OFFSET, no-PK, computed projection,
dropped-PK-column) exactly as they are; an ineligible *body* over an MV source is still
rejected by those, just no longer pre-empted by a blanket MV-over-MV reject.

## What does NOT change

- The synchronous, in-transaction, DML-boundary maintenance policy. No commit-time path.
- The `MaintenancePlan` arm bodies. Cascade consumes whatever arm each plan specifies via the
  uniform `applyMaintenanceToLayer` → `BackingRowChange[]` path.
- `rowTimeBySource` / `rowTime` map structures and `findRowTimeCoveringStructure` /
  `lookupCoveringConflicts` signatures. Covering enforcement keeps keying off the user source
  table and the constraint's `coveringStructureName`; a covering MV that happens to also be a
  source for a further MV still enforces unchanged.

## Coordination with sibling maintenance-surface tickets

This edits the same `database-materialized-views.ts` / `dml-executor.ts` / `manager.ts`
surface as several active tickets — whoever lands second reconciles:

- `incremental-maintenance-plan-abstraction` (1.5) — provides `MaintenancePlan`,
  `applyMaintenancePlan`, `buildMaintenancePlan`, the equivalence harness. **Hard prereq.**
- `materialized-view-rowtime-per-statement-batching` (3) — provides the per-statement
  backing-connection cache / maintenance context `maintainRowTime` threads. **Hard prereq**
  so the cascade threads the cache rather than re-resolving per step (avoids rework, and is
  exactly the mechanism semantic-edge-B relies on).
- `rowtime-mv-minor-cleanups` — renames the `{ op; oldRow?; newRow? }` change payload to a
  named discriminated union across `maintainRowTime` / `applyMaintenancePlan` / the
  DML-executor hook. If it has landed, reuse that union for both the inbound `change` and the
  new `BackingRowChange` (they are the same shape — unify, don't duplicate).
- `materialized-view-rowtime-residual-recompute` (3.2) / `-prefix-delete-lateral-tvf` (3.4)
  add new arms. Cascade is arm-agnostic and needs neither; once they land, a chain may mix
  arms and the harness's chain assertions exercise it. No ordering dependency.

## Acceptance

- An MV-over-MV chain (2+ levels) reflects a source insert/update/delete transparently and
  mid-transaction (reads-own-writes), and rolls back as a unit.
- The create-time MV-over-MV rejection is removed; the diagnostics spec asserts acceptance.
- Covering-MV UNIQUE enforcement still works when the covering MV sits in a chain; an
  intra-statement enforcement scan on a downstream backing observes all prior cascade rows of
  the same statement.
- The maintenance-equivalence harness, extended to chains, holds `read(MV_backing) ==
  evaluate(body)` for **every** MV in the chain after each mutation and after rollback.

## Tests (key cases + expected outputs)

### `53-materialized-views-rowtime.sqllogic` — new "MV-over-MV cascade" section
- 2-level chain `mv2 over mv1 over t`: insert into `t` → both `select * from mv1` and
  `select * from mv2` reflect the new row with no refresh. Update a projected column and
  delete; both levels track. Expected: `mv2` rows equal `select <mv2-body> from mv1-body`.
- Predicate transition through the chain: `mv1 as select id,x from t where x>0`, `mv2 as
  select id,x from mv1`; an update moving a `t` row from `x=5` to `x=-1` removes it from both
  `mv1` and `mv2`; the reverse re-adds it.
- 3-level chain to prove arbitrary depth and DAG order.
- Mid-transaction `begin; insert …; select * from mv2;` shows the row before commit;
  `rollback;` then `select * from mv2;` shows it gone — and `mv1` likewise (whole-chain
  atomic rollback).

### `materialized-view-diagnostics.spec.ts`
- Replace the `rejects a materialized view over a materialized view …` test: now
  `create materialized view mv_over as select id, v from mv_base;` **succeeds**, and after an
  insert into `g`, `select * from mv_over` reflects it. Keep the "never leaks `_mv_…`" spirit
  for whatever diagnostics remain.

### `test/incremental/maintenance-equivalence.spec.ts` (substrate harness, extend)
- Add a chain arbitrary: build `t`, `mv1 over t`, `mv2 over mv1` (and optionally `mv3 over
  mv2`) from the covering-index body arbitraries; run a random insert/update/delete sequence
  on `t`; after **each** mutation assert, as multisets, `rows(select * from mv1) ==
  rows(<mv1-body> against t)` AND `rows(select * from mv2) == rows(<mv2-body> against mv1's
  body against t)`; then `ROLLBACK` and assert both again. This is the chain oracle the
  acceptance names.

### `54-covering-mv-enforcement.sqllogic`
- A covering MV that is also a chain link (or a downstream covering MV): a bulk
  `insert into t values (…dup within one statement…)` must detect the duplicate via the
  downstream backing reflecting the earlier same-statement row (ABORT/IGNORE/REPLACE all
  resolve correctly), proving cascade kept the backing intra-statement-consistent. Note: if a
  *downstream* MV proving a constraint on `t` requires the lens prover to see through the
  chain to `t`, that proving is the lens layer's concern (`lens-prover-and-constraint-
  attachment`); this ticket only guarantees the backing is row-time *consistent* so such
  enforcement is possible. Cover the realistic in-reach case (covering MV at level 1 whose
  backing then cascades) and document any lens-proving dependency for the deeper case.
