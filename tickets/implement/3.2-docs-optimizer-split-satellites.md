## Rules of the move

- **Cut at headings, never at line numbers.** Line offsets shift under you; the
  prereq ticket already moved 10,000 words out of this file.
- **Promote depth by one** in each satellite (top section becomes `# `). Slugs are
  depth-independent, so an anchor survives promotion — only the filename in each
  inbound link changes.
- **Leave a stub, not a hole.** Every removed `##` section leaves behind a
  one-to-three-sentence summary and a link, in the hub, at its original position.
  A reader skimming the hub must still learn that asof-scan recognition exists.
- **Apply the three-vocabulary sort from `docs/doc-conventions.md`** as each block
  passes through your hands: delete narrative history; condense why-nots into a
  `## Rejected alternatives` section in the receiving satellite; move
  future/planned prose to `docs/todo.md` or an existing backlog ticket, leaving a
  one-liner under `## Current limitations`. The `## Known Issues` section is a
  concentration of the latter — it names three open tickets by path. Reduce it to
  pointers.
- **Do not extract invariants.** That is `docs-invariants-optimizer`. Here you
  move and prune; you do not restate.
- **Do not rewrite the code any doc describes,** and do not fix a doc's substantive
  claims about behavior beyond correcting an obviously-dead symbol name. A wrong
  claim you find is a finding for the review handoff, not a scope extension.

## Edge cases & interactions

- **Slug collisions after promotion.** `### Overview` appears under both
  "Join Optimization with QuickPick" and "Retrieve-based Push-down Architecture";
  `### Registration` appears in several rule write-ups; `### Integration Points`
  appears twice. Once promoted into a satellite these may collide, and the
  checker's `-1` suffix will make a link resolve to the wrong heading *silently
  and successfully*. Audit the promoted outline of every satellite for duplicate
  heading text before running the checker. This is the failure mode most likely to
  ship a wrong link that passes the gate.
- **`#assertion-derived-premises`, `#coverage-proving`,
  `#effective-key-proving-body-proves-it`** are `####`-depth anchors — the plan-time
  outline scan only enumerated `#`–`###`, so their exact home is unverified.
  Locate each before assuming which satellite it belongs to; two of them are cited
  from `materialized-views.md` and `lens.md`, which `docs-mv-split` is also
  touching.
- **`## Visited Tracking Architecture` is a real 1,200-word section, not a
  container.** Nine `###` deep-dives were misfiled under it. When you lift them
  out, the surviving section must read as a coherent standalone — check its last
  subsection (`### Integration with Advanced Optimizations`) does not turn into a
  dangling reference to sections that left the file.
- **`## Component Reference` is 6,589 words**, almost all of it the rule catalog.
  What remains after the catalog leaves (intro, Plan Node Hierarchy, Virtual Table
  Integration, Debugging and Tracing) is a few hundred words. Either keep it as a
  thin `## Component Reference` in the hub or dissolve it — do not leave a
  `##` heading whose only content is "see the satellites."
- **The hub must stay under 12,000 words** with no ratchet entry. If it does not,
  the grouping is wrong: the likeliest culprit is Engineering Considerations
  (2,225 words), whose physical-properties material could join
  `optimizer-fd.md`'s neighbourhood — but prefer trimming history to inventing a
  ninth doc.
- **Six new files land in one commit.** The review pass reads a large diff. Order
  the work satellite-by-satellite (cut → stub → repoint → `yarn docs:check`) so
  each is individually green before starting the next, rather than one big cut
  followed by one big repair.
- **`docs/review.html` / `docs/review.md`** are frozen artifacts naming old
  optimizer sections. Exempt. Do not update.
- **Interaction with `docs-mv-split`** (sibling, no ordering constraint between
  them): both edit `materialized-views.md`'s outbound links into `optimizer.md`.
  Whichever lands second must re-run `yarn docs:check` and fix the crossing; the
  checker is what catches it. Neither ticket may assume the other's state.
- **Interaction with `docs-stability-tiers`** (separate plan ticket): it wants a
  tier banner in each topic doc's header. Six new topic docs are six new banners.
  Nothing to do here beyond keeping each satellite's header conventional (h1,
