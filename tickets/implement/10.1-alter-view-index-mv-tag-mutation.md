# ALTER VIEW / ALTER INDEX / ALTER MATERIALIZED VIEW … SET TAGS

## Why

`alter-table-tag-mutation` closes the tag-mutation gap for the three sites under `ALTER TABLE`.
The remaining tagged catalog objects — **views**, **materialized views**, and **indexes**
(`CatalogView.tags`, `CatalogMaterializedView.tags`, `CatalogIndex.tags`) — have **no `ALTER`
statement at all**: the parser's top-level `ALTER` dispatch only accepts `TABLE`
(`parser.ts:343` → `alterTableStatement`). So their tags are write-only at create time, and the
declarative differ both fails to detect tag drift and has no in-place primitive to apply it (it
would otherwise have to drop+recreate — which for a **materialized view** means a full, needless
re-materialization of the body). This ticket adds the missing in-place tag primitive for all three.

## Design

Follow the `SET TAGS` model established by the prereq ticket: **whole-set replacement**, empty list
clears, catalog-only mutation (no module/data round-trip — re-register the schema object and fire
the matching change event), reserved-tag validation at the right site, and differ detection that
emits the in-place `SET TAGS` form.

### New statements

```sql
alter view v               set tags (cacheable = true);
alter materialized view mv set tags (owner = 'analytics');
alter index idx_name       set tags (purpose = 'search');
alter index idx set tags ();   -- clear
```

Top-level `ALTER` dispatch (parser.ts:343) currently advances and unconditionally calls
`alterTableStatement`. Replace with a small dispatch: `TABLE` → existing path; `VIEW` →
`alterViewStatement`; `MATERIALIZED VIEW` → `alterMaterializedViewStatement`; `INDEX` →
`alterIndexStatement`. Each new parser method consumes the object name then `SET TAGS` + reuses
`parseTags()`. v1 scope for these three statements is **`SET TAGS` only** — no other alter verb
(structural changes to views/MVs/indexes remain drop+recreate via the declarative pipeline).

New AST nodes (parser/ast.ts), each minimal:

```ts
export interface AlterViewStmt extends AstNode { type: 'alterView'; name: IdentifierExpr; action: { type: 'setTags'; tags: Record<string, SqlValue> } }
export interface AlterMaterializedViewStmt extends AstNode { type: 'alterMaterializedView'; name: IdentifierExpr; action: { type: 'setTags'; tags: Record<string, SqlValue> } }
export interface AlterIndexStmt extends AstNode { type: 'alterIndex'; name: IdentifierExpr; action: { type: 'setTags'; tags: Record<string, SqlValue> } }
```

(Keep an `action` envelope even though `setTags` is the only member, so the structural-alter
follow-ups have a home.) Add them to the top-level statement union.

### Plan node(s) + emitters

One generic node is cleanest: `SetObjectTagsNode` (VoidNode) carrying `{ objectKind: 'view' |
'materializedView' | 'index'; schemaName; name; tags }`, `computePhysical → { readonly: false }`,
with a matching emitter `emitSetObjectTags` registered in `emit/emitter.ts`. The emitter looks the
object up, validates reserved tags at the right site (`view-ddl` for view/MV, `physical-index` for
index), swaps in the new `tags` (or `undefined` when empty), re-registers it, and fires the
matching change event (`table_modified`-style for the backing artifacts / the relevant notifier
event for views/MVs/indexes — match whatever the create path emits).

### SchemaManager helpers

Add `setViewTags`, `setMaterializedViewTags`, `setIndexTags` mirroring `setTableTags`: catalog-only
swap, clear on empty, `NOTFOUND` on missing object. (Index lookup is by index name; resolve the
owning table from `IndexSchema`.)

### Differ + migration DDL

The view / MV / index loops in `computeSchemaDiff` currently only create/drop (and MV
rebuild-on-body-hash). Add tag-drift detection that does **not** force a drop+recreate:

- New `SchemaDiff` buckets, e.g. `viewTagsChanges`, `materializedViewTagsChanges`,
  `indexTagsChanges`, each `Array<{ name: string; tags: Record<string, SqlValue> }>`.
- For a declared object that **name-matches** an actual one (no create/drop), compare tags via
  `stableStringify` (excluding `quereus.id` / `quereus.previous_name`, consistent with the prereq).
- `generateMigrationDDL` emits `ALTER VIEW <n> SET TAGS (...)` / `ALTER MATERIALIZED VIEW <n> SET
  TAGS (...)` / `ALTER INDEX <n> SET TAGS (...)` in the alter phase (after creates, before/with the
  table-alter block — ordering is independent since these are leaf metadata writes).

Note: an MV whose **body** changed still drops+recreates (existing `bodyHash` path) — a tag-only
change must take the new in-place path and **not** trigger a rebuild. Guard so the two are mutually
exclusive per object.

### Stringify

Add `alterViewToString` / `alterMaterializedViewToString` / `alterIndexToString` to ast-stringify
and wire them into the statement stringifier so emit-roundtrip covers the new statements.

## Edge cases & interactions

- **MV body-change vs tag-change**: never emit both a rebuild and a `SET TAGS` for the same MV in
  one diff. If the body changed, the recreate already carries the declared tags; only emit `SET
  TAGS` when the body hash matches but tags drifted.
- **Implicit covering-structure indexes**: a UNIQUE constraint's auto-built index is omitted from
  the catalog unless `quereus.expose_implicit_index = true`. `ALTER INDEX … SET TAGS` should target
  only user-visible indexes; setting tags on an implicit covering index is out of scope (its tags
  live on the originating constraint — use `ALTER TABLE … ALTER CONSTRAINT … SET TAGS` from the
  prereq ticket). Confirm a hidden implicit index name errors `NOTFOUND` here.
- **Name resolution / schema qualification**: views, MVs, and indexes are schema-scoped; honor a
  qualified `schema.name`. Index names are unique per schema (resolve owning table internally).
- **Clear semantics**: empty list → `tags` becomes `undefined` (round-trips with no `with tags`).
- **Schema hash unaffected** (tags excluded from the hash) — assert for each object kind.
- **Reserved-tag validation** runs at `view-ddl` / `physical-index` sites — a typo'd `quereus.*`
  key fails loudly.
- **`alter view` vs updatable-view tags**: `quereus.update.*` tags on a view are behavioral; a
  `SET TAGS` that drops/changes them changes write-through routing — desirable and version-tracked,
  but note it in docs so it isn't a surprise.
