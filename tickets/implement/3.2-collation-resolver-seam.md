# One official way to resolve a collation name, per database

## Background for a reader with no context

A *collation* is a named text-sorting rule (`BINARY` = raw byte order, `NOCASE` =
case-insensitive, `RTRIM` = ignore trailing spaces, plus whatever the embedder
registers). SQL columns, indexes, `ORDER BY`, and `UNIQUE` all reference a
collation *by name*; the engine has to turn that name into a comparison function.

Quereus stores collations **per `Database` instance** (`database.ts` ~133), so two
open databases can register different rules under the same name — there is a test
locking that isolation (`test/collation-isolation.spec.ts`). But `util/comparison.ts`
also carries an older **process-global** registry (~16), and several low-level
comparison paths still resolve names against *that*. A name registered on a
database is invisible to those paths; they fall back to `BINARY` and emit a
debug-level log nobody sees.

## What this ticket does

This is the **seam** ticket. It introduces the one resolution primitive that the
three follow-on tickets (memory vtab, store + isolation, core callers) then adopt.
It does **not** delete the global registry — `3.6-retire-global-collation-registry`
does that once nothing reads it.

### `CollationResolver`

Add to `packages/quereus/src/types/logical-type.ts`, beside `CollationFunction`:

```ts
/**
 * Resolves a collation name to its comparison function for one specific
 * database. Throws `QuereusError(StatusCode.ERROR)` when the name is not
 * registered on that database — an unresolvable collation is never silently
 * downgraded to BINARY, because byte-order results would be wrong and invisible.
 *
 * Names are case-insensitive (see `normalizeCollationName`).
 */
export type CollationResolver = (collationName: string) => CollationFunction;
```

Export it from `src/index.ts`.

### `Database.getCollationResolver()`

Add to `core/database.ts`:

```ts
/**
 * The canonical collation resolver for this database. Stable identity across
 * calls (bound once), reads the live per-database registry, so a collation
 * registered after the resolver is handed out is visible to later calls.
 */
getCollationResolver(): CollationResolver;
```

Implementation: a lazily-created bound closure stored on the instance. Body:
`BINARY` fast-path → `this._getCollation(name)` → on miss,
`throw new QuereusError(\`no such collation sequence: ${name}\`, StatusCode.ERROR)`.

Add `getCollationResolver(): CollationResolver` to the `DatabaseInternal`
interface (`core/database-internal.ts`) so `quereus-store` and
`quereus-isolation` can reach it through their existing `this.db as
DatabaseInternal` cast.

### `builtinCollationResolver`

Also add to `util/comparison.ts` a database-free resolver over the three
built-ins only:

```ts
/**
 * Resolves only the built-in collations (BINARY / NOCASE / RTRIM). For standalone
 * utility code and tests that have no `Database`. Returns `undefined` for any other
 * name — callers must decide whether that is an error or a reason to bail.
 */
export function builtinCollationResolver(name: string): CollationFunction | undefined;
```

Implement it as a `switch` on the upper-cased name, not a `Map` — it must outlive
the deprecated global registry that `3.6-retire-global-collation-registry` deletes.
`Database.registerDefaultCollations` keeps its own entries (it pairs each built-in
with a key normalizer and a `replicable` stamp); do not try to share state between
the two.

### Why throw, not warn

The plan ticket left this open. Resolution: **throw**.

- `planner/building/schema-resolution.ts` `resolveCollation` (~210) — the
  build-time path — *already* throws `Collation not found: X`. Two paths that
  answer the same question must not disagree.
- Column-level collation names are validated at DDL time
  (`schema/table.ts` ~202, pinned by `test/logic/41.7-alter-column-collate.sqllogic`
  and `test/logic/102.1-unique-edge-cases.sqllogic`), so an unresolvable name at
  comparison time means either an internal bug or a persisted table whose custom
  collation was never registered on this connection. Both deserve an error.
- SQLite errors with `no such collation sequence: X` in exactly this case. Match
  the wording so the message is searchable.
- Silent `BINARY` fallback in these paths produces wrong sort order, wrong
  `UNIQUE` enforcement, and wrong index seeks with no signal at all. That is the
  defect this ticket family exists to close.

**Accepted tradeoff:** opening a persisted database whose DDL names a custom
collation now throws when the embedder has not re-registered that collation,
where it previously returned byte-ordered rows. That is the correct, SQLite-matching
behavior, and it is louder than the silent corruption it replaces. Document it in
`docs/sql.md` under COLLATE: *custom collations must be registered on every
`Database` that opens a table declaring them, before the table is queried.*

### `EmissionContext.resolveCollation`

Rewrite `runtime/emission-context.ts` (~194) to delegate:

- Keep the `BINARY` fast path and keep recording the collation dependency via
  the existing `getCollation()` (which snapshots the collation for plan
  invalidation) — do not lose dependency tracking.
- Replace the `log(...) + return BINARY_COLLATION` miss branch with the
  database resolver's throw.

If a test surfaces an emitter passing a name that is not a registered collation
(watch `effectiveCollationOfTypes` in `runtime/emit/join.ts` ~49, and the
`attr.type.collationName` reads in `distinct.ts` / `set-operation.ts` /
`async-gather.ts`), **fix the producer of that name** — do not restore the silent
fallback.

## Edge cases & interactions

- **Case-insensitivity:** `getCollationResolver()('nocase')` must resolve.
  `_getCollation` already upper-cases; confirm and keep. `normalizeCollationName`
  stays the canonicalizer for DDL.
- **Empty / whitespace name:** `''` must throw, not silently resolve to BINARY.
- **Resolver identity is stable, results are not cached.** A collation registered
  *after* an index's comparator was built does not rebuild that index. This is
  pre-existing behavior; leave a `NOTE:` comment at `Database.registerCollation`
  saying so rather than filing work for it.
- **Overwriting a built-in:** `db.registerCollation('NOCASE', ...)` is legal today
  (`test/collation-isolation.spec.ts` ~65 relies on it). The resolver's `BINARY`
  fast path means `BINARY` alone cannot be overridden — that is existing
  `EmissionContext` behavior; keep it, and state it in the resolver's doc comment.
- **Two databases, same name, different functions:** each resolver must see only
  its own database's registration. `test/collation-isolation.spec.ts` covers the
  emitter path; add a direct unit assertion on `getCollationResolver()` for both
  databases.
- **Closed database:** `getCollationResolver()` on a closed `Database` — the
  resolver closure holds `this`. Decide and document: reading the map after close
  is harmless, so do **not** add a `checkOpen()` to the resolver itself (it runs in
  hot comparison paths); `registerCollation` keeps its `checkOpen()`.
- **Hot path cost:** the resolver is called at comparator-construction time by
  every follow-on ticket, never per-row. Verify no per-row call slips in;
  `test/performance-sentinels.spec.ts` guards some of these paths.

## Tests

- `getCollationResolver()('BINARY' | 'NOCASE' | 'RTRIM')` returns the built-ins.
- `getCollationResolver()('nocase')` resolves case-insensitively.
- `getCollationResolver()('NOPE')` throws `QuereusError` matching
  `/no such collation sequence: NOPE/`.
- Two `Database` instances, each registering `REVERSE` with a different
  comparator, get different functions from their own resolvers.
- A collation registered after `getCollationResolver()` was obtained is visible.
- Existing suites stay green: `test/collation-isolation.spec.ts`,
  `test/collation-normalizer.spec.ts`, `test/logic/41.7-alter-column-collate.sqllogic`,
  `test/logic/102.1-unique-edge-cases.sqllogic`.
