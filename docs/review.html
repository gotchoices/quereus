Quereus Code & Design Review

Full-monorepo review — July 2026. Five parallel deep-dive passes: core/parser, planner/optimizer, runtime/vtab, satellite packages (store → isolation → sync), and architecture/test strategy. All findings verified against source; line numbers from main at time of review.

Executive Summary

Verdict: Quereus is an unusually well-engineered engine with a coherent theoretical spine — FD tracking → view updateability → materialized views → lenses form one deliberate dependency chain, and the layered test strategy (soundness property nets, round-trip laws, seeded fuzz, deterministic concurrency tests) is well above typical for a project this size. The core design choices (VTab-centric federation, async core, attribute-based context, single PlanNode hierarchy) are sound and verified in code, not just in docs.

The biggest risks are not design errors. They are: (a) total absence of CI for a ~134K-line engine plus 15 satellite packages; (b) a handful of genuine correctness bugs in the persistence/sync chain (key-encoding sort order, sync watermark advancement) that silently produce wrong results or permanent data loss; (c) a systemic cost-model contract violation that distorts join ordering; and (d) a research-program-sized feature surface (lenses, migration, parallel runtime) maintained on single-committer bandwidth, with mega-docs (optimizer.md alone 281 KB) already drifting from code.

Severity distribution: 18 high, ~25 medium, ~20 low across ~80 verified findings. The high-severity cluster concentrates in quereus-store key encoding, quereus-isolation transaction semantics, and the sync protocol seams — the layers that underpin persistence and replication.

Strategic recommendations (ranked)

  1. Stand up CI now. One GitHub Actions workflow: lint + build + test + test:store + test:fork-strict on push. Highest-ROI item in the entire review; test:store and fork-strict are currently excluded from default yarn test and nothing guarantees they ever run.
  2. Fix the persistence-layer key encodings before anything is stored durably at scale (findings 1, 2, 7 above). These silently return wrong query results and become data-migration problems the moment real data exists. Add a mixed int/real range test and a KVStore conformance suite run against LevelDB, IndexedDB, and memory stores.
  3. Make the isolation layer honest. Either implement snapshot + write-write conflict detection (base-row fingerprints at first touch, fail the second committer) or fix the documentation in both packages to say "read-committed + read-your-own-writes". Coordinate multi-table flush into one underlying transaction.
  4. Harden the sync watermark/ack protocol. Advance lastSyncHLC only from server-ordered responses; serialize commit handling through a promise chain; validate HLC drift before commit; stream tombstones in snapshots; only kill auto-reconnect on fatal (auth) errors. Extract one shared protocol module with a protocolVersion handshake — the client and coordinator codecs have already drifted.
  5. Pick one cost-model convention and enforce it in the plan validator (self-cost only, or totals-everywhere). Then memoize getTotalCost() — plan construction is currently O(depth²).
  6. Declare stability tiers. Core SQL + vtab + views: stable. Materialized views: beta. Lenses / migration / parallel runtime: experimental. Contains the blast radius of the research tracks; consider freezing the parallel track until a federated consumer pays its rent.
  7. Mechanize the context-shadowing invariant. The runtime's shared mutable RowContextMap with last-set-wins semantics plus hand-applied per-emitter discipline is the engine's most likely source of silent wrong-row bugs. Add a debug-mode assertion analogous to QUEREUS_FORK_STRICT.
  8. Delete the dead scaffolding. The dead global rule registry (which swallows exceptions, unlike the live path), PredicateAnalysis/CapabilityRegistry, SQLite-era config types, deprecated global collation registry, dead REPL methods. Half-dead parallel paths with divergent semantics are traps.
  9. Shrink and true-up the mega-docs. Four concrete doc drifts verified (below); 281 KB optimizer.md and 204 KB materialized-views.md are past the point of being reviewable against code — extract normative invariants from narrative history.

1. Core, Parser, Common, Util

Assessment. Unusually disciplined engine code. The error taxonomy (QuereusError subclasses with status codes, cause-chaining, AbortError interop) is thoughtful; wrapAsyncIterator's commit/rollback separation is a genuinely clean solution to a hard iterator-lifecycle problem; comparison.ts hot paths show real perf awareness. Weak spots: drifted seams (two collation registries, two equality functions with divergent semantics), soft statement-lifecycle corners (inconsistent mutex, non-resetting reset(), leakable listeners), and TODO placeholders that quietly became user-visible behavior.

Broken getBlockSql() — "[object Object]"

core/statement.ts:79,133 · robustness

AST statements have no toString(); public getBlockSql(), reconstructed originalSql, and the planning log all emit garbage. astToString in emit/ast-stringify.ts already exists — use it at both sites.

Parser error re-wrap strips subclass identity and location

parser/parser.ts:106–119 · robustness

parseAll matches e.name === 'QuereusError', but every subclass overrides name — a ParseError gets logged as unhandled and re-wrapped without location info. Fix: if (e instanceof QuereusError) throw e;.

key-serializer equality invariant does not hold

util/key-serializer.ts:46–59 · correctness

Doc promises "equal values ⇔ identical keys", but 5n vs 5 and true vs 1 get different tags, and every distinct JSON object serializes to 'o:[object Object]' — wrongly merging unequal values in PARTITION BY / hash joins, wrongly splitting equal ones in store UNIQUE re-validation. Normalize numeric classes to one tag; JSON.stringify object values.

LIKE/GLOB compile a fresh RegExp per row

util/patterns.ts:16–30, callers in runtime/emit/binary.ts, func/builtins/string.ts · performance

Two .replace passes + new RegExp per row scanned. Memoize compiled patterns (small LRU) and pre-compile in emitLikeOp when the pattern is a literal.

Statement lifecycle soft spots

core/statement.ts:164–213, 390–421 · robustness

(a) iterateRows() skips the exec mutex that run()/get()/all() acquire — concurrent statements can interleave with implicit-transaction lifecycle. (b) Schema-change listener leaks when a recompile yields no dependencies (unsubscribe is conditional). (c) reset() merely warns and clears busy while iteration may be in flight — second iteration can start concurrently. (d) db.get() binds parameters twice.

Dead SQLite-era config types mislead plugin authors

common/types.ts:111–176 · dead code

DatabaseInfo/DatabaseConfig (WAL, journal mode, page size) have zero references and describe features Quereus doesn't have. Delete.

Two collation registries, silent BINARY fallback

util/comparison.ts:14,67–100 vs core/database.ts:376 · design

Deprecated global registry is still the live fallback inside compareSqlValues; per-database registrations are invisible to util-layer resolvers (debug-log-only BINARY fallback). Needs a migration plan; at minimum surface the fallback once per name. Related: sqlValuesEqual duplicates compareSqlValuesFast with divergent semantics (5n === 5 false via ===) and a stale doc comment — delete it.

Smaller items

various · cleanliness/performance

needsParens rebuilds its precedence table per binary node (hot in canonical-DDL hashing) — hoist to module constants; table omits <>/IS. OBJECT comparison JSON.stringifies both sides per compare inside sorts. Lexer builds lexemes char-by-char (slice-to-terminator for the no-escape case) and carries a dead parseInt try/catch. Token.literal?: any and two (stmt as any).withClause casts violate the no-any rule. Latches is a process-global static keyed by string — shared across all Database instances, no timeout/diagnostics. database-materialized-views.ts at 3,462 lines needs the same decomposition its siblings got. validateParameterTypes named-param ?? fallback misfires on legitimate null bindings.

2. Planner & Optimizer

Assessment. The pass framework is clean (iterative worklist traversals, depth + rules-fired budgets, per-pass memo cache); sideEffectMode audit discipline is enforced at registration with soundness rationales; FD/EC/IND propagation is unusually rigorous. Weaknesses: the cost-model contract violation below, a dead parallel rule-application path with different error semantics, a priority field that pretends to govern ordering but doesn't, and three competing type-discrimination idioms.

Cost model: children double-counted, exponentially by depth

nodes/plan-node.ts:707,776; ~20 node constructors (filter.ts:29, join-node.ts:119, sort.ts:44, …) · design/correctness

estimatedCost is documented as self-cost-excluding-children, but constructors bake in source.getTotalCost(); getTotalCost() then adds children again — a leaf under d operators is weighted 2d. QuickPick join enumeration (rule-quickpick-enumeration.ts:109) consumes this, so left-deep vs bushy comparisons are systematically distorted. Pick one convention; enforce with a validator check. Also: getTotalCost() is uncached recursion invoked in constructors → O(depth²) plan construction, re-paid on every withChildren re-mint across three passes. Memoize (nodes are immutable).

Dead parallel rule-application path with divergent semantics

optimizer.ts:1059–1113, framework/registry.ts:261 · cleanliness/robustness

optimizeNode/optimizeChildren are never called; they're the sole consumers of the global RuleRegistry path — whose applyRules swallows rule exceptions while the live PassManager propagates. Two near-duplicate loops with different error and ordering semantics. Delete the dead path and registry class.

Materialization advisory: swallowed errors + double analysis, fired from 12 anchors

rules/cache/rule-materialization-advisory.ts:43–76; optimizer.ts:967–1000 · robustness/performance

Catch-all return null masks advisory bugs (violates no-silent-exceptions); the probe loop transforms per-child then throws results away and re-analyzes the whole tree. Registered on 12 node types in a bottom-up pass, each firing walks the full subtree — O(n²)-flavored plan time on statement-heavy plans. Run once at plan root; let unexpected errors propagate.

priority is dead metadata on the live path

framework/pass.ts:520; optimizer.ts:117,413 · design

Live pass iterates rules in registration order; priority sorting exists only in the dead registry. Comments admit it. Either sort pass.rules by priority (making annotations real) or remove the field. Related: the ~900-line registerRulesToPasses should become a table-driven manifest with a startup assertion that priorities are non-decreasing.

Three competing type-discrimination idioms

framework/characteristics.ts:323–443 and 40 planner files · cleanliness/robustness

instanceof (179 occurrences), nodeType string checks, and duck-typed as any detectors coexist. Concrete defects: isColumnBindingProvider tests typeof method === 'string' (dead branch — methods typeof to 'function'); isAggregateFunction keys off mere property presence, so any node growing a same-named member silently acquires the capability. Pick one canonical mechanism. Also dead: PredicateAnalysis (whose TODO stub returns true — the unsafe direction) and CapabilityRegistry (registered into, never queried).

Plan-time performance: declining rules retried, heavyweight recompute per re-mint, residual O(n) scans

framework/pass.ts:508–544; nodes/filter.ts:59–165; physical-utils.ts:41, rule-async-gather-zip-by-key.ts:581 · performance

(a) Only transforming rules are marked applied — one transform on a Filter re-runs its full ~9-rule battery including the 1300-line query-rewrite matcher. Track per-(node,rule) declines. (b) FilterNode.computePhysical re-runs constraint extraction (1457-line module) 3–4× per pass re-mint and hand-builds an attr index instead of the existing cache. (c) Several findIndex-in-loop attribute scans survive despite getAttributeIndex() existing precisely to kill them.

Recursive property computation undermines the iterative-traversal guarantee

plan-node.ts:776,780,825; characteristics.ts:38 · robustness

Pass traversal is deliberately iterative to avoid stack overflow, but get physical(), getTotalCost(), visit(), and subtreeHasSideEffects are unbounded recursion over the same trees — a plan deep enough to need the depth budget still overflows the first time a rule touches .physical.

Cross-rule protocol rides untyped moduleCtx with two drifting local types

rules/retrieve/rule-grow-retrieve.ts:49 vs rules/access/rule-select-access-path.ts:1000 · robustness

IndexStyleContext declared twice with divergent field types (residualPredicate?: PlanNode vs ScalarPlanNode); consumer forced into as unknown as casts. Move to one shared module with a type guard.

Selectivity hardcoded at 0.5 while a full stats layer sits unreachable

nodes/filter.ts:50–57; stats/ · design

Histograms and OptContext.stats exist, but node-level computePhysical has no context parameter, so filters can never use them. Route selectivity through a Physical-pass rule that has OptContext rather than widening computePhysical. Also low: convergence model silently prevents rules re-firing on their own output (filter-merge loops internally; undocumented contract for new rule authors); planner/debug.ts any usage; dead OptimizationContext surface.

3. Runtime, VTab, Schema, Types, Func

Assessment. Coherent, well-documented-in-code architecture: plan-time specialization in emitters, pre-resolved collations, O(1) attribute-indexed context, MVCC memory vtab with carefully reasoned Halloween protection, end-to-end cancellation, and the strict-fork harness is a genuinely good concurrency-contract enforcement idea. Weaknesses: pervasive per-row async overhead, inconsistent iterator cleanup, a scheduler in six near-identical copies, memory-vtab specifics leaked into generic runtime.

Unconditional await per row on scalar sub-programs

runtime/emit/filter.ts:30, project.ts:39, join.ts:92 · performance

Every row pays await predicate(rctx) — one per projection column — even when the sub-scheduler returned synchronously (the common case). k microtask hops per row for a k-column projection. Check instanceof Promise and only await then; shared resolveMaybe helper.

Memory scan path stacks 3–5 async generators per row

vtab/memory/layer/safe-iterate.ts:7, scan-layer.ts:26, table.ts:249, runtime/emit/scan.ts:107 · performance

safeIterate is async function* over a fully synchronous BTree; scanLayer wraps it recursively; query() re-yields with for await … yield instead of yield*; emitSeqScan wraps again. Each layer is a promise round-trip per row. Make safeIterate sync, adapt at the boundary, yield* everywhere else.

tee()/merge() leak source iterators

runtime/async-util.ts:43–118, 238 · robustness

Early consumer break never calls srcIterator.return() — the source's finally (row slots, vtab disconnect) never runs. merge() additionally re-wraps every pending promise per loop iteration (O(sources) allocations per row, unbounded handler chains). Refcount consumers in tee; index-tagged promises + try/finally closing live iterators in merge.

CHECK truthiness misses bigint/string zero

runtime/emit/constraint-check.ts:368, deferred-constraint-queue.ts:104 · correctness

result === false || result === 0 — a CHECK evaluating to 0n or '0' passes, diverging from FilterNode/isTruthy semantics. Use isTruthy at both sites.

Per-execution and per-run overhead on prepared statements

core/statement.ts:323; runtime/emitters.ts:170; runtime/emit/scan.ts:75 · performance

(a) Plan is cached but emitPlanNode + new Scheduler rerun per execution — whole-tree closure allocation on every reuse. (b) validateCapturedSchemaObjects re-validates the entire schema snapshot on every validated-instruction run — O(instructions × snapshot), and re-fires per outer row inside un-cached NLJ inners, which also re-run module.connect/disconnect per outer row. Validate once per statement execution; cache the instruction tree keyed on the emission-context fingerprint.

Scheduler abandons in-flight promises on throw; six near-identical loops

runtime/scheduler.ts:105–438 · robustness/cleanliness

In runAsync, promises parked in instrArgs are never awaited if an earlier instruction throws → unhandled rejections (process-fatal under strict mode). And the fix must land six times: runOptimized/runWithTracing/runWithMetrics plus async twins duplicate the core loop. Sweep abandoned promises with allSettled on throw; collapse to one hook-parameterized loop.

Coordinated-commit sibling path is last-writer-wins

vtab/memory/layer/manager.ts:453–479 · robustness

When a sibling connection already committed a layer for the same table during coordinated commit, the pending layer overwrites it — sibling's changes vanish from the committed chain. Rebase onto the sibling head or throw BUSY.

ParallelDriver.closeAll() violates the iterator protocol it polices

runtime/parallel-driver.ts:228–249 · robustness

it.return() invoked while a next() is still pending — async generators queue this safely, but arbitrary vtab AsyncIterator implementations may not. Await the pending pull first, or document the generator requirement.

Memory-vtab specifics hardcoded in generic runtime

runtime/utils.ts:141–153 · design

getVTable special-cases vtabModuleName === 'memory' and imports memory-vtab classes into module-agnostic runtime — third-party modules can't get connection injection. Add an optional VirtualTable.adoptConnection() hook.

Smaller items

various · cleanliness/design

AND/OR can't short-circuit (eager param evaluation; expensive scalar subqueries always run — emit callback-style like CASE when non-trivial). File-wide eslint-disable no-explicit-any in emitters.ts/utils.ts; run as InstructionRun casts everywhere (generic Instruction<TArgs> would fix). Dead: shadowName ("UNWIRED / DEAD"), no-op cleanupUnreferencedLayers calling global.gc() in production path. Multi-seek dedup allocates a full BTree as a seen-set. Context-leak diagnostic checks before async execution completes — can never observe the leaks it exists for. Mixed tabs/spaces in emit/scan.ts; ~60 duplicated lines in scan-layer.ts.

4. Satellite Packages

Assessment. The store → isolation → sync chain contains the review's most dangerous findings: two key-encoding sort-order bugs at the foundation, an isolation layer overselling its guarantee, and a sync watermark protocol that permanently loses changes under ordinary packet loss. The CRDT core in quereus-sync (LWW column versions, tombstone blocking, three-phase admission, quarantine) is carefully reasoned; the failures are at the seams. quereus-plugin-leveldb is the strongest reviewed package; quoomb-cli the messiest.

Store (quereus-store)

Key encoding: mixed INTEGER/REAL prefixes break sort order

src/common/encoding.ts:110–115 · correctness

Whole numbers (3.0) route to TYPE_INTEGER (0x01), fractional to TYPE_REAL (0x02), so in a REAL PK column every whole number sorts before every fractional one. getBestAccessPlan marks range filters handled — WHERE x >= 2.5 skips x = 3. Under-fetch: silently wrong results. Fix: single order-preserving numeric encoding (SQLite4-style) or encode by declared physical type; add a mixed int/real range test.

BLOB keys: varint length prefix is not memcmp-ordered

src/common/encoding.ts:271–278 · correctness

[0x01,0x02] compares greater than [0x03] (length-first). Same handled-filter consequence: pushed blob-PK ranges return wrong rows. Escape-encode with terminator (reuse the TEXT 0x00/0x01 escaping).

UNIQUE enforcement full-scans the table per constraint per row; secondary indexes written but never read

src/common/store-table.ts:1451,636; store-module.ts:1856 · performance/design

Bulk inserts with UNIQUE constraints are O(n²) — while writers pay per-index delete+put maintenance and query() only supports PK scans. Implementing the index-scan read arm fixes both.

Codec and lifecycle drag

serialization.ts:24–49, transaction.ts:215, store-module.ts:1061,1753,1973 · performance/robustness

JSON row codec with per-call new TextEncoder(); triplicate slow bytesToHex. Failed commit fires neither onCommit nor onRollback (stats delta leaks into next txn); renameTable deletes stats; tableKey.split('.') mis-routes quoted identifiers containing dots. CREATE INDEX / ALTER PK buffer the entire table. 565-line alterTable in a 2,689-line file.

Isolation (quereus-isolation)

Documented snapshot isolation not delivered; multi-table commits torn

isolation-module.ts:106, isolated-table.ts:315,1325, isolated-connection.ts:49 · design/correctness

Reads merge the overlay with the live shared underlying table — other connections' commits visible mid-transaction; no write-write conflict detection (last flusher wins). Each table flushes in its own underlying transaction — table B failure leaves table A durably committed. Either implement (fingerprints at first touch + coordinated flush) or fix the docs in both packages.

Modified-PK set: JSON.stringify(pk) throws on bigint, ignores collation; tombstone revival skips UNIQUE checks

isolated-table.ts:391,716 · robustness

Secondary-index scans with pending changes crash on bigint PKs; NOCASE PK case-change yields duplicate rows in the merge. INSERT reviving a same-transaction tombstone returns early without checkMergedUniqueConstraints, then flushes trustedWrite: true — UNIQUE collision surfaces as INTERNAL at commit or corrupts the store.

String-parses the store's private idxStr wire format

isolated-table.ts:449,1022 · design

Parsing "idx=name(n);plan=2" couples packages through an unversioned string; unrecognized formats silently degrade to full scans. The engine is missing a small public seam: a structured index descriptor on FilterInfo/BestAccessPlanResult. UNIQUE checks here also full-scan the underlying (push an equality FilterInfo instead). Only 4 test specs for an isolation layer — thin.

Sync (quereus-sync, sync-client, sync-coordinator)

Watermark advanced on fire-and-forget broadcasts → permanent change loss

sync-coordinator …/coordinator-service.ts:631–672 + sync-client.ts:319–324 · robustness

Coordinator broadcast failures are only logged; the client sets lastSyncHLC = max(hlc) for pushed batches too. Miss batch X (HLC 5), receive Y (HLC 6) → next get_changes sinceHLC=6 skips X forever. Advance the watermark only from server-ordered changes responses.

Unawaited commit handler violates the documented change-log invariant

quereus-sync/src/sync/sync-manager-impl.ts:255 · correctness

void manager.handleTransactionCommit(batch) — two rapid commits interleave; second's dedup reads can precede first's kvBatch.write(), violating the invariant collectChangesSince documents as load-bearing. Serialize through a promise chain.

Snapshots omit tombstones; drift check runs after commit

snapshot-stream.ts:72,272; change-applicator.ts:513, hlc.ts:329 · robustness

Snapshot bootstrap (including every S3 restore) clears receiver tombstones — straggler writes resurrect deleted rows, de-converging replicas. Clock-drift rejection executes after data + CRDT metadata commit — a peer 60s ahead durably lands unbeatable far-future LWW wins, then the batch "fails". Stream a tombstone chunk type; validate HLC wallTime in phase 1.

Client kills auto-reconnect on any server error message

sync-client.ts:260–268 · robustness

Coordinator sends error for transient per-request failures; the client sets intentionalDisconnect = true unconditionally — one flaky apply ends sync for the process lifetime. Related fragility: stale-socket handlers reject new connects, uncorrelated apply_result acks, silent send() drops, in-memory-only lastSentHLC (every restart re-sends full history).

Wire protocol duplicated, drifted, unversioned

sync-client/src/serialization.ts vs sync-coordinator/src/common/serialization.ts · design

Codec copies already disagree (missing-schemaMigrations tolerance, typed vs Record<string,unknown>); message unions have diverged (resume_snapshot/snapshot_chunk on one side only); no protocol version anywhere. One shared protocol module in @quereus/sync + protocolVersion handshake. Also: change-log pruning never wired into production; store-manager close/acquire race hands out closed LevelDB handles; resume_snapshot trusts an unvalidated client checkpoint; auth token in WS query string; S3 snapshot fully stringified in memory despite a streaming comment; no socket backpressure.

Plugins & UI

IndexedDB plugin diverges from its LevelDB twin behind the same contract

quereus-plugin-indexeddb/src/store.ts:156,263, manager.ts:79,191 · robustness

Iterate materializes the full result set (LevelDB streams); batch not cleared after write (reuse double-applies); failed open cached forever; racy upgrade serialization → VersionError. These stores are meant to be interchangeable — build a shared KVStore conformance suite run against LevelDB, IndexedDB, and memory.

various · cleanliness

plugin-loader: protocol allowlist sits beside rather than inside dynamicLoadModule; config channel flattens objects to JSON strings that IndexedDB plugin casts back as objects (user cache config silently ignored); web/CLI/loader each re-implement the plugin-load loop. quoomb-cli: ~70 lines of dead REPL methods, scaffold remnants, pervasive any, raw identifier interpolation into PRAGMA table_info(${tableName}), dead npm readline dep. quoomb-web: five copy-paste fetch* actions. shared-ui: exports a string constant — drop or implement.

5. Architecture & Design Assessment

What's sound

  • VTab-centric federation — deeply integrated, not a bolt-on: join-elimination rationale, getBestAccessPlan, and the module concurrency contract are all real and verified. Hidden cost: every read pays the async-iterator protocol per row even for in-memory data, and the "generous" perf sentinels won't catch gradual per-row creep (see runtime findings).
  • FD tracking — "sound by construction, best-effort complete" is the correct asymmetry, and the Key Soundness property net directly checks the thing that matters. Exemplary. Residual risk: uniqueness truth lives on three surfaces reconciled by one utility pair; any consumer bypassing fd-utils reintroduces the inconsistency.
  • Single PlanNode hierarchy + 1:1 emitter mirror — simple, testable; sideEffectMode registration enforcement is real discipline.
  • Key-based addressing — clean Third-Manifesto position with known write-amplification consequences for keyless tables. No concerns.

What's fragile

  • Attribute-based context is the strongest and most fragile part simultaneously. Stable attribute IDs are the right call, but the runtime rests on a shared mutable RowContextMap with last-set-wins semantics; the stale-parent-context shadowing footgun is documented in runtime.md and mitigated by hand-applied per-emitter discipline that differs per emitter. Every new streaming emitter is a chance to reintroduce a silent wrong-row bug. That the parallel track needed QUEREUS_FORK_STRICT plus a static allowlist of mutation sites confirms the tension between shared-mutable context and the parallelism ambitions. Recommendation: debug-mode runtime assertion for stale-context wins.
  • Scope. Load-bearing: vtab framework, planner, FD framework, view updateability (MVs, lenses, and migration all sit on it — the Round-Trip Laws suite is the right mitigation), declarative schema. Research-grade but real: lenses (implemented with prover, compiler, logic tests — publishable-research correctness surface relative to user base). Speculative: the parallel runtime track — well-built, deterministic tests are genuinely good, but its payoff exists only for high-latency federated vtabs while its cost (fork contract, connection locks) is imposed on the whole runtime. Clearest candidate for "pause until demand justifies".
  • Feature creep verdict: the core is not yet endangered — 249 sqllogic files and the property nets keep the SQL engine honest, and advanced features are additive layers. The endangered resource is maintainer attention: 800 KB+ of topic docs already drifting, and every layer added to the FD→lens chain raises the cost of changing anything beneath it.

6. Test Strategy & Tooling

Strengths (verified): 249 logic-test files; property tests including Key Soundness (Tier 1 + per-node Tier 2) and View Round-Trip Laws; seeded reproducible fuzzing with optimizer differential testing; golden plans; perf sentinels; deterministic gate-driven (not wall-clock) concurrency tests; Stryker mutation-testing infra; store-backed re-run.

No CI

repo root · process

No .github directory. Release safety is a 5-second setTimeout reminder in the root release script. test:store and test:fork-strict are excluded from default yarn test; without CI, nothing guarantees they run. Single largest risk in the project.

Coverage unevenness and tooling gaps

workspace · testing/tooling

quereus-isolation: 4 specs — thin for an isolation layer (vs sync 26, store 35). plugin-loader: no test script. quoomb-web/cli/shared-ui: vitest --passWithNoTests — effectively untested. No end-to-end sync test through the coordinator; no browser-environment execution for the IndexedDB plugin; no cross-package version-skew testing. Root build is a sequential 16-script cd-chain — composite: true is set but tsc -b project references are unused. Root yarn lint covers only @quereus/quereus while five packages actually have lint scripts (AGENTS.md's "only quereus has lint" is itself stale) and store/isolation/sync/plugins/coordinator have none. Dependency hygiene is good (six small runtime deps, pinned resolutions).
